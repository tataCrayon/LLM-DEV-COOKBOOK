# Agent 基础范式：ReAct / Plan-and-Execute / Reflexion / LATS

> 一个 Agent 拿到任务后，内部"怎么想、怎么做、怎么纠错"——这就是控制范式要回答的问题。本文梳理四种经典范式（加一种工业组合），从最简的 ReAct 到最复杂的树搜索，帮你在实际项目中做出正确的架构选型。

## 0. 开场问题

同一个任务："在 monorepo 中找到所有使用了已废弃 API 的文件，替换成新 API，确保测试通过"。

你至少有四种策略让 Agent 完成它：

1. **边想边做（ReAct）**：读一个文件→看看废弃 API→改一个→跑测试→读下一个→...
2. **先规划后执行（Plan-and-Execute）**：先扫描全部文件列出所有需要改的地方→制定替换计划→逐个执行→最后统一验证
3. **做了再反思（Reflexion）**：先试着改，改完发现测试挂了，反思"我遗漏了什么"，把反思结论记住，重新来过
4. **多路探索择优（LATS）**：同时生成三种替换方案，对每种评分，选最好的继续深入，像下棋一样搜索最优路径

四种策略都能完成任务，但成本、速度、可靠性、适用场景完全不同。选错了会怎样？用 ReAct 做 50 文件任务，Agent 到第 30 个就上下文爆了；用 LATS 做 3 文件的简单替换，花了 10 倍 token 却没比 ReAct 好多少。

## 1. ReAct：最简且最普遍的范式

### 1.1 核心机制

Yao et al. (2022) 提出的 ReAct 是一个极其简单的循环：

```
while not done:
    Thought: 我应该做什么？（自然语言推理）
    Action: 调用工具 X，参数 Y
    Observation: 工具返回了 Z
    → 回到 Thought
```

三步循环，思考和行动交替进行。模型看到上一步的 observation 后，再决定下一步做什么。

### 1.2 为什么它成了默认选项

arXiv 2604.03515（2025，Coding Agent 架构源码分类学）分析了 13 个开源 coding agent 的源码，发现 **7/13 使用 sequential ReAct 作为主控制循环**（SWE-agent、OpenHands、Codex CLI、Gemini CLI、mini-swe-agent、Cline、OpenCode）。原因简单：

- **实现最简**：一个 while 循环 + tool dispatch 就够了
- **与 tool use 原生协议天然匹配**：三大提供商的 API 本身就是 "模型输出 tool_call → 你执行 → 返回 tool_result → 模型继续" 的循环
- **通用性强**：不需要预知任务结构，对任何任务都能"边探索边做"

Codex CLI 的实际实现甚至比原版 ReAct 更简洁——没有显式的 "Thought" 步骤，直接依赖模型在 tool_call 前的隐式推理（因为现代模型足够强，不需要 prompt 模板来强制 "Let me think..."）。

### 1.3 ReAct 的工程问题

2025 年的生产实践暴露了 ReAct 的系统性缺陷：

**上下文膨胀**：每一步的 thought + action + observation 都堆积在上下文里。一个 50 步任务，光工具输出可能就占 100K+ token。这正是 1.3 篇讲的 context engineering 问题——JetBrains 的研究证明 observation masking 能压缩 52% 成本而不损失效果。

**无全局视野**：ReAct 是纯"贪心"策略——每一步只基于当前观察决策，没有全局计划。对简单任务这够用，对需要统筹的任务（"改 50 个文件但要保持一致性"）会导致前后矛盾。

**重试浪费**：Towards Data Science 的分析指出 ReAct agent "浪费了 90% 的重试"——当 Agent 遇到错误时，它会反复用相同（或相似）的策略重试，而非退一步想"我是不是方向错了"。

**无回溯能力**：一旦走错路（比如修改了不该改的文件），ReAct 没有内置的"撤销"或"回到之前的状态重新选择"机制。它只能在后续步骤中"修复前面的错误"，这经常导致修复链条越来越长。

### 1.4 现代 ReAct 的改良

生产系统不会使用原版 ReAct，而是加了各种工程补丁：

| 改良 | 做法 | 对应哪个范式的思想 |
|---|---|---|
| 上下文压缩 | Observation masking / compaction | Context Engineering |
| 预算控制 | max_steps + token budget | Loop Engineering |
| 错误分类路由 | 不同错误触发不同策略 | Reflexion 的弱形态 |
| 子任务分解 | 大任务拆成多个小 ReAct | Plan-and-Execute 的弱形态 |
| 工具权限控制 | 分阶段开放不同工具 | Phased loop（AutoCodeRover 式） |

这说明：**纯 ReAct 在工业中几乎不存在——它总是和其他范式的片段组合使用**。arXiv 论文的发现也印证了这一点："11/13 agents compose multiple primitives rather than relying on a single control structure"。

## 2. Plan-and-Execute：分离规划与执行

### 2.1 核心机制

```
Step 1 — Planning:
    分析任务 → 生成有序子任务列表
    [Task 1: 扫描所有文件找废弃 API]
    [Task 2: 对每个文件生成替换 patch]
    [Task 3: 应用 patch]
    [Task 4: 运行测试验证]

Step 2 — Execution:
    for task in plan:
        execute(task)
        if failed:
            replan(remaining_tasks, failure_info)
```

关键区别：**Planner 和 Executor 是分离的**。Planner 负责全局决策（做什么、什么顺序、资源如何分配），Executor 负责具体执行（怎么做）。这是经典的管理者/工人分离。

### 2.2 与 ReAct 的定量对比

基于 Dev.to 的实测对比：

| 维度 | ReAct | Plan-and-Execute |
|---|---|---|
| 任务完成准确率 | ~85% | ~92% |
| Token 消耗 | 2000-3000/任务 | 3000-4500/任务 |
| API 调用数 | 3-5 | 5-8 |
| 成本（GPT-4 级） | $0.06-0.09 | $0.09-0.14 |
| 响应延迟 | 快 | 慢（规划开销） |
| 复杂任务处理 | 中等 | 强 |

7% 的准确率提升换来约 50% 的成本增加——这就是核心 tradeoff。

### 2.3 什么时候用 Plan-and-Execute

- **子任务间有依赖关系**：Task B 需要 Task A 的结果
- **需要一致性保证**：50 个文件的 API 替换需要遵循同一套规则
- **长 horizon 任务**：超过 10 步、跨越多个文件/系统的任务
- **需要可审计性**：计划本身就是审计记录——人类可以在执行前 review 计划

### 2.4 Re-planning：计划不是死的

好的 Plan-and-Execute 系统不会死板地执行初始计划。LangGraph 的 Plan-and-Execute 模板在每步执行后有一个"需要重新规划吗？"的检查点。触发重新规划的条件：

- 某一步执行失败
- 执行结果与预期不符（比如发现文件数量比预估多 3 倍）
- 发现新信息改变了问题的性质

Re-planning 的成本是一次额外的 LLM 调用（给 Planner 当前进度和新信息，让它修订后续步骤）。这比 ReAct 的盲目重试高效得多。

### 2.5 工业实例

**AutoCodeRover / Prometheus**（arXiv 论文中归类为"Phased loop"）：不是简单的两阶段，而是多阶段，每个阶段有不同的工具可用。比如 AutoCodeRover 先用 SBFL（Spectrum-Based Fault Localization）定位 bug，再用编辑工具修复——定位阶段不允许写文件，修复阶段不允许再做大范围搜索。

**Codex Cloud**（/goal 命令）：用户设定一个持久目标，系统维护一个可验证的终态定义 + 一个动态计划，每步执行后根据验证结果调整计划。

## 3. Reflexion：从失败中学习

### 3.1 核心机制

Shinn et al. (2023, NeurIPS) 提出的 Reflexion 给 Agent 加了一个"短期记忆"来存储失败经验：

```
for attempt in range(max_attempts):
    result = execute_task(task, reflections)
    if evaluate(result) == success:
        return result
    reflection = reflect(task, result, failure_reason)
    reflections.append(reflection)  # 累积经验
```

关键创新：**Agent 不只是重试，而是先用自然语言"反思"为什么失败了，把反思结论作为额外上下文带入下一次尝试**。这是"verbal reinforcement learning"——用语言代替梯度来更新策略。

### 3.2 与简单重试的本质差异

简单重试：失败 → 用完全相同的 prompt 再跑一次（期望随机性带来不同结果）。

Reflexion：失败 → 分析原因 → 生成"下次应该怎么做"的指导 → 带着这个指导重新执行。

区别在于信息流：Reflexion 把**失败的教训**显式地注入到后续尝试的上下文中，而非依赖模型"凭记忆自动学习"。

### 3.3 适用场景

Reflexion 最有价值的场景是**有明确验证信号但解法不唯一**的任务：

- **代码生成**：测试通过/不通过是清晰的二值信号。Agent 写了代码→跑测试→失败→反思"哪行逻辑错了"→重写。原论文在 HumanEval 上把 pass@1 从 80% 提到 91%
- **问答推理**：答案对/错可以验证。Agent 先回答→检查答案→错了→反思"我遗漏了什么前提"→重新推理
- **搜索任务**：成功条件可定义。Agent 搜索→没找到→反思"关键词不对，应该搜 X 而非 Y"→重新搜索

**不适合**的场景：无法明确验证结果的任务（如"写一段优美的诗"）——因为 evaluate 函数给不出有意义的失败信号。

### 3.4 工程实现要点

**反思的质量决定了 Reflexion 的有效性**。一个好的 reflection prompt 应该引导模型：

```python
REFLECT_PROMPT = """任务: {task}
我的尝试: {attempt_summary}
失败原因: {error_message}

分析:
1. 我具体在哪一步出了问题？
2. 根本原因是什么（不是表面症状）？
3. 下次应该怎么做才能避免这个问题？

生成一条简洁的经验教训（一句话）："""
```

**反思的累积需要管理**：如果 5 次尝试积累了 5 条 reflection，全部塞入下一次的上下文可能适得其反（互相矛盾的教训会让模型困惑）。实践中保留最近 2-3 条最相关的。

**与 Self-Consistency 的互补**：Reflexion 是纵向的（多次尝试、逐步改进），Self-Consistency 是横向的（多路并行、投票选优）。两者可以组合：先并行生成 3 个方案，选最好的；如果全部失败，进入 Reflexion 循环。

## 4. LATS：语言 Agent 树搜索

### 4.1 核心机制

Zhou et al. (2023, ICML 2024) 把蒙特卡洛树搜索（MCTS）引入 LLM Agent 决策：

```
              root (初始状态)
             /    |    \
        Action1  Action2  Action3
        /          |         \
    State1a    State2a    State3a
    Score:0.7  Score:0.3  Score:0.8  ← LLM 评估打分
       |                      |
   Action1a              Action3a    ← 选高分继续展开
       |                      |
   State1aa             State3aa
   Score:0.9            Score:0.6    ← 再评估
       ↓
    最终选择这条路径
```

四步循环（经典 MCTS）：
1. **Selection**：从根节点出发，沿高分路径向下选择
2. **Expansion**：到达叶节点时，用 LLM 生成 N 个可能的下一步（展开子节点）
3. **Evaluation**：用 LLM 或外部信号对每个新节点评分
4. **Backpropagation**：把分数向上传播，更新父节点的统计量

### 4.2 与其他范式的关系

| 范式 | 探索策略 | 回溯能力 | Token 成本 |
|---|---|---|---|
| ReAct | 贪心（每步选一） | ❌ 无 | 低 |
| Reflexion | 序列重试 | 部分（重新开始） | 中 |
| LATS | 树搜索（多路展开 + 择优） | ✅ 完整（任意节点回溯） | 高 |

LATS 本质上是把 ReAct 和 Reflexion 统一到树搜索框架里：
- ReAct 是 LATS 中每条路径上的单步操作
- Reflexion 对应 LATS 的 backpropagation（失败路径的负分向上传播，后续搜索会避开）

### 4.3 关键结果

原论文在 HumanEval（代码生成）上：
- ReAct: 78.2%
- Reflexion: 91.0%  
- LATS: **94.4%**

在 WebShop（网络购物任务）上也有类似优势。但代价是 token 消耗约为 ReAct 的 3-5 倍。

### 4.4 什么时候值得用 LATS

LATS 的高成本意味着只在特定场景值得：

- **解空间离散且有限**：比如"有 5 种可能的 fix 方案"——展开 5 个节点比盲试 5 次高效
- **有廉价的评估函数**：如果评估一个节点需要跑完整测试套件（2 分钟），LATS 的每次展开都很贵；如果评估只是"看代码是否语法正确"（毫秒级），LATS 的价值就体现出来
- **错误代价高**：金融交易、不可逆操作——多花 token 在"想清楚再做"上比"做了再改"更划算
- **任务有多个等效优解**：不是"唯一正确答案"而是"很多种做法，有些明显更好"

### 4.5 工业实践中的简化形式

完整 MCTS 在生产中很少直接用（太贵、太复杂）。arXiv 论文发现实际系统用的是简化变体：

- **Agentless**：扁平独立采样（~20 个候选方案，多数投票）——没有树结构，就是 "sample N times, pick best"
- **DARS-Agent**：深度优先 + LLM 作为 critic 选择分支——但没有分数回传
- **Moatless Tools**：完整 MCTS + 奖励回传（-100 到 +100 的评分范围）

这是一个复杂度连续谱：

```
Simple Retry → Self-Consistency → Best-of-N → Greedy Tree → Full MCTS
   ReAct          Wang 2022        Agentless     DARS        Moatless/LATS
```

生产中选择在这个谱上的位置取决于 **eval 成本 × 错误代价 / token 预算**。

## 5. 第五种范式：Generate-Test-Repair

arXiv 论文识别出的第五个控制原语，在 coding agent 中尤其常见：

```
generate(code) → test(code) → if failed: repair(code, error) → test again
```

它和 Reflexion 的区别在于：
- Reflexion 的 "reflect" 是**高层策略调整**（"我该换个思路"）
- Generate-Test-Repair 的 "repair" 是**局部修补**（"这行报错了，改这行"）

实际上这就是日常 coding 的循环——写代码→跑测试→看报错→改代码。大多数 coding agent 的内循环就是这个模式。它可以嵌套在 ReAct 或 Plan-Execute 之内。

## 6. 组合使用：真实系统的范式叠加

没有任何工业级 Agent 只用一种范式。以 Codex CLI 为例，它的实际控制流是：

```
┌─────────────────────────────────────────────┐
│  外层：Loop Engineering（调度器）            │
│  cron/event 触发 → 新 session               │
├─────────────────────────────────────────────┤
│  中层：隐式 Plan（初始化）                   │
│  读 progress file → 选下一个任务             │
├─────────────────────────────────────────────┤
│  内层：ReAct + Generate-Test-Repair         │
│  think → edit → test → fix → think → ...   │
├─────────────────────────────────────────────┤
│  横切：Context Management                    │
│  auto-compact / prefix caching               │
└─────────────────────────────────────────────┘
```

## 7. 选型决策矩阵

| 条件 | 推荐范式 | 理由 |
|---|---|---|
| 步骤少（<10）、任务简单 | ReAct | 开销最低，实现最快 |
| 步骤多（>10）、子任务有依赖 | Plan-and-Execute | 需要全局视野避免前后矛盾 |
| 有明确验证信号、允许重试 | Reflexion | 从错误中学习比盲试高效 |
| 解空间小、评估便宜、错误代价高 | LATS（或简化 Best-of-N） | 多路探索比序列试错更可靠 |
| 复杂生产任务 | 组合使用 | Plan 选任务 + ReAct 执行 + GTR 修复 |

**成本敏感度**：如果 token 预算紧张，优先用 ReAct + observation masking。如果预算充裕且追求可靠性，用 Plan-Execute + Reflexion。LATS 只在高价值 / 高错误代价场景使用。

**模型能力依赖**：

- ReAct 对模型要求最低——GPT-4o 级别就够
- Plan-and-Execute 要求模型有强分解能力——reasoning model 或 GPT-5 级别更好
- Reflexion 要求模型有自我批评能力——高质量模型的 reflection 比低质量模型有效得多
- LATS 要求模型能做评估打分——需要 LLM-as-judge 能力

## 8. 完整代码示例：Plan-and-Execute with Reflexion

```python
"""
演示：Plan-and-Execute + Reflexion 的组合架构
场景：替换 monorepo 中废弃 API 的调用
"""
import json
from dataclasses import dataclass, field
from anthropic import Anthropic

client = Anthropic()

@dataclass
class TaskPlan:
    """由 Planner 生成的任务计划"""
    tasks: list[dict]  # [{"id": 1, "description": "...", "status": "pending"}]
    overall_goal: str
    
@dataclass
class ReflectionMemory:
    """Reflexion 的经验记忆"""
    lessons: list[str] = field(default_factory=list)
    
    def add(self, lesson: str):
        self.lessons.append(lesson)
        # 只保留最近 5 条，避免上下文膨胀
        if len(self.lessons) > 5:
            self.lessons = self.lessons[-5:]
    
    def to_context(self) -> str:
        if not self.lessons:
            return ""
        return "\n<past_lessons>\n" + "\n".join(f"- {l}" for l in self.lessons) + "\n</past_lessons>"


# ===== Planner =====
def create_plan(goal: str, context: str = "") -> TaskPlan:
    """Planner：将大目标分解为有序子任务"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=2000,
        system=[{
            "type": "text",
            "text": """你是任务规划专家。将给定目标分解为有序的、可执行的子任务。
每个子任务应该：
- 足够小，单步可完成
- 有明确的完成标准
- 与其他子任务之间的依赖关系清晰
输出 JSON 数组。""",
            "cache_control": {"type": "ephemeral"}
        }],
        messages=[{"role": "user", "content": f"目标：{goal}\n\n上下文：{context}\n\n请生成执行计划（JSON数组，每项含 id, description, depends_on）："}],
        tools=[{
            "name": "submit_plan",
            "description": "提交任务计划",
            "input_schema": {
                "type": "object",
                "properties": {
                    "tasks": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "id": {"type": "integer"},
                                "description": {"type": "string"},
                                "depends_on": {"type": "array", "items": {"type": "integer"}},
                            },
                            "required": ["id", "description"]
                        }
                    }
                },
                "required": ["tasks"]
            }
        }],
        tool_choice={"type": "tool", "name": "submit_plan"}
    )
    
    for block in response.content:
        if block.type == "tool_use":
            tasks = [{"id": t["id"], "description": t["description"], "status": "pending"} 
                     for t in block.input["tasks"]]
            return TaskPlan(tasks=tasks, overall_goal=goal)
    
    return TaskPlan(tasks=[], overall_goal=goal)


# ===== Executor（内部用 ReAct）=====
def execute_task(task: dict, memory: ReflectionMemory, tools: list) -> dict:
    """执行单个子任务，使用 ReAct 循环"""
    messages = [{"role": "user", "content": f"""执行以下任务：
{task['description']}
{memory.to_context()}
完成后用 report_result 工具报告结果。"""}]
    
    max_steps = 10
    for step in range(max_steps):
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=2048,
            system="你是代码修改执行者。逐步完成任务，每步一个工具调用。",
            tools=tools + [{
                "name": "report_result",
                "description": "报告任务执行结果",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "success": {"type": "boolean"},
                        "summary": {"type": "string"},
                        "error": {"type": "string"}
                    },
                    "required": ["success", "summary"]
                }
            }],
            messages=messages,
        )
        
        messages.append({"role": "assistant", "content": response.content})
        
        if response.stop_reason == "end_turn":
            return {"success": False, "summary": "Agent 未显式报告结果", "error": "no_report"}
        
        # 处理工具调用
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                if block.name == "report_result":
                    return block.input  # 任务完成
                # 执行其他工具
                result = run_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })
        
        if tool_results:
            messages.append({"role": "user", "content": tool_results})
    
    return {"success": False, "summary": "超过最大步数", "error": "max_steps_exceeded"}


# ===== Reflector =====
def reflect_on_failure(task: dict, result: dict, memory: ReflectionMemory) -> str:
    """Reflexion：分析失败原因并生成经验教训"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=500,
        messages=[{"role": "user", "content": f"""任务失败，请分析原因。

任务：{task['description']}
结果：{json.dumps(result, ensure_ascii=False)}
已有教训：{memory.to_context()}

用一句话总结：下次遇到类似情况应该怎么做？（不要重复已有教训）"""}]
    )
    lesson = response.content[0].text.strip()
    memory.add(lesson)
    return lesson


# ===== Re-planner =====
def replan_if_needed(plan: TaskPlan, completed: list, failed_task: dict, failure_info: str) -> TaskPlan:
    """判断是否需要重新规划，如果需要则生成新计划"""
    remaining = [t for t in plan.tasks if t["status"] == "pending"]
    
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1000,
        messages=[{"role": "user", "content": f"""目标：{plan.overall_goal}
已完成：{json.dumps(completed, ensure_ascii=False)}
刚失败的任务：{json.dumps(failed_task, ensure_ascii=False)}
失败原因：{failure_info}
剩余计划：{json.dumps(remaining, ensure_ascii=False)}

问题：原计划是否需要调整？
如果需要，输出新的剩余任务列表（JSON）。
如果不需要（只是执行问题，重试即可），输出 "NO_REPLAN"。"""}]
    )
    
    text = response.content[0].text
    if "NO_REPLAN" in text:
        return plan
    
    # 尝试解析新计划...（简化处理）
    return plan


# ===== 主编排循环 =====
def run_plan_execute_reflect(goal: str, tools: list, max_retries: int = 3):
    """完整的 Plan-Execute-Reflect 循环"""
    
    # Phase 1: Planning
    print("[Planning] 生成执行计划...")
    plan = create_plan(goal)
    print(f"  计划包含 {len(plan.tasks)} 个子任务")
    
    memory = ReflectionMemory()
    completed = []
    
    # Phase 2: Execute with Reflexion
    for task in plan.tasks:
        if task["status"] != "pending":
            continue
        
        print(f"\n[Executing] Task {task['id']}: {task['description']}")
        
        for attempt in range(max_retries):
            result = execute_task(task, memory, tools)
            
            if result.get("success"):
                task["status"] = "completed"
                completed.append(task)
                print(f"  ✓ 成功 (attempt {attempt+1})")
                break
            else:
                print(f"  ✗ 失败 (attempt {attempt+1}): {result.get('error', 'unknown')}")
                
                if attempt < max_retries - 1:
                    # Reflexion: 反思并学习
                    lesson = reflect_on_failure(task, result, memory)
                    print(f"  💡 教训: {lesson}")
                    
                    # 考虑是否需要 replan
                    plan = replan_if_needed(plan, completed, task, lesson)
        else:
            # 所有重试都失败
            task["status"] = "failed"
            print(f"  ⚠️ Task {task['id']} 最终失败，跳过")
    
    # Phase 3: Summary
    success_count = sum(1 for t in plan.tasks if t["status"] == "completed")
    print(f"\n[Summary] {success_count}/{len(plan.tasks)} 任务完成")
    return plan


def run_tool(name: str, input_data: dict) -> str:
    """工具执行占位（实际连接文件系统/测试框架）"""
    return json.dumps({"status": "ok", "output": f"executed {name}"})
```
