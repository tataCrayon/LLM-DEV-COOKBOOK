# 8.2 可靠性设计：Retry / Circuit Breaker / Idempotency / Graceful Degradation

> 8.1 选好了部署模式，本篇把"模式 B/C/D 在生产中怎么不死"讲透。Agent 的可靠性 = 传统分布式可靠性 + LLM 特有不确定性 + 高昂副作用代价 三者叠加。

---

## §0 真问题：单点失败如何引发雪崩

复盘一次真实事故（综合多个团队 2025 公开总结）：

```
T+0    OpenAI us-east 区 5xx 率从 0.1% 飙到 8%
T+1m   客户端代码无脑 retry 3 次，所有 in-flight 请求放大 4×
T+2m   client-side 并发飙升，单 Pod connection pool 耗尽
T+3m   K8s liveness probe 失败，Pod restart，所有 in-memory 任务丢失
T+4m   任务队列堆积 20w 条，worker fan-out 重试，Provider 限流升级
T+6m   级联打垮 Vector DB（每次重试都走全链路）
T+12m  团队手动改配置降级到 Claude，部分恢复
T+38m  OpenAI 自愈，但队列积压还需 2 小时消化
事故等级：P0；财务损失 = LLM 浪费 $3,800 + 误发邮件赔付 $7,200
```

可看出 4 类典型问题同时发生：

1. **Retry storm**：固定次数重试，无指数退避无熔断，把瞬时故障放大成持续过载；
2. **无幂等性**：重试期间触发了重复副作用（发邮件、扣款、写订单）；
3. **无降级路径**：主模型挂 = 整个 Agent 挂，没有"降级模型 / 降级工具 / 缓存 / 模板"梯度；
4. **无熔断**：客户端继续狂打已经在着火的 Provider，自取灭亡。

工业界对 Agent 可靠性的共识公式：

```
可靠性 = 幂等 × 重试控制 × 熔断 × 限流 × 降级 × 超时 × 观测
```

7 项缺一即"看起来还行，关键时刻翻车"。Maxim AI 的 [Production Guide](https://www.getmaxim.ai/articles/retries-fallbacks-and-circuit-breakers-in-llm-apps-a-production-guide/) 把前三者称为 LLM 应用的"三件套"，本文在此基础上扩展到 7 个维度。

## §1 第一性原理：错误三分类决定一切

```
                  ┌─ Transient (瞬时) → Retry
Error ─类型识别─┼─ Capacity (容量/限流) → Retry + Backoff + Circuit Break
                  ├─ Persistent (永久) → Fail Fast, No Retry
                  └─ Semantic (语义错误) → Fix Prompt, No Retry
```

| 错误码/类型 | 分类 | 处理 |
|---|---|---|
| 429 | Capacity | 指数退避 + jitter + 尊重 `Retry-After` |
| 500 / 502 / 503 / 504 | Transient | 指数退避 + jitter |
| Network timeout / connection reset | Transient | 同上 |
| 400 / 401 / 403 / 404 | Persistent | **不要重试**，直接错误日志 + 上抛 |
| 422（Pydantic / schema 失败） | Semantic | 不要简单重试；重写 prompt 或 self-correction（详见 §6） |
| Tool 报"订单不存在" | Persistent | 业务错误，不要重试 |
| LLM 输出格式错 | Semantic | structured output / function calling 强约束（见 1.6） |

**Anti-rule**："所有错误都 retry 3 次"。这条是大多数 Agent 故障的根因。

## §2 Retry：指数退避 + Jitter，是底线不是上限

```python
import asyncio, random, httpx
from typing import Awaitable, Callable, TypeVar

T = TypeVar("T")
RETRYABLE_STATUS = {408, 425, 429, 500, 502, 503, 504}
PERMANENT_STATUS = {400, 401, 403, 404, 422}

class PermanentError(Exception): ...

async def retry_with_backoff(
    fn: Callable[[], Awaitable[T]],
    *,
    max_attempts: int = 5,
    base: float = 0.5,
    cap: float = 30.0,
    timeout_per_call: float = 30.0,
) -> T:
    last_exc = None
    for attempt in range(max_attempts):
        try:
            return await asyncio.wait_for(fn(), timeout=timeout_per_call)
        except httpx.HTTPStatusError as e:
            code = e.response.status_code
            if code in PERMANENT_STATUS:
                raise PermanentError(code) from e
            if code not in RETRYABLE_STATUS:
                raise
            # 优先 Retry-After，否则 exponential + decorrelated jitter
            retry_after = e.response.headers.get("Retry-After")
            if retry_after:
                delay = float(retry_after)
            else:
                delay = min(cap, base * (2 ** attempt))
                # decorrelated jitter (AWS 推荐，比 full jitter 收敛快)
                delay = random.uniform(base, delay * 3)
            last_exc = e
            await asyncio.sleep(delay)
        except (httpx.TimeoutException, httpx.ConnectError) as e:
            delay = min(cap, base * (2 ** attempt))
            delay = random.uniform(base, delay * 3)
            last_exc = e
            await asyncio.sleep(delay)
    raise last_exc
```

**关键决策点**：

- `max_attempts` 控制在 **3-5**，超过则用 fallback 而非更多 retry；
- **jitter 必加**：固定间隔重试会让所有失败 client 同步重试，形成"惊群"二次打击；
- **per-call timeout 必加**：没有超时的 retry 等于把雪崩拖长；
- **respect Retry-After**：Provider 已告诉你多久后再试，强制覆盖本地退避；
- **idempotency_key 透传**：retry 必须带上同样的幂等键（见 §4），否则不是 retry 而是"重复下单"；
- **不要在 client 和 gateway 双重 retry**：典型放大错误。生产做法是 client 只做 1 次，重试全部交给 gateway。

## §3 Circuit Breaker：保护下游、保护自己

熔断三态：

```
        失败率 > 阈值
CLOSED ────────────► OPEN
   ▲                   │
   │ 半开期成功 N 次     │ 等待 cooldown
   │                   ▼
   └─── HALF_OPEN ◄────┘
```

参数建议（生产经验值）：

| 参数 | 推荐 | 说明 |
|---|---|---|
| `failure_threshold` | 5 次或 50% 错误率 | 滚动窗口内 |
| `rolling_window` | 60 秒 | 太短抖动多，太长反应慢 |
| `open_timeout` | 30 秒 | OPEN 状态持续 |
| `half_open_max_calls` | 3-5 | 半开期试探请求数 |
| `success_threshold` | 2 次 | 成功多少次关闭 |
| `minimum_throughput` | 20 | 滚动窗口内最少多少调用才统计失败率（防止低流量误判） |

**粒度**：必须按 **(provider, model, endpoint)** 维度独立熔断，不能"OpenAI 整个 provider"一锅端。

```python
import time
from dataclasses import dataclass, field

@dataclass
class CircuitBreaker:
    failure_threshold: float = 0.5
    minimum_throughput: int = 20
    rolling_window: float = 60.0
    open_timeout: float = 30.0
    success_threshold: int = 2
    state: str = "CLOSED"
    samples: list = field(default_factory=list)  # [(ts, success_bool), ...]
    opened_at: float = 0
    half_open_successes: int = 0

    def _gc(self):
        now = time.time()
        self.samples = [s for s in self.samples if now - s[0] < self.rolling_window]

    def allow(self) -> bool:
        now = time.time()
        if self.state == "OPEN":
            if now - self.opened_at >= self.open_timeout:
                self.state = "HALF_OPEN"
                self.half_open_successes = 0
                return True
            return False
        return True

    def record(self, success: bool):
        now = time.time()
        if self.state == "HALF_OPEN":
            if success:
                self.half_open_successes += 1
                if self.half_open_successes >= self.success_threshold:
                    self.state = "CLOSED"
                    self.samples.clear()
            else:
                self.state = "OPEN"
                self.opened_at = now
            return
        self.samples.append((now, success))
        self._gc()
        if len(self.samples) >= self.minimum_throughput:
            fail_rate = sum(1 for _, s in self.samples if not s) / len(self.samples)
            if fail_rate >= self.failure_threshold:
                self.state = "OPEN"
                self.opened_at = now
```

**反模式**：

- 单一全局熔断器把所有 LLM 调用打包 → 一个 endpoint 不健康连累全部；
- `minimum_throughput` 没设，低流量场景一次失败就熔断；
- 熔断打开时直接返回 5xx 给用户 → 应该走 §6 的降级路径。

## §4 幂等性：Agent 时代最被低估的可靠性基础

Agent 与传统 RPC 服务的最大区别：**它会自己重试，且每次重试代价高昂**（LLM 钱 + 工具副作用）。**所有可能被重放的步骤必须幂等**。

### 4.1 三类幂等需求

| 步骤类型 | 幂等性 | 实现 |
|---|---|---|
| LLM 调用 | 内容上**不强求**（输出本就随机），但成本上要去重 | 调用前查 trace cache：`(task_id, step_id, prompt_hash)` 命中则直返 |
| 工具调用（读） | 天然幂等 | 不必处理 |
| 工具调用（写） | **必须**幂等 | idempotency_key = `(task_id, step_id)` |
| Agent step transition | 必须幂等 | 用 outbox pattern 写状态机 |

### 4.2 idempotency_key 设计

```python
def make_idempotency_key(task_id: str, step_id: str, params_hash: str) -> str:
    # task_id 隔离任务；step_id 隔离任务内步骤；params_hash 防止 prompt 改了但 step_id 没变
    return f"{task_id}:{step_id}:{params_hash[:8]}"

await stripe.PaymentIntents.create(
    amount=amount, currency="usd",
    metadata={"task_id": task_id},
    idempotency_key=make_idempotency_key(task_id, step_id, params_hash),
)
```

**严禁**：
- `idempotency_key = uuid4()` —— 等于没设；
- `idempotency_key = task_id` —— 任务内多步会冲突；
- 没有 `params_hash` —— Agent 中途改了 prompt 但 step_id 没变，会复用旧结果。

### 4.3 工具侧没原生幂等怎么办

很多内部业务 API 没设计 idempotency_key。三种补救：

1. **在 Agent gateway 加一层**：用 `(idempotency_key, response)` 写 Redis，TTL 24h，命中直接返回缓存；
2. **业务侧加去重表**：DB 上一张 `executed_actions(idempotency_key PK, executed_at, result)`，写入前 INSERT-or-skip；
3. **配合人类回路（HITL）**：高风险操作不重试，进入待审队列让人类决定（详见 6.3）。

### 4.4 Outbox pattern：状态与副作用的原子化

Agent 写状态机时容易出现"DB 写成功但消息没发 / 消息发了但 DB 没写"。Outbox 把消息发送转成 DB 事务的一部分：

```
BEGIN TX
  UPDATE task SET step=42, status='paid' WHERE id=:t;
  INSERT INTO outbox(event_type, payload) VALUES ('payment.done', :p);
COMMIT
```

后台 outbox-poller 读未发事件 → 发到 Kafka/SQS → 标记 sent。挂掉重启不丢消息也不发重复（消息 id 是 event 主键）。

## §5 限流：基于 token 而非 request

LLM 调用的限流和传统 API 不一样：[TrueFoundry](https://www.truefoundry.com/blog/rate-limiting-ai-agents-preventing-llm-api-exhaustion) 指出三层防护是工业界共识：

### Layer 1：Token Bucket（请求量）

**粒度**：建议 `(tenant, workload, model)` 三元组，而非只按 user。否则一个失控的脚本会卡住该用户所有合法工作。

```python
# Redis Lua 原子扣减（生产实测可用）
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])  -- tokens per second
local now = tonumber(ARGV[3])
local cost = tonumber(ARGV[4])

local data = redis.call("HMGET", key, "tokens", "ts")
local tokens = tonumber(data[1]) or capacity
local last = tonumber(data[2]) or now
tokens = math.min(capacity, tokens + (now - last) * refill_rate)
if tokens >= cost then
    tokens = tokens - cost
    redis.call("HMSET", key, "tokens", tokens, "ts", now)
    redis.call("EXPIRE", key, 3600)
    return 1
end
redis.call("HMSET", key, "tokens", tokens, "ts", now)
return 0
```

### Layer 2：Cost Velocity Circuit Breaker（成本异常）

Token bucket 看请求数，但一个长 prompt 可能比 100 个短 prompt 还贵。**对 LLM 网关而言，预算才是真正稀缺资源**：

```python
def estimate_cost(req: LLMRequest, model_pricing: dict) -> float:
    in_tokens = count_tokens(req.messages)
    out_tokens_max = req.max_tokens or 1024
    pricing = model_pricing[req.model]
    return in_tokens * pricing["input"] + out_tokens_max * pricing["output"]

# 滚动 5 分钟成本，相对预算 10× 则熔断
if rolling_cost_5min(tenant) > 10 * tenant.budget_per_5min:
    raise CostVelocityCircuitOpen()
```

**TrueFoundry 强调**：成本阈值要 **相对于预算**（如 "10× planned rate"）而非静态美元值，以适应模型价格变化。

### Layer 3：Spike Detection（单次异常）

Agent 失控经典症状：context 单调膨胀，每步 prompt 翻倍。一个真实案例：4000 token 起始，每步翻倍，第 5 步 128000 token，单步成本 32×。

```python
if request_cost > 50 * tenant.avg_request_cost:
    raise SuspiciousSpike(f"cost {request_cost:.4f} >> avg {tenant.avg_request_cost:.4f}")
```

### 客户端 vs 网关 vs Provider

| 层 | 谁限流 | 干啥 |
|---|---|---|
| Provider | OpenAI/Anthropic | 绝对配额，组织级 |
| Gateway | 你的 LiteLLM/Bifrost/自建 | 按租户、按工作负载细粒度 |
| Client | 业务代码 | 一般不做，简化代码 |

**反模式**：双重限流（client + gateway 都做），导致 quota 浪费、调试困难。**统一在 gateway 做**。

## §6 Graceful Degradation：四级降级阶梯

[Zylos 2026 研究](https://zylos.ai/research/2026-02-20-graceful-degradation-ai-agent-systems/) 提出四级响应：

| 级别 | 触发 | 行动 | 响应时间预算 |
|---|---|---|---|
| L1 | 主模型 429 / 低置信度 | 切到同 provider 的便宜/小模型 | <2s |
| L2 | 整个 provider 不可用 | 跨 provider fallback | <10s |
| L3 | 多个 provider 失败 / 工具不可用 | 工具降级（缓存、规则、人工） | <30s |
| L4 | 系统级故障 | 入队 + 通知用户重试 + 告警 | 立即返回兜底 |

### 6.1 模型降级链

```python
MODEL_FALLBACK = {
    "task_routing": ["gpt-5", "gpt-5-mini", "claude-sonnet-4.5", "gpt-4o-mini"],
    "summarization": ["claude-haiku", "gpt-4o-mini", "local-llama-3-8b"],
    "code_gen": ["claude-opus-4.5", "claude-sonnet-4.5", "gpt-5"],
}

async def call_with_fallback(task: str, prompt: str):
    chain = MODEL_FALLBACK[task]
    for model in chain:
        try:
            if not breaker[model].allow():
                continue
            resp = await retry_with_backoff(lambda: llm.call(model, prompt))
            breaker[model].record(True)
            return resp
        except Exception:
            breaker[model].record(False)
            continue
    return template_fallback(task, prompt)
```

**关键工程点**：

- **跨 provider 降级才能扛区域性故障**：纯模型降级在 provider 整体挂时无效；
- **保留行为一致性**：fallback 模型的 output 格式可能不同，下游解析必须容错（structured output 在这里救命）；
- **fallback 链宜短**（≤3）：太长会引入额外失败模式与延迟尾巴。

### 6.2 工具降级表

按能力分级——必需/增强/可选——给每一层定义"降级行为"：

| 能力 | 等级 | 降级行为 |
|---|---|---|
| 语言推理 | 必需 | 始终可用（本地模型兜底） |
| 知识检索 | 必需 | 用训练知识 + 显式不确定性标注 |
| 实时网搜 | 增强 | 禁用，提示信息可能过时 |
| 向量检索 | 增强 | 回退到 BM25 / 关键词搜 |
| 代码执行 | 增强 | 推理但不执行 |
| 数据库查询 | 增强 | 用带 staleness 标注的缓存 |
| 图像生成 | 可选 | 跳过并说明 |

### 6.3 模板与缓存降级

最末端：常见问题的 deterministic template、语义缓存命中（即使陈旧也要标注）。

**Anti-rule**：silent degradation —— 用户必须**知道**当前在降级模式（"实时数据暂不可用，以下基于昨日缓存"），否则后续误信导致更大损失。

### 6.4 Self-correction：语义错误的特殊处理

Schema 不匹配、工具调用参数错——retry 同 prompt 没用。三种做法：

1. **Structured output / function calling**：从源头杜绝（见 1.6）；
2. **Reflection loop**：把错误信息加回 prompt 让 LLM 自己改；上限 1-2 次，否则陷死循环；
3. **降级到更强模型**：小模型搞不定的语义任务，临时切到大模型。

```python
async def call_with_self_correction(prompt: str, schema: BaseModel, max_fix: int = 1):
    for attempt in range(max_fix + 1):
        try:
            raw = await llm.call(prompt)
            return schema.parse_raw(raw)
        except ValidationError as e:
            if attempt >= max_fix:
                raise
            prompt += f"\n上次输出报错：{e}\n请按 schema 重新输出。"
```

## §7 超时与隔离

### 7.1 多层超时预算

```
client → API GW → app → LLM gateway → Provider
  30s     25s     22s      20s          18s
```

每一层 **比下一层多 2-5s**，避免下游还在等而上游已超时浪费 token。LLM 调用本身建议 **per-call timeout 30-60s**（含 streaming TTFT），**total timeout = 单步 × 最大步数 + buffer**。

### 7.2 Bulkhead（舱壁）

不同优先级的 Agent 任务用独立资源池：

```python
# 三个独立 connection pool + 独立 semaphore
pools = {
    "interactive": httpx.AsyncClient(limits=httpx.Limits(max_connections=200)),
    "batch":       httpx.AsyncClient(limits=httpx.Limits(max_connections=50)),
    "background":  httpx.AsyncClient(limits=httpx.Limits(max_connections=20)),
}
sems = {
    "interactive": asyncio.Semaphore(100),
    "batch":       asyncio.Semaphore(20),
    "background":  asyncio.Semaphore(5),
}
```

**目的**：后台批跑卡 LLM 时，不会把交互对话的连接池也吃光（8.1 §7 反模式直接对应）。

### 7.3 Hedged requests（对冲请求）

延迟敏感场景：发主请求 + 200ms 后发副本，任一回来即采纳，取消另一个。

```python
async def hedged(coro1, coro2, hedge_delay=0.2):
    t1 = asyncio.create_task(coro1)
    try:
        return await asyncio.wait_for(asyncio.shield(t1), hedge_delay)
    except asyncio.TimeoutError:
        t2 = asyncio.create_task(coro2)
        done, pending = await asyncio.wait({t1, t2}, return_when=asyncio.FIRST_COMPLETED)
        for p in pending: p.cancel()
        return done.pop().result()
```

适用：低 RPS、对 P99 敏感的接口。**高 RPS 慎用**——成本翻倍。

## §8 可观测性指标

可靠性配套的 SLO 指标（与 7.3 联动）：

| 指标 | 目标 | 告警 |
|---|---|---|
| Task success rate | ≥99% | <97% |
| Step retry count p99 | ≤2 | >5 |
| Circuit breaker open events / hour | 0 | >3 |
| Idempotent dedup hits / 1k req | <10 | >100（说明上游重试过激） |
| Fallback model invocation % | <5% | >20% |
| Cost velocity 5min / budget | <2× | >5× |
| Stale cache served % | <1% | >5%（说明实时工具长时间挂） |

**关键看板**：除了基础指标，必须有"degradation tree"看板：每一级降级触发了多少次，每次持续多久。

## §9 反模式

| 反模式 | 后果 |
|---|---|
| Client + Gateway 双重 retry | 错误放大 9× |
| 固定间隔 retry 无 jitter | 惊群、二次打击 |
| `max_retries=10` | 平均延迟拖到 30s+，Provider 看你像 DDoS |
| 永久错误（400/404）也 retry | 浪费 token、误把业务错当临时故障 |
| 全局一个熔断器 | 单 endpoint 故障导致全模型停摆 |
| 工具调用不带 idempotency_key | 重试导致重复扣款 |
| Idempotency_key = uuid4() | 等于没设 |
| 单层限流只看 request 数 | 长 prompt 撑爆预算 |
| Silent degradation | 用户在不知情下接受劣化输出 |
| Fallback 链 6 级 | 延迟尾巴爆炸，每级都带 retry |
| Reflection loop 不限轮数 | 死循环烧钱 |
| 没分 bulkhead | 后台任务把对话挤超时 |
| 超时层级倒挂 | 上游超时但下游仍在烧 token |

---

## §10 S0 → S6 路线

| 阶段 | 关键里程碑 |
|---|---|
| S0 | 没 retry / 没 timeout / 一挂全挂 |
| S1 | 加 retry + exponential backoff + jitter + per-call timeout |
| S2 | 加 idempotency_key（最大单笔收益） |
| S3 | 加 circuit breaker per (provider, model) |
| S4 | 加多 provider fallback + 工具降级表 |
| S5 | 加成本速度熔断 + bulkhead + outbox pattern |
| S6 | 加 self-correction + hedged request + 全链路 SLO 看板 |

**S1-S3 解决 80% 的事故**，不要直接跳 S6。

---

## §11 一页总结

把 8.2 压缩成一句话：**Agent 可靠性的本质是"识别 4 类错误（瞬时/容量/永久/语义）+ 按错误类型选 7 种工具（retry/熔断/限流/幂等/降级/超时/隔离）+ 副作用永远幂等"，绝不要"所有错误 retry 3 次"。**

承接关系：
- **8.1**：模式 B/C/D 在生产中怎么不死靠本章；
- **8.3 成本**：成本速度熔断是预算治理的核心；
- **8.4 延迟**：超时预算分层和 hedged request 在这里详谈；
- **6.3 权限**：高风险工具走 HITL 而非重试；
- **6.7 Kill Switch**：熔断 + Kill Switch 是同一谱系（自动 vs 手动）；
- **7.3 可观测**：可靠性指标必须 SLO 化；
- **3.4 Self-correction**：与 §6.4 联动。
