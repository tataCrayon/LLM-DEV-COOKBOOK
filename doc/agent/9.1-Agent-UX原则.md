# 9.1 Agent UX 原则：透明性 / 可预期性 / 可控性

> "信任不是默认存在的——它是 Agent 在每一步交互中累积或消耗的货币。"

## §0 真问题：为什么用户抛弃你的 Agent，不是因为它不聪明

四个典型故障场景。

**场景 A**：一个企业 Agent 帮用户生成合同摘要。用户点了按钮，然后……60 秒后冒出一面墙的文字。没有进度、没有解释"我正在读文档"、没有中间状态。用户第 30 秒已经关掉了窗口。**Agent 不是太慢——它是沉默**。

**场景 B**：一个 coding Agent 自动 refactor 了 120 个文件。用户发现后大惊——没有确认、没有 diff 预览、没有回滚按钮。技术能力满分，但信任归零。**Agent 不是太强——它是不可控**。

**场景 C**：另一个 Agent 的 UI 把"模型正在思考"、"工具调用结果"、"最终回答"全部混在一条聊天流里。用户不知道哪些是过程、哪些是结论。**Agent 不是信息不够——它是结构不清**。

**场景 D**：SAP 的一项 107 人 conjoint 实验（[arXiv:2606.18716](https://arxiv.org/html/2606.18716v1)）发现：在 1139 次界面选择任务中，把 agent reasoning 从"隐藏"变为"可见"，用户选择该界面的概率**净增 37.79 个百分点**（p<0.05）。透明性对用户偏好的因果效应，超过了 accountability 标识（+22.46pp）和 stop/pause 按钮（+5.11pp）。

四个场景揭示一件事：Agent UX 失败的根因不是能力问题，而是**交互契约**问题。用户不信任黑箱，即使黑箱里装着 o3。

本文回答：设计 Agent 界面的底层原则是什么？怎么让用户**愿意**把控制权交给 Agent？

## §1 第一性原理：为什么 Agent UX ≠ Chatbot UX

### 1.1 交互范式跃迁

| 维度 | Chatbot | Assistant（Siri/Alexa） | Agent |
|---|---|---|---|
| 发起方 | 用户发问 | 用户指令 | 双方都可 |
| 决策主体 | 用户 | 用户 | **Agent 代决策** |
| 影响范围 | 输出文字 | 执行单步命令 | **多步自主操作** |
| 错误后果 | 重新问 | 重做 | **可能不可逆** |
| 核心 UX 挑战 | 回答质量 | 理解意图 | **控制权边界** |

**Agent 的 UX 本质**：用户把**行动权**而不仅仅是**信息权**交给了系统。这意味着 UX 必须解决三个传统界面不需要面对的问题——

1. **透明性（Transparency）**：Agent 在做什么、为什么这么做、用了什么数据？
2. **可预期性（Predictability）**：Agent 下一步会做什么、不会做什么？
3. **可控性（Controllability）**：用户随时能暂停、介入、修改、回滚？

这三根支柱撑起了全部 Agent UX 设计的地基。

### 1.2 信任经济学

Microsoft Design 2025 的 Agent UX 原则文档（[microsoft.design](https://microsoft.design/articles/ux-design-for-agents/)）提出核心洞察：**"A certain level of Agent uncertainty is expected. Uncertainty is a key element of agent design."**

用户并不需要 Agent 100% 正确——他们需要知道 Agent**知道自己什么时候不确定**。这意味着：

| 设计姿态 | 信任效应 | 示例 |
|---|---|---|
| Agent 说"我确定" + 结果错了 | 信任崩塌（overconfident = dangerous） | 幻觉输出无任何标记 |
| Agent 说"我不确定" + 请求人工确认 | 信任稳定（honest agent） | "我对这步有 60% 信心，是否继续？" |
| Agent 显示推理过程 + 用户自己判断 | 信任增长（transparent agent） | Thinking 流 + 引用来源 |
| Agent 默默做对了 | 信任微增 | 后台自动完成简单任务 |
| Agent 默默做错了 + 被用户发现 | 信任暴跌 | 自动改了文件用户没同意 |

**关键定律**：信任的消耗速度是积累速度的 5-10 倍（类似客户服务中的"一个差评需要十个好评来对冲"）。

### 1.3 三原则的相互关系

```
          ┌─────────────────────┐
          │   User Trust        │
          │   (accumulated)     │
          └──────────┬──────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
        ▼            ▼            ▼
┌──────────────┐┌──────────────┐┌──────────────┐
│ Transparency ││Predictability││Controllability│
│ (看见)        ││ (预见)        ││ (干预)        │
└──────────────┘└──────────────┘└──────────────┘
        │            │            │
        │    互相增强  │            │
        ├────────────┼────────────┤
        │            ▼            │
        │    当三者协同时            │
        │    → Trust 复利增长       │
        │    当任一缺失时            │
        │    → Trust 崩塌          │
        └─────────────────────────┘
```

- **透明性支撑可预期性**：看得见过程，才能预测下一步
- **可预期性支撑可控性**：知道下一步会做什么，才知道在哪里干预
- **可控性反哺信任**：能随时停，才敢放手

## §2 原则一：透明性——让 Agent 的决策"可见"

### 2.1 透明性的 4 个层次

不是所有信息都需要同时呈现。透明性有层次——Fuselab Creative 2026 总结为"progressive disclosure of reasoning"（[fuselabcreative.com](https://fuselabcreative.com/ui-design-for-ai-agents/)）：

| 层次 | 内容 | 呈现方式 | 触发条件 |
|---|---|---|---|
| **L0 状态** | "正在执行" / "已完成" / "等待确认" | 顶部 status badge | 始终可见 |
| **L1 计划** | 要做哪几步 | step list / progress bar | 多步任务默认展示 |
| **L2 推理** | 为什么选这个方案 | 折叠面板 / thinking 流 | 用户展开或高置信度任务隐藏 |
| **L3 数据溯源** | 用了哪些文档 / 调了哪些 API | citation / tool-use log | 用户点击详情 |

**核心原则**：默认展示 L0+L1，让 L2+L3 可达。**永远不要让 Agent 沉默超过 3 秒**。

### 2.2 "确认屏不是透明性"

Fuselab 2026 的一个尖锐批判："A confirmation screen alone is not transparency. It is a fait accompli."——给用户一个"确定执行？"按钮，不等于透明。

真正的透明是：
- **Plan-and-Execute 模式**：执行前展示计划，用户能修改任意步骤
- **Tool-use disclosure**：调了哪些外部系统、返回了什么
- **Confidence signaling**：这步我有多确定

反例：Agent 说"我将删除这些文件，确定？"——用户只看到要删的文件列表，但看不到**为什么**要删、**还有没有**其他方案。这是伪透明。

### 2.3 信心信号的设计

SAP 2026 的实验（arXiv:2606.18716）发现：binary confidence（高/低）比数值百分比更有效——"users decided faster"。

设计建议：

| 场景 | 呈现方式 | 原因 |
|---|---|---|
| Agent 很确定（>90%） | 直接执行 + 绿色 √ | 不需要打断用户 |
| Agent 中等确定（50-90%） | 黄色⚠️ + "建议执行，点击查看推理" | 给用户选择权 |
| Agent 不确定（<50%） | 红色 ? + 暂停 + 请求人工决策 | 必须升级 |

**反模式**：显示"83.7% confidence"——用户无法区分 83% 和 87% 意味着什么差别，假精度制造决策瘫痪。

### 2.4 透明性随时间衰减

arXiv:2606.18716 的访谈发现："transparency needs decrease as trust increases over time"。这和 Claude Code 的纵向数据一致——用户的 auto-approve 率从前 50 次的 20% 增长到 750 次后的 40%+（[arXiv:2604.14228](https://arxiv.org/html/2604.14228v1)）。

**设计含义**：透明性不应该是固定的，而应该是可调的：
- **新用户**：默认展示 L0-L2，所有操作要确认
- **熟练用户**：只展示 L0+L1，低风险操作自动执行
- **专家用户**：只展示异常和失败

这就是"Progressive Delegation"模式（§4 详述）。

## §3 原则二：可预期性——让用户能"预见"下一步

### 3.1 心智模型对齐

Google PAIR Guidebook 的核心论点：AI 系统最大的 UX 失败是"用户不知道它能做什么、不能做什么"（[pair.withgoogle.com](https://pair.withgoogle.com/guidebook/)）。

Agent 场景下，心智模型问题更严重——因为 Agent 的能力边界**动态变化**（取决于工具、权限、上下文）。

**设计策略**：

| 策略 | 实现 | 效果 |
|---|---|---|
| **能力声明** | 开场白展示"我能做 X/Y/Z" | 设定边界 |
| **边界标注** | Agent 说"这超出我权限/能力" | 防止过高期望 |
| **计划预览** | 执行前展示意图 | 允许用户校正 |
| **行为一致性** | 同类任务→同类处理模式 | 建立模式识别 |
| **否定确认** | 明确说"我不会做 X" | 消除顾虑 |

### 3.2 Plan-and-Execute 模式

这是 2025-2026 Agent UX 最广泛采用的模式：

```
用户请求: "帮我重构 auth 模块"

Agent 展示计划:
┌─────────────────────────────────────────┐
│ 📋 Plan (3 steps)                       │
│                                         │
│ ① 分析现有 auth 代码 (read-only)         │
│ ② 生成新的模块结构 (create 5 files)      │
│ ③ 迁移测试并验证 (run tests)             │
│                                         │
│ [✅ 批准全部] [✏️ 修改] [❌ 取消]          │
└─────────────────────────────────────────┘
```

用户能：
- 批准整个计划
- 删除某些步骤
- 添加前置步骤
- 修改某步的参数

**执行时**，每步完成标记 ✓，当前步脉动高亮，用户能随时 pause。

### 3.3 行为一致性——Agent 的"人格"

用户需要能预测 Agent 在类似场景下的行为。这要求：

- **相同类型输入 → 相同类型处理流程**（不是随机选工具）
- **错误处理一致**：同类错误→同类恢复策略（见 8.2）
- **语气一致**：不因为模型温度波动而在"严谨/随意"间摇摆
- **风险处理一致**：高风险操作总是要确认，不是有时问有时不问

Anthropic agent 设计经验（[dev.to](https://dev.to/kazuya_dev/aws-reinvent-2025-what-anthropic-learned-building-ai-agents-in-2025-aim277-16lc)）："Err on the side of being too vague rather than being too specific"——对 Agent 的指令设计也遵循一致性：**宁可行为保守一致，不要聪明但不可预测**。

### 3.4 "不做什么"比"能做什么"更重要

Microsoft Design 原则明确指出 Agent 的禁区："Agents are not designed to replace or belittle people"。

在 UX 层面，**显式的边界比隐式的能力更重要**：

| 边界类型 | UX 实现 |
|---|---|
| 权限边界 | "我没有写文件的权限，需要你授权" |
| 能力边界 | "这个任务需要人工判断，我给你准备了材料" |
| 风险边界 | "这个操作不可逆，我不会自动执行" |
| 合规边界 | "这涉及敏感数据，需要额外确认" |

用户看到清晰的边界反而更信任——因为他们知道 Agent**不会**越界。

## §4 原则三：可控性——让用户随时能"干预"

### 4.1 可控性的 5 个维度

arXiv:2606.18716 的 conjoint 实验虽然只测了 stop/pause 一个维度（+5.11pp），但深度访谈里 95% 参与者认为"humans must confirm critical business decisions"，86% 要求"ability to intervene, override, or disengage"。

完整可控性包含 5 个维度：

| 维度 | 含义 | UX 组件 |
|---|---|---|
| **Pause** | 暂停执行 | ⏸️ 按钮，当前步完成后停 |
| **Override** | 手动接管某一步 | "我自己来"按钮 |
| **Redirect** | 改变后续计划 | 计划编辑器 |
| **Undo/Rollback** | 撤销已完成步骤 | 回滚到 checkpoint |
| **Kill** | 强制终止 | 🛑 emergency stop |

### 4.2 Approval Fatigue：为什么不能每步都问

Claude Code 的数据极具说明力（arXiv:2604.14228）：**用户批准了 93% 的权限请求**——说明绝大多数确认是"点确定"的仪式。

如果每步都要确认，会发生：
- **确认变成肌肉记忆**：用户不再读内容直接 approve
- **关键确认被淹没**：真正需要审视的操作混在大量无害操作中
- **效率崩溃**：Agent 的自主性优势被确认流程吞噬

**正确的设计是分层控制**（下一节详述）。

### 4.3 分层控制模型

Claude Code 的 7 档权限光谱是个好范例。简化为 3 层：

| 层级 | Agent 自主执行 | 需要确认 | 需要详细审查 |
|---|---|---|---|
| **绿区** | 读操作、搜索、分析 | — | — |
| **黄区** | — | 创建文件、小范围修改 | — |
| **红区** | — | — | 删除、外部 API 调用、不可逆操作 |

分区依据是**后果的可逆程度**：

- 可逆 + 低影响 → 绿区（自动）
- 可逆 + 高影响 → 黄区（一键确认）
- 不可逆 or 外部影响 → 红区（强制审查+确认）

**这个分区是可定制的**——随着用户信任增长，黄区可以升入绿区（Progressive Delegation）。

### 4.4 干预成本要低

一个关键原则：**干预必须比不干预更容易**。否则用户会因为懒惰而放弃控制权——直到出事。

| 设计 | 干预成本 | 效果 |
|---|---|---|
| Stop 按钮永远可见 | 低 | 用户安心 |
| 暂停后能原地继续（不需重新开始） | 低 | 鼓励探索 |
| 修改一步不用重做全部 | 低 | 细粒度控制 |
| 撤销需要找到隐藏菜单 | 高 | 用户放弃 |
| 回滚需要理解 git | 高 | 非技术用户不可控 |

## §5 三原则在产品中的落地模式

### 5.1 对话流 vs 活动流的分离

Fuselab 2026 的关键洞察："Separating the conversation thread from the activity stream. Combining both produces an interface that fails as both."

| 流 | 内容 | 用途 | 交互 |
|---|---|---|---|
| **对话流** | 用户请求 + Agent 自然语言回复 | 沟通意图、解释 | 打字、语音 |
| **活动流** | 步骤执行、工具调用、状态变更 | 监控、审计 | 批准/拒绝/暂停 |

合并的问题：
- 工具调用结果插在对话里，破坏可读性
- 用户分不清哪些是"过程"哪些是"结论"
- 回看历史时无法快速定位某次操作

**参考实现**：Cursor 的 chat + terminal split；Claude Code 的 thinking blocks 折叠；Devin 的 browser/editor/terminal/plan 四面板。

### 5.2 六大 Agent UX 模式总结

| 模式 | 解决什么问题 | 源自 |
|---|---|---|
| **Plan-and-Execute** | 可预期性：执行前展示意图 | Fuselab 2026、Cursor |
| **Confidence Signaling** | 透明性：Agent 有多确定 | SAP 2026 实验 |
| **Progressive Delegation** | 可控性 + 效率：信任渐进 | Claude Code（20%→40% auto-approve） |
| **Conversation / Activity Split** | 透明性 + 可读性 | Fuselab 2026、Devin |
| **Tool-Use Disclosure** | 透明性：用了什么工具 | Microsoft Design |
| **Recovery Routing** | 可控性：失败时怎么修 | Fuselab 2026 |

### 5.3 错误恢复三段式

Fuselab 2026 提出的"structured error recovery"设计——Agent 报错时必须包含 3 部分：

```
┌──────────────────────────────────────────┐
│ ❌ Error: API 调用失败                    │
│                                          │
│ 📍 What happened:                        │
│    Stripe API 返回 429 Too Many Requests │
│                                          │
│ 🔍 Why:                                  │
│    30 秒内发送了 100+ 请求，超过速率限制  │
│                                          │
│ 🛠️ What to try:                          │
│    [重试 (30s 后)] [手动处理] [跳过此步]  │
└──────────────────────────────────────────┘
```

**不是** "出了点问题，请重试" 这种无信息的 toast。

## §6 信任校准：防止 Over-reliance 和 Under-reliance

### 6.1 双重风险

arXiv:2604.10530 的 Reliance-Control Framework 指出 AI 使用的两个极端：

| 风险 | 表现 | 后果 |
|---|---|---|
| **Over-reliance** | 用户盲目接受 Agent 输出 | 认知退化、错误放大、团队沟通减少 |
| **Under-reliance** | 用户拒绝使用 Agent | 生产力损失、工具投入浪费 |
| **Appropriate reliance** | 用户根据场景动态调整依赖程度 | 最佳结果 |

### 6.2 校准设计干预

| 干预 | 效果 | 来源 |
|---|---|---|
| **设计摩擦（Friction）** | 降低过度依赖，但可能降低总参与度 | Collins et al. |
| **来源标注** | 有不一致时能降低依赖（好）；一致时增加依赖（也好） | Kim et al. |
| **Disclaimer / 不确定性高亮** | 实验证明**无效**（400人实验） | Bo et al. |
| **交互模式匹配** | 新手用高控制模式、专家用低控制模式 | arXiv:2604.10530 |
| **自适应推荐** | 系统根据用户能力调整展示方式 | Ma et al. |

**核心洞察**：单纯的"免责声明"和"置信度百分比"几乎没用。真正有效的是**结构性干预**——通过工作流设计（而非提示语）来校准信任。

### 6.3 17% 问题

Claude Code 论文的一个数据引人深思：使用 AI coding 辅助的开发者在理解力测试中**低 17%**。

这不是否定 Agent，而是说明 UX 需要**主动防止认知退化**：
- 让用户看到推理，而不只是结果
- 关键决策让用户自己做（Agent 准备材料但不做判断）
- 定期让用户"手动做一次"以维持能力

## §7 设计决策矩阵：用三原则做产品决策

| 产品决策 | 透明性 角度 | 可预期性 角度 | 可控性 角度 | 建议 |
|---|---|---|---|---|
| Agent 要不要显示 thinking？ | ✅ 显示推理提升信任 | ✅ 用户能预测下一步 | 中立 | 默认折叠可展开 |
| 多步任务要不要先展示计划？ | ✅ 呈现意图 | ✅ 用户能校正 | ✅ 允许修改 | 是，Plan-and-Execute |
| 低风险操作要不要确认？ | 中立 | ❌ 反复确认降低可预期性 | ❌ 制造 approval fatigue | 否，绿区自动 |
| Agent 应该记住用户偏好吗？ | ✅ 但要让用户知道记了什么 | ✅ 行为更一致 | ✅ 用户能编辑/删除 | 是，且可审计 |
| 失败时显示技术细节吗？ | 分层：L0 给普通用户，L3 给开发者 | 一致的错误处理模式 | 提供多条恢复路径 | Progressive disclosure |
| 要不要展示其他 Agent 的参与？ | ✅ multi-agent 必须可见 | ✅ 谁负责什么 | ✅ 用户能决定哪些 agent 参与 | 是（Microsoft Design） |

## §8 反模式表

| # | 反模式 | 违反原则 | 后果 | 解药 |
|---|---|---|---|---|
| 1 | 黑箱发射（输入→长时间沉默→输出） | 透明性 | 用户 30s 后离开 | 实时 status + thinking 流 |
| 2 | 伪透明（确认屏=fait accompli） | 透明性 | 用户无法做有信息的决策 | 展示推理 + 替代方案 |
| 3 | 假精度（"83.7% 置信度"） | 可预期性 | 决策瘫痪 | 二值信号（高/低） |
| 4 | 每步都确认（Approval Fatigue） | 可控性 | 93% 盲批，关键确认被淹没 | 分层权限（绿/黄/红） |
| 5 | 沉默失败（Agent 卡死不说话） | 透明性+可控性 | 用户不知道发生了什么 | 超时→主动报告+恢复选项 |
| 6 | 能力暗示过度（"我什么都能做"） | 可预期性 | 用户期望崩塌 | 显式边界声明 |
| 7 | 对话+活动混流 | 透明性 | 用户分不清过程和结论 | 分离两条流 |
| 8 | 不可逆操作无回滚 | 可控性 | 信任归零 | Checkpoint + undo |
| 9 | 固定透明度（新手专家同一界面） | 三者都违反 | 新手困惑/专家烦躁 | 自适应透明度 |
| 10 | 单一恢复路径（"请重试"） | 可控性 | 用户被困 | 结构化 3 段式 + 多路径 |

## §9 S0 → S5 演进路线

| 阶段 | 透明性 | 可预期性 | 可控性 | 产品示例 |
|---|---|---|---|---|
| **S0** | 无状态、无过程 | 不知道能做什么 | 无暂停/回滚 | 早期 ChatGPT |
| **S1** | Loading spinner | 展示能力范围 | Stop generating | ChatGPT 2023 |
| **S2** | Tool-use 显示 | 计划预览 | 分步确认 | Cursor Tab |
| **S3** | Thinking 流 + Citation | Plan-and-Execute | 绿黄红分层 + checkpoint | Claude Code |
| **S4** | 自适应透明度 + confidence | Progressive Delegation | 用户自定义权限档位 | 成熟内部平台 |
| **S5** | Multi-agent 可见 + 审计 | 行为一致+边界清晰 | 细粒度回滚+干预即续 | 未来态 |

**你在哪？** 判断标准：用户能不能在 Agent 执行中途，**不打断不重启**地修改一步？如果不能，你还在 S0-S1。

## §10 一页总结：抄走这页就够

| 问题 | 答案 |
|---|---|
| **Agent UX 三原则？** | 透明性（看见）+ 可预期性（预见）+ 可控性（干预） |
| **三原则怎么排优先级？** | SAP 实验：透明性 > accountability > stop/pause（但三者都必须有） |
| **透明性怎么做不过度？** | 4 层 progressive disclosure：L0 状态始终可见 → L3 数据溯源按需展开 |
| **怎么防 approval fatigue？** | 分层权限（绿/黄/红），只在红区强制审查 |
| **怎么防 over-reliance？** | 结构性干预（工作流设计）>提示性干预（disclaimer）——后者实验证明无效 |
| **信任随时间怎么变？** | Progressive Delegation：auto-approve 率从 20% 到 40%+ |
| **错误怎么呈现？** | 3 段式：What happened + Why + What to try，不是"请重试" |
| **对话和操作混在一起？** | 分离 conversation thread 和 activity stream |
| **第一性原则** | Agent UX 的本质是控制权的动态协商，不是信息展示 |
