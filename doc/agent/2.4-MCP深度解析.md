# MCP（Model Context Protocol）深度解析

> 当你的 Agent 只有三把工具，把 JSON Schema 写死在 system prompt 里就行。但当工具膨胀到 30 个、300 个、分布在不同团队维护的不同服务上时，你需要的不再是"怎么调一个函数"，而是一套**工具发现、连接、鉴权、生命周期管理**的标准协议。MCP 就是这套协议。

## 0. 从一个工程事故说起

某团队构建了一个内部 AI 助手，集成了 Slack、Jira、GitHub、数据库查询四种能力。四个工具的 schema 全部硬编码在 system prompt 里，四种鉴权方式散落在不同配置文件中。某天 Jira API 升级了字段名，工具定义没同步——Agent 开始用旧字段名调用，API 返回 400，Agent 反复重试导致 rate limit 触发，连带把其他三个工具的调用也阻塞了。

根本原因：工具与 Agent 之间缺乏一个**动态发现、版本协商、独立生命周期**的中间层。MCP 的设计初衷就是填补这个空白。

## 1. MCP 是什么：一句话定位

Model Context Protocol 是 Anthropic 于 2024 年 11 月发布的开放协议，定义了 **LLM 应用（Host/Client）与外部能力提供方（Server）之间**的标准通信方式。它基于 JSON-RPC 2.0，提供工具调用、资源读取、提示模板三大原语，以及鉴权、生命周期管理、能力协商等基础设施。

2025 年 12 月，Anthropic 将 MCP 捐赠给 Linux Foundation 下属的 **Agentic AI Foundation（AAIF）**，OpenAI 和 Block 作为联合创始人，AWS、Google、Microsoft、Cloudflare、Bloomberg 作为支持成员加入治理。这标志着 MCP 从 Anthropic 内部项目升级为行业中立标准。

截至 2026 年中，MCP 生态数据：

- Python + TypeScript SDK 月下载量合计约 **9700 万**
- 注册表中有近 **2000** 个 Server 条目
- 客户端支持：Claude、ChatGPT、Gemini、Microsoft Copilot、GitHub Copilot、Cursor、Windsurf、VS Code、Zed

## 2. 架构三角色：Host / Client / Server

```
┌─────────────────────────────────────────────┐
│                   Host                       │
│  (Claude Desktop / IDE / 自研 AI 应用)       │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Client A │  │ Client B │  │ Client C │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
└───────┼──────────────┼──────────────┼────────┘
        │              │              │
   JSON-RPC 2.0   JSON-RPC 2.0   JSON-RPC 2.0
        │              │              │
  ┌─────▼─────┐  ┌────▼──────┐  ┌───▼───────┐
  │ Server:   │  │ Server:   │  │ Server:   │
  │ GitHub    │  │ Database  │  │ Slack     │
  └───────────┘  └───────────┘  └───────────┘
```

**Host** 是面向用户的 LLM 应用，负责模型交互、权限仲裁、用户体验。一个 Host 内部可以维护多个 Client 实例。

**Client** 是 Host 内部的连接器，每个 Client 与恰好一个 Server 建立 1:1 的有状态连接。Client 负责协议握手、能力协商、消息路由。

**Server** 是能力提供方，暴露工具（Tools）、资源（Resources）、提示模板（Prompts）给 Client。Server 可以是本地进程（stdio），也可以是远程服务（HTTP）。

这个三角色分离的设计意味着：Server 不知道自己被哪个模型调用、Host 可以自由替换底层模型、Client 作为中间层承担安全边界（鉴权、审计、限流）。

## 3. 协议生命周期

MCP 连接分三个阶段：

### 3.1 初始化（Initialization）

三步握手：

```jsonc
// 1. Client → Server: initialize 请求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "roots": { "listChanged": true },
      "sampling": {}
    },
    "clientInfo": {
      "name": "my-ai-app",
      "version": "1.0.0"
    }
  }
}

// 2. Server → Client: 返回自身能力
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": { "subscribe": true, "listChanged": true },
      "prompts": { "listChanged": true }
    },
    "serverInfo": {
      "name": "github-server",
      "version": "2.1.0"
    }
  }
}

// 3. Client → Server: initialized 通知
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

版本协商规则：Client 发送自己支持的最新版本，Server 如果兼容则回传相同版本号，否则回传自己的最新版本。Client 若无法处理 Server 的版本则应断开连接。

### 3.2 运行阶段（Operation）

双方只能使用在初始化阶段成功协商的能力。Client 调用 `tools/list`、`tools/call`、`resources/read` 等方法，Server 可以通过 `notifications/tools/list_changed` 通知 Client 工具列表变更。

### 3.3 关闭（Shutdown）

没有专门的关闭消息——通过传输层终止连接（关闭 stdin、断开 HTTP 连接）。

## 4. 三大原语 + 三大客户端能力

### 4.1 Server 提供的原语

| 原语 | 控制方 | 用途 | 典型场景 |
|------|--------|------|----------|
| **Tools** | 模型调用 | 可执行函数，有 inputSchema | 创建 issue、查询数据库、发消息 |
| **Resources** | 应用/用户引用 | 只读数据，URI 标识 | 文件内容、配置项、数据集 |
| **Prompts** | 用户选择 | 模板化工作流 | "代码审查"、"数据分析"模板 |

**Tools** 是最核心的原语。每个 Tool 声明 `name`、`description`、`inputSchema`（JSON Schema），Client 通过 `tools/call` 传入参数，Server 返回 `content`（文本/图片/音频块）。2025-06-18 版新增了 `outputSchema` 和 `structuredContent` 字段，允许工具返回结构化 JSON 而非仅纯文本。

**Resources** 是被动数据暴露。URI 格式如 `file:///path`、`postgres://db/table`。支持静态列表和动态模板（`weather://cities/{city}`）。Client 可以订阅资源变更。

**Prompts** 是面向用户的模板，不是给模型的 system prompt。用户在 UI 中选择一个 Prompt 模板，Client 填充参数后构造完整消息序列。

### 4.2 Client 提供的能力

| 能力 | 用途 | 典型场景 |
|------|------|----------|
| **Sampling** | Server 请求 LLM 生成 | 工具执行中需要模型判断 |
| **Roots** | Server 询问工作目录边界 | 文件系统工具需要知道根路径 |
| **Elicitation** | Server 向用户请求输入 | 需要用户确认/补充信息 |

**Sampling** 是最有争议的能力——它允许 Server 反向请求 Client 调用 LLM。Server 发送 `sampling/createMessage` 请求，包含 `messages`、`modelPreferences`（含 `costPriority`/`speedPriority`/`intelligencePriority` 三个 0-1 维度）、`maxTokens` 等参数。Client 可以修改 prompt、选择不同模型、要求用户审核。规范强调"SHOULD always be a human in the loop"。

> 注：2026-07-28 RC 版已将 Sampling 标记为 deprecated，建议新实现直接集成 LLM Provider API。这反映了社区对 Sampling 安全风险的持续担忧。

**Elicitation**（2025-06-18 新增）允许 Server 在执行中途请求用户输入。Schema 限定为扁平结构（string/number/enum/boolean），Client 自动生成表单。用户可以 accept、reject 或 cancel。

## 5. 传输层：从 SSE 到 Streamable HTTP

MCP 定义了两种传输机制：

### 5.1 stdio（本地传输）

Client 以子进程方式启动 Server，通过 stdin/stdout 交换 JSON-RPC 消息。特点：零网络开销、启动快、天然安全隔离（进程级）。适合本地工具如文件操作、Git、数据库客户端。

关键陷阱：stdout 专用于协议消息，任何 `console.log` 或 `print` 都会破坏 JSON-RPC 流。诊断信息必须写 stderr。

### 5.2 Streamable HTTP（远程传输）

2025-03-26 版引入，替代已废弃的 SSE（Server-Sent Events）双端点方案。

**为什么废弃 SSE？** 旧方案需要两个端点（`/sse` 接收 + `/messages` 发送），像"用两部电话打一通电话"——协调逻辑复杂、故障点多、连接断开后无法恢复、持久连接空闲时浪费资源、与 HTTP/2 和 HTTP/3 存在兼容性问题。

**Streamable HTTP 设计：**

- 单端点（通常 `/mcp`）处理所有请求
- 短请求走普通 request/response，长任务动态升级为流式
- 支持真正双向通信：Server 可在同一通道推送通知
- 内置会话 ID 管理，状态跨连接持久化
- 设计中的断点续传能力：连接中断后可从断点恢复

```
旧 SSE 架构：                    新 Streamable HTTP 架构：

Client ──GET──► /sse (长连接)     Client ──POST──► /mcp
Client ──POST─► /messages          │
                                    ├─ 短请求: 同步 response
                                    ├─ 长请求: 升级为 stream
                                    └─ Server push: 同通道推送
```

**向后兼容**：TypeScript SDK 1.10.0（2025 年 4 月）起支持双传输。Client 可先尝试 Streamable HTTP，失败后降级到 SSE。Server 可同时暴露 `/mcp`（新）和 `/sse` + `/messages`（旧）。

### 5.3 协议版本头

2025-06-18 起，每个 HTTP 请求必须包含 `MCP-Protocol-Version: 2025-06-18` 头。版本不匹配返回 `400 Bad Request`。若头缺失，Server 默认按 `2025-03-26` 处理以保持兼容。

## 6. 鉴权体系演进

MCP 的鉴权经历了三个阶段：

| 时间 | 机制 | 特点 |
|------|------|------|
| 2025-03 | OAuth 2.1 + 动态客户端注册（DCR） | 标准 OAuth 流程，每个 Client 向每个 Server 注册 |
| 2025-06 | MCP Server 正式归类为 OAuth Resource Server + RFC 8707 Resource Indicators | Token 绑定到特定 Server，防止跨服务重放 |
| 2025-11 | Client ID Metadata Documents（CIMD）取代 DCR 为默认 | 一个 Client 连 1000 个 Server 时不需要逐一注册 |

**本地 stdio Server** 继承 Host 应用的系统权限，不需要额外鉴权。

**远程 HTTP Server** 使用完整 OAuth 2.1 流程：

```
┌────────┐     ┌─────────────────┐     ┌──────────────────┐
│ Client │     │ Authorization   │     │   MCP Server     │
│        │     │ Server          │     │ (Resource Server) │
└───┬────┘     └────────┬────────┘     └────────┬─────────┘
    │                    │                       │
    │  1. PKCE + resource=https://mcp.example   │
    │─────────────────► │                       │
    │                    │                       │
    │  2. access_token (audience-bound)          │
    │◄─────────────────  │                       │
    │                    │                       │
    │  3. Authorization: Bearer <token>          │
    │──────────────────────────────────────────►│
    │                    │                       │
    │  4. Validate token audience matches self   │
    │                    │              ─────────│
```

关键安全要求：
- **PKCE 强制**：所有客户端必须使用 PKCE 防止授权码截获
- **Resource Indicators（RFC 8707）**：请求 token 时必须指定 `resource` 参数，将 token 绑定到目标 Server，防止 token 在不同 Server 间被重放
- **短期 token + Refresh Rotation**：public client 必须使用刷新令牌轮换

## 7. 2025-06-18 版规范重要变更

相比 2025-03-26 版，主要新增：

1. **Structured Tool Output**：`structuredContent` 字段 + `outputSchema` 声明，工具可返回 JSON Schema 校验的结构化数据，同时保留 `content` 文本用于向后兼容
2. **Elicitation**：Server 中途请求用户输入的标准化机制
3. **Resource Links in Tool Results**：工具返回中可包含 `resource_link` 内容类型，指向 URI 而非内联全部数据
4. **OAuth Resource Server 分类 + RFC 8707**：安全模型形式化
5. **协议版本头强制**：每个 HTTP 请求必须带 `MCP-Protocol-Version`
6. **JSON-RPC Batching 移除（Breaking Change）**：不再允许批量 JSON-RPC 调用
7. **`title` 字段**：人类可读的显示名称，区别于机器用的 `name`
8. **`_meta` 元数据字段**：扩展点，供实现方附加自定义数据

## 8. 完整代码示例：构建一个 MCP Server

以下用 Python（FastMCP）和 TypeScript 双语展示一个数据库查询 Server 的实现：

### Python 实现

```python
"""
MCP Server: 数据库查询服务
支持 SQL 查询、表结构浏览、查询模板
"""
from mcp.server.fastmcp import FastMCP
import sqlite3
from contextlib import contextmanager

mcp = FastMCP(
    "db-query-server",
    version="1.0.0",
)

DB_PATH = "/data/analytics.db"


@contextmanager
def get_connection():
    conn = sqlite3.connect(DB_PATH)
    conn.row_factory = sqlite3.Row
    try:
        yield conn
    finally:
        conn.close()


# ─── Tools ──────────────────────────────────────────────────────

@mcp.tool()
def query_database(sql: str, params: list[str] | None = None) -> str:
    """Execute a read-only SQL query against the analytics database.

    Args:
        sql: SELECT statement to execute. Only read operations are allowed.
        params: Optional list of positional parameters for parameterized queries.
    """
    if not sql.strip().upper().startswith("SELECT"):
        return "Error: Only SELECT queries are allowed for safety."

    with get_connection() as conn:
        cursor = conn.execute(sql, params or [])
        columns = [desc[0] for desc in cursor.description]
        rows = cursor.fetchall()

        if not rows:
            return "Query returned no results."

        # 格式化为 markdown 表格
        header = "| " + " | ".join(columns) + " |"
        separator = "| " + " | ".join(["---"] * len(columns)) + " |"
        body = "\n".join(
            "| " + " | ".join(str(row[col]) for col in columns) + " |"
            for row in rows
        )
        return f"{header}\n{separator}\n{body}\n\n({len(rows)} rows)"


@mcp.tool()
def list_tables() -> str:
    """List all tables in the analytics database with row counts."""
    with get_connection() as conn:
        tables = conn.execute(
            "SELECT name FROM sqlite_master WHERE type='table' ORDER BY name"
        ).fetchall()

        results = []
        for table in tables:
            name = table["name"]
            count = conn.execute(f"SELECT COUNT(*) as c FROM [{name}]").fetchone()["c"]
            results.append(f"- {name} ({count:,} rows)")

        return "\n".join(results) if results else "No tables found."


@mcp.tool()
def describe_table(table_name: str) -> str:
    """Get the schema (columns, types) for a specific table.

    Args:
        table_name: Name of the table to describe.
    """
    with get_connection() as conn:
        columns = conn.execute(f"PRAGMA table_info([{table_name}])").fetchall()
        if not columns:
            return f"Table '{table_name}' not found."

        lines = [f"## {table_name}\n"]
        lines.append("| Column | Type | Nullable | Default | PK |")
        lines.append("| --- | --- | --- | --- | --- |")
        for col in columns:
            nullable = "YES" if not col["notnull"] else "NO"
            default = col["dflt_value"] or ""
            pk = "✓" if col["pk"] else ""
            lines.append(f"| {col['name']} | {col['type']} | {nullable} | {default} | {pk} |")

        return "\n".join(lines)


# ─── Resources ──────────────────────────────────────────────────

@mcp.resource("db://tables")
def get_tables_resource() -> str:
    """Complete list of available database tables."""
    return list_tables()


@mcp.resource("db://tables/{table_name}/schema")
def get_table_schema(table_name: str) -> str:
    """Schema definition for a specific table."""
    return describe_table(table_name)


# ─── Prompts ────────────────────────────────────────────────────

@mcp.prompt()
def data_analysis(question: str) -> str:
    """Generate a data analysis workflow for a business question.

    Args:
        question: The business question to analyze (e.g., "What's our MoM revenue growth?")
    """
    return f"""You are a data analyst with access to the analytics database.

The user wants to answer: {question}

Follow this workflow:
1. First call list_tables to understand what data is available
2. Call describe_table for relevant tables to understand the schema
3. Formulate and execute SQL queries using query_database
4. Synthesize findings into a clear answer with supporting data

Important:
- Only use SELECT queries
- Use parameterized queries when handling user-provided values
- If a query fails, examine the error and try an alternative approach
- Present numbers with appropriate formatting (percentages, currency, etc.)
"""


# ─── 启动 ──────────────────────────────────────────────────────

if __name__ == "__main__":
    # 本地开发用 stdio；生产部署用 streamable-http
    import sys

    if "--http" in sys.argv:
        mcp.run(transport="streamable-http", host="0.0.0.0", port=3050)
    else:
        mcp.run(transport="stdio")
```

### TypeScript 实现（等效）

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import Database from "better-sqlite3";

const server = new McpServer({
  name: "db-query-server",
  version: "1.0.0",
});

const DB_PATH = "/data/analytics.db";

// ─── Tools ──────────────────────────────────────────────────────

server.tool(
  "query_database",
  "Execute a read-only SQL query against the analytics database",
  {
    sql: z.string().describe("SELECT statement to execute"),
    params: z.array(z.string()).optional().describe("Positional parameters"),
  },
  async ({ sql, params }) => {
    if (!sql.trim().toUpperCase().startsWith("SELECT")) {
      return {
        content: [{ type: "text", text: "Error: Only SELECT queries allowed." }],
        isError: true,
      };
    }

    const db = new Database(DB_PATH, { readonly: true });
    try {
      const stmt = db.prepare(sql);
      const rows = stmt.all(...(params || []));

      if (rows.length === 0) {
        return { content: [{ type: "text", text: "Query returned no results." }] };
      }

      const columns = Object.keys(rows[0] as object);
      const header = `| ${columns.join(" | ")} |`;
      const separator = `| ${columns.map(() => "---").join(" | ")} |`;
      const body = rows
        .map((row: any) => `| ${columns.map((c) => String(row[c])).join(" | ")} |`)
        .join("\n");

      return {
        content: [{ type: "text", text: `${header}\n${separator}\n${body}\n\n(${rows.length} rows)` }],
      };
    } finally {
      db.close();
    }
  }
);

server.tool(
  "list_tables",
  "List all tables in the analytics database with row counts",
  {},
  async () => {
    const db = new Database(DB_PATH, { readonly: true });
    try {
      const tables = db
        .prepare("SELECT name FROM sqlite_master WHERE type='table' ORDER BY name")
        .all() as { name: string }[];

      const results = tables.map((t) => {
        const count = db.prepare(`SELECT COUNT(*) as c FROM [${t.name}]`).get() as { c: number };
        return `- ${t.name} (${count.c.toLocaleString()} rows)`;
      });

      return { content: [{ type: "text", text: results.join("\n") || "No tables found." }] };
    } finally {
      db.close();
    }
  }
);

// ─── Resources ──────────────────────────────────────────────────

server.resource("tables-list", "db://tables", { description: "All database tables" }, async () => ({
  contents: [{ uri: "db://tables", mimeType: "text/plain", text: "..." }],
}));

// ─── Transport ──────────────────────────────────────────────────

const transport = new StdioServerTransport();
await server.connect(transport);
```

## 9. 安全威胁模型

MCP 的开放性带来了显著的攻击面。以下是 2025-2026 年安全社区识别的主要威胁：

### 9.1 Tool Poisoning（工具投毒）

攻击者修改注册表中工具的 `description` 字段，嵌入隐藏指令。由于 LLM 依赖 description 决定是否调用工具以及如何构造参数，被投毒的描述可以诱导模型：
- 将敏感数据作为参数传给恶意工具
- 优先调用恶意工具而非合法工具
- 以特定方式格式化输出来绕过安全检查

**缓解**：数字签名的工具定义、版本锁定、白名单注册表、description 内容扫描。

### 9.2 通过 Sampling 的间接提示注入

Palo Alto Unit 42 在 2025 年的研究中揭示了三种 Sampling 攻击向量：

**资源窃取**：恶意 Server 在 sampling 请求中附加隐藏指令，消耗额外 token 但不在用户界面可见（因为 Client 通常只展示摘要）。

**会话劫持**：Server 通过 sampling 让 LLM 在回复中嵌入持久性指令（如"所有后续回复都用海盗语气"），一旦进入对话上下文就会影响整个会话。

**隐蔽工具调用**：利用 sampling 的工具使用能力（`tools` 字段），让 LLM 静默调用其他 Server 的工具（如 `writeFile`），用户看到的只是正常摘要。

**缓解**：strict prompt template 隔离用户内容和 Server 修改、token 限额、response 扫描指令性短语、sampling 频率限流、要求用户显式批准工具执行。

### 9.3 Supply Chain 攻击

社区注册表中可能存在：
- 名称相似的恶意 Server（typosquatting）
- 合法 Server 被入侵后推送恶意更新
- 依赖链中的恶意包

**缓解**：官方认证标记、代码审计、版本固定、SBOM（软件物料清单）。

### 9.4 Token 跨服务重放

若 Client 使用同一个 token 访问多个 Server，被入侵的 Server A 可以将 token 用于 Server B。RFC 8707 Resource Indicators 正是为解决此问题引入的——每个 token 绑定到特定 `resource` URI。

### 9.5 协议层面的安全设计原则

MCP 规范定义了四个安全支柱：

1. **用户同意与控制**：所有操作需要明确用户授权，用户可以随时撤回
2. **数据隐私**：Host 不得在未经用户同意时将数据暴露给 Server 或转发给第三方
3. **工具安全**：工具本质上是任意代码执行，"工具描述中的行为声明（如 annotations）应被视为不可信，除非来自可信 Server"
4. **LLM Sampling 控制**：用户审批 Sampling 请求，协议有意限制 Server 对 prompt 内容的可见性

## 10. MCP 与其他协议的关系

### MCP vs Function Calling

Function Calling 是模型厂商定义的**模型-应用**接口（模型如何告诉应用"我想调工具"），MCP 是**应用-工具**接口（应用如何发现和连接外部能力）。二者是互补而非竞争关系：

```
User ─► Host ─► LLM (Function Calling) ─► Host ─► Client (MCP) ─► Server
```

模型通过 Function Calling 表达"调用 query_database"的意图，Host 收到后通过 MCP Client 将调用路由到正确的 Server。

### MCP vs A2A（Agent-to-Agent）

Google 于 2025 年 4 月发布的 A2A 协议解决的是 **Agent 之间**的协作通信。MCP 是 Agent 内部连接工具的"USB-C 接口"，A2A 是 Agent 之间的"网络协议"。一个 Agent 内部用 MCP 连工具，对外用 A2A 与其他 Agent 协作。

### MCP vs LangChain Tools / OpenAI Plugins（已废弃）

| 维度 | MCP | LangChain Tools | OpenAI Plugins |
|------|-----|----------------|----------------|
| 标准化 | 开放规范，跨厂商 | 框架特定 API | 厂商锁定（已废弃） |
| 传输 | stdio / Streamable HTTP | 进程内函数调用 | HTTP REST |
| 动态发现 | ✓ 运行时 list + subscribe | ✗ 编译时绑定 | ✓ manifest.json |
| 鉴权 | OAuth 2.1 + RFC 8707 | N/A（进程内） | OAuth |
| 生命周期 | 有状态连接 + 能力协商 | 无 | 无状态 |

## 11. 生产部署模式

### 模式一：本地工具集群（stdio）

适合 IDE 场景。Host 启动多个 Server 子进程，每个 Server 一个工具域。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    },
    "git": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-git"]
    },
    "database": {
      "command": "python",
      "args": ["./servers/db_server.py"],
      "env": { "DB_URL": "postgresql://..." }
    }
  }
}
```

### 模式二：远程 Server + Gateway

企业场景，工具以微服务形式部署，前置 API Gateway 统一鉴权、限流、审计。

```
Client ──► API Gateway (OAuth + Rate Limit + Audit Log)
               │
       ┌───────┼───────┐
       ▼       ▼       ▼
   Server A  Server B  Server C
   (CRM)    (ERP)     (Analytics)
```

### 模式三：Serverless MCP

每个工具调用触发一个 Function（如 AWS Lambda / Cloudflare Worker）。Streamable HTTP 的无状态模式天然适配。E2B 和 Modal 已提供 MCP-over-Serverless 的托管方案。

## 12. 选型决策清单

在决定是否引入 MCP 时，参考以下判断条件：

**适合引入 MCP 的场景：**
- 工具数量 > 5 且分属不同团队/服务维护
- 需要动态工具发现（工具列表运行时变化）
- 需要统一鉴权模型（特别是远程工具）
- 构建平台级 AI 产品（多租户、多模型）
- 希望工具实现与 Agent 框架解耦

**不需要 MCP 的场景：**
- 2-3 个固定工具，schema 硬编码即可
- 纯进程内工具调用（如 LangChain 的 `@tool` 装饰器直接返回结果）
- 对延迟极其敏感的场景（MCP 的协议开销约增加 5-15ms RTT）
