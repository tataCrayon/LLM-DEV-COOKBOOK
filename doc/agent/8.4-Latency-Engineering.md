# 8.4 Latency Engineering：TTFT vs E2E / Streaming / Speculative / Parallel Tool Calls / Cache 命中工程

> 8.3 把成本压了下来，本篇把延迟也压下来。Agent 的延迟有两套指标（TTFT 和 E2E），优化手段各不相同。本篇按"用户视角"和"系统视角"两套时间线拆解。

---

## §0 真问题：30 秒的"答得正确"输给 3 秒的"答得粗糙"

工业界对 Agent 延迟的两个反直觉发现：

1. **用户对"开始反馈的时间"远比"总完成时间"敏感**。Perplexity 把检索过程暴露给用户后，端到端时间没变但用户满意度显著上升（[LangChain blog](https://www.langchain.com/blog/how-do-i-speed-up-my-agent)）。
2. **Agent 串行工具调用是最大延迟黑洞**。一个 5 步 Agent，每步等 LLM 3s + 工具 1s = 串行 20s；同样任务并行后只需 6s（[LLMCompiler](https://arxiv.org/abs/2312.04511) 实测 3.7× 加速、9% 准确率反而提升）。

错误的延迟优化（典型反模式）：

- **盯着 E2E 不看 TTFT**：换更快模型把总耗时从 8s 压到 4s，TTFT 还是 2s，用户感受没变；
- **盲目并行**：把有依赖关系的工具调用强行并行，得到错误结果；
- **省 token 省错地方**：压输出 token 收益 1×，输入 token 压一半（cache）收益 4-10×；
- **没看清瓶颈**：用 LLM 流式解决了响应延迟，实际瓶颈是工具调用网络 RTT。

正确的第一性原理：

```
延迟可观测公式：

User-perceived latency = TTFT (time-to-first-token / first useful signal)
                       + Streaming throughput (后续 token 速率)
                       + Tool tail latency (P99 工具回路)
                       + Tool serialization overhead (串行回环数)

E2E latency  = Σ (LLM_inference + Tool_RTT + Network)
            ≈ Σ(prefill + decode × N_out) + Σ(tool_calls × tool_p99)
```

**两套指标各自的优化抓手**：
- **TTFT 优化** = prompt cache + 短 prefix + 流式 + speculative + 模型选型；
- **E2E 优化** = 并行工具调用 + 减少回环数 + 长输出压缩 + speculative decoding。

## §1 第一性原理：拆解延迟的 4 大组件

```
User send query  ─────────────────────────────────► User see first useful output
                  │                                       │
                  ├── (1) Prefill: input → KV cache       │ TTFT 主要在这
                  │      与输入长度成线性                  │
                  ├── (2) Decode: token-by-token          │
                  │      与输出长度成线性，每 token ~20ms  │
                  ├── (3) Tool call RTT                   │
                  │      工具的 P99 才是瓶颈              │
                  └── (4) Inter-call orchestration        │
                          串行还是并行？                  │
```

**关键数字（2026 主流模型实测）**：

| 模型 | Prefill speed (token/s) | Decode speed (token/s) | TTFT @ 10K input | Cost ratio |
|---|---|---|---|---|
| GPT-5 | ~4000 | ~100 | ~2.5s | 1.0× |
| Claude Sonnet 4.5 | ~5000 | ~80 | ~2.0s | 0.8× |
| Gemini 2.5 Flash | ~10000 | ~250 | ~1.0s | 0.1× |
| Llama-3.3 70B (vLLM) | ~20000 | ~150 | ~0.5s | self-hosted |
| Groq Llama-3.3 | ~50000 | ~500 | ~0.2s | 0.5× |

注：实际数字随负载、context 长度、batch size 变化。生产用前必跑自己的 benchmark。

## §2 TTFT 优化：用户感知的第一战线

### 2.1 Prompt Cache：TTFT 与 cost 的双重武器

8.3 已经讲了成本侧，这里讲延迟侧：

[Don't Break the Cache (arXiv 2601.06007)](https://arxiv.org/abs/2601.06007) 实测命中 prompt cache 后 TTFT 改善 **13%-31%**。原理：prefill 阶段被绕过，直接从 cached KV 进入 decode。

工程要点（与 8.3 互补）：

- **Prefix 越稳定 TTFT 越好**：cache 命中要求 prefix byte-for-byte 一致；
- **动态内容塞末尾**：当前时间、用户 ID 这种"每次都变"的字段绝对不能放前缀；
- **降级保底**：cache miss 的 P99 是 cache hit 的 3-5×，必须把"miss 也能接受"作为 SLO 设计；
- **预热**：知道接下来会用某长 prompt，提前发一次 dummy 请求 prime cache。

### 2.2 Streaming：让 TTFT 从"等结果"变成"等首词"

不是技术加速而是 UX 加速。具体技术形态：

1. **Token streaming**：SSE/WebSocket 推 LLM 输出 token，TTFT 直接降到 prefill+1 个 token 的时间；
2. **Status streaming**：推送 Agent 中间状态（"正在搜索...", "找到 3 篇相关..."），让用户看到进度而非空白；
3. **Plan streaming**：先吐出"我打算这样做（计划列表）"，再开始执行；
4. **Citation streaming**：边生成边推 inline citation，用户可以提前点开看引用。

```python
async def agent_stream(query: str) -> AsyncIterator[Event]:
    yield Event(type="status", content="理解问题...")
    plan = await llm.call("plan", query)
    yield Event(type="plan", content=plan)
    for step in plan.steps:
        yield Event(type="status", content=f"执行：{step.description}")
        async for token in llm.stream_call("execute", step):
            yield Event(type="token", content=token)
    yield Event(type="done")
```

详见 9.4 关于 Thinking 流式呈现的 UX 原则。

### 2.3 Speculative Decoding：从模型推理本身压 TTFT/decode

LLM 推理层面的两类 speculative：

**(a) Token-level speculative decoding（推理引擎层）**

用小 draft 模型生成 N 个候选 token，大模型一次性 verify，命中即批量接受。vLLM/SGLang/TGI 都已实现。典型 1.5-3× decode 加速、TTFT 影响小。**自托管必开**（详见 8.5）。

**(b) Action-level speculation（Agent 层，2025 新方向）**

[SPAgent (arXiv:2511.20048)](https://arxiv.org/html/2511.20048v1) 把 speculative 上升到 Agent 动作级：

- **激进期**：早期步骤推理需求低，跳过 reasoning 直接采样 k 个候选动作（命中率 73.4% in first step）；
- **验证期**：后期复杂步骤，主路径正常 reason-action，副路径并行执行推测动作；
- 端到端加速 **1.08-1.65×**，在线 serving 平均 24.2%；
- Action Buffer 跨请求复用工具结果，命中率约 40%。

工程门槛较高，**先实现 token-level speculative + 并行工具调用**，再考虑 action-level。

### 2.4 模型选型：直接换更快的

[LangChain blog] 强调：响应时间 ∝ 输入长度。把"快"作为一级要求时：

- **Groq / Cerebras / SambaNova**：硬件特化，开源模型 5-10× 提速；
- **Gemini Flash**：闭源最快档之一；
- **Claude Haiku / GPT-5 nano**：同代模型 mini/nano 版；
- **自托管 + vLLM/SGLang**：见 8.5。

但**模型选型不是孤立决策**，要结合 8.3 的 cascade router，把"快 model"放在 cascade 第一层，绝大多数请求 1-2s 拿结果。

### 2.5 上下文压缩

输入越长 prefill 越慢。三类压缩：

1. **LLMLingua / LongLLMLingua**：基于信息熵的 token 级压缩，2.5-5× 压缩比，质量损失可控；
2. **摘要式压缩**：长会话历史用便宜模型摘要后塞入；
3. **RAG 替代 stuff**：把 100k context 换成"top-5 chunks" 检索 + 短上下文（详见 4.x）。

## §3 E2E 优化：并行工具调用是头号红利

### 3.1 LLMCompiler：把工具调用编译成 DAG

[LLMCompiler (Kim et al. 2023)](https://arxiv.org/abs/2312.04511) 的三件套：

1. **Planner**：让 LLM 一次性输出工具调用 DAG，节点 = 工具，边 = 数据依赖；
2. **Task Fetching Unit**：拓扑分析，依赖满足即派发；
3. **Executor**：BSP/Pregel wave-based 调度，同波次并发。

报告 **3.7× 加速、6.7× 成本下降、9% 准确率提升**（vs ReAct 串行）。准确率提升的反直觉原因：**LLMCompiler 避免了 ReAct 中"看到上一步结果后改变主意"的混乱**。

### 3.2 何时并行 vs 何时串行

| 场景 | 策略 |
|---|---|
| 多个独立数据采集（API/DB/搜索） | **总是并行** |
| Fan-out / Fan-in（先散后合） | **扇出并行，聚合串行** |
| 多候选试探（speculative） | 若浪费算力 < 串行延迟则并行 |
| 真依赖链（后者读前者结果） | 不能并行，可流水线 |
| 高副作用（写、转账） | **串行**，且加 §5 锁 |

LangChain LangGraph 基于 BSP/Pregel 做"deterministic concurrency with full support for cycles"，是当前主流实现。Salesforce W&D 框架的实验数据：**每轮 3 个并行工具**在 BrowseComp/HLE/GAIA 上是最优 sweet spot；超过 3 个收益递减。

### 3.3 并行实现的 6 个陷阱

[Zylos 2026 研究](https://zylos.ai/research/2026-04-23-parallel-tool-calling-optimization-ai-agents)：

1. **波次被慢调用阻塞**：必须 per-call timeout（`asyncio.wait_for`），不要整波 timeout；
2. **上下文洪泛**：并行结果同时回填会撑爆 context；要在 reduce 阶段汇总/过滤；
3. **重试无界**：失败工具需断路器 + 指数退避（与 8.2 联动）；
4. **低并发利用率**：planner 生成的"计划"实际上 70% 是串行依赖；要监控**每波实际并发度**指标；
5. **Mock 测试失真**：用 0 延迟 mock 跑测试，发现不了并行 bug；测试中要注入真实延迟；
6. **Provider 限制**：OpenAI `tool_choice` 不支持指定并行组合；Claude 4.6 在 Batch + 大工具集时降级。

### 3.4 并行调用的代码模式

```python
import asyncio
from typing import Awaitable

async def run_wave(tasks: list[Awaitable], per_call_timeout: float = 5.0):
    """同一波次的工具并发执行，单个超时不影响其他"""
    async def _safe(coro):
        try:
            return await asyncio.wait_for(coro, per_call_timeout)
        except asyncio.TimeoutError:
            return ToolError("timeout")
        except Exception as e:
            return ToolError(str(e))
    
    return await asyncio.gather(*(_safe(t) for t in tasks))


async def agent_compiled(query: str):
    plan = await llm.plan(query)  # 输出 DAG
    results = {}
    for wave in topological_waves(plan):
        coros = [tool.call(node.params, deps=results) for node in wave]
        wave_results = await run_wave(coros, per_call_timeout=5.0)
        for node, res in zip(wave, wave_results):
            results[node.id] = res
    return await llm.summarize(results)
```

### 3.5 减少 LLM 回环

每多一次 LLM round-trip 就多一个 RTT + prefill + decode。压回环的 4 种手段：

1. **Pre-plan**：把多步规划一次性出来，减少"看一步问一次"的回合制；
2. **批量 reflect**：5 步动作执行完后做一次 reflection，而非每步都反思；
3. **Tool calling 中嵌入 reasoning**：一个 LLM 调用同时输出"思考 + 工具调用"，少一次回合；
4. **Skip-if-confident**：高置信场景直接执行，跳过 confirmation 步骤。

### 3.6 隐藏耦合检测

[Tian Pan 2026 文章](https://tianpan.co/blog/2026-04-10-parallel-tool-calls-hidden-coupling)指出一个真实陷阱：**LLM 生成的"并行"工具调用看似独立，实际有隐藏依赖**。例如：

```
parallel_calls:
  - search("California weather")
  - search("trip to California")
```

第二个查询的真正语义需要第一个的结果（"如果加州下雨，找不下雨的目的地"），但 LLM 把它写成了并行。**对策**：

- 在 schema 层让工具显式声明 `requires` 字段；
- 给 LLM few-shot 例子展示"看似并行实则串行"的反例；
- 加 post-hoc verification：跑并行后，让 LLM 二审"结果是否一致 / 是否需要重做"。

## §4 工具侧延迟：被忽视的 P99

Agent 延迟尾巴 80% 在工具侧（数据库、HTTP、外部 API），但很多人盯着 LLM 不放。

### 4.1 工具调用 RTT 优化清单

| 手段 | 收益 |
|---|---|
| HTTP keep-alive / connection pool | 单次 -50-200ms |
| HTTP/2 多路复用 | 长链路 -30% |
| 工具 client side cache（短 TTL） | 命中减少一次 RTT |
| 工具响应压缩（gzip / br） | 大 payload -50% |
| 地域亲和（Agent 与工具同 region） | 跨洋 RTT -200-500ms |
| Tool gateway 内置 fallback / circuit break | 见 8.2 |
| 批量工具调用（merged tool） | N 次 → 1 次 |

### 4.2 Tool result truncation

返回 100KB 工具结果 → LLM 输入暴涨 → prefill 慢 → TTFT 升。**所有工具结果必须在 LLM 入口做截断**：

```python
def truncate_tool_result(result: str, max_tokens: int = 2000) -> str:
    if count_tokens(result) <= max_tokens:
        return result
    head = result[:max_tokens * 2]
    tail = result[-500:]
    return f"{head}\n...[truncated]...\n{tail}"
```

### 4.3 Tool result caching

8.3 讲了 LLM 缓存，工具结果同样可缓存：

```python
@cache_with_ttl(ttl_seconds=300, key_fn=lambda req: f"{req.tool}:{hash(req.params)}")
async def call_tool(req: ToolCall):
    ...
```

注意：**写操作绝不可缓存**（与 8.2 幂等性区分）。

## §5 SLO 与监控：先量化再优化

### 5.1 关键指标

| 指标 | 目标 | 含义 |
|---|---|---|
| TTFT p50 | <800ms | 用户开始看到反馈 |
| TTFT p99 | <2.5s | 长尾用户 |
| First useful chunk | <1.5s | streaming 首段有意义内容 |
| E2E p50 | <8s (chatbot) / <60s (Agent) | 任务完成 |
| E2E p99 | <30s / <300s | 长尾 |
| Tool call P99 / tool | 每工具独立监控 | 找瓶颈工具 |
| 平均回环数 per task | ≤5 | 衡量 Agent 设计效率 |
| 平均并发度 per wave | ≥1.8 | 衡量并行设计有效性 |
| Cache hit ratio | ≥70% | 见 8.3 |

### 5.2 Waterfall 视图

LangSmith / Langfuse / Phoenix 都提供 trace waterfall。看 waterfall 时关键问题：

1. **哪些步骤可以并行但被写成了串行？**
2. **哪个工具的 P99 抬高了整体延迟？**
3. **LLM prefill 时间是否随 input 长度线性？说明 cache 没命中。**
4. **是否有空白 gap？**说明编排框架本身在等什么。

### 5.3 延迟预算分解

E2E SLO 8s 反向分配：

```
TTFT 1s + (5 工具 × 1s P99 工具) + (5 LLM × 0.6s decode) + 余量 1s = 8s
                                    │
                                    └── 串行 vs 并行决定了实际倍数
                                        并行后 5 工具 → 1.5s（max + jitter）
                                        E2E ≈ 1 + 1.5 + 3 + 1 = 6.5s
```

每项都有独立 SLO，超阈值各自告警。

## §6 实战：把延迟压一半的组合拳

某客服 Agent（综合 Anthropic、LangChain 2025 案例）：

| 阶段 | 优化 | TTFT | E2E |
|---|---|---|---|
| 基线 ReAct | - | 2.5s | 22s |
| 加 prompt cache（90% 命中） | TTFT -30% | 1.7s | 19s |
| 加 streaming | 感知 -60% | 0.5s 感知 | 19s |
| 改 LLMCompiler 并行 | E2E -50% | 0.5s | 9s |
| 加 cascade router (Haiku first) | TTFT -50% (75% 流量) | 0.3s | 7s |
| 加 tool client cache + http/2 | -20% tail | 0.3s | 5.5s |

最终：**TTFT 0.5s → 0.3s，E2E 22s → 5.5s，cost -65%**。

## §7 反模式

| 反模式 | 后果 |
|---|---|
| 盯着 E2E 不看 TTFT | 用户感觉没变好 |
| 盲目并行有依赖的工具 | 结果错乱 |
| 没看 waterfall 就优化 | 优化了非瓶颈环节，无效 |
| LLM 输出不流式 | 用户等空白 8s |
| 一次工具结果 100KB 灌给 LLM | prefill 慢 5×、token 烧 5× |
| 工具 client 不开 keep-alive | 每次新建 TCP，多 100ms |
| 并行无 per-call timeout | 一个慢工具阻塞整波 |
| 测试用 0 延迟 mock | 上线发现并行 bug |
| 跨洋 LLM + 跨洋工具 | 来回 500ms RTT 叠加 |
| 反思每一步 | 多 50% 回环 |

## §8 S0 → S6 路线

| 阶段 | 关键动作 | TTFT | E2E |
|---|---|---|---|
| S0 | 同步 ReAct，无优化 | 3s | 30s |
| S1 | 加 streaming（UX） | 感知 0.5s | 30s |
| S2 | 加 prompt cache + cache-friendly prompt | 真实 0.8s | 25s |
| S3 | 工具并发执行（gather/LangGraph wave） | 0.8s | 12s |
| S4 | LLMCompiler-style DAG planning | 0.8s | 7s |
| S5 | Cascade router + 选型优化（Haiku 兜底） | 0.4s | 5s |
| S6 | Action-level speculative + 工具 cache + http/2 | 0.3s | 3-4s |

## §9 一页总结

把 8.4 压成一句话：**Agent 延迟优化 = TTFT 与 E2E 双指标分治，TTFT 靠 prompt cache + streaming + 模型选型，E2E 靠并行工具调用（LLMCompiler DAG）+ 减回环 + 工具侧 RTT 优化；永远先看 waterfall 找瓶颈再优化，不要盲目优化非瓶颈环节。**

承接关系：
- **8.3 成本**：prompt cache 同时省钱与降 TTFT，是双赢手段；
- **8.5 模型选型**：自托管 + vLLM 是降 latency 的另一战线；
- **8.2 可靠性**：并行执行的 per-call timeout、断路器、幂等性都来自 8.2；
- **9.4 UX**：streaming 与 thinking 呈现是用户感知层；
- **3.4 工具调用**：依赖图设计的工具 schema 来自 3.4；
- **7.3 可观测性**：waterfall trace 与 SLO 看板来自 7.3。
