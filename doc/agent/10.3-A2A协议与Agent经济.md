## §0 真问题：当 Agent 数量超过 100 个，谁负责让它们对话与结算？

2026 年中，几个真实场景同时发生：

- 一家供应链公司有 200+ 内部 Agent，每个团队用不同框架（LangGraph、CrewAI、AutoGen、自研），它们要协同完成 "客户下单 → 库存 → 物流 → 财务" 全链路。
- ChatGPT Instant Checkout 上线，AI Agent 在 Etsy 卖家页面**直接代用户下单**。问题：商家如何确认是"被 AI 授权的用户购买"而不是滥用？
- API 厂商发现：付费用户是 Agent，Agent 调用频次极高、模式离散，传统订阅模型崩溃，需要按调用计费、不要 API Key、不要账户。

这三件事共同指向一个新生层：**Agent 经济基础设施**。它由两条主线组成：

1. **A2A（Agent-to-Agent 通信）**：Agent 之间如何发现、对话、协调任务。
2. **Agentic Payments（Agent 经济）**：Agent 之间或 Agent 与服务之间如何授权、签名、结算。

如果说 MCP 解决"Agent 如何调用工具"，那么 A2A + Agentic Payments 解决"Agent 如何与**别的 Agent 与服务**互动"。

> **判句**：MCP 是 Agent 的 USB-C，A2A 是 Agent 之间的 HTTP，AP2/x402/MPP 是 Agent 之间的 SWIFT/Visa。

---

## §1 第一性原理：为什么需要新协议？

直接用 HTTP REST / gRPC 不行吗？短期可以，长期会撞上五堵墙：

1. **发现性**：Agent 数量爆炸式增长，没有统一的 Agent Card / Manifest，调用方不知道对方有什么能力。
2. **会话语义**：HTTP 一问一答适合"工具调用"，不适合"长任务+中断+多轮"的 Agent 协作。Agent 任务可能跑几分钟到几小时，需要 status update / artifact streaming / input-required 中断态。
3. **跨框架**：LangGraph 内存模型 vs CrewAI vs AutoGen 各异，**不能要求双方共享内部状态**，需要一个对外的"任务 + 消息 + 工件"中立模型。
4. **可验证身份**：Agent 自己声称的"我代表 Alice 公司"不可信，需要密码学签名（Signed Agent Cards / Mandates）。
5. **机器原生支付**：API Key + 订阅模型不适合按调用 / 按 token / 按任务的细粒度计费。

> **判句**：A2A 不是又一个 REST 包装，它把 **任务（Task）、消息（Message）、工件（Artifact）** 作为一等公民，把长任务、中断、签名身份当作协议基石。

## §2 A2A 协议：技术骨架

[A2A v1.0](https://a2a-protocol.org/latest/specification/) 已由 Google 捐献给 Linux Foundation（[官宣](https://www.linuxfoundation.org/press/linux-foundation-launches-the-agent2agent-protocol-project-to-enable-secure-intelligent-communication-between-ai-agents)），截至 2026 年 4 月支持机构 **150+**，GitHub 22,000+ stars，AWS Bedrock AgentCore / Azure AI Foundry / Google Cloud 均原生集成（[LF 2026-04 press release](https://www.linuxfoundation.org/press/a2a-protocol-surpasses-150-organizations-lands-in-major-cloud-platforms-and-sees-enterprise-production-use-in-first-year)）。

### 2.1 三种传输绑定

| Binding | 用途 |
|---|---|
| **JSON-RPC 2.0 over HTTPS** | 默认 / 调试友好 |
| **gRPC over HTTP/2** | 高性能场景 |
| **HTTP+JSON/REST** | 网关/防火墙友好 |

流式更新统一用 **Server-Sent Events (SSE)**（gRPC 用原生流）。

### 2.2 核心对象模型

```
AgentCard               # 身份 + 能力 + 端点（签名可选）
 └─ AgentSkill[]
 └─ AgentCapabilities   # streaming / push / extended

Task                    # 服务端生成 ID, 有生命周期
 ├─ Message[]           # role: "user" | "agent"
 │   └─ Part[]          # text | file | data
 ├─ Artifact[]          # 输出物
 └─ TaskStatus          # WORKING/INPUT_REQUIRED/AUTH_REQUIRED/
                        #  COMPLETED/FAILED/CANCELED/REJECTED
```

### 2.3 八个核心方法

| 方法 | 说明 |
|---|---|
| `message/send` | 发起对话，返回 Task 或直接 Message |
| `message/stream` | 同上，但 SSE 流式 |
| `tasks/get` | 查询任务状态 / 历史 / 工件（轮询） |
| `tasks/list` | 分页+过滤 |
| `tasks/cancel` | 幂等取消 |
| `tasks/subscribe` | 订阅已存在任务的事件流 |
| `pushNotificationConfig/*` | webhook 配置 |
| `agentCard/extended` | 鉴权后的扩展 Agent Card |

### 2.4 一次最小 A2A 会话

```http
POST /a2a HTTP/1.1
Content-Type: application/json
A2A-Version: 1.0

{
  "jsonrpc": "2.0",
  "id": "1",
  "method": "message/stream",
  "params": {
    "message": {
      "role": "user",
      "parts": [{"type": "text", "text": "请预订下周一去上海的高铁，预算 800"}]
    }
  }
}
```

服务端 SSE 流式返回：

```
event: task
data: {"task": {"id": "tk_001", "status": {"state": "WORKING"}, ...}}

event: statusUpdate
data: {"taskId":"tk_001","state":"INPUT_REQUIRED",
       "message":{"role":"agent","parts":[{"type":"text","text":"找到 3 班，选哪班？"}]}}

… 客户端补一次 message/send →

event: statusUpdate
data: {"taskId":"tk_001","state":"COMPLETED",
       "artifacts":[{"name":"ticket","parts":[{"type":"data","data":{...}}]}]}
```

**关键点**：
- 长任务原生支持中途 `INPUT_REQUIRED`（呼应 9.3 的中断 UX）
- 跨框架协作 **不共享内部内存**，只共享 Task/Message/Artifact
- 一次会话可以挂载多个 Task，通过 `contextId` 串联

### 2.5 A2A 与 MCP 的分工

| 维度 | MCP | A2A |
|---|---|---|
| 主要场景 | Agent ↔ Tool / Data Source | Agent ↔ Agent |
| 信任模型 | 同一 trust domain | **跨组织 / 跨厂商** |
| 身份 | 通常 in-process / API Key | 签名 Agent Card / OAuth2 / mTLS |
| 任务模型 | 工具调用（同步为主） | 长任务、流式、中断、工件 |
| 隐喻 | "Agent 的 USB-C" | "Agent 的 HTTP" |

> **架构判断**：MCP 处理"我用什么工具"，A2A 处理"我向哪个 Agent 委托子任务"。它们互补，不冲突。

### 2.6 身份与授权

A2A v1.0 关键升级：
- **Signed Agent Card**：canonicalization + 签名，验证发布方
- **SecurityScheme**：APIKey / HTTPAuth / OAuth2 / OpenIdConnect / **mTLS**
- **In-Task Authorization**：任务执行中途升级权限（step-up auth）
- 未授权资源返回 **not-found** 而非 forbidden（防探测）

这条与 6.3 Agent 权限模型直接对接：A2A 上的每一次跨组织调用都是一次身份+授权 + Scope 的协商。

## §3 Agent 经济：从"Agent 能对话"到"Agent 能结算"

光会对话不够。真实经济交互需要"我授权你花这笔钱"、"你完成后我付你钱"。2025-2026 出现了四个互补协议，分别占据不同层。

### 3.1 四协议总览

| 协议 | 主导方 | 层 | 核心机制 | 结算 | 状态（2026-06） |
|---|---|---|---|---|---|
| **ACP** Agentic Commerce Protocol | OpenAI + Stripe | 商户结账 | 4 个 REST 接口 (Create/Update/Complete/Cancel Checkout) + SharedPaymentToken | 法币 (Stripe) | ChatGPT Instant Checkout 上线，US Etsy |
| **AP2** Agent Payments Protocol | Google + 60+ partners | 授权/信任 | **Mandates**（ECDSA + JSON-LD 签名）：Intent / Cart / Payment | 路由无关 | 规范发布，A2A x402 extension 生产可用 |
| **x402** | Coinbase / Cloudflare | HTTP 原生结算 | 复活 HTTP 402 状态码 + 钱包签名 + USDC | 稳定币（多链） | Stripe Base 集成（2026-02），Cloudflare 生产 |
| **MPP** Machine Payments Protocol | Stripe + Tempo | 会话级流支付 | Sessions：预授权 + 持续流式微支付，链下记账 | 稳定币（Tempo）+ 法币（SPT）+ 卡 | 主网 2026-03-18，100+ 服务接入 |

来源：[Crossmint 2026 综述](https://www.crossmint.com/learn/agentic-payments-protocols-compared)、[Forrester Stripe 2026](https://www.forrester.com/blogs/stripe-sessions-2026-stripe-is-rearchitecting-payments-for-an-agentic-ai-economy/)、[Mastercard Agent Pay 2026](https://www.mastercard.com/us/en/news-and-trends/press/2026/june/mastercard-launches-agent-pay-for-machines.html)。

### 3.2 AP2 Mandates：把"授权"做成可签可验的数据结构

AP2 的核心创新是 **Mandate**——加密签名的 JSON-LD 文档：

```json
// Intent Mandate（用户写的"授权书"）
{
  "@context": "https://google.com/ap2/v1",
  "type": "IntentMandate",
  "principal": "did:web:alice.example",
  "agent": "did:web:travel-agent.com",
  "constraints": {
    "category": "high-speed-rail",
    "max_amount_cny": 800,
    "valid_until": "2026-07-01T00:00:00Z",
    "destinations": ["上海"]
  },
  "signature": { "alg": "ES256", "value": "..." }
}

// Cart Mandate（Agent 执行前回到用户身边确认）
{
  "type": "CartMandate",
  "items": [...],
  "total": 768,
  "human_approval_required": true,
  "signature": { ... }
}

// Payment Mandate（告诉支付网络：这是 Agent 操作的）
{
  "type": "PaymentMandate",
  "agent_signal": true,
  "underlying_intent": "<intent-mandate-hash>",
  "signature": { ... }
}
```

**为什么这么设计**？三个目标：
1. **可审计**：任何一次扣款都可追溯到 Intent → Cart → Payment 三层签名
2. **可申诉**：Cart 超出 Intent 边界即视为越权
3. **网络可识别**：支付网络看到 `agent_signal=true` 可以做不同风控

### 3.3 x402：让 HTTP 自身会收钱

x402 让 server 在响应 `402 Payment Required` 时附带支付条款，客户端钱包签名后 retry，无需 API Key、无需账户、无需 OAuth：

```
GET /api/llm-completion HTTP/1.1

→ HTTP/1.1 402 Payment Required
  X-PAYMENT-REQUIRED: {
    "amount": "0.005",
    "currency": "USDC",
    "chain": "base",
    "recipient": "0x..."
  }

→ GET /api/llm-completion
  X-PAYMENT-SIGNED: <signed-payment>
← HTTP/1.1 200 OK
  ...
```

对架构师的意义：**Agent 调用 API 时的计费成为协议级原语**，不需要预先开账户，按调用付费，链上即结算。

### 3.4 MPP：流式扣费（agentic streaming bill）

MPP 用 "Session" 让 Agent 预授权一个上限，然后链下记账、持续扣费、批量上链：

- 适合：长跑 Agent，每秒 N 次调用
- 不适合：偶尔大额一次性支付
- 与 x402 的关系：x402 是一次性支付、MPP 是流式订阅，**互补不冲突**

### 3.5 实战组合：一个端到端例子

```
[User]
  ↓ Intent Mandate (AP2)
[Personal Agent A]
  ↓ A2A: message/stream
[Travel Agent B (在另一公司)]
  ↓ 调用 MCP 工具
[Booking API]
  ↓ 402 Payment Required (x402)
  ← USDC on Base
[Train Reservation System]
  → 返回 Artifact (Ticket)
[A 收到 Cart Mandate 需要人审 → 用户在 UI 中点击批准]
  ↓ Payment Mandate
[支付路由 → MPP session or x402]
```

> **判句**：未来 Agent 系统是 **A2A（沟通）+ MCP（工具）+ AP2（授权）+ x402/MPP（结算）** 的复合体。

## §4 Agent 身份：A2A 时代的新一等公民

A2A 让 Agent 跨组织对话，意味着**身份必须可验证、可吊销、可审计**。这与 [10.5 Agent 标准与治理演进](./10.5-Agent标准与治理.md) 中将讨论的 Agent Identity 议题深度耦合。

| 身份维度 | 含义 | 当前方案 |
|---|---|---|
| **Agent 自身** | 这个 Agent 是 X 公司开发的、版本号 v1.2 | Signed Agent Card / DID |
| **代理关系** | 这个 Agent 代表 Alice 行使权限 | OAuth2 token / Mandate |
| **任务上下文** | 当前任务的可信链路 | Task ID + signed audit log |
| **执行环境** | 这个 Agent 跑在哪个受控环境 | Attestation（TEE / TPM） |

**反模式提醒**：很多团队只做了第 1 项（自身身份），漏掉了 2-4。结果："我是合法 Agent" ≠ "我有权花这笔钱" ≠ "这次调用是合规的"。

## §5 反模式

| 反模式 | 后果 | 替代 |
|---|---|---|
| 跨厂商 Agent 直接共享 prompt/memory | 框架强耦合、信息泄露 | 走 A2A Task/Message/Artifact 中立模型 |
| 自定义 REST 假装是 A2A | 没有 Agent Card / 生命周期 / 流式 → 集成失败 | 接 A2A SDK（Python/JS/Java/Go/.NET） |
| 用 API Key 模型给 Agent 计费 | 一个 Agent 调度 1000 次/秒，账户/限流崩溃 | x402 / MPP 按调用付费 |
| 用静态 OAuth scope 给所有 Agent 操作授权 | 越权操作无法追溯 | AP2 Mandates 分 Intent/Cart/Payment 三层 |
| 在 A2A 端点上不实现 INPUT_REQUIRED | 长任务到一半就 fail | 实现完整 TaskStatus 状态机 |
| 把 MCP 当 A2A 用（或反之） | 跨组织信任、长任务、流式都缺 | MCP 处理工具，A2A 处理 Agent 协作 |
| Mandate 签名但不验证 chain-of-trust | 伪造 Intent → 套出 Cart | 每层 Mandate 验证上一层 hash |
| 用人类 KYC 模型处理 Agent | UX 灾难 / 不可执行 | 走 Agent-specific 身份层（Mastercard Agent Pay 模型） |

## §6 S0 → S5 落地路线图

### S0：现状
- 内部 Agent 通过 REST 调来调去，无统一规范
- 没有 Agent 间计费机制

### S1：内部 A2A 试点（1-2 个月）
- 选 2-3 个 Agent 改造为 A2A endpoint
- 用 JSON-RPC binding + SSE 流式
- Agent Card 静态发布在内部 registry

### S2：跨团队 A2A 标准化（3-6 个月）
- 全公司统一 Agent Card schema
- 引入 OAuth2 + Signed Agent Card
- 长任务、INPUT_REQUIRED、Artifact 都接通

### S3：MCP × A2A 双轨（6-9 个月）
- 工具调用走 MCP，Agent 间走 A2A
- 团队各自维护 Agent Card，公共 registry 提供发现

### S4：跨组织 + 计费（9-12 个月）
- 与外部供应商 / 合作伙伴的 Agent 互通
- 接入 x402（API 调用）或 MPP（流式服务）
- 涉及高金额操作引入 AP2 Mandates

### S5：参与生态治理（12+ 个月）
- 跟进 LF A2A WG、AP2 spec 修订
- 自研扩展提交回上游
- 与 [10.5](./10.5-Agent标准与治理.md) 中 EU AI Act、Agent Identity 标准衔接

## §7 一页总结

**判句**：MCP 是 USB-C，A2A 是 HTTP，AP2/x402/MPP 是 SWIFT/Visa。三层缺一不可。

**三件事必须记住**：
1. **A2A 与 MCP 互补**，前者跨组织、长任务、流式；后者同信任域、工具调用、同步为主。
2. **Agent 经济四协议各占一层**：ACP 商户结账 / AP2 授权 / x402 HTTP 原生结算 / MPP 会话流支付。
3. **Mandate** 是 AP2 的核心创新——把"授权"做成可签可验的数据结构，Intent→Cart→Payment 三层可追溯。

**做对的事**：
- 跨厂商 Agent 协作直接用 A2A SDK，不自己造 REST
- Agent Card 一定要签名
- 高金额/高风险操作用 AP2 Mandates
- API 计费考虑 x402，避免 API Key 模式

**警惕**：
- 把"自身身份"等同于"代理权限"
- 假装是 A2A 的私有 REST
- Mandate 不验链路就直接信任
- 用人类 KYC 模型给 Agent 做合规
