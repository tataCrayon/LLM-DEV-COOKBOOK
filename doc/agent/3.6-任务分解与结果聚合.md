# 3.6 任务分解与结果聚合：Decomposition / Map-Reduce / Debate / Consensus / Voting / Synthesis

> **写在前面**：本文讨论 Multi-Agent 系统里**信息怎么进、怎么分、怎么并、怎么合**——从任务分解（Decomposition）开始，到结果聚合（Reduce / Vote / Consensus / Debate / Synthesize）结束。承接 3.5（角色与边界）、3.1（编排模式），与 3.7（可靠性）共同构成 Multi-Agent 工程三联画。

## §0 真问题：5 个 Researcher 跑完，最后一步把答案搞错了

把 Multi-Agent 工程化的人，撞到的真问题往往不在"怎么并行"，而在"怎么合"：

第一，**并行 worker 返回了 5 份相互矛盾的结果**——Researcher A 说 X 公司 CFO 是 Alice，Researcher B 说是 Bob，Researcher C 完全没找到。最后写报告的 Synthesizer 选了第一个看到的，或求平均把 Alice 0.6 Bob 0.4 输出。结论错。

第二，**Map 阶段 "task overlap"**——5 个 Researcher 都在调研同一段时期同一公司，时间和 token 全浪费在重复工作。Anthropic 在 multi-agent research blog 里直接讲过这个坑："one investigated 2021 chip crisis while two redundantly explored 2025 supply chains"。

第三，**Reduce 阶段 context 爆**——10 个 worker 各返回 50K tokens，Synthesizer 单 context 装不下。简单 concat 不可行，必须分层 reduce 或外置 memory。

第四，**Voting 在低 N 上是噪声放大器**——3 个 agent 投票，2:1 的结果只是噪声偏移，不构成"集成增强"。自一致性（self-consistency）要 N≥10 才稳定，但成本 10×。

第五，**Debate 增加成本但不一定提升精度**——给两个 Agent 对辩，3 轮过去消耗 4× 计算量，结果有时反而被强烈表达者带偏（多个 2025 研究复盘）。

第六，**Synthesizer 偏向"中庸答案"**——LLM 作为 Synthesizer 倾向于选既不偏激也不冷门的答案，导致正确但反共识的结论被 reduce 掉。Wisdom of crowds 反过来变成 herd toward mediocrity。

第七，**任务分解时没有显式 schema，worker 不知道返回什么**——Planner 写"go research the supply chain"，但没说"返回 5 个供应商名字 + 每个的市场份额数字"。Worker 各自发挥，Reduce 时字段对不上。

本文系统化讲这些。

## §1 第一性原理：分解-聚合的四类模式

**模式 1：Linear Chain**（线性分解）
```
Task → Step1 → Step2 → Step3 → Result
```
最简单。一个 Agent 接一个 Agent 顺序传递。失败模式：context 累积、误差累积、不能并行。适合**步骤明确且 strict 依赖**的工作流（如 ETL pipeline）。

**模式 2：Map-Reduce**（横向分解 + 纵向聚合）
```
Task ───┬─ Worker1 ┐
        ├─ Worker2 ├→ Reducer → Result
        ├─ Worker3 ┘
        └─ Worker4
```
并行高，吞吐高。失败模式：worker 任务重叠、reduce 时 context 爆、worker schema 不一致。适合**广度搜索/批处理/可并行检索**——A-MapReduce 范式。

**模式 3：Tree Search / Hierarchical**（树形分解）
```
Root Task → Subtask1 → SubSubtask1.1
                    → SubSubtask1.2
          → Subtask2 → SubSubtask2.1
```
深度可控。失败模式：tree 深度不收敛、回溯成本高。适合**真正递归的问题**（数学证明、形式化分解）。

**模式 4：Debate / Voting / Consensus**（多观点 → 一答案）
```
Task ───┬─ Agent1 ┐
        ├─ Agent2 ├→ Vote / Debate / Synthesize → Result
        ├─ Agent3 ┘
```
不分解任务，分解视角。失败模式：N 太小则噪声、N 太大则成本爆、易被多数派吸引。适合**判断/选择/没有 ground truth 的开放问题**。

**核心区别**：Map-Reduce 假设 worker **任务不同**（分工），Debate/Voting 假设 worker **任务相同**（多视角）。两者的聚合算法完全不同。

## §2 分解（Decomposition）的工程做法

### 2.1 显式 Schema 驱动的任务分解

错误示范：
```python
plan = planner.complete(f"Decompose: {task}")  # 返回散文
worker_tasks = parse_loosely(plan)
```

推荐做法：
```python
class SubTask(BaseModel):
    worker_role: Literal["researcher", "analyst", "writer"]
    objective: str                # 1-2 句明确任务
    input_data: dict
    expected_output_schema: dict  # JSON schema
    success_criterion: str        # 可执行 assert
    max_budget_tokens: int
    depends_on: list[int] = []    # 显式依赖

class DecompositionPlan(BaseModel):
    subtasks: list[SubTask]
    aggregation_strategy: Literal["concat", "reduce", "vote", "debate", "synthesize"]

plan = planner.complete(
    messages=[...],
    response_format=DecompositionPlan,
    reasoning_effort="medium"
)
```

要点：
- **每个 subtask 必须带 expected_output_schema**——这是 Reduce 阶段不出错的前提。
- **success_criterion 是可执行 assert**——worker 完成时立刻验证。
- **aggregation_strategy 提前声明**——Planner 决定怎么聚合，而非临场判断。
- **depends_on 显式 DAG**——可识别可并行 vs 必须串行。

### 2.2 防 Overlap：Task Matrix

A-MapReduce 的核心技巧：把任务分解成**矩阵 M_q + 模板 P_q**：

```
M_q = [
  {company: "Apple",  year: 2025, attribute: "CFO"},
  {company: "Apple",  year: 2025, attribute: "Revenue"},
  {company: "Google", year: 2025, attribute: "CFO"},
  ...
]
P_q = "Find {attribute} of {company} in {year}"
```

每行 = 一个原子任务。**结构化的分解避免了散文 plan 的 overlap 与 omission**。A-MapReduce 在 WideSearch benchmark 上 Item F1 67.81% 显著超过非结构化 multi-agent baseline 51-55%（[A-MapReduce, arXiv:2602.01331](https://arxiv.org/html/2602.01331v1)）。

### 2.3 Reasoning Model 做 Planner

Planner 用 reasoning model（o3 / GPT-5 / Claude Opus thinking / R1）效果显著好于普通模型。Anthropic Research System 用 Opus 4 做 LeadResearcher、Sonnet 4 做 worker——正是这个分工。原因：**分解任务 = 多约束推理**，正是 reasoning model 强项。

### 2.4 Decomposition 的失败信号

监控以下指标：
- **task overlap rate**：worker 任务的语义相似度（embedding cosine > 0.8 即可疑）。
- **subtask completion rate**：success_criterion 通过率。低于 80% 时 plan 出了问题。
- **plan depth**：超过 4 层时大概率是 reasoning model 在过度分解（YAGNI）。
- **worker count vs task complexity**：Anthropic 经验法则——简单 1 个 worker，中等 2-4 个、10-15 calls 每个；复杂 10+ worker。

## §3 聚合（Aggregation）的五种模式与实测对比

### 3.1 Concat（拼接）

最朴素：把 worker 输出连起来。
- **优点**：零信息损失。
- **缺点**：context 爆、可读性差。
- **适用**：worker 数 ≤ 3 且每个输出 ≤ 2K tokens。

### 3.2 Hierarchical Reduce（分层规约）

```python
def hierarchical_reduce(results: list[str], batch=5):
    while len(results) > 1:
        batches = [results[i:i+batch] for i in range(0, len(results), batch)]
        results = [reducer.summarize(b) for b in batches]
    return results[0]
```

- **优点**：可扩展到任意 N。
- **缺点**：信息逐层流失；早期 batch 的 noisy 信号难恢复。
- **适用**：worker 数 > 5 的 Map-Reduce 场景。

### 3.3 Voting（投票）

```python
from collections import Counter

def majority_vote(answers: list[str]) -> tuple[str, float]:
    counts = Counter(answers)
    winner, n = counts.most_common(1)[0]
    return winner, n / len(answers)
```

- **优点**：噪声鲁棒（在 N ≥ 5 时）。
- **缺点**：要求答案**可枚举且离散**；N 小时是噪声放大器。
- **适用**：分类任务、选择题、布尔判断。
- **关键**：N 必须 ≥ 5，最好 7-10。Self-consistency（[Wang 2023](https://arxiv.org/abs/2203.11171)）要求 N=40。

### 3.4 Debate（辩论）

```
Agent A gives initial answer.
Agent B critiques A's answer.
Agent A responds to critique.
... (N rounds)
Judge selects winner or synthesizes.
```

- **优点**：在数学推理、批判性思考上能提升 5-15% 精度（早期论文）。
- **缺点**：成本 4-10×；2025 后续研究显示**强表达者偏置严重**——更自信的 Agent 在 Judge 评分时占优，与正确性无关；某些任务上反而比 single agent 差。
- **适用**：长论证任务、伦理判断、需要 stress-test 答案的场景。
- **关键发现**：[Voting or Consensus, ACL Findings 2025](https://aclanthology.org/2025.findings-acl.606.pdf) 数据显示——**投票（vote）在可枚举答案任务上稳赢；consensus 在开放性任务上更好；debate 高方差**。

### 3.5 Consensus（共识）

```
All agents propose.
All agents iteratively refine until convergence (or k rounds).
```

- **优点**：开放性任务上质量稳定；共识过程本身具有可解释价值。
- **缺点**：收敛慢；可能困于 group thinking（所有 agent 同模型时尤其严重）。
- **适用**：写作、设计、决策类任务。
- **关键**：consensus 必须配模型多样性——同模型 consensus 就是 echo chamber。

### 3.6 Synthesize（综合）

LLM 作为 Synthesizer 接所有 worker 输出，产出新答案。
- **优点**：可处理结构差异大的输入；可加入 metadata（confidence、source）。
- **缺点**：Synthesizer 是单点（出错就全错）；倾向于中庸答案。
- **适用**：Open-domain QA、Research Report、Deep Search。
- **关键**：Synthesizer 必须用 reasoning model + 显式让它**保留 worker 间分歧而非求平均**。

### 3.7 决策矩阵

| 任务类型 | 推荐聚合 | 备注 |
|---------|---------|------|
| 数学 / 单选题 / 分类 | **Voting (N≥7)** | self-consistency 标配 |
| 代码生成 | Voting on test pass / pick first that passes | majority on text 没意义 |
| 事实检索 / Wide Search | **Hierarchical Reduce** | 配 source-aware merge |
| 写作 / 创意 | **Synthesize**（同模型）或 **Consensus**（多模型） | 避免 vote |
| 复杂分析 / Deep Research | **Synthesize by reasoning model** | Anthropic 范式 |
| 红队 / 对抗博弈 | **Debate**（必要） | 投票/共识会失去对抗价值 |
| 工程决策 | **Debate + Human Judge** | 不要让 LLM 做 Judge |

## §4 工程实现

### 4.1 完整 Map-Reduce 骨架

```python
class MapReduceOrchestrator:
    def __init__(self, planner, workers, reducer):
        self.planner = planner    # reasoning model
        self.workers = workers     # dict[role -> Worker]
        self.reducer = reducer     # reasoning model
    
    def run(self, task: str) -> Result:
        # 1. Plan
        plan: DecompositionPlan = self.planner.decompose(task)
        validate_plan(plan)         # 检查 overlap、schema 完整性
        
        # 2. Map (parallel)
        results: list[WorkerResult] = []
        with ThreadPoolExecutor(max_workers=10) as ex:
            futures = [
                ex.submit(self._run_subtask, st) 
                for st in plan.subtasks if not st.depends_on
            ]
            for f in as_completed(futures):
                r = f.result()
                if r.success:
                    results.append(r)
                else:
                    # Worker 失败重试或降级
                    results.append(self._fallback(r))
        
        # 3. Reduce
        return self._aggregate(results, plan.aggregation_strategy)
    
    def _run_subtask(self, st: SubTask) -> WorkerResult:
        worker = self.workers[st.worker_role]
        with Sandbox(tools=worker.spec.allowed_tools, budget=st.max_budget_tokens):
            output = worker.execute(st.input_data, schema=st.expected_output_schema)
        if not check_criterion(output, st.success_criterion):
            return WorkerResult(success=False, error="criterion failed")
        return WorkerResult(success=True, data=output)
    
    def _aggregate(self, results, strategy):
        if strategy == "vote":
            return majority_vote([r.data for r in results])
        elif strategy == "reduce":
            return hierarchical_reduce([r.data for r in results])
        elif strategy == "synthesize":
            return self.reducer.synthesize(results, preserve_disagreement=True)
        elif strategy == "debate":
            return debate_round(results, judge=self.reducer)
```

要点：
- **Plan validation**——overlap、schema、budget 上限。
- **并行+sandbox**——每个 worker 独立 capability。
- **失败回退**——单 worker 失败不击穿整体（见 3.7 可靠性）。
- **聚合策略由 Planner 声明**，runtime 仅执行。

### 4.2 Synthesizer 的 "保留分歧" prompt 模板

```
你是 Research Synthesizer。下面是 N 个 Researcher 对同一问题的独立调研结果。

要求：
1. 总结所有 Researcher 都同意的事实，标注 "consensus: N/N agree"
2. **保留并明确标注分歧**，例如 "Researcher 1, 3 认为 X；Researcher 2 认为 Y"，不要求平均
3. 标注每个事实的来源 Researcher ID 与 confidence
4. 对存在分歧但**重要**的事实，建议进一步调研而非武断结论
5. 不要为了答案完整而虚构未涉及的信息
```

这个 prompt 显著降低 Synthesizer 的"中庸偏置"——多个 2025 实验表明，明确告知"保留分歧"能让 Synthesizer 的 factual accuracy 提升 8-15%（基于 Anthropic Research Eval 内部数据，公开 blog 隐含）。

### 4.3 Debate 实施细节

```python
def debate_round(agents: list[Agent], task: str, rounds=3, judge=None):
    transcripts = {a.id: [] for a in agents}
    
    # Round 1: independent answers
    for a in agents:
        a_resp = a.answer(task)
        transcripts[a.id].append(a_resp)
    
    # Rounds 2..n: see others' answers, refine
    for r in range(1, rounds):
        for a in agents:
            others = {oid: t[-1] for oid, t in transcripts.items() if oid != a.id}
            a_resp = a.refine(task, my_history=transcripts[a.id], others=others)
            transcripts[a.id].append(a_resp)
    
    # Judge picks winner（用 reasoning model）
    return judge.evaluate(task, transcripts)
```

关键：
- **Judge 不能是参与者之一**——否则自利偏置。
- **Judge 用 reasoning model + structured criterion**。
- **rounds 通常 3-4 即收敛**——更多轮成本剧增但收益递减。
- 关注 **disagreement persistence**：到第 N 轮还不收敛 → 这个题目本身可能没有正确答案 → 提交人类。

### 4.4 Self-Consistency 工程实现

```python
def self_consistency(model, prompt, n=10, temperature=0.7):
    """适用于答案可枚举的任务（数学题数字、分类标签等）"""
    answers = []
    for _ in range(n):
        resp = model.complete(prompt, temperature=temperature)
        ans = extract_final_answer(resp)
        if ans is not None:
            answers.append(ans)
    return majority_vote(answers)
```

要点：
- 温度必须 > 0（否则 N 个完全一样）。
- N=10-40 取决于成本预算。
- 仅对**有标准答案**的任务有效——开放性写作上 self-consistency 无意义。
- 推理模型上 self-consistency 性价比下降——reasoning 已经内化了多路径采样。

## §5 反模式

**反模式 1：plan 是散文，worker schema 不一致**。Reduce 时字段对不齐，硬合并必错。Schema-driven decomposition 是底线。

**反模式 2：Planner 用普通模型**。Planner 是 reasoning-intensive task；用 GPT-4o-mini 做 Planner 配 Claude Sonnet 做 worker，是 cost backward 配置。

**反模式 3：所有 worker 用同模型同 prompt 然后 vote**。这是 echo chamber——投票相当于一个 agent 多采样，没有真正多样性。要么换模型，要么换 retrieval source。

**反模式 4：N=3 投票**。2:1 在分类任务的统计意义极弱。N≥5 起步。

**反模式 5：Debate 让 LLM 做 Judge**。LLM Judge 易被"自信表达者"偏置；要么用 reasoning model 做 Judge + 显式 criterion，要么人类 Judge。

**反模式 6：Synthesizer 直接 concat 所有 worker 输出**。Reasoning model 对 50K+ context 注意力下降；必须先 hierarchical reduce。

**反模式 7：Reduce 不保留 source provenance**。最终报告里"Apple CFO is Bob"——但不知道这个事实来自哪个 worker、哪个引用、哪个时间。出错时无法 trace。每个 fact 必须带 citation。

**反模式 8：忽视 worker overlap**。10 个 Researcher 调研同一主题——成本 10×、信息没增加。Decomposition 阶段必须显式不相交。

**反模式 9：Debate 走 5+ 轮**。研究表明 3-4 轮后收益递减且开始 deteriorate（agent 互相 echo）。3 轮收尾。

**反模式 10：用 vote 解决创意/写作任务**。"3 个 writer 投票选最好的标题"——vote 没有信号，因为每个 writer 投自己。改用 Synthesize 或人类选。

**反模式 11：Map-Reduce 当万能解**。Cognition / Anthropic 都强调：大部分代码任务用单 agent 长跑，不需要 map-reduce。识别真正的"宽搜索/广收集"才用。

## §6 S0→S5 演进路线

**S0 - 单 Agent 长跑**：一个 Agent 顺序做完。简单任务的默认。

**S1 - Sequential Chain**：Plan → Research → Write → Review 串联。Linear chain。引入显式 schema。

**S2 - Map-Reduce 并行（schema-driven）**：Plan 输出 task matrix，worker 并行，reducer 聚合。引入 success criterion + budget + sandbox。

**S3 - Hierarchical Aggregation**：Reduce 分层；Synthesizer 保留分歧；citation 全程透明。

**S4 - 多策略混合**：根据 task type 自动选 vote / debate / synthesize。Planner 输出 aggregation_strategy。

**S5 - 经验型分解（A-MapReduce）**：保留过往执行 trajectory，下次相似 query 用历史 hint 加速 plan。把 Multi-Agent 系统升级到 self-improving。

## §7 一页总结

第一性原理：Multi-Agent 系统的核心问题是**信息流**——任务怎么分（Decomposition）、worker 怎么并、结果怎么合（Aggregation）。Decomposition 决定 worker 是否做重复工作；Aggregation 决定结果质量上限。

工程不变量：**Schema-driven decomposition**（task matrix + expected output schema）、**Planner 用 reasoning model**、**Worker 用 cheap model**、**Synthesizer 显式保留分歧**、**Citation 全程透明**。

聚合策略选择：可枚举/分类用 **Voting (N≥7)**；事实检索用 **Hierarchical Reduce**；写作/创意用 **Synthesize**；开放问题用 **Consensus**（多模型）；对抗/红队用 **Debate**。

最危险的反模式：plan 散文；N=3 vote；LLM Judge；Synthesizer 直接 concat；同模型 echo chamber；Map-Reduce 当万能解。

最强信号：A-MapReduce 在 WideSearch 上 67.81% 显著超过非结构化 baseline 51-55%，**结构化分解+经验复用是 2026 多 agent 系统的胜出点**。

向后衔接：3.5（角色边界）解决"谁来做"；本文（3.6）解决"怎么分怎么合"；3.7（可靠性）解决"出错怎么办"。三篇互锁。下游与 5.2 RAG（retrieval 子模块）、6.2 Guardrails（worker 输出护栏）、8.3 成本工程（多 agent 成本预算）联动。
