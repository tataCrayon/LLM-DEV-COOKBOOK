# 7.5 领域专项 Benchmark 深度解读：SWE-bench / τ-bench / GAIA / WebArena / OSWorld / MLE-Bench / HLE

> **学习目标**：把 2026 年 7 个主流 Agent benchmark 解剖到"任务/数据集/评分/SOTA/局限"五位，建立"按业务域选 benchmark"的对照表，识别公开榜的可游戏化漏洞，明确**哪些 benchmark 可用作选型基线、哪些只能当学术参考**。
>
> **前置阅读**：[7.1 评估框架](./7.1-Agent评估框架.md) §7（公开榜 vs 自建）

---

## 0. 真问题：为什么"benchmark 70%"几乎等于没说

PM 在产品需求里写："这个 Agent 要达到行业领先水平——SWE-bench 70%+"。听起来很专业。但 **SWE-bench 哪个变体？Verified 还是 Full？哪个 harness？是 Bash-Only 还是 Claude Code 完整 scaffolding？500 题里只覆盖 12 个 Python repo，跟你的 Java/Go/Rust 业务关系多大？**

2026 年最容易踩的 3 个坑：

| 坑 | 例子 |
|---|---|
| **变体混淆** | SWE-bench (2294) vs SWE-bench Verified (500) vs SWE-bench Lite (300) vs Live；分数差 10-20pp |
| **Harness 不可比** | Lab 自报分用了私有 scaffolding；统一 Bash-only harness 下分数低 5-15pp |
| **领域错位** | 客服业务跟 SWE-bench 没关系；HLE 是学术考试不是 Agent benchmark |

**第一性原理**：公开 benchmark 是**模型/方法能力的相对比较器**——它**永远不该被当成你业务质量的预测器**。把这件事记牢，本章其他内容才能正确使用。

## 2. SWE-bench 家族：代码修复的事实标准

### 2.1 三个变体

| 变体 | 题量 | 谁出资 | 关键特点 |
|---|---|---|---|
| **SWE-bench (Full)** | 2,294 | Princeton/USC/CMU | 12 个流行 Python repo 的 GH issue + 对应 PR |
| **SWE-bench Verified** | **500** | **OpenAI 出资人工筛** | 移除歧义/不可解题，**事实上的标准榜** |
| **SWE-bench Lite** | 300 | 社区子集 | 单文件修复，便宜跑通快测试 |
| **SWE-bench Live** | 持续 | [live-swe-agent.github.io](https://live-swe-agent.github.io/) | 新 issue 持续注入，防训练集泄露 |
| **Multimodal / Multilingual** | - | 扩展 | UI bug、Java/Go/Rust |

> **业界数据**（[Simon Willison, 2026.02](https://simonwillison.net/2026/Feb/19/swe-bench/)）：Verified 500 集的 repo 分布——django 231、sympy 75、sphinx 44、matplotlib 34、sklearn 32、astropy/xarray 各 22。**单 django 占 46%**——你的业务如果不是 Django，参考价值打折。

### 2.2 评分

- **% Resolved**：模型生成 patch → 应用到 repo → 跑该 issue 对应的测试套件全过即"resolved"。
- **完全执行级**，不看文本相似度——τ-bench 与之同流。
- 官方 leaderboard 区分**自报**（lab 用任意 harness）和 **Bash-Only**（统一 mini-swe-agent ~9K 行 Python 跑所有模型，公平横评）。

### 2.3 SOTA 与陷阱

| 模型 | Bash-Only Verified % Resolved |
|---|---|
| Claude 4.5 Opus (high reasoning) | **76.8%** |
| Gemini 3 Flash | 75.8% |
| MiniMax M2.5 | 75.8% |
| Claude Opus 4.6 | 75.6% |
| GPT-5.2 | 72.8% |

**陷阱清单**：
1. **变体混淆**：Lab 公告写 "SWE-bench 65%" 但其实是 Verified；Full 上的同模型可能只有 30-40%。
2. **Harness gap**：自报数 ≠ Bash-only 数；Anthropic Claude Code 自家 harness 可能多 5-10pp。
3. **数据泄露**：base model 训练集很可能见过这些 PR——SWE-bench Live 才是真"零泄露"。
4. **Python only**：12 repo Python 项目，泛化到 Java/Go/Rust 业务**不可外推**。
5. **测试覆盖**：通过测试 ≠ 真正修复；存在"通过原测但破坏新场景"的情况（[arXiv:2506.17208](https://arxiv.org/html/2506.17208v1)）。

**业务使用建议**：
- 选 coding Agent 时**优先看 Bash-Only Verified + Live**。
- 内部务必自建一个对你业务语言/repo 的小型 benchmark（30-100 题）作为校准。

## 3. τ-bench / τ²-bench：客服 Agent 唯一靠谱的公开基线

### 3.1 任务设计

- **域**：retail（电商）、airline（机票）、telecom（电信，τ²-bench 新增）。
- **可执行环境**：真实 DB schema + 一组工具（search_order / cancel / refund / book / ...）。
- **用户模拟器**：LLM 扮演用户，按 persona 与意图与 agent 对话。
- **核心创新**（vs 旧 benchmark）：**检查 DB 终态，不只检查文本**。Agent 说"已取消订单"但 DB 里订单还活着 → 判失败。

### 3.2 τ²-bench 的两大升级（[arXiv:2506.07982](https://arxiv.org/abs/2506.07982)）

1. **Dual-control（Dec-POMDP）**：用户也能调用工具操作环境（如自己改密码、自己查询），不再是"被动回答 agent 提问的 NPC"。这更贴近真实客服——客户不是只会回答问题。
2. **pass^k 指标**：k 次重复尝试都成功的概率。单次 success rate 高但 pass^5 低 = 不稳定，业务上无法部署。

**关键发现**："agents show significant performance drops when shifting from no-user to dual-control, highlighting the challenges of guiding users."——单方控制下表现好不代表能处理真客户。

### 3.3 SOTA 与启示

- retail / airline 单控：~60-70%（top 模型）。
- telecom 双控：~45-55%（明显更难）。
- pass^5 比 pass^1 通常掉 15-30pp——**单次准确率不等于工程可用**。

**业务使用建议**：
- 客服 Agent 必看 τ-bench；尤其多轮、需 DB 写入的场景。
- 自建 benchmark 时**复用 τ-bench 的方法论**：定义 schema、工具、user simulator、检查终态。
- pass^k 是 Agent 必须报告的指标，不是 nice-to-have。

## 4. GAIA：通用 assistant 的"高考"

### 4.1 设计

- **450 题**（公开 val 165，私榜 285）。
- 三级难度：L1（"好 LLM 应能搞定"）、L2、L3（"能力跃迁分界"）。
- 任务覆盖：推理、多模态、web 浏览、tool use、长文档处理。
- **答案唯一**（exact match 可评）——这是 GAIA 的核心便利。

### 4.2 SOTA（[HAL Princeton 榜](https://hal.cs.princeton.edu/gaia)）

| 系统 | 总 Acc | L1 | L2 | L3 | 成本 |
|---|---|---|---|---|---|
| HAL Generalist + Claude Sonnet 4.5 | **74.55%** | 82.07% | 72.68% | 65.39% | $178/run |
| HAL Generalist + Claude Sonnet 4.5 High | 70.91% | - | - | - | $179 |
| HAL + Claude Opus 4.1 High | 68.48% | - | - | - | $562 |

**关键观察**：
- 同一 base 模型 + 不同 reasoning 设置 + 不同 harness → 5-10pp 差异。
- **L3 仍只 65%**——"通用 assistant"在专家级任务上还差得远。
- 成本 $178/run（165 题）≈ **$1/题**。"GAIA 跑分"本身就贵。

### 4.3 业务使用建议

- 看 GAIA 选**通用 assistant 类**（如 Manus、Devin、企业内"啥都问"助手）的基线模型。
- **不适合**做客服/代码/特定 vertical 的判断。
- 评估**端到端成本**：GAIA 暴露了"答对一题花多少钱"——对成本敏感的产品必须看。

## 5. WebArena / Visual WebArena / OSWorld：操作类 Agent

### 5.1 WebArena

- **812 个 web 任务**，4 类自托管站点（shop / gitlab / reddit-like / map）。
- 评分：URL/state match + 程序化 check。
- 优势：**完全自托管**，可重现性强；不会因为线上站点改版让结果失效。
- SOTA：~40-50%（top 系统）。
- **Visual WebArena** 扩展加入图像识别（截图理解）。

### 5.2 OSWorld

- **369 个真实桌面任务**（Ubuntu / Windows / macOS）。
- 任务示例：在 Calc 里改公式、在 Chrome 改设置、用 Terminal 跑命令、跨 app 完成工作流。
- 评分：终态文件 / 进程 / 注册表 match。
- SOTA：~30-40%（明显更难，桌面交互维度高）。

### 5.3 业务使用建议

- **GUI Agent / RPA 类产品**：WebArena 是基线选型；OSWorld 是上限测试。
- **注意 reproducibility**：OSWorld 跨 OS 配置易出环境差异，要锁版本。
- 自建 benchmark 时学其"**终态 match**"思路——不要让模型"自由发挥"输出文字，必须验证现实世界变化。

## 6. MLE-Bench：ML Engineer Agent 的"奖牌赛"

- [OpenAI MLE-Bench](https://openai.com/index/mle-bench/)，2024 末发布。
- **75 个 Kaggle 比赛**离线复刻。
- 评分：用历史排行榜分布判断 Agent 提交的分数能拿铜/银/金 → **medal rate** 是主指标。
- SOTA：medal rate ~30%（GPT/Claude top 模型 + 多 agent scaffolding）。

**业务使用建议**：
- 评估"AI 工程师"产品（如 Sakana ShinkaAgent / OpenHands ML）的核心基线。
- 不适合非 ML 工程业务。
- **重要警示**：medal rate 与"模型本身能力"耦合于具体竞赛分布——75 个项目难代表整个 ML 工程。

## 7. HLE：测的不是 Agent，是基座模型推理上限

- **2,500 多模态题**，100+ 学科，500+ 机构 1000+ 专家出题（[agi.safe.ai](https://agi.safe.ai/)）。
- **目的**：MMLU 已经被刷到 90%+，需要新的"前沿能力天花板"；HLE 是回应。
- 评分：accuracy + **calibration error**（模型给答案 + 0-100% 置信度，penalize 过度自信）。
- SOTA（2026 H1）：

| 模型 | HLE Acc | Calibration Err |
|---|---|---|
| Gemini 3 Pro | **38.3%** | 57.2% |
| GPT-5 | 25.3% | 50.0% |
| Grok 4 | 24.5% | 56.4% |
| Claude 4.5 Sonnet | 13.7% | 65.0% |

**关键定位**：
- HLE **不是 Agent benchmark**——没有工具、没有多步、没有副作用。
- 它衡量"基座模型的纯推理 / 知识广度上限"。
- Agent 工程师看 HLE 的唯一价值：**预判 vendor 模型升级的 ceiling**——如果你的 Agent 主要瓶颈是基座推理能力，HLE 涨分预示未来 Agent 能力提升。

## 8. 选型决策矩阵

按业务域 → 主选基线 → 二级参考：

```
代码修复 / 软件工程 Agent
    主选: SWE-bench Verified (Bash-Only) + SWE-bench Live
    二级: Multilingual SWE-bench (Java/Go/Rust 业务)
    自建: 你业务 repo 的 30-100 题闭环

客服 / 多轮工具 Agent
    主选: τ²-bench (dual-control + pass^k)
    二级: τ-bench retail/airline
    自建: 复用方法论；用真实 DB schema + user simulator

通用 assistant / 跨域助手
    主选: GAIA (按 L1/L2/L3 分层报告 + 端到端成本)
    二级: HLE (基座模型推理 ceiling)
    自建: 真实流量分桶

浏览器 / GUI Agent
    主选: WebArena / Visual WebArena
    二级: OSWorld (桌面)
    自建: 你目标网站的 50-100 任务 + 终态 check

ML / Data Science Agent
    主选: MLE-Bench
    二级: HLE 的科学子集
    自建: 你领域常见 Kaggle/内部数据集

科研 / 知识密集 Agent
    主选: HLE
    二级: GAIA L3
    自建: 学科专家出题
```

## 9. 公开榜的 5 种"作弊"与防御

1. **数据泄露**：base 模型预训练时见过测试集 → 选 **Live / Hold-out** 变体（SWE-bench Live、HLE 私榜）。
2. **Harness 套娃**：自报分用昂贵 scaffolding；选 **统一 harness（Bash-Only / HAL Generalist）**。
3. **过拟合 prompt 工程**：榜上专门优化的 prompt，泛化弱；自建 hidden 子集做校准。
4. **Cherry-pick 报告**：只报 best of N，不报 pass^k / median；要求 **pass^5 + std**。
5. **指标狭义化**：只报 accuracy，不报成本/延迟；要求 **accuracy/$ Pareto 前沿**。

> "Lab 自报 vs 第三方复现差距 5-15pp 是常态"——[Simon Willison, 2026](https://simonwillison.net/2026/Feb/19/swe-bench/)。

## 10. 自建 Benchmark 的最佳实践（汇总）

把本系列 [7.1](./7.1-Agent评估框架.md) §7 + [7.2](./7.2-Agent测试工程.md) §4 + 本章 §8 浓缩为一张清单：

```
✅ 真实流量采样（≥ 70%），分桶：happy path / 长尾 / 失败原因
✅ 执行级评分：DB / 文件 / API 终态，不只看文本
✅ 多合法路径：不强制工具调用顺序
✅ 难度分层：L1 / L2 / L3 分别报分
✅ Pass^k：至少 k=3，最好 k=5
✅ Cost & Latency 强制报：accuracy/$ Pareto
✅ Hold-out / Live 子集防泄露
✅ Per-fail-mode 分布：知道改进空间
✅ 月度扩充：失败 case 自动入库；红队产物（[6.4](./6.4-RedTeaming合规审计与人机信任边界.md)）自动入库
```

## 11. 反模式速查

| 反模式 | 后果 | 替代 |
|---|---|---|
| 用 HLE 评 Agent | 测错维度 | 看业务域选 benchmark |
| 只看自报分 | 5-15pp 虚高 | Bash-Only / HAL Generalist 统一 harness |
| 把 benchmark 分数当业务质量 | 上线打脸 | benchmark 选模型，自建 benchmark 决定上线 |
| pass^1 当唯一指标 | 部署不稳定 | pass^5 + std |
| accuracy 不算 cost | 烧钱也不知道 | accuracy/$ Pareto |
| 不分层报分 | L3 灾难被 L1 平均掩盖 | per-level 报告 |
| 公开榜不防泄露 | 真实能力被高估 | Live / 私榜 / hold-out |
| benchmark 一年不更 | drift 后失效 | 月度扩充失败 case + 红队产物 |

## 12. 一页总结

> **Public benchmark 的最大价值是"横向比较"，最大风险是"被当成业务质量预测器"**。

- **代码 Agent** 看 SWE-bench Verified (Bash-Only) + Live；记 Lab 自报与第三方差 5-15pp。
- **客服 Agent** 看 τ²-bench dual-control + pass^k；不要看单次 success。
- **通用 assistant** 看 GAIA 分层 + 端到端 cost。
- **GUI Agent** 看 WebArena (web) / OSWorld (desktop)。
- **ML Agent** 看 MLE-Bench medal rate。
- **基座推理 ceiling** 看 HLE（**不是 Agent benchmark**）。
- **5 种作弊** + 5 种防御要逐条 check。
- 上线决策**永远以自建 benchmark 为准**，公开榜只是选型与对外 PR。
