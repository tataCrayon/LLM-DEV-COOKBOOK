# 9.5 异步与后台 Agent UX：Devin/Cursor Background 范式 / 通知中断 / 多任务面板 / 进度可视化

> "后台 Agent 的 UX 悖论：你看不见它在工作——但你必须相信它在正确地工作。"

## §0 真问题：Agent 在后台跑，用户在焦虑

四个场景。

**场景 A**：开发者给 Devin 分配了 3 个 bug fix，然后去开会。40 分钟后回来：3 个 PR 已经提好了。但中间发生了什么？有没有跑偏？有没有引入新 bug？**用户不在场时的信任怎么建立？**

**场景 B**：用户同时给后台 Agent 分配了 5 个任务。其中第 3 个卡住了（需要人工输入），但前端只显示"进行中"——用户不知道该介入，其他 4 个也被阻塞。**多任务管理没有有效优先级**。

**场景 C**：后台 Agent 完成了任务，在凌晨 3 点发了通知。用户早上醒来看到 7 条 Agent 通知混在几十条其他通知里——**通知被淹没**。

**场景 D**：后台 Agent 跑了 2 小时，过程中做了 47 步操作。完成后用户看到一个 PR diff 有 800 行。怎么 review？**异步结果的审查成本太高**。

后台 Agent 是 9.2 中 L4（审批者模式）的 UX 挑战——当用户不实时注视 Agent 时，9.1 的三原则（透明性/可预期性/可控性）全部需要重新设计。

本文回答：后台 Agent 怎么设计 UX 才能让用户**放心离开**又**高效回来**？

## §1 第一性原理：同步交互 → 异步委托的范式转变

### 1.1 两种 Agent 交互范式

| 维度 | 同步（Foreground） | 异步（Background） |
|---|---|---|
| 用户状态 | 注视屏幕 | 离开/做别的 |
| 反馈循环 | 实时（秒级） | 延迟（分钟-小时） |
| 信任基础 | 看见过程 | 看见结果+审计 |
| 中断方式 | 随时暂停/接管 | 通知触发回来 |
| 主要焦虑 | "它在做什么" | "它有没有做对" |
| 核心 UX 问题 | 透明性 | 可审计性+通知策略 |

**异步模式的本质**：用户把"审查过程"的权利推迟到"审查结果"——这需要 Agent 在过程中**自主判断何时升级**。

### 1.2 6 种后台 Agent 的交付模式

2026 年主流产品的交付模式（[ssojet.com](https://ssojet.com/blog/background-coding-agents)）：

| 产品 | 触发方式 | 交付物 | 进度追踪 |
|---|---|---|---|
| **Devin** | Slack / Web / API | PR + 全程录屏 | 实时面板 |
| **OpenAI Codex** | IDE / Web | Diff / PR | 完成通知 |
| **GitHub Copilot Agent** | Issue assign / @mention | PR | PR 出现 |
| **Cursor Cloud Agents** | IDE kick-off | Branch + 通知 | IDE 内 |
| **Google Jules** | Web app | PR | Web 面板 |
| **Claude Code (CI)** | @mention / CI event | PR comment | PR 线程 |

**共性**：Pull Request 是主流交付物。PR 天然支持审查、评论、回滚——是异步协作的成熟载体。

### 1.3 "Fire-and-Forget" 不是好 UX

虽然这些产品都是"分配任务→等结果"模型，但**纯 fire-and-forget 不够**：
- 用户不知道任务卡住了还是在跑
- 遇到需要决策的岔路口没人做判断
- 完成后审查压力太大（缺过程信息）

好的后台 Agent UX 不是"完全离开"，而是**"低频检查+关键时刻召回"**。

## §2 通知策略：何时打扰用户

### 2.1 通知的 4 个级别

| 级别 | 何时触发 | 通知方式 | 示例 |
|---|---|---|---|
| **Silent** | 正常进度更新 | 仅更新面板/日志 | "Step 3/7 完成" |
| **Passive** | 里程碑完成 | Badge / 面板高亮 | "任务 A 完成" |
| **Active** | 需要用户关注 | Push 通知 | "任务 B 有问题" |
| **Urgent** | 需要立刻介入 | 强通知+声音 | "外部 API 权限过期" |

**核心原则**：**绝大多数事件应该是 Silent 或 Passive**。只有真正需要用户做决策时才 Active/Urgent。

### 2.2 通知疲劳防御

如果 Agent 每完成一步都通知，用户很快就 mute 了。设计策略：

| 策略 | 实现 |
|---|---|
| **聚合通知** | 5 分钟内的多条更新合并为一条摘要 |
| **上下文感知** | 用户在开会→攒着不发，会后一起推 |
| **升级阈值** | 只有卡住 > 5 分钟才升级为 Active |
| **用户偏好** | 允许设置"只在完成/失败时通知" |
| **Do Not Disturb** | 尊重系统 DND 状态 |

### 2.3 何时"召回"用户

最关键的设计决策：**什么情况下必须打断用户？**

| 触发条件 | 紧急度 | 处理 |
|---|---|---|
| 任务完成 | Passive | Badge + 等用户回来看 |
| 遇到需要决策的岔路口 | Active | Push + 清晰问题描述 |
| 需要凭据/权限 | Active | Push + 一键授权链接 |
| 严重错误（影响其他系统） | Urgent | 强通知 + 自动暂停 |
| 超时（卡住超过 N 分钟） | Active | "我卡住了，需要帮助" |
| 即将执行高风险操作 | Active | 暂停+预览+等审批 |

### 2.4 通知的信息密度

好的后台 Agent 通知不是"任务完成"这种空洞消息。它应该是一个**micro-summary**：

```
坏：✅ 任务完成
好：✅ Bug #1234 已修复：修改了 auth.ts 中的 token 过期逻辑，
    添加了 2 个测试。PR #567 待 review。预计影响：低风险。
    [查看 PR] [查看 Agent 日志]
```

```
坏：⚠️ 任务出错
好：⚠️ Bug #1235 修复卡住：第 3 步需要访问 staging DB，
    但连接凭据已过期。
    选择：[提供新凭据] [跳过此步] [终止任务]
```

### 3.2 面板信息层级

| 信息 | 始终可见 | 展开可见 |
|---|---|---|
| 任务名称 + 状态 | ✅ | — |
| 当前步骤 / 进度 | ✅ | — |
| 耗时 | ✅ | — |
| 快捷操作（Pause/View） | ✅ | — |
| 详细步骤列表 | — | ✅ |
| Cost 消耗 | — | ✅ |
| 完整日志 | — | ✅ |

### 3.3 优先级排序

面板中任务的排序不应该是"先来后到"，而应该是**需要注意力的优先排**：

1. 🚨 **Blocked**（需要用户操作）→ 最顶部
2. ⚠️ **Warning**（可能有问题）→ 次之
3. ✅ **Completed**（待审查）→ 第三
4. 🔄 **Running**（正常进行中）→ 最底

### 3.4 批量操作

多任务场景需要批量能力：

| 操作 | 场景 |
|---|---|
| 全部暂停 | 下班前暂停所有任务 |
| 按优先级恢复 | 只跑 P0 任务 |
| 批量 approve | 多个任务等审批 |
| 资源调度 | 并发数限制：同时最多 3 个 |

## §4 进度可视化：让等待有形

### 4.1 进度的 3 种粒度

| 粒度 | 呈现 | 适用 |
|---|---|---|
| **任务级** | 进度条 (Step 3/7) | 多任务面板总览 |
| **步骤级** | 当前步名称 + 预估剩余 | 详情面板 |
| **实时级** | Live terminal/browser 流 | 用户主动深入 |

### 4.2 预估时间

用户在后台等待时最想知道："还要多久？"

```python
def estimate_remaining(task, current_step):
    """预估剩余时间"""
    # 基于历史同类任务
    historical_avg = task_stats.avg_duration(task.type)
    
    # 基于当前进度
    elapsed = now() - task.start_time
    progress_ratio = current_step / task.total_steps
    
    if progress_ratio > 0:
        projected = elapsed / progress_ratio
        remaining = projected - elapsed
    else:
        remaining = historical_avg
    
    # 给出范围而非精确值（诚实）
    return TimeEstimate(
        min=remaining * 0.7,
        max=remaining * 1.5,
        confidence="medium" if progress_ratio > 0.3 else "low"
    )
```

**UX 呈现**：
- ✅ "预计还需 5-10 分钟"（范围）
- ❌ "预计还需 7 分钟 23 秒"（假精度）
- ✅ 低置信度时说"难以估计，通常此类任务需要 10-30 分钟"

## §5 结果交付与审查

### 5.1 结果交付的最佳载体

| 载体 | 优点 | 缺点 | 适合 |
|---|---|---|---|
| **Pull Request** | 可审查/可评论/可回滚 | 需要 Git 平台 | 代码类 |
| **Branch diff** | 轻量、IDE 内直接看 | 无评论/讨论 | 小改动 |
| **Artifact（文件）** | 通用 | 无版本对比 | 文档、报告 |
| **消息摘要** | 即时 | 细节不够 | 快速结论 |

### 5.2 审查成本降低策略

800 行 diff 怎么 review？不能原样丢给用户。后台 Agent 应该**辅助审查**：

| 策略 | 实现 | 效果 |
|---|---|---|
| **变更摘要** | Agent 自己总结改了什么+为什么 | 审查时间 -50% |
| **测试报告** | 自动运行测试 + 报告通过/失败 | 信心 +++ |
| **影响分析** | 哪些下游可能受影响 | 聚焦高风险区 |
| **分层 diff** | 按逻辑分组（不按文件） | 更好理解 |
| **关键决策高亮** | 标注 Agent 做了哪些判断 | 抓住要点 |

```
┌──────────────────────────────────────────────────┐
│ ✅ Task Complete: Refactor payment module         │
│                                                  │
│ 📝 Summary:                                      │
│ 将 payment.ts (245 行) 拆分为 3 个模块：          │
│ • payment-gateway.ts — 外部支付网关对接          │
│ • payment-validator.ts — 输入验证逻辑            │
│ • payment.ts — 协调层（精简至 45 行）            │
│                                                  │
│ 🧪 Tests: 12/12 passed (including 4 new)         │
│                                                  │
│ ⚠️ Key decisions made by Agent:                  │
│ 1. 选择按职责拆分（非按数据流拆分）[查看推理]    │
│ 2. 保留了 legacy callback 接口以兼容 [查看推理]  │
│                                                  │
│ 📊 Impact: 无破坏性变更，接口保持不变            │
│                                                  │
│ [Review PR] [View full log] [Approve & merge]    │
└──────────────────────────────────────────────────┘
```

### 5.3 Cursor 的信任模型

Cursor Cloud Agents 的文档（[stevekinney.com](https://stevekinney.com/courses/ai-development/cursor-background-agents)）明确指出："Treat the branch as untrusted work from another developer: run the tests again and read the changed files."

**这是正确的心智模型**：后台 Agent 的输出等同于一个 Junior Developer 的 PR——需要 review，但不需要每一行都手把手。

## §6 隔离与安全

### 6.1 为什么后台 Agent 需要更强隔离

同步 Agent 用户在场——出问题能立刻停。后台 Agent 无人值守——必须从架构层防止损害。

| 隔离维度 | 机制 | 产品示例 |
|---|---|---|
| **代码隔离** | 独立 branch，不直接改 main | 所有主流产品 |
| **环境隔离** | 独立 VM/容器，不跑在本地 | Cursor Cloud、Devin、Jules |
| **网络隔离** | 只允许白名单出站 | Codex sandbox |
| **权限隔离** | 最小权限原则 | 只给需要的 repo 写权限 |
| **时间隔离** | 超时自动终止 | 最长运行时间限制 |
| **成本隔离** | 单任务预算上限 | 防止无限循环烧钱 |

### 6.2 凭据管理

后台 Agent 需要访问外部服务（DB、API、云平台）。凭据怎么给？

| 方案 | 安全性 | 便捷性 |
|---|---|---|
| 用户每次手动输入 | 高 | 低（打断异步） |
| 加密存储+按需注入 | 中-高 | 高 |
| 短期 token（15min） | 高 | 中（可能过期需续） |
| OAuth scope 限制 | 高 | 高（但需前置配置） |

**推荐**：环境变量 + 短期 token + 过期时升级通知用户。

## §7 长时间运行的特殊挑战

### 7.1 Context Drift

Zylos 2026 数据：context drift 导致 65% 的多步任务失败。后台 Agent 运行时间更长，问题更严重。

应对策略（与 8.4 延迟工程的 context 压缩呼应）：

| 策略 | 实现 |
|---|---|
| 定期目标重申 | 每 N 步重新注入原始任务描述 |
| Memory 蒸馏 | 压缩长 context 为精炼摘要 |
| Checkpoint + 新 session | 每小时重启 context（用 checkpoint 恢复状态） |
| 环境快照 | 将文件系统状态而非对话历史作为真实来源 |

### 7.2 代码库漂移

后台 Agent 在独立 branch 工作时，main 可能已经被其他人修改：

```
     main:  A → B → C → D (others pushed while agent worked)
              \
     agent:   → X → Y → Z (agent's work based on commit B)
```

Agent 完成时需要：
1. Rebase/merge main 的最新变化
2. 检查冲突
3. 重新运行测试
4. 如有冲突→通知用户

### 7.3 成本失控防护

后台 Agent 最大的"暗风险"——在循环中烧钱却不产出有效结果。

```python
class CostGuard:
    """后台 Agent 成本熔断器"""
    
    def __init__(self, budget_usd, max_duration_min, max_steps):
        self.budget = budget_usd
        self.max_duration = max_duration_min
        self.max_steps = max_steps
        self.spent = 0
        self.steps = 0
    
    def check(self, step_cost, elapsed_min):
        self.spent += step_cost
        self.steps += 1
        
        if self.spent > self.budget:
            raise CostLimitExceeded(
                f"预算 ${self.budget} 已用尽（已花 ${self.spent:.2f}）"
            )
        
        if elapsed_min > self.max_duration:
            raise TimeLimitExceeded(
                f"运行已超过 {self.max_duration} 分钟"
            )
        
        if self.steps > self.max_steps:
            raise StepLimitExceeded(
                f"已执行 {self.steps} 步，超过上限 {self.max_steps}"
            )
        
        # 检测循环（重复操作检测）
        if self.detect_loop():
            raise LoopDetected("Agent 可能陷入循环")
```

## §8 反模式表

| # | 反模式 | 后果 | 解药 |
|---|---|---|---|
| 1 | 纯 fire-and-forget 无进度 | 用户焦虑不知状态 | 多任务面板 + 进度条 |
| 2 | 每步都 push 通知 | 通知疲劳→mute | 4 级通知 + 聚合 |
| 3 | 结果只给一个大 diff | 审查成本爆炸 | 摘要+测试+关键决策高亮 |
| 4 | 卡住了不升级 | 用户不知道任务 blocked | 超时→Active 通知 |
| 5 | 无成本上限 | 循环烧钱 | CostGuard 熔断器 |
| 6 | 后台 Agent 直接改 main | 不可逆风险 | 独立 branch + PR |
| 7 | 凭据永久存储无过期 | 安全隐患 | 短期 token + 自动续期/升级 |
| 8 | 无运行时间上限 | Context drift + 永不结束 | max_duration + checkpoint 重启 |
| 9 | 完成通知无摘要 | 用户不知道值不值得看 | micro-summary 通知 |
| 10 | 多任务无优先级 | Blocked 任务被淹没 | 注意力优先排序 |
| 11 | 无法"窥视"实时状态 | 只能等结果 | View Live 模式 |
| 12 | 不处理 main 漂移 | Merge 时冲突爆炸 | 定期 rebase + 冲突检测 |

## §9 S0 → S5 演进路线

| 阶段 | 后台能力 | 通知 | 多任务 | 审查辅助 |
|---|---|---|---|---|
| **S0** | 无后台运行 | — | — | — |
| **S1** | 单任务后台 | 完成/失败通知 | 无 | 原始 diff |
| **S2** | 多任务队列 | 4 级通知 | 列表面板 | 摘要 |
| **S3** | 并行任务+隔离 | 智能升级+DND | 优先级面板 | 摘要+测试+影响 |

| **S5** | 多 Agent 后台编排 | 跨 Agent 协调通知 | 全局任务图 | 自动审查 Agent |

| **后台 Agent 最大 UX 挑战？** | 用户不在场时的信任建立——透明性要从"实时看过程"变成"事后可审计" |
| **通知怎么不烦人？** | 4 级分层（Silent/Passive/Active/Urgent）+ 聚合 + DND |
| **什么时候必须打断用户？** | 需要决策（岔路口）/ 需要权限 / 严重错误 / 超时卡住 |
| **多任务怎么管？** | 面板 + 按"需要注意力"排序（Blocked > Warning > Done > Running） |
| **结果怎么审查不累？** | 摘要+测试+关键决策高亮+分层 diff |
| **怎么防烧钱？** | 预算上限 + 时间上限 + 步数上限 + 循环检测 |
| **后台需要什么隔离？** | 独立 branch + 容器 + 白名单网络 + 短期凭据 + 成本熔断 |
| **正确心智模型？** | 把后台 Agent 当"一个 Junior Dev 的 PR"——需要 review，但不需要监工 |
| **第一性原则** | 后台 Agent 的 UX 核心是"低频检查+关键召回"，不是完全放手 |

## §11 与其他模块的连接

- **接 9.1 UX 原则**：后台场景下透明性从"实时看"变成"事后审计+摘要"
- **接 9.2 自主度**：后台 Agent 通常是 L4（审批者），通知策略是 L4 控制权的 UX 实现
