# 1.6 结构化输出工程：JSON Mode / Constrained Decoding / Outlines / xgrammar / Pydantic

> **写在前面**：本文系统化讲 Agent 时代的"让模型说人话之外的话"——把 LLM 的自由文本压成数据库可入、API 可消费、下游可断言的结构化格式。覆盖三条技术路线：**Prompt-only**（让模型"自觉"输出 JSON）、**Provider 原生 strict / response_format**（OpenAI / Anthropic / Gemini）、**Engine-level Constrained Decoding**（Outlines / xgrammar / Guidance / lm-format-enforcer / llama.cpp grammar / vLLM）。结合 2025-2026 的实测数据与推理模型时代的新约束。

## §0 真问题：Agent 跑两小时后吐了个 trailing comma

把 LLM 接到生产管线里，第一周就会撞墙的真问题：

第一，**JSON 不可解析的尾部 bug**。模型在长输出里漏一个引号、加一个逗号、提前截断，整个 batch 失败。即使 prompt 里写"only output JSON"也只能压到 95%-99% 通过率——剩下的 1-5% 在百万级调用里就是几万条脏数据。

第二，**Schema 通过但语义错误**。例如分类任务字段类型对、enum 值对，但**所有样本 confidence 都是 0.99**——模型摸到了"安全分布"。这种错误 schema validator 看不到。

第三，**推理模型加了一层乱码**。o1 / R1 / Claude Extended Thinking 经常把 reasoning 与 final answer 混在一起输出，结构化解析容易把 thinking 误解析成数据；GPT-5 的 `verbosity=high` 也会包裹更多 markdown 围栏。

第四，**Tool Use 与 response_format 是两套 API**。同一个 Pydantic model 在 tool 调用与最终输出里要写两套适配。多 provider 时 schema 还得分别适配 OpenAI strict、Claude tool_input_schema、Gemini responseSchema 三种方言。

第五，**约束太强会掉精度，约束太弱会丢可靠性**。XGrammar 在 GitHub Hard 数据集上常 under-constrained（schema 通过但结构错），Outlines 在大 enum / minItems 上经常 compile timeout，OpenAI strict 上跨表 oneOf 不支持。

第六，**Reasoning model 与 strict 的兼容性**。GPT-5 系列 `reasoning_effort=minimal` 不支持 parallel tool calls；Claude Extended Thinking 在 strict JSON schema 模式下 thinking 块要求与最终输出独立结构……碰一次坑都是数小时排查。

本文给出一套**分层框架**：Prompt-only / Provider 原生 / Engine-level，从问题出发对应到工程方案。

## §1 第一性原理：结构化输出的三种实现机制

### 1.1 Prompt-only：纯文本约束

最朴素的做法：prompt 里写"Return strictly valid JSON conforming to this schema: ..."，模型靠 prior 估计 next-token 分布，靠 instruction-following 自觉服从。

机制：**完全在模型外部，无 decoding 干预**。

代价与上限：
- 通过率上限：GPT-4 90-95%，GPT-5 95-99%，Claude Sonnet 4 类似。
- 长输出退化：>2K tokens 后通过率显著下降（中间会"忘了 schema"）。
- 推理模型反而更差：Reasoning 输出过程会插入 markdown / commentary。

适用场景：**只用一次、低风险**。Demo、小工具、原型。**不要**用在 production。

### 1.2 Provider 原生 strict / response_format：服务端约束

OpenAI 2024-08 推 strict mode，承诺 100% schema 遵从；Anthropic、Gemini 跟进类似能力。

机制：在模型 backend 里**对 logits 做 mask** + **schema 编译为有限状态机** + **fall-back 到拒答**。OpenAI strict 限制 JSON Schema 子集（`additionalProperties: false` 必须、`required` 必须列全所有 properties、有限 `oneOf` 用 discriminator）。Anthropic 通过 `tool_use` 的 `input_schema` 实现等价能力；Claude 4.x 也提供独立的 structured outputs（`output_format`）。Gemini `responseSchema` 同理。

代价：
- **不可移植**——schema 方言彼此不同；多 provider Agent 要写适配层。
- **被限制的 JSON Schema 子集**：高级关键字（`patternProperties`、复杂 `oneOf`、`if/then/else`）通常不支持。
- **冷启动延迟**：第一次提交新 schema 时 backend 要编译/缓存（OpenAI ~10s，Anthropic 缓存 24h——schema 改一字就失效，[Wilkins 2026](https://collinwilkins.com/articles/structured-output)）。
- **不支持 self-host**：闭源 SaaS only。

适用场景：**大多数 production**。强烈推荐先用 provider 原生，失败再考虑 engine-level。

### 1.3 Engine-level Constrained Decoding：客户侧约束

把 schema 编译成 grammar，在每个 decode step 用 grammar 限制可选 token 集合。代表工具：

- **Outlines**：把 JSON Schema 转 regex，再转 FSM。覆盖广，但 regex 编译慢，大 enum / minItems 会爆。
- **xgrammar**（MLC AI 2024）：pushdown automaton 加速 grammar mask 计算，TPOT 增加极小，已集成 vLLM、SGLang、TensorRT-LLM。
- **Guidance**：微软系，用 token healing 技巧避免边界 token 错误。JSONSchemaBench 显示综合最强。
- **lm-format-enforcer**：纯 Python，集成 vLLM / Transformers，trade-off 在易用 vs 性能。
- **llama.cpp GBNF**：本地推理的事实标准，GBNF 语法自带 JSON schema 转换器。
- **vLLM Structured Outputs**：v0.7+ 同时支持 outlines、xgrammar、lm-format-enforcer 三个 backend。

机制：每步 decode 时拿当前已生成的 prefix，用 grammar 算出"合法 next token 集合"，把不合法 token 的 logits 设为 -∞，再做 sampling/argmax。

代价：
- **必须能控制 inference**——SaaS API 不行，自托管 vLLM/SGLang/llama.cpp 才用得上。
- **Coverage vs 通过率的权衡**：xgrammar 偏 permissive（under-constrained 多），Guidance 偏 strict 但 compile error 也较多。
- **Reasoning model 兼容性差**：自托管的 R1/Qwen QwQ 跑 grammar mask 时常和 `<think>` 标签冲突——要么先关 thinking 再开 grammar，要么 grammar 里显式写 thinking 区段。

适用场景：**自托管推理、严苛 schema 要求、追求最低延迟**。

### 1.4 三者的本质区别

|  | Prompt-only | Provider 原生 | Engine-level |
|---|---|---|---|
| 约束位置 | 模型外 | 模型 backend | 解码器 |
| 通过率 | 95-99% | ~100% schema 形式 | ~100% schema 形式 |
| 性能成本 | 0 | 编译开销摊销后 ~0 | TPOT +5-30% |
| Schema 表达力 | 任意 | 受限子集 | xgrammar/Outlines 较全 |
| Self-host | ✅ | ❌ | ✅ |
| Multi-provider | ✅ | ❌ | 不适用 |

## §2 数据：JSONSchemaBench 2025-01 关键发现

引自 Geng et al. 的 [JSONSchemaBench, arXiv:2501.10868](https://arxiv.org/abs/2501.10868)，约 1 万真实 JSON schema，涵盖 GlaiveAI（function call）、Kubernetes、Snowplow、GitHub schemas。

**效率（Llama-3.1-8B, llama.cpp backend）**：

| Dataset | Framework | TPOT (ms) |
|---|---|---|
| GlaiveAI | LM only | 15.40 |
| | Guidance | 6.37 |
| | llama.cpp GBNF | 29.98 |
| | Outlines | 30.33 |

**反直觉**：Guidance 由于 token healing，**TPOT 比无约束还低**——约束信息提前压缩了 sampling 空间。"Constrained decoding can speed up the generation process by 50%."

**Coverage / Compliance Rate**（empirical/compliance）：

| Dataset | Guidance | llama.cpp | Outlines | XGrammar | OpenAI |
|---|---|---|---|---|---|
| GitHub Hard | 0.41 / 0.69 | 0.39 / 0.63 | 0.03 / 0.06 | 0.28 / 0.41 | 0.09 / 1.00 |
| Kubernetes | 0.91 / 0.92 | 0.76 / 0.78 | 0.57 / 0.58 | 0.07 / 0.58 | 0.21 / 1.00 |
| JsonSchemaStore | 0.30 / 0.88 | 0.38 / 0.69 | 0.09 / 0.24 | 0.33 / 0.43 | 0.06 / 1.00 |

**关键 takeaway**：闭源服务（OpenAI/Gemini）coverage 低（拒受复杂 schema）但 compliance 高（接受了就 100% 合规）；开源 engine 反过来，coverage 高但 under-constrained 风险显著。

**质量（下游任务正确率）**：

| Framework | GSM8K | Last Letters |
|---|---|---|
| LM only | 80.1% | 50.7% |
| XGrammar | 83.7% | 51.2% |
| Outlines | 81.6% | 53.3% |
| Guidance | 83.8% | 54.0% |

**反直觉**：约束解码不仅没掉精度，反而**提升下游正确率 ~4%**——之前"constrained decoding 损害质量"的说法被反驳。机制：grammar 把模型从语法歧义里救出来，让它专注于内容。

## §3 决策矩阵：怎么选

| 你的情况 | 推荐方案 |
|---|---|
| 单 provider（OpenAI / Anthropic / Gemini）+ 简单 schema | Provider 原生 strict |
| 单 provider + 复杂 schema（patternProperties / 大 oneOf） | 拆 schema → provider 原生；不行再换 Instructor + 重试 |
| 多 provider，schema 共用 | **Pydantic / Zod 模型 + Instructor / LangChain `with_structured_output`** |
| 自托管 vLLM / SGLang | **xgrammar**（默认） |
| 自托管 llama.cpp / Ollama | **GBNF grammar** |
| 追求最高 schema compliance | **Guidance** |
| 用 Outlines | 别用了——大 schema 编译超时是已知问题 |
| Reasoning model + 必须结构化 | provider 原生 + thinking summary 单独处理；自托管见 §4.5 |
| 通过率敏感 + 不可重试 | strict + Instructor 双保险 |

## §4 工程实施：Pydantic + Instructor 范式

### 4.1 单一真实定义

```python
from pydantic import BaseModel, ConfigDict, Field
from typing import Literal

class IssueClassification(BaseModel):
    """Bug tracker triage 用结构化输出"""
    model_config = ConfigDict(extra='forbid')  # 关键：禁止额外字段
    
    reasoning: str = Field(description="先想清楚再回答")  # 关键：放在 answer 前
    priority: Literal["P0", "P1", "P2", "P3"]
    component: Literal["frontend", "backend", "infra", "ml", "other"]
    is_regression: bool
    suspected_cause: str = Field(max_length=300)
    confidence: float = Field(ge=0.0, le=1.0)
```

字段顺序原则：**reasoning 字段必须在 answer 字段之前**，否则 reasoning model 会"先决定答案再编造理由"——这是 [Wilkins 2026](https://collinwilkins.com/articles/structured-output) 指出的常见陷阱。

### 4.2 跨 provider 调用

```python
import instructor
from openai import OpenAI
from anthropic import Anthropic

# OpenAI strict
oai_client = instructor.from_openai(OpenAI())
result = oai_client.chat.completions.create(
    model="gpt-5",
    response_model=IssueClassification,
    messages=[{"role": "user", "content": issue_text}],
    max_retries=2,
)

# Anthropic
ant_client = instructor.from_anthropic(Anthropic())
result = ant_client.messages.create(
    model="claude-sonnet-4-5",
    response_model=IssueClassification,
    max_tokens=2048,
    messages=[{"role": "user", "content": issue_text}],
    max_retries=2,
)
```

Instructor 把 Pydantic model 在底层自动转换：OpenAI → strict tool；Anthropic → tool_use schema；Gemini → responseSchema。失败时把 ValidationError 注入 repair prompt 重试。

### 4.3 Validator-first 流水线

```python
from prometheus_client import Counter, Histogram

validation_failures = Counter("llm_structured_validation_fails", "fail count", ["model", "schema"])
confidence_dist = Histogram("llm_confidence", "field-level distribution")

def safe_classify(issue_text: str, max_retries=1):
    for attempt in range(max_retries + 1):
        try:
            result = oai_client.chat.completions.create(
                model="gpt-5",
                response_model=IssueClassification,
                messages=[{"role": "user", "content": issue_text}],
                max_retries=0,  # 由我们自己管 retry
            )
            # 语义校验（schema 之外）
            assert 0.05 < result.confidence < 0.99, "suspect confidence saturation"
            confidence_dist.observe(result.confidence)
            return result
        except (ValidationError, AssertionError) as e:
            validation_failures.labels(model="gpt-5", schema="IssueClassification").inc()
            if attempt == max_retries:
                # 永远 fallback 到一个安全默认值，不让 Agent 死掉
                return IssueClassification(
                    reasoning="validation_fallback",
                    priority="P2", component="other", is_regression=False,
                    suspected_cause="parsing failed", confidence=0.5
                )
```

要点：
- **Schema 通过 ≠ 任务成功**。confidence 分布、enum 频次必须监控。
- **失败回退到安全默认**，绝不让 ValidationError 击穿上游。
- **限制 retry 次数**。Wilkins 法则："如果常需要 2 次以上重试，是 prompt 或 schema 出问题，不是 retry 出问题。"

### 4.4 与 Function Calling 的关系

Function Calling 本质就是 strict structured outputs 的一种特殊形态——把"调用什么工具 + 传什么参数"用 JSON schema 表达。所以现代 Agent 设计里：

- **outputs**（最终结果）：用 `response_format` / `structured_outputs`
- **tool calls**（中间动作）：用 `tools` + `tool_choice`
- 两者**可以并存**——OpenAI Responses API、Anthropic Messages API 都支持

不要混用：把所有最终输出都伪装成"submit_final_answer" tool 是反模式（增加 token 成本，混淆 monitor）。

### 4.5 Reasoning Model 时代的新约束

**1. Thinking 字段不要塞进 schema**。Reasoning 已经在模型内部做了，schema 里再加 `reasoning_chain: list[str]` 是双重消耗。要可解释性就用 `reasoning.summary`（OpenAI）或 `thinking` block（Anthropic），**它们不是 user schema 的一部分**。

**2. GPT-5 `verbosity` 不要拉满**。`verbosity=high` 在 structured output 上不会让 schema 更精确，只会塞更多 markdown 注释——可能触发 strict 失败。给 production 用 `verbosity=low`。

**3. Claude Extended Thinking 与 strict JSON 共用时**：必须把 thinking 块当作前缀输出，再让模型在 final assistant message 里给 JSON。不要试图"让 thinking 也变 JSON"——会破坏 thinking 的语义。

**4. 自托管 R1 + xgrammar**：在 `<think>...</think>` 区段关闭 grammar mask，区段外打开。SGLang 2025-Q3 之后支持 `region-based grammar`，可显式声明 thinking 区段。

**5. Streaming + structured outputs**：xgrammar / Outlines / Guidance 都支持流式输出有效前缀，但解析端要用增量 JSON parser（如 `json-stream-parser`、Python 的 `ijson`），不能等整体到位再 `json.loads`。

## §5 反模式

**反模式 1：用 regex 解析 LLM JSON**。永远会有 edge case。直接换 Instructor。

**反模式 2：在 prompt 里贴 OpenAPI 完整 schema**。3000 行 schema 喂进 prompt 是 token 浪费 + 性能下降。**应该用 Pydantic / Zod 写 model，由库自动生成精简 schema**。

**反模式 3：reasoning 字段放在 answer 之后**。模型会先决定答案再编理由。Field order matters.

**反模式 4：大 enum (>100) 用 enum 类型**。会让 Outlines 编译超时；Guidance/xgrammar 也慢。改成 string + post-validation。

**反模式 5：嵌套深度 >3**。每加一层嵌套，模型出错率非线性上升。Flatten it.

**反模式 6：不监控字段分布**。所有样本 confidence=0.99、所有 category 都是 "other"，schema validator 不报错但任务已经废了。dashboard 必须包含字段分布、enum 频次、空值率。

**反模式 7：把 strict 当 100% 保险**。Strict 只保证**格式**对，不保证**内容**对。模型完全可以输出 schema-valid 但语义错误的结果。

**反模式 8：每个调用都用 strict**。strict 在 schema 简单时无开销，在 schema 复杂时编译延迟显著。简单任务用 prompt + Pydantic validate 已经够了。

**反模式 9：Gemini schema 调试用静默忽略**。Gemini 对不支持的 schema 关键字**静默忽略**——你以为约束生效了其实没有。**必须 end-to-end 测**，包括 negative case。

**反模式 10：schema 改完直接上**。Anthropic 的 24h grammar cache 失效后下一次调用首跳延迟会显著上升；OpenAI strict 编译也要 ~10s。schema 改动要在低峰期发布或 warm-up。

## §6 S0→S5 演进路线

**S0 - 字符串解析**：`json.loads` + try/except + 失败时丢弃。可用于 demo。

**S1 - Pydantic + 单次 retry**：定义 Pydantic model，用 model.parse_raw + 一次自我修复 prompt 重试。已经覆盖 95% 场景。

**S2 - Instructor + Provider strict**：用 Instructor 自动适配 OpenAI strict / Anthropic tool_use。retry=2，加 fallback。生产可用。

**S3 - 监控 + 语义校验**：监控字段分布、enum 频次、confidence 分布、空值率。语义级 assertion（不只是类型）。

**S4 - Multi-provider 路由**：同一个 Pydantic schema 跑多个 provider，根据成本/延迟/质量选最优。配合 prompt cache。

**S5 - Self-host + Engine-level**：对延迟、隐私敏感的场景上 vLLM + xgrammar；对 schema compliance 极端敏感的场景上 Guidance。配合 reasoning model（R1/QwQ）做本地推理。

## §7 一页总结

第一性原理：结构化输出本质是**约束的位置**——prompt 端（外部）、provider 端（backend）、decoder 端（grammar mask）。三个位置的成本-能力-可移植性 trade-off 决定选型。

工程不变量：**Pydantic/Zod 单一真实定义**、**Validator-first 流水线**、**字段分布监控**（不是只看类型）、**失败回退到安全默认**、**reasoning 字段在 answer 之前**。

最危险的反模式：用 regex 解析 JSON；3000 行 schema 喂 prompt；reasoning 字段顺序倒置；strict 当 100% 保险；不监控字段分布。

最强信号：Constrained decoding 在多数 benchmark 上**反而提升下游精度 ~4%**——只要用对工具（Guidance、xgrammar）。

向后衔接：与 1.4 Function Calling/Tool Use 协议是孪生关系（tools 是 structured outputs 的子集）；与 2.5 Tool Schema 设计原则、3.6 任务分解与结果聚合（Worker 间数据流契约）、6.2 Guardrails（output 层护栏）共同构成 Agent 的"数据契约"层。
