# 9.3 中断、暂停、接管、回滚的 UX 模式

> "好的 Agent 不只是能做对——它还能被优雅地打断，并从断点继续。"

## §0 真问题：Agent 被打断后的四种死法

**场景 A**：用户在 Agent 多步执行到第 3 步时按了"暂停"。Agent 停了，但用户想继续时发现——**必须从头开始**。中间步骤的结果全丢了。

**场景 B**：Agent 正在调用外部 API（发邮件），用户紧急按了 Stop。但消息已经发出去了——**中断太晚，不可逆操作已经执行**。

**场景 C**：Agent 做了一系列文件修改，用户审查后发现第 2 步就错了。但现在已经到第 5 步，**没有办法只回滚第 2 步而保留 3-5**。要么全撤销，要么手动一个个改。

**场景 D**：用户接管了 Agent 的一步（手动操作），完成后想让 Agent 继续后面的步骤。结果 Agent 不认识用户手动产生的状态——**接管后无法续接**。

四种死法的共同根因：**Agent 的状态管理不支持中断/恢复语义**。这不是 UX 表层问题——是系统架构问题在 UX 层面的暴露。

本文回答：怎么设计一个"打断友好"的 Agent 交互系统？

---

## §1 第一性原理：控制权的 4 种操作

用户对 Agent 执行的控制权本质上有且只有 4 种操作：

| 操作 | 含义 | 类比 |
|---|---|---|
| **Pause（暂停）** | 当前步完成后停住，保留全部状态 | 暂停视频 |
| **Interrupt（中断）** | 立刻停止当前步，丢弃当前步的部分结果 | 强制终止进程 |
| **Takeover（接管）** | 用户替代 Agent 执行某一步 | 从 AI 手里拿回方向盘 |
| **Rollback（回滚）** | 撤销已完成的步骤，回到之前的状态 | git revert |

这 4 种操作的**组合**覆盖了所有用户干预场景：

| 场景 | 操作组合 |
|---|---|
| "等一下，让我想想" | Pause |
| "停！别继续了" | Interrupt |
| "这步我自己来" | Pause → Takeover → Resume |
| "第 2 步错了，重来" | Rollback(step 2) → Resume from step 2 |
| "全部撤销" | Rollback(all) |
| "方向不对，换个方案" | Pause → Redirect（修改计划）→ Resume |

## §2 Pause（暂停）：3 种粒度

### 2.1 步间暂停（Step-boundary Pause）

**最常用、最安全**。Agent 完成当前步骤后，不启动下一步。

```
Steps: [✅ Done] [✅ Done] [⏸️ Paused here] [⬜ Pending] [⬜ Pending]
                                    ↑
                              用户按了暂停
```

实现要点：
- Agent 执行循环在每步之间检查 `pause_flag`
- 当前步骤的所有状态都已持久化（checkpoint 已写）
- 恢复时直接执行下一步，无需重做任何工作

```python
class AgentExecutor:
    def run(self, plan):
        for i, step in enumerate(plan.steps):
            # 步间暂停检查
            if self.pause_requested:
                self.checkpoint(i, plan.state)
                self.notify_user("已暂停，随时可继续")
                yield PauseEvent(step_index=i, state=plan.state)
                return
            
            result = self.execute_step(step)
            plan.state.update(result)
            self.checkpoint(i + 1, plan.state)
```

### 2.2 步内暂停（Mid-step Pause）

某些步骤执行时间很长（大文件处理、多 API 调用）。用户可能需要在步骤内部暂停。

**问题**：步内暂停的状态一致性更难保证。

设计策略：**把大步骤拆成子步骤**，暂停只发生在子步骤边界。

```python
def execute_large_step(self, step):
    """大步骤分解为可暂停的子步骤"""
    sub_steps = self.decompose(step)
    for sub in sub_steps:
        if self.pause_requested:
            self.checkpoint_partial(step, sub.index)
            return PartialResult(completed=sub.index)
        sub.execute()
```

### 2.3 思考中暂停（Thinking Pause）

Agent 正在 LLM 推理中（等 API 返回）。这时暂停意味着：
- 放弃当前正在生成的 response
- 或者等 response 完成后暂停

**推荐**：等 response 完成 → 然后在"执行 tool call 之前"暂停。原因：LLM 调用本身无副作用（不改变外部世界），但 tool call 可能有副作用。

```
Timeline:
  User → Request → [LLM thinking...] → Response → [暂停点] → Tool Call → Result
                                                      ↑
                                              用户按暂停，在这里停
```

## §3 Interrupt（中断）：紧急制动

### 3.1 中断的时间窗口问题

中断和暂停的关键区别：**中断可能在不可逆操作执行中途发生**。

```
Timeline:
  Step 1 (read) → Step 2 (write file) → Step 3 (send email)
                                              ↑
                                         如果中断在这里...
                                         邮件可能已发出
```

### 3.2 Pre-commit / Post-commit 设计

借鉴数据库事务思想：

| 阶段 | 中断行为 |
|---|---|
| **Pre-commit**（准备阶段，未执行副作用） | 安全中断，丢弃当前步 |
| **Post-commit**（副作用已执行） | 中断只能停止后续步骤，当前步不可撤 |

**设计含义**：每个有副作用的 step 分成 "prepare" 和 "execute" 两个子阶段：

```python
class Step:
    def prepare(self) -> PreviewResult:
        """生成预览，但不执行。可安全中断。"""
        return PreviewResult(
            action="send_email",
            preview=self.render_email(),
            reversible=False
        )
    
    def execute(self) -> ExecuteResult:
        """真正执行。一旦开始，应视为已提交。"""
        return self.api.send_email(self.payload)
    
    def rollback(self) -> Optional[RollbackResult]:
        """如果可逆，执行回滚。"""
        if self.reversible:
            return self.api.undo(self.execution_id)
        return None  # 不可逆
```

### 3.3 Emergency Stop 的实现

有些场景需要"**立刻**停止一切"——即使当前步可能处于不一致状态。

| 组件 | 行为 |
|---|---|
| 前端 | 🛑 大红按钮，始终可见 |
| Agent Loop | 检查 `kill_flag`，立刻退出 |
| Running Tool | 取消 HTTP 请求（abort signal） |
| Post-interrupt | 进入"受损状态恢复"流程 |

```python
async def execute_with_emergency_stop(self, step):
    try:
        async with asyncio.timeout(step.timeout):
            result = await step.execute()
    except asyncio.CancelledError:
        # Emergency stop triggered
        await self.enter_recovery_mode(step)
        raise AgentInterrupted(
            step=step,
            state="partial",
            recovery_options=self.assess_recovery(step)
        )
```

### 3.4 中断后的状态诊断

中断后系统必须告诉用户当前状态：

```
┌──────────────────────────────────────────┐
│ ⚠️ Agent 已中断                           │
│                                          │
│ 已完成步骤：                              │
│   ✅ Step 1: 读取文件 (可逆)              │
│   ✅ Step 2: 修改 config.json (可逆)      │
│   ⚠️ Step 3: 发送通知 (不可逆，已执行)    │
│                                          │
│ 未执行步骤：                              │
│   ⬜ Step 4: 更新数据库                   │
│   ⬜ Step 5: 清理临时文件                 │
│                                          │
│ 选择：                                    │
│   [🔄 从 Step 4 继续] [↩️ 回滚到 Step 1]  │
│   [✏️ 修改计划后继续] [🛑 放弃全部]       │
└──────────────────────────────────────────┘
```

## §4 Takeover（接管）：人机控制权交接

### 4.1 接管协议

接管不是"用户手动做了某件事 Agent 不知道"——它是一个**协议化的控制权转移**。

完整的接管流程：

```
Agent → [暂停于 Step N] 
     → 通知用户："Step N 准备就绪，控制权交给你"
     → 提供上下文：目标、约束、可用工具
User → [手动执行 Step N]
     → 通知 Agent："我完成了，这是结果"
     → 提供结果的结构化描述
Agent → [验证结果满足 Step N 的 post-condition]
     → [从 Step N+1 继续]
```

### 4.2 接管时的上下文传递

Tianpan.co 2026 的 Escalation Protocol（[tianpan.co](https://tianpan.co/blog/2026-04-10-escalation-protocol-agent-to-human-handoffs)）指出：交接时最大的浪费是"人工花 15 分钟重建上下文，而结构化 payload 30 秒就能传达"。

**Handoff payload 三层结构**：

| 层 | 内容 | 目的 |
|---|---|---|
| **Action History** | 之前每步做了什么、得到什么、为什么 | 上下文 |
| **Current Context** | 当前状态、需要决策的问题、可选方案 | 聚焦 |
| **Structured Question** | 明确的选项/操作要求 | 加速决策 |

```python
@dataclass
class HandoffPayload:
    # Layer 1: History
    completed_steps: list[StepSummary]
    key_decisions: list[Decision]
    
    # Layer 2: Current
    current_state: dict
    escalation_reason: str
    available_options: list[Option]
    
    # Layer 3: Question
    question: str  # "请选择支付网关：Stripe vs Adyen"
    urgency: str   # "low" | "medium" | "high"
    deadline: Optional[datetime]
```

### 4.3 接管后的续接

接管最难的部分：**用户做完后，Agent 怎么理解新状态？**

三种策略：

| 策略 | 适用场景 | 复杂度 |
|---|---|---|
| **结构化输出** | 用户通过表单提交结果 | 低 |
| **状态 diff** | Agent 检测文件/系统变化 | 中 |
| **LLM 理解** | Agent 读用户的自然语言描述来更新状态 | 高 |

最稳健的方案是**结构化 + LLM fallback**：

```python
async def resume_after_takeover(self, step, user_result):
    # 优先用结构化结果
    if user_result.structured:
        new_state = self.apply_structured(user_result.data)
    else:
        # Fallback: LLM 理解用户描述
        new_state = await self.llm_interpret(
            context=self.state,
            user_description=user_result.text,
            expected_schema=step.output_schema
        )
    
    # 验证 post-condition
    if not step.post_condition.check(new_state):
        raise TakeoverIncomplete(
            missing=step.post_condition.unmet(new_state)
        )
    
    self.state.update(new_state)
    return self.continue_from(step.index + 1)
```

### 4.4 Partial Takeover：不完全接管

有时用户只想改 Agent 结果的一部分：

- Agent 生成了代码，用户想改其中 3 行
- Agent 规划了 5 步，用户想改第 3 步的参数
- Agent 选了工具 A，用户想换成工具 B

**设计**：让用户能**编辑** Agent 的中间产出，而不是"接管整步"。

```
Agent 生成：config = {host: "db-prod", port: 5432, ssl: true}
用户修改：config = {host: "db-staging", port: 5432, ssl: false}
Agent 继续：用修改后的 config 执行后续步骤
```

这对应 9.2 中 L2（协作者）的交互模式——"用户可以直接修改 Agent 的产出"。

## §5 Rollback（回滚）：撤销已完成的步骤

### 5.1 回滚的 3 层体系

| 层 | 机制 | 适用 | 粒度 |
|---|---|---|---|
| **应用层回滚** | Undo stack（操作反向执行） | 文件修改、配置变更 | 步骤级 |
| **系统层回滚** | Checkpoint restore（状态快照恢复） | 复杂状态机 | 快照级 |
| **基础设施层回滚** | Git revert / DB transaction rollback | 代码、数据 | commit 级 |

### 5.2 Checkpoint 设计

Indium 2026（[indium.tech](https://www.indium.tech/blog/7-state-persistence-strategies-ai-agents-2026/)）研究发现：**运行超过 4 小时的 Agent 如果没有状态持久化，有 90% 更高的完全失败风险**。

Checkpoint 需要保存：

```python
@dataclass
class Checkpoint:
    id: str
    timestamp: datetime
    step_index: int
    
    # Agent 内部状态
    agent_state: dict        # 变量、中间结果
    plan: Plan               # 当前计划（可能被修改过）
    memory: list[Message]    # 对话/推理历史
    
    # 外部状态快照
    file_snapshots: dict[str, bytes]  # 被修改文件的备份
    tool_states: dict[str, Any]       # 工具状态
    
    # 元数据
    reversible_steps: list[int]       # 哪些步可逆
    external_effects: list[Effect]    # 不可逆外部影响
```

### 5.3 回滚的可行性分析

不是所有步骤都能回滚。Agent 必须在**执行前**告知用户回滚可行性：

| 操作类型 | 回滚可行性 | 回滚方式 |
|---|---|---|
| 创建文件 | ✅ 完全可逆 | 删除创建的文件 |
| 修改文件 | ✅ 完全可逆（有备份） | 恢复备份 |
| 删除文件 | ⚠️ 条件可逆（有备份/git） | 从备份恢复 |
| API 读取 | ✅ 无需回滚（无副作用） | — |
| 发送消息 | ❌ 不可逆 | 只能发更正 |
| 数据库写入 | ⚠️ 有事务时可逆 | ROLLBACK |
| 外部支付 | ❌ 不可逆（需人工退款） | 人工介入 |

### 5.4 选择性回滚 vs 全量回滚

用户可能需要"只撤销第 3 步，保留第 4-5 步"。这在有依赖关系时很复杂：

```
Step 1 → Step 2 → Step 3 → Step 4 → Step 5
                     ↑ 
              用户想回滚这步

如果 Step 4 依赖 Step 3 的输出 → 必须级联回滚 4+5
如果 Step 4 不依赖 Step 3   → 可以只回滚 3
```

**依赖分析**是选择性回滚的前提：

```python
def can_selective_rollback(self, target_step):
    """判断是否可以只回滚目标步骤"""
    dependents = self.find_dependents(target_step)
    
    if not dependents:
        return SelectiveRollback(steps=[target_step])
    
    # 有依赖步骤，必须级联
    cascade = [target_step] + dependents
    return CascadeRollback(
        steps=cascade,
        message=f"步骤 {target_step} 有 {len(dependents)} 个依赖步骤，"
                f"需要一起回滚：{[s.name for s in cascade]}"
    )
```

### 5.5 回滚 UX：让用户看见影响范围

```
┌──────────────────────────────────────────┐
│ ↩️ 回滚确认                              │
│                                          │
│ 你要回滚 Step 3: "修改支付配置"            │
│                                          │
│ ⚠️ 影响范围：                             │
│   • Step 3 的修改将被撤销                 │
│   • Step 4 依赖 Step 3，也将被撤销        │
│   • Step 5 不受影响（独立步骤）            │
│                                          │
│ 回滚后状态：                              │
│   回到 Step 2 完成时的状态                │
│   可以从 Step 3 重新执行                  │
│                                          │
│ [确认回滚] [只回滚 Step 3（实验性）] [取消] │
└──────────────────────────────────────────┘
```

## §6 状态持久化架构

### 6.1 为什么需要持久化

没有持久化 = 没有 pause/resume/rollback。这三种操作都依赖 Agent 能从任意断点恢复状态。

### 6.2 推荐架构：Graph-based Checkpoint

LangGraph 风格的图状态管理在 2025-2026 成为主流（Indium 2026）：

```python
class GraphCheckpointStore:
    """图结构 + checkpoint 的状态管理"""
    
    def __init__(self, backend="postgres"):
        self.backend = backend
    
    def save_checkpoint(self, node_id, state, metadata):
        """每个图节点完成时保存"""
        checkpoint = {
            "node_id": node_id,
            "state": serialize(state),
            "timestamp": now(),
            "parent_checkpoint": self.current_checkpoint_id,
            "metadata": metadata
        }
        self.backend.write(checkpoint)
        self.current_checkpoint_id = checkpoint["id"]
    
    def restore(self, checkpoint_id):
        """恢复到任意 checkpoint"""
        cp = self.backend.read(checkpoint_id)
        return deserialize(cp["state"])
    
    def fork(self, checkpoint_id, new_plan):
        """从某个 checkpoint 分叉执行新计划"""
        state = self.restore(checkpoint_id)
        new_branch = self.create_branch(checkpoint_id)
        return ExecutionContext(state=state, plan=new_plan, branch=new_branch)
    
    def list_checkpoints(self, branch=None):
        """列出所有 checkpoint（用于 rollback 选择）"""
        return self.backend.query(branch=branch, order="desc")
```

### 6.3 Checkpoint 频率 vs 性能

| 策略 | Checkpoint 频率 | 开销 | 适用场景 |
|---|---|---|---|
| 每步 | 每个 step 完成 | 中（IO 多） | 默认推荐 |
| 关键节点 | 只在有副作用的步骤前后 | 低 | 性能敏感 |
| 定时 | 每 N 秒 | 可控 | 长运行步骤 |
| 事件驱动 | 外部状态变更时 | 最低 | 最小化 IO |

**推荐**：默认每步 checkpoint + 不可逆操作前额外 checkpoint。

### 6.4 有状态 vs 无状态恢复

Tianpan.co 的 Escalation Protocol 区分两种恢复模式：

| 模式 | 机制 | 恢复时间 | 适用 |
|---|---|---|---|
| **Stateful snapshot** | 完整内存快照 | 微秒级 | 短暂中断（分钟） |
| **Stateless checkpoint** | 序列化到存储 → 反序列化恢复 | 秒级 | 长时间中断（小时/天） |
| **Hybrid（推荐）** | Redis 短期 + DB 长期 | 依赖场景 | 生产系统 |

## §7 "干预即续"（Interrupt-and-Resume）的工程实现

### 7.1 核心原则：干预不应重启

**最重要的 UX 原则**：用户的任何干预操作（暂停、接管、修改计划、回滚一步）都不应该导致"从头开始"。

实现这个原则需要：
1. **不可变执行历史**：每步结果是 append-only 的，不被后续步骤覆盖
2. **状态派生而非状态积累**：当前状态从执行历史 replay 得到，不是 mutable 的
3. **计划是可编辑的**：用户可以在执行中修改剩余计划

### 7.2 Append-only 执行日志

Claude Code 的设计（arXiv:2604.14228）："mostly append-only JSONL session transcripts, favoring auditability over query power"。

```python
class ExecutionLog:
    """不可变的执行历史"""
    
    def append(self, event: Event):
        """只追加，不修改"""
        self.journal.write(event.serialize() + "\n")
    
    def derive_state(self, up_to_step=None):
        """从日志派生当前状态"""
        state = InitialState()
        for event in self.replay(up_to=up_to_step):
            state = state.apply(event)
        return state
    
    def fork_from(self, step_index):
        """从某步分叉：保留历史，开始新分支"""
        return ExecutionLog(
            parent=self,
            fork_point=step_index,
            events=[]  # 新分支从空开始
        )
```

### 7.3 完整的 Interrupt-and-Resume 流程

```python
class InterruptibleAgent:
    def __init__(self):
        self.log = ExecutionLog()
        self.checkpoint_store = GraphCheckpointStore()
    
    async def run(self, plan):
        for i, step in enumerate(plan.steps):
            # 检查控制信号
            signal = self.check_control_signal()
            
            if signal == Signal.PAUSE:
                self.checkpoint_store.save(i, self.log.derive_state())
                return AgentPaused(resumable_from=i)
            
            if signal == Signal.INTERRUPT:
                return AgentInterrupted(
                    completed=i,
                    state=self.log.derive_state()
                )
            
            if signal == Signal.TAKEOVER:
                # 等待用户完成这步
                user_result = await self.wait_for_user(step)
                self.log.append(TakeoverEvent(step=i, result=user_result))
                continue
            
            if signal == Signal.ROLLBACK:
                target = signal.target_step
                self.log.append(RollbackEvent(to_step=target))
                # 恢复到目标步骤的状态
                plan.rewind_to(target)
                # 从目标步骤重新开始
                return await self.run_from(plan, target)
            
            # 正常执行
            result = await self.execute_step(step)
            self.log.append(StepCompleteEvent(step=i, result=result))
            self.checkpoint_store.save(i + 1, self.log.derive_state())
    
    async def resume(self, checkpoint_id):
        """从 checkpoint 恢复执行"""
        state = self.checkpoint_store.restore(checkpoint_id)
        plan = state.remaining_plan
        return await self.run(plan)
```

```
└──────────────────────────────────────────────────┘
```

### 8.2 中断后的恢复选项

| 状态 | 用户可选操作 | 最安全选项 |
|---|---|---|
| 暂停于 Step N | 继续 / 修改计划 / 回滚 / 终止 | 继续 |
| 中断于 Step N（步内） | 重做 Step N / 跳过 / 接管 / 终止 | 重做 |
| 回滚到 Step M | 从 M 重新执行 / 修改 M 后执行 / 终止 | 修改后重新执行 |
| 接管完成 | Agent 继续 / 人工继续全部 | Agent 继续 |

### 8.3 不可逆操作的预警

在执行不可逆操作**之前**（不是之后）必须给用户最后一次干预机会：

```
┌────────────────────────────────────────┐
│ ⚠️ 下一步包含不可逆操作                 │
│                                        │
│ Step 5: 向客户发送确认邮件              │
│                                        │
│ 一旦执行无法撤销。                      │
│ 收件人：alice@example.com              │
│ 内容预览：[展开查看]                    │
│                                        │
│ [📧 发送] [✏️ 编辑] [⏭️ 跳过] [⏸️ 暂停] │
└────────────────────────────────────────┘
```

**这不是 9.1 批判的"fait accompli 确认屏"**——因为它提供了完整预览 + 多条路径（编辑/跳过/暂停），而不只是"确定/取消"。

## §9 反模式表

| # | 反模式 | 后果 | 解药 |
|---|---|---|---|
| 1 | 暂停=从头开始 | 用户浪费所有前序工作 | Checkpoint + resume |
| 2 | 中断后状态未知 | 用户不知道发生了什么 | 中断后立刻展示状态诊断 |
| 3 | 不可逆操作无预警 | 后悔无药 | 不可逆步骤前强制预审 |
| 4 | 接管后 Agent 不识新状态 | 只能重启 | 结构化 handoff + LLM fallback |
| 5 | 回滚只能全量 | 5 步里只错 1 步也要全撤 | 依赖分析+选择性回滚 |
| 6 | 无 checkpoint 直接 run | 崩溃=全部丢失 | 每步 checkpoint |
| 7 | 步骤间状态用 mutable 变量 | 回滚时状态不一致 | Append-only log + 状态派生 |
| 8 | Emergency Stop 后无恢复路径 | Agent 死在原地 | 受损状态诊断 + 恢复向导 |
| 9 | 接管时无上下文传递 | 人工花 15 分钟重建 | 三层 handoff payload |
| 10 | Checkpoint 太少 | 回滚只能到很远之前 | 每步+不可逆前 double checkpoint |

## §10 S0 → S5 演进路线

| 阶段 | Pause | Interrupt | Takeover | Rollback | 实现 |
|---|---|---|---|---|---|
| **S0** | 无 | 关闭窗口 | 不支持 | 手动改 | 无状态 |
| **S1** | Stop generating | 丢弃当前 | 不支持 | 不支持 | Stateless |
| **S2** | 步间暂停 | 步间中断 | 不支持 | 全量回滚 | 基本 checkpoint |
| **S3** | 步间+步内 | Pre/post-commit | 简单接管 | 级联回滚 | Graph checkpoint |
| **S4** | 实时暂停 | 精确中断 | 结构化 handoff | 选择性回滚 | Append-only + fork |
| **S5** | 多 Agent 暂停 | 跨 Agent 中断 | 跨 Agent 接管 | 分布式回滚 | 分布式事务 |

## §11 一页总结：抄走这页就够

| 问题 | 答案 |
|---|---|
| **4 种控制操作？** | Pause / Interrupt / Takeover / Rollback |
| **暂停在哪里停？** | 步间（安全）> 步内（需子步骤拆分）> 思考中（等 response 完成后暂停） |
| **中断怎么处理副作用？** | Pre-commit 安全中断 / Post-commit 只停后续 |
| **接管后怎么续接？** | 三层 handoff payload + 结构化结果 + post-condition 验证 |
| **回滚粒度？** | 依赖分析：无依赖→选择性回滚，有依赖→级联回滚 |
| **状态怎么存？** | Append-only log + graph checkpoint + 每步持久化 |
| **核心原则？** | 干预不应重启——任何操作后都能从断点继续 |
| **不可逆操作？** | 执行前预审 + 预览 + 多路径选择（不只是确定/取消） |
| **Emergency Stop？** | 始终可见 + 立刻响应 + 中断后状态诊断 + 恢复向导 |

## §12 与其他模块的连接

- **接 9.1 UX 原则**：本文是 9.1 可控性 5 维（Pause/Override/Redirect/Undo/Kill）的完整工程实现
- **接 9.2 自主度梯度**：不同自主度级别的中断机制不同——L1 忽略即可、L4 需要 emergency stop
- **接 8.1 部署架构**：Durable Execution（Restate/Temporal）天然支持 checkpoint + resume
- **接 8.2 可靠性**：checkpoint 是 idempotency 的基础；circuit breaker 是自动中断
- **接 9.5 异步后台**：后台 Agent 的 pause/resume 更复杂（用户不在线）
- **接 5.3 Multi-Agent**：多 Agent 中断需要协调——一个 Agent 暂停时其他怎么办
