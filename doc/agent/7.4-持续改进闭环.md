## 0. 真问题：为什么"线下涨点 → 上线掉点"

线下回归套件 500 case，新 prompt 比旧 prompt 高 4 个百分点。PM 拍板上线。一周后业务指标下跌 2pp，customer satisfaction 跌 6pp。复盘发现三件事：

1. **分布不一致**：回归套件 80% 是 happy path，而生产长尾里 30% 是新出现的"退订+换货"组合。
2. **Novelty effect 反向**：用户第一周看到风格变化，差评率短暂上涨。
3. **模型悄悄换了**：vendor 在第 4 天小版本升级，agent 输出风格漂移，没人察觉。

第一性原理：**线下评估是"昨天的考试"，线上世界每天换题**。持续改进闭环必须做到三件事：

```
A/B Test       ────→ "新版本是不是统计显著地更好？"（决策门）
Online Eval    ────→ "现在线上质量是多少？"（实时仪表）
Drift Detect   ────→ "输入/输出分布在悄悄变吗？"（预警器）
                              │
                              └──→ 触发 retrain/repromp/rollback（与 6.7 联动）
```

三者缺一不可：A/B 决定"上不上"，Online Eval 决定"现在好不好"，Drift Detect 决定"很快会不会变坏"。

## 1. A/B Test：Agent 时代的三个新麻烦

### 1.1 比微服务时代难在哪

| 维度 | 微服务 A/B | Agent A/B |
|---|---|---|
| 用户体验差异 | 像素/文案级 | 整段对话/工具链不同 |
| 单次成本 | µs 级 | **秒级 + token 计费** |
| 主指标方差 | 较小 | 大（长对话引入大噪声） |
| 用户感知风险 | 很低 | "Agent 说错话"立刻影响信任 |
| 决策周期 | 几小时-几天 | 通常需要 1-2 周（高方差 + 学习效应） |

### 1.2 实验设计四要素

**1. 单元（unit）**：默认 `user_id` 哈希分桶；多轮会话场景必须 `session_id` 一致性——同会话不能跨 arm，否则用户体验割裂。

**2. 主指标（OEC, Overall Evaluation Criterion）**：业务真北 + 防护指标。

| OEC 类型 | 示例 |
|---|---|
| 主指标 | task completion rate / 转化率 / NPS |
| 护栏（guardrail） | p95 延迟 / 单 task 成本 / 安全事件率 / kill switch 触发率 |
| 健康指标 | DAU / retention / 客诉率 |

**主指标至少有一个，护栏指标至少 3-5 个**——只看主指标的 A/B 是"用 KPI 换风险"。

**3. 样本量与最小可检差异 (MDE)**：

```python
# Agent 场景 MDE 计算（双侧 z-test，简化版）
def sample_size(baseline: float, mde: float, alpha=0.05, power=0.8):
    from scipy.stats import norm
    p1, p2 = baseline, baseline + mde
    p_bar = (p1 + p2) / 2
    z_a = norm.ppf(1 - alpha/2)
    z_b = norm.ppf(power)
    n = ((z_a * (2 * p_bar * (1 - p_bar))**0.5 + z_b * (p1*(1-p1)+p2*(1-p2))**0.5) / (p2-p1))**2
    return int(n) + 1

# 例：baseline 70%，想检测 +2pp 提升
sample_size(0.70, 0.02)  # ≈ 8600/组
```

实务经验：Agent 主指标方差大，**实际样本量取理论值 1.5-2×**。

**4. 实验周期**：至少跨完整一个业务周期（B2C 7 天 / B2B 14 天），覆盖**周内/周末**、**早晚高峰**、**至少 1 次模型自动更新窗口**。

### 1.3 严重坑：novelty / primacy effect

新版本上线初期的指标偏离不一定是真效果：

- **Novelty effect**：用户对新功能短暂兴奋，指标虚高，1-2 周回落。
- **Primacy effect**：用户对老版本有肌肉记忆，新版本暂时使用障碍，指标虚低。

**应对**：实验周期不少于 7 天；做"早期 vs 晚期"分时段分析，如果**只有早期显著**就不能下结论。

### 1.4 Interleaving：用更少流量做更敏锐的对比

经典 A/B 把用户切两组各看一个版本。**Interleaving** 给每个用户**同时**展现 A 和 B 的结果（聊天 A 一句、B 一句，或对工具调用结果做 A/B 混排），看用户选/接受哪个。

**优势**：

| 维度 | A/B | Interleaving |
|---|---|---|
| 样本需求 | 高 | **5-100× 更少**（[Netflix 论文](https://research.netflix.com/publication/Interleaving)） |
| 实验周期 | 7-14d | 数天 |
| 用户体验侵入 | 0 | 略增（看到不同风格） |

**Agent 场景适合 interleaving 的情形**：
- 检索结果排序 A/B（直接混排 top-k）
- 工具推荐 A/B（agent 给出 2 个选项让用户选）
- 文案/总结 A/B（同界面对比）

**不适合**：
- 多步对话主流程（割裂 UX）
- 副作用工具（不能"两个版本都退一次款"）

### 1.5 顺序检验：早停而不犯错

经典固定样本量 t-test **不允许 peeking**——每多看一次就抬高假阳性率。Agent 实验贵且慢，急需早停能力。解法：**Sequential Testing**（mSPRT、Always-Valid p-values、贝叶斯 A/B）。

```python
# 简版：mSPRT 早停
def msprt_log_lr(x_a, x_b, theta=0.02, sigma2=0.05):
    """x_a / x_b: 累积观察；theta: 设计差异；sigma2: 方差估计"""
    n = min(len(x_a), len(x_b))
    diff = sum(x_a[:n]) - sum(x_b[:n])
    return (theta * diff - n * theta**2 / 2) / sigma2

# 阈值 log(1/alpha) 触发即可停止
```

实务建议：用现成框架（Optimizely Stats Accelerator、自研 Sequential SDK），别手撸概率论。

### 1.6 因果纯度三大杀手

| 杀手 | 现象 | 防御 |
|---|---|---|
| **SUTVA 破坏** | A/B 用户互相影响（多 agent 共享 memory） | 按 session 隔离 memory namespace |
| **样本污染** | 用户在 A 组但因 cache 命中拿到 B 结果 | cache key 加 arm 标识 |
| **生存者偏差** | 失败用户提前离开，留下的"看似"好转 | 用 intent-to-treat（ITT），不丢弃中途退出 |

## 2. Online Evaluation：实时为 1-5% 流量打分

### 2.1 架构：从 trace 到 score

```
Agent (prod) ─OTel─→ Collector ─sample─→ Eval Queue
                                              │
                                       ┌──────┴──────┐
                                       ▼             ▼
                                  Judge Worker  Detector Worker
                                       │             │
                                       ▼             ▼
                                  Score Span    Drift Score
                                       │             │
                                       └──────┬──────┘
                                              ▼
                                       Dashboard + Alert
```

### 2.2 三档采样

| 档 | 采样率 | Judge 内容 | 成本 |
|---|---|---|---|
| **轻档** | 100% | 确定性指标（tool correctness / latency / token） | 0 |
| **中档** | 5% | 便宜 LLM Judge（goal completion / faithfulness，用小模型） | 低 |
| **重档** | 0.5% + 异常升采 | 大模型 / Agent-as-Judge（trajectory 全检） | 中高 |

**异常升采规则**：错误 trace、低 judge 分、kill switch 触发、用户负反馈 trace → 全部进重档。

### 2.3 用户反馈通道

被动等 judge 不够。**显式 + 隐式反馈双通道**：

```python
@dataclass
class UserSignal:
    trace_id: str
    explicit: dict | None    # 👍/👎 / "this was wrong" / 1-5 星
    implicit: dict           # 是否复制、是否重新提问、对话时长、是否切人工
```

显式反馈样本少但准；隐式反馈样本多但噪。**双通道做交叉校准**——隐式与显式高一致时可大胆扩大隐式权重。

### 2.4 Judge 自身的校准

Online Eval 用便宜小模型 judge 是工程需求，但便宜的 judge 偏差更大。**每月对照** 1 次：

```python
def monthly_judge_calibration():
    sample = sample_traces(n=200, stratified_by="task_type")
    cheap_scores = [cheap_judge(t) for t in sample]
    gold_scores = [agent_as_judge(t) for t in sample]    # 月度 1 次全量重档
    kappa = cohen_kappa(cheap_scores, gold_scores)
    if kappa < 0.6:
        alert("Online judge calibration drifted")
        # 触发 judge prompt 重训 / 升级模型
```

## 3. Drift Detection：质量塌方之前的"地震预报"

按 [AWS 2026 指南](https://docs.aws.amazon.com/prescriptive-guidance/latest/gen-ai-lifecycle-operational-excellence/prod-monitoring-drift.html) 与 [Future AGI 2026](https://futureagi.com/blog/model-vs-data-drift-how-to-identify-and-handle-it/)，Agent 上至少要监控四类漂移：

| 类型 | 定义 | Agent 例子 |
|---|---|---|
| **Data drift（covariate shift）** | P(X) 变了，P(Y\|X) 不变 | 用户问题语种从中文偏向英文 |
| **Concept drift** | P(Y\|X) 变了（最难测） | "好投资"的标准变了；监管口径变了 |
| **Embedding drift** | 输入/检索结果向量分布漂移 | 知识库新增大量新主题 |
| **Output / behavior drift** | Agent 输出风格、长度、工具选择分布变了 | vendor 模型小版本升级后 |

### 3.1 输入漂移：双层检测

**Layer 1：统计层（便宜，秒级）**

[AWS](https://docs.aws.amazon.com/prescriptive-guidance/latest/gen-ai-lifecycle-operational-excellence/prod-monitoring-drift.html) 与 [Evidently AI](https://www.evidentlyai.com/blog/embedding-drift-detection) 推荐：

```python
methods = {
    "psi": "PSI > 0.2 即报警（适合特征/标量）",  # 信贷领域起源
    "ks": "Kolmogorov-Smirnov（标量分布）",
    "wasserstein": "高维 / 嵌入空间首选",       # AWS 主推
    "mmd": "Maximum Mean Discrepancy（核方法）",
    "classifier": "训练一个分类器分辨 ref vs prod，AUC=drift score",  # Evidently 推荐默认
}
```

**Agent 场景关键经验**：
- **PSI/KS 对嵌入维度太高**，不直接适用——要么先降维（PCA/UMAP），要么用 Wasserstein/MMD/classifier。
- **基线必须代表"稳定期"**：取上线后第 1 个月的 representative sample 做 reference。
- **每特征独立阈值**：通用 PSI > 0.2 是起点，关键特征要更严格。

**Layer 2：语义层（贵但可解释）**

统计告警只说"变了"，不说"为什么"。AWS 建议用 LLM-as-Judge 做语义分类：

```python
def explain_drift(reference_samples, drifted_samples):
    prompt = f"""
    Reference prompts: {sample(reference_samples, 20)}
    Drifted prompts:   {sample(drifted_samples, 20)}
    
    Classify the primary reason:
    (A) New topic emergence  (B) User intent shift
    (C) Complexity increase  (D) Language style change  (E) Spam/abuse spike
    
    Return: {{"category": ..., "evidence": ...}}
    """
    return llm_judge(prompt)
```

这一步把"alert"变成"action"——告警说"15% 流量是新主题：A 类移动套餐促销活动"比"PSI=0.27"有用 1000 倍。

### 3.2 输出漂移：监控 agent 自己的行为变化

模型 vendor 静默升级是 Agent 时代的隐形地震。监控**输出端**的统计签名：

```python
output_signals = [
    "avg_output_token_length",      # 突变 = vendor 改了系统行为
    "tool_call_distribution",       # 调用工具的频次分布
    "json_parse_fail_rate",         # 格式遵循能力
    "refusal_rate",                 # vendor 加严了 safety policy
    "language_detected",            # 输出语言分布
    "trajectory_avg_steps",         # 平均步数（绕路 vs 直接）
]
```

每天对比 7 日滑动均值——任何信号偏离 ±2σ 触发 alert。

### 3.3 Concept Drift：最难，但有信号

> Concept drift "user prompts might look statistically similar (data drift low), but the desired answer has changed (concept drift high)."（[AWS](https://docs.aws.amazon.com/prescriptive-guidance/latest/gen-ai-lifecycle-operational-excellence/prod-monitoring-drift.html)）

**只能靠下游业务信号反推**：

| 信号 | 表征 concept drift |
|---|---|
| 同样回答的用户负反馈率上升 | 答案"以前对、现在不对" |
| 客服 escalation 率上升 | 用户对"标准答案"不再满意 |
| 知识库引用文档命中率不变但点踩率上升 | 内容过时 |
| 业务规则更新通知 | 法规/政策变更（人工 trigger） |

工程做法：把这些信号合并为一个"concept staleness score"，与 retrain trigger 关联。

### 3.4 Retrain / Reprompt 触发器

完整的触发逻辑（[Future AGI](https://futureagi.com/blog/model-vs-data-drift-how-to-identify-and-handle-it/)）：

```python
def should_take_action(metrics) -> str | None:
    # 极端：任意硬错（安全 / kill switch）→ 立即回滚
    if metrics.kill_switch_triggered or metrics.safety_violation:
        return "ROLLBACK_NOW"

    # PSI > 0.2 + 离线评估在新 slice 上跌 3-5pp → 触发动作
    if metrics.psi > 0.2 and metrics.fresh_eval_drop >= 0.03:
        if metrics.is_lm_frozen:
            return "REPROMPT_OR_RETUNE_RETRIEVER"
        else:
            return "RETRAIN_FINETUNE"

    # 单纯 PSI 漂移但 eval 不掉 → 观察
    if metrics.psi > 0.2 and metrics.fresh_eval_drop < 0.03:
        return "OBSERVE_AND_ANNOTATE"

    return None
```

**重要纪律**：Agent 的"retrain"通常不是 base model 训练（base 是冻结的 SaaS），而是 **prompt 调整 + retriever 调整 + few-shot 数据更新**。这是 LLM 时代的"持续学习"形态。

## 4. 与 Kill Switch / 红队联动：完整改进飞轮

把四个子系统串成 **闭环**：

```
         ┌─────────────────────────────────────────────────────┐
         │                                                     │
         ▼                                                     │
  ┌──────────────┐    drift / low eval     ┌──────────────┐    │
  │ Production   │ ─────────alert───────→  │ On-call      │    │
  │ Agent        │                          │ Investigate  │    │
  └──────────────┘                          └──────────────┘    │
         │                                         │            │
         │ trace                                   │ root cause │
         ▼                                         ▼            │
  ┌──────────────┐    │           ┌─────────────────────┐       │
  │ Trace store  │    │           │ Reprompt / Retune / │       │
  │ + Eval store │    │           │ Retriever update    │       │
  └──────────────┘    │           └─────────────────────┘       │
         │            │                   │                     │
         │  failure   │                   │ candidate           │
         │  cases     │                   ▼                     │
         ▼            │           ┌──────────────┐              │
  ┌──────────────┐    │           │ A/B + Shadow │              │
  │ Regression   │ ◀──┘           │ Experiment   │ ─wins─→  Roll-out
  │ Suite        │                └──────────────┘              │
  │ + Red Team   │                                              │
  └──────────────┘                                              │
         │                                                      │
         │ block bad release                                    │
         └──────────────────────────────────────────────────────┘
```

**每一环都通过 6.7 Kill Switch 接到自动止血**：异常 → drift alert → 流量分级降级 → 严重 → 自动回滚 → 立 incident。

## 5. 实践模板：每周发版仪式

把"持续改进"具象成可重复的**周仪式**：

| 日 | 动作 | 谁 |
|---|---|---|
| **周一** | 拉取上周 trace / eval / drift 报告；列改进点 top 3 | AI 工程师 |
| **周二** | 把改进点变成 prompt/retriever/tool diff；本地跑回归套件 | AI 工程师 |
| **周三** | Shadow 24h；mutation testing pass；A/B 实验设计 review | AI 工程师 + PM |
| **周四** | A/B 上线 1%；OEC + 护栏指标观察 | 全员 |
| **周五** | 看显著性 + novelty；决定扩 5% / 回滚 / 继续观察 | PM |
| **下周一** | 扩 25% → 100%（或回滚归档）；总结写入 daily memory | 全员 |

**纪律**：
- 没有 A/B 不上线。
- A/B 没护栏指标全绿不扩量。
- 任何一次回滚必须有 5-Whys 复盘。
- 每月 1 次 judge 校准、drift baseline 重置、回归套件月度扩充。

## 6. 反模式

| 反模式 | 后果 | 替代 |
|---|---|---|
| 线下涨点直接全量 | 长尾爆炸 / novelty 误判 | 必经 A/B + shadow 验证 |
| 只看主指标，不看护栏 | 转化率涨但延迟翻倍/成本翻倍 | OEC + 3-5 护栏 |
| 不设最小样本量就 peek | 假阳性率拉满 | sequential testing 或固定样本 |
| 跨 session 切桶 | 用户体验割裂、SUTVA 破坏 | session_id 一致性分桶 |
| 用静态阈值监控漂移 | 季节性误报海 | 滑动基线 + 自适应阈值 |
| PSI/KS 监控嵌入 | 高维下统计力差 | Wasserstein / MMD / classifier |
| 只告"drift 发生了" | oncall 无从下手 | 双层（统计 + LLM 语义分类） |
| Vendor 升级不监控输出漂移 | 静默劣化 | 6 维 output signals 日度比 |
| 隐式反馈直接当 ground truth | 噪音 | 与显式做交叉校准 |
| 把"持续改进"做成月度 | 漂移已变成事故 | 周节奏发版仪式 |

## 7. 一页总结

> **Agent 的持续改进 = 三件事的合奏**：A/B 给"上不上"打分（决策门）；Online Eval 给"现在好不好"打分（仪表）；Drift Detect 给"快变坏了吗"打分（雷达）。

- **A/B**：unit=user/session 一致性；OEC = 主指标 + 3-5 护栏；样本量 ×1.5-2；至少 7d；用 sequential testing 早停；防 novelty/primacy/SUTVA。
- **Interleaving**：检索/排序/工具推荐场景比 A/B 节省 5-100× 样本；多步副作用主流程不适用。
- **Online Eval**：100% 确定性 + 5% 便宜 judge + 0.5% 重档；用户显式 + 隐式反馈双通道；月度 judge 校准。
- **Drift**：data（Wasserstein/MMD/classifier）+ concept（下游业务信号）+ embedding（基线 + 双层告警）+ output（vendor 静默升级监控）；统计告警 + LLM 语义分类。
- **触发动作**：base model 冻结时 retrain ≈ reprompt + retune retriever + few-shot 更新。
- **飞轮**：trace → eval → drift → A/B → roll-out → 回滚 incident → 红队补 case → 回归套件，每一环跟 [6.7 Kill Switch](./6.7-KillSwitch与可控性工程.md) 联动。
- **节奏**：每周 1 次发版仪式，没护栏不扩量，月度做 judge/baseline 校准。

不可观测的 Agent 没法改进。不能 A/B 的 Agent 改进不可信。不能 drift detect 的 Agent 等同盲飞。三件齐备，发版周期才能从季度收缩到周。
