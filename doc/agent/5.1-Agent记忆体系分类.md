## 0. 开场问题：为什么 Conversation History ≠ Memory？

设想这样一个真实场景。你给一个 Customer Support Agent 接入了"会话历史"——把过去 30 天用户的所有对话拼到 system prompt 里，告诉自己"现在它有记忆了"。一周后系统出现以下症状：

- **症状 A**：用户 9 月说过"我不吃辣"，10 月点餐时 Agent 仍然推荐"麻辣香锅"。
- **症状 B**：用户问"上次那个退款进展如何"，Agent 把同一个用户 3 个月前的另一笔退款翻出来当成"上次"。
- **症状 C**：用户每次对话都要等 12 秒首字才出来，token 成本是 chatbot 的 8 倍。
- **症状 D**：用户在 A 设备说"以后用英文回我"，B 设备打开还是中文。

这四个症状不是 Bug，是**架构错误**——它们分别对应四类**根本不同**的记忆需求被错配到了同一个机制（Conversation History）上：

| 症状 | 实际需要的记忆类型 | 错配在哪 |
|---|---|---|
| A：忘记偏好 | **Semantic Memory**（事实/偏好） | 偏好被淹没在历史里，没有提取成结构化条目 |
| B：上下文混淆 | **Episodic Memory**（带时序的事件） | 历史没有按事件分段，没有时间锚点 |
| C：高延迟高成本 | **Working Memory**（当前上下文） | 把所有历史当作当前上下文，违反规模假设 |
| D：跨会话/跨设备共享 | **Long-term Memory**（持久化层） | 没有 namespace/作用域设计 |

**第一性原理**：记忆不是"把上下文存起来"，记忆是**根据访问模式（access pattern）、生命周期（lifecycle）、可写性（write policy）的不同，把信息分层存储到不同基质（substrate）上**。把所有信息塞进 conversation history，本质上是把 working memory、semantic memory、episodic memory 全部混在 LLM 的注意力机制里——LLM 既要做推理，又要做检索，又要做去噪，又要做相关性排序。这违反了 Anthropic 在 [《Building effective agents》](https://www.anthropic.com/research/building-effective-agents) 中反复强调的"职责分离"原则。

本篇要构建的，是一套**让 Agent 真正拥有记忆**的认知架构。

## 1. 第一性原理：记忆的四个区分维度

在进入分类法之前，先把"记忆"这个词拆开。任何一个记忆系统都要同时回答四个正交问题：

### 1.1 时间维度：Working vs Long-term

- **Working Memory**：单次决策周期（decision cycle）内活跃的信息——刚刚的工具返回、当前的 plan、待回填的变量。CoALA 论文把它定义为"the central hub connecting different components of a language agent"。
- **Long-term Memory**：跨周期、跨会话持久存在的信息。可以是用户偏好（语义）、过去的某次成功轨迹（情景）、Agent 自己学到的工作流（程序）。

**关键差异**：Working Memory 的访问代价是 O(1)（在 prompt 里直接读），但容量受 context window 严格约束；Long-term Memory 的访问代价是 O(检索调用)（要走 vector / graph / k-v lookup），但容量理论无限。

### 1.2 内容维度：Fact vs Experience vs Procedure

这是来自认知心理学（Tulving 1972）经典分类的工程对应：

| 心理学概念 | Agent 对应 | 示例 |
|---|---|---|
| Semantic Memory（语义记忆） | **事实**（facts） | "用户偏好 Python 后端"、"公司退款政策是 7 天" |
| Episodic Memory（情景记忆） | **经历**（experiences） | "2025-10-12 用户咨询了 A 产品，最终下单"、"上次重启服务后问题消失" |
| Procedural Memory（程序记忆） | **技能**（procedures） | "处理退款的标准流程"、"代码 review 必须先跑 lint 再跑 test" |

LangChain 官方文档（2025）将这三类作为 long-term memory 的标准分类，并明确说："Semantic memory is different from 'semantic search'——前者是心理学（存储事实），后者是检索方法"（[LangChain Memory Overview](https://docs.langchain.com/oss/python/concepts/memory)）。

### 1.3 写入维度：Hot Path vs Background

记忆何时被写入：

- **Hot Path（热路径写入）**：用户对话过程中实时写入。优点是立刻可读、对用户透明（"我记住了"）；缺点是增加延迟、Agent 必须 multitask。代表实现：ChatGPT 的 `save_memories` 工具调用。
- **Background（背景写入）**：异步批处理写入。优点是不阻塞响应、可以批量优化；缺点是有"刷新延迟"、并行会话间可能脏读。代表实现：LangGraph 的 `memory-service` 模板。

LangChain 文档总结的取舍："Need instant recall + transparency → hot path. Optimize latency/quality → background."

### 1.4 作用域维度：四级 Namespace

Mem0 的 [State of AI Agent Memory 2026](https://mem0.ai/blog/state-of-ai-agent-memory-2026) 报告把作用域形式化为四个层级：

```
user_id        # 单用户跨 session
agent_id       # 单 agent 实例（多用户共享）
run_id/session_id  # 单次执行/对话
app_id/org_id  # 应用级/组织级共享
```

这是多租户 Agent 的核心。同一条"喜欢简洁回答"的记忆，写在 `user_id` 下是用户偏好，写在 `agent_id` 下是 Agent 的统一风格——同样的内容，作用域不同，行为完全不同。

## 2. 经典分类法：CoALA 的四层记忆模型

要建立工程化的记忆系统，绕不开 Princeton 在 2024 年发表的 **CoALA（Cognitive Architectures for Language Agents）**（[arXiv:2309.02427](https://arxiv.org/abs/2309.02427)）。它把分散在 ReAct、Voyager、Generative Agents 等系统里的记忆设计**统一成了一套形式化框架**。理解 CoALA = 理解工业界 90% 的记忆架构语言。

### 2.1 四层记忆的完整定义

CoALA Section 4.1 给出的标准定义（[arXiv HTML v3](https://arxiv.org/html/2309.02427v3)）：

| 类型 | CoALA 原文定义 | 工程实现形态 | 写者 |
|---|---|---|---|
| **Working Memory** | "Maintains active and readily available information as symbolic variables for the current decision cycle" | Prompt + scratchpad + agent state | Agent 自身（每个 cycle 自动） |
| **Episodic Memory** | "Stores experience from earlier decision cycles" | Trajectory log / few-shot example store | Learning action 写入 |
| **Semantic Memory** | "Stores an agent's knowledge about the world and itself" | KB / Vector DB / KG | 初始化导入 + Learning action |
| **Procedural Memory** | "Implicit knowledge in LLM weights + explicit knowledge in agent code" | LLM 权重 + Prompt / 工具实现 / Decision procedure | 训练（隐式）+ Reflection 重写（显式） |

CoALA 明确警告："Writing to procedural memory is significantly riskier than writing to episodic or semantic memory."——这是工业界后来"prompt 自动优化"系统（如 LangMem prompt optimizer）必须加 reflection / scoring gate 的理论根源。

### 2.2 记忆的"流动方向"

CoALA 把动作空间分成两大类，**记忆的流动**完全由这些动作驱动（Section 4.2-4.5）：

```
                    ┌─────────────────────────────────────┐
                    │         Working Memory              │
                    │   (active symbolic variables)       │
                    └──┬──────┬──────────▲────────▲───────┘
       Grounding read  │      │ Retrieval│        │ Reasoning
        (external)     │      │  reads   │        │ reads/writes
                       ▼      ▼          │        │
                  ┌──────────┐  ┌────────┴────────┴───────┐
                  │ External │  │     Long-term Memory     │
                  │   Env    │  │  ┌─────────┐ ┌────────┐  │
                  └──────────┘  │  │Episodic │ │Semantic│  │
                                │  └─────────┘ └────────┘  │
                                │  ┌──────────────────┐    │
                                │  │   Procedural     │    │
                                │  └──────────────────┘    │
                                └────────▲────────▲────────┘
                                         │        │
                                Learning │        │ Learning
                                 (writes)│        │  (writes)
```

四类动作的语义：

1. **Grounding（外部动作）**：调工具、查 API、与人/Agent 对话——读 external environment，结果写入 working memory。
2. **Retrieval（检索）**：从 long-term memory 读到 working memory。可以是规则匹配、稀疏检索（BM25）、稠密检索（embedding）。
3. **Reasoning（推理）**：LLM 调用本身——既读又写 working memory（典型例子：CoT 的中间步骤被回填到 working memory）。
4. **Learning（学习）**：把 working memory 的内容写到 long-term memory。这是**记忆形成的唯一通道**。

**架构启示**：在你的 Agent 里，每一次 LLM call 都对应一个 Reasoning action；每一次工具调用对应一个 Grounding action；每一次"想要记住"的瞬间，必须有一个显式的 Learning action——否则那条信息只存在于本轮 working memory 中，回合结束就消失了。

### 2.3 经典 Agent 在 CoALA 框架下的位置

CoALA Section 5 把几个标杆系统映射到记忆维度（表 2），帮助我们看清"不同 Agent = 不同记忆配置"：

| 系统 | Working | Episodic | Semantic | Procedural |
|---|---|---|---|---|
| **ReAct** | ✅ (prompt scratchpad) | ❌ | ❌ | ❌ |
| **SayCan** | ✅ | ❌ | ❌ | ✅ (551 个固定 skill) |
| **Voyager** | ✅ | ❌ | ❌ | ✅ (hierarchical code skill library) |
| **Generative Agents** | ✅ | ✅ (events) | ✅ (reflections) | ❌ |
| **Tree of Thoughts** | ✅ (tree state) | ❌ | ❌ | ❌ |

读法：ReAct 之所以"没记忆"，是因为它**只有 working memory**——每次任务开始 prompt 都是干净的。Voyager 之所以能在 Minecraft 里"学技能"，是因为它实现了 **procedural memory 的显式写入**（把验证通过的代码追加到 skill library）。Generative Agents 之所以能模拟"长期生活"，是因为它同时实现了 **episodic memory（事件流）+ semantic memory（reflection 提炼出的人物画像）**。

**实践含义**：当有人说"我要做有记忆的 Agent"，第一个问题不是"用 Pinecone 还是 Weaviate"，而是**"你需要 CoALA 四类中的哪几类？"**。绝大多数生产场景只需要 Working + Semantic，不需要 Episodic（少量场景需要 few-shot 时除外），更不需要 Procedural（除非你在做 self-improving agent）。

## 3. 工程映射：LangChain / LangMem 的标准实现

CoALA 是理论框架，LangChain 团队 2025 年在此基础上发布的 [LangMem SDK](https://www.langchain.com/blog/langmem-sdk-launch) 把它工程化为了**几乎成为业界事实标准的 API**。理解 LangMem，就理解了 80% 的生产实现。

### 3.1 Short-term vs Long-term 的边界

LangChain 文档将 working memory 的工程边界定义为 **thread scope（线程作用域）**：

```python
# Short-term: thread-scoped, 由 checkpointer 持久化
# 包含 message history + 其他 state（上传的文件、检索的文档、生成的产物）
from langgraph.checkpoint.postgres import PostgresSaver

checkpointer = PostgresSaver.from_conn_string("postgresql://...")
graph = workflow.compile(checkpointer=checkpointer)

# 同一个 thread_id 内的所有调用共享 state
config = {"configurable": {"thread_id": "user_123_session_abc"}}
graph.invoke({"messages": [...]}, config=config)
```

```python
# Long-term: cross-thread, 由 store 持久化，按 namespace 组织
from langgraph.store.postgres import PostgresStore

store = PostgresStore.from_conn_string("postgresql://...", index={"embed": embed_fn})
graph = workflow.compile(checkpointer=checkpointer, store=store)

# namespace 决定作用域
namespace = ("user_123", "preferences")
store.put(namespace, "language", {"value": "Chinese"})
```

**这个二元划分是工程上最重要的设计决策**：thread-scoped 用 checkpointer，cross-thread 用 store。混用会导致跨会话脏数据、并发覆盖等问题。

### 3.2 Semantic Memory 的两种模式：Profile vs Collection

LangChain 文档对 semantic memory 给出了**两种互斥的存储模式**，选错代价极高。

#### Profile 模式：单一 JSON 文档不断重写

```python
# Profile: 一个用户对应一个 JSON 文档，整体 PUT 覆盖
user_profile_schema = {
    "name": "string",
    "preferences": {
        "language": "string",
        "verbosity": "concise | detailed",
        "code_style": "string"
    },
    "dietary_restrictions": ["string"],
    "timezone": "string"
}

# 写入：每次都是完整覆盖
store.put(("user_123", "profile"), "main", new_profile)
```

**优点**：结构清晰、随时可序列化为单一对象、易于 schema 校验。
**致命缺点**：随着 profile 字段增多，每次 LLM 重写都有 **"overwrite errors"**——某次更新可能"忘记"原来已有的字段。LangChain 推荐用 [Trustcall](https://github.com/hinthornw/trustcall) 这类基于 JSON Patch 的库做差量更新，而不是整体重写。

#### Collection 模式：增长式条目列表

```python
# Collection: 每条事实是一条独立 memory document
namespace = ("user_123", "facts")

# 每个新事实是新一条
store.put(namespace, "fact_001", {"content": "Prefers Python over TypeScript for backends", "ts": "2026-06-24"})
store.put(namespace, "fact_002", {"content": "Has a dog named Mochi", "ts": "2026-06-24"})

# 检索时按相关性搜索
results = store.search(namespace, query="programming preferences", limit=5)
```

**优点**：增量友好、易于 semantic search、不会"overwrite"。
**缺点**：Agent 必须主动**去重 / 更新 / 删除**已有条目（"User likes Python" 和 "Prefers Python for backends" 是同一条还是两条？），上下文组装时还要做 reranking。

**决策矩阵**：

| 场景 | 推荐 |
|---|---|
| 字段固定、结构稳定（地址、联系方式、订阅状态） | **Profile** |
| 偏好/兴趣/见解，会持续累积 | **Collection** |
| 跨用户的全局事实（公司政策、产品目录） | **Collection** |
| 单用户的核心身份（angular 项目里的 author 信息） | **Profile** |

实战推荐：**两者并存**。Profile 存"结构性事实"（schema 稳定），Collection 存"自由形态见解"（schema 可变）。

### 3.3 Episodic Memory：Few-shot Example Retrieval

LangChain 把 episodic memory 工程化为 **"按相关性检索的 few-shot 示例库"**：

```python
from langgraph.store.memory import InMemoryStore

# Episode 结构：input + trajectory + outcome
episode = {
    "user_query": "How do I refund order 12345?",
    "trajectory": [
        {"tool": "lookup_order", "args": {"id": "12345"}, "result": "..."},
        {"tool": "check_refund_policy", "args": {"order_id": "12345"}, "result": "..."},
        {"tool": "issue_refund", "args": {"id": "12345", "amount": 99.0}, "result": "ok"}
    ],
    "outcome": "success",
    "score": 0.95
}

store.put(("agent_support", "episodes"), "ep_001", episode)

# 推理时检索相似 episode 作为 few-shot
similar = store.search(
    ("agent_support", "episodes"),
    query=current_user_query,
    filter={"outcome": "success"},
    limit=3
)
prompt = build_prompt_with_examples(current_query, similar)
```

值得注意的是 [LangMem SDK Launch](https://www.langchain.com/blog/langmem-sdk-launch) 明确写道："LangMem doesn't yet support opinionated utilities for episodic memory."——业界对 episodic memory 的"标准做法"还没达成共识，原因是它**高度业务相关**：客服 agent 的 episode 是工单轨迹，代码 agent 的 episode 是 commit 序列，无法用同一套抽象覆盖。

**实战建议**：episodic memory 通常不要早做。先用 working + semantic 把基础 agent 跑通，等观察到"重复在做相同任务"或"用户反复问类似问题"时，再引入 episodic 做 few-shot retrieval。

### 3.4 Procedural Memory：Prompt 自我重写

这是 CoALA 警告"最危险"的一类。LangMem 提供的实现是 **Prompt Optimizer**（[LangMem SDK Launch](https://www.langchain.com/blog/langmem-sdk-launch)）：

```python
from langmem import create_prompt_optimizer

optimizer = create_prompt_optimizer(
    "anthropic:claude-3-5-sonnet-latest",
    kind="metaprompt",  # 或 "gradient" / "prompt_memory"
    config={"max_reflection_steps": 3}
)

# trajectories: 历史轨迹 + 评分
trajectories = [
    (messages_run_1, {"score": 0.5, "comment": "Was too verbose"}),
    (messages_run_2, {"score": 0.9, "comment": "Perfect concision"}),
]

improved_prompt = optimizer.invoke({
    "trajectories": trajectories,
    "prompt": current_system_prompt
})
```

三种算法的取舍：

| 算法 | 步数 | 适用 |
|---|---|---|
| `metaprompt` | 多步反思 + meta-prompt 生成更新 | 复杂场景，需要"思考时间" |
| `gradient` | 分离 critique 和 propose | 需要可解释的"为什么改" |
| `prompt_memory` | 单步直接改 | 简单场景，低成本 |

**为什么危险**：Procedural memory 一旦写错，所有后续行为都会偏离。CoALA 论文明确建议 procedural memory 的**写入必须经过 reflection 和评分门控**。在生产中，至少要做到：

1. **新 prompt 必须先在 sandbox/staging 跑 N 次 evaluation** 才能 promote 到 production
2. **保留历史版本**（git 化 prompt 仓库）以便快速回滚
3. **A/B 测试**新旧 prompt，至少 5% 流量灰度

这条架构原则的根源是 4.4（Loop 治理）讨论过的 **Verification Burden**：Procedural memory 的写入相当于在 production 改代码，必须有对应的 CI/CD 安全网。

## 4. 工业实现对比：Mem0 / Letta / Zep / LangMem

理论框架到这里清晰了。但生产选型时，工程师面对的是一组**已经做出不同取舍的产品**：Mem0、Letta（前 MemGPT）、Zep（Graphiti）、LangMem。它们都自称"agent memory"，但**底层架构和擅长场景完全不同**。

### 4.1 架构差异速览

基于 [State of AI Agent Memory 2026](https://mem0.ai/blog/state-of-ai-agent-memory-2026) 和 [AI Agent Memory Architectures 2026 分析](https://zylos.ai/research/2026-04-05-ai-agent-memory-architectures-persistent-knowledge/) 的横向对比：

| 系统 | 核心抽象 | 存储基质 | 杀手锏 |
|---|---|---|---|
| **Mem0** | Extraction LLM + Vector + (optional) Graph | Vector DB + Graph | "Extract & consolidate" 流水线，token 效率 |
| **Letta**（前 MemGPT） | LLM-controlled paging（OS 虚拟内存类比） | Core / Archival 双层 + 函数调用 | LLM 自己决定 "page in/out" |
| **Zep / Graphiti** | Temporal Knowledge Graph | Graph DB 为主 | 时序边、事实版本管理 |
| **LangMem** | Memory Manager + Prompt Optimizer | 任意 store（LangGraph 原生） | Procedural memory（prompt 自演化） |

### 4.2 LoCoMo 2026 基准：当前 SOTA 表现

LoCoMo（Long Conversation Memory）是 2025 年发布的标准基准，包含 **1,540 道题**覆盖 single-hop、multi-hop、open-domain、temporal 四类。2026 年初的代表性分数：

| 系统 | LoCoMo Judge Score | 备注 |
|---|---|---|
| **MAGMA** | 0.70 | 最新 SOTA |
| **Mem0** | 0.66+ | 论文报告 vs OpenAI Memory 提升 26% |
| **Nemori** | 0.59 | |
| **A-MEM** | 0.58 | |
| **MemoryOS** | 0.553 | |
| **Full-context（baseline）** | ~0.52 | 把全部历史塞进 context |

来自 [Mem0 论文 arXiv:2504.19413](https://arxiv.org/abs/2504.19413) 的关键数字（在 LoCoMo 上对比 full-context）：

- **p95 延迟降低 91%**（路径：检索 vs 全量读取）
- **token 成本降低 90%+**（6,956 tokens/query vs ~26,000）
- **LLM-as-Judge 准确率提升 26%**（vs OpenAI Memory）
- **Graph 变体**比 base Mem0 再高 ~2%

**第一个非平凡的洞察**：LoCoMo 上 SOTA 是 0.70 左右——这意味着即使是当前最强的 agent memory 系统，在"长对话记忆"任务上**仍然有 30% 的错误率**。生产部署时必须假设 memory **会出错**，对应的容错策略（人工 override、可纠错、可解释）必须设计在内。

**第二个洞察**来自 BEAM 基准：在 1M tokens 时分数 64.1，10M tokens 时 48.6——**context 扩大 10× 性能损失 ~25%**。这从反方向证明了"光靠扩 context window 解决不了 agent 记忆"。

### 4.3 选型决策矩阵

| 你的场景 | 推荐 |
|---|---|
| 已经用 LangGraph / LangChain | **LangMem**（同栈零迁移、Procedural memory 强） |
| 偏 Chat / 客服，需要 user-level long-term memory | **Mem0**（提取-巩固流水线成熟） |
| 多 agent / 长 horizon、需要"无限上下文"幻觉 | **Letta**（page in/out 哲学） |
| 强时序关系（金融、医疗、合规） | **Zep / Graphiti**（时序图、事实版本） |
| 本地/隐私敏感 | **OpenMemory MCP**（local-first, Claude Desktop / Cursor 兼容） |
| 不想引入额外依赖 | 自己实现：Postgres + pgvector + 简单 namespace |

**反模式提醒**：不要"一上来就上 Knowledge Graph"。[State of AI Agent Memory 2026](https://mem0.ai/blog/state-of-ai-agent-memory-2026) 报告显示，95% 的 2025 年 RAG 系统仍然只用 vector embedding——不是因为 KG 不好，而是因为 KG 的运维成本（ontology 维护、entity disambiguation）通常远高于其带来的检索质量提升。Graph 应该是 vector 已经吃到瓶颈后的**第二阶段**优化。

## 5. Mini 实现：从零搭一个四层记忆 Agent

理论说完了，给一个**最小可运行的四层记忆 Agent 骨架**。我们用 LangGraph + LangMem 实现，重点展示四类记忆的协同。

```python
"""
四层记忆 Agent 骨架
- Working Memory: LangGraph State（thread-scoped）
- Episodic Memory: Store namespace "episodes"，按相关性检索 few-shot
- Semantic Memory: Store namespace "facts"，Collection 模式
- Procedural Memory: Store namespace "instructions"，单 prompt 重写
"""

from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.store.postgres import PostgresStore
from langgraph.graph.message import add_messages
from langmem import create_memory_manager, create_prompt_optimizer


# ---- State 定义（Working Memory）----
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_id: str
    retrieved_facts: list
    retrieved_episodes: list


# ---- 三个 long-term 子系统 ----
fact_manager = create_memory_manager(
    "anthropic:claude-3-5-sonnet-latest",
    instructions="Extract durable user preferences and facts. Skip ephemeral context.",
    enable_inserts=True,
)

prompt_optimizer = create_prompt_optimizer(
    "anthropic:claude-3-5-sonnet-latest",
    kind="metaprompt",
    config={"max_reflection_steps": 3},
)


# ---- 节点 1：检索（Retrieval action）----
def retrieve_memories(state: AgentState, *, store):
    user_id = state["user_id"]
    query = state["messages"][-1].content

    # 拉 semantic memory
    facts = store.search(
        (user_id, "facts"), query=query, limit=5
    )

    # 拉 episodic memory（仅成功案例）
    episodes = store.search(
        (user_id, "episodes"), query=query,
        filter={"outcome": "success"}, limit=3
    )

    return {"retrieved_facts": facts, "retrieved_episodes": episodes}


# ---- 节点 2：推理（Reasoning + Grounding）----
def reason_and_act(state: AgentState, *, store):
    # 拉 procedural memory（system prompt）
    instr = store.get((state["user_id"], "instructions"), "system_prompt")
    system_prompt = instr.value["prompt"] if instr else DEFAULT_SYSTEM_PROMPT

    # 组装最终 prompt：procedural + semantic + episodic + working
    prompt = build_prompt(
        system=system_prompt,
        facts=state["retrieved_facts"],
        examples=state["retrieved_episodes"],
        messages=state["messages"],
    )
    response = llm.invoke(prompt)
    return {"messages": [response]}


# ---- 节点 3：Background 学习（Learning action）----
async def consolidate_memory(state: AgentState, *, store):
    """
    在响应返回给用户之后异步执行。
    1. 提取新事实写入 semantic memory
    2. 把当前 trajectory 存入 episodic memory（待人工/自动评分）
    3. 不在这里写 procedural memory——那条路径独立、有评测门控
    """
    user_id = state["user_id"]

    # 1. Semantic: 增量提取
    existing = store.search((user_id, "facts"), limit=50)
    new_or_updated = fact_manager.invoke({
        "messages": state["messages"],
        "existing": existing,
    })
    for item in new_or_updated:
        store.put((user_id, "facts"), item.id, {"content": item.content})

    # 2. Episodic: 写入（outcome 待评分 worker 更新）
    episode_id = f"ep_{int(time.time())}"
    store.put((user_id, "episodes"), episode_id, {
        "messages": [m.dict() for m in state["messages"]],
        "outcome": "pending",
        "ts": now_iso(),
    })


# ---- 节点 4：离线 Procedural 优化（独立 cron / nightly job）----
def optimize_procedural_memory(user_id: str, *, store):
    """
    完全独立于热路径。从 episodic memory 拉评分后的轨迹，
    用 prompt optimizer 提议新 prompt，经过评测门控才 promote。
    """
    eps = store.search(
        (user_id, "episodes"),
        filter={"outcome": {"$in": ["success", "failure"]}, "scored": True},
        limit=100,
    )
    trajectories = [(ep.value["messages"], {"score": ep.value["score"]}) for ep in eps]

    current = store.get((user_id, "instructions"), "system_prompt")
    current_prompt = current.value["prompt"] if current else DEFAULT_SYSTEM_PROMPT

    new_prompt = prompt_optimizer.invoke({
        "trajectories": trajectories,
        "prompt": current_prompt,
    })

    # ⚠️ 关键：评测门控。不通过就不写。
    if not passes_eval_gate(new_prompt, eval_set):
        log.warning("Prompt optimization rejected by eval gate")
        return

    store.put((user_id, "instructions"), "system_prompt", {
        "prompt": new_prompt,
        "version": next_version(),
        "promoted_at": now_iso(),
        "supersedes": current_prompt[:200],
    })


# ---- 图编排 ----
workflow = StateGraph(AgentState)
workflow.add_node("retrieve", retrieve_memories)
workflow.add_node("act", reason_and_act)
workflow.add_node("consolidate", consolidate_memory)  # 实际部署用后台 worker

workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "act")
workflow.add_edge("act", "consolidate")  # 或者发到队列
workflow.add_edge("consolidate", END)

checkpointer = PostgresSaver.from_conn_string(POSTGRES_URL)
store = PostgresStore.from_conn_string(POSTGRES_URL, index={"embed": embed_fn})
graph = workflow.compile(checkpointer=checkpointer, store=store)
```

**这个骨架展示了四个关键设计决策**：

1. **Working memory（`AgentState`）只持有 thread-scoped 数据**，跨会话信息全部在 store。
2. **三类 long-term 各占独立 namespace**，可独立 TTL、独立备份、独立删除（用户行使"被遗忘权"时只删 facts/episodes namespace，不动 instructions）。
3. **Semantic / Episodic 在 hot path 之后异步 consolidate**，不阻塞用户响应。
4. **Procedural 完全独立 pipeline**，强制经过 eval gate——这是 CoALA "risky write" 警告的工程兑现。

## 6. 进阶话题：遗忘、巩固、漂移

记忆系统永远不会"只增不减"，否则就变成 [arXiv 2603.11768](https://arxiv.org/html/2603.11768v1) 论文里描述的灾难场景。这篇 2026 年的 **SSGM（Self-Stabilizing Governed Memory）** 论文形式化了几个绕不开的问题：

### 6.1 Memory Drift 的三种形态

| 漂移类型 | 表现 | 触发机制 |
|---|---|---|
| **Semantic Drift** | "我喜欢微辣"→反复 summarize→变成"我喜欢鬼椒翼" | 迭代式 summarization 的有损压缩 |
| **Procedural Drift** | Agent 学到一个 workaround，逐渐固化成"标准流程" | 强化学习/RLHF 在 trajectory 上的隐式偏置 |
| **Goal Drift** | 长期交互中目标从 A 渐变到 A' | 累积偏置 + 用户漂移 + reflection 错误 |

论文给出的关键定理（Theorem 1, Bounded Semantic Drift）：

- **朴素系统**：漂移按 O(T · ε_step) 累积（T = 时间步、ε_step = 单步误差）
- **加上每 N 步做一次 reconciliation**：漂移降到 O(N · ε_step)

**工程含义**：必须周期性把 evolved memory 与 ground truth（用户原话、原始事件日志）做对账。生产实现就是 "**Mutable Active Graph + Append-only Immutable Episodic Log**" 双存储——前者可改、后者只读，定期用后者校准前者。

### 6.2 遗忘机制：Priority Decay

最简单也最实用的遗忘策略，用 Weibull 衰减函数为每条记忆打分：

```
w(Δτ) = exp(−(Δτ/η)^κ)
```

- `Δτ`：自上次访问以来的时间
- `η`：寿命参数（用户偏好 vs 事件 vs 操作技能不同）
- `κ`：形状参数（控制衰减曲线）

低于阈值 `θ_fresh` 的条目进入"垃圾回收候选"。但**注意**：Procedural memory 不应自动衰减（删 prompt 风险太大），通常只对 episodic 做积极 GC、对 semantic 做"标记 stale 但不删"。

### 6.3 SSGM 的四条治理原则

值得作为生产 checklist：

1. **Pre-Consolidation Validation**：写入前用 Truth Maintenance System 校验，与"core facts"逻辑矛盾的拒写（公式：`ΔM ∧ M_core ⊨ ⊥` 时拒绝）。
2. **Temporal & Provenance Grounding**：每条记忆带 cryptographic provenance + 时间戳，retrieval 时按新鲜度加权。
3. **Access-Scoped Retrieval**：检索不能只靠语义相似度，必须叠加 ABAC（attribute-based access control）。这是合规场景的硬要求。
4. **Reversible Reconciliation**：上面提到的双存储——任何写入可回滚。

这四条与 4.4 章 Loop 治理里的 **Verification Burden / Comprehension Debt** 是同一个方法论族在记忆领域的延伸。

## 7. 反模式与红线

总结一份"看到就拒签"的清单：

| 反模式 | 为什么错 | 正确做法 |
|---|---|---|
| 把全部 conversation history 塞进 system prompt | 三类记忆混在一起、成本 8×、lost-in-the-middle | 拆成 working + 异步写入的 long-term |
| 用一个 Pinecone collection 存所有用户/所有事实/所有 episode | 检索精度暴跌、隔离失败 | 按 `(user_id, kind)` 严格 namespace |
| Hot path 写入 + 同步 embedding | 用户响应延迟 +500ms ~ +2s | Background worker + 队列 |
| 让 LLM 自由重写 system prompt 后立即生效 | Procedural drift 一发不可收 | Sandbox + eval gate + 灰度 |
| 用单一 JSON profile 存 50+ 字段并整体覆盖 | overwrite errors 累积 | Trustcall / JSON Patch，或拆成 Collection |
| 不区分 user_id 和 session_id | "上次的退款"被错锚 | 四级 namespace 严格区分 |
| 不带时间戳的 semantic memory | 无法解决 temporal obsolescence | 每条事实必带 `valid_from` / `valid_until` |
| 把 episodic memory 当 audit log 用 | 两套需求互相干扰（一个要遗忘，一个要永久） | 两套独立存储（Mutable + Immutable） |

## 8. 进阶路线图：从 0 到生产级记忆系统

最后给一个**渐进式路线图**，配合你的 agent 成熟度逐步演进，不要一上来就追 SOTA：

| 阶段 | 配置 | 上线条件 | 不做什么 |
|---|---|---|---|
| **S0 Stateless** | 仅 working memory（prompt 内） | MVP 验证流程 | 不做任何 long-term |
| **S1 Profile** | + 单 JSON profile（Trustcall 增量） | 有"用户偏好"明确需求 | 不做 episode、不做 vector |
| **S2 Collection** | + Collection 模式 semantic memory + vector | 偏好/兴趣需要积累 | 暂不引入 KG |
| **S3 Episodic** | + few-shot episode 检索 | 观察到任务重复性 | 不要早做 prompt 自演化 |
| **S4 Governance** | + TTL / Decay / 双存储 / Provenance | 进入合规/医疗/金融场景 | |
| **S5 Procedural** | + Prompt Optimizer + Eval Gate | 有完善 eval pipeline | |
| **S6 Graph** | + Knowledge Graph（Zep/Graphiti） | Vector 已经吃到精度瓶颈 | |

实操建议：**至少在 S2 阶段停留 3-6 个月**积累数据，再决定要不要 S3+。绝大多数 agent 应用最终止于 S2-S3，不需要 S5/S6。
