# Prompt Engineering 系统化：从散装技巧到决策框架

> 本文不是又一篇"10 个 Prompt 技巧合集"。它的目标是把散装技巧组织成一个可以在工程决策中复用的框架——面对一个具体任务时，你应该按什么逻辑选择 prompt 策略、组合哪些技术、如何验证效果。

## 0. 开场问题

假设你要让 LLM 帮你做这三件事：

1. 对 500 条客服工单做情感分类（positive / neutral / negative）
2. 把一段自然语言需求转换成 SQL 查询
3. 分析一份 20 页合同中的风险条款并给出修改建议

三件事都是"写个 prompt 让模型干"，但它们对 prompt 的要求完全不同。情感分类需要的是格式稳定和批量一致性；SQL 生成需要的是精确遵循 schema 和语法；合同分析需要的是深度推理和引证。如果你用同一种"请帮我做 X"的 prompt 模板去处理三者，前两个会因为输出格式不稳定而需要大量后处理，第三个会因为推理深度不足而漏掉关键风险。

Prompt Engineering 系统化要解决的就是这个问题：**给定任务特征，如何做出正确的 prompt 设计决策。**

## 1. Prompt 的物理学：Token 预算与注意力衰减

在讲技巧之前，先理解约束。

### 1.1 上下文窗口是硬性预算

每个 prompt 本质上是在争夺有限的 token 预算。一次 API 调用的 token 消耗 = system prompt + 历史消息 + 用户输入 + 工具定义 + 模型输出。窗口是硬约束——超了就截断或报错。但即使在窗口之内，1.1 篇讲过的注意力 U 形衰减也意味着：**放在 prompt 中间的信息，被模型"看到"的概率显著低于头尾。**

工程含义：

- 最重要的指令放开头（system prompt 首段）和结尾（紧贴用户问题前）
- 大段参考资料用 XML/Markdown 标记包裹，给模型"目录感"
- 超过模型有效注意力半径的内容，宁可分次喂而非一次塞

### 1.2 Prompt Caching 改变了成本方程

Anthropic（2024 年推出，2025 年已全面生产化）的 prompt caching 机制在工程上有深远影响。标注了 `cache_control: {"type": "ephemeral"}` 的 system prompt 块，在 5 分钟 TTL 内的后续请求只按 10% 的价格收费。这意味着：

- **长 system prompt 不再是成本灾难**——一个 8000 token 的 AGENTS.md 如果每分钟被调用一次，90% 的调用只花 1/10 的钱
- **把稳定信息前置到 system prompt，把变化信息后置到 messages**——这不只是组织习惯，而是成本优化策略
- OpenAI 的 cached input（GPT-4o / o1 系列）同样按 50% 折扣计费，但缓存策略是自动的、不可控的

结论：现代 prompt 设计应该按"缓存友好"来分层——越稳定的放越前面。

## 2. 决策树：给定任务，选择什么 Prompt 策略

把常见任务按两个维度分类：**推理深度**（任务需要多少步逻辑链）和**输出确定性**（输出格式是否需要严格可预测）。

```
                    输出确定性要求
                    低                高
推理     ┌─────────────────┬─────────────────┐
深度     │  Q3: 创意写作    │  Q4: 合规检查    │
要求  高 │  开放推理 +      │  深度推理 +      │
         │  自由格式        │  结构化输出      │
         ├─────────────────┼─────────────────┤
         │  Q1: 翻译/摘要   │  Q2: 分类/抽取   │
      低 │  简单指令 +      │  简单指令 +      │
         │  自由格式        │  结构化输出      │
         └─────────────────┴─────────────────┘
```

每个象限对应不同的 prompt 策略组合：

**Q1（低推理 + 低确定性）**：简洁指令 + 少量示例即可。不需要 CoT，不需要 JSON mode。典型任务：翻译、风格改写、摘要。

**Q2（低推理 + 高确定性）**：Structured Output / JSON Schema 约束 + few-shot 示例。关键在于格式稳定，而非推理深度。典型任务：情感分类、NER、日志解析。

**Q3（高推理 + 低确定性）**：Chain-of-Thought 或直接用 reasoning model。输出可以是自然语言。关键在于让模型有足够的"思考空间"。典型任务：代码 review、策略分析、开放性问题解答。

**Q4（高推理 + 高确定性）**：这是最难的象限——既要深度推理，又要输出格式严格可预测。典型任务：合同风险分析（要求 JSON 结构化输出 + 引证条款号）、SQL 生成（要求语法正确 + 逻辑正确）。需要组合使用：reasoning model + structured output + verification step。

## 3. 核心技术栈：七种 Prompt 技术的适用场景与局限

### 3.1 Zero-shot 直接指令

最简单的形态——直接告诉模型做什么，不给示例。

**适用**：模型已经对该任务有强先验（翻译、摘要、代码补全）。

**陷阱**：输出格式不可预测。同一个 prompt 在不同温度、不同模型版本下可能返回完全不同的结构。

**工程建议**：生产环境中几乎不应该只用 zero-shot——至少加一个输出格式说明。

### 3.2 Few-shot 示例驱动

提供 2-5 个输入→输出的示例对，让模型从模式中归纳。

**适用**：分类、抽取、格式转换。Few-shot 的真正价值不在于"教模型做什么"（它通常已经会），而在于**锁定输出格式和边界判断**。

**2025 新认知**：Anthropic 的 context engineering 文档明确指出，示例是"模型的图片胜千言"（examples are the model's equivalent of pictures worth a thousand words）——但不应该穷举所有 edge case，而应该选择"多样化的典型样本"。

**反模式**：
- 示例过多（>10 个）导致 token 浪费，且后面的示例可能落入注意力盲区
- 示例间存在矛盾（模型会困惑而非报错）
- 示例只覆盖"正常"情况，没覆盖边界/拒绝场景

### 3.3 Chain-of-Thought（CoT）

让模型在给出答案前先展示推理过程。Wei et al. (2022, Google) 的奠基论文证明 CoT 在 PaLM 540B 上把 GSM8K 数学题正确率从 17.9% 提升到 56.9%。

**2025 年的新情况**：

CoT 已经分化为两个世界：

1. **手动 CoT**（在 prompt 中写"Let's think step by step"或提供推理示例）——仍然对非推理模型有效（GPT-4o、Claude Sonnet）
2. **内置 CoT**（o1/o3/GPT-5.5 的 reasoning tokens、Claude extended thinking）——模型在隐藏层自动做链式推理，不需要你提示

**关键发现（2025 共识）**：

- 对推理模型说"think step by step"是**反模式**——它要么浪费 token（模型在可见输出和隐藏推理中重复推理），要么限制模型选择更优路径（ACL 2025，COLING 2025 Chain-of-X 综述）
- CoT 会放大幻觉：如果第一步推理出错，后续步骤会"精心论证一个错误结论"
- 推理 token 有边际效益递减：10,000 token 的思考比 1,000 token 高 15-25%，但 100,000 token 不会比 50,000 好多少
- 简单任务上 CoT 可能**降低**准确率（overthinking——模型 second-guess 一个本来对的直觉答案）

**决策规则**：
- 如果任务需要 >3 步逻辑链 → 用 CoT 或 reasoning model
- 如果用 reasoning model → **不要在 prompt 里加 CoT 指令**，改用 reasoning_effort/budget_tokens 参数控制深度
- 如果任务是分类/抽取/格式转换 → 不要用 CoT，直接 few-shot

### 3.4 Self-Consistency（多路投票）

Wang et al. (2022) 提出：对同一问题多次采样（temperature > 0），取多数答案。可以在不改 prompt 的情况下提升 10-15% 可靠性。

**适用**：单次答案不够可靠、且有一个"可验证正确性"的任务——数学题、事实性判断、代码 bug 定位。

**成本公式**：N 次采样 = N 倍 token 消耗。所以只在**高价值、低频**的判断上使用。Agent 场景中，Self-Consistency 可以作为 Sensor 的一种实现——让多个 sub-agent 独立分析同一段代码，投票出结论。

**2025 进展**：现代 reasoning model 内部已经做了类似的 "internal self-consistency"（生成多条推理路径，内部选最优），所以外部 self-consistency 在 o3/Claude extended thinking 上的边际收益在下降。

### 3.5 Structured Output（结构化输出）

2025 年结构化输出已从"nice-to-have"变成生产必需品。三大提供商都提供了原生支持：

| 提供商 | 机制 | 保证级别 |
|---|---|---|
| OpenAI | `response_format: {type: "json_schema", json_schema: {...}}` | 100% schema 合规（通过 constrained decoding） |
| Anthropic | Tool use 的 input_schema（JSON Schema） | 高度可靠但非 100%（概率采样） |
| vLLM/本地 | Outlines / XGrammar / guided decoding | 100%（grammar-constrained） |

**工程要点**：

- OpenAI 的 Structured Outputs（大写 S）使用 constrained decoding，数学上保证输出 100% 符合你的 JSON Schema——这在 Agent 工具调用中极其重要
- Anthropic 的 tool use 本质上就是结构化输出——`tools` 参数中的 `input_schema` 告诉模型必须输出什么结构
- 本地部署用 vLLM 的 guided decoding（支持 JSON Schema、正则表达式、CFG），可以做到 100% 格式合规

**反模式**：在 prompt 里用自然语言描述 JSON 格式（"请输出一个包含 name、age、score 字段的 JSON"）——这在 2023 年是标准做法，在 2025 年是浪费。用原生 schema 约束。

### 3.6 Reasoning Model 专用策略

2025-2026 年最大的 prompt engineering 范式转变：**推理模型需要一套反直觉的 prompt 写法。**

核心原则（综合 OpenAI o3、Claude extended thinking、GPT-5.5 的官方指导）：

**说目标，不说步骤**——给 destination，不给 route。推理模型的优势就在于自己规划路径，你写的步骤反而成为约束。

**给约束，不给流程**——边界条件（字数限制、技术栈要求、预算上限）是好的；"先做 A，再做 B，再做 C"是有害的。

**不要 few-shot 推理示例**——模型可能锚定你的推理模式而非发现更优路径。few-shot 仍可用于格式/分类，但不要示范推理过程。

**用参数控制深度，不用文字控制**——"请深入思考"没用。GPT-5.5 用 `reasoning_effort: "high"/"medium"/"low"`；Claude 用 `budget_tokens: 10000`。

**要求自验证**——明确写"验证你的结论"比"仔细思考"有效得多。这触发模型在推理链末尾加一轮自检，而非增加前面的推理长度。

```python
# GPT-5.5 推理模型调用示例
response = client.chat.completions.create(
    model="gpt-5.5",
    reasoning_effort="high",  # 参数控制，非 prompt 文字
    messages=[
        {"role": "system", "content": "你是合同风险分析专家。"},
        {"role": "user", "content": f"""
分析以下合同的法律风险。

约束：
- 只关注中国法律管辖范围
- 只输出中/高风险条款
- 对每条风险给出修改建议
- 验证每条风险是否有明确的法律依据

合同内容：
{contract_text}
"""}
    ],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "risk_analysis",
            "schema": {
                "type": "object",
                "properties": {
                    "risks": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "clause_number": {"type": "string"},
                                "risk_level": {"enum": ["medium", "high"]},
                                "description": {"type": "string"},
                                "legal_basis": {"type": "string"},
                                "suggestion": {"type": "string"}
                                },
                            "required": ["clause_number", "risk_level", "description", "legal_basis", "suggestion"]
                        }
                    }
                },
                "required": ["risks"]
            }
        }
    }
)
```

### 3.7 Meta-prompting / Prompt Chaining

当单个 prompt 无法完成时，拆成多步：

- **Sequential chaining**：前一步的输出是后一步的输入（翻译 → 校对 → 格式化）
- **Parallel fan-out**：多个 prompt 独立处理同一输入的不同方面，再合并（类似 MapReduce）
- **Conditional routing**：先用一个分类 prompt 判断类型，再路由到专用 prompt

这是从 Prompt Engineering 到 Context Engineering / Harness Engineering 的过渡地带。当你开始做 chaining，你其实已经在写 Harness 了——只是用的是 prompt 而非代码来编排。

## 4. System Prompt 架构学

好的 system prompt 不是一段话，而是一个**分层结构**。Anthropic 在 2025 年的 context engineering 文档中给出了明确的架构建议：

### 4.1 分层结构

```xml
<system>
<!-- Layer 1: Identity & Tone -->
<identity>
你是 X 公司的法务助理 AI。你的沟通风格是专业但平易近人的。
</identity>

<!-- Layer 2: Core Capabilities & Boundaries -->
<capabilities>
你可以：分析合同、检索法规、生成修改建议。
你不可以：提供最终法律意见、代替律师签字。
</capabilities>

<!-- Layer 3: Output Specifications -->
<output_format>
所有分析结果以 JSON 格式输出，schema 见 tool 定义。
对话式回复使用中文，引用法条使用原文。
</output_format>

<!-- Layer 4: Domain Knowledge (cacheable) -->
<knowledge>
{公司合同模板}
{常用法条摘要}
{历史 case 摘要}
</knowledge>

<!-- Layer 5: Behavioral Guardrails -->
<guardrails>
如果用户要求你做超出能力范围的事，明确说明边界。
如果合同涉及刑事风险，建议用户直接咨询律师。
</guardrails>
</system>
```

### 4.2 设计原则

Anthropic 把好的 system prompt 描述为需要找到一个 **Goldilocks zone**：

- **太底层（brittle）**：写成硬编码的 if-else（"如果用户说 X，回复 Y"）——覆盖不了所有情况，脆弱
- **太高层（vague）**：写成通用指导（"请专业地回答"）——假设了模型不具备的共识
- **恰好**：足够具体以引导行为，又足够灵活以提供强启发式

实操建议：

1. 先用最简 prompt 跑最强模型，看哪里出错
2. 针对错误模式逐条加指令（而非一次写一堆"以防万一"的规则）
3. 每条指令都应该对应一个可观测的失败模式——如果你说不出"不加这条会出什么问题"，就别加

### 4.3 缓存友好的分层

结合 prompt caching，system prompt 应该按变化频率排列：

```
稳定度高 → 放前面（被缓存）
─────────────────────────
Identity & Tone          ← 几乎不变
Capabilities & Boundaries ← 版本间变
Domain Knowledge         ← 按项目变
Output Format            ← 按 API 版本变
─────────────────────────
变化频率高 → 放后面（每次新算）
Guardrails（可能根据用户行为动态调整）
```

## 5. 从 Prompt Engineering 到 Context Engineering 的过渡

Prompt Engineering 处理的是"单次调用内怎么说"。但在 Agent 场景中，单次 prompt 只是冰山一角——真正影响输出质量的是**整个上下文窗口的组成**：

- System prompt（稳定层）
- Tool definitions（能力描述）
- Message history（会话记忆）
- Retrieved documents（RAG 检索结果）
- Intermediate results（之前步骤的输出）

Anthropic 在 2025 年正式把这叫做 **Context Engineering**——"识别能最大化期望输出概率的最小高信号 token 集合"。这已经不是"怎么写 prompt"的问题，而是"怎么管理注意力预算"的问题。

核心差异：

| 维度 | Prompt Engineering | Context Engineering |
|---|---|---|
| 作用域 | 单次调用 | 整个 Agent 生命周期 |
| 优化目标 | 措辞/格式 | Token 预算分配 |
| 关键技术 | CoT/few-shot/structured output | Compaction/RAG/sub-agent/caching |
| 失败模式 | 格式错误/幻觉 | 上下文溢出/注意力稀释 |

这是 1.3 篇要展开的内容。这里只标记边界：**当你发现 prompt 写得再好也不够、需要动态拼装上下文时，你已经跨入了 Context Engineering 的领地。**

## 6. 完整代码示例：一个分类任务的 Prompt 工程全流程

以"客服工单情感分类"为例，演示从 naive prompt 到 production-ready 的演进：

```python
"""
演示：情感分类任务的 Prompt Engineering 演进路径
从 zero-shot → few-shot → structured output → batch + verification
"""
from anthropic import Anthropic

client = Anthropic()

# ===== Stage 1: Naive zero-shot（不可用于生产）=====
def classify_naive(text: str) -> str:
    """问题：输出格式不稳定，可能返回 'Positive'/'positive'/'积极' 等变体"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=50,
        messages=[{"role": "user", "content": f"对以下文本做情感分类：{text}"}]
    )
    return response.content[0].text  # 格式不可预测


# ===== Stage 2: Few-shot + 格式锚定 =====
FEW_SHOT_EXAMPLES = """
输入: "快递三天就到了，包装完好，非常满意！"
输出: positive

输入: "功能一般般吧，没什么特别的"
输出: neutral

输入: "等了两周还没发货，客服也联系不上，差评"
输出: negative

输入: "东西不错但物流太慢了"
输出: neutral
"""

def classify_fewshot(text: str) -> str:
    """改进：格式稳定，但仍需 parse 文本输出"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=20,
        system="你是情感分类器。只输出 positive/neutral/negative 三选一，不要输出其他任何内容。",
        messages=[{"role": "user", "content": f"{FEW_SHOT_EXAMPLES}\n输入: \"{text}\"\n输出:"}]
    )
    return response.content[0].text.strip()


# ===== Stage 3: Tool use 实现结构化输出（生产推荐）=====
CLASSIFICATION_TOOL = {
    "name": "submit_classification",
    "description": "提交情感分类结果",
    "input_schema": {
        "type": "object",
        "properties": {
            "sentiment": {
                "type": "string",
                "enum": ["positive", "neutral", "negative"],
                "description": "情感极性"
            },
            "confidence": {
                "type": "number",
                "minimum": 0,
                "maximum": 1,
                "description": "置信度 0-1"
            },
            "key_phrase": {
                "type": "string",
                "description": "支撑判断的关键短语"
            }
        },
        "required": ["sentiment", "confidence", "key_phrase"]
    }
}

SYSTEM_PROMPT = """你是情感分类器。分析用户提供的客服工单文本，判断情感极性。

判断标准：
- positive: 明确表达满意、推荐、好评
- negative: 明确表达不满、投诉、差评
- neutral: 中性描述、混合情感、或无法明确判断

用 submit_classification 工具提交结果。confidence < 0.7 时倾向标记为 neutral。"""

def classify_structured(text: str) -> dict:
    """生产级：结构化输出 + 置信度 + 可解释性"""
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=200,
        system=[{
            "type": "text",
            "text": SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"}  # 缓存 system prompt
        }],
        tools=[CLASSIFICATION_TOOL],
        tool_choice={"type": "tool", "name": "submit_classification"},
        messages=[{"role": "user", "content": f"工单内容：{text}"}]
    )
    # 直接从 tool_use block 拿到结构化结果
    for block in response.content:
        if block.type == "tool_use":
            return block.input  # {'sentiment': 'positive', 'confidence': 0.92, 'key_phrase': '非常满意'}
    return {"sentiment": "neutral", "confidence": 0.0, "key_phrase": ""}


# ===== Stage 4: 批量处理 + Self-consistency 验证 =====
def classify_batch_with_verification(texts: list[str], threshold: float = 0.8) -> list[dict]:
    """
    批量分类 + 低置信度二次验证。
    对 confidence < threshold 的样本，用不同温度重新采样 3 次取多数。
    """
    results = []
    uncertain = []

    # 第一轮：标准分类
    for i, text in enumerate(texts):
        result = classify_structured(text)
        result["index"] = i
        result["text"] = text
        if result["confidence"] >= threshold:
            results.append(result)
        else:
            uncertain.append(result)

    # 第二轮：对不确定样本做 self-consistency
    for item in uncertain:
        votes = []
        for _ in range(3):
            retry = classify_structured(item["text"])
            votes.append(retry["sentiment"])
        # 多数投票
        from collections import Counter
        majority = Counter(votes).most_common(1)[0]
        item["sentiment"] = majority[0]
        item["confidence"] = majority[1] / 3.0
        item["verification"] = "self_consistency_3x"
        results.append(item)

    results.sort(key=lambda x: x["index"])
    return results
```

这个演进路径体现了 prompt engineering 系统化的核心逻辑：**不是"哪个技巧最酷"，而是"当前的失败模式需要什么层级的解法"。**

## 7. 反模式清单

工程中高频出现的 prompt 反模式：

**"万能 system prompt"综合征**——把所有能想到的规则塞进一个巨大 system prompt，以为覆盖面越广越好。实际上：无关规则消耗注意力预算，互相矛盾的规则让模型困惑，很多规则对应的场景永远不会出现。正确做法：每条规则对应一个真实的失败案例，否则删掉。

**"请一步一步思考"的仪式化**——2022 年的技巧在 2025 年变成了 cargo cult。对 reasoning model 说这句话不仅没用还有害。对非 reasoning model，也应该替换成更具体的指令（"先列出涉及的法条，再逐条分析适用性"比泛泛的"一步一步"有效得多）。

**示例过拟合**——few-shot 示例全是"正常情况"，模型在边界情况下表现离谱。至少 30% 的示例应该覆盖 edge case 和拒绝场景。

**用自然语言描述格式**——"请返回 JSON 格式，包含以下字段..."在有原生 structured output 支持的时代是浪费且不可靠的。

**忽略温度对可复现性的影响**——生产环境做分类/抽取用 temperature=0（或接近 0），做创意生成用 >0。混用场景（"既要创意又要格式稳定"）需要拆成两步。
