# Multi-Agent 设计模式：从编排到涌现

> 单 Agent 能做的事有天花板——上下文窗口塞不下所有信息、一个 prompt 覆盖不了所有场景、一次推理链很难同时兼顾深度和广度。Multi-Agent 的核心命题不是"多个模型比一个强"，而是**分治、隔离、并行**——通过架构手段把复杂度拆到每个 Agent 能稳定处理的粒度。

## 0. 从一个真实的生产瓶颈说起

某电商客服 AI 用单 Agent 处理所有场景——退款、物流查询、商品咨询、投诉升级。system prompt 长达 8000 tokens，工具列表 25 个，分支逻辑全靠自然语言描述。当"退款政策变更"更新时，改一句 prompt 导致"物流查询"场景的工具选择准确率从 94% 掉到 81%——因为模型在过长的指令中迷失了优先级。

解法：拆为 4 个专业 Agent + 1 个路由 Agent。每个 Agent 的 prompt <2000 tokens、工具 3-5 个、职责单一。退款策略更新只改退款 Agent 的 prompt，其他场景完全不受影响。上线后整体准确率从 87% 提升到 95%，且各场景可以独立 A/B 测试。

这就是 Multi-Agent 的核心价值：**关注点分离带来的可维护性和稳定性**。

## 1. 第一性原理：为什么需要多 Agent

### 1.1 单 Agent 的三重天花板

| 天花板 | 症状 | Multi-Agent 如何解决 |
|--------|------|---------------------|
| **上下文容量** | 工具定义 + 领域知识 + 对话历史溢出窗口 | 每个 Agent 只加载自己需要的上下文 |
| **指令干扰** | 一个场景的优化破坏另一个场景 | 隔离 prompt，互不干扰 |
| **串行瓶颈** | 多步任务必须顺序执行，延迟累加 | 无依赖的子任务并行执行 |

### 1.2 Multi-Agent 不是银弹

Anthropic 在"Building Effective Agents"中的核心原则是：**从最简方案出发，只在复杂度确实带来可测量收益时才升级**。很多场景下，单 Agent + 更好的 prompt/工具设计就够了。引入 Multi-Agent 意味着更高的延迟、更多的 token 消耗、更复杂的调试。

判断标准：如果你的单 Agent prompt 超过 5000 tokens、工具超过 8 个、或者不同子任务的质量指标出现跷跷板效应，才考虑拆分。

## 2. 五大核心模式

### 2.1 路由模式（Router / Triage）

```
         ┌─── Agent A (退款)
User ──► Router ─┼─── Agent B (物流)
         └─── Agent C (咨询)
```

**机制**：一个轻量 Agent（或传统分类器）对输入做意图识别，将请求分发到专业 Agent。各专业 Agent 有独立的 prompt、工具集和领域知识。

**关键设计要素**：
- Router 的分类准确率决定系统上限
- 分类错误时需要回退机制（Agent 可以声明"这不是我的事"并退回 Router）
- Router 可以用 LLM，也可以用传统 NLU（更快、更便宜、更可控）

**Anthropic 的建议**：路由的核心价值是"关注点分离"——不同类别的输入适合不同的 prompt 和工具组合。没有路由时，"优化一类输入可能损害其他类输入的表现。"

**适用场景**：
- 客服（多类型工单）
- API 网关（多后端服务）
- 模型级联（简单问题→小模型，复杂问题→大模型）

### 2.2 编排者-工作者模式（Orchestrator-Workers）

```
              ┌─── Worker A (搜索)
User ──► Orchestrator ─┼─── Worker B (分析)
              └─── Worker C (写作)
              │
              ▼
         综合结果返回
```

**机制**：中央编排者（Orchestrator）接收任务，动态拆解为子任务，分派给 Worker Agent，收集结果后综合输出。

**与路由的关键区别**：路由是"一进一出"（请求被分发到一个 Agent），编排是"一进多出"（任务被拆成多个子任务并行/串行执行后聚合）。

**Anthropic 的定位**：适合"无法预先定义子任务"的复杂场景——子任务的种类和数量由编排者根据具体输入动态决定。

**OpenAI Agents SDK 的实现**：通过 `Agent.as_tool()` 机制，Orchestrator 将专业 Agent 包装为工具调用，保持控制权在 Orchestrator 手中。Orchestrator 拥有最终输出权，可以组合多个 Worker 的结果。

**适用场景**：
- 代码修改（分析影响范围→修改多文件→运行测试→修复失败）
- 研究任务（搜索多源→筛选→综合→输出报告）
- 数据处理 pipeline（提取→清洗→转换→加载）

### 2.3 层级模式（Hierarchical）

```
              Strategic Coordinator
              /                    \
    Domain Supervisor A      Domain Supervisor B
    /        |        \      /        |        \
Worker  Worker  Worker  Worker  Worker  Worker
```

**机制**：多级树形结构。高层做战略决策，中层做战术协调，底层执行具体任务。每一级都是上一级的"工作者"，同时是下一级的"编排者"。

**核心优势**：
- 上下文隔离：没有任何单一 Agent 需要持有全局信息
- 对数级扩展：20+ Agent 通过 3 级层级组织，每个节点只管 3-5 个下属
- 故障隔离：一个分支的失败不影响其他分支

**核心代价**：
- 每层至少一次 LLM 调用的延迟累加（3 层 × 2s = 6s 最低启动延迟）
- 逐层总结时的信息损失（"电话游戏"效应）

**LangChain 的基准测试发现**：Supervisor 模式（层级的简化形式）的主要性能损失来自"翻译"——Supervisor 在转达 Worker 结果时的改写会丢失细节。解法：`forward_message` 工具让 Supervisor 直接转发原文。

**适用场景**：
- 大型企业系统（多业务域协同）
- 代码库级重构（按模块分治）
- 多轮复杂调研（多领域信息收集+综合）

### 2.4 群体模式（Swarm）

```
  Agent A ←→ Agent B
    ↕            ↕
  Agent C ←→ Agent D
   (无中心协调者，基于共享状态自组织)
```

**机制**：完全去中心化。没有编排者，Agent 作为对等节点，基于共享状态（黑板/消息总线）做局部决策，通过 handoff 协议在 Agent 之间传递控制权。协调行为从简单的局部规则中**涌现**。

**OpenAI Swarm 的 Handoff 设计**：Agent 可以声明 `handoffs` 列表，当模型判断当前任务不属于自己时，主动将控制权转交给更合适的 Agent。被转交的 Agent 成为新的"活跃 Agent"。

**LangChain 基准测试结论**：Swarm 在性能上略优于 Supervisor（因为没有中间翻译层），但要求所有 Agent 互相知道彼此的存在——对第三方/跨团队 Agent 不友好。

**核心特征**：
- 高容错：任何 Agent 故障不影响全局
- 并行探索：50 个假设可以同时验证
- 难以调试：需要分布式追踪 + 共享状态回放
- 需要显式终止条件（最大迭代/质量阈值/超时）

**适用场景**：
- 大规模并行探索（竞品调研、假设验证）
- 对话式多角色系统（每个 Agent 一个角色）
- 自修复系统（Agent 互相监控和接管）

### 2.5 事件驱动模式（Event-Driven）

```
                Event Bus (Kafka / Message Queue)
               /          |          \
  Agent A (produce)  Agent B (consume+produce)  Agent C (consume)
```

**机制**：Agent 之间通过事件总线通信，不直接调用。每个 Agent 订阅感兴趣的事件类型，处理后发布新事件。解耦程度最高。

**Confluent 定义的四种事件驱动子模式**：

| 子模式 | 结构 | 特点 |
|--------|------|------|
| Orchestrator-Worker（事件版） | 中心协调 + 事件分发 | Kafka partition 做负载均衡 |
| Hierarchical（事件版） | 多层递归委托 | 每个非叶节点同时是 worker 和 orchestrator |
| Blackboard | 共享知识库（Kafka topic） | Agent 异步贡献，知识增量积累 |
| Market-Based | 投标/竞拍机制 | 解决多 Agent 资源分配和优先级 |

**关键优势**：
- 不可变事件日志提供完整审计轨迹
- Agent 可独立扩缩容
- 故障恢复通过日志回放实现
- 天然适配微服务架构

**适用场景**：
- 高吞吐异步处理（数据 pipeline、告警处理）
- 需要持久化审计的场景（金融、合规）
- Agent 由不同团队独立开发/部署

## 3. 并行化与评估器-优化器

除了上述拓扑模式，Anthropic 还定义了两个跨拓扑的**协作模式**：

### 3.1 并行化（Parallelization）

不属于某种特定拓扑——任何模式中都可以应用。两种变体：

**分段（Sectioning）**：将任务拆为独立子任务并行执行。例如：一个 Agent 处理用户查询，同时另一个 Agent 做内容安全审核。

**投票（Voting）**：对同一任务运行多次（可能用不同 prompt），取多数/最佳结果。例如：3 个 Agent 分别审查代码漏洞，取至少 2 个都标记的结果。

### 3.2 评估器-优化器（Evaluator-Optimizer）

一个 Agent 生成输出，另一个 Agent 评估并反馈，迭代改进直到满足标准。类似人类的"写作→审阅→修改"循环。

**OpenAI Agents SDK 的实现**：`while` 循环中交替运行 generator agent 和 evaluator agent，直到评估通过。

**适用条件**（Anthropic）：
- 有明确的评估标准
- LLM 的输出确实能从反馈中改善
- LLM 本身能产出有意义的反馈

## 4. 模式选择决策矩阵

| 条件 | 推荐模式 | 理由 |
|------|----------|------|
| 输入有明确类别，需分流处理 | **路由** | 最简实现，隔离关注点 |
| 子任务不可预知，需动态拆解 | **编排者-工作者** | 灵活度最高 |
| Agent 数量 >20，跨多个领域 | **层级** | 对数级扩展，上下文隔离 |
| 需要并行探索大问题空间 | **群体** | 无瓶颈，高容错 |
| 高吞吐异步，跨团队独立部署 | **事件驱动** | 解耦度最高，可独立扩缩 |
| 同一任务需多视角/高置信 | **投票并行** | 提升可靠性 |
| 输出需要迭代打磨 | **评估器-优化器** | 适合有明确质量标准的场景 |

**关键洞察**：生产系统几乎都是**混合模式**。例如：层级结构中的底层团队内部用 Mesh 协作，管线式流程中的某一步用 Swarm 做并行搜索。模式是可组合的构件，不是互斥选项。

## 5. 完整代码示例：Orchestrator-Workers + Evaluator 模式

```python
"""
Multi-Agent Orchestrator: Research Report Generator
模式：Orchestrator 动态分派 Worker，最终由 Evaluator 审核质量

架构：
  User → Orchestrator → [Worker_Search, Worker_Analyze, Worker_Write] → Evaluator → Output
"""
import asyncio
from dataclasses import dataclass, field
from enum import Enum
from typing import Any


class AgentRole(Enum):
    ORCHESTRATOR = "orchestrator"
    SEARCHER = "searcher"
    ANALYZER = "analyzer"
    WRITER = "writer"
    EVALUATOR = "evaluator"


@dataclass
class AgentMessage:
    """Agent 间通信的标准消息格式"""
    from_agent: str
    to_agent: str
    content: str
    metadata: dict = field(default_factory=dict)
    task_id: str = ""


@dataclass
class SubTask:
    """编排者分解出的子任务"""
    id: str
    description: str
    assigned_to: AgentRole
    dependencies: list[str] = field(default_factory=list)
    result: str | None = None
    status: str = "pending"  # pending / running / completed / failed


class BaseAgent:
    """Agent 基类：统一接口"""

    def __init__(self, name: str, role: AgentRole, system_prompt: str):
        self.name = name
        self.role = role
        self.system_prompt = system_prompt

    async def process(self, message: AgentMessage) -> AgentMessage:
        raise NotImplementedError


class SearcherAgent(BaseAgent):
    """搜索 Worker：负责信息检索"""

    def __init__(self):
        super().__init__(
            name="searcher",
            role=AgentRole.SEARCHER,
            system_prompt=(
                "You are a research specialist. Given a search query, "
                "find relevant information from available sources. "
                "Return structured findings with source citations."
            ),
        )

    async def process(self, message: AgentMessage) -> AgentMessage:
        # 实际实现：调用搜索 API / RAG / Web Search
        query = message.content
        # 模拟搜索结果
        result = f"[Search results for '{query}']\n- Finding 1: ...\n- Finding 2: ...\n- Source: ..."
        return AgentMessage(
            from_agent=self.name,
            to_agent=message.metadata.get("reply_to", "orchestrator"),
            content=result,
            task_id=message.task_id,
        )


class AnalyzerAgent(BaseAgent):
    """分析 Worker：负责数据分析和推理"""

    def __init__(self):
        super().__init__(
            name="analyzer",
            role=AgentRole.ANALYZER,
            system_prompt=(
                "You are an analytical specialist. Given raw data or findings, "
                "identify patterns, draw insights, and structure conclusions. "
                "Be precise about confidence levels."
            ),
        )

    async def process(self, message: AgentMessage) -> AgentMessage:
        data = message.content
        result = f"[Analysis of provided data]\n- Key insight 1: ...\n- Pattern: ...\n- Confidence: high"
        return AgentMessage(
            from_agent=self.name,
            to_agent=message.metadata.get("reply_to", "orchestrator"),
            content=result,
            task_id=message.task_id,
        )


class WriterAgent(BaseAgent):
    """写作 Worker：负责内容生成"""

    def __init__(self):
        super().__init__(
            name="writer",
            role=AgentRole.WRITER,
            system_prompt=(
                "You are a technical writer. Given research findings and analysis, "
                "produce clear, well-structured content. "
                "Use concrete examples and maintain logical flow."
            ),
        )

    async def process(self, message: AgentMessage) -> AgentMessage:
        brief = message.content
        result = f"[Draft report based on brief]\n## Summary\n...\n## Findings\n...\n## Recommendations\n..."
        return AgentMessage(
            from_agent=self.name,
            to_agent=message.metadata.get("reply_to", "orchestrator"),
            content=result,
            task_id=message.task_id,
        )


class EvaluatorAgent(BaseAgent):
    """评估器：审核输出质量"""

    def __init__(self, criteria: list[str]):
        super().__init__(
            name="evaluator",
            role=AgentRole.EVALUATOR,
            system_prompt=(
                "You are a quality evaluator. Assess the provided content against "
                "specific criteria. Return a pass/fail verdict with actionable feedback."
            ),
        )
        self.criteria = criteria

    async def process(self, message: AgentMessage) -> AgentMessage:
        content = message.content
        # 实际实现：LLM 评估 content 是否满足 self.criteria
        # 模拟评估结果
        feedback = {
            "verdict": "pass",  # or "fail"
            "score": 0.85,
            "feedback": "Good structure. Consider adding more specific data points in section 2.",
            "criteria_met": self.criteria[:3],
            "criteria_missed": [],
        }
        return AgentMessage(
            from_agent=self.name,
            to_agent="orchestrator",
            content=str(feedback),
            metadata={"verdict": feedback["verdict"], "score": feedback["score"]},
            task_id=message.task_id,
        )


class OrchestratorAgent:
    """
    编排者：动态任务分解 + Worker 调度 + 评估循环

    流程：
    1. 接收用户请求
    2. 分解为子任务（动态，基于 LLM 规划）
    3. 按依赖关系调度 Worker（无依赖的并行执行）
    4. 收集结果并综合
    5. 提交 Evaluator 审核
    6. 如果不通过，将反馈交给 Writer 修改（最多 3 轮）
    """

    def __init__(self):
        self.workers: dict[AgentRole, BaseAgent] = {
            AgentRole.SEARCHER: SearcherAgent(),
            AgentRole.ANALYZER: AnalyzerAgent(),
            AgentRole.WRITER: WriterAgent(),
        }
        self.evaluator = EvaluatorAgent(criteria=[
            "Factual accuracy with citations",
            "Clear logical structure",
            "Actionable conclusions",
            "Appropriate depth for target audience",
        ])
        self.max_revision_rounds = 3

    async def plan(self, user_request: str) -> list[SubTask]:
        """
        任务规划：将用户请求分解为子任务序列
        实际实现应调用 LLM 做动态规划
        """
        # 模拟规划结果
        return [
            SubTask(id="t1", description=f"Search for: {user_request}", assigned_to=AgentRole.SEARCHER),
            SubTask(id="t2", description="Analyze search findings", assigned_to=AgentRole.ANALYZER, dependencies=["t1"]),
            SubTask(id="t3", description="Write report from analysis", assigned_to=AgentRole.WRITER, dependencies=["t2"]),
        ]

    async def dispatch(self, task: SubTask, context: dict[str, str]) -> str:
        """分派子任务到对应 Worker"""
        worker = self.workers[task.assigned_to]

        # 构建消息，注入依赖任务的结果作为上下文
        dep_context = "\n---\n".join(
            f"[{dep_id} result]: {context[dep_id]}" for dep_id in task.dependencies if dep_id in context
        )
        content = f"{task.description}\n\nContext:\n{dep_context}" if dep_context else task.description

        message = AgentMessage(
            from_agent="orchestrator",
            to_agent=worker.name,
            content=content,
            metadata={"reply_to": "orchestrator"},
            task_id=task.id,
        )

        response = await worker.process(message)
        return response.content

    async def run(self, user_request: str) -> str:
        """执行完整编排流程"""
        print(f"[Orchestrator] Received request: {user_request[:80]}...")

        # Step 1: 规划
        tasks = await self.plan(user_request)
        print(f"[Orchestrator] Planned {len(tasks)} subtasks")

        # Step 2: 按依赖顺序执行（无依赖的并行）
        context: dict[str, str] = {}
        completed = set()

        while len(completed) < len(tasks):
            # 找出可执行的任务（依赖已满足）
            runnable = [
                t for t in tasks
                if t.id not in completed and all(d in completed for d in t.dependencies)
            ]

            if not runnable:
                raise RuntimeError("Deadlock: no runnable tasks but not all completed")

            # 并行执行所有可运行任务
            results = await asyncio.gather(*[self.dispatch(t, context) for t in runnable])

            for task, result in zip(runnable, results):
                context[task.id] = result
                completed.add(task.id)
                print(f"[Orchestrator] Completed subtask {task.id} ({task.assigned_to.value})")

        # Step 3: 获取最终输出（最后一个任务的结果）
        final_output = context[tasks[-1].id]

        # Step 4: 评估循环
        for round_num in range(1, self.max_revision_rounds + 1):
            print(f"[Orchestrator] Evaluation round {round_num}")

            eval_message = AgentMessage(
                from_agent="orchestrator",
                to_agent="evaluator",
                content=final_output,
                task_id="eval",
            )
            eval_response = await self.evaluator.process(eval_message)

            if eval_response.metadata.get("verdict") == "pass":
                print(f"[Orchestrator] Evaluation passed (score: {eval_response.metadata.get('score')})")
                break
            else:
                print(f"[Orchestrator] Revision needed: {eval_response.content[:100]}")
                # 将评估反馈交给 Writer 修改
                revision_message = AgentMessage(
                    from_agent="orchestrator",
                    to_agent="writer",
                    content=f"Please revise based on feedback:\n{eval_response.content}\n\nOriginal:\n{final_output}",
                    metadata={"reply_to": "orchestrator"},
                    task_id="revision",
                )
                revision_response = await self.workers[AgentRole.WRITER].process(revision_message)
                final_output = revision_response.content

        return final_output


# ─── 运行 ──────────────────────────────────────────────────────

async def main():
    orchestrator = OrchestratorAgent()
    result = await orchestrator.run(
        "Research the current state of AI agent frameworks in 2026, "
        "comparing LangGraph, CrewAI, and OpenAI Agents SDK. "
        "Focus on production readiness and multi-agent support."
    )
    print(f"\n{'='*60}\n[Final Output]\n{result}")


if __name__ == "__main__":
    asyncio.run(main())
```

## 6. 反模式与陷阱

### 6.1 "一个 Agent 所有工具"

把 30 个工具全给一个 Agent，依赖它自己选对——这是最常见的反模式。MLflow 研究指出：超过 8 个工具时，选择准确率开始显著下降。

### 6.2 过早引入 Multi-Agent

只有 3 个工具、prompt 2000 tokens，不存在场景冲突——强行拆为 Multi-Agent 只是增加了延迟和调试难度。

### 6.3 Supervisor 的"电话游戏"

LangChain 基准测试揭示：Supervisor 模式的主要性能损失不在 Worker 质量，而在 Supervisor 转述 Worker 结果时的信息丢失。解法：让 Supervisor 直接转发原始消息，或只做路由不做改写。

### 6.4 无终止条件的 Swarm

去中心化 Agent 系统如果没有显式的收敛判定（max iterations / quality threshold / timeout / token budget），会陷入无限循环。这不是"可能"，而是"一定会"。

### 6.5 忽略延迟累加

3 层层级 × 每层 2 次 LLM 调用 × 每次 2s = 最低 12s 响应时间。用户容忍度有限。设计时必须考虑是否真的需要每一层。
