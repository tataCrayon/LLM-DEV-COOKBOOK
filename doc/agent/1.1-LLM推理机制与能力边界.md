# 1.1 LLM 推理机制与能力边界——Agent 工程师的物理基础

> **为什么从这里开始？**  
> Agent 工程师不需要会训练模型，但必须理解模型推理时**到底在算什么**。否则你写的 prompt 会无意中触发性能悬崖、设计的工具调用会撞上注意力衰减、调出的"长上下文 Agent"会在中间段失忆。本篇从 Transformer 推理过程开始，把一系列反直觉的工程现象讲清楚——它们都不是模型 bug，而是架构的物理后果。

---

## 0. 一个让人警觉的现象

把同一段 30 KB 的代码塞给 GPT-4 / Claude，让它"找出第 17 个函数的第 3 个参数的默认值"。如果你把这段代码放在 prompt 的开头或结尾，它几乎都能答对；但如果你把它埋在 prompt 的中间——同样的内容、同样的模型、同样的问题——准确率会断崖式下降。

这不是模型"懒"，而是 Transformer 注意力机制的**物理性质**。理解这一类现象，是 Agent 工程的起点。本篇要回答四组核心问题：

1. 推理时模型到底在算什么？为什么 prompt 越长越贵？
2. 上下文窗口"看起来越来越长"，为什么用起来还是不靠谱？
3. 幻觉到底从哪里来？为什么"越聪明的模型越爱猜"？
4. 模型能力的硬边界在哪里？什么任务再调 prompt 也救不了？

## 1. 推理过程：Prefill 与 Decode 的两阶段世界

### 1.1 一次模型调用的内部时间线

当你调一次 `client.chat.completions.create(...)`，模型内部会经历两个截然不同的阶段：

```
                ┌─────────────────────────┐
                │  Prefill 阶段（一次性） │
prompt(N tokens)│  把 prompt 全部读进去   │  
─────────────►  │  并行计算 N 个位置的    │
                │  attention             │
                │  ↓                     │
                │  生成 KV Cache         │
                └─────────────────────────┘
                            │
                            ▼
                ┌─────────────────────────┐
                │  Decode 阶段（逐 token）│
                │  每次只读 1 个新 token  │   生成
                │  + 复用之前所有 KV     │ ──► output
                │  生成下一个 token      │   (M tokens)
                │  循环 M 次             │
                └─────────────────────────┘
```

这两个阶段的资源画像完全不同：

| 维度 | Prefill | Decode |
|---|---|---|
| 触发次数 | 1 次 | M 次（输出 token 数） |
| 单步处理 token 数 | N（prompt 长度） | 1 |
| 计算特性 | 高度并行，**算力受限**（compute-bound） | 串行，**显存带宽受限**（memory-bound） |
| GPU 利用率 | 高（接近峰值 FLOPS） | 低（10%-30% 常见） |
| 优化技巧 | Flash Attention、batch | KV cache、speculative decoding、量化 |

**Agent 设计层面的直接启示**：

- **Prompt 长度的代价不是均匀的**。Prefill 是一次性 O(N²) 的注意力代价（朴素实现），但能在大批 GPU 算力上并行摊薄；真正贵的是 decode 阶段——每输出一个 token 都要把整个 KV cache 从显存搬一遍。所以"输入 100K + 输出 100 token"远比"输入 1K + 输出 10K"便宜。
- **Agent 的 ReAct 循环天然落在 decode 慢路径上**。每一步 thought-action-observation 都涉及多轮短输出，且每次都要带着累积越来越长的历史重新跑 prefill。这就是为什么"工具调用密集型 Agent"延迟感受比"单轮 RAG"差很多。

### 1.2 KV Cache：让 Decode 不至于退化成 O(N²·M)

如果没有 KV cache，每生成一个 token 都要重新算一遍前面所有 token 的 K（key）和 V（value）。引入 KV cache 后，decode 每一步只需要算新 token 的 K/V，并把它**追加**到 cache 上。复杂度从 O(N²·M) 降到 O(N·M)。

但 KV cache 不是免费的，它是**显存的主要消耗者**：

```
KV Cache 大小（单 sequence）
= 2 × num_layers × num_heads × head_dim × seq_len × dtype_bytes
= 2 × num_layers × hidden_dim × seq_len × dtype_bytes
```

以 Llama 3 70B（80 layers, hidden 8192, FP16）为例，**128K 上下文**的 KV cache 单卡需要：

```
2 × 80 × 8192 × 128_000 × 2 bytes ≈ 335 GB
```

一张 H100 才 80 GB。这就是为什么"超长上下文"在工程上需要 PagedAttention（vLLM 提出）、KV cache 分页、跨 GPU 切分、量化（KV Cache INT8/INT4）等一系列技巧才能做出来。

**Agent 工程师的对应直觉**：

- **Prompt Caching 是真金白银**。Anthropic、OpenAI、阿里云都推出了 prompt caching 功能。它的本质就是把 prefill 阶段的 KV cache 在多轮调用之间复用——相同前缀只算一次，后续命中按 ~10% 价格收费。Agent 系统里**把固定不变的部分（system prompt、工具描述、project context）放在最前面**，能直接砍掉一大块成本。
- **KV Cache 是有效上下文的物理上限**。模型说"支持 1M 上下文"和"用 1M 上下文还便宜"是两回事。

### 1.3 一个数量级感

来看一组对 Agent 工程师有用的近似数字（以 H100 + Llama 3 70B 为例）：

| 操作 | 时间量级 |
|---|---|
| Prefill 1K tokens | ~50ms |
| Prefill 100K tokens | ~5s |
| Decode 1 token | ~20-30ms |
| 网络往返（同 region） | ~10ms |
| 工具调用执行（如 grep） | 10ms - 数秒 |

如果一个 Agent 任务需要 20 步 ReAct，每步 prompt 5K tokens、输出 200 tokens：
- Prefill 总计 ≈ 20 × 250ms = 5s
- Decode 总计 ≈ 20 × 200 × 25ms = 100s
- **延迟 95% 都在 decode 上**

这就是为什么所有 Agent 框架都在追求"减少步数"和"streaming reaction"——延迟的本质矛盾在 decode。

## 2. 注意力机制与上下文的反直觉现象

### 2.1 注意力的物理本质

回到 attention 公式（忽略多头）：

```
Attention(Q, K, V) = softmax(QK^T / √d_k) · V
```

对每个位置 i，模型计算一个长度为 N 的权重向量 `softmax(q_i · K^T)`，然后对所有位置的 V 加权求和。这个权重向量是一个**概率分布**——所有位置加起来等于 1。

这有两个直接的工程后果：

**后果一：注意力是被稀释的**。N=2K 时，平均每个位置分到的权重是 1/2000；N=128K 时降到 1/128000。当目标信息埋在 100K tokens 中间，它需要**显著高于平均**的相关性才能浮上来——而这恰恰是模型在长距离上最弱的能力。

**后果二：位置信息靠位置编码（RoPE/ALiBi）注入**。这些编码在训练分布外的位置上会衰减。这就是"长上下文模型"扩展到训练长度 4× 之后性能骤降的根源。

### 2.2 Lost in the Middle：实证现象

Liu et al. 2023 在 *Lost in the Middle: How Language Models Use Long Contexts* 中给出了一个里程碑实验：把"包含答案的文档"放在 20 个干扰文档组成的列表中，依次让答案文档出现在第 1、5、10、15、20 个位置，测准确率。

**结果是经典的 U 形曲线**：开头和结尾位置准确率最高，中间位置准确率最低，且降幅可以达到 20+ 个百分点。论文还发现：**即便是声称"支持长上下文"的模型，这个 U 形也不会消失**。

这个发现至今仍是 Agent / RAG 工程的核心约束。后续工作（如 NeurIPS 2024 的 *Found in the Middle*）从位置编码、训练数据分布等角度给出了部分缓解方法，但**根本性的位置偏置依然存在**。

### 2.3 Agent 工程的对应工程化

把上述物理现象翻译成可执行规则：

1. **关键信息靠近两端**。RAG 召回结果 reranking 时，把最相关的放最前或最后；system prompt 中"绝对约束"放最末（"永远不要执行 rm -rf"）。
2. **每轮压缩 vs 全量带回历史**。长任务里的中段历史会变成"地铁中间座位"——记不住但占空间。Compaction（上下文压缩）不是性能优化技巧，是正确性保障。
3. **结构化好过自由文本**。XML/JSON 分隔的结构让模型用"结构定位"代替"语义定位"，可以部分绕开位置偏置。Anthropic 长期推荐 `<context>...</context>` 风格不是审美，是物理需求。
4. **工具描述放在系统提示固定位置**。用 Anthropic / OpenAI 的 prompt cache，工具描述固定在最前；让模型反复看到相同位置的同样信息，等于用频率对抗位置衰减。

```python
# 反例：把所有东西平铺
prompt = f"用户问题：{q}\n\n相关文档：{doc1}\n{doc2}\n...\n{doc20}\n\n回答："

# 正例：结构化 + 关键信息上下夹击
prompt = f"""<task>{q}</task>

<retrieved_context>
<doc rank="1">{best_doc}</doc>     # 召回最相关的放最前
<doc rank="2">{second_best}</doc>
...
<doc rank="20">{least_relevant}</doc>
</retrieved_context>

<task_repeat>{q}</task_repeat>      # 在最后再问一遍

请基于 retrieved_context 回答 task。"""
```

### 2.4 上下文不是"越长越好"，而是"匹配模型的有效注意力分布"

行业术语 "**effective context length**"（有效上下文长度）专门描述这个现象：模型宣称支持 128K，但在某些任务上 32K 之后准确率就开始掉。NIAH（Needle in a Haystack）测试是行业事实标准——在长文中插入一句"特殊事实"，看模型能不能召回。

实践经验：

- **GPT-4 Turbo / GPT-4o**：声明 128K，实测 32-64K 是甜蜜点，超过 100K 性能可见下降。
- **Claude 3.5 Sonnet / Opus**：声明 200K，是当前 effective context 最长的模型之一，但仍存在中间衰减。
- **Gemini 1.5 Pro**：声明 1M-2M，NIAH 表现接近完美，但复杂推理在长上下文下仍弱化。

**Agent 设计准则**：你设计的 Agent 在生产环境会随着对话/任务推进上下文越来越长。**永远要有一个 "compaction trigger"**——超过某阈值（比如 effective context 的 60%）就触发摘要压缩，而不是依赖模型"自己处理"。

## 3. 幻觉的根因：从训练目标到评估激励

### 3.1 一个被误解了多年的问题

"幻觉"这个词在工业界被反复讨论，但很多人对其根因的理解停留在"训练数据有错"或"模型不够大"。OpenAI 2025 年 9 月的论文 *Why Language Models Hallucinate*（Kalai, Nachum, Vempala et al.）给出了一个更本质的视角，值得 Agent 工程师好好读一遍。

### 3.2 论文核心论点

> **"Language models hallucinate because the training and evaluation procedures reward guessing over acknowledging uncertainty."**

幻觉不是 bug，是**激励机制下的最优策略**。这个论点分两层：

**预训练层面：交叉熵损失天然产生错误**

论文给出了一个新颖的归约：把生成问题 reduce 到二分类问题，证明：
```
generative_error_rate ≳ 2 × IIV_misclassification_rate
```

具体的不可避免错误来源：
- **单例事实（singletons）**：训练数据中只出现过一次的事实（比如某个人的生日），如果占比 20%，base model 至少在 20% 的此类问题上会幻觉。这是数学上的下界，不是训练技巧能解决的。
- **表征瓶颈**：tokenizer 把字符级结构隐藏了。让模型数 "DEEPSEEK" 里有几个 D，它看到的是 token id `[12345, 67890]` 而不是字符序列——天然算不准。
- **GIGO**：训练数据里的错误会被学进去。

**后训练层面：评估激励错位**

这是论文最尖锐的部分。作者审视了主流 benchmark（MMLU、GPQA、SWE-bench、Humanity's Last Exam 等），发现几乎全部使用 0/1 二值评分：答对 1 分，答错或"我不知道"都是 0 分。

在这种评分体系下：
- 一个老老实实说"不确定"的模型 → 平均得分低
- 一个永远自信乱猜的模型 → 平均得分高（因为偶然蒙对会加分）

> **"A model that honestly signals uncertainty will lose to one that always guesses."**

只要 leaderboard 还按 0/1 算，所有 lab 就都会训出爱猜的模型。这是一个**竞争压力下的纳什均衡**，不是技术问题。

### 3.3 OpenAI 给的处方

不要再造新的 hallucination benchmark（被主流 benchmark 淹没），而是改造主流 benchmark 本身：

1. **显式置信度阈值**：在 prompt 里加 "answer only if you're > t confident; wrong = -t/(1-t) points; correct = 1; IDK = 0"。
2. **行为校准**：要求模型给出"在置信度 ≥ t 下最有用的回答"，可以在多个 t 上 audit。
3. **集成进既有 leaderboard**：直接改 SWE-bench、MMLU 的评分规则。

### 3.4 Agent 工程师的实操层面

幻觉对 Agent 系统影响巨大——一次工具调用幻觉就可能 `rm -rf`、付错钱、发错邮件。结合论文论点，工程层面有几条铁律：

1. **永远给"我不知道"留一个出口**。在 Agent 的 system prompt 里明确：
   ```
   If uncertain, output {"action": "ask_user", "question": "..."}
   instead of guessing. Guessing wrong has higher cost than asking.
   ```
2. **Tool call 必须强 schema**。函数签名 + 类型 + enum，让模型只能在合法集合里选；非法字段直接 reject 重生成。这等于把"猜"的成本变成无穷大。
3. **High-stakes 操作要 confidence gate**。删除文件、转账、删数据库前，让另一个模型/工具复核——本质上是用集成对抗个体的 reward hacking。
4. **Context Engineering 是抗幻觉首选**。模型在没有事实时只能编，但在有事实时**会优先用**。RAG 不是为了让模型"懂更多"，而是把"不该编的部分"从必须猜变成可以查。

```python
# 反例：让模型自由发挥
def answer(question):
    return llm(f"Answer: {question}")

# 正例：上下文 + 拒绝兜底 + 引用要求
def answer(question, context_docs):
    return llm(f"""<context>
{format_docs(context_docs)}
</context>

<question>{question}</question>

Rules:
1. Answer ONLY based on <context>. Cite the doc id.
2. If <context> is insufficient, output exactly: {{"answer": null, "reason": "insufficient context"}}
3. Do not guess. Wrong guess > "I don't know" in our eval.
""")
```

## 4. 模型能力的硬边界：什么任务调 prompt 也救不了

### 4.1 三类硬边界

不是所有问题都能靠 prompt engineering 解决。识别"模型注定做不好"的任务，是 Agent 架构师的关键技能。常见硬边界：

**边界一：精确计数与字符级操作**

由于 BPE tokenizer，"数 D"、"反转字符串"、"数 r 在 strawberry 中出现几次"这种任务对模型是不公平的——它压根没看见字符。Agent 工程对策：**遇到此类任务直接调 Python**。Code Interpreter / function calling 是补丁，不是绕路。

**边界二：可靠的多步算术与精确数值推理**

模型在 token 空间做"近似算术"，长链路计算误差累积。GPT-4o 算 7 位数乘法准确率不到 50%。Agent 对策：**所有数值计算外包给计算器/SQL/代码**。

**边界三：跨长上下文的精确状态跟踪**

例如让模型管理一个 50 步对话中的"用户偏好状态"，每步都准确。注意力 + 位置偏置注定它会在中段失忆。对策：**外部状态文件（State Outside Context Pattern）**，让模型每步读写 `state.json` 或数据库，而非把状态留在 chat history。

### 4.2 一个判别框架：四个问题决定能否用 LLM

设计 Agent 时对每个子任务问四个问题：

1. **Verifiable cheaply?**（结果能不能被便宜地验证）—— 能 → LLM 可用，不能 → 谨慎。
2. **Tolerant to occasional error?**（偶尔错一下能否承受）—— 能 → LLM 自身可处理；不能 → 必须配 sensor/guard。
3. **Length of reasoning chain?**（推理链有多长）—— 短 → 直接 LLM；长 → 拆成多步 + 中间持久化。
4. **Determinism required?**（是否需要可重复结果）—— 是 → 用 temperature=0 + 强 schema + 工具兜底；否 → LLM 创造性输出可接受。

这套问题决定了 Agent 应该把"哪些环节交给模型"、"哪些必须留给确定性代码"——也就是 Harness Engineering 里 *Computational vs Inferential* 取舍的物理依据。

### 4.3 Capability Frontier Watch：保持对边界的更新

模型能力边界是会随着模型升级而**移动**的。架构师必须持续跟踪：

- **OpenAI / Anthropic / Google 的能力发布说明**：每次新模型发布关注它在 *frontier benchmarks*（GPQA Diamond, FrontierMath, ARC-AGI, SWE-bench Verified）上的提升。
- **METR Task Suite**：第三方测评 Agent 长时任务能力，每一代模型上去几个百分点。
- **能力突变**：某些能力（如 GPT-4 的 in-context learning）会在某代模型突然出现。这通常意味着可以重新设计原本只能"workaround"的部分。

**架构原则**：Agent 系统应该让"模型能力"成为**可替换的依赖**——通过抽象层（liteLLM、langchain LLM 接口）解耦，使新模型一上线就能 A/B 测试，而不需要改业务代码。

## 5. 把这一切收束到 Agent 工程的设计准则

把前面四节的物理现象、激励现象、能力边界统一翻译成 Agent 架构师可以挂在墙上的工程准则：

| 物理/认知现象 | 设计准则 |
|---|---|
| Decode 是 memory-bound 慢路径 | 减少 ReAct 步数；尽量 streaming；并行化独立子任务 |
| Prefill 受益于 prompt cache | 固定前缀（system + tools + project context）放最前 |
| KV cache 是显存主消耗 | Compaction 是必备，不是优化项 |
| 注意力 U 形偏置 | 关键信息夹两端；结构化分隔；高优结果置顶/置底 |
| Effective context < advertised | 设 trigger，到 60% 阈值压缩 |
| 0/1 评估激励猜测 | system prompt 显式给"不知道"出口；tool 强 schema |
| 单例事实必然幻觉 | RAG 提供事实；要求引用来源；high-stakes 加 verifier |
| 字符级/数值计算硬边界 | 调 Python/SQL/计算器，不要让模型算 |
| 长链路状态跟踪不可靠 | State Outside Context：外部 state 文件 |
| 模型边界会移动 | 抽象 LLM 层，让模型可替换 |

## 6. 一个完整的最小 Agent 调用样例（体现以上原则）

下面是一个生产级最小 Agent 调用片段（伪代码风格，重在体现原则）：

```python
from anthropic import Anthropic

client = Anthropic()

# 原则1: 固定前缀放最前，命中 prompt cache
SYSTEM_PROMPT = """You are a code analysis agent.

<rules>
1. Use tools for all file operations - never guess file content.
2. If you cannot determine the answer with high confidence, call ask_user tool.
3. For arithmetic / counting, always call run_python tool.
4. Cite file paths and line numbers in your answer.
</rules>

<tools_description>
... (long tool descriptions with examples)
</tools_description>
"""  # 这一整块缓存复用

# 原则2: 强 schema 工具定义
TOOLS = [
    {
        "name": "read_file",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"],
        },
    },
    {
        "name": "ask_user",  # 给 "不知道" 留出口
        "input_schema": {
            "type": "object",
            "properties": {"question": {"type": "string"}},
            "required": ["question"],
        },
    },
    {
        "name": "run_python",  # 数值/字符级计算外包
        "input_schema": {
            "type": "object",
            "properties": {"code": {"type": "string"}},
            "required": ["code"],
        },
    },
]

def run_agent(user_task, context_files):
    history = []
    step = 0
    
    while step < MAX_STEPS:  # 原则3: 步数预算
        # 原则4: 关键信息夹两端
        messages = [
            {"role": "user", "content": f"<task>{user_task}</task>"},
            *history,
            # 原则5: 上下文超 60% 阈值就压缩
            *(maybe_compact(history) if estimate_tokens(history) > 0.6 * CTX_LIMIT else []),
            {"role": "user", "content": f"<task_reminder>{user_task}</task_reminder>"},
        ]
        
        response = client.messages.create(
            model="claude-sonnet-4-5",
            system=[
                {"type": "text", "text": SYSTEM_PROMPT, "cache_control": {"type": "ephemeral"}}  # 启用缓存
            ],
            tools=TOOLS,
            messages=messages,
            max_tokens=2048,
        )
        
        if response.stop_reason == "end_turn":
            return response.content
        
        # 工具调用 → 执行 → 把结果加回 history
        for block in response.content:
            if block.type == "tool_use":
                tool_result = execute_tool(block.name, block.input)
                history.append({"role": "assistant", "content": response.content})
                history.append({
                    "role": "user",
                    "content": [{
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": tool_result,
                    }]
                })
        
        step += 1
    
    raise BudgetExceeded(f"Agent did not finish in {MAX_STEPS} steps")
```

这段代码里**每一行设计**都对应前面讨论过的物理/认知现象。它不是"最完美的 Agent"，而是一个最小但**没有违反任何物理规律**的 Agent。

## 7. 小结与下一篇的衔接

Agent 工程的所有"奇怪经验"，都能从这一篇里讲到的物理基础推出来：

- 为什么 prompt cache 这么重要 → Prefill 太贵
- 为什么 ReAct Agent 慢 → Decode 是 memory-bound
- 为什么 RAG 召回要 reranking → Lost in the Middle
- 为什么必须有 compaction → KV cache 显存上限
- 为什么"幻觉问题"治不好 → 评估激励错位
- 为什么强 schema 工具调用比自由文本好 → 把"猜"的成本变无穷大
- 为什么数值计算必须调 Python → tokenizer 物理边界

下一篇 **1.2 Prompt Engineering 系统化** 会承接这一篇——把"如何利用这些物理性质把模型用好"的具体技术（CoT、few-shot、self-consistency、structured output）串成一条决策链，而不是零散的"小技巧合集"。
