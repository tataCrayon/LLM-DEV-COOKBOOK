# Workspace 四原语与预算策略：让长跑 Agent 不出轨

> Loop Engineering 的 5+1 building blocks 解决了"如何让 Agent 持续跑"，但没解决"如何让 Agent 跑得起、停得下、改得回、可追溯"。本文聚焦 Workspace（第六个 Block）的四个核心原语，以及围绕它们建立的预算与终止策略——这是从"demo 式 loop"升级到"production-ready loop"的关键缺口。

## 0. 开场问题

你的团队部署了一个夜跑 Agent：每天凌晨自动扫描代码仓库，发现可优化点后提交 MR。运行一周后，你发现：

1. **预算失控**：一次扫描花了 47 美元（Revenium 案例数据），Agent 在一个根本不应该被它处理的目录里循环了 6 小时
2. **改动无人能审**：Agent 提了 23 个 MR，每个改动跨越多个模块，code reviewer 无法在合理时间内审完，开始机械点 approve
3. **故障不可回滚**：Agent 改坏了一段缓存逻辑，事故 4 小时后才被发现——回溯时发现日志里只有"AI 提交"，无法知道是哪次 loop、用了什么上下文、为什么改这里
4. **重启就丢状态**：Agent 跑到一半 OOM，重启后不知道已经处理过哪些文件，从头开始
5. **Agent 越权**：本应只读的目录里出现了写操作——因为权限是"整个仓库可读可写"

这些问题的共同点：**它们都不是模型问题**。模型该懂的它都懂——是 Workspace 没建好。

Workspace 不是"Agent 的工作目录"那么简单。它是一组让长跑 Agent 在生产环境里**可控、可审、可恢复**的原语。Addy Osmani（2025）将其总结为 Loop Engineering 第六个缺失的 Block：

> "Memory 回答'Agent 会不会记得'，Workspace 回答'Agent 能不能在受治理的边界内行动'。Workspace ≠ Memory，它增加 permission、attribution、reversibility、structure。"

## 1. Workspace 的四个核心原语

从生产实践提炼，一个完整的 Agent Workspace 由四个原语构成：

| 原语 | 解决什么问题 | 典型实现 |
|---|---|---|
| **Identity & Scope（身份与边界）** | Agent 是谁？能动哪里？ | OAuth Service Account、Scoped Token、Path Allowlist |
| **State & Persistence（状态与持久化）** | 中断后能恢复到哪？ | Event Log、Snapshot、Checkpoint、TODO.md |
| **Write Surface（治理化写入面）** | 改动如何审计、回滚？ | Shadow Git、PR-only Write、Reversible Diff |
| **Budget Envelope（预算包络）** | 何时必须停？ | Token Budget、Tool Call Budget、Wall-clock、Cost Cap |

这四个原语相互独立，但只有四个齐备，Workspace 才是"production-ready"的。下面逐一拆解。

## 2. 原语一：Identity & Scope

### 2.1 为什么需要

传统观念里，"Agent" 是一个抽象主体。但生产环境必须问清楚：**这个 Agent 是用谁的身份在动手？**

考虑两种灾难场景：

- 场景 A：Agent 用 root 凭证操作 Kubernetes，幻觉删了生产 namespace
- 场景 B：Agent 用某个工程师的 GitHub token 提交代码，几个月后该工程师离职，token 被吊销，所有 Agent 历史 commit 失去作者归属

两个问题的根因都是 **Agent 没有自己的身份**——它"借用"了人类或系统的身份。

### 2.2 设计模式

生产级 Agent 必须满足：

```
每个 Agent（或子 Agent）= 1 个独立身份 + 1 个 scoped credential 集合
```

三层 scope 设计：

| 层 | 控制什么 | 示例 |
|---|---|---|
| Path Scope | 能读/写哪些路径 | `read: /repo/**`, `write: /worktrees/<agent_id>/**` |
| Tool Scope | 能调哪些工具 | `allow: [git_commit, run_tests]`, `deny: [kubectl_delete, prod_deploy]` |
| Resource Scope | 能消费多少资源 | `max_tokens: 1M/day`, `max_cost: $50/run`, `max_wall_clock: 4h` |

每次 Agent 行动都必须**显式校验** scope，违规直接拒绝执行——而不是事后审计。

### 2.3 命名与归因

每个 Agent 身份的 commit、API 调用、消息都必须带上**机器可解析的归因**：

- `Author: agent:nightly-refactor-bot@v1.3 (parent: human:alice)`
- 包含 Agent 类型、版本、触发者
- 故障调查时可一键过滤"哪个 Agent、哪个版本、哪个 run 改的"

Anthropic（2025）在内部 Multi-Agent Research System 中采用类似方案：每个 Subagent 拥有独立 trace ID，全链路日志可重建任何一次 run 的完整决策路径。

## 3. 原语二：State & Persistence

### 3.1 为什么需要

LLM 是无状态的——上下文窗口一关，所有"思考"消失。长跑 Agent 必须把状态存到**外部**，否则任何中断都意味着从头开始。

### 3.2 状态分层

不同状态有不同的持久化策略：

| 状态类型 | 频率 | 持久化方式 | 恢复语义 |
|---|---|---|---|
| 工作记忆（Working Memory） | 每个 LLM 调用 | Context Window | 重启后丢失 |
| 任务状态（Task State） | 每个工具调用后 | TODO.md / 任务表 | 重启后从断点恢复 |
| 决策日志（Event Log） | 每个决策点 | 不可变 Append-Only Log | 重启后可完整回放 |
| 检查点（Checkpoint） | 关键里程碑 | 快照 + 元数据 | 重启可回退到任意 checkpoint |
| 长期知识（Knowledge） | 跨 run | 向量库 / 知识图谱 | 永久保留 |

**关键原则**：**TODO.md 是长跑 Agent 最强的状态原语**。它满足三个属性：

1. **人机共读**：人类可以直接看到 Agent 当前进度
2. **断点续跑**：Agent 重启后读 TODO.md 即可知道下一步
3. **可手工干预**：人可以编辑 TODO.md 来引导 Agent

Claude Code 的 `CLAUDE.md` + 内置 TodoList、Cursor 的 `.cursorrules`、OpenAI Codex 的 `/goal` 状态——本质都是这个原语的不同实现。

### 3.3 Event Sourcing 模式

对于必须满足合规审计的场景（金融、医疗），State 必须用 **Event Sourcing** 实现：

```
所有 Agent 决策 → Append-Only Event Log → 可通过 replay 重建任意时刻状态
```

Event Log 的特性：

- 不可变：写入后不能修改（防止事后篡改）
- 全序：每个事件有严格时间戳和序号
- 自包含：每个事件包含完整上下文（输入、决策、输出）
- 可 replay：从任意 event 开始可以重建后续状态

这是 Anthropic（2025）建议长跑 Agent "resume from failure points rather than restarting" 的实现基础。

## 4. 原语三：Write Surface（治理化写入面）

### 4.1 为什么需要

让 Agent 直接 `git commit -m "..." && git push` 是 demo 友好的设计，是生产灾难。一旦发生：

- 改动 review 不充分
- 故障归因困难
- 回滚需要手动 revert

正确做法是把所有写操作收敛到一个**受治理的写入面**。

### 4.2 三种治理化写入模式

| 模式 | 实现 | 适用场景 |
|---|---|---|
| **Shadow Workspace** | Agent 在隔离副本（worktree / branch / sandbox）中工作，最终通过 PR 提交 | 代码改动、配置变更 |
| **Staged Write + Approval** | Agent 写入"待审"区，人工或规则引擎批准后才生效 | 数据库变更、生产部署 |
| **Reversible Action** | 所有写操作必须配套 undo 操作 | 客户工单、订单状态 |

**Shadow Git** 是 Coding Agent 的标准做法。arXiv 2604.03515（2025）分析的 13 个 Coding Agent 中，11 个采用某种形式的 shadow workspace——Devin 用 Firecracker microVM、Claude Code 用 Docker + git worktree、Codex 用临时 branch。

### 4.3 写入策略的代码实现

```python
"""
Workspace 治理化写入面：所有 Agent 写操作必须通过 WriteSurface
"""
from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import Callable, List, Optional
import uuid

class WriteStatus(Enum):
    PENDING = "pending"
    APPROVED = "approved"
    REJECTED = "rejected"
    APPLIED = "applied"
    ROLLED_BACK = "rolled_back"

@dataclass
class WriteIntent:
    """写入意图：Agent 想做什么改动，但还没真做"""
    id: str
    agent_identity: str
    target_path: str
    operation: str  # "create" | "update" | "delete"
    diff: str
    rationale: str  # Agent 为什么要做这个改动
    rollback_action: Callable[[], None]  # 必须提供回滚函数
    created_at: datetime = field(default_factory=datetime.utcnow)
    status: WriteStatus = WriteStatus.PENDING


class PolicyViolation(Exception):
    pass


class WriteSurface:
    """治理化写入面：拦截所有写操作，应用策略，记录审计"""
    
    def __init__(self, identity: str, scope: dict):
        self.identity = identity
        self.scope = scope  # {"allowed_paths": [...], "max_diff_lines": N, ...}
        self.intents: List[WriteIntent] = []
        self.event_log: List[dict] = []
    
    def propose_write(self, target_path: str, operation: str, 
                      diff: str, rationale: str, 
                      rollback: Callable) -> WriteIntent:
        """Agent 不直接写——只能 propose"""
        # 1. 边界检查
        if not any(target_path.startswith(p) for p in self.scope["allowed_paths"]):
            raise PolicyViolation(
                f"{self.identity} attempted write outside scope: {target_path}"
            )
        
        # 2. 大小检查
        diff_lines = diff.count("\n")
        if diff_lines > self.scope.get("max_diff_lines", 500):
            raise PolicyViolation(
                f"Diff too large ({diff_lines} lines), exceeds max"
            )
        
        # 3. 危险操作检查
        if operation == "delete" and not self.scope.get("can_delete", False):
            raise PolicyViolation(f"{self.identity} cannot delete files")
        
        intent = WriteIntent(
            id=str(uuid.uuid4()),
            agent_identity=self.identity,
            target_path=target_path,
            operation=operation,
            diff=diff,
            rationale=rationale,
            rollback_action=rollback,
        )
        self.intents.append(intent)
        self._log_event("propose_write", intent)
        return intent
    
    def approve(self, intent_id: str, approver: str, 
                checks_passed: List[str]) -> None:
        """规则引擎或人工批准"""
        intent = self._find(intent_id)
        if not checks_passed:
            raise ValueError("Cannot approve without passed checks")
        intent.status = WriteStatus.APPROVED
        self._log_event("approve", intent, approver=approver, 
                        checks=checks_passed)
    
    def apply(self, intent_id: str, executor: Callable[[WriteIntent], None]):
        """执行已批准的写入"""
        intent = self._find(intent_id)
        if intent.status != WriteStatus.APPROVED:
            raise PolicyViolation(f"Intent {intent_id} not approved")
        executor(intent)
        intent.status = WriteStatus.APPLIED
        self._log_event("apply", intent)
    
    def rollback(self, intent_id: str, reason: str):
        """回滚已应用的写入"""
        intent = self._find(intent_id)
        if intent.status != WriteStatus.APPLIED:
            return
        intent.rollback_action()
        intent.status = WriteStatus.ROLLED_BACK
        self._log_event("rollback", intent, reason=reason)
    
    def _find(self, intent_id: str) -> WriteIntent:
        for i in self.intents:
            if i.id == intent_id:
                return i
        raise KeyError(f"Intent {intent_id} not found")
    
    def _log_event(self, event_type: str, intent: WriteIntent, **kwargs):
        """不可变事件日志（实际生产环境应写入持久化存储）"""
        self.event_log.append({
            "timestamp": datetime.utcnow().isoformat(),
            "event_type": event_type,
            "intent_id": intent.id,
            "agent": intent.agent_identity,
            "path": intent.target_path,
            "operation": intent.operation,
            **kwargs,
        })


# --- 使用示例 ---
if __name__ == "__main__":
    surface = WriteSurface(
        identity="agent:nightly-refactor@v1.3",
        scope={
            "allowed_paths": ["/repo/src/", "/repo/tests/"],
            "max_diff_lines": 200,
            "can_delete": False,
        }
    )
    
    # Agent 提出写入意图
    intent = surface.propose_write(
        target_path="/repo/src/utils.py",
        operation="update",
        diff="+def helper(): pass",
        rationale="抽取重复代码到 helper 函数",
        rollback=lambda: print("git revert ..."),
    )
    
    # 规则引擎批准（基于 lint+test 通过）
    surface.approve(intent.id, approver="rule_engine", 
                    checks_passed=["lint", "tests", "type_check"])
    
    # 应用
    surface.apply(intent.id, executor=lambda i: print(f"Applying {i.diff}"))
    
    # 故障时回滚
    surface.rollback(intent.id, reason="downstream test failed")
    
    print(f"\n审计日志（{len(surface.event_log)} 条）:")
    for e in surface.event_log:
        print(f"  {e}")
```

这个模式的核心：**Agent 永远不直接写，只 propose；写操作必须经过 approve；所有动作都进 event log**。

## 5. 原语四：Budget Envelope

### 5.1 为什么需要

没有预算上限的 Agent 是无底洞。Revenium（2025）报告了一个真实案例："The $47K Agent Loop"——一个客服 Agent 因为消息循环 bug，在 8 小时内消费了 47000 美元 token 费用，直到信用卡被云厂商风控冻结才停。

预算不只是成本控制——**预算是 Agent 的硬终止条件**。没有预算的 Loop 没有"完成"的定义，只有"被强制停止"的定义。

### 5.2 多维预算

生产级 Agent 必须同时管理多个独立预算：

| 预算维度 | 单位 | 典型上限 | 触发机制 |
|---|---|---|---|
| Token | 输入+输出 token 数 | 1M token/run | 累计达到上限 → 终止 |
| Tool Call | 工具调用次数 | 100 次/run | 累计达到上限 → 终止 |
| Wall-clock | 真实时间 | 4 小时/run | 时间到 → 终止 |
| Cost | 美元 | $50/run | 估算成本达到 → 终止 |
| Iteration | Loop 迭代次数 | 50 次 | 达到上限 → 终止 |
| Tool-specific | 特定工具次数 | 10 次搜索/run | 单工具上限 → 禁用该工具 |

任何一个预算超限都应触发终止——不能只看单一维度。

### 5.3 预算感知的工具选择（BATS 模式）

arXiv 2511.17006（2025）提出 **BATS（Budget-Aware Test-time Scaling）**：将剩余预算显式注入 Agent 的推理上下文，让它根据预算调整行为。

预算感知的策略层级：

| 剩余预算比例 | Agent 行为模式 |
|---|---|
| ≥70%（充裕） | 广度探索：3-5 个并行查询，多 URL 浏览，多假设 |
| 30-70%（中等） | 收敛探索：2-3 个精炼查询，1-2 个目标 URL |
| 10-30%（紧张） | 单点突破：1 个查询，最多 1 次浏览，验证关键事实 |
| <10%（临界） | 强制收尾：直接输出当前最佳答案，禁止新工具调用 |

实证数据：相同准确率下，BATS 比无预算感知的 ReAct 减少约 40% 搜索调用、降低约 31% 总成本。

### 5.4 Budget Tracker 实现

```python
"""
多维预算包络：硬终止 + 软引导
"""
from dataclasses import dataclass, field
from typing import Dict, List
import time

@dataclass
class BudgetEnvelope:
    """多维预算包络"""
    max_tokens: int = 1_000_000
    max_tool_calls: int = 100
    max_iterations: int = 50
    max_wall_clock_seconds: int = 14400  # 4h
    max_cost_usd: float = 50.0
    
    # 实时计数
    used_tokens: int = 0
    used_tool_calls: int = 0
    used_iterations: int = 0
    started_at: float = field(default_factory=time.time)
    accumulated_cost: float = 0.0
    
    # 单工具配额
    tool_quotas: Dict[str, int] = field(default_factory=dict)
    tool_used: Dict[str, int] = field(default_factory=dict)


class BudgetExhausted(Exception):
    pass


class BudgetTracker:
    """预算跟踪器：硬终止 + 软引导"""
    
    def __init__(self, envelope: BudgetEnvelope):
        self.envelope = envelope
    
    def check_and_consume(self, tokens: int = 0, tool: str = None, 
                          cost: float = 0.0) -> str:
        """检查预算，返回 regime 标签或抛出 BudgetExhausted"""
        e = self.envelope
        
        # 累加消耗
        e.used_tokens += tokens
        if tool:
            e.used_tool_calls += 1
            e.tool_used[tool] = e.tool_used.get(tool, 0) + 1
        e.accumulated_cost += cost
        
        # 检查每个维度
        wall = time.time() - e.started_at
        violations = []
        if e.used_tokens >= e.max_tokens:
            violations.append("tokens")
        if e.used_tool_calls >= e.max_tool_calls:
            violations.append("tool_calls")
        if e.used_iterations >= e.max_iterations:
            violations.append("iterations")
        if wall >= e.max_wall_clock_seconds:
            violations.append("wall_clock")
        if e.accumulated_cost >= e.max_cost_usd:
            violations.append("cost")
        if tool and tool in e.tool_quotas:
            if e.tool_used[tool] >= e.tool_quotas[tool]:
                violations.append(f"tool:{tool}")
        
        if violations:
            raise BudgetExhausted(f"Budget exhausted: {violations}")
        
        # 计算剩余比例（取所有维度最低值）
        remaining_ratios = [
            1 - e.used_tokens / e.max_tokens,
            1 - e.used_tool_calls / e.max_tool_calls,
            1 - e.used_iterations / e.max_iterations,
            1 - wall / e.max_wall_clock_seconds,
            1 - e.accumulated_cost / e.max_cost_usd,
        ]
        min_remaining = min(remaining_ratios)
        
        # 返回 regime
        if min_remaining >= 0.7:
            return "abundant"
        elif min_remaining >= 0.3:
            return "moderate"
        elif min_remaining >= 0.1:
            return "tight"
        else:
            return "critical"
    
    def to_context_block(self) -> str:
        """注入 Agent context 的状态块"""
        e = self.envelope
        wall = time.time() - e.started_at
        return f"""
=== Budget Status ===
Tokens:     {e.used_tokens:,} / {e.max_tokens:,} ({e.used_tokens*100//e.max_tokens}%)
Tool calls: {e.used_tool_calls} / {e.max_tool_calls}
Iterations: {e.used_iterations} / {e.max_iterations}
Wall time:  {wall:.0f}s / {e.max_wall_clock_seconds}s
Cost:       ${e.accumulated_cost:.2f} / ${e.max_cost_usd:.2f}
=====================
"""
```

**关键设计**：`to_context_block()` 把预算状态注入每次 LLM 调用的 system prompt，让模型自己感知"我还剩多少"——这比纯外部硬终止更高效，因为 Agent 会主动调整行为。

### 5.5 软终止 vs 硬终止

预算策略需要分层：

- **软警告**（≤30% 剩余）：注入 context，引导 Agent 收敛
- **软终止**（≤10% 剩余）：禁止新工具调用，只允许输出最终答案
- **硬终止**（达到上限）：抛出 `BudgetExhausted`，强制结束

软终止的优势：Agent 有机会"优雅退出"，输出当前最佳结果，而非粗暴中断丢失所有进度。

## 6. 四原语的协同：完整 Workspace 实现

四个原语不是独立组件，它们必须协同工作：

```
Identity & Scope ──┐
                   ├─→ 决定 Write Surface 接受哪些操作
State & Persistence ┤
                   ├─→ 决定 Budget 计算的起点（重启时不重置）
Write Surface ─────┤
                   ├─→ 写操作消耗 Budget
Budget Envelope ───┘
                   └─→ 触发终止时，State 持久化最终状态
```

完整的 Workspace 装配示意：

```python
class Workspace:
    """完整 Workspace：四原语装配"""
    def __init__(self, agent_identity: str, scope: dict, 
                 budget: BudgetEnvelope):
        self.identity = agent_identity
        self.scope = scope
        self.write_surface = WriteSurface(agent_identity, scope)
        self.budget_tracker = BudgetTracker(budget)
        self.state_store = StateStore(path=f"workspace/{agent_identity}/")
    
    def execute_action(self, action_type: str, **kwargs):
        """统一执行入口：所有 action 都过四原语"""
        # 1. 检查预算
        regime = self.budget_tracker.check_and_consume(
            tokens=kwargs.get("estimated_tokens", 0),
            tool=action_type,
            cost=kwargs.get("estimated_cost", 0.0),
        )
        
        # 2. 写操作走治理面
        if action_type.startswith("write_"):
            intent = self.write_surface.propose_write(**kwargs)
            return intent
        
        # 3. 状态变更落到持久化层
        if action_type == "update_state":
            self.state_store.append_event(kwargs)
        
        # 4. 注入 budget regime 到下次 LLM 调用
        return {"regime": regime, ...}
```

## 7. 典型场景的 Workspace 配置

不同场景的四原语取值差异巨大。下面是几个典型配置：

### 7.1 IDE Coding Copilot（如 Cursor）

| 原语 | 配置 |
|---|---|
| Identity | 用户身份本身（IDE 内嵌） |
| Scope | 当前打开文件 + 用户授权扩展 |
| State | IDE session 内存，不持久化 |
| Write Surface | IDE diff UI，用户 accept/reject |
| Budget | 单次请求 token 上限（用户感知不到） |

特点：Workspace 极简——因为用户在场，每次都能立即审核。

### 7.2 异步 AI Worker（如 Devin）

| 原语 | 配置 |
|---|---|
| Identity | 独立 service account（每个任务一个 microVM） |
| Scope | 仓库可读写、生产环境只读、密钥隔离 |
| State | 跨会话持久化 + Event Log + Snapshot |
| Write Surface | PR-only（不能直接 push 到主分支） |
| Budget | $50/任务、4h wall-clock、200 工具调用 |

特点：Workspace 最重——因为用户不在场，所有治理必须在 Workspace 层兜底。

### 7.3 夜跑批处理 Agent（如自动 dependency upgrade）

| 原语 | 配置 |
|---|---|
| Identity | bot account，commit 带 [bot] 标记 |
| Scope | 依赖文件 + lock 文件，禁止改业务代码 |
| State | TODO.md（剩余仓库列表）+ event log |
| Write Surface | PR-only + 自动等待 CI 全绿才允许合并 |
| Budget | $5/仓库、1h/批次、自动跳过失败仓库 |

特点：Scope 极窄、Write Surface 强约束、Budget 严格——因为没人盯着。

### 7.4 多 Agent 协作系统（如 Anthropic Research）

| 原语 | 配置 |
|---|---|
| Identity | Lead Agent + N 个 Subagent，各自独立身份 |
| Scope | Subagent 只能读分配的子任务输入，写子任务输出文件 |
| State | 共享文件系统（subagent 写 → lead 读） |
| Write Surface | 文件系统是输出面，无 PR 概念 |
| Budget | 每个 subagent 独立预算 + 全局预算（Lead 监控） |

特点：每个 Agent 都有独立 Workspace，但共享文件系统作为通信介质——这是 Anthropic（2025）规避"电话游戏"token 膨胀的关键设计。

## 8. 反模式

### 反模式一：共享身份的 Multi-Agent

所有 subagent 共用一个 token，commit 都署名同一个 bot。后果：故障定位时无法区分是哪个 subagent 改坏的，code review 失去维度信息。

**正解**：每个子 Agent 一个独立身份，commit 署名包含 agent_id 和 parent_id。

### 反模式二：State 只在内存

Agent 进程崩溃后所有进度丢失。重启从 0 开始。后果：长任务永远跑不完——任何 4h 内的中断都会让 4h 工作白费。

**正解**：每个工具调用后持久化状态，至少做到"重启可断点续跑"。

### 反模式三：Write 直通生产

Agent 写权限直连生产数据库/Git 主分支。后果：一次幻觉 = 一次事故。

**正解**：所有写操作必须先进 Shadow Workspace 或 Pending 队列，经审核（人工或规则引擎）后才生效。

### 反模式四：单一 Token 预算

只看 token 不看时间不看成本。后果：Agent 在便宜模型上死循环，token 没超但时间和真实账单已经爆。

**正解**：多维预算同时检查，任一维度超限即终止。

### 反模式五：硬终止无优雅退出

预算耗尽时直接抛异常退出。后果：当前进度未持久化，下次重启重做。

**正解**：分层终止——软警告 → 软终止（强制收敛）→ 硬终止（保存状态后退出）。

## 9. Workspace 的渐进式建设

不是所有项目从第一天就需要四原语全配。建议的演进路径：

| 阶段 | 必备原语 | 触发升级的信号 |
|---|---|---|
| 阶段 0：Demo | 任意 | （demo 用，不上生产） |
| 阶段 1：内测 | Budget Envelope | 出现一次失控成本 |
| 阶段 2：私测 | + Write Surface | 出现一次需要回滚的事故 |
| 阶段 3：公测 | + State & Persistence | 长任务中断频繁 |
| 阶段 4：GA | + Identity & Scope | 多 Agent / 多用户场景上线 |

**核心信号**：每个原语都对应一个具体的生产事故类型。如果你还没遇到这类事故，可以推迟引入；一旦遇到，必须立即补齐——因为同类事故一定会再发生。

## 10. 总结：Workspace 是 Loop 的"安全带"

回到 Loop Engineering 的整体框架（2.2 已建立）：

- **Automations、Worktrees、Skills、Connectors、Sub-agents、Memory** 解决"如何让 Loop 跑起来"
- **Workspace 四原语** 解决"如何让 Loop 跑得起、停得下、改得回、可追溯"

打个比方：5+1 building blocks 让你造出了一辆能跑的车，但 Workspace 是这辆车的安全带、刹车、ABS、和事故记录仪——demo 时这些可以省，上路必须有。

Addy Osmani（2025）的判断很犀利：

> "如果你不能确定谁做了改动、并且不打开终端就能回滚，那你的 loop 不是自治系统——它是一个延迟的事故。"

四个原语，按需组合，按事故升级。不要追求一次到位，但要清楚每个原语解决的是哪类事故——这样当事故发生时，你知道下一步该补哪个。

下一篇 4.3 将基于这个 Workspace 框架，对比 Codex `/goal`、Claude Code `/loop`、Devin 三家工业级 Loop 实现的具体差异。
