# Agent 通信与协调：共享状态 vs 消息传递 vs A2A 协议

> Multi-Agent 系统的拓扑（路由、层级、群体）定义了"谁和谁连接"，但真正决定系统行为的是**Agent 之间如何交换信息、如何达成一致**。通信机制是 Multi-Agent 系统的"神经网络"——选错了，再好的拓扑也跑不起来。

## 0. 从一个协调失败说起

某 AI 研究系统用 3 个 Agent 做竞品分析：Searcher 搜索资料、Analyzer 提取洞察、Writer 输出报告。三者共享一个消息列表，Searcher 每次搜索都把完整结果（~3000 tokens）追加到共享状态。当 Searcher 完成 8 轮搜索后，共享状态已膨胀到 24000 tokens——Analyzer 的上下文窗口中 70% 都是原始搜索结果，留给"分析"的空间只剩 30%。结果：Analyzer 开始跳过早期搜索结果，输出质量断崖式下降。

根本原因：**全量共享状态**没有区分"什么信息该传给谁"。Analyzer 不需要原始搜索结果的全文，只需要 Searcher 提取的结构化发现。

修复方案：改为**接口化消息传递**——每个 Agent 只暴露结构化输出（findings summary），不暴露内部工作记录。共享状态从 24000 tokens 降到 3000 tokens，Analyzer 准确率恢复到预期水平。

## 1. 三种通信范式

Multi-Agent 通信可以归纳为三种根本不同的范式：

| 范式 | 核心抽象 | 类比 | 典型实现 |
|------|----------|------|----------|
| **共享状态** | 所有 Agent 读写同一块数据结构 | 白板上写字，所有人都能看到 | LangGraph shared state, Blackboard |
| **消息传递** | Agent 之间发送结构化消息 | 发邮件，收件人才能看到 | Handoff, Agent-as-Tool, Event Bus |
| **协议互操作** | 跨系统/跨厂商的标准化通信 | HTTP REST/gRPC，不关心对方实现 | A2A, MCP（工具层） |

三者不互斥——大多数生产系统混合使用。

## 2. 共享状态（Shared State）

### 2.1 机制

所有 Agent 对同一块状态对象（通常是消息列表或 key-value store）做读写操作。LangGraph 的核心抽象就是这个：图的"state"是所有节点共享的数据结构，节点（Agent）通过修改 state 来通信。

```python
# LangGraph 共享状态模型
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph
from langgraph.graph.message import add_messages

class SharedState(TypedDict):
    messages: Annotated[list, add_messages]  # 所有 Agent 共享的消息列表
    current_agent: str
    task_status: str

# Agent A 写入 state
def agent_a(state: SharedState) -> dict:
    # 处理后将结果写入 messages
    result = do_work(state["messages"])
    return {"messages": [{"role": "assistant", "content": result}]}

# Agent B 读取 state（能看到 Agent A 写的内容）
def agent_b(state: SharedState) -> dict:
    all_history = state["messages"]  # 包含 Agent A 的输出
    analysis = analyze(all_history)
    return {"messages": [{"role": "assistant", "content": analysis}]}
```

### 2.2 两种变体

**完全共享**（LangGraph Multi-Agent Collaboration）：所有 Agent 的每一步操作都写入同一个消息列表，每个 Agent 都能看到其他 Agent 的完整工作过程。优点是完全透明，缺点是上下文快速膨胀。

**选择性共享**（LangGraph Supervisor）：每个 Agent 维护自己的内部工作记录（scratchpad），只有最终结果被写入全局状态。Supervisor 只看到各 Agent 的结论，看不到过程。

```
完全共享：
  Agent A 工作记录 ──→ Global State ←── Agent B 工作记录
                         ↑
                    Agent C 全都能看到

选择性共享：
  Agent A [internal scratchpad] ──(final result only)──→ Global State
  Agent B [internal scratchpad] ──(final result only)──→ Global State
                                                            ↑
                                                    Supervisor 只看结论
```

### 2.3 优劣分析

| 维度 | 优势 | 劣势 |
|------|------|------|
| 透明度 | 完全可追溯，调试直观 | 信息过载，"噪声"淹没"信号" |
| 实现复杂度 | 最简单——加个列表就行 | 并发写入需要冲突解决 |
| 上下文效率 | 无需格式转换 | 线性增长，不可控 |
| 耦合度 | 无需定义接口 | 所有 Agent 依赖同一数据结构 |

### 2.4 适用场景

- Agent 数量少（2-4 个）
- 需要完全透明的协作过程（如 pair programming）
- 快速原型验证
- 所有 Agent 属于同一开发团队/同一系统

## 3. 消息传递（Message Passing）

### 3.1 机制

Agent 之间通过结构化消息通信，每个 Agent 只接收发给自己的消息，只暴露结构化输出。类似微服务之间通过 API 调用通信。

### 3.2 三种实现模式

**Handoff（控制权转移）**：当前活跃 Agent 将控制权连同相关上下文转交给下一个 Agent。OpenAI Agents SDK 的核心机制——Agent 声明 `handoffs` 列表，模型判断后主动转交。

```python
# OpenAI Agents SDK Handoff 模式
from agents import Agent, handoff

refund_agent = Agent(
    name="Refund Specialist",
    instructions="Handle refund requests...",
    tools=[process_refund, check_order_status],
)

triage_agent = Agent(
    name="Triage",
    instructions="Route customer queries to the right specialist.",
    handoffs=[handoff(refund_agent, description="For refund-related queries")],
)
# 当 triage_agent 判断是退款问题时，自动 handoff 到 refund_agent
```

**Agent-as-Tool（函数调用式）**：Orchestrator 将其他 Agent 包装为工具，通过标准工具调用接口发送请求、接收结果。控制权始终在 Orchestrator 手中。

```python
# OpenAI Agents SDK Agent-as-Tool 模式
from agents import Agent

research_agent = Agent(name="Researcher", instructions="...", tools=[web_search])
analysis_agent = Agent(name="Analyzer", instructions="...", tools=[data_process])

orchestrator = Agent(
    name="Orchestrator",
    instructions="Coordinate research and analysis tasks.",
    tools=[
        research_agent.as_tool(description="Delegate research queries"),
        analysis_agent.as_tool(description="Delegate data analysis"),
    ],
)
# Orchestrator 保持控制权，按需调用子 Agent
```

**Event-Driven（事件发布/订阅）**：Agent 通过事件总线发布消息，其他 Agent 订阅感兴趣的事件类型。完全解耦，适合异步长任务。

```python
# Event-driven 伪代码
class EventBus:
    def __init__(self):
        self._subscribers: dict[str, list[Callable]] = {}

    def subscribe(self, event_type: str, handler: Callable):
        self._subscribers.setdefault(event_type, []).append(handler)

    def publish(self, event_type: str, payload: dict):
        for handler in self._subscribers.get(event_type, []):
            handler(payload)

bus = EventBus()

# Searcher 发布搜索结果
bus.publish("search.completed", {"findings": [...], "query": "..."})

# Analyzer 订阅搜索结果
bus.subscribe("search.completed", lambda payload: analyze(payload["findings"]))
```

### 3.3 消息格式设计

Agent 之间的消息应该是**结构化且自描述的**：

```python
@dataclass
class AgentMessage:
    """标准 Agent 间消息"""
    id: str                    # 消息唯一 ID
    from_agent: str            # 发送方标识
    to_agent: str              # 接收方标识
    intent: str                # 消息意图（request/response/notification/handoff）
    content: dict              # 结构化内容
    context: dict | None       # 上游上下文（按需携带）
    timestamp: float
    correlation_id: str        # 关联 ID，追踪完整调用链
```

关键设计原则：
- **最小信息原则**：只传接收方需要的信息，不传内部工作记录
- **自描述**：消息自身携带足够元数据，接收方无需查询外部状态
- **关联追踪**：correlation_id 贯穿整条调用链，支持分布式追踪

### 3.4 优劣分析

| 维度 | 优势 | 劣势 |
|------|------|------|
| 上下文效率 | Agent 只接收需要的信息 | 需要定义接口/schema |
| 可扩展性 | 新 Agent 接入只需实现接口 | 接口变更需要协调 |
| 隔离性 | Agent 内部实现自由 | 调试需要跨 Agent 追踪 |
| 并发安全 | 天然无冲突 | 需要处理超时和失败 |

### 3.5 适用场景

- Agent 数量多（5+）
- 不同 Agent 由不同团队维护
- 需要精确控制信息流
- 异步长任务
- 需要第三方 Agent 集成

## 4. A2A 协议：跨系统互操作

### 4.1 为什么需要 A2A

MCP 解决了 Agent 如何连接工具的标准化，但当**不同组织/不同框架构建的 Agent 之间**需要协作时，需要另一层协议。A2A（Agent-to-Agent）就是这层——由 Google 于 2025 年 4 月发布的开放协议。

```
MCP 的定位：Agent ↔ Tool（"Agent 的 USB-C"）
A2A 的定位：Agent ↔ Agent（"Agent 的 HTTP"）
```

### 4.2 核心概念

**Agent Card**（`/.well-known/agent.json`）：Agent 的数字身份证，描述能力、端点、鉴权方式。类似 `robots.txt` 的发现机制。

```json
{
  "name": "Travel Booking Agent",
  "description": "Books flights, hotels, and rental cars",
  "url": "https://travel-agent.example.com/a2a",
  "version": "1.0.0",
  "capabilities": {
    "skills": ["flight_booking", "hotel_reservation", "car_rental"],
    "streaming": true,
    "pushNotifications": true
  },
  "authentication": {
    "schemes": ["bearer"]
  }
}
```

**Task**：协作的核心对象，有完整生命周期：

```
submitted → working → completed
                  ↘ input-required → (补充信息) → working
                  ↘ failed
                  ↘ canceled
```

**Message**：任务中交换的内容，由 Parts 组成（文本、图片、视频、交互式元素）。支持 UX 能力协商——客户端 Agent 可以声明支持的展示能力。

**Artifact**：任务完成后产出的交付物（文件、数据、报告等）。

### 4.3 通信机制

基于 JSON-RPC 2.0 + HTTP，支持三种交互模式：

| 模式 | 机制 | 适用场景 |
|------|------|----------|
| 请求/响应 | 标准 JSON-RPC over HTTP | 快速任务（<30s） |
| 流式 | SSE (Server-Sent Events) | 需要实时进度反馈 |
| 推送通知 | Webhook + 认证 | 长任务（小时/天级） |

核心方法：
- `tasks/send` — 创建或更新任务
- `tasks/get` — 查询任务状态
- `tasks/cancel` — 取消任务
- `tasks/sendSubscribe` — 订阅任务实时更新

### 4.4 A2A vs MCP：互补定位

| 维度 | MCP | A2A |
|------|-----|-----|
| 连接什么 | Agent ↔ Tool/Data | Agent ↔ Agent |
| 通信粒度 | 函数调用级（调一个工具） | 任务级（委托一个目标） |
| 发现机制 | Server 注册 + 能力协商 | Agent Card（/.well-known/） |
| 状态管理 | 有状态连接（Client-Server） | Task 生命周期（无状态 HTTP） |
| 自主度 | Tool 被动执行 | Agent 自主决策执行方式 |
| 协作模式 | 垂直（Agent 调 Tool） | 水平（Agent 委托 Agent） |

**组合使用场景**：一个 Travel Agent 内部通过 MCP 连接航班 API、酒店 API（工具层），对外通过 A2A 暴露"预订旅行"能力给其他 Agent 调用（协作层）。

### 4.5 治理与安全

A2A 的安全模型：
- 与 OpenAPI 认证方案对齐（Bearer、OAuth、API Key）
- JWT 支持多并发会话
- RSA 密钥对用于服务间认证
- 基于角色的访问控制
- 速率限制（token bucket / leaky bucket / rolling window）

## 5. 协调机制深度剖析

通信解决"信息怎么流动"，协调解决"决策怎么做出"。

### 5.1 控制流协调

| 策略 | 描述 | 实现 |
|------|------|------|
| **中心化** | 一个节点决定谁做什么 | Orchestrator/Supervisor |
| **委托链** | 控制权顺序传递 | Handoff chain |
| **竞争式** | 多 Agent 竞标任务 | Market-Based |
| **涌现式** | 局部规则产生全局行为 | Swarm/Blackboard |

### 5.2 状态一致性

当多个 Agent 并发操作时，状态一致性是核心挑战：

**乐观并发**：Agent 先执行，冲突时回滚重试。适合冲突概率低的场景。

**悲观锁定**：Agent 操作前先获取锁。适合冲突代价高的场景（如金融操作）。

**事件溯源**：所有操作记录为不可变事件序列，通过回放重建状态。Kafka-based 事件驱动系统天然支持。

**最终一致**：允许短暂不一致，最终收敛。大多数 AI Agent 系统采用此模型——Agent 的推理本身就是非确定性的，强一致性意义不大。

### 5.3 失败处理与恢复

| 失败类型 | 策略 | 示例 |
|----------|------|------|
| Agent 超时 | 超时 + 降级 | Searcher 超时后 Orchestrator 用已有结果继续 |
| Agent 错误 | 重试 + 回退 | 3 次重试后切换到备用 Agent |
| 部分失败 | 补偿事务 | Writer 失败后回滚 Analyzer 的中间状态 |
| 级联失败 | 断路器 | 一个 Agent 持续失败时切断，防止雪崩 |

### 5.4 终止条件设计

Multi-Agent 系统必须有显式终止条件，否则会无限循环：

```python
class TerminationPolicy:
    """终止条件组合"""

    def __init__(
        self,
        max_iterations: int = 10,
        max_tokens: int = 100_000,
        timeout_seconds: float = 300,
        quality_threshold: float = 0.8,
    ):
        self.max_iterations = max_iterations
        self.max_tokens = max_tokens
        self.timeout_seconds = timeout_seconds
        self.quality_threshold = quality_threshold
        self._start_time = time.time()
        self._iteration = 0
        self._tokens_used = 0

    def should_terminate(self, quality_score: float | None = None) -> tuple[bool, str]:
        self._iteration += 1

        if self._iteration >= self.max_iterations:
            return True, f"Max iterations ({self.max_iterations}) reached"
        if self._tokens_used >= self.max_tokens:
            return True, f"Token budget ({self.max_tokens}) exhausted"
        if (time.time() - self._start_time) >= self.timeout_seconds:
            return True, f"Timeout ({self.timeout_seconds}s) exceeded"
        if quality_score and quality_score >= self.quality_threshold:
            return True, f"Quality threshold ({self.quality_threshold}) met"

        return False, ""
```

## 6. 完整代码示例：混合通信架构

```python
"""
混合通信 Multi-Agent 系统
- Orchestrator ↔ Workers: 消息传递（Agent-as-Tool 模式）
- Workers 内部: 共享 scratchpad
- 外部 Agent: A2A 协议接口

演示场景：竞品分析系统
"""
import asyncio
from dataclasses import dataclass, field
from typing import Any, Protocol
from abc import abstractmethod
import json
import time


# ─── 通信层抽象 ──────────────────────────────────────────────────

class CommunicationChannel(Protocol):
    """通信通道协议"""

    @abstractmethod
    async def send(self, message: "InterAgentMessage") -> "InterAgentMessage":
        ...


@dataclass
class InterAgentMessage:
    """统一的 Agent 间消息"""
    id: str
    from_agent: str
    to_agent: str
    intent: str  # "request" | "response" | "handoff" | "notification"
    content: dict[str, Any]
    correlation_id: str = ""
    timestamp: float = field(default_factory=time.time)


# ─── 共享状态通信（团队内部）──────────────────────────────────────

class SharedScratchpad:
    """团队内部共享状态"""

    def __init__(self):
        self._state: dict[str, Any] = {}
        self._history: list[dict] = []

    def write(self, agent: str, key: str, value: Any) -> None:
        self._state[key] = value
        self._history.append({"agent": agent, "key": key, "action": "write", "ts": time.time()})

    def read(self, key: str) -> Any:
        return self._state.get(key)

    def get_summary(self) -> dict:
        """返回摘要而非全部（控制上下文膨胀）"""
        return {k: str(v)[:200] for k, v in self._state.items()}


# ─── 消息传递通信（跨 Agent）──────────────────────────────────────

class DirectChannel:
    """直接消息通道（进程内模拟）"""

    def __init__(self):
        self._agents: dict[str, "WorkerAgent"] = {}

    def register(self, agent: "WorkerAgent") -> None:
        self._agents[agent.name] = agent

    async def send(self, message: InterAgentMessage) -> InterAgentMessage:
        target = self._agents.get(message.to_agent)
        if not target:
            return InterAgentMessage(
                id=f"err-{message.id}",
                from_agent="system",
                to_agent=message.from_agent,
                intent="response",
                content={"error": f"Agent '{message.to_agent}' not found"},
                correlation_id=message.correlation_id,
            )
        return await target.handle_message(message)


# ─── Worker Agent（使用消息传递接收指令，内部可用共享状态）─────────

class WorkerAgent:
    """Worker Agent 基类"""

    def __init__(self, name: str, description: str):
        self.name = name
        self.description = description
        self.scratchpad = SharedScratchpad()  # 内部工作区

    async def handle_message(self, message: InterAgentMessage) -> InterAgentMessage:
        """处理收到的消息"""
        # 内部工作在 scratchpad 中进行
        self.scratchpad.write(self.name, "last_request", message.content)

        result = await self._process(message.content)

        # 只返回结构化结果，不暴露 scratchpad
        return InterAgentMessage(
            id=f"resp-{message.id}",
            from_agent=self.name,
            to_agent=message.from_agent,
            intent="response",
            content={"result": result, "status": "success"},
            correlation_id=message.correlation_id,
        )

    async def _process(self, request: dict) -> str:
        raise NotImplementedError


class ResearchWorker(WorkerAgent):
    def __init__(self):
        super().__init__("researcher", "Searches and collects information")

    async def _process(self, request: dict) -> str:
        query = request.get("query", "")
        # 模拟搜索
        findings = f"Found 5 relevant sources for '{query}'"
        self.scratchpad.write(self.name, "raw_results", f"[raw data for {query}]")
        # 只返回结构化摘要，不返回 raw data
        return findings


class AnalysisWorker(WorkerAgent):
    def __init__(self):
        super().__init__("analyzer", "Analyzes data and extracts insights")

    async def _process(self, request: dict) -> str:
        data = request.get("data", "")
        insights = f"Key insight: {data[:50]}... shows pattern X"
        self.scratchpad.write(self.name, "analysis_steps", "[internal reasoning]")
        return insights


# ─── A2A 接口层（对外暴露）──────────────────────────────────────

class A2AInterface:
    """A2A 协议接口：将内部系统暴露为标准 A2A Agent"""

    def __init__(self, orchestrator: "CompetitorAnalysisOrchestrator"):
        self.orchestrator = orchestrator

    def get_agent_card(self) -> dict:
        """返回 Agent Card（/.well-known/agent.json）"""
        return {
            "name": "Competitor Analysis Agent",
            "description": "Performs deep competitor research and produces structured reports",
            "url": "https://analysis.internal.example.com/a2a",
            "version": "1.0.0",
            "capabilities": {
                "skills": ["competitor_analysis", "market_research"],
                "streaming": True,
                "pushNotifications": False,
            },
            "authentication": {"schemes": ["bearer"]},
        }

    async def handle_task(self, task_request: dict) -> dict:
        """处理 A2A tasks/send 请求"""
        task_id = task_request.get("id", "t-001")
        message = task_request.get("message", {})
        query = message.get("parts", [{}])[0].get("text", "")

        # 内部编排
        result = await self.orchestrator.run(query)

        return {
            "id": task_id,
            "status": {"state": "completed"},
            "artifacts": [{
                "name": "analysis_report",
                "parts": [{"type": "text", "text": result}],
            }],
        }


# ─── Orchestrator（混合通信）─────────────────────────────────────

class CompetitorAnalysisOrchestrator:
    """
    编排者：
    - 对 Worker 用消息传递（结构化请求/响应）
    - Worker 内部用共享状态（scratchpad）
    - 对外用 A2A 接口
    """

    def __init__(self):
        self.channel = DirectChannel()
        self.researcher = ResearchWorker()
        self.analyzer = AnalysisWorker()
        self.channel.register(self.researcher)
        self.channel.register(self.analyzer)
        self.a2a = A2AInterface(self)

    async def run(self, user_query: str) -> str:
        """执行完整分析流程"""
        # Step 1: 通过消息传递委托搜索
        search_msg = InterAgentMessage(
            id="msg-001",
            from_agent="orchestrator",
            to_agent="researcher",
            intent="request",
            content={"query": user_query, "max_sources": 10},
            correlation_id="corr-001",
        )
        search_response = await self.channel.send(search_msg)

        # Step 2: 将搜索结果（摘要）传给分析
        analysis_msg = InterAgentMessage(
            id="msg-002",
            from_agent="orchestrator",
            to_agent="analyzer",
            intent="request",
            content={"data": search_response.content["result"], "focus": "competitive_advantage"},
            correlation_id="corr-001",
        )
        analysis_response = await self.channel.send(analysis_msg)

        # Step 3: 综合输出
        return f"## Competitor Analysis: {user_query}\n\n{analysis_response.content['result']}"


# ─── 运行示例 ──────────────────────────────────────────────────

async def main():
    system = CompetitorAnalysisOrchestrator()

    # 内部调用（消息传递）
    result = await system.run("Analyze LangGraph vs CrewAI market position")
    print(f"[Internal result]\n{result}\n")

    # 外部 A2A 调用
    a2a_request = {
        "id": "task-ext-001",
        "message": {"parts": [{"type": "text", "text": "Analyze OpenAI vs Anthropic agent strategy"}]},
    }
    a2a_response = await system.a2a.handle_task(a2a_request)
    print(f"[A2A response]\n{json.dumps(a2a_response, indent=2)}")


if __name__ == "__main__":
    asyncio.run(main())
```

## 7. 通信范式选择决策树

```
需要跨组织/跨厂商 Agent 互操作？
  ├─ 是 → A2A 协议
  └─ 否 → 同一系统内部
            ├─ Agent 数量 ≤ 4 且需要完全透明？
            │   └─ 是 → 共享状态
            ├─ 需要精确控制信息流？
            │   └─ 是 → 消息传递
            ├─ 高吞吐异步？
            │   └─ 是 → 事件驱动（消息传递的异步变体）
            └─ 快速原型？
                └─ 共享状态（最简实现）
```
