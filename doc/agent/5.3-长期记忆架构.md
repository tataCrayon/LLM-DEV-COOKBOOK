# 长期记忆架构：MemGPT / Letta / Hierarchical Memory

> 本文是 Agent 系列第 5.3 篇。**5.1 解决"记忆有哪几种"，5.2 解决"知识怎么检索"，5.3 解决"长期记忆怎么落地成可运行的系统"。** 我们把视线从"记忆分类"和"RAG 工程"收回到一个更具体的问题：**当对话已经超过 100 万 token、跨越数百次会话、用户期待 Agent 像同事一样"记得我们上次聊过什么"时——这套系统的运行时长什么样？**
>
> 答案的源头是 **MemGPT（Packer et al., 2023）**：把 LLM 当作操作系统，把 context window 当作 RAM，把外部存储当作 disk，由模型自己通过 function call 完成 page-in / page-out。两年过去，MemGPT 已经演化为商业化产品 **Letta**，引入了 **memory blocks**、**sleep-time compute** 等新原语；**MemoryOS（EMNLP 2025）** 进一步把它扩展为 STM/MTM/LPM 三层分页架构；**Mem0 (ECAI 2025)** 用 92.5/LoCoMo 的成绩证明了"提取-压缩-再检索"路线在工业落地上的优势。
>
> 这一篇会从一个真问题切入——**"为什么把所有历史 dump 到 prompt 里行不通？"**——逐层剥开虚拟上下文、记忆块、sleep-time compute、三层分页等关键架构，给出生产级 Agent Memory Runtime 的完整设计蓝图与代码骨架。

## 0. 真问题：长上下文窗口能否替代长期记忆？

2026 年很多模型已经原生支持 1M~10M token 上下文（Gemini 2.5、GLM-4.6、Qwen3-Max-Preview-2M 等），常有人问："既然窗口够大，直接把所有历史塞进去不就行了？"

直接看数据：

| 维度 | 全量 full-context（10M） | 长期记忆系统（如 Mem0/Letta） |
|---|---|---|
| 单次成本 | ~26,000+ tokens prompt | ~6,956 tokens prompt（LoCoMo 基准） |
| 延迟 | TTFT 通常 5-20s | TTFT 1-3s |
| 注意力衰减 | "Lost in the Middle"，长文本中段召回急剧下降 | 通过精准检索 top-k，注意力集中 |
| 跨会话身份一致性 | 必须每次重传，否则不连贯 | 持久化 user profile、persona |
| 隐私可控性 | 历史全量入 prompt，敏感信息泄露风险 | 可在记忆层做过滤/遮蔽 |
| BEAM 10M 准确率 | 大多模型在 10M 尺度上失败 | Mem0 算法 48.6 / 显著高于 baseline |

> *Source*: Mem0, *State of AI Agent Memory 2026*。

**结论**：长上下文窗口是必要条件，不是充分条件。它解决了"能塞下"的问题，但解决不了"应该塞什么"、"怎么持久化"、"怎么跨会话保持一致"、"怎么控成本控延迟"四个问题。这些恰恰是长期记忆架构要解决的。

> 一句话总结：**Long-context ≠ Long-term memory**。前者是 RAM 的容量，后者是从 RAM 到 disk 的虚拟内存系统。

## 1. 第一性原理：把 LLM 看作操作系统

MemGPT 论文（Packer et al., 2023）提出了一个被后续所有长期记忆系统继承的隐喻——**LLM as Operating System**。映射如下：

| OS 概念 | LLM 对应概念 | 举例 |
|---|---|---|
| RAM | Context window | 主上下文（system prompt + 工作集 + 当前消息） |
| Disk | External store | Vector DB / KV / 文件系统 |
| Page table | Memory map | 哪些信息当前在 context、哪些在 disk |
| Page fault | Recall miss | 模型发现自己缺信息，触发检索 |
| Page-in / Page-out | Read / Evict | 把信息从 disk 召回 / 把陈旧信息写回 disk |
| Interrupt | Function call | LLM 主动调用 `read_memory` / `write_memory` |
| Process | Agent | 一个有状态、有身份、可暂停可恢复的实例 |

这个隐喻的威力在于：**它让记忆问题从"prompt 工程问题"升级为"系统问题"**。一旦你把 Agent 看作进程，自然就需要状态持久化、生命周期管理、隔离、调度——这恰恰是 Letta 把自己定位成"agent runtime"而不是"memory library"的根本原因。

## 2. MemGPT：虚拟上下文管理的奠基设计

### 2.1 双层结构

MemGPT 把上下文分成两层：

```
┌──────────────────────────────────────────────────────────┐
│                    Main Context（在 LLM 窗口内）            │
│  ┌──────────────┬───────────────┬──────────────────────┐  │
│  │ System Inst  │ Working Ctx   │ FIFO Queue           │  │
│  │ (静态)        │ (Core Memory) │ (recent messages)    │  │
│  └──────────────┴───────────────┴──────────────────────┘  │
└────────────────────────────▲─────────────────────────────┘
                             │  function calls
┌────────────────────────────┴─────────────────────────────┐
│                External Context（窗口外）                   │
│  ┌────────────────────────┬────────────────────────────┐ │
│  │  Recall Storage         │  Archival Storage          │ │
│  │  (历史完整对话日志)       │  (任意结构化/向量化文档)     │ │
│  └────────────────────────┴────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

* **Main Context** = LLM 实际看到的 token，进一步分为：
  * **System instruction**：固定不变（角色、规则、工具定义）
  * **Working context（核心记忆）**：模型可读写的小块"工作 RAM"，存 persona、用户画像
  * **FIFO queue**：最近若干轮消息
* **External Context** = 外部存储：
  * **Recall storage**：完整对话历史，按时间/主题可搜索
  * **Archival storage**：任意外部知识（文档、笔记、向量库），通过工具调用访问

### 2.2 关键机制 1：Self-edit by function call

MemGPT 的最大创新——**让 LLM 自己当 OS 调度器**。系统暴露一组 function：

* `core_memory_append(content)` / `core_memory_replace(label, content)` — 改写工作上下文
* `conversation_search(query)` / `conversation_search_date(start, end)` — 查 recall storage
* `archival_memory_insert(content)` / `archival_memory_search(query)` — 读写归档

每轮对话结束前，模型自主决定：当前哪些信息值得提升到 core memory？哪些 message 该写回 archival？哪些过期了？这些决策不靠外部 heuristic，而是由 LLM 自己 reason。

### 2.3 关键机制 2：Heartbeat 与中断

模型可以请求一个 **heartbeat**：返回结果后立刻继续推理（不等用户输入），用于多步内部操作（先 search，再 append，再 reply）。这把 Agent 从"一问一答"升级为"一问多步内部 loop 后再答"。

### 2.4 关键机制 3：Pressure-driven eviction

当 FIFO queue 快满时，系统注入"memory pressure" warning，触发模型主动召回 / 转储。这是 MemGPT 把 OS 内存压力机制翻译成 LLM 行为的精巧设计。

### 2.5 局限

* **Eviction 全靠模型 judgment**，token 成本高（每次决策都要推理）
* **FIFO queue 是平铺结构**，对话长了会出现 topic mixing（话题串扰）
* **没有显式跨会话身份/技能层**，长期一致性不强

> *Source*: Packer et al., *MemGPT: Towards LLMs as Operating Systems*, [arXiv:2310.08560](https://arxiv.org/abs/2310.08560)；MemoryOS 论文亦明确指出 MemGPT "flat FIFO queue causes topic mixing as dialogue length grows"。

## 3. Letta：从 MemGPT 走向 Stateful Agent Runtime

2024-2026 年间，MemGPT 商业化为 **Letta**。它保留了虚拟上下文范式，但做了三件大事：(1) 重新设计 memory blocks 模型；(2) 把自己定位为 runtime；(3) 引入 sleep-time compute。

### 3.1 三层记忆原语

| 层级 | 类比 | 特征 | 典型内容 |
|---|---|---|---|
| **Core Memory** | RAM | 永远在窗口内、模型可直接读写 | persona、人格设定、当前用户画像、技能列表 |
| **Recall Memory** | Disk Cache | 完整对话历史，按需召回 | 历次会话日志，按时间/语义搜索 |
| **Archival Memory** | Cold Storage | 大规模知识库 | 文档、长文本、向量化片段，通过工具访问 |

> *Source*: Vectorize, *Hindsight vs Letta (MemGPT): Agent Memory Compared (2026)*。

Letta 的关键升级是把 core memory 抽象成**显式 Memory Blocks**——每个 block 是一个有标签、有语义边界的小段 markdown，例如：

```
[block: persona]
你是一名资深 AI Agent 应用架构师，回答风格务实、数据导向...

[block: human]
姓名：张三；研究方向：长期记忆；最近一次对话：2026-06-22 关于 Letta vs Mem0...

[block: skills]
- 擅长 Python / TypeScript
- 熟悉 LangGraph / LangChain
- 不熟悉 Rust 语法

[block: rethink_memory_block]   # 用于 sleep-time compute
（由 sleep-time agent 改写）
```

每个 block 都可以是 read-only 或 read-write，支持 ACL 与版本控制。Letta 进一步推到了 "**Context Repositories: Git-based Memory**"——把记忆当作可 diff、可回滚的版本化资源。

### 3.2 Stateful Agent：从 library 到 runtime

> "Your agents don't just *use* Letta for memory — they *run inside* Letta." — Letta Docs

Letta 不是一个被你的进程导入的库，而是你的 Agent 进程的**宿主**。它负责：

* **Agent 状态持久化**：Agent ID、memory blocks、recall index、tool definitions 都存在 Postgres + 向量库中
* **Loop 调度**：一次 send_message 实际触发一个内部 step loop（思考→工具→修改 memory→回复）
* **可观测性**：ADE（Agent Development Environment）提供 memory state 实时检视
* **多 Agent 编排**：每个 Agent 是独立 stateful 进程，可互相 message

这个定位非常关键——它意味着 Letta 把 **Agent 状态从应用层下沉到了基础设施层**，类似 Kubernetes 把"进程编排"从 Linux 下沉到集群层。

### 3.3 Sleep-time Compute：让 Agent 在空闲时"做梦"

2025 年 4 月 Letta 团队发表了 *Sleep-time Compute: Beyond Inference Scaling at Test-time*（[arXiv:2504.13171](https://arxiv.org/abs/2504.13171)）。核心思想：

> **在用户没有提问的"空闲时段"，让另一个 LLM 进程对已有 context 进行预处理，把推理结果写入 memory block；用户来问时，主 Agent 直接读这个 block，不再重新推理。**

实现上，Letta 暴露两个新工具：

* `rethink_memory(target_block)` — sleep-time agent 重写指定 block，整合新推理结果
* `finish_rethinking_memory()` — 终止 offline 推理 loop（最多 10 次 rethink）

性能数据（论文给出）：

| 指标 | 数值 |
|---|---|
| Stateful GSM-Symbolic：等准确率下 test-time 计算 | **降低 ~5×** |
| Stateful GSM-Symbolic：scale sleep-time 后准确率 | **+13%** |
| Stateful AIME：scale sleep-time 后准确率 | **+18%** |
| Multi-Query GSM-Symbolic：10 个 query 共享同一 context | **平均成本降低 2.5×** |
| SWE-Features：低 budget 下 test-time tokens | **降低 ~1.5×** |

> 经验法则：**当 query 可从 context 中预测（用 Llama2-70B logprob 衡量）时，sleep-time compute 效果最佳**。如果 query 完全不可预测，sleep-time 可能不如 test-time scaling。

这是长期记忆架构在 2026 年最大的范式跃迁之一——**记忆不再是被动存储，而是后台持续被 LLM 自身 refine 的动态资产**。

## 4. MemoryOS：把 OS 隐喻推到极致的三层分页

EMNLP 2025 主会议论文 *Memory OS of AI Agent*（[arXiv:2506.06326](https://arxiv.org/abs/2506.06326)）把 MemGPT 的双层扩展到三层：

```
┌────────────────────────────────────────────────────┐
│ Short-Term Memory (STM)                            │
│   - 单元：dialogue page = (query, response, ts)     │
│   - 结构：FIFO queue + dialogue chain               │
│   - 容量：固定长度，溢出按 FIFO evict               │
└────────────────────────┬───────────────────────────┘
                         │ (page out by similarity grouping)
┌────────────────────────▼───────────────────────────┐
│ Mid-Term Memory (MTM)                              │
│   - 单元：segment = 同主题 dialogue pages 集合      │
│   - 结构：Segmented Paging（OS 分段+分页）           │
│   - 评分：semantic + Jaccard keyword similarity     │
│   - 热度：retrieval freq × interaction × recency    │
└────────────────────────┬───────────────────────────┘
                         │ (heat ≥ threshold → promote)
┌────────────────────────▼───────────────────────────┐
│ Long-term Persona Memory (LPM)                     │
│   - User profile：static 属性 + dynamic KB + traits │
│   - Agent persona：跨会话稳定身份                    │
└────────────────────────────────────────────────────┘
```

### 4.1 关键创新：Segmented Paging

MemGPT 的 FIFO 是平铺的；MemoryOS 把语义相近的 dialogue pages **聚成 segment**，每个 segment 是一个分页单元。这解决了 MemGPT 的 topic mixing 问题——eviction 与 promotion 都以"主题块"为粒度，而非单条消息。

### 4.2 关键创新：Heat-based Promotion

从 MTM 升级到 LPM 不靠时间，而靠 **heat score**：

```
heat(segment) = α · retrieval_freq + β · interaction_volume + γ · exp(-λ · age)
```

热度高的 segment 进入 LPM，成为长期 persona 的一部分；冷的被淘汰。这本质上是 LRU + LFU 的混合策略。

### 4.3 Benchmark 数据（LoCoMo）

| 系统 | 平均 F1 | 平均 Token | 时序问题 F1 |
|---|---|---|---|
| MemGPT | 29.13 | 16,977 | 9.15 |
| **MemoryOS** | **36.23** | **3,874** | **20.02 (+118.8%)** |

> *Source*: 论文 Table 4-5（GPT-4o-mini 基线）。

**两个关键启示**：

1. **三层比双层显著更高效**：token 量降低 ~4.4×、F1 提升 24%
2. **时序问题（multi-session 跨时间） 是 hierarchical 架构的最大受益场景**

**选型建议**（2026 年视角）：

* **做产品级伴侣 / 个性化 Agent**：Letta（runtime + sleep-time + memory blocks）
* **做需要严格主题归并的多轮长对话系统**：参考 MemoryOS 的 STM/MTM/LPM 思想自建
* **做高吞吐 API 服务**（如 chatbot 平台）：Mem0（async writes + 多 scope tag）
* **想理解原理 / 自研**：先读 MemGPT 论文，再读 MemoryOS

## 6. 生产级 Agent Memory Runtime 蓝图

把上面三种范式吸收后，可以画出一套 2026 工业实践推荐的运行时架构：

```
                        ┌──────────────────┐
   user msg ────────────▶│  Inference Agent │◀────── core memory blocks
                        │   (test-time)    │           (always in ctx)
                        └────┬─────────┬───┘
                             │         │
                tool call    │         │  page-fault
            ┌────────────────▼─┐     ┌─▼─────────────────────┐
            │ Recall Search    │     │ Archival Search        │
            │ (历次对话, 时序)   │     │ (RAG: hybrid+rerank)   │
            └────────┬─────────┘     └─────────┬──────────────┘
                     │                         │
                     ▼                         ▼
            ┌──────────────────────────────────────────────┐
            │             Hierarchical Memory Store         │
            │  STM(FIFO) → MTM(segments) → LPM(profile)    │
            └────────────────────┬─────────────────────────┘
                                 │ async
                                 ▼
                        ┌──────────────────┐
                        │ Sleep-time Agent │  cron / idle-trigger
                        │  (background)    │  rethink_memory(...)
                        └──────────────────┘
```

### 6.1 数据模型骨架（Pydantic）

```python
from pydantic import BaseModel, Field
from typing import Literal
from datetime import datetime
import uuid

class MemoryBlock(BaseModel):
    """Letta-style memory block, always in main context."""
    block_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    label: Literal["persona", "human", "skills", "goals", "rethink"]
    content: str
    read_only: bool = False
    version: int = 0  # 用于 git-style diff
    updated_at: datetime = Field(default_factory=datetime.utcnow)


class DialoguePage(BaseModel):
    """MemoryOS-style STM unit."""
    page_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    user_query: str
    agent_response: str
    timestamp: datetime
    tools_called: list[str] = []
    embedding: list[float] | None = None


class Segment(BaseModel):
    """MTM topic-clustered group."""
    segment_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    pages: list[str]  # page_ids
    topic_summary: str
    heat: float = 0.0
    last_access: datetime = Field(default_factory=datetime.utcnow)


class PersonaProfile(BaseModel):
    """LPM user / agent persistent identity."""
    user_id: str
    static_attrs: dict          # 姓名、职业、长期兴趣
    dynamic_kb: list[str]       # 动态知识：偏好、决策、最近状态
    traits: dict[str, float]    # 演化中的特质打分
```

### 6.2 主流程伪代码

```python
class AgentRuntime:
    def __init__(self, agent_id, llm, store):
        self.agent_id = agent_id
        self.llm = llm
        self.store = store          # 持久化层（Postgres + 向量库）
        self.core_blocks = store.load_blocks(agent_id)
        self.stm = store.load_stm(agent_id, limit=20)

    def step(self, user_msg: str) -> str:
        # 1. 组装 main context
        main_ctx = self._compose_context(user_msg)
        
        # 2. LLM 推理 (允许工具调用)
        for turn in range(MAX_INNER_TURNS):
            resp = self.llm.invoke(
                main_ctx,
                tools=[
                    self.core_memory_replace,
                    self.recall_search,
                    self.archival_search,
                    self.archival_insert,
                ],
            )
            if not resp.tool_calls:
                break
            # 工具调用 → 把结果写回 main_ctx，进入下一轮
            main_ctx = self._apply_tool_results(main_ctx, resp.tool_calls)
        
        # 3. 持久化 STM
        page = DialoguePage(
            user_query=user_msg,
            agent_response=resp.content,
            timestamp=datetime.utcnow(),
        )
        self.stm.append(page)
        self.store.save_page(self.agent_id, page)
        
        # 4. STM 满 → 触发 segmentation 与 promotion
        if len(self.stm) > STM_CAPACITY:
            self._page_out_to_mtm()
        
        # 5. 异步触发 sleep-time（如果队列空闲）
        sleep_time_queue.enqueue(self.agent_id)
        
        return resp.content

    def _page_out_to_mtm(self):
        """STM → MTM：相似 page 聚成 segment。"""
        old_pages = self.stm.pop_oldest_n(STM_EVICT_BATCH)
        segments = cluster_by_similarity(old_pages, threshold=0.75)
        for seg in segments:
            existing = self.store.find_similar_segment(seg.topic_summary)
            if existing:
                existing.pages.extend(p.page_id for p in seg.pages)
                self.store.save_segment(existing)
            else:
                self.store.save_segment(seg)

    def _promote_to_lpm(self):
        """MTM → LPM：热度高的 segment 升级为 persona 知识。"""
        hot = self.store.top_segments_by_heat(self.agent_id, k=5)
        for seg in hot:
            if seg.heat > LPM_THRESHOLD:
                summary = self.llm.summarize_for_persona(seg)
                self.store.update_persona(self.agent_id, summary)
                seg.heat = 0  # 重置防止反复升级
```

### 6.3 Sleep-time Worker

```python
async def sleep_time_worker(agent_id: str, store):
    """空闲时段后台改写 memory blocks，摊销 inference 成本。"""
    blocks = store.load_blocks(agent_id)
    rethink = next(b for b in blocks if b.label == "rethink")
    
    sleep_agent = build_sleep_time_agent(
        tools=[rethink_memory_block, finish_rethinking_memory],
        max_calls=10,
    )
    
    # 把当前所有 STM/MTM 摘要喂给 sleep-time agent
    context = store.compose_offline_context(agent_id)
    
    new_block_content = await sleep_agent.run(
        f"基于以下对话历史，预先推理用户可能的下次提问，"
        f"把可复用的结论 / 中间结果 / 修正写入 rethink_memory_block:\n{context}"
    )
    
    rethink.content = new_block_content
    rethink.version += 1
    store.save_block(rethink)
```

## 7. 反模式与陷阱

| 反模式 | 表现 | 修正 |
|---|---|---|
| **All-in-prompt** | 把所有历史塞 1M context | 用三层分页 + heat eviction |
| **Memory write hot path** | 每次回答都同步写记忆 → p95 暴涨 | async 写入 + batch flush（Mem0 推荐） |
| **No scope** | 多用户共用一个记忆空间 | 强制 user_id / agent_id / session_id scope tag |
| **Topic mixing** | 平铺 FIFO 导致检索串话 | segmentation by topic（MemoryOS） |
| **Stale persona** | LPM 写入后从不更新 | 设置 staleness check + 周期 reconciliation |
| **No versioning** | 记忆被覆盖后无法回滚 | git-style block versioning（Letta Context Repositories） |
| **Sleep-time on unpredictable queries** | 改写无关内容反而干扰 | 仅当 query logprob 高时启用 |
| **Skip eval** | 上线前不跑 LoCoMo / LongMemEval | 强制 CI 集成评估 |
