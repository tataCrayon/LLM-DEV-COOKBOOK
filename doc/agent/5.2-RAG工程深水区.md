# 5.2 RAG 工程深水区：Chunking / Hybrid Search / Reranking / GraphRAG

> **本篇定位**：5.1 把 Agent 记忆的"分类学"讲清楚了——Semantic Memory 在生产里通常对应一套 RAG 系统。但绝大多数团队的 RAG 仍停留在"OpenAI Embedding + Pinecone + Top-K=5"的入门配置，被一系列**基础却致命**的问题反复折磨：召回不准、上下文割裂、关键词查不到、跨文档推理失败。本篇沿着检索流水线四个层次（Chunking → Embedding/Indexing → Retrieval → Reranking），结合 2024-2026 出现的几条范式级新方法（**Anthropic Contextual Retrieval**、**Late Chunking**、**RRF**、**HippoRAG2 / Microsoft GraphRAG**），把"为什么我的 RAG 不行"的工程根因和应对策略系统化讲清楚。

## 0. 开场问题：为什么我的 RAG 表现这么差？

设想这样一个真实场景：你给团队搭了一个内部知识库 RAG，技术栈是教科书式的 "**OpenAI text-embedding-3-large + Pinecone + top-k=5**"。上线一周后，反馈如下：

- **症状 A**：用户问 "ACME Corp 在 Q2 2023 营收增长多少？"，召回的 chunk 是 "The company's revenue grew by 3% over the previous quarter."——chunk 里**没有公司名、没有季度信息**，embedding 相似度排不上去。
- **症状 B**：用户搜 "TS-999 错误码"，**完全召回不到**——明明文档里有这个字符串。
- **症状 C**：用户问 "Project X 的预算是多少？" 召回了 5 个文档片段，每个都说了一部分，但拼起来 LLM 答不全——上下文被切断了。
- **症状 D**：用户问 "我们公司 2024 年所有项目的总预算"——这是个**多文档聚合**问题，向量检索完全没法处理。
- **症状 E**：上线 3 个月后，索引里有 200 万 chunk，**top-k=5 检索出来 60% 是无关的**——召回宽，精度低。

这五个症状不是 Bug，是 RAG **架构不完整**——它们分别暴露了检索流水线四个层次的关键缺陷：

| 症状 | 根因 | 对应解决方案 |
|---|---|---|
| A：chunk 上下文割裂 | Chunking 没保留全局上下文 | **Contextual Retrieval / Late Chunking** |
| B：精确字符串召回失败 | 纯向量检索丢失词法信号 | **Hybrid Search（BM25 + Dense）** |
| C：相关 chunk 拼接不全 | Top-K 召回不全 + 跨 chunk 不连续 | **大 K 召回 + Reranker** |
| D：多文档聚合失败 | 向量检索的拓扑盲性 | **GraphRAG / HippoRAG** |
| E：精度被召回淹没 | 缺少 Reranker 层 | **Reranking + 召回-精排两阶段** |

**第一性原理**：生产级 RAG 不是"向量 + Top-K"，而是一条**流水线**：

```
Documents
   │
   ├─► Chunking（保留上下文）
   ├─► Embedding（高质量稠密 + 稀疏 BM25 双索引）
   ├─► Indexing（vector + keyword + 可选 graph）
   │
   ▼
Query
   │
   ├─► Retrieval（hybrid / RRF 融合，召回 top-100~150）
   ├─► Reranking（cross-encoder 精排，缩到 top-20）
   ├─► (可选) Graph augmentation（多跳推理）
   │
   ▼
Generation（LLM with tight context）
   │
   ▼
Evaluation（RAGAS / DeepEval：Faithfulness / Relevancy / Context Precision/Recall）
```

每一层都有**经过测量**的最佳实践和失败模式。下面逐层展开。

## 1. Chunking：从"切开"到"保留上下文"

### 1.1 七种 Chunking 策略与适用面

[Firecrawl 2026 chunking strategies guide](https://www.firecrawl.dev/blog/best-chunking-strategies-rag) 总结了当前在用的七种主流策略：

| 策略 | 召回基准 | 成本 | 适用场景 | 反模式 |
|---|---|---|---|---|
| **Fixed-size**（固定字符/token） | 低 | 极低 | MVP / 原型 | 切断句子、忽略结构 |
| **Recursive Character**（按 `\n\n`/`\n`/空格递归） | 85-90% recall@400 | 低 | **80% 应用的默认值** | 极少 |
| **Sentence-based**（NLP 句法切分） | 中 | 低 | Q&A、对话 | 格式不规范文本 |
| **Page-level**（按页切分） | NVIDIA 2024 基准 0.648 准确率 | 低 | PDF 报告、财报 | 非分页内容 |
| **Semantic**（按语义相似度切分） | 比 recursive 高 2-3pp | 高（每句嵌入） | 高价值长文档 | NAACL 2025 称"成本不值" |
| **LLM / Agentic** | 最高 | 极高 | 高价值实验 | 大规模时不可行 |
| **Late Chunking**（先嵌入全文后切分） | NFCorpus +6.5 nDCG@10 | 中（需长 ctx 模型） | 跨指代/合同/论文 | 短文档无收益 |

**2026 年默认配置**：
- Recursive Character Splitting，**400-512 tokens**，10-20% overlap（注意：[2026.01 系统性分析](https://www.firecrawl.dev/blog/best-chunking-strategies-rag) 发现 "overlap 没有可测收益"，先测再用）
- Factoid 查询：256-512 tokens
- 分析类查询：1024+ tokens
- 文档类型路由：PDF→page-level、Web→recursive、Code→code-aware separators (`\n\nclass `、`\n\ndef `)

**反直觉的一条**（[Vecta 2026.02 基准](https://www.firecrawl.dev/blog/best-chunking-strategies-rag)）：**recursive 512-token 分块（69% 准确率）显著优于 semantic 分块（54%）**——不要默认认为"语义分块更聪明"，semantic chunking 的边界检测在嘈杂数据上反而引入噪声。

### 1.2 Anthropic Contextual Retrieval：2024 年最重要的范式更新

来自 [Anthropic 2024 Contextual Retrieval](https://www.anthropic.com/engineering/contextual-retrieval) 的核心洞察：**chunk 在切出来之前，必须由 LLM 给它生成一段"上下文"，prepend 到 chunk 文本前再做 embedding 和 BM25 索引**。

#### 提示词模板（Anthropic 官方）

```
<document>
{{WHOLE_DOCUMENT}}
</document>
Here is the chunk we want to situate within the whole document
<chunk>
{{CHUNK_CONTENT}}
</chunk>
Please give a short succinct context to situate this chunk within the
overall document for the purposes of improving search retrieval of the
chunk. Answer only with the succinct context and nothing else.
```

效果（用 1 - recall@20 衡量召回失败率，baseline 5.7%）：

| 配置 | 失败率 | 相对降低 |
|---|---|---|
| Baseline（vector only） | 5.7% | — |
| **+ Contextual Embeddings** | 3.7% | **−35%** |
| **+ Contextual BM25** | 2.9% | **−49%** |
| **+ Reranking** | 1.9% | **−67%** |

#### 关键工程数字

- 生成 context 的 LLM：Claude 3 Haiku
- 输出长度：50-100 tokens
- **预处理一次性成本：每百万 document tokens 仅 $1.02**——靠的是 Anthropic 的 prompt caching（document 缓存一次，所有 chunk 复用）
- 顶级 embedding：Voyage / Gemini Text 004（Anthropic 实测最佳）

**实战建议**：做 RAG 一定要测一次 contextual retrieval 的提升幅度。$1.02/M tokens 对绝大多数知识库都是**一次性可承受成本**，而把召回失败率降低 49%-67% 几乎是其他所有工程优化加起来都很难达到的提升。

### 1.3 Late Chunking：对长文档的另一条路

[Jina AI 2024 Late Chunking](https://jina.ai/news/late-chunking-in-long-context-embedding-models/) 提出的另一种方案：

```
传统流程：
  Document → split into chunks → embed each chunk

Late Chunking：
  Document → embed FULL DOCUMENT (token-level) → mean-pool by chunk boundaries
```

**机制**：chunk 边界**不再决定 attention 范围**。每个 token 都看到了全文，最后才按 chunk boundary 做 mean pooling——这样 chunk embedding **天然带全文上下文**。

效果：NFCorpus nDCG@10 +6.5 分；但短文档（如 Quora）零收益。

**适用边界**：
- 必须用支持 mean pooling 的长上下文 embedding（jina-embeddings-v3 8192 ctx 是代表）
- 文档必须能塞进 context window（>32K 的合同/论文不行，需要先 split）
- 跨指代/代词/缩写多的文档（合同、技术文档、学术论文）

### 1.4 Contextual Retrieval vs Late Chunking 决策

| 场景 | 选择 |
|---|---|
| 短文档（< embedding ctx） | **Late Chunking**（更便宜、效果接近） |
| 长文档 / 不能整体塞进 ctx | **Contextual Retrieval** |
| 高价值知识库（一次预处理永久受益） | **两者叠加**（Anthropic 论文表明可叠加） |
| 实时新增、无法离线预处理 | 只能用 Late Chunking |
| 成本敏感、文档量极大 | Late Chunking（无 LLM 调用） |

## 2. Hybrid Search：词法 + 语义双轨

### 2.1 为什么需要 Hybrid

向量检索（dense）擅长**语义相似**，但对以下场景几乎无能：

- **精确字符串**："TS-999"、"order #12345"、"v3.2.1"
- **专有名词 / 缩写**："ACME"、"NATO"、"K8s"
- **代码符号**：`useState`、`handleClick`
- **罕见词 / 长尾术语**：embedding 训练分布之外

BM25（[完整 BM25 公式](https://superlinked.com/blog/optimizing-rag-with-hybrid-search-reranking)）正是补这一段的——稀疏检索（sparse retrieval）按词法相似度打分，对精确匹配近乎完美。

```
BM25(q, D) = Σ_{q_i ∈ q} IDF(q_i) · (TF(q_i, D) · (k₁ + 1)) /
                                     (TF(q_i, D) + k₁ · (1 - b + b · |D|/avgdl))
```

### 2.2 融合：RRF vs 加权平均

两类融合策略各有适用：

#### Reciprocal Rank Fusion (RRF)

```
RRF(d) = Σ_systems 1 / (k + rank_system(d))
```

`k` 通常取 60。优点：**不需要分数尺度对齐**（dense 和 BM25 的 score 范围完全不同），只看排名。Elasticsearch 8.9+、Weaviate、Qdrant 都内置 RRF。

#### 加权平均（Alpha Weighting）

```
H(q, d) = (1 - α) · BM25_score(q, d) + α · vector_score(q, d)
```

- α = 1：纯向量
- α = 0：纯关键词
- Weaviate 默认 α = 0.5

但**加权前必须做 score normalization**（min-max 或 z-score），否则尺度差异会让一边完全压制另一边。这是工程上最常见的坑。

### 2.3 决策矩阵

| 场景 | 推荐 |
|---|---|
| 通用知识库（混合查询模式） | **RRF**（默认值，无需调参） |
| 技术文档、代码（含大量精确符号） | **Hybrid，BM25 权重提高（α=0.3-0.4）** |
| 纯自然语言问答 | 纯 dense 即可 |
| 多语言场景 | 测多语言 BM25 + 多语言 embedding |
| 罕见缩写 / 专有名词高 | **必须 Hybrid** |

[Anthropic 论文](https://www.anthropic.com/engineering/contextual-retrieval) 在自家场景实测：从 Contextual Embeddings 单跑（−35% 失败率）升级到 + Contextual BM25，进一步降到 −49% 失败率——**BM25 提供的 14pp 提升基本来自精确字符串场景**。

## 3. Reranking：召回-精排两阶段架构

### 3.1 为什么必须做 Reranking

Embedding 是**双塔模型**（query 和 doc 独立编码），效率高但精度有上限。Reranker 是 **cross-encoder**（query 和 doc 拼起来一起进模型），精度高但慢——所以两阶段：

```
Query
  │
  ├─► Retrieval（hybrid，召回 top-100~150）  ← 快、保召回
  │
  ├─► Reranker（cross-encoder，精排 → top-20） ← 慢、保精度
  │
  └─► LLM 生成
```

[Anthropic 数据](https://www.anthropic.com/engineering/contextual-retrieval)：召回 150 → 精排 20，是他们论文里的实测最佳配置；加 reranker 把失败率从 2.9% 进一步降到 1.9%。

### 3.2 2026 年 Reranker 选型矩阵

参考 [ZeroEntropy 2026 Reranker Guide](https://zeroentropy.dev/articles/ultimate-guide-to-choosing-the-best-reranking-model-in-2025/)：

| 类型 | 代表 | NDCG@10 | 延迟 | 成本 / M tokens | 适用 |
|---|---|---|---|---|---|
| **Cross-Encoder** | Cohere Rerank 3.5 / Voyage / Jina / **ZeroEntropy** | 0.85+ | 200ms-2s | $0.025-0.050 | **生产搜索 / RAG（默认）** |
| **Pointwise LLM** | gpt-4o-mini as judge | 0.70+ | 1-3s | $0.50-5 | 原型阶段 |
| **Listwise LLM** | RankGPT / RankZephyr | 0.90+（小 k 时最高） | >5s | $0.50-5 | 研究、专业小领域 |

**关键洞察**：cross-encoder 的"95% 的 LLM reranker 准确率 + 3× 延迟优势"——生产环境绕不开 cross-encoder。LLM reranker 在用户响应路径上不可行（用户 3 秒后就放弃）。

### 3.3 主流 Cross-Encoder 横评

| 提供方 | 模型 | 特点 | 适用 |
|---|---|---|---|
| **Cohere** | Rerank 3.5 | 闭源、强多语言、低延迟 | 通用首选 |
| **Voyage** | rerank-2.5 | 指令式、agent 场景调优 | 对话 / Agent |
| **Jina** | jina-reranker-v2 / -m0（多模态） | 开源权重、PDF/图像友好 | 自部署、多模态 |
| **ZeroEntropy** | zerank-1 | 标榜行业最高精度，灵活部署 | 高精度需求 |
| **Salesforce** | LlamaRank 4B / 8B | Llama 基底、本地部署 | 数据主权场景 |
| **BAAI** | bge-reranker-v2-m3 | 完全开源、多语言 | 自主可控 / 中文 |

### 3.4 K 值选型（候选数）

ZeroEntropy 给出按域调优建议：

| 场景 | k（候选数） | 理由 |
|---|---|---|
| LLM 聊天 | ~50 | 速度优先 |
| Web 搜索 | 100-200 | 全面性优先 |
| 大多数应用 sweet spot | **50-75** | 性价比最佳 |
| Finance（合规高精） | 100-150 | 不能漏 |
| E-commerce | 50-75 | 转化加权 |
| Q&A | 25-50 | 答案聚焦 |

成本论据：用 reranker 把 75→20 candidate，再喂 GPT-4o，**省 72% 成本 + 保留 95% 精度**——这是为什么生产 RAG 必须有 reranker 的财务理由。

## 4. GraphRAG：什么时候真的值得？

向量检索的根本缺陷：**对图拓扑盲**。问 "公司 A 的所有子公司的 2024 营收总和" 这类**多跳聚合**问题，向量检索只能召回相关片段，无法做"实体跳转 + 聚合"——这就是 GraphRAG 的用武之地。

### 4.1 GraphRAG 三种主流实现

| 系统 | 来自 | 核心机制 | 适用 |
|---|---|---|---|
| **Microsoft GraphRAG** | Microsoft Research, 2024 | LLM 抽实体/关系 → 社区检测 → 层级摘要 → local/global 双查询 | 跨文档 summarization、社区级 |
| **HippoRAG2** | OSU, 2024 | 受海马体启发，PageRank + 知识图谱节点 | 多跳 single-fact |
| **LightRAG** | HKU, 2024 | 双层（实体/关系）retrieval、低成本 | 折中方案 |
| **LinearRAG** | 2025 | 线性化图、低开销 | 大规模索引 |
| **Fast-GraphRAG** | 2024 | 极简实现 | 快速验证 |

### 4.2 WildGraphBench 实测数据（2026.02）

[arXiv 2602.02053 WildGraphBench](https://arxiv.org/html/2602.02053v1) 用 1,197 道题（Wikipedia 12 个 top-level 主题，外部引用 URL 作为嘈杂语料）实测：

| 方法 | Single-fact 准确率 | Multi-fact F1 | Summary F1 |
|---|---|---|---|
| NaiveRAG（baseline） | 66.87 | 35.08 | **15.84** ← 最佳 |
| BM25 | 41.38 | 20.94 | 12.66 |
| **HippoRAG2** | **71.51** ← 最佳 | 39.27 | 13.39 |
| MS GraphRAG (global) | 56.52 | **47.64** ← 最佳 | 13.78 |
| LightRAG (hybrid) | 61.32 | 40.84 | 14.61 |
| LinearRAG | 47.53 | 35.60 | 9.69 |
| Fast-GraphRAG | 35.83 | 25.65 | 10.56 |

**三个反直觉结论**：

1. **Single-fact 查询**：HippoRAG2 仅领先 NaiveRAG 4.6 分，**绝大多数 GraphRAG 方法反而落后于 NaiveRAG**——单点查询 GraphRAG 不值得。
2. **Multi-fact 聚合**：MS GraphRAG（global）领先 NaiveRAG 12.5 分（47.64 vs 35.08），这是 GraphRAG 真正的甜区。
3. **Summarization**：所有 GraphRAG 都**不如 NaiveRAG**。原因：GraphRAG 流水线（实体抽取、稀疏化、遍历预算）在嘈杂长上下文下"过度关注高层陈述、牺牲细粒度召回"。

### 4.3 GraphRAG 决策矩阵

| 场景 | 推荐 |
|---|---|
| 单事实查询、Q&A | **NaiveRAG / Hybrid + Rerank**（GraphRAG 不值得） |
| 跨文档多跳推理（"X 和 Y 的关系是什么"） | **HippoRAG2 / MS GraphRAG（local）** |
| 高层社区摘要（"公司治理结构概览"） | **MS GraphRAG（global）** |
| 长文档摘要 | **NaiveRAG**（GraphRAG 反而拖累） |
| 强结构化领域（生物、金融、合规） | **GraphRAG / KG-RAG**（领域 ontology 已存在） |
| 探索性新领域 | **先 NaiveRAG**，等观察到多跳需求再加 |

**[2026 GraphRAG Buyer's Guide](https://medium.com/@tongbing00/graphrag-in-2026-a-practical-buyers-guide-to-knowledge-graph-augmented-rag-43e5e72d522d) 的核心建议**：GraphRAG 是"vector RAG 已经吃到瓶颈"后的第二阶段优化。一上来就上 GraphRAG 的成本（构建图、维护 ontology、调试遍历）几乎都不值得。

### 4.4 HybridRAG：结合 vector 和 graph

实际生产里，最 robust 的配置是 **HybridRAG**：

```
Query
  ├─► Vector Retrieval（top-50）
  ├─► Graph Traversal（从 query 实体出发 1-2 跳，top-20 三元组）
  ├─► RRF / 加权融合
  └─► Reranker → LLM
```

这条路线在 [2026 HybridRAG 实测](https://arxiv.org/html/2604.01733v1) 中表现优于纯 vector 或纯 graph。

## 5. 评估：RAGAS / DeepEval 与生产指标

### 5.1 四大核心指标

无论用 [RAGAS](https://docs.ragas.io/) 还是 [DeepEval](https://deepeval.com/)，生产级 RAG 评估都绕不开四个指标：

| 指标 | 定义 | 衡量什么 | 失败模式 |
|---|---|---|---|
| **Faithfulness** | LLM 答案是否能从检索的 context 推出 | 是否幻觉 | 答案 = 真实但 context 没说 |
| **Answer Relevancy** | LLM 答案是否回应了 query | 离题度 | 答相关知识但没回答问题 |
| **Context Precision** | 检索的 context 中相关比例 | 检索精度 | 噪声多 |
| **Context Recall** | 真实答案需要的 context 是否都召回了 | 检索召回 | 漏关键文档 |

### 5.2 二维诊断矩阵

| Context Recall | Context Precision | 诊断 | 应对 |
|---|---|---|---|
| 高 | 高 | 检索系统 OK | 看 generation 层 |
| 高 | **低** | 检索召回太宽 | **加 Reranker / 降 K** |
| **低** | 高 | 检索召回不足 | **改 Chunking / 升级 Embedding / 加 Hybrid** |
| 低 | 低 | 系统性烂 | 重做 |
| Faithfulness 低 + Context 高 | — | LLM 幻觉 | **prompt 加 "根据 context"约束 / 换更强模型** |
| Answer Relevancy 低 + Faithfulness 高 | — | 答非所问 | **改 prompt / query rewriting** |

### 5.3 RAGAS vs DeepEval

| 维度 | RAGAS | DeepEval |
|---|---|---|
| 起点 | 学术（Exploding Gradients） | 工业（Confident AI） |
| 指标数量 | 4 核心 + 扩展 | 14+ |
| LLM-as-judge | ✅ | ✅ |
| 单元测试集成 | 一般 | **强**（pytest 风格） |
| CI/CD | 需自己写 | 内置 |
| 推荐 | 离线 benchmark | **生产 CI** |

## 6. 端到端生产蓝图

把上面所有内容拧成一条**可直接落地**的生产架构：

```python
"""
生产级 RAG 蓝图
- Chunking: Recursive 512 tokens (PDF 走 page-level)
- Pre-processing: Anthropic Contextual Retrieval（Claude Haiku 生成 context）
- Embedding: Voyage-3 / Gemini text-004（dense）+ BM25（sparse）
- Indexing: Qdrant（dense）+ OpenSearch（sparse）+ 可选 Neo4j（graph）
- Retrieval: Hybrid via RRF, top-100
- Reranking: Cohere Rerank 3.5 → top-20
- Generation: GPT-4o / Claude Sonnet with strict citation prompt
- Evaluation: DeepEval in CI + 5% online LLM-as-judge sampling
"""

from typing import List
from dataclasses import dataclass


# ---- 1. Chunking + Contextual Pre-processing ----
def preprocess_document(doc: str, doc_id: str) -> List["Chunk"]:
    raw_chunks = recursive_split(doc, target_size=512, overlap=0)  # 已被验证 overlap 可省
    contextualized = []
    for chunk in raw_chunks:
        # Anthropic prompt + prompt caching（document 整体缓存）
        context = anthropic_haiku.invoke(
            CONTEXTUAL_PROMPT.format(WHOLE_DOCUMENT=doc, CHUNK_CONTENT=chunk),
            cache_document=True,  # 关键：90% 节省
        )
        text_for_index = f"{context}\n\n{chunk}"
        contextualized.append(Chunk(
            id=f"{doc_id}#{hash(chunk)}",
            doc_id=doc_id,
            text=text_for_index,
            original=chunk,
            context=context,
        ))
    return contextualized


# ---- 2. 双索引写入 ----
def index_chunks(chunks: List["Chunk"]):
    # Dense
    embeddings = voyage.embed([c.text for c in chunks], model="voyage-3")
    qdrant.upsert("rag_dense", chunks, embeddings)
    # Sparse
    opensearch.bulk_index("rag_sparse", chunks)


# ---- 3. Hybrid Retrieval with RRF ----
def hybrid_retrieve(query: str, top_n: int = 100) -> List["Chunk"]:
    q_emb = voyage.embed([query], model="voyage-3")[0]
    dense_hits = qdrant.search("rag_dense", q_emb, limit=top_n)
    sparse_hits = opensearch.search("rag_sparse", query, size=top_n)
    return rrf_merge(dense_hits, sparse_hits, k=60)[:top_n]


def rrf_merge(list_a, list_b, k=60):
    scores = {}
    for rank, hit in enumerate(list_a):
        scores[hit.id] = scores.get(hit.id, 0) + 1 / (k + rank + 1)
    for rank, hit in enumerate(list_b):
        scores[hit.id] = scores.get(hit.id, 0) + 1 / (k + rank + 1)
    by_id = {h.id: h for h in list_a + list_b}
    return [by_id[i] for i, _ in sorted(scores.items(), key=lambda x: -x[1])]


# ---- 4. Reranking ----
def rerank(query: str, candidates: List["Chunk"], top_k: int = 20) -> List["Chunk"]:
    resp = cohere.rerank(
        model="rerank-english-v3.5",
        query=query,
        documents=[c.text for c in candidates],
        top_n=top_k,
    )
    return [candidates[r.index] for r in resp.results]


# ---- 5. 生成（strict citation）----
SYSTEM_PROMPT = """You are a RAG assistant. Answer ONLY using the provided context.
Cite each claim with [chunk_id]. If the context is insufficient, say "I don't know."
Do NOT use external knowledge."""

def generate(query: str, top_chunks: List["Chunk"]) -> str:
    context = "\n\n".join(f"[{c.id}] {c.original}" for c in top_chunks)
    return llm.invoke(
        system=SYSTEM_PROMPT,
        user=f"Context:\n{context}\n\nQuestion: {query}",
    )


# ---- 6. 端到端 ----
def rag(query: str) -> str:
    candidates = hybrid_retrieve(query, top_n=100)
    top = rerank(query, candidates, top_k=20)
    return generate(query, top)
```

## 7. 反模式清单（按出现频率排序）

| # | 反模式 | 损害 | 正确做法 |
|---|---|---|---|
| 1 | "OpenAI Embedding + Pinecone + top-K=5" 一把梭 | 召回失败率 5-10× 于优化版 | 至少加 Hybrid + Reranker |
| 2 | 不做 Reranking | 60-90% 上下文是噪声 | 100→20 是 RAG 默认配置 |
| 3 | 纯向量（无 BM25） | 精确字符串/缩写零召回 | RRF 融合 |
| 4 | Chunk 直接切 1000-tokens 不做 contextual | 关键词丢失上下文 | Contextual Retrieval 或 Late Chunking |
| 5 | 一上来就 GraphRAG | 成本高、单事实查询反而退化 | 先 NaiveRAG，看到多跳需求再上 |
| 6 | 没有 golden set / 没有评估 | 改完不知道好不好 | 至少 100 条 golden set + DeepEval CI |
| 7 | Hybrid score 不做归一化就加权 | 一边完全压制另一边 | RRF 或 min-max 归一化 |
| 8 | 用 LLM 做 reranker 走线上 | P95 >5s | 用 cross-encoder reranker |
| 9 | 不区分 query 类型用同一套 K | factoid 浪费、analytical 召回不全 | Query 分类后路由 |
| 10 | 索引一次永不更新 | 文档变更后命中失效 | 增量索引 + 删除策略 |
| 11 | Chunk overlap 设 30%-50% | 索引膨胀、可能无效 | 先测后定，多数场景可降到 0-10% |
| 12 | Embedding 模型一锁三年 | 落后业界 30+pp | 半年评估一次（Voyage / Gemini / OpenAI / 开源） |

## 8. 渐进式建设路线图

| 阶段 | 配置 | 上线条件 | 预计 Recall@20 |
|---|---|---|---|
| **R0 MVP** | OpenAI embedding + Pinecone + top-5 | 验证可行性 | 70-85% |
| **R1 + Hybrid** | + BM25 / OpenSearch + RRF | 用户抱怨"搜不到名词" | 85-92% |
| **R2 + Reranker** | + Cohere Rerank（100→20） | 上下文噪声多 | 92-96% |
| **R3 + Contextual** | + Anthropic Contextual Retrieval | 跨 chunk 上下文割裂明显 | 96-98% |
| **R4 + 评估闭环** | + DeepEval CI + golden set + online sampling | 准备规模化 | — |
| **R5 + Graph（可选）** | + HippoRAG2 / MS GraphRAG | 多跳查询占比 >20% | Multi-fact +12pp |
| **R6 + 多模态** | + 多模态 embedding（Voyage multimodal / Jina v4） | 含图/PDF/视频 | — |

**实操原则**：每个阶段停留 1-3 个月观察，不要跳级。R3 之后的提升空间最有限——真正的差异往往在 R0→R1→R2 这三步。
