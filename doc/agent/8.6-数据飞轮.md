# 8.6 数据飞轮：Logs → Auto-labeling → Eval 集 → SFT/DPO/RFT → 持续改进闭环

> "生产数据是 Agent 的护城河——但只有当你能把它变成训练信号时才是。"

## §0 真问题：为什么大多数团队的"日志"不是数据资产，而是负债？

来看四个典型场景。

**场景 A**：一个 RAG Agent 上线 6 个月，累计 200 万条对话日志。团队想用这些数据微调一个开源模型替代 GPT-4o 以降本，结果发现：日志里没有"用户满意"的明确信号，只有最后一次点击或会话结束；正反例分布极度不均（错误案例埋在长尾里）；每条 trace 包含 30+ 步工具调用，找不到关键决策步。最终团队雇了一个外包标注团队，花了 4 个月才整理出 10K 高质量样本——比基础模型微调成本还高。

**场景 B**：一家 fintech 的客服 Agent 切换了底模，离线 eval 准确率 +3%，上线一周用户投诉量翻倍。原因：离线 eval 集是 6 个月前的版本，覆盖不了最近爆款活动引入的新意图。**Eval 集没有跟着生产数据演化**。

**场景 C**：另一家公司搭了"日志→人工标注→SFT"的标准流水线，每月迭代一次。但 6 个月后发现：模型在新版上越改越窄，对边缘 case 鲁棒性下降。复盘发现，**标注员只标 LLM 已经处理得不错的"简单错误"**，真正难的歧义 case 被默认跳过——飞轮在加固"模型已会的部分"，对未覆盖空间反而失忆。

**场景 D**：Airbnb 的搜索 Agent 团队，对一个新功能"客服建议"做端到端改进。在 AITL（AI-in-the-Loop）模式之前，从日志收集→人工评估→SFT→上线，cycle 是 3 个月。引入 LLM 虚拟评判员（HITL r>0.90）+ 自动化训练流水线后，**cycle 缩短到几周**，Recall@75 +11.7%，Citation +38.1%，用户偏好 60.12% vs 基线 33.32%（[arXiv:2510.06674](https://arxiv.org/abs/2510.06674)）。

四个场景揭示一件事：**数据飞轮不是把日志倒进 S3 就能转起来的。它是一个工程系统，包含 5 个齿轮：捕获 → 筛选 → 标注 → 评估 → 训练，每一齿都得啮合**。任何一环失效，飞轮就空转——甚至倒转（场景 C）。

本文回答：怎么把 Agent 的生产 trace 变成可持续改进的训练信号？

## §1 第一性原理：飞轮是什么、不是什么

### 1.1 数据飞轮的本质：从"调用即燃料"到"调用即资产"

传统模型时代，调用成本是固定支出。Agent 时代不一样——**每一次调用都自带 5 类信号**：

| 信号 | 出处 | 直接用途 |
|---|---|---|
| **输入分布** | 用户 query + 上下文 | 监控漂移、覆盖率 |
| **轨迹结构** | tool calls / 步数 / 失败 | 路由优化、子任务模式 |
| **结果质量** | 用户反馈 / 下游事件 | 偏好标签、reward |
| **失败模式** | 异常、空回、人工 escalate | 反例集、护栏训练 |
| **成功要素** | 用户 adopt 的版本 | 正例集、参考答案 |

飞轮的目标，是让这 5 类信号**自动沉淀为可执行的训练/eval 数据，而不需要每次都召集外包标注大军**。

**关键判断**：如果你的日志只能用于"看 dashboard"，没能反推训练或 eval 数据，那它就是负债——存储成本天天涨，没创造价值。

### 1.2 飞轮的 5 个齿轮

```
┌─────────────────────────────────────────────────────────┐
│  Production Agent                                       │
│       │                                                 │
│       ▼                                                 │
│  ① Trace Capture（workload_id + 全 step 结构化）          │
│       │                                                 │
│       ▼                                                 │
│  ② Filtering & Routing（rule + LLM virtual judge 过滤）  │
│       │                                                 │
│       ▼                                                 │
│  ③ Auto-labeling（LLM-as-judge 做粗标，HITL 抽检校准）    │
│       │                                                 │
│       ├──→ Eval Set (class-aware stratified split)      │
│       │                                                 │
│       └──→ Training Set                                 │
│              │                                          │
│              ▼                                          │
│  ④ Training Loop（SFT → DPO/ORPO → RFT/GRPO）            │
│              │                                          │
│              ▼                                          │
│  ⑤ Replacement Decision（A/B + HITL，不是 autopilot）    │
│              │                                          │
│              └──→ 回到 Production                       │
└─────────────────────────────────────────────────────────┘
```

每一齿都有失败模式（见 §6），整体设计哲学是 NVIDIA Data Flywheel Blueprint 的一句话——**"Flywheel as a flashlight, not an autopilot"**（飞轮是手电筒不是自动驾驶）：它高亮哪些模型可以被替换、哪些数据要补，但**替换决策永远由人来定**。

### 1.3 飞轮 vs 离线训练 vs 持续学习

| 维度 | 离线训练 | 数据飞轮 | 持续学习（online） |
|---|---|---|---|
| 数据来源 | 标注 dataset | 生产 trace + LLM 标注 + 人工抽检 | 实时流 |
| 更新频率 | 季度/年 | 周/月 | 实时 |
| 风险 | 数据陈旧 | 反馈循环偏置 | 灾难性遗忘、prompt 投毒 |
| 成本结构 | 一次性大额 | 持续中等 | 持续高（监控+护栏） |
| 适合阶段 | 冷启动 | 已上线、有日志 | 极少数高频极成熟场景 |

**结论**：99% 的团队应该在数据飞轮，而不是 online learning。后者的护栏成本+偏置风险，远超大多数业务收益。

## §2 齿轮 ①：Trace 捕获——schema 决定下游天花板

### 2.1 别再写 `print(response)` 了：production trace schema

NVIDIA Data Flywheel Blueprint 的 schema 是个好起点（[GitHub](https://github.com/NVIDIA-AI-Blueprints/data-flywheel)）：

```json
{
  "trace_id": "abc-123",
  "timestamp": "2026-06-25T10:30:00Z",
  "workload_id": "customer_support_v2.intent_classifier",
  "client_id": "tenant_xyz",
  "request": {
    "model": "gpt-4o",
    "messages": [...],
    "tools": [...],
    "temperature": 0.0
  },
  "response": {
    "content": "...",
    "tool_calls": [...],
    "finish_reason": "stop",
    "usage": {"prompt_tokens": 1200, "completion_tokens": 80}
  },
  "outcome": {
    "user_action": "adopted | edited | rejected | ignored",
    "downstream_event": "ticket_resolved | escalated | abandoned",
    "latency_ms": 850,
    "cost_usd": 0.024
  },
  "trace_links": ["parent_trace_id", "next_trace_id"]
}
```

四个关键字段决定下游能力：

| 字段 | 缺失代价 |
|---|---|
| `workload_id` | 无法按"角色"切分数据（一个 Agent 内多个子任务混在一起，训练样本污染） |
| `client_id` | 多租户场景无法做合规切分、隔离评估 |
| `outcome.user_action` | 没有偏好标签，只能 LLM-as-judge，质量差一截 |
| `trace_links` | 多 Agent 协同场景无法重建完整任务（见 3.7 multi-agent） |

### 2.2 workload_id 是关键

一个生产 Agent 内部往往有 5-20 个 LLM 调用点（intent classifier、router、planner、tool selector、summarizer……）。**每一个调用点的最佳替代模型可能完全不同**：

- intent classifier：开源 1B 模型 + 几条 ICL 就够
- planner：必须 frontier
- summarizer：mid-tier 加 SFT 就行

如果不打 `workload_id`，你最后只能得到一个"全 Agent 替换决策"——粒度太粗，永远做不出真正的成本优化。

NVIDIA Blueprint 的 HR Chatbot 案例里，正是因为按 workload 切分后，发现**只有 intent classifier 这一个 workload 能用 Llama-3.2-1b 替代 Llama-3.1-70b，准确率 98%，成本节省 98.6%**（其他 workload 还得用大模型）。

### 2.3 捕获的 4 个反模式

| 反模式 | 后果 |
|---|---|
| 只存最终 response，不存中间 step | 失败归因不到具体 tool 调用 |
| Tool 调用结果不存 raw payload，只存摘要 | 无法复用做 retrieval 训练 |
| Timestamp 用应用层时间，不用 LLM provider 时间 | 无法准确算 latency / 调试 cache hit |
| PII 不脱敏直接 dump 到训练集 | GDPR / 合规风险，模型记忆敏感数据 |

捕获层接 7.3 可观测性（trace/span/event 三层）和 6.6 Memory&Data Security（PII 脱敏管线）。**没有可观测性基础，飞轮转不起来**。

## §3 齿轮 ②：筛选——VJ 过滤把噪声挡在外面

百万级 trace 里能直接进训练的不到 5%。原始日志里有：

- **垃圾输入**：用户瞎打的"测试 123"、prompt injection 尝试
- **模糊场景**：连人工都说不清对错的边界 case
- **重复**：高度相似的 query 反复出现（cache miss 但语义重复）
- **失败 trace**：上游错误传到 LLM，不应学
- **PII 污染**：必须脱敏或丢弃

### 3.1 Airbnb 的 VJ Filter

Airbnb AITL 论文（arXiv:2510.06674）提出 **Verification Judge (VJ)**——独立于打标 LLM 的另一个 judge 模型，用 rejection sampling 思想：

```
candidate_label = LabelingLLM(trace)
verification = VJ(trace, candidate_label)
if verification.confidence < 0.7 or verification.disagree:
    reject sample
```

实测拒绝 14.3% - 34.5% 样本，**保留下来的样本与人工标注 r > 0.90**。

```python
def vj_filter(trace, candidate_label, threshold=0.7):
    """Verification Judge filter"""
    vj_prompt = f"""
    Trace: {trace}
    Proposed label: {candidate_label}
    
    Tasks:
    1. Verify the label is consistent with trace evidence.
    2. Output {{verdict: 'accept' | 'reject', confidence: 0-1}}
    """
    result = vj_model.call(vj_prompt)
    return result.verdict == "accept" and result.confidence >= threshold
```

**关键点**：VJ 用的是**不同**的底模（避免同一个模型的偏置自我加强）；用更小、更便宜的 judge 模型而不是 frontier（成本可控且性能足够，AITL 用 Mixtral 8x7B 做 VJ）。

### 3.2 规则层 vs LLM 层

筛选不是只有 LLM 一种武器。先用规则把 80% 的明显垃圾过滤掉，再让 LLM 处理灰区：

| 层 | 规则示例 | 处理量 | 成本 |
|---|---|---|---|
| L0 协议层 | 拒绝 schema 不完整 trace | ~5% | 0 |
| L1 启发式 | query 长度<3 char、纯表情、重复字符 | ~30% | 极低 |
| L2 去重 | MinHash/SimHash 语义去重 | ~20% | 低 |
| L3 LLM 评分 | "这条 trace 是否值得训练？" | ~40% | 中 |
| L4 VJ verify | 第二个 judge 验证标签一致性 | ~5% | 中 |

剩下 ~10% 是真正的高价值样本，进 ③ 标注。

### 3.3 反模式：用 ChatGPT 一把梭

**最常见错误**：写个 prompt "请判断这条对话是好样本吗" 喂给 GPT-4，拿输出当训练数据。

为什么不行：
- 同模型偏置：用 GPT 标的数据训新模型，新模型只是在拟合 GPT
- 没有 ground truth 校准，r 系数不知道
- 成本爆炸：百万条全过 GPT-4 = $20K+

正确姿势：**先抽样人工标 500-1000 条做 ground truth → 算 LLM judge 与人工的 agreement (Cohen's κ 或 Pearson r) → 只有 r > 0.85 才能上线 → 持续抽检（每周 50-100 条）**。

## §4 齿轮 ③：自动标注——LLM-as-Verifier，不是 Generator

### 4.1 标注什么：4 类标签

Airbnb AITL 论文的 4 类标签是个好范式：

| 标签类型 | 输入 | 输出 | 用途 |
|---|---|---|---|
| **Pairwise Preference** | 同 query 两个候选 response | A>B / B>A / Tie | DPO / ORPO 训练 |
| **Adoption with Critique** | response | adopted? + 改进点 | SFT + RFT 信号 |
| **Retrieval Relevance** | query + retrieved doc | 相关度 0-3 | retriever 微调 |
| **Missing Knowledge** | failed trace | 缺什么 KB 条目 | RAG 知识库补全 |

这 4 类标签覆盖了 SFT/DPO/RFT 三种训练路径的全部数据需求。

### 4.2 LLM-as-Verifier 范式

**核心原则**：让 LLM 做"是否对"的判断，不让 LLM 自己生成"应该是什么"。原因：

- Verifier 任务比 generator 任务简单一个数量级（PRM800K 论文证实）
- 输出空间约束在 {0,1} 或 0-3 等小范围，方差低
- 可以用更小、更便宜的 judge 模型

```python
def label_adoption(trace, response, vj_threshold=0.7):
    """LLM-as-verifier 风格的 adoption 标注"""
    prompt = f"""
    User asked: {trace.query}
    Agent response: {response}
    User's next action: {trace.outcome.user_action}
    
    Question: did the user adopt the response?
    Reasoning then output JSON: {{adopted: bool, confidence: 0-1, critique: "..."}}
    """
    
    label = primary_judge.call(prompt)  # e.g. Mixtral 8x7B
    
    # VJ filter
    if not vj_filter(trace, label, vj_threshold):
        return None
    
    return label
```

### 4.3 HITL 校准循环

Airbnb 论文里最关键的工程贡献：HITL 不是"人工每条都看"，而是**周期性校准 + 抽检**。

```
Day 1:  人工标 500 条 → 训 LLM judge → 算 r
Day 7:  LLM judge 标 50K 条 → 人工抽 100 条 verify → r 仍 > 0.90？
Day 30: 重新校准（重要！分布会漂移）
```

校准频率不固定，关键指标：**LLM judge 与人工的 disagreement rate**。一旦 disagreement > 10%，立刻全量重训 judge 或人工接管。

### 4.4 Reward Model：从 Label 到 Reward

DPO/RFT 训练需要 reward signal，而 reward 不一定来自显式标签：

| Reward 来源 | 适用场景 | 风险 |
|---|---|---|
| 用户显式反馈（👍/👎） | 频率低，但信号干净 | 样本量不足 |
| 用户隐式行为（adopt/edit/abandon） | 全量但有噪声 | edit 长度 ≠ 质量 |
| LLM judge 评分 | 可扩展，但需 VJ | 同模型偏置 |
| 单元测试 / 工具执行结果 | 代码、计算类任务最强信号 | 仅限可验证任务 |
| 业务下游事件（订单成交、ticket 关闭） | 最直接、但延迟长 | 归因难 |

**最佳实践**：组合 reward——`R = w1 * user_action + w2 * llm_judge + w3 * downstream_event`，每个分量都过 VJ。

## §5 齿轮 ④：训练循环——SFT → DPO/ORPO → RFT/GRPO

### 5.1 训练范式光谱

| 范式 | 数据需求 | 算力 | 收益 | 何时用 |
|---|---|---|---|---|
| **Prompt Engineering** | 0 | 0 | 基线 | 总是先试 |
| **Few-shot / ICL** | 5-50 例 | 0 | +5-10% | 任务结构稳定 |
| **RAG** | 知识库 | 中 | +20-50% | 知识类任务 |
| **LoRA SFT** | 1K-10K 高质 | GPU 小时 | +10-30% | 风格、格式、领域术语 |
| **QLoRA SFT** | 同上 | 更小（4-bit） | 同上 | 显存受限 |
| **DPO** | 1K-5K 偏好对 | GPU 小时 | +5-15% | 偏好对齐 |
| **ORPO** | 偏好对 | 一阶段（SFT+DPO合并） | +10% | 简化流水线（AITL 用） |
| **RFT (RL Fine-Tuning)** | reward function + 几千 trace | 多 GPU 天 | +15-40% | 可验证任务（代码、数学） |
| **GRPO / DAPO / GSPO** | 同 RFT，分组采样 | 更省 | +20-30% | 大规模 RL（DeepSeek R1 用） |
| **MARFT** (Multi-Agent RFT) | 多 Agent trace | 高 | 早期阶段 | 多 Agent 协同优化 |

### 5.2 Agent 场景的训练优先级

| Agent 痛点 | 首选范式 | 数据 |
|---|---|---|
| 工具选错 / 调错参数 | SFT on adopted tool calls | trace.tool_calls where outcome=adopted |
| 输出格式不稳 | SFT + Schema validation | <100 例就够 |
| 风格偏差 / 安全偏好 | DPO/ORPO | pairwise preference |
| 复杂规划失败 | RFT with task success reward | downstream_event |
| 多步推理 | GRPO with stepwise reward | step-level adoption |

**Airbnb AITL 配方**：Mixtral 8x7B + **ORPO + QLoRA**——单阶段、低显存、生产可部署。结果：用户偏好 60.12% vs 基线 33.32%。

### 5.3 训练数据构造：class-aware stratified split

最大的坑：随机切 train/eval。**业务的长尾分布会让 eval 集严重偏离生产真实分布**。

NVIDIA Blueprint 的方案是 **class-aware stratified split**：

```python
def stratified_split(traces, classes, test_ratio=0.1):
    """按 workload 类别分层抽样"""
    train, test = [], []
    for cls in classes:
        cls_traces = [t for t in traces if t.workload_id == cls]
        # 确保 test 集每类至少 50 条
        n_test = max(50, int(len(cls_traces) * test_ratio))
        random.shuffle(cls_traces)
        test.extend(cls_traces[:n_test])
        train.extend(cls_traces[n_test:])
    return train, test
```

### 5.4 RFT/GRPO 的工程注意

RFT 是 2026 年最被高估也最被低估的方向：

**高估**：很多团队以为 RFT = 万能升级，结果发现没有可验证 reward 就训不出来。
**低估**：在可验证场景（代码、SQL、数学、工具调用），RFT 能直接超越 SFT 上限。

工程要点（参考 OpenAI RFT API 公开实践、MARFT arXiv:2504.16129）：
- **Reward function 必须 deterministic + cheap**——每一步训练要调用千万次，LLM judge 太慢
- **Cold start 用 SFT**——RFT from scratch 几乎不收敛
- **GRPO 比 PPO 简单**：去掉 critic 网络，分组归一化 advantage，省 50% 显存
- **Reward hacking 是常态**：定期人工抽看 trajectory，看模型是不是在"作弊"满足 reward

## §6 齿轮 ⑤：替换决策——HITL，不是 Autopilot

### 6.1 何时替换：3 个判据

新模型训完，要不要替换生产？三条硬指标：

| 判据 | 阈值 | 工具 |
|---|---|---|
| **离线 eval 不退化** | 主指标 ≥ 基线 95% | NeMo Evaluator / 自建 |
| **生产 A/B 不退化** | 5-10% 流量 7-14 天，用户体验指标无降 | 流量切分 |
| **失败模式审查** | 人工 review 100 条边缘 case | HITL 抽检 |

**任何一条不过都不替换**——这就是 NVIDIA "flashlight not autopilot" 的实操含义。

### 6.2 A/B 流量切分模板

```python
def route(request):
    if request.client_id in CANARY_TENANTS:
        return new_model
    if hash(request.trace_id) % 100 < CANARY_PERCENT:
        return new_model
    return baseline_model
```

切分原则：
- **小流量起步**：1% → 5% → 25% → 100%，每档跑 24-72h
- **按 client_id 切，不按 trace_id**：避免同一用户在两个模型间反复横跳，体验崩
- **金丝雀监控指标**：除主指标外，监控 p95 latency、错误率、用户重试率

### 6.3 回滚红线

哪些信号一出现立刻回滚（自动化，不等人审）：

| 信号 | 阈值 |
|---|---|
| p95 latency 涨 > 30% | 立刻 |
| 错误率涨 > 50% | 立刻 |
| 用户取消率涨 > 20% | 立刻 |
| 高危错误（PII 泄露、policy 违规） | 立刻 + 报警 |
| 主指标 z-score < -2 持续 1h | 立刻 |

### 6.4 飞轮闭环：从生产回 ①

新模型上线后，它产生的新 trace 又回到 ① 捕获。**这里有一个微妙陷阱**：

新模型的 trace 会偏离原分布（因为它的行为变了），如果直接进训练集，会形成"自我蒸馏正反馈"——模型只学自己。

防御：
- 训练集保留 30%+ **历史多模型 trace**（多样性蓄水池）
- 用 OOD detection 监控新 trace 是否偏离训练分布过多
- 定期混入"对抗 case"（手工构造的 hard sample）

## §7 反模式：飞轮空转或倒转的 12 种姿势

| # | 反模式 | 后果 | 解药 |
|---|---|---|---|
| 1 | 日志没 workload_id | 训练样本污染、替换决策粗粒度 | schema 设计阶段必加 |
| 2 | LLM judge 没 ground truth 校准 | r 未知，垃圾进垃圾出 | 抽样人工标 500 条算 r |
| 3 | Judge = Training model（同一个） | 自我加强偏置 | judge 用不同底模 |
| 4 | 没 VJ 二次验证 | 标签噪声高，训练不收敛 | rejection sampling |
| 5 | 随机切 train/eval | 长尾分布失真 | class-aware stratified |
| 6 | Eval 集冻结 6 个月不变 | 测不出生产新意图退化 | 月度滚动注入 |
| 7 | 只标"成功"trace | 模型不会处理 corner case | 强采样 escalate/failed |
| 8 | RFT reward 不 deterministic | 训练发散 | 单测/工具结果做 reward |
| 9 | Cold start 直接 RFT | 不收敛 | SFT warmup 再 RL |
| 10 | A/B 按 trace_id 切 | 用户体验跳变 | 按 client_id |
| 11 | 飞轮全自动无 HITL | 漂移 + 偏置无人发现 | 每月人工抽检 100 条 |
| 12 | 新模型 trace 直接进训练 | 自我蒸馏正反馈 | 30% 历史样本蓄水池 + OOD 监控 |

## §9 一页总结：抄走这页就够

| 决策 | 选择 |
|---|---|
| **第一步该做什么？** | Trace schema 加 workload_id + client_id + outcome.user_action |
| **怎么过滤百万日志？** | 规则 L0-L2 → LLM judge L3 → VJ L4，分层过滤 |
| **LLM judge 怎么选？** | 用不同于训练模型的底模，做 verifier 而非 generator |
| **怎么知道 judge 靠谱？** | 抽样人工标 500-1000 条算 Pearson r，要 > 0.85 |
| **训什么范式？** | SFT 风格/格式 → DPO/ORPO 偏好 → RFT/GRPO 可验证任务 |
| **训完怎么决定替换？** | 离线 eval + 1-25% A/B + HITL 失败审查，三关全过 |
| **飞轮转太快怎么防？** | 训练集留 30% 历史，OOD 监控偏移，强制 HITL 抽检 |
| **第一性原则** | Flashlight not autopilot——飞轮高亮机会，决策由人定 |
