## 0. 真问题：为什么"加日志"在 Agent 时代不够

线上 Agent 出问题，工程师第一反应是 `grep error`。日志里只有：

```
2026-06-25T10:01:00Z agent.run started req_id=abc123
2026-06-25T10:01:32Z agent.run finished req_id=abc123 ok=true
```

中间 32 秒里发生了什么？跑了几次 LLM？token 烧了多少？哪个工具失败重试了 3 次？哪步 retrieval 命中率为 0？日志答不出来——而这正是 Agent 故障的全部信息密度所在。

老问题在 Agent 场景被指数级放大：

| 维度 | 微服务时代 | Agent 时代 |
|---|---|---|
| 一次请求步数 | 5-10 个 RPC | 10-100+ 步推理+工具调用 |
| 失败类型 | 错误码 / 异常 | "看似成功但语义错"（hallucination/绕路） |
| 主要成本 | CPU / 网络 | **token**（直接挂账单） |
| 延迟主导 | DB / IO | **token 数量**（[Uptrace, 2026](https://uptrace.dev/blog/opentelemetry-ai-systems)） |
| 调试关键 | 调用链 | **完整 prompt + 输出 + 中间状态** |
| 信号类型 | metrics + logs | + **eval scores + trajectory** |

第一性原理：**Agent 可观测性 = 经典三件套（Logs/Metrics/Traces）+ 两件新装备（Token Economics + Eval Scores）**。前三件 OpenTelemetry 已经标准化（[GenAI Semantic Conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/)），后两件是 2026 年还在演进的"Agent 专属"。

## 1. 数据模型：Span 树是 Agent 可观测性的"DNA"

### 1.1 Trace 结构：一次 agent.run 的完整解剖

OpenTelemetry GenAI Conventions 定义的层级（[Uptrace, 2026](https://uptrace.dev/blog/opentelemetry-ai-systems)）：

```
agent.run                                  ← 根 span，task 维度
├── gen_ai.embeddings                      ← 嵌入调用（query）
├── db.vector_search                       ← retrieval span
├── gen_ai.chat (planner)                  ← 规划 LLM
│   ├── agent.tool_call name=lookup_order  ← 工具子 span
│   ├── agent.tool_call name=process_refund
│   │   └── http.client                    ← 工具内部出站
│   └── gen_ai.chat (reflection)           ← Reflexion 反思
└── gen_ai.chat (final_answer)             ← 收尾生成
```

**关键设计原则**：
1. **agent.run 是根，task_id 贯穿**：跨服务用 W3C `traceparent` 透传。
2. **每次 LLM 调用一个 `gen_ai.chat` span**：input/output、token、cost 全记在 span attributes/events。
3. **每次工具调用一个 `agent.tool_call` span**：包含 `tool.name`、`tool.args`（脱敏后）、`tool.result`、`tool.success`。
4. **Retrieval 单独 span**：`db.vector_search`，记录 query、top-k、命中文档 ids、score 分布。
5. **Eval span 旁挂**：从同一个 trace_id 异步发评估 span，关联但不阻断。

### 1.2 必备 gen_ai.* 属性

```python
# OTel GenAI 标准属性（核心子集）
attrs = {
    # 系统标识
    "gen_ai.system": "openai",                    # openai / anthropic / google / mistral / ...
    "gen_ai.operation.name": "chat",              # chat / embeddings / text_completion
    "gen_ai.request.model": "gpt-4o-2024-08-06",
    "gen_ai.response.model": "gpt-4o-2024-08-06",
    # 请求参数
    "gen_ai.request.temperature": 0.0,
    "gen_ai.request.max_tokens": 1024,
    "gen_ai.request.top_p": 1.0,
    # 用量与延迟
    "gen_ai.usage.input_tokens": 412,
    "gen_ai.usage.output_tokens": 87,
    "gen_ai.response.finish_reasons": ["stop"],
    # 流式延迟
    "gen_ai.server.time_to_first_token": 0.42,
    # Agent 扩展（非标准但通行）
    "agent.task_id": "tsk_xyz",
    "agent.step_index": 3,
    "agent.tool.name": "process_refund",
    "agent.tool_calls_total": 5,
}
```

### 1.3 Prompt 内容放哪里——隐私 vs. 可调试

> **绝对纪律**：prompt content 进 **span events**（可被 Collector drop），**不进 span attributes**（默认索引、有 PII 风险）。

```python
# ✅ 正确：内容放 events
span.add_event("gen_ai.user.message", attributes={
    "gen_ai.prompt.content": user_input,    # 可在 Collector 阶段过滤
    "gen_ai.prompt.role": "user",
})
span.add_event("gen_ai.assistant.message", attributes={
    "gen_ai.completion.content": assistant_output,
    "gen_ai.completion.role": "assistant",
})

# ❌ 错误：内容放 attributes（一律索引、暴露 PII）
span.set_attribute("input", user_input)
```

Collector 端按环境一键脱敏：

```yaml
# otel-collector.yaml
processors:
  transform/redact:
    trace_statements:
      - context: spanevent
        statements:
          # 生产环境删除明文 prompt
          - delete_matching_keys(attributes, "gen_ai\\.(prompt|completion)\\.content")
  attributes/pii:
    actions:
      - key: agent.tool.args.email
        action: hash
```

## 2. 四件套指标：Token / Latency / Quality / Cost

### 2.1 Token Economics（Agent 独有）

| 指标 | 类型 | 用途 |
|---|---|---|
| `gen_ai.client.token.usage` | Counter（dim：input/output/cached） | 计费 + 异常调用检测 |
| `gen_ai.tokens_per_task` | Histogram | task 维度成本分布 |
| `gen_ai.cache_hit_ratio` | Gauge | prompt cache 命中率（影响成本） |

**重要**：**input / output / cached** 必须分开计数，绝不能合并成 total——三者单价不同（output 通常 3-5×input；cached input 1/10）。["Separate input/output counters; combining hides cost structure."](https://uptrace.dev/blog/opentelemetry-ai-systems)

```python
# 单 span 记录
input_tokens.add(412, {"gen_ai.token.type": "input", "gen_ai.request.model": "gpt-4o"})
output_tokens.add(87, {"gen_ai.token.type": "output", "gen_ai.request.model": "gpt-4o"})
cost_total.add(412 * 2.5e-6 + 87 * 1.0e-5, {"gen_ai.request.model": "gpt-4o"})
```

### 2.2 Latency 三件套

| 指标 | 来源 | 阈值参考 |
|---|---|---|
| `gen_ai.client.operation.duration` | LLM 单次调用 histogram | p95 < 5s（chat），p99 < 30s（[Uptrace](https://uptrace.dev/blog/opentelemetry-ai-systems)） |
| `gen_ai.server.time_to_first_token` | 流式首 token 延迟 | p95 < 1.5s（用户体验关键） |
| `agent.run.duration` | 端到端 task | p95 < 业务 SLO |

把 **TTFT**（time-to-first-token）单独建仪表盘——它直接决定流式 UX，比总延迟更敏感。

### 2.3 Quality 在线打分

线上不可能跑全量 agent-judge（贵），但**1-5% 采样 + 异常升采样**是工程基线：

```python
def trace_finished_handler(trace):
    sampled = should_sample(trace, base_rate=0.05, error_boost=1.0)  # 错 100% 采
    if sampled:
        # 异步打分，不阻断主流程
        eval_queue.send({
            "trace_id": trace.id,
            "judges": ["task_completion", "tool_correctness", "faithfulness"],
        })
```

eval 结果作为 **新 span 旁挂**到原 trace（共用 trace_id），让 dashboard 能聚合"task 完成度"分布。

### 2.4 Cost

`cost_per_task` 是**最被忽视**的产品指标。三个必备：

```
1. p50/p95/p99 cost per task        ← 单次成本分布
2. cost_per_user_per_day            ← 用户级别（防滥用）
3. cost_per_outcome                 ← 成功 task 的成本（"为有效产出付了多少钱"）
```

`cost_per_outcome = total_cost / sum(task_completion_score)`——它把"贵但有效"和"便宜但白跑"区分开。

## 3. 采样策略：100% 全采是奢侈，0% 是失明

参考 [Uptrace 2026](https://uptrace.dev/blog/opentelemetry-ai-systems)：

| 场景 | 采样率 | 实现 |
|---|---|---|
| 开发 / staging | 100% | Head-based |
| 生产成功 trace | 5-10% | Head-based |
| 生产错误 trace | **100%** | Tail-based（Collector 端） |
| 高 token (>2K input) | **100%** | Tail-based |
| Agent run（多步、单价高） | **建议 100%** | Head-based |
| eval 异常（judge score < 阈值） | **100%** 升采 | Tail-based |

**Tail sampling 的关键**：必须用 OpenTelemetry Collector 的 `tail_sampling` processor，不能在 SDK 里做（SDK 看不到 trace 是否成功）。

```yaml
processors:
  tail_sampling:
    decision_wait: 30s
    policies:
      - name: errors
        type: status_code
        status_code: { status_codes: [ERROR] }
      - name: high_cost
        type: numeric_attribute
        numeric_attribute: { key: gen_ai.usage.input_tokens, min_value: 2000 }
      - name: low_eval
        type: numeric_attribute
        numeric_attribute: { key: agent.eval.task_completion, max_value: 0.5 }
      - name: prob_5pct
        type: probabilistic
        probabilistic: { sampling_percentage: 5 }
```

## 4. 主流平台对比：6 个该知道的，1 个该选的

来源：[Laminar 2026 排行](https://laminar.sh/article/2026-04-23-top-6-agent-observability-platforms)、[Langfuse vs LangSmith vs Arize 2026](https://devops.gheware.com/blog/posts/langfuse-vs-langsmith-arize-comparison-2026.html)。

### 4.1 平台矩阵（按维度而非排名）

| 平台 | 强项 | 弱项 | OSS | OTel 原生 | 自托管 |
|---|---|---|---|---|---|
| **LangSmith** | LangGraph 紧耦合，IDE 调试，断点修改状态 | 仅 SaaS 实质开源；按 trace 计价（$0.50/1k base，$2.50/1k extended） | 闭源 | OTLP 端点 2026.03 上线 | 仅企业版 |
| **Langfuse** | prompt 管理 + eval + dataset 全栈，OSS 友好 | UI 重型，agent trajectory 不如 Laminar | MIT | 是 | 全功能 |
| **Arize Phoenix** | OpenInference 约定、本地 `pip install` 即用，eval 库丰富 | 企业版（Arize AX）才有完整功能 | Elastic 2.0 | 是 | OSS 部分全功能 |
| **Braintrust** | eval-first，回归对比、scorer 矩阵第一 | 偏单纯 eval，trace 体验弱 | 闭源 | 部分 | 仅企业 |
| **Laminar** | agent 调试 / SQL over traces / session replay | 生态新 | Apache 2.0 | 原生 | 全功能 |
| **W&B Weave** | 已有 W&B ML 工作流的延伸 | 不擅长非 ML 团队 | 闭源 | 部分 | 仅企业 |

### 4.2 选型决策树

```
是否已重度用 LangGraph？
├─ Yes → LangSmith（生态 lock-in 但调试体验最好）
└─ No
   │
   是否需要 100% OSS + 自托管？
   ├─ Yes
   │   ├─ 重 prompt 迭代 → Langfuse
   │   ├─ 重 trace 调试 → Laminar
   │   └─ 重 eval / OpenInference → Arize Phoenix
   └─ No
       ├─ Eval-first，多 prompt/model 对比 → Braintrust
       ├─ 已在 W&B 生态 → W&B Weave
       └─ 综合最强 SaaS → LangSmith / Langfuse Pro
```

### 4.3 反 lock-in 的工程选择

**强烈建议**：

> **应用层只对接 OpenTelemetry**，不要直接 `import langsmith`。所有平台的 ingest 口都接 OTLP。

```python
# ✅ 正确：仅依赖 OTel SDK
from opentelemetry import trace
tracer = trace.get_tracer("my-agent")

# 配置 exporter（环境变量切换）
# OTEL_EXPORTER_OTLP_ENDPOINT=https://api.langsmith.com/otel
# OTEL_EXPORTER_OTLP_ENDPOINT=https://us.cloud.langfuse.com/api/public/otel

# ❌ 反模式：直接调平台 SDK
# from langsmith import Client
# client = Client()
# client.create_run(...)
```

切换平台只改环境变量，几乎零代码改动——平台是工具，不该是绑架。

### 5.2 自动注入 + 手动补强

```python
# main.py - 启动时一键注入
from opentelemetry.instrumentation.openai import OpenAIInstrumentor
from opentelemetry.instrumentation.anthropic import AnthropicInstrumentor
from opentelemetry.instrumentation.langchain import LangChainInstrumentor

OpenAIInstrumentor().instrument()
AnthropicInstrumentor().instrument()
LangChainInstrumentor().instrument()

# Agent 自定义 span 手动补强
from opentelemetry import trace
tracer = trace.get_tracer(__name__)

def agent_run(task_input: str, task_id: str):
    with tracer.start_as_current_span("agent.run") as root:
        root.set_attribute("agent.task_id", task_id)
        root.set_attribute("agent.input.length", len(task_input))
        for step in plan(task_input):
            with tracer.start_as_current_span(f"agent.step.{step.kind}") as s:
                s.set_attribute("agent.step.kind", step.kind)
                s.set_attribute("agent.step.index", step.index)
                if step.kind == "tool_call":
                    s.set_attribute("agent.tool.name", step.tool)
                    s.set_attribute("agent.tool.args.size", len(json.dumps(step.args)))
                step.execute()
```

### 5.3 在线 eval 旁路

```python
# eval_worker.py - 异步消费 trace 并打分
async def eval_handler(trace_id: str):
    trace = await load_trace(trace_id)
    judges = [
        ("task_completion", goal_aware_judge),
        ("tool_correctness", deterministic_tool_check),
    ]
    for name, judge in judges:
        score = await judge(trace)
        # 用 same trace_id 写一个 child span，关联但不阻断
        with tracer.start_as_current_span(
            "agent.eval", context=context_from_trace_id(trace_id)
        ) as s:
            s.set_attribute("agent.eval.metric", name)
            s.set_attribute("agent.eval.score", score)
```

| # | 仪表盘 | 关键 panel | 告警 |

| 7 | Eval Scores | task_completion / faithfulness 分布 | 周环比 < -2pp |

## 7. 反模式速查

| 反模式 | 后果 | 替代 |
|---|---|---|
| 把 prompt 内容写 attribute | 索引爆炸、PII 泄露 | 写 span events，Collector 脱敏 |
| 只有 logs 没有 trace | 多步问题无从定位 | OTel trace 全覆盖 |
| input/output token 合计 | 成本黑盒 | 三向分（input/output/cached） |
| 100% 采样上生产 | 存储爆炸 | head 5% + tail 100%（错/高 token/低 eval） |
| 直接 vendor SDK 写采集 | lock-in；切换成本高 | OTel SDK + OTLP exporter |
| 不记 trace_id 上下文 | 跨服务对不上 | W3C traceparent 透传 |
| eval 异步落不到 trace 上 | dashboard 无法关联 | eval 写 child span，复用 trace_id |
| latency 只看 p95 总延迟 | 忽视流式体验 | 单独看 TTFT |
| cost dashboards 只有"总和" | 无法定位单 task | per-task cost histogram + cost_per_outcome |
| 没有 kill switch 触发面板 | 6.7 联动失效 | 面板 + page 告警 |

## 9. 一页总结

> **Agent 可观测性 = OpenTelemetry GenAI 标准（Spans/Metrics/Events）+ Token 经济学 + Eval 旁挂**。三层都对，才能在 Agent 时代继续做工程。

- **Span 树是 DNA**：`agent.run` 根 → `gen_ai.chat` / `agent.tool_call` / `db.vector_search` 子。
- **prompt 内容进 events**，Collector 脱敏，绝不放 attributes。
- **Token 必须三向分**（input/output/cached），cost_per_outcome 比 cost_per_task 更值钱。
- **采样策略**：head 5% + tail 100%（错/高 token/低 eval）。
- **平台选型**：应用层只对 OTel，平台是 exporter；选型按生态（LangGraph→LangSmith；OSS 自托管→Langfuse/Phoenix/Laminar；eval-first→Braintrust）。
- **9 张仪表盘**：health / latency / token / cost / tools / retrieval / eval / errors / drift——少一张都是裸奔。
- **eval 旁挂**：同 trace_id 的 child span，让"质量"成为 dashboard 一等公民。

可观测性不是事故后翻日志，是事故发生**前 5 分钟**就把告警打到 oncall 的能力。
