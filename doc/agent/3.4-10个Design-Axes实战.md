# 10 个 Design Axes 实战：用架构坐标系做出正确的设计决策

> 给你 10 个 Agent，你能构建一个系统。但给你 10 个轴，你能构建一万个系统——并知道哪个是对的。本文不再解释"什么是 Design Axes"（2.2 已完成），而是展示如何在真实场景中用这 10 根轴做设计决策、发现架构缺陷、进行系统对比。

## 0. 开场问题

你的团队要为一个金融机构搭建"智能审贷系统"。产品经理给了需求：

1. 用户提交贷款申请后，系统自动完成反欺诈检查、征信报告分析、收入核验、资产评估、最终给出审批建议
2. 单笔审贷必须在 30 秒内完成
3. 所有中间决策必须可追溯，满足监管审计要求
4. 某些步骤可以并行（反欺诈和征信同时跑），某些必须串行（资产评估依赖收入核验结果）
5. 模型产生的"拒绝"决策必须经人工确认

架构师甲说："用一个强模型，一次调用出结果。"架构师乙说："上 multi-agent，每个检查项一个 Agent。"架构师丙说："Orchestrator-Worker 模式，加 HITL 门禁。"

三个人都没错，也都没对——因为他们在用"模式名"交流，而非用**具体的设计维度**交流。当你改用 10 个 Design Axes 逐一标注时，问题自然有答案：

| Axis | 架构师甲 | 架构师乙 | 架构师丙 |
|---|---|---|---|
| Topology | 单 Agent | 扁平 Multi | Orchestrator-Worker |
| Coordination | 无 | 共享状态 | 消息传递 |
| Session Span | 单轮 | 单轮 | 单轮 |
| Control Loop | 单次推理 | 各自 ReAct | Orchestrator 顺序/并行调度 |
| Autonomy | 全自主 | 全自主 | HITL 门禁（拒绝决策） |
| Execution Isolation | 无 | 无 | 容器隔离 |
| Memory | 无 | 短期共享 | 事件日志 |
| Tool Type | 只读查询 | 只读 + 写决策 | 只读查询 + 写审批意见 |
| Verification | 无 | 各自自验 | 外部 rule engine 验证 |
| Compaction | 无 | 无 | 不需要（单轮短任务） |

现在差异一目了然。架构师甲的方案在 Autonomy 和 Verification 上有合规缺陷——全自主 + 无验证无法满足监管要求。架构师乙在 Execution Isolation 上有风险——多 Agent 共享环境可能产生数据泄漏。架构师丙的方案最接近需求，但还需要明确 Coordination 和 Memory 的具体实现。

**10 个轴不是给你答案，而是强制你问出正确的问题。**

## 1. Axes 回顾与"坐标值域"

2.2 文章定义了 10 个轴的名称和问题。本文进一步给每个轴定义具体的**值域光谱**——来自 arXiv 2604.03515（Widyasari et al., 2025）对 13 个生产级 Coding Agent 的源码逆向分析，以及 Anthropic（2025）、LangChain（2025）的工程实践：

| 组 | Axis | 值域光谱（从简到复） | 关键 trade-off |
|---|---|---|---|
| 结构 | Topology | 单 Agent → Router → Orchestrator-Workers → Hierarchical → Swarm → Event-Driven | 复杂度 vs 并行度 |
| 结构 | Coordination | 无 → 共享 Scratchpad → 消息传递（Handoff）→ A2A Protocol → 混合 | 耦合度 vs 灵活性 |
| 结构 | Session Span | 单轮 → 多轮会话 → 跨天持久化 → 异步触发 | 状态管理成本 vs 任务连续性 |
| 行为 | Control Loop | 固定管道 → ReAct → Plan-Execute → 分阶段 → DFS Tree → MCTS | 可预测性 vs 探索能力 |
| 行为 | Autonomy | 全人工 → Copilot → HITL 门禁 → 受限自主 → 全自主 | 安全性 vs 效率 |
| 行为 | Execution Isolation | 无隔离 → 子进程 → 平台沙箱 → Docker → microVM → Shadow Git | 开销 vs 安全性 |
| 资源 | Memory | 无 → 会话内上下文 → 外部短期存储 → 持久化知识库 → 跨 Agent 共享记忆 | Token 成本 vs 连续性 |
| 资源 | Tool Type | 纯只读 → 读 + 可逆写 → 读 + 不可逆写 → 自主创建工具 | 能力 vs 风险 |
| 资源 | Verification | 无 → 自检（LLM-as-Judge）→ 确定性验证（测试/类型检查）→ 外部验证 → 人工审核 | 延迟 vs 可靠性 |
| 资源 | Compaction | 无 → 硬截断 → 滑动窗口 → 选择性丢弃 → LLM 摘要 → LLM 主动发起 | 信息保留 vs Token 预算 |

这张表是架构师的**罗盘**。下面我们用它导航四个真实案例。

## 2. 案例一：Anthropic Multi-Agent Research System

Anthropic 在 2025 年公开了其多 Agent 研究系统的工程细节（Anthropic Engineering Blog, 2025），该系统在内部评测中比单 Agent Opus 4 高出 90.2%。用 10 轴标注其架构：

| Axis | 取值 | 理由 |
|---|---|---|
| Topology | Orchestrator-Workers | Lead Researcher（Opus 4）协调，Subagents（Sonnet 4）执行 |
| Coordination | 文件系统消息传递 | Subagents 将输出写入文件系统，Lead 通过引用读取——避免"电话游戏"token 膨胀 |
| Session Span | 多轮 + 异步触发 | 复杂查询 Lead 产生多轮 subagent 迭代 |
| Control Loop | Plan-Execute + 动态规模调整 | Lead 先规划，根据查询复杂度 spawn 1-10+ 个 subagent |
| Autonomy | 受限自主 | 无人工门禁，但 Lead 内置"effort-scaling"启发式约束探索范围 |
| Execution Isolation | 独立上下文窗口 | 每个 Subagent 拥有独立 200K context，互不干扰 |
| Memory | 外部短期存储（Memory 文件） | Lead 将计划写入 Memory 文件以应对 context truncation |
| Tool Type | 只读查询 + 可逆写（搜索 + 文件写入） | 搜索外部知识 + 写结构化输出 |
| Verification | LLM-as-Judge（CitationAgent） | 独立 Agent 验证引用正确性 |
| Compaction | LLM 摘要 + 新 Subagent 接力 | 长对话通过 summarization + 新鲜 Subagent 的 handoff 管理 |

**架构洞察**：

Anthropic 的关键决策是在 Coordination 轴上选择"文件系统传递"而非"上下文内拼接"。传统做法是让 Subagent 的输出直接拼入 Orchestrator 的 context，但这会导致 token 随 agent 数量线性膨胀。文件引用模式使 Lead 只需处理摘要和指针，将 token 开销从 O(n × output_size) 降到 O(n × reference_size)。

这个选择在 Compaction 轴上产生连锁效应——因为 Lead 的 context 不会被 subagent 输出淹没，所以不需要激进的 compaction 策略。**轴之间不是独立的，一个维度的取值会约束其他维度的可选范围。**

## 3. 案例二：Coding Agent 架构谱系

arXiv 2604.03515 逆向分析了 13 个开源 Coding Agent 的完整源码（2025），发现在 10 轴上呈现出鲜明的聚类和分歧：

### 收敛区（共识已形成）

**Tool Type**：所有 13 个 Agent 的工具能力收敛到五类——读取、搜索、编辑、执行、验证。即使工具数量差异巨大（Aider 0 个 vs Moatless Tools 37 个），能力类别一致。

**Execution Isolation**：全部采用某种形式的隔离。最轻量的是 stateless subshell，最重的是 Docker + shadow Git。没有任何生产级 Agent 敢在用户机器上无隔离运行写操作。

**Verification**：每个 Agent 都内置了至少一层确定性验证（lint/test/typecheck），纯 LLM-as-Judge 无一采用。Bui（2026）的结论是"tool reliability matters more than model capability"。

### 分歧区（仍在探索）

**Control Loop**：13 个 Agent 中 11 个采用多种控制原语的组合——没有一个只用纯 ReAct 或纯 Plan-Execute。这说明行业尚未找到单一最优 loop 结构，组合式设计是当前 best practice。

**Compaction**：策略差异最大，从硬截断到 LLM 主动发起的 summarization 都有。这是唯一一个"正确答案"明显依赖具体用例的轴。

**Memory**：从"无记忆"（Agentless，每次独立运行）到"LLM-authored + 跨工具持久化"（Claude Code 的 CLAUDE.md），跨度极大。选择高度依赖 Session Span——单轮 Agent 不需要 Memory，持久化 Agent 必须有。

### 轴间约束关系

论文揭示了一个关键发现：**轴不是独立正交的，而是存在约束传播**。例如：

- Control Loop = Tree Search → Execution Isolation 必须支持分支恢复（shadow Git 或快照）
- Topology = Multi-Agent → Coordination 不能为"无"
- Session Span = 跨天持久化 → Memory 至少需要外部存储，Compaction 成为必选
- Tool Type = 不可逆写 → Verification 至少需要确定性验证层

这些约束构成一张**兼容性矩阵**。违反约束的组合，即使逻辑上可能，工程上不可持续。

## 4. 案例三：企业审批流 Agent

回到开场的审贷场景，结合 Catalect（2025）的企业架构决策框架，我们用 5 个诊断问题判断是否需要 Multi-Agent：

1. 是否涉及 2 个以上企业系统？→ **是**（反欺诈系统、征信中心、银行核心系统、影像平台）
2. 是否有可并行的步骤？→ **是**（反欺诈 ∥ 征信 ∥ 收入核验可并行）
3. 单步失败是否会级联？→ **是**（征信查询超时不应阻塞整个审批）
4. 合规是否要求步骤级审计？→ **是**（银保监会要求全流程可追溯）
5. 是否需要扩展？→ **是**（后续要接入更多检查项）

5/5 命中——Multi-Agent 是正确选择。接下来用 10 轴精确规格化：

```python
"""
审贷 Agent 系统的 10 轴架构规格
"""
from dataclasses import dataclass, field
from enum import Enum
from typing import List

class TopologyType(Enum):
    ORCHESTRATOR_WORKERS = "orchestrator-workers"

class CoordinationType(Enum):
    EVENT_LOG = "event-sourced-log"  # 所有决策写入不可变事件流

class SessionSpanType(Enum):
    SINGLE_ROUND_BOUNDED = "single-round-30s-sla"

class ControlLoopType(Enum):
    PARALLEL_SCATTER_GATHER = "parallel-scatter-gather"

class AutonomyLevel(Enum):
    HITL_ON_REJECT = "hitl-gate-on-rejection"

class IsolationType(Enum):
    CONTAINER_PER_WORKER = "container-per-worker"

class MemoryType(Enum):
    EVENT_SOURCED_AUDIT_LOG = "immutable-event-log"

class ToolTypeLevel(Enum):
    READ_PLUS_REVERSIBLE_WRITE = "read-query + reversible-decision-write"

class VerificationType(Enum):
    RULE_ENGINE_PLUS_HUMAN = "deterministic-rules + human-on-reject"

class CompactionType(Enum):
    NOT_NEEDED = "n/a-single-round"


@dataclass
class AgentArchitectureSpec:
    """10 轴架构规格：审贷系统"""
    topology: TopologyType = TopologyType.ORCHESTRATOR_WORKERS
    coordination: CoordinationType = CoordinationType.EVENT_LOG
    session_span: SessionSpanType = SessionSpanType.SINGLE_ROUND_BOUNDED
    control_loop: ControlLoopType = ControlLoopType.PARALLEL_SCATTER_GATHER
    autonomy: AutonomyLevel = AutonomyLevel.HITL_ON_REJECT
    execution_isolation: IsolationType = IsolationType.CONTAINER_PER_WORKER
    memory: MemoryType = MemoryType.EVENT_SOURCED_AUDIT_LOG
    tool_type: ToolTypeLevel = ToolTypeLevel.READ_PLUS_REVERSIBLE_WRITE
    verification: VerificationType = VerificationType.RULE_ENGINE_PLUS_HUMAN
    compaction: CompactionType = CompactionType.NOT_NEEDED


@dataclass
class DesignDecisionRecord:
    """每个轴选值的设计决策记录（满足审计要求）"""
    axis: str
    chosen_value: str
    rationale: str
    rejected_alternatives: List[str] = field(default_factory=list)
    constraints_from: List[str] = field(default_factory=list)  # 受哪些轴约束

# --- 设计决策记录示例 ---

decisions = [
    DesignDecisionRecord(
        axis="Topology",
        chosen_value="Orchestrator-Workers",
        rationale="需要中心调度以满足 30s SLA 和并行控制；"
                  "Swarm 无法保证时序；Hierarchical 增加不必要的层级",
        rejected_alternatives=["Single Agent", "Swarm", "Hierarchical"],
        constraints_from=[]
    ),
    DesignDecisionRecord(
        axis="Coordination",
        chosen_value="Event-Sourced Log",
        rationale="不可变事件日志同时满足审计合规和 Worker 间数据传递；"
                  "共享状态有并发竞争风险",
        rejected_alternatives=["Shared State", "Direct Message Passing"],
        constraints_from=["Topology=Orchestrator"]
    ),
    DesignDecisionRecord(
        axis="Autonomy",
        chosen_value="HITL Gate on Rejection",
        rationale="监管要求拒绝决策必须人工确认；"
                  "通过决策可全自动（降低延迟）",
        rejected_alternatives=["Full Autonomous", "All HITL"],
        constraints_from=["Verification=Rule Engine + Human"]
    ),
    DesignDecisionRecord(
        axis="Verification",
        chosen_value="Rule Engine + Human",
        rationale="确定性规则引擎覆盖合规边界（如 LTV > 80% 自动拒）；"
                  "人工覆盖边缘情况；纯 LLM-as-Judge 不满足监管确定性要求",
        rejected_alternatives=["LLM-as-Judge only", "No verification"],
        constraints_from=["Autonomy=HITL on Reject"]
    ),
    DesignDecisionRecord(
        axis="Execution Isolation",
        chosen_value="Container per Worker",
        rationale="金融数据隔离要求：反欺诈 Agent 不应访问征信原始报告；"
                  "容器边界天然实现最小权限",
        rejected_alternatives=["Shared environment", "microVM (overkill)"],
        constraints_from=["Tool Type=Read+Write (需要隔离写操作)"]
    ),
]


# --- 架构合规性检查器 ---

COMPATIBILITY_RULES = {
    # (axis, value_pattern): [(required_axis, allowed_value_patterns)]
    ("topology", "multi"): [
        ("coordination", "!none"),  # Multi-Agent 必须有协调机制
    ],
    ("tool_type", "irreversible"): [
        ("verification", "!none"),  # 不可逆写必须有验证层
        ("execution_isolation", "!none"),  # 不可逆写必须有隔离
    ],
    ("session_span", "persistent"): [
        ("memory", "!none"),  # 持久会话必须有记忆
        ("compaction", "!none"),  # 持久会话必须有压缩策略
    ],
    ("control_loop", "tree_search"): [
        ("execution_isolation", "snapshot_capable"),  # 树搜索需要可快照隔离
    ],
}


def check_compatibility(spec: AgentArchitectureSpec) -> List[str]:
    """检查 10 轴取值间的兼容性，返回违规列表"""
    violations = []
    
    # Example check: Multi-Agent topology requires coordination
    if "multi" in spec.topology.value or "orchestrator" in spec.topology.value:
        if "none" in spec.coordination.value.lower():
            violations.append(
                f"VIOLATION: Topology={spec.topology.value} 要求 Coordination != None"
            )
    
    # Tool with write requires verification
    if "write" in spec.tool_type.value.lower():
        if "none" in spec.verification.value.lower():
            violations.append(
                f"VIOLATION: Tool Type 含写操作时必须配置 Verification"
            )
    
    return violations


# --- 使用示例 ---
if __name__ == "__main__":
    spec = AgentArchitectureSpec()
    print("=== 审贷系统架构规格 ===")
    for axis_name in spec.__dataclass_fields__:
        value = getattr(spec, axis_name)
        print(f"  {axis_name:20s} = {value.value}")
    
    violations = check_compatibility(spec)
    if violations:
        print(f"\n⚠️ 发现 {len(violations)} 个兼容性问题:")
        for v in violations:
            print(f"  - {v}")
    else:
        print("\n✅ 所有轴取值兼容性检查通过")
    
    print("\n=== 设计决策记录 ===")
    for d in decisions:
        print(f"\n[{d.axis}] → {d.chosen_value}")
        print(f"  理由: {d.rationale}")
        if d.rejected_alternatives:
            print(f"  排除: {', '.join(d.rejected_alternatives)}")
        if d.constraints_from:
            print(f"  约束来源: {', '.join(d.constraints_from)}")
```

这段代码不仅是架构设计工具，更是**设计决策的可执行文档**——每个轴的选值都带有 rationale，满足企业架构治理的追溯要求。

## 5. 案例四：对比分析——Cursor vs Devin vs Claude Code

使用 10 轴对比三个代表性产品级 Coding Agent，数据来源为 arXiv 2604.03515 源码分析和各产品公开文档：

| Axis | Cursor (Tab/Agent) | Devin | Claude Code |
|---|---|---|---|
| Topology | 单 Agent（IDE 内嵌） | Orchestrator-Workers（Planner + Executor + Reviewer） | 单 Agent（终端内嵌） |
| Coordination | 无（单 Agent） | 内部消息传递 | 无（单 Agent） |
| Session Span | 单轮~短多轮 | 跨天持久化（异步任务） | 多轮会话 + headless 模式 |
| Control Loop | ReAct（用户驱动迭代） | Plan-Execute + Review Loop | ReAct + 工具检查循环 |
| Autonomy | Copilot（需要用户 accept） | 受限自主（异步，可中途介入） | Copilot → 可配置全自主 |
| Execution Isolation | IDE 沙箱 | Firecracker microVM | Docker + Shadow Git |
| Memory | 项目级 .cursorrules | 跨会话持久化 Knowledge | CLAUDE.md + Memory Bank |
| Tool Type | 读 + 可逆写（IDE diff） | 读 + 不可逆写（真实环境操作） | 读 + 可逆写（Git backed） |
| Verification | 实时 lint + 用户审核 | 自动测试 + AI Review + 用户审核 | Hook 触发 lint/test + 用户审核 |
| Compaction | IDE 控制上下文窗口 | LLM 摘要 + 事件日志 | 选择性丢弃 + CLAUDE.md 沉淀 |

**洞察**：

**Devin 的差异化来自 Session Span 和 Execution Isolation 的组合选择。** 跨天持久化需要 microVM 隔离（普通容器重启会丢状态），而 microVM 的成本只有在长任务中才能被分摊。这两个轴的协同选择定义了 Devin 的产品形态——异步、自治、长周期。

**Claude Code 的差异化来自 Memory 和 Verification 的组合。** CLAUDE.md 是用户可编辑的持久记忆（Guide），PostToolUse Hook 是确定性验证（Sensor）。两者配对正是 2.2 中 Böckeler 的核心主张：Guide + Sensor 必须成对出现。

**Cursor 的差异化来自 Autonomy 的极致简化。** 通过将 Autonomy 锚定在 Copilot 级别，它不需要复杂的 Memory、Compaction、或 Execution Isolation——用户的实时 accept/reject 本身就是最强的 Verification 和 Rollback。

**关键教训**：10 轴不存在"每个都选最右边"的最优解。产品形态决定了你在哪些轴上"向右推"、在哪些轴上"保持左侧"——推得太远的轴会在其他轴上产生级联约束。

## 6. 反模式：违反兼容性的真实灾难

### 反模式一：全自主 + 无验证 + 不可逆写

**典型案例**：早期部署的自动化 DevOps Agent，拥有 `rm -rf`、`kubectl delete` 等不可逆工具权限，设定为全自主模式且无确定性验证层。

**10 轴诊断**：
- Autonomy = 全自主
- Tool Type = 不可逆写
- Verification = 无（或仅 LLM 自检）

**兼容性违规**：不可逆写 + 全自主 + 无验证是"致命三角"。任何一个 LLM 幻觉都会直接变成生产事故，且不可恢复。

**修复路径**：至少将 Verification 推到"确定性验证"（dry-run + 人工确认不可逆操作），或将 Autonomy 回退到 HITL 门禁。

### 反模式二：Multi-Agent + 共享环境 + 无 Coordination

**典型案例**：多个 Agent 操作同一个 Git 仓库，没有锁机制也没有事件日志协调。

**10 轴诊断**：
- Topology = Multi-Agent
- Execution Isolation = 无隔离（共享环境）
- Coordination = 无

**后果**：Agent A 正在编辑 `main.py`，Agent B 同时 rebase 了分支。结果是 merge conflict、代码丢失、状态不一致。

**修复路径**：最小修复是加 Coordination（悲观锁或文件级租约）。更好的方案是推 Execution Isolation 到 Shadow Git（每个 Agent 独立工作副本）。

### 反模式三：持久会话 + 无 Compaction + 无 Memory

**典型案例**：长时间运行的客服 Agent，对话持续数小时但既没有 context 压缩也没有外部记忆存储。

**10 轴诊断**：
- Session Span = 长多轮
- Memory = 无
- Compaction = 无

**后果**：对话超过 200K token 后，早期信息被硬截断。用户说"我之前提到过我的订单号是 XXX"，Agent 完全没有记忆——不是忘了，是被物理截断了。

**修复路径**：Session Span = 持久化时，必须至少选择 Compaction = LLM 摘要（保留关键信息），理想情况下配合 Memory = 外部存储（将关键事实持久化到 context 之外）。

## 7. 实战决策流程：从需求到 10 轴规格

将 10 轴实战化为一个可执行的架构决策流程：

### Phase 1：约束驱动排除（5 分钟）

首先识别**硬约束**——不可协商的需求：

- 有合规审计要求 → Verification ≥ 确定性验证 且 Memory = 事件日志（审计需要）
- 有 SLA 时间约束 → Session Span 固定 且 Control Loop 不能用 MCTS（太慢）
- 涉及不可逆操作 → Verification ≥ 确定性验证 且 Execution Isolation ≥ 容器
- 处理敏感数据 → Execution Isolation ≥ 容器隔离

硬约束会立刻排除大量无效组合，将设计空间从理论上的 5^10 ≈ 一千万种缩减到可管理的范围。

### Phase 2：约束传播（10 分钟）

已选定的轴值会约束其他轴。按以下传播规则推导：

```
Topology ∈ {Multi-Agent} 
  → Coordination ≠ None
  → 如果 Coordination = 共享状态 → Execution Isolation 需要处理并发
  → 如果 Worker 数量动态 → Control Loop 需要支持并行调度

Session Span ∈ {持久化}
  → Memory ≠ None
  → Compaction ≠ None
  → 如果 Memory = 外部存储 → 需要额外的一致性保证

Tool Type ∈ {不可逆写}
  → Verification ≥ 确定性验证
  → Execution Isolation ≥ 容器（限制爆炸半径）
  → Autonomy ≤ 受限自主（或配合强验证）

Control Loop ∈ {Tree Search}
  → Execution Isolation 必须支持快照/恢复
  → Memory 需要存储分支状态
```

### Phase 3：产品形态选择（15 分钟）

在约束排除后的可行域内，根据产品定位选择：

| 产品形态 | 典型轴倾向 |
|---|---|
| IDE Copilot | Topology=单, Autonomy=Copilot, Session=短, Isolation=轻 |
| 异步 AI Worker | Topology=Orchestrator, Autonomy=受限自主, Session=持久, Isolation=重 |
| 对话式助手 | Topology=单/Router, Autonomy=Copilot, Session=多轮, Memory=持久 |
| 批量处理 Pipeline | Topology=Pipeline, Autonomy=全自主, Verification=确定性, Compaction=无需 |
| 研究/探索 Agent | Topology=Orchestrator, Control Loop=Tree/MCTS, Memory=外部, Verification=混合 |

### Phase 4：记录与验证（10 分钟）

为每个轴的最终取值写一条 Decision Record（如案例三的 `DesignDecisionRecord`），然后运行兼容性检查。如果有违规，回到 Phase 2 调整。

**整个流程 40 分钟内完成**，输出是一份可机器解析的架构规格 + 设计决策记录。这比"画一个架构图然后开始写代码"可靠得多——因为架构图不会告诉你哪些组合是不可行的。

## 8. Design Axes 作为团队沟通语言

10 轴的另一个关键价值：**它把主观的架构争论变成客观的维度选择讨论**。

回到开场问题——三个架构师的分歧，本质上只是在 3 个轴（Topology、Autonomy、Verification）上取值不同。一旦用轴表达，讨论焦点从"你的方案 vs 我的方案"变成"这个轴应该选什么值"，每个值的 trade-off 是明确的。

LangChain（2025）在其多 Agent 架构选择指南中提出了类似的思路：不要先选模式（Subagents / Skills / Handoffs / Router），而是先判断需求特征（是否需要并行、是否需要状态传递、是否需要用户交互），然后特征自然指向模式。

FlowForge（arXiv 2507.15559, 2025）的用户研究进一步验证：当给设计者提供显式的设计空间可视化（每个轴的取值范围和组合关系）时，他们产出的架构多样性显著提升，且更少陷入"设计固定"（design fixation）——即总是回到自己最熟悉的模式。

**启示**：在团队 Design Review 中引入 10 轴表格，要求每个方案都标注自己的取值，比比较"方案 A vs 方案 B"的自由讨论高效一个数量级。

## 9. 从 10 轴到 12 轴：演化方向

arXiv 2604.03515（v2, 2025）实际上已经将维度扩展到了 12 个，增加了：

- **Multi-Model Routing**：单模型 → 分类器链 → Actor-Critic → 角色分配。主要驱动力是成本优化——复杂推理用强模型，格式化输出用小模型。
- **Persistent Memory**：无 → 静态项目指令（如 CLAUDE.md）→ LLM 生成记忆 → 跨工具兼容文件。

这两个维度正在快速成熟。Multi-Model Routing 随着模型价格持续下降和性能梯度缩小，可能会逐渐被"一个足够好的模型"取代，变成运维优化而非架构决策。但 Persistent Memory 正在成为定义产品形态的核心轴——OpenAI Codex 的 `/setup` + knowledge files、Claude Code 的 CLAUDE.md + Memory Bank、Cursor 的 .cursorrules 都是在这个轴上竞争。

未来可能出现的第 13 轴：**Agent Identity / Trust**——在 A2A 协议普及后，Agent 需要可验证的身份和信任等级。这将从"Agent 内部设计"扩展到"Agent 间治理"的领域。

## 10. 总结：轴比模式更有用

回到文章开头的命题：

模式（Pattern）告诉你"长什么样"——Orchestrator-Workers、Swarm、Router。但模式是**输出**，不是**输入**。你不能反过来从模式出发设计系统——那是事后合理化。

轴（Axes）告诉你"在哪里做选择"——Topology、Coordination、Verification。轴是**输入**，是设计过程中的决策点。当你沿每个轴明确取值后，模式会自然浮现。

实战方法论：

1. 用硬约束排除不可行组合
2. 用约束传播推导必选取值
3. 在可行域内根据产品形态选择
4. 记录决策理由，运行兼容性检查
5. 用 10 轴表格作为团队沟通的共同语言

当你下一次面对架构决策时，不要问"我们应该用什么模式"，而是问"这 10 个轴，每个应该选什么值，为什么"。答案会比模式名精确十倍。
