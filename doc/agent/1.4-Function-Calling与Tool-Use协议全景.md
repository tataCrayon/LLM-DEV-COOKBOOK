# Function Calling / Tool Use 协议全景

> LLM 本身只能输出文本。Tool Use 是让模型从"说话"变成"做事"的关键机制——它定义了模型如何表达"我想调用某个函数"的意图、外部系统如何执行这个调用并返回结果、以及如何把这个循环跑起来。本文覆盖三大提供商的协议细节、MCP 作为通用工具层标准的崛起、以及 Tool Schema 设计的工程实践。

## 0. 开场问题

你有一个客服 Agent，需要它能够：查订单状态、发退款、查库存、转人工。四个能力对应四个 API。问题来了：

1. 模型怎么知道有这四个工具可用？
2. 模型决定调某个工具时，参数怎么传？格式怎么保证正确？
3. 一次回复里能不能同时调多个工具？
4. 工具返回的结果怎么喂回模型？
5. 如果你有 100 个工具，怎么避免每次都把 100 个定义塞进上下文？

这五个问题就是 Tool Use 协议层要解决的全部内容。

## 1. 工具调用的基本循环

所有提供商的 tool use 都遵循同一个基本循环：

```
User Message
    ↓
┌─────────────┐
│   LLM       │  ←── tool definitions（告诉模型有什么工具）
│  "我要调用  │
│   tool X"   │  ──→ tool_use / function_call（模型输出）
└─────────────┘
    ↓
┌─────────────┐
│  Runtime    │  执行实际函数
│  (你的代码) │
└─────────────┘
    ↓
tool_result / function response
    ↓
┌─────────────┐
│   LLM       │  基于工具返回继续推理
│  "根据结果  │
│   我认为..." │
└─────────────┘
    ↓
Final Response to User
```

循环可以重复多次——模型看到工具结果后可能决定再调另一个工具，直到它认为信息足够给出最终回答。

## 2. 三大提供商协议对比

### 2.1 OpenAI：从 function_calling 到 tools

OpenAI 最早引入 function calling（2023-06），后在 2023-11 升级为 `tools` 参数（支持并行工具调用）。到 2025 年，`functions` 参数已被标记为 legacy，所有新代码应使用 `tools`。

**定义工具：**

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_order_status",
            "description": "查询订单当前状态。需要订单ID。",
            "parameters": {
                "type": "object",
                "properties": {
                    "order_id": {
                        "type": "string",
                        "description": "订单编号，格式如 ORD-20250101-XXXX"
                    }
                },
                "required": ["order_id"]
            },
            "strict": True  # 启用 Structured Outputs，保证 100% schema 合规
        }
    }
]
```

**tool_choice 控制：**

| 值 | 行为 |
|---|---|
| `"auto"` | 模型自主决定是否调用工具（默认） |
| `"none"` | 禁止工具调用，强制文本回复 |
| `"required"` | 模型必须调用至少一个工具 |
| `{"type": "function", "function": {"name": "X"}}` | 强制调用指定工具 |

**并行工具调用（Parallel Tool Calls）：**

GPT-4o / GPT-5 系列支持在单次回复中输出多个 `tool_calls`。例如模型可能同时调用 `get_order_status` 和 `check_inventory`。你需要并行执行它们，把两个结果都放入后续消息中：

```python
# 模型返回中可能有多个 tool_calls
for tool_call in response.choices[0].message.tool_calls:
    result = execute(tool_call.function.name, json.loads(tool_call.function.arguments))
    messages.append({
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": json.dumps(result)
    })
```

**Strict Mode（结构化输出保证）：**

当 `strict: True` 时，OpenAI 使用 constrained decoding（受限解码）保证输出 100% 符合 JSON Schema。代价是首次使用新 schema 时有几秒延迟（schema 需要编译），且 schema 有额外限制（不支持 `additionalProperties: true`、不支持某些复杂引用模式）。

### 2.2 Anthropic：Tool Use 原生集成

Anthropic 的 tool use 设计与 OpenAI 的核心差异在于：**工具调用是 content block 的一种类型，而非独立的 message 字段**。这使得模型可以在同一条消息中混合文本和工具调用。

**定义工具：**

```python
tools = [
    {
        "name": "get_order_status",
        "description": "查询订单当前状态。需要订单ID。当用户询问订单进度时使用。",
        "input_schema": {
            "type": "object",
            "properties": {
                "order_id": {
                    "type": "string",
                    "description": "订单编号，格式如 ORD-20250101-XXXX"
                }
            },
            "required": ["order_id"]
        }
    }
]
```

**tool_choice 控制：**

| 值 | 行为 |
|---|---|
| `{"type": "auto"}` | 模型自主决定（默认） |
| `{"type": "any"}` | 必须调用某个工具（任意） |
| `{"type": "tool", "name": "X"}` | 强制调用指定工具 |
| `{"type": "none"}` | 禁止工具调用 |

**响应结构：**

```python
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "我的订单 ORD-20250101-A1B2 到哪了？"}]
)

# response.content 是一个 block 列表：
# [
#   TextBlock(type="text", text="让我帮你查一下订单状态。"),
#   ToolUseBlock(type="tool_use", id="toolu_xxx", name="get_order_status", 
#                input={"order_id": "ORD-20250101-A1B2"})
# ]

# 执行工具后，把结果放回：
messages.append({"role": "assistant", "content": response.content})
messages.append({
    "role": "user",
    "content": [{
        "type": "tool_result",
        "tool_use_id": "toolu_xxx",
        "content": "订单已发货，预计明天送达"
    }]
})
```

### 2.3 Anthropic 2025 三大进阶能力

Anthropic 在 2025 年 11 月发布了三个 advanced tool use 功能，直接解决了大规模工具场景的工程痛点：

**Tool Search Tool（工具搜索工具）：**

问题：当你有 50+ 个工具时，全部定义塞进上下文消耗大量 token 且让模型选择困难。

解法：给工具标记 `defer_loading: true`，它们的定义不会出现在初始上下文中。Claude 在需要时通过内置的 tool_search 工具动态发现和加载相关工具定义。

效果：token 使用减少 85%，工具选择准确率从 79.5% 提升到 88.1%。

**Programmatic Tool Calling（程序化工具调用，PTC）：**

问题：Agent 需要连续调用多个工具并对结果做处理（如"查询 10 个用户的订单，汇总统计"），传统方式需要 10 次 LLM ↔ 工具往返。

解法：Claude 生成 Python 代码，在 sandbox 中执行，代码内可以调用工具。中间结果留在 sandbox 变量中，不回到模型上下文。只有最终 stdout 返回给 Claude。

效果：复杂研究任务 token 降低 37%，准确率提升 5 个百分点。

**Tool Use Examples（工具使用示例）：**

问题：JSON Schema 能描述参数类型但不能示范"什么情况下该填什么值"。

解法：在工具定义中加 `input_examples` 字段，给 1-5 个真实调用示例。

效果：复杂参数处理准确率从 72% 提升到 90%。

### 2.4 Google Gemini：Function Calling + 内置工具

Gemini 的 function calling 协议与 OpenAI 类似但有独特之处：

**内置工具（Built-in Tools）：**
- `google_search_retrieval` / `google_search`：模型可以主动搜索 Google
- `code_execution`：模型可以在 sandbox 中运行 Python 代码
- 两者可以与自定义函数同时使用

**Function Calling Mode：**
- `AUTO`：模型决定是否调用
- `ANY`：必须调用（可指定允许的函数子集 `allowed_function_names`）
- `NONE`：禁止调用

**独特特性：自动函数调用（Automatic Function Calling）**：在 Gemini SDK 中，如果你同时传入函数定义和实际的 Python 函数实现，SDK 可以自动执行工具调用并把结果回填，无需手动编写循环。

## 3. 协议对比总结

| 维度 | OpenAI | Anthropic | Gemini |
|---|---|---|---|
| 定义位置 | `tools[].function` | `tools[]` | `tools[].function_declarations` |
| Schema 格式 | JSON Schema | JSON Schema | OpenAPI-like (subset) |
| 并行调用 | ✅（`parallel_tool_calls`） | ✅（多 tool_use blocks） | ✅ |
| 强制调用 | `tool_choice: required` | `tool_choice: {type: "any"}` | `mode: ANY` |
| 严格输出 | `strict: true`（100%保证） | 高可靠但非 100% | 无原生 strict |
| 内置工具 | Code interpreter, file search | Computer use, bash, text editor | Google Search, Code execution |
| 动态发现 | ❌ | ✅（Tool Search Tool） | ❌ |
| 程序化调用 | ❌ | ✅（PTC） | ✅（auto function calling） |

## 4. MCP：Tool Use 的下一层抽象

### 4.1 从"硬编码工具"到"可发现工具"

传统 function calling 有一个根本问题：**工具定义是硬编码在应用代码中的**。你的 Agent 要接入 GitHub，你就要自己写 GitHub 的工具定义和执行逻辑；要接入 Slack，再写一套。每个应用都在重复做同样的集成工作。

MCP（Model Context Protocol）是 Anthropic 在 2024 年 11 月发布的开放协议，解决的就是这个问题：**让工具提供方（Server）标准化地暴露能力，让工具消费方（Client/Host）标准化地发现和调用。**

### 4.2 架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  MCP Host   │     │ MCP Client  │     │ MCP Server  │
│ (Claude,    │────▶│ (协议桥接)  │────▶│ (GitHub,    │
│  IDE, App)  │     │             │     │  Slack, DB) │
└─────────────┘     └─────────────┘     └─────────────┘
                         │
                    JSON-RPC 2.0
                    over stdio / HTTP
```

**三种原语：**

| 原语 | 控制方 | 用途 |
|---|---|---|
| **Resources** | 应用控制 | 可读数据（文件、API 响应）——类似 GET |
| **Tools** | 模型控制 | 可执行动作——类似 POST，模型决定何时调用 |
| **Prompts** | 用户控制 | 预定义的交互模板，用户手动触发 |

### 4.3 传输层：stdio vs Streamable HTTP

**stdio（本地模式）**：Client 在本地启动 Server 进程，通过 stdin/stdout 通信。一个 Server 配一个 Client。适合开发者工具（IDE 插件、本地 Agent）。优点是简单；缺点是需要本地安装 runtime，不支持浏览器，升级麻烦。

**Streamable HTTP（远程模式）**：Server 作为 HTTP 服务部署，多个 Client 可以连接。适合 SaaS 场景。优点是集中管理、即时更新、浏览器兼容；缺点是需要处理认证和网络问题。

### 4.4 MCP 与 Function Calling 的关系

MCP 不是 function calling 的替代品——它是**工具发现和执行的标准化层**。Agent 仍然通过 function calling / tool use 协议来表达"我要调用工具"的意图，但工具的**来源**从"应用内硬编码"变成了"从 MCP Server 动态获取"。

```
传统方式:
  App 硬编码 tools[] → LLM → App 执行

MCP 方式:
  MCP Server 暴露 tools → Client 发现 → 转为 tools[] 传给 LLM → LLM → Client 转发 → Server 执行
```

### 4.5 MCP 的工程价值

对 Agent 应用架构师来说，MCP 的核心价值在于：

**解耦**：工具的定义、实现、更新都由 Server 方负责。你的 Agent 代码不需要知道 GitHub API 怎么认证、Slack 的 rate limit 是多少——这些都封装在 Server 内。

**可组合**：一个 Host 可以同时连接多个 MCP Server（GitHub + Slack + Database + 内部系统），组合出复杂能力，而不需要在应用层一一集成。

**生态效应**：Server 写一次，所有支持 MCP 的 Host 都能用。截至 2025 年底，主流 IDE（Cursor、Windsurf、Claude Code）和 AI 平台都支持 MCP Client。

**安全审计**：MCP 2025-03 规范要求 OAuth 2.1 认证，Server 作为 Authorization Server 管理令牌。虽然实现上仍有摩擦（很多第三方不支持 Dynamic Client Registration），但至少有了统一的安全框架。

## 5. Tool Schema 设计：被忽视的工程实践

工具定义的质量直接影响模型的工具选择准确率。这不是"写个 description 就行"的事——它是一门微型的 API 设计。

### 5.1 Description 的黄金法则

Anthropic 的 context engineering 文档给出了 tool description 的设计原则：

**像给工程师写 API 文档一样写 description**——不是"这个工具可以做 X"，而是"在什么场景下使用这个工具、它的前置条件是什么、它会返回什么"。

**好的 description：**
```
"查询指定订单的当前物流状态。
使用场景：当用户询问订单配送进度时。
前置条件：需要有效的订单ID（格式 ORD-YYYYMMDD-XXXX）。
返回：物流状态对象，包含 status、location、estimated_delivery。
注意：如果订单未发货，返回 status='pending' 而非报错。"
```

**差的 description：**
```
"获取订单状态"
```

### 5.2 参数设计原则

**最小化必填参数**——每多一个 required 参数，模型出错概率就增加一份。如果某个参数有合理默认值，设为 optional。

**用 enum 约束可选值**——比起让模型猜"状态码可能是什么"，直接给 `"enum": ["pending", "shipped", "delivered", "cancelled"]`。

**description 比 type 重要**——模型从 description 理解参数的语义，从 type 理解格式。一个没有 description 的参数，即使类型正确，模型也可能传入语义错误的值。

**避免嵌套过深**——超过 3 层的嵌套对象会显著增加模型的错误率。如果业务上需要复杂结构，考虑拆成多个工具。

### 5.3 工具集治理

Anthropic 的启发式规则很实用：**如果一个人类工程师也分不清两个工具在什么情况下用哪个，模型也分不清。**

**避免功能重叠**——如果 `search_users` 和 `find_user_by_email` 功能高度重叠，合并成一个带 optional 参数的工具。

**工具数量控制**——研究表明 10 个以内的工具，模型选择准确率最高。超过 20 个时准确率明显下降。Anthropic 的 Tool Search Tool 就是为了解决 50+ 工具场景的。

**分组与命名空间**——当工具数量大时，使用命名前缀（`github.create_pr`、`github.list_issues`、`slack.send_message`）帮助模型建立分类。

### 5.4 Token 成本意识

每个工具定义都占 token。一个完整的工具定义（name + description + schema）通常消耗 200-500 token。20 个工具就是 4000-10000 token——这是实实在在的成本，且每次调用都要重复支付（除非用了 prompt caching 或 tool search）。

设计含义：
- 稳定的工具集用 prompt caching（放在 system 最前面）
- 大规模工具集用 deferred loading / tool search
- 定期审计：删除不再使用的工具定义

## 6. 内置工具：模型自带的能力

三大提供商都在模型层面提供了预置工具，无需自定义：

### 6.1 代码执行

| 提供商 | 工具名 | 能力 |
|---|---|---|
| OpenAI | Code Interpreter | Python sandbox，支持文件 I/O |
| Anthropic | code_execution (PTC) | Python sandbox，可组合自定义工具 |
| Gemini | code_execution | Python sandbox，支持 numpy/matplotlib |

代码执行的核心价值：让模型从"说代码"变成"跑代码"——模型生成代码、在 sandbox 中执行、看到结果、基于结果继续推理。这是 Agent 能做计算密集型任务的关键。

### 6.2 搜索

| 提供商 | 工具名 | 能力 |
|---|---|---|
| OpenAI | web_search | 模型自主搜索网络 |
| Gemini | google_search | Google Search grounding |
| Anthropic | (无原生，但有 MCP server) | 通过 MCP 连接搜索服务 |

### 6.3 计算机操作（Anthropic 独有）

Anthropic 的 `computer_use` 工具让 Claude 可以操作桌面环境：移动鼠标、点击、键盘输入、截屏。这是目前最接近"通用 Agent"的能力——理论上任何人类能在电脑上做的事，它都能做。

实践限制：延迟高（每步需要截屏-分析-操作）、容易出错（UI 变化敏感）、成本高（每步消耗大量 token 处理图像）。适合做自动化测试、UI 验证，不适合做高频实时操作。

        ], ensure_ascii=False)
    elif name == "get_order_detail":
        return json.dumps({
            "order_id": input_data["order_id"],
            "status": "shipped",
            "items": [{"name": "机械键盘", "qty": 1, "price": 299.0}],
            "tracking": {"carrier": "顺丰", "number": "SF1234567890", "estimated": "2025-06-24"},
        }, ensure_ascii=False)
    elif name == "initiate_refund":
        return json.dumps({"success": True, "refund_id": "REF-001", "message": "退款已提交审核，预计3个工作日内到账"})
    elif name == "transfer_to_human":
        return json.dumps({"success": True, "queue_position": 3, "estimated_wait": "2分钟"})
    return json.dumps({"error": f"未知工具: {name}"})


# ===== Agent 主循环 =====
def run_customer_service_agent(user_message: str, max_turns: int = 10):
    """
    运行客服 Agent。
    max_turns: 防止无限循环的预算控制（每次工具调用+回复算一轮）
    """
    messages = [{"role": "user", "content": user_message}]
    
    for turn in range(max_turns):
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            system=[{
                "type": "text",
                "text": SYSTEM_PROMPT,
                "cache_control": {"type": "ephemeral"}
            }],
            tools=TOOLS,
            messages=messages,
        )
        
        # 把 assistant 的完整响应加入历史
        messages.append({"role": "assistant", "content": response.content})
        
        # 如果模型决定结束（不调用工具）
        if response.stop_reason == "end_turn":
            final_text = "".join(
                block.text for block in response.content if block.type == "text"
            )
            return {"response": final_text, "turns_used": turn + 1}
        
        # 处理工具调用
        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"  [Turn {turn+1}] Calling: {block.name}({json.dumps(block.input, ensure_ascii=False)})")
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            
            # 把所有工具结果一起放回（支持 parallel tool calls）
            messages.append({"role": "user", "content": tool_results})
    
    return {"response": "[预算耗尽，请转人工处理]", "turns_used": max_turns}


# ===== 使用示例 =====
if __name__ == "__main__":
    # 示例 1: 简单查询
    result = run_customer_service_agent("我最近买的键盘到哪了？")
    print(f"回复: {result['response']}")
    print(f"使用轮次: {result['turns_used']}")
    
    # 示例 2: 需要多步的操作
    result = run_customer_service_agent("那个键盘我不想要了，帮我退款")
    print(f"回复: {result['response']}")

## 8. 设计决策清单

面对一个新的 Agent 项目，工具层按以下顺序决策：

**第一步：确定工具来源**
- 纯自建 API → 直接用各提供商的 tool use 格式定义
- 集成第三方服务 → 优先找现成 MCP Server
- 混合场景 → MCP Server（第三方）+ 自定义 tools（业务逻辑）

**第二步：确定工具数量和加载策略**
- ≤10 个工具 → 全部塞入 tools 参数，用 prompt caching
- 10-50 个 → 按场景分组，通过 routing agent 动态加载子集
- 50+ 个 → 用 Tool Search Tool（Anthropic）或自建 RAG-over-tools

**第三步：确定输出可靠性要求**
- 格式必须 100% 可靠（用于程序消费）→ OpenAI strict mode 或 tool use（Anthropic tool use 本质就是结构化输出）
- 可以容忍偶尔格式异常 → 标准 tool use + 客户端校验 + 重试

**第四步：确定安全边界**
- 只读操作 → 低风险，可以自动执行
- 写操作（创建/修改/删除）→ 需要确认机制（Hooks / HITL gate）
- 不可逆操作（退款/删除/发布）→ 必须人工审批

**第五步：确定并行策略**
- 工具间有依赖（B 需要 A 的结果）→ 顺序调用
- 工具间独立 → 启用 parallel tool calls，减少往返
