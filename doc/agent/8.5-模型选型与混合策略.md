# 8.5 模型选型与混合策略：Self-hosted（vLLM/SGLang）vs Managed / 开源 vs 闭源 / 级联路由 / 数据主权

> 一个能用的 Agent 通常用了 2-5 个不同的模型组合。本篇把"用哪个模型"这道架构题拆成 5 个独立维度，给出工程决策矩阵以及自托管推理栈的关键选型。

---

## §0 真问题：单一供应商的 5 个隐性账单

很多团队一开始用 OpenAI/Anthropic 单家 API，几个月后会发现一连串问题：

1. **数据合规红线**：金融/医疗/政府客户上来就问"数据有没有出境"，单一外部 API 直接断送 deal；
2. **价格震荡**：Provider 每 6 个月调一次价（向上或向下），单点依赖导致预算不可预测；
3. **故障同时下线**：OpenAI 2024-2025 多次区域级故障 3+ 小时，单依赖 = 完全停摆；
4. **速率上限触顶**：批量任务在配额前撞墙，Tier 5 客户也照样 429；
5. **任务-模型不匹配**：所有请求走 GPT-5，简单分类也烧 $0.01；
6. **微调封闭**：闭源模型给不了真正意义上的领域微调（除部分 SFT 接口）；
7. **数据飞轮断裂**：8.6 要把日志变 SFT 数据，闭源模型不能消化自己的 trace；

正确的第一性原理：

```
                  ┌─ 数据敏感度（出不出境）
                  ├─ 任务难度（要不要 frontier）
                  ├─ QPS / Token Volume（高频 → 自托管划算）
模型选型矩阵 = f(│
                  ├─ 延迟 SLO（自托管 + vLLM 提升上限）
                  ├─ 微调需求（是否需要领域 SFT/DPO/RFT）
                  └─ 团队能力（运维 GPU 集群是不是负担）
                                  )
```

**唯一正确答案**：**混合**。前沿闭源跑高层规划/复杂推理，自托管/小开源模型跑高频执行子任务，按任务路由。下面把这道选型题拆成 5 个独立维度。

## §1 维度一：开源 vs 闭源（Capability vs Control）

### 1.1 2026 主流模型梯队

| 梯队 | 闭源代表 | 开源代表 | 关键差距 |
|---|---|---|---|
| 顶尖 | Claude Opus 4.5 / GPT-5 / Gemini 3 Pro | DeepSeek V3.x / Qwen3-Max / Llama 4 Behemoth | SWE-bench Verified、长上下文，闭源 +5-10pp |
| 中端 | Claude Sonnet 4.5 / GPT-5 mini | Qwen3-235B / DeepSeek V3 / Llama 4 Scout | 多数 Agent 场景持平 |
| 轻量 | Claude Haiku / GPT-5 nano | Qwen3-30B / Llama 4 Maverick 17B | 简单任务无差异 |
| 工具调用专项 | Claude (强 function calling) | Qwen3-Coder / Codestral / GLM-4 | 闭源在 function calling 可靠性仍领先 |

**单步可靠性差异在多步 Agent 中指数级放大**（[MindStudio 2026](https://www.mindstudio.ai/blog/open-source-vs-closed-source-ai-models-agentic-workflows)）："A model that's 90% reliable on a single step is only 59% reliable across five sequential steps." 这就是为什么很多 Agent 跑长链路时即使在 benchmark 上接近的开源模型，体感差很多。

### 1.2 选型决策表

| 场景 | 推荐 |
|---|---|
| 数据 must on-prem（金融/医疗/政府） | 开源自托管 |
| Frontier reasoning（深度推理、SWE 重任务） | 闭源（Opus 4.5 / GPT-5） |
| 高频低复杂度（分类、摘要、提取） | 开源中端（Qwen3-30B） |
| 需要领域微调 | 开源（多数闭源不支持真微调） |
| 早期 PoC、团队没有 ML Eng | 闭源 API |
| 多语言、超长上下文 | 闭源（Gemini 1M+ context） |
| 工具调用密集 | 闭源（Claude）或微调开源 |
| 月成本 >$50K | 开始考虑自托管 |

### 1.3 行业典型组合

观察 2025-2026 多个公开案例（Cursor、Devin、Perplexity、Anthropic Customers）：

- **Planner / Critic** = Frontier 闭源（少调用、高质量）
- **Executor** = Sonnet/Haiku 级 或 自托管 70B-class（高频）
- **Embedder / Classifier / Reranker** = 自托管小开源模型（极高频）
- **Image / Multimodal** = 通常闭源或专用模型（Gemini / Sonnet / GPT-4o）
- **Guardrail / Safety** = 小专用分类器（Llama Guard 4 / Prompt Guard 2）

## §2 维度二：Managed vs Self-hosted（API vs 自托管）

### 2.1 拐点公式

```
自托管划算条件：

monthly_token_volume × token_price > GPU_cost + Eng_cost + reliability_premium

经验值：
- < 5M tokens/day → Managed API
- 5M - 50M tokens/day → 看 GPU 自有 vs 租赁
- > 50M tokens/day → Self-host 显著划算
- > 500M tokens/day → Self-host 必选（且省 60-80%）
```

[MindStudio 2026] 数据：自托管 Qwen3 72B / Llama 4 Scout 在规模下可节省 **80-95% per-inference cost**。

### 2.2 自托管的"隐藏账单"

不要只算 GPU 成本，还有：

| 隐藏成本 | 估算 |
|---|---|
| ML Engineer salary | 2-4 人，年 $400k-1M |
| GPU idle time | 自托管利用率 60-70%（顶配也只能到 80%） |
| 模型升级 / re-deploy | 每代模型 1-2 周工程 |
| 故障运维 | OnCall + Incident, 季度性 |
| Capacity buffer | 防限流要 1.5-2× 实际峰值 |
| 多区域 / 容灾 | 主备 GPU 加 50%+ |

实际盈亏拐点比"纯算 token 单价"要高 30-50%。

### 2.3 三种自托管运营模式

| 模式 | 控制力 | 复杂度 | 成本 |
|---|---|---|---|
| 裸金属 GPU + vLLM/SGLang 自部 | 最高 | 最高 | 最低单价 |
| 托管 GPU（Modal/Together/Fireworks/Replicate） | 中 | 中 | 中 |
| Managed inference（Bedrock/Vertex/Azure OAI） | 低 | 低 | 高 |

中间档（Modal / Together / Fireworks / Replicate / DeepInfra）是 **被严重低估的甜蜜区**：吃到自托管 80% 的成本节省，几乎零运维。

## §3 维度三：推理引擎选型（vLLM / SGLang / TensorRT-LLM）

### 3.1 三大主流引擎

| 引擎 | 起源 | 核心创新 | 适用 |
|---|---|---|---|
| vLLM | 2023.6 UC Berkeley → PyTorch Foundation | PagedAttention + Continuous Batching | 通用、社区最大 |
| SGLang | 2024.1 → PyTorch | RadixAttention（前缀复用）+ CPU 调度优化 | 重复 prompt 多、Agent 场景 |
| TensorRT-LLM | NVIDIA 闭源生态 | 深度内核优化 + 硬件绑死 | NVIDIA + 极致性能 |

[Inclusion AI 2025 评测](https://www.inclusion-ai.org/blog/llm-landscape-vllm-sgl/) 关键发现：

- **功能趋同**：两者都支持 continuous batching、PagedAttention、RadixAttention、chunked prefill、speculative decoding、disaggregated serving、CUDA graphs；
- **生态差异**：vLLM 贡献者 2× SGLang，Red Hat 是骨干企业；SGLang 由 xAI / Skywork / Oracle / LinkedIn 推动；
- **性能**：阿里云 2025 评测在 Qwen 系列上 SGLang 略胜（因 RadixAttention 在重复 prompt 场景受益）；
- **响应**：vLLM issue 12h-3d，SGLang 3-5d。

### 3.2 Agent 场景特殊性：选 SGLang

Agent 应用的 prompt 模式：

- 同一 system prompt + tool defs 反复使用（90%+ 重复）；
- 多步 trajectory，每步前缀都包含前文；
- 多 Agent 协作时 shared context 多。

**SGLang 的 RadixAttention 天生为这种模式优化**：跨请求共享 KV cache prefix，prefill 计算大幅削减。Agent 场景推荐 SGLang。

通用 chatbot / 完全异质请求场景 vLLM 更稳。

### 3.3 关键配置（无论选哪个）

```yaml
# vLLM/SGLang 通用关键参数
max_num_seqs: 256              # 单 batch 最大请求数，越大吞吐越高、延迟越长
gpu_memory_utilization: 0.9    # KV cache 给多少
max_model_len: 32768           # 上下文上限，过长浪费 KV
enable_prefix_caching: true    # vLLM 必开（SGLang 默认开）
enable_chunked_prefill: true   # 长 input 必开，避免阻塞 decode
speculative_model: meta-llama/...-1B  # 必开 speculative 1.5-3× decode 提速
num_speculative_tokens: 5
```

### 3.4 性能压测必做

不要相信宣传数字，自己跑：

```bash
# vLLM 自带 benchmark
python benchmarks/benchmark_serving.py \
    --backend vllm \
    --model meta-llama/Llama-3.3-70B-Instruct \
    --dataset-name sharegpt \
    --num-prompts 1000 \
    --request-rate 10
```

关键指标：**P99 TTFT、P99 ITL、tokens/s/GPU、单卡 RPS**。在自己真实流量分布下跑。

## §4 维度四：级联路由架构

这部分与 8.3 §4 联动，本节聚焦"哪些模型组成 cascade"的工程决策。

### 4.1 三层模型架构

```
Layer 1 (Cheap Fast)        Qwen3-30B / Haiku / GPT-5 nano
   │ 处理 60-80% 流量
   │ 阈值通过率高
   ▼
Layer 2 (Mid)               Sonnet 4.5 / GPT-5 mini / DeepSeek V3
   │ 处理 15-30% 流量
   ▼
Layer 3 (Frontier)          Opus 4.5 / GPT-5 / Gemini 3 Pro
     处理 < 5% 流量
```

### 4.2 跨供应商 vs 单供应商内部

| 类型 | 优点 | 缺点 |
|---|---|---|
| 单供应商内 cascade（如 Haiku→Sonnet→Opus） | 行为一致、prompt 复用 | 供应商故障同步停摆 |
| 跨供应商 cascade（如 Qwen→Sonnet→GPT-5） | 抗故障、价格弹性 | prompt 适配多家、维护成本高 |
| 混合（开源 + 闭源） | 兼顾敏感数据 + 兜底质量 | 最复杂 |

**生产推荐**：开源自托管做 Layer 1（处理 60%+ 流量、敏感数据可留内部），闭源做 Layer 2/3 兜底。

### 4.3 Router 模型本身

Router 也是模型。两种实现：

- **LLM Router**：让一个便宜 LLM 判断难度 → 派单（成本 ~$0.0001/req）；
- **Trained Classifier Router**：拿历史数据训一个 1-3B 模型（成本 ~ms 推理，几乎 0）。

中大规模生产推荐 trained classifier，长期可靠且廉价。

## §5 维度五：数据主权与合规

### 5.1 数据流向矩阵

| 部署方式 | 数据流向 | 适用合规 |
|---|---|---|
| OpenAI/Anthropic API | 数据出 → 供应商基础设施 | 一般商用，签 DPA |
| Anthropic via AWS Bedrock | 数据进 AWS Region 不出 | HIPAA、GDPR-EU |
| Azure OpenAI | 数据进 Azure tenant | HIPAA、GDPR-EU、FedRAMP |
| Vertex AI Gemini | 数据进 GCP tenant | 同上 |
| Self-host 开源 | 完全留内部 | 任何合规、出口管制、机密 |

**关键陷阱**：API 不留数据 ≠ 训练不用数据。要在合同 / 控制台 settings 显式关闭 "data used for training"。GDPR 跨境（如 EU 数据用 US 模型）需要 SCCs。

### 5.2 中国大陆合规额外约束

- 算法备案（互联网信息服务深度合成 / 生成式 AI）；
- 数据出境评估（PIPL）；
- 大模型备案（生成式 AI 服务管理办法）；
- 关键信息基础设施运营者的本地化要求。

很多场景下**只能用境内可用模型 + 境内 GPU + 境内 vector DB**，决策被强约束。

### 5.3 PII 处理

无论模型本地还是 API：

- **入口 redact**：在 prompt 进 LLM 前过 PII 检测，遮蔽身份证 / 手机 / 邮箱（与 6.2 联动）；
- **trace redact**：7.3 的 OTel Collector 处必须 redact；
- **prompt cache 隔离**：8.3 提到的多租户隔离也是合规要求。

## §6 微调决策：什么时候自己训

### 6.1 微调梯度

```
Prompt-only           （0 训练，最低门槛）
   ↓ 不够
Few-shot / RAG         （0 训练，用上下文）
   ↓ 不够
Prompt tuning / Soft prompt  （minimal 训练）
   ↓ 不够
LoRA / QLoRA           （轻量微调，常用）
   ↓ 不够
Full SFT               （重量级）
   ↓ 还不够
DPO / KTO              （偏好对齐）
   ↓ 还不够
RFT / GRPO / PPO       （Reinforcement Fine-Tuning，2025-2026 新热点）
```

**关键判断**：先把 prompt + RAG + few-shot + cascade 做到极致，能解决问题就不要训。**90% 团队的"需要微调"实际是 prompt engineering 没做够**。

### 6.2 微调真正必要的 3 个场景

1. **领域专属术语 / 文档风格**：通用模型怎么 prompt 也说不对（医疗、法律、金融特定表达）；
2. **大量结构化输出要求**：通用模型 95% 准但你需要 99.5%，SFT 一组 schema 例子能拉满；
3. **极小模型部署**：要在边缘 / 移动端跑 1-3B，必须微调到任务上才能用。

### 6.3 微调技术选型

| 技术 | 适合 | 资源 |
|---|---|---|
| LoRA / QLoRA | 90% 的微调场景 | 单卡 A100/H100 可训 70B |
| Full SFT | 需要彻底改变模型行为 | 多卡 H100 + 数据 100k+ |
| DPO | 偏好对齐（哪个回答更好的二选一） | 中等 |
| RFT (GRPO/PPO) | Agent 多步任务、可定义 reward | 高，但 2025 平台化（OpenAI RFT、HF TRL） |

## §7 决策矩阵：把 5 维拍平

| 场景 | 模型层 1 | 模型层 2 | 模型层 3 | 部署 | 引擎 |
|---|---|---|---|---|---|
| 通用 chatbot（消费级） | Haiku | Sonnet | Opus | API | - |
| SaaS B2B（中小） | Haiku + 自托管 Qwen3-30B | Sonnet | Opus | Modal/Together | vLLM |
| 大型企业内部 Agent | 自托管 Qwen3-30B | Sonnet via Bedrock | Opus via Bedrock | 自部 + Bedrock | SGLang |
| 金融 / 医疗 / 政府 | 自托管 Qwen3 / Llama 4 | 自托管 70B/235B | 自托管 235B | 完全自托管 | SGLang/vLLM |
| Deep research Agent | Sonnet（高频步骤） | Opus（关键推理） | - | API | - |
| SWE Agent | Sonnet（planner） | Opus（hard solve） | Qwen3-Coder 32B（lint/format） | API + 自托管 | SGLang |
| 边缘 / 移动端 | 微调 Qwen3-7B / Llama 4 17B | - | - | 设备本地 | llama.cpp / vLLM |

## §8 反模式

| 反模式 | 后果 |
|---|---|
| 所有请求走 frontier 模型 | 月账单 10× 实际需要 |
| 早期就自托管 | 团队精力都在 ops，业务停滞 |
| 自托管不开 speculative | 浪费 1.5-3× 单卡能力 |
| 跨 region 部署 + 不开 prompt cache | TTFT 飙升 + cost 飙升 |
| 单供应商生产，无 fallback | 一次故障停 4 小时 |
| 不签 DPA 就跑用户数据 | 合规炸弹 |
| 误以为 API "不留数据"=不训练 | settings 没关，数据被用 |
| 微调 = 万能解 | 训了 SFT 还是 prompt 问题 |
| Router 用 GPT-5 当裁判 | 路由本身比执行还贵 |
| 自托管利用率 30% | TCO 比 API 还贵 |
| 跨供应商 cascade 不维护 prompt 适配 | 一家变 schema 全链路崩 |

## §9 S0 → S6 路线

| 阶段 | 关键里程碑 |
|---|---|
| S0 | 单一闭源 API 跑 demo |
| S1 | 加 fallback provider（跨家） + 签 DPA |
| S2 | 加 Haiku / nano 做廉价 cascade 第一层 |
| S3 | 引入自托管小模型（30B 级）跑高频任务 |
| S4 | 上自部推理（vLLM/SGLang）+ speculative + prefix cache |
| S5 | 跨供应商 + 自托管混合 cascade，trained router |
| S6 | 领域微调闭环（与 8.6 数据飞轮联动） |

90% 团队应该停在 S2-S3，盲目跳 S4-S6 = TCO 暴涨。

## §10 一页总结

把 8.5 压成一句话：**模型选型不是"选一个最好的"，而是"按数据敏感度 × 任务难度 × QPS × 延迟 × 微调需求 × 团队能力"6 维路由组合；前沿闭源做规划/复杂推理，自托管开源做高频执行子任务，自托管推理选 vLLM/SGLang（Agent 场景优先 SGLang），从 API → cascade → mixed → fine-tune 渐进演化。**

承接关系：
- **8.3 成本**：cascade 路由的成本逻辑；
- **8.4 延迟**：自托管 + vLLM/SGLang 是 TTFT 优化的另一路；
- **8.6 数据飞轮**：开源模型才能完成 trace → SFT → 闭环；
- **6.3 权限**：多供应商凭据隔离；
- **7.4 持续改进**：模型升级时的 shadow + canary；
- **5.3 长上下文**：选模型时必须看真实 context 性能而非纸面长度。
