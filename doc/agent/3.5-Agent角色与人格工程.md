# 3.5 Agent 角色与人格工程：Persona / Role / Capability Boundary / Specialization 反模式

> **写在前面**：本文不讲"如何让 Agent 像 Marketing Manager 那样说话"那种 cosplay 教程，而是讲 **Multi-Agent 系统里 Persona / Role / Capability Boundary 三个概念的工程实质**，以及为什么 2023-2024 流行的"5 个 Agent 配 5 个 Persona 角色扮演"在 2025-2026 已经被业界判定为反模式。承接 3.1 设计模式、3.3 框架对比；为 3.6 分解聚合、3.7 可靠性铺垫。

## §0 真问题：为什么 "5 个 Persona 拼凑一个 Marketing Workflow" 跑不稳

2023 年 AutoGen / CrewAI 火起来之后，多人涌入"Multi-Agent = 多个 Persona 协作"的范式。典型架构：

```
- PlannerAgent: 你是项目经理，规划任务...
- ResearcherAgent: 你是研究员，认真调研...
- WriterAgent: 你是营销文案写手，文笔活泼...
- ReviewerAgent: 你是 senior editor，严格挑错...
- PublisherAgent: 你是发布管理员，最终决定...
```

跑两周后会出现的真问题：

第一，**Planner 不知道 Researcher 实际能干什么**。Planner 只看到 ResearcherAgent 的 system prompt，但实际 Researcher 受限于工具（无网络、context 8K）。Planner 制定的任务超出 Researcher 实际能力时，Researcher 会"角色扮演成功"——伪造结果。

第二，**Writer 重写 Reviewer 的反馈**。Reviewer 提的修改意见在被 Writer 读到时，Writer 解读为"Reviewer 是另一个角色，我有创作自由"，于是部分接受部分忽略，导致回归问题反复出现。

第三，**Persona 越多越没法 debug**。出问题时不知道是 Planner 没规划清楚、Researcher 没研究透、还是 Writer 没听懂。每个 Agent 都"做了自己该做的"，但**最终输出仍然不达标**。

第四，**Multi-Persona 加成的实际收益小**。Cognition 的 Walden Yan 在 2025-06 《Don't Build Multi-Agents》里直说：把 Devin 内部改成 5 个 sub-agent 互发消息，结果更糟（[Cognition Latent.Space 访谈, 2026-04](https://www.latent.space/p/cognition)）。同期 Anthropic 公开的 multi-agent research system 是**orchestrator + worker** 模式，**没有 Persona**——只有任务边界和工具边界。

第五，**Persona 把"职位"幻觉化**。"你是 senior engineer with 10 years experience" 这种声明对模型行为的影响远小于工具集 + 任务约束。"角色"是一种文化符号，模型并不真有职业经验，靠 prompt 模拟出来的"职业判断"反而更不可靠。

那 Persona 到底有没有用？什么时候用？Multi-Agent 的正确"角色化"是什么？这是本文要回答的。

## §1 第一性原理：角色、人格、能力边界的三层模型

把"Agent 是谁"拆成三层：

### 1.1 Persona（人格）：表达层

Persona 控制**输出风格**：语气、用词、详细度、专业度。
- 适合：面向用户的对话型 Agent（客服、营销、教学）。
- 工程实质：prompt 后半段的若干句指令 + few-shot 例子。
- 评价标准：用户感知。可主观打分。

### 1.2 Role（角色）：任务层

Role 控制**做什么任务**：负责什么阶段、输出什么 artifact、用什么 metric。
- 适合：Multi-Agent 系统里的分工边界。
- 工程实质：**任务 spec + 输入 schema + 输出 schema + 评估 criterion**。
- 评价标准：客观指标。任务完成率、输出 schema 合规率。

### 1.3 Capability Boundary（能力边界）：约束层

Capability Boundary 控制**能做什么**：可用工具、可访问数据源、可调用其他 agent、权限范围。
- 适合：所有 production Agent。
- 工程实质：tool 列表 + 权限 scope + retrieval source + max budget。
- 评价标准：security audit + blast radius 控制。

**关键洞察**：这三层在 2023-2024 的 Persona-driven 设计里被搅在一起——`"You are a senior security engineer with deep knowledge of..."` 这种 prompt 同时混了 Persona（表达）、Role（任务）、Capability（能力）的暗示，但**没有任何强制约束**。结果是模型按 prompt 文学性的方式拼凑这三层，而非工程性地满足约束。

**正确的工程做法**：
- Persona 写最少（甚至不写），只控制风格。
- Role 写成 **task spec**：输入是什么、输出 schema 是什么、成功标准是什么。
- Capability Boundary 写成 **代码层强制约束**：tool 白名单、retrieval scope、权限 IAM。

## §2 数据：单 Agent vs Multi-Agent 的实测对比

### 2.1 Anthropic Research System (2025-06)

Anthropic 公开的 multi-agent research system（[How we built our multi-agent research system, 2025-06](https://www.anthropic.com/engineering/multi-agent-research-system)）核心数据：

- LeadResearcher（Claude Opus 4）+ Subagents（Claude Sonnet 4）**单一研究 eval 上比 single-agent Opus 4 提升 90.2%**。
- Token 使用量解释 80% 的性能方差；token + tool calls + model 三个因子解释 95%。
- Multi-agent 系统**用 token 约是单聊 15×**，单 agent 约 4×。
- 任务并行：lead 启动 3-5 subagents 同时跑，3+ tools 并行，**复杂查询时间砍 90%**。

但 Anthropic 自己注明**这不是普适解**：multi-agent 适用于"高度可并行 + 信息超 single context + 大量复杂工具"的任务；**不适用**于"高 inter-agent 依赖 + 共享上下文 + 实时协作"的任务（如绝大多数代码任务）。

### 2.2 Cognition Devin (2025-06 → 2026-04)

Walden Yan 2025-06 写《Don't Build Multi-Agents》：踩坑后的反思——给 Devin 加 MCP 工具能互相发消息、互相 spawn，结果一团乱。

2026-04 在 Latent.Space 访谈里更新结论：
- **依然 manager + sub-agent**，而非 peer-to-peer。
- Sub-agent 主要价值是 **context compression**，本质是"高级 tool call"。
- 长跑单 agent（一个 Devin 跑几小时重建一个 Windows-like OS）**比 fan-out 多 agent 效果更好**。
- 例外：跨模型 orchestration（OpenAI + Anthropic + Google 协作）在 e2e 测试这种任务里偶尔不可替代。
- 真正改变是模型成熟到能 **disagree**——"Devin 现在会反驳我"——这是 multi-agent 协作的前提。

### 2.3 学术界 (MASS, MAST, ChatDev 后续)

[Emergent Coordination in Multi-Agent Language Models, arXiv:2510.05174, 2025-10](https://arxiv.org/html/2510.05174v3) 表明：
- 多个 LLM agent 之间能涌现简单的协调行为（分工、轮流发言）。
- 但**冲突解决、deadlock 打破、共识形成**依然依赖外部 orchestrator，而非纯涌现。
- 给每个 agent 独立 persona 并不显著提升协作效率；任务边界清晰度才是关键。

### 2.4 综合判断

| 场景 | Single-Agent | Multi-Agent (orchestrator + worker) | Multi-Persona (无 orchestrator) |
|------|------------|----------------------------------|--------------------------------|
| 代码 / SWE 任务 | ✅ 主流 | ⚠️ 有限场景 | ❌ 反模式 |
| Deep Research / 跨源调研 | ❌ context 不够 | ✅ Anthropic 范式 | ❌ |
| 客服 / Single-turn 对话 | ✅ | ❌ 过度工程 | ❌ |
| 长时间自治任务（Devin/Codex 类） | ✅ 长跑 | ⚠️ 短期任务 | ❌ |
| 复杂多模态协作（语音+视觉+文本） | ❌ | ✅ 模态各自专攻 | ❌ |
| 红队 / 对抗博弈（Debate） | ❌ | ✅ 故意制造对立 | ⚠️ Persona 可作为博弈标识 |

## §3 工程做法：Role-Spec-Driven 而非 Persona-Driven

### 3.1 Role 写成代码而非散文

**反模式**：
```python
researcher_agent = Agent(
    name="Researcher",
    system_prompt="""You are a senior research analyst with 10+ years of 
    experience. You excel at deep web research, synthesizing diverse sources, 
    and producing high-quality reports..."""
)
```

**推荐做法**：
```python
@dataclass
class RoleSpec:
    name: str
    input_schema: type[BaseModel]      # Pydantic
    output_schema: type[BaseModel]
    allowed_tools: list[str]            # 强制白名单
    retrieval_scope: list[str]          # 强制 retrieval namespace
    max_budget_tokens: int
    max_budget_tools: int
    success_criterion: Callable[[Any], bool]   # 客观判断
    persona_hint: str = ""              # 可选，只控风格

researcher_role = RoleSpec(
    name="researcher",
    input_schema=ResearchQuery,
    output_schema=ResearchFindings,
    allowed_tools=["web_search", "url_fetch"],
    retrieval_scope=["public_web", "arxiv"],
    max_budget_tokens=50_000,
    max_budget_tools=20,
    success_criterion=lambda x: len(x.citations) >= 3 and x.confidence > 0.6,
    persona_hint="Use neutral, factual tone."
)
```

差别：
- 输入 / 输出 schema **强制定义**——下游知道能拿到什么字段。
- 工具 / scope / budget **可在 runtime 强制执行**，不靠模型"自觉"。
- success_criterion 是**可执行代码**，不是模型自评。
- Persona 退化为可选风格 hint。

### 3.2 Orchestrator 主导任务分发

Orchestrator-Worker 模式的关键是：**Orchestrator 看得到 Worker 的 RoleSpec（schema + 工具 + budget），而非 Persona prompt**。

```python
class Orchestrator:
    def __init__(self, workers: dict[str, RoleSpec]):
        self.workers = workers
    
    def decompose(self, task: str) -> list[Subtask]:
        # LeadResearcher 用 reasoning model 拆任务
        return reasoning_model.complete(
            messages=[{"role": "user", "content": build_decompose_prompt(
                task=task, 
                available_workers=[w.summary() for w in self.workers.values()]
            )}],
            reasoning_effort="medium",
            response_format=list[Subtask]
        )
    
    def dispatch(self, subtask: Subtask) -> WorkerResult:
        role = self.workers[subtask.worker_name]
        # 强制约束在外层
        with Sandbox(tools=role.allowed_tools, budget=role.max_budget_tokens):
            result = run_worker(role, subtask.input)
        assert role.success_criterion(result), f"Worker failed: {result}"
        return result
```

注意：约束在 Sandbox 层强制，不靠 worker prompt 自觉。这是 Capability Boundary **代码化**。

### 3.3 Persona 的少数合理使用场景

- **用户感知重要**：教学 Agent 需要友善耐心；客服 Agent 需要专业克制；
- **对抗博弈需要立场**：Debate / Devil's Advocate / Red Team 中 Persona 是显式工具，让多个 agent **故意持不同立场**；
- **品牌一致性**：企业 Agent 用 Persona 控制语气、措辞、避忌词。

这三类用 Persona 的本质是：**Persona 是为了影响 surface output**，而不是为了"模拟职业能力"。

### 3.4 Capability Boundary 的工程实现

```python
# 1. 工具白名单：在 tool dispatcher 层 enforce
def execute_tool(agent_role: str, tool_name: str, args: dict):
    if tool_name not in ROLE_TOOL_WHITELIST[agent_role]:
        raise PermissionError(f"{agent_role} cannot use {tool_name}")
    return TOOL_REGISTRY[tool_name](**args)

# 2. Retrieval scope：在 vector DB query 层 enforce  
def retrieve(agent_role: str, query: str):
    namespaces = ROLE_RETRIEVAL_SCOPE[agent_role]
    return vector_db.search(query, namespace=namespaces)

# 3. Token budget：在 model client 层 enforce
class BudgetEnforcedClient:
    def __init__(self, base_client, max_tokens):
        self.base = base_client
        self.budget = max_tokens
        self.spent = 0
    
    def complete(self, **kwargs):
        if self.spent >= self.budget:
            raise BudgetExceeded()
        resp = self.base.complete(**kwargs)
        self.spent += resp.usage.total_tokens
        return resp

# 4. IAM scope：worker 拿到的 credential 在 sandbox 内有限权限
worker_creds = STSClient.assume_role(
    role_arn=f"arn:aws:iam::xxx:role/{agent_role}-worker",
    duration_seconds=3600
)
```

**这才是"Role" 的工程含义**——不是 prompt 里写"你是研究员"，而是 IAM、tool whitelist、retrieval scope、budget 在系统层把"能做什么"卡死。

## §4 决策矩阵：选 Persona 还是 Role-spec

| 你的情境 | 推荐 |
|---------|------|
| 客服 / 单轮对话 Bot | 一个 Persona，不要 Role 分层 |
| Devin / Codex 代码 Agent | **长跑单 Agent**，sub-task spawn 时用 RoleSpec 包装 |
| Deep Research / 跨源调研 | Orchestrator + worker（Anthropic 范式），RoleSpec 强制 |
| Multi-modal Agent（视觉+文本+语音） | 模态 worker 分立，每个 worker = 1 个 Role |
| Debate / Red Team / 对抗推演 | 显式 Persona + Role + 立场 |
| Workflow 自动化（如 RPA） | Pipeline 阶段 = Role，不需要 Persona |
| 内部 dev tool | 单 Agent + 工具丰富，不要 Multi-Agent |

## §5 反模式与陷阱

**反模式 1：把"职位 + 经验年数"写进 prompt**。"You are a senior security engineer with 15 years experience" 这种 declaration 对模型的实际能力没影响，但会让 prompt 变得冗长。

**反模式 2：5+ Persona 互相 talk**。出错时无法归因。改为 orchestrator + worker。

**反模式 3：Persona 与 Capability 混淆**。`"You are an ethical hacker who can only test public targets"`——但工具层没限制，模型 prompt-injected 后就跑到内网了。正确做法：**Capability 在代码层强制**。

**反模式 4：让 Agent 自己声明 Role**。`"What role would you like to play for this task?"`——把 Role 决策让模型自己做。Orchestrator 必须显式分配。

**反模式 5：忽视 Sub-agent 的 context compression 价值**。Cognition 观点：sub-agent **最大的价值是把 100K 调研结果压成 2K 给 lead**，而不是真协作。把它当 tool call 用最实用。

**反模式 6：Persona 累加 = 能力累加**。"You are an expert in SQL + Python + React + DevOps" 不会让模型四个领域都强；现实是**probably 都中等**。专家 prompt 不创造能力，能力来自训练。

**反模式 7：Multi-Agent 用同模型同 prompt**。如果 5 个 worker 跑同样模型 + 几乎同样 prompt，结果会高度相关——多样性丧失。要么换不同模型，要么换不同温度，要么用不同 retrieval scope。

**反模式 8：忽视 sub-agent 的 cost**。Anthropic 数据：multi-agent 用 15× token。给 worker 限 budget、限步数；要么用更便宜的模型做 worker（Sonnet/Haiku），Orchestrator 用 Opus。

**反模式 9：让 worker 互相发消息**。Cognition 实证：peer-to-peer 通信导致 deadlock、循环、context drift。改为通过 orchestrator 中转，或共享 memory（如 Anthropic 的 memory.md）。

**反模式 10：Multi-Agent 当首选**。Anthropic、Cognition 都强调：**先把 single agent 做好；只有当单 agent 撞到 context window 或 tool 复杂度上限时，再考虑 multi-agent**。

## §6 S0→S5 演进路线

**S0 - 单 Agent + Persona prompt**：写一个 system prompt 包括角色 + 任务 + 风格。可作为 prototype。

**S1 - 单 Agent + 工具白名单 + 输出 schema**：把 Persona 中"能做什么"剥离到 tool 白名单，"输出什么"剥离到 Pydantic schema。Persona 只剩风格。

**S2 - RoleSpec 化**：把任务、输入、输出、工具、budget、success criterion 全部代码化。Persona 退化为可选 hint。

**S3 - Orchestrator + Worker（按需）**：当任务超过单 agent context 或工具复杂度时，引入 orchestrator-worker。Orchestrator 看 RoleSpec 而非 Persona。

**S4 - Capability Boundary IAM 化**：用 STS、IAM Role、Tool Whitelist、Retrieval Namespace 把"能做什么"从 prompt 层迁到基础设施层。**Even if prompt is jailbroken, capabilities still enforced.**

**S5 - Multi-Modal / Multi-Model Orchestration**：不同模态（视觉、语音、文本）、不同模型（GPT-5、Claude、Gemini）作为 worker；orchestrator 协调跨模态/跨模型协作。当下最前沿。

## §7 一页总结

第一性原理：Agent 的"角色"应拆成三层——**Persona（表达）/ Role（任务）/ Capability Boundary（约束）**。2023-2024 流行的 Persona-driven Multi-Agent 把三层搅在 prompt 里，2025-2026 已被业界判定为反模式。

正确做法：Role 是 **schema + tool + budget + success criterion 的代码化定义**；Capability Boundary 在 IAM/tool whitelist/sandbox 层强制；Persona 只控风格，可有可无。

Multi-Agent 选型：**Single-agent 长跑** 是默认（Cognition Devin），**Orchestrator + Worker** 是有限场景（Anthropic Research），**Multi-Persona peer-to-peer** 是反模式。

工程不变量：Orchestrator 看 RoleSpec 不看 Persona；Sub-agent 主要价值是 context compression；Capability 在代码层强制；Single agent 先做好再考虑 multi。

最危险的反模式：写"You are a senior X with 10 years experience"；让 5 个 Persona 互发消息；Persona 与 Capability 混淆；让模型自己 declare role。

向后衔接：本文与 3.6 任务分解聚合（如何把 task 拆给 worker）和 3.7 Multi-Agent 可靠性（worker 故障与级联）是三位一体；与 6.3 Agent 权限模型互锁（Capability Boundary 的 IAM 实施）；与 5.1 记忆体系联动（shared memory 替代消息总线）。
