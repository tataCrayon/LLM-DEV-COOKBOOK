# 框架深度对比：LangGraph / CrewAI / AutoGen / OpenAI Agents SDK

> 选框架不是选"最好的"，而是选"最匹配你的约束条件的"。约束条件包括：团队技术栈、模型锁定与否、上线时间压力、工作流复杂度、是否需要人类介入、可观测性要求。本文对四个主流框架做架构级对比，帮你在 15 分钟内做出决策。

## 0. 先看一张全景表

| 维度 | LangGraph | CrewAI | AutoGen / AG2 | OpenAI Agents SDK |
|------|-----------|--------|---------------|-------------------|
| **发布方** | LangChain (2024) | CrewAI Inc (2024) | Microsoft Research (2023→AG2 2024) | OpenAI (2025-03) |
| **核心抽象** | 有向图（节点+边+状态） | 角色（Agent+Task+Crew） | 对话（多 Agent 聊天） | 原语（Agent+Handoff+Guardrail） |
| **设计哲学** | 显式 > 隐式，图即架构 | 模拟人类团队协作 | 多轮辩论涌现质量 | 最少抽象，最快上手 |
| **抽象层级** | 低（你定义每个节点和边） | 高（声明角色即用） | 中（定义 Agent 和对话模式） | 低-中（声明式 Agent + 命令式调度） |
| **模型支持** | 全模型（agnostic） | 全模型 | 全模型 | 仅 OpenAI |
| **语言** | Python + JavaScript | Python | Python + .NET | Python |
| **学习曲线** | 中-高 | 最低 | 中 | 低 |
| **生产就绪度** | 最高 | 中 | 中（AG2 仍在成熟） | 高（但需自建周边） |
| **热度（月搜索）** | ~27,100 | ~14,800 | ~8,000 | 快速增长（新发） |

## 1. LangGraph：图即架构

### 1.1 核心设计

LangGraph 把 Agent 工作流建模为**有类型状态的有向图**。节点是计算单元（Agent / 函数 / 子图），边定义控制流（含条件路由），状态是在节点间流动的类型化对象。

```python
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
    next_agent: str

graph = StateGraph(State)
graph.add_node("researcher", researcher_agent)
graph.add_node("writer", writer_agent)
graph.add_node("reviewer", reviewer_agent)

# 条件路由：基于状态决定下一步
graph.add_conditional_edges("researcher", route_fn, {"write": "writer", "more_search": "researcher"})
graph.add_edge("writer", "reviewer")
graph.add_conditional_edges("reviewer", review_fn, {"approve": END, "revise": "writer"})

graph.set_entry_point("researcher")
app = graph.compile(checkpointer=memory_checkpointer)
```

### 1.2 杀手级特性

**Checkpointing + Time-Travel Debugging**：每个节点执行后自动保存状态快照。可以从任意中间状态恢复执行、回退到某一步重跑、或者在某个节点暂停等待人工审核。这在需要审计轨迹的场景（金融、医疗）中是不可替代的。

**Sub-graph 组合**：一个节点本身可以是另一个 LangGraph——支持层级式架构，团队级子图嵌套在系统级图中。

**Human-in-the-Loop**：可以在任意节点插入 `interrupt_before` 或 `interrupt_after`，暂停执行等待人工输入。

**Per-node Streaming**：每个节点的 LLM 输出可以独立流式传输到前端。

### 1.3 Multi-Agent 拓扑

LangGraph 原生支持三种拓扑：

- **Collaboration（完全共享状态）**：所有 Agent 读写同一消息列表
- **Supervisor（中心调度）**：`langgraph-supervisor` 包，Supervisor 节点路由到 Worker 子图
- **Swarm（去中心化）**：`langgraph-swarm` 包，Agent 之间直接 handoff

### 1.4 劣势

- **样板代码多**：即使两个 Agent 的简单流程也需要定义 State schema、节点函数、边、条件路由、编译
- **学习曲线陡峭**：图的思维模型需要适应
- **过度工程风险**：直线流程不需要图

### 1.5 最适合

- 复杂分支工作流（多条件路由、循环、并行分支）
- 需要 checkpoint / 回滚 / 审计轨迹的场景
- 团队已在 LangChain 生态中
- 需要 model-agnostic

## 2. CrewAI：模拟人类团队

### 2.1 核心设计

CrewAI 的抽象层级最高——你定义的是"角色"而不是"节点和边"。每个 Agent 有 role（职位）、goal（目标）、backstory（背景设定）。Task 定义具体工作，Crew 把 Agent 和 Task 组装起来。

```python
from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role="Senior Research Analyst",
    goal="Find cutting-edge AI agent developments",
    backstory="You are a veteran tech researcher with 10 years experience...",
    tools=[web_search, arxiv_search],
    llm="gpt-4o",
)

writer = Agent(
    role="Technical Writer",
    goal="Produce clear, engaging technical content",
    backstory="You are a published author specializing in AI topics...",
    llm="claude-sonnet-4",
)

research_task = Task(
    description="Research the latest multi-agent framework developments in 2026",
    agent=researcher,
    expected_output="Structured research findings with citations",
)

writing_task = Task(
    description="Write a comprehensive comparison article based on the research",
    agent=writer,
    expected_output="A 2000-word article in markdown",
    context=[research_task],  # 依赖 research_task 的输出
)

crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, writing_task],
    process=Process.sequential,  # or hierarchical, consensual
)

result = crew.kickoff()
```

### 2.2 三种执行模式

| 模式 | 机制 | 适用 |
|------|------|------|
| Sequential | Task 按顺序执行，前一个的输出自动成为后一个的上下文 | 管线式流程 |
| Hierarchical | 自动创建 Manager Agent 做任务分派和质量控制 | 需要管理层 |
| Consensual | Agent 投票决策 | 需要多视角共识 |

### 2.3 Flows（编排层）

CrewAI Flows 是更高层的编排抽象——用 Python 装饰器定义步骤间的依赖和条件分支，将多个 Crew 组合为复杂工作流。类似 LangGraph 的图但用声明式代码。

### 2.4 杀手级特性

**最快原型**：20 行代码跑起来一个多 Agent 系统。从想法到可演示的 demo 最快路径。

**角色设定的心智模型**：非技术人员也能理解——"研究员搜资料，写手写文章，编辑审稿"。

**Model-agnostic**：支持 OpenAI、Anthropic、Ollama、任何 OpenAI-compatible API。

### 2.5 劣势

- **缺少 checkpointing**：长任务中断后无法从中间状态恢复
- **Agent 通信粗粒度**：Agent 之间只通过 Task 输出间接传递信息，无法精细控制中间交互
- **调试困难**：Agent 内部决策过程不透明
- **团队往往迁移**：原型用 CrewAI 验证思路，生产时迁移到 LangGraph 获取更多控制力

### 2.6 最适合

- 快速验证多 Agent 想法
- 团队技术能力有限，需要高层抽象
- 简单的顺序/层级任务
- Demo 和 PoC 场景

## 3. AutoGen / AG2：多轮辩论

### 3.1 核心设计

AutoGen 的核心假设是：**多个 Agent 的多轮对话可以涌现出比单次调用更高质量的输出**。像一场辩论赛——写手写初稿、评论员挑毛病、写手改稿、评论员再审……直到满意。

AG2（AutoGen v0.4 重写）引入了事件驱动核心、async-first 执行和可插拔编排策略。

```python
from autogen import AssistantAgent, UserProxyAgent, GroupChat, GroupChatManager

coder = AssistantAgent(
    name="Coder",
    system_message="You write Python code to solve problems.",
    llm_config={"model": "gpt-4o"},
)

reviewer = AssistantAgent(
    name="Reviewer",
    system_message="You review code for bugs, performance, and style issues.",
    llm_config={"model": "gpt-4o"},
)

executor = UserProxyAgent(
    name="Executor",
    human_input_mode="NEVER",
    code_execution_config={"work_dir": "workspace"},
)

# GroupChat：多 Agent 自由对话
group_chat = GroupChat(
    agents=[coder, reviewer, executor],
    messages=[],
    max_round=10,
    speaker_selection_method="auto",  # LLM 决定谁说话
)
manager = GroupChatManager(groupchat=group_chat)

executor.initiate_chat(manager, message="Write a web scraper for HN front page")
```

### 3.2 GroupChat 协调

`speaker_selection_method` 控制谁下一个发言：
- `"auto"` — LLM 根据上下文决定
- `"round_robin"` — 轮流
- `"random"` — 随机
- 自定义函数 — 基于任务状态决定

### 3.3 杀手级特性

**代码执行内置**：`UserProxyAgent` 可以直接执行 Agent 生成的代码，形成"写→跑→看结果→改"的闭环。

**辩论式质量提升**：Writer + Critic 模式天然适合需要高质量输出的场景。研究表明多轮迭代在代码生成、文案写作、事实核查场景中有显著提升。

**AG2 的事件驱动**：新架构支持异步处理、插拔式编排策略、更好的并发。

### 3.4 劣势

- **Token 成本爆炸**：4 Agent × 5 轮 = 最少 20 次 LLM 调用。每轮对话的完整历史都在上下文中。
- **延迟高**：不适合实时场景
- **收敛不确定**：辩论可能陷入无尽循环（"A 说改这里，B 说别改"）
- **AG2 仍在成熟**：社区分裂为 AutoGen 0.2（稳定但旧）和 AG2（新但不稳定）

### 3.5 最适合

- 代码生成 + 自动测试 + 自动修复循环
- 高质量文本创作（文章、报告、翻译）
- 研究型任务（不赶时间，要深度）
- 需要多视角审阅的场景

## 4. OpenAI Agents SDK：最少抽象

### 4.1 核心设计

2025 年 3 月发布，替代实验性的 Swarm 框架。设计哲学是"enough primitives to be expressive, few enough to learn quickly"。三个核心原语：

- **Agent**：LLM + instructions + tools + handoffs
- **Handoff**：显式控制权转移
- **Guardrail**：输入/输出校验（可并行执行）

```python
from agents import Agent, Runner, handoff, InputGuardrail, GuardrailFunctionOutput

# 定义 Guardrail
def check_toxicity(input: str) -> GuardrailFunctionOutput:
    # 实际实现：调用内容审核 API
    return GuardrailFunctionOutput(output_info={"safe": True}, tripwire_triggered=False)

# 定义专业 Agent
billing_agent = Agent(
    name="Billing Specialist",
    instructions="Handle billing inquiries. You can check invoices, process refunds...",
    tools=[check_invoice, process_refund],
)

tech_agent = Agent(
    name="Tech Support",
    instructions="Handle technical issues. You can check system status, create tickets...",
    tools=[check_status, create_ticket],
)

# 定义路由 Agent（通过 handoff 分发）
triage_agent = Agent(
    name="Triage",
    instructions="Route customers to the right specialist based on their query.",
    handoffs=[
        handoff(billing_agent, description="For billing/payment/refund queries"),
        handoff(tech_agent, description="For technical issues and bug reports"),
    ],
    input_guardrails=[InputGuardrail(guardrail_function=check_toxicity)],
)

# 运行
result = await Runner.run(triage_agent, input="I need a refund for my last order")
print(result.final_output)
```

### 4.2 两种 Multi-Agent 模式

**Handoff（控制权转移）**：Triage Agent 判断后把控制权彻底交给专业 Agent。适合"一个问题找一个人解决"的场景。

**Agent-as-Tool（保持控制权）**：Orchestrator 把其他 Agent 包装为工具，调用后结果回到 Orchestrator。适合需要综合多方结果的场景。

### 4.3 杀手级特性

**内置 Tracing**：每次 run 自动生成完整 trace，可发送到 OpenAI Dashboard 或自定义后端。无需额外集成 LangSmith / Langfuse。

**Guardrails 并行执行**：输入校验和 LLM 调用同时进行，不增加延迟。Guardrail 触发时中断执行。

**极简 API**：从 Agent 定义到运行只需 4 行代码。

### 4.4 劣势

- **模型锁定**：只支持 OpenAI 模型（GPT-4o、o3、GPT-5）
- **Python Only**：无 TypeScript/JS SDK
- **缺少 checkpointing**：没有内置的中间状态持久化
- **Handoff 扩展性**：超过 8-10 个 Agent 类型后 handoff 图变得难以管理
- **自建周边**：没有内置 memory、RAG、向量存储——需要自己搭

### 4.5 最适合

- 团队已 all-in OpenAI
- 需要最快上手 + 内置可观测
- 客服路由类场景（清晰的分类→分发→处理）
- 不需要复杂分支或循环

## 5. 决策流程图

```
需要 model-agnostic？
├─ 否 → 只用 OpenAI？
│        ├─ 是 → OpenAI Agents SDK
│        └─ 否 → (评估具体模型厂商 SDK)
└─ 是 → 工作流复杂度？
          ├─ 简单顺序/层级 → CrewAI（最快原型）
          ├─ 复杂分支/循环/人工介入 → LangGraph
          └─ 多轮迭代/辩论/代码生成 → AutoGen

补充条件：
- 需要 checkpoint + 审计？ → LangGraph（唯一内置）
- Token 预算有限？ → 避免 AutoGen（最费 token）
- 团队偏好声明式？ → CrewAI
- 团队偏好命令式控制？ → LangGraph 或 OpenAI Agents SDK
```

## 6. 代码风格对比：同一个任务四种实现

任务：**"搜索最新 AI 新闻 → 分析趋势 → 写摘要"**

### LangGraph（显式图）

```python
# 定义状态、3 个节点函数、条件边、编译、运行
# 约 50-80 行代码，完全控制每一步
graph = StateGraph(State)
graph.add_node("search", search_node)
graph.add_node("analyze", analyze_node)
graph.add_node("write", write_node)
graph.add_edge(START, "search")
graph.add_edge("search", "analyze")
graph.add_edge("analyze", "write")
graph.add_edge("write", END)
app = graph.compile()
```

### CrewAI（声明角色）

```python
# 定义 Agent（角色+目标）、Task（描述+期望输出）、Crew
# 约 20-30 行代码，最少样板
crew = Crew(agents=[researcher, analyst, writer], tasks=[search_task, analyze_task, write_task])
result = crew.kickoff()
```

### AutoGen（对话式）

```python
# 定义 Agent、GroupChat、Manager，发起对话
# 约 30-40 行代码，但运行时 token 消耗最大
group_chat = GroupChat(agents=[searcher, analyst, writer], max_round=6)
manager = GroupChatManager(groupchat=group_chat)
user.initiate_chat(manager, message="Find and summarize latest AI news")
```

### OpenAI Agents SDK（原语式）

```python
# 定义 Agent + tools，Runner.run
# 约 15-25 行代码（最简），但限制 OpenAI
agent = Agent(name="News Analyst", tools=[web_search, summarize])
result = await Runner.run(agent, "Find latest AI news, analyze trends, write summary")
```

## 7. 生产部署考量

| 维度 | LangGraph | CrewAI | AutoGen | OpenAI SDK |
|------|-----------|--------|---------|------------|
| **中间状态恢复** | ✅ Checkpointer | ❌ | ❌ | ❌ |
| **失败重试** | ✅ 从最近 checkpoint | 手动 | 手动 | 手动 |
| **可观测性** | LangSmith | Langfuse | Langfuse | 内置 Tracing |
| **流式输出** | ✅ per-node | ✅ | ✅ | ✅ |
| **并发/横向扩展** | ✅（无状态部署 + 外部 state） | 有限 | ✅（AG2 async） | 有限 |
| **成本控制** | 手动 | 手动 | 高（多轮对话） | 手动 |
| **版本管理** | LangSmith Hub | 代码版本 | 代码版本 | 代码版本 |

## 8. 何时不用框架

一个重要的元问题：你是否真的需要框架？

**不需要框架的场景**：
- 单 Agent + 3-5 个工具 → 直接用模型 API + 简单循环
- 直线型管线（A→B→C）→ 普通 Python async/await
- 已有成熟的工作流引擎（Temporal / Prefect / Airflow）→ 在其中集成 LLM 调用

**需要框架的信号**：
- 需要条件分支/循环/人工介入
- Agent 数量 > 3 且需要协调
- 需要中间状态持久化/恢复
- 需要内置可观测性
- 团队多人协作，需要统一抽象
