# 工业级 Loop 实现对比：Codex `/goal` vs Claude Code `/loop` vs Devin

> 三家工业级产品对"如何让 Agent 持续运行"给出了三种不同的答案：Codex 把 Loop 做成"持久化目标"（state-driven），Claude Code 把 Loop 做成"周期性提示词"（schedule-driven），Devin 把 Loop 做成"端到端任务"（task-driven）。本文从 4.2 的 Workspace 四原语出发，系统对比三家的实现差异、设计哲学与适用场景。

## 0. 开场问题

你的团队用三种不同的 Loop 工具试图解决同一个问题——"每天凌晨自动检查依赖更新并提交 MR"。三组工程师的实现差异巨大：

- 工程师 A 用 **Codex `/goal`**：写了一句"goal: 每天检查所有依赖更新，提交 MR 并保持测试通过"，启动后让 Codex 一直跑直到 budget 耗尽
- 工程师 B 用 **Claude Code `/loop`**：写了 `/loop 1d 检查依赖更新并提交 MR`，让它每 24 小时跑一次同样的提示词
- 工程师 C 用 **Devin**：通过 Slack `@devin 帮我每天处理依赖更新` 创建任务

一周后回看：

- A 的 Codex 在第三天就自报 `goal_status: achieved` 并停止了——但实际只升级了 3/47 个依赖
- B 的 Claude Code 每天忠实地跑，但每次都是"全新视角"——上次跑过 react 升级，这次又重新评估 react，重复劳动
- C 的 Devin 每天在 Slack 里贴一份漂亮的报告，但在 Cognition 云端跑，本地仓库根本没改

三个人都没"用错"，但都没"用对"——因为他们没意识到这三个 `/goal` `/loop` `Devin` **不是同一类东西**。它们是三种根本不同的 Loop 抽象。

理解这个差异，必须把它们放到 4.2 提出的 Workspace 四原语框架里逐项对比：

## 1. 三家的设计哲学

| 维度 | Codex `/goal` | Claude Code `/loop` | Devin |
|---|---|---|---|
| Loop 抽象 | Persistent State Machine（持久状态机） | Recurring Prompt（周期提示词） | End-to-End Task（端到端任务） |
| 主导者 | Runtime（运行时管理状态） | Scheduler（调度器管理时间） | Agent（代理自主管理任务） |
| 终止条件 | 显式 `update_goal(achieved)` 或 budget 耗尽 | 7 天自动过期 / 终端关闭 / 手动停 | Agent 自报 done / ACU 耗尽 |
| 目标对象 | 一个长期目标 | 一个周期性检查 | 一个具体工单 |
| 适合场景 | Long-horizon 单一任务 | 持续监控 + 实时响应 | 隔离的中等复杂度需求 |

**核心差异的一句话总结**：

- Codex `/goal` = "盯住一件事，直到完成"
- Claude Code `/loop` = "每隔一段时间问同一个问题"
- Devin = "丢给一个独立工程师做完"

## 2. 沿 Workspace 四原语逐项对比

### 2.1 Identity & Scope

| | Codex `/goal` | Claude Code `/loop` | Devin |
|---|---|---|---|
| Agent 身份 | 用户本地身份（CLI） | 用户本地身份（CLI） | 独立 service account（Cognition 云） |
| 凭证 | 用户 OpenAI key + 本地凭证 | 用户 Anthropic key + 本地凭证 | 项目级集成凭证（GitHub App / GitLab Bot） |
| 边界 | 当前仓库 + 用户授权 | 当前仓库 + 用户授权 + 单 session | Cognition 云沙箱（隔离） |
| Commit 归因 | 用户身份（默认） | 用户身份（默认） | `devin-ai-integration[bot]` |

**关键差异**：Devin 是**唯一**真正实现"独立身份"的。Codex 和 Claude Code 都依附于用户身份——这意味着如果开发者离职，所有 Agent 历史 commit 失去归属。Devin 的 bot identity 是其差异化定位的核心。

### 2.2 State & Persistence

#### Codex `/goal`：State-First 设计

Codex 把"目标"做成一个**首类持久化对象**。这是其最显著的架构特征——OpenAI（2025）将 `/goal` 拆成 5 个 PR（#18073–#18077）覆盖：persistence、app-server API、model tools、runtime continuation、TUI controls，体现的是"separation of powers"理念。

```
thread_goals 表 = {
    goal_id, objective_text, status, 
    cumulative_tokens, elapsed_time, 
    snapshot_data
}
```

**5 个 lifecycle states**：

- `pursuing`（默认）— runtime 每轮注入 `continuation.md` 模板
- `paused` — 手动挂起，状态全保留
- `achieved` — 成功终态（必须经 audit 确认）
- `unmet` — Agent 判定无法达成（缺依赖、冲突等）
- `budget-limited` — token 预算耗尽，触发 graceful wrap-up

`update_goal` 是**唯一的 model-facing 状态转换工具**。Pause/resume/clear 仅用户和 runtime 可控——模型无法自行 seize 这些状态。这是一个深思熟虑的权限分离：让模型只能宣告"我做完了"，不能自行决定"我们换个目标"。

#### Claude Code `/loop`：Stateless Iteration

Claude Code 的 `/loop` 几乎反向——**没有持久化目标**，只是一个周期性 prompt 触发器：

```
/loop 5m check if the staging deploy at localhost:3000 is responding
```

每次迭代继承当前 session 的 conversational context（这是它和 cron 的核心差异），但跨 session 不持久化。终端关闭即结束，最长 7 天自动过期，单 session 最多 50 个并发 loop。

**这意味着 `/loop` 不是"长跑 Agent"——它是"短跑 Agent 的循环"**。它假设你打开终端，让 Claude Code 在后台周期性醒来，做一次检查就睡。

#### Devin：Task-Bound State

Devin 的状态绑定在"任务"对象上：每个任务有独立 sandbox、独立 session、独立 ACU 计费。任务内状态丰富（包含 plan、artifacts、conversation），但任务结束即终结——没有"持续目标"概念。

Devin 2.0 引入的 **Interactive Planning**（Augment Code, 2026）让用户在执行前修改计划，但计划批准后 Devin 进入"自主执行 + 完成时审核"模式，期间不可见中间状态。

#### 对比小结

| 状态特性 | Codex `/goal` | Claude Code `/loop` | Devin |
|---|---|---|---|
| 跨 turn 持久 | ✅ | ✅（session 内） | ✅（任务内） |
| 跨 session 持久 | ✅ | ❌ | ✅（任务跨天） |
| 跨任务持久 | ❌（一次一 goal） | N/A | ❌（一次一 task） |
| 状态可外部查询 | ✅（app-server API） | ❌ | ✅（dashboard + Slack） |
| 状态可被人编辑 | 部分（pause/clear） | ❌ | 部分（Interactive Planning） |

### 2.3 Write Surface

#### Codex `/goal`

- 工作面：用户本地仓库直接读写
- 默认无 PR-only 约束（用户可配置）
- 通过 `--ask-for-approval` 设置交互门禁
- Audit 通过 `update_goal` 调用历史
- 回滚靠 git

**风险点**：默认本地直写，是"demo 友好但生产不安全"的设计。生产使用需用户自配 worktree + PR-only。

#### Claude Code `/loop`

- 工作面：用户本地仓库
- 默认带 PostToolUse Hook（lint/test 自动触发）
- CLAUDE.md 作为 Guide 注入
- 写操作受 Hook 反馈控制
- 回滚靠 git

**特点**：Claude Code 的 Hook 机制是其差异化优势——每次工具调用后自动触发外部检查，等价于内置的 Sensor。但 `/loop` 只是周期性触发器，写入治理仍依赖底层 Claude Code 的能力。

#### Devin

- 工作面：Cognition 云沙箱（与本地完全隔离）
- 默认 PR-only：Devin 通过 GitHub PR 提交，**永不直接 push**
- 内置 Devin Search / Wiki / DeepWiki 用于代码理解
- Audit 通过任务 dashboard
- 回滚靠 PR revert

**优势**：唯一一个**默认就是 PR-only** 的产品。Devin 的写入治理是内置的，不需要用户额外配置——这是其面向企业市场定位的体现。

### 2.4 Budget Envelope

| 预算维度 | Codex `/goal` | Claude Code `/loop` | Devin |
|---|---|---|---|
| 主预算单位 | Token Budget（100k - 10M） | 时间间隔 + 7 天过期 | ACU（Agent Compute Units） |
| ACU 含义 | N/A | N/A | ~15 分钟 active work |
| 多维预算 | Token + 时间（statusline 显示） | 间隔 + 过期 + 50 并发 | ACU + Cognition 服务限额 |
| 软终止机制 | budget_limited → 注入 wrap-up 模板 | 无（直接停） | 无（任务超 ACU 即停） |
| 用户可见 | TUI footer 实时显示 | 终端日志 | Dashboard + Slack 通知 |

**Codex 的 budget_limited 是最优雅的设计**：当 token 预算即将耗尽时，runtime **不是简单 abort**，而是注入一个 `budget_limit.md` 模板，引导 Agent 优雅收尾——例如总结当前进度、写下"剩余事项"、提交 WIP commit。这正是 4.2 中讨论的"软终止"模式的工业实现。

Claude Code `/loop` 的预算几乎是"时间维度的一刀切"——7 天过期或终端关闭。它不适合长任务，本质上是"轻量周期任务"。

Devin 的 ACU 是**最贴近商业现实的预算单位**——每个 ACU 约 15 分钟 active work，对应一个相对稳定的"算力 + 时间"组合，方便企业做 capacity planning。

### 2.5 完整四原语对比

| 原语 | Codex `/goal` | Claude Code `/loop` | Devin |
|---|---|---|---|
| Identity | 用户身份 | 用户身份 | 独立 bot 身份 ✅ |
| Scope | 本地仓库（用户配置） | 本地仓库 + session 内 | Cognition 云沙箱（强隔离）✅ |
| State | 持久化 goal 对象 ✅ | session 内 context | 任务内状态 |
| Write Surface | 本地直写（默认） | 本地直写 + Hook | PR-only ✅ |
| Budget | Token + 时间 + 软终止 ✅ | 时间间隔 + 过期 | ACU |

**关键洞察**：没有任何一家在所有四个原语上都"最强"——它们各有权衡：

- **Codex** 在 State 和 Budget 上最先进（持久化 goal 对象 + 软终止），但 Identity 和 Write Surface 最弱（依赖用户身份和配置）
- **Claude Code** 在 Hook 触发的 Verification 上最强，但 State 持久化最弱
- **Devin** 在 Identity 和 Write Surface 上最强（bot 身份 + PR-only），但 State 不可跨任务持久

## 3. 三种 Loop 抽象的工程含义

### 3.1 Codex `/goal`：Verifiable End State

Codex 设计中最深刻的部分是 **"audit-first completion"**。`continuation.md` 模板强制要求模型：

> 不接受代理信号（proxy signals）作为完成证据。测试通过、TypeScript 全绿、文件存在——都不是证据。模型必须从原始 objective 构建 checklist，逐项直接验证，然后才能调 `update_goal(achieved)`。

文章里有句话总结得好：**"OpenAI 把每个 Ralph loop 用户血的教训硬编码进去了"**——Agent 倾向于在表面信号达标时宣布胜利，而真实需求并未达成。

这个设计的代价是：**Codex `/goal` 不适合开放性任务**。如果你的目标是"提升代码质量"，Agent 永远找不到 audit-first 的证据，会一直跑直到 budget 耗尽。它适合的是"有明确 deliverable 的目标"——如"迁移 src/legacy/auth.ts 到 async/await"。

### 3.2 Claude Code `/loop`：Watchdog with a Brain

Claude Code `/loop` 的核心定位是 **"watchdog with a brain"**——会思考的看门狗。它的优势不在跑得久，而在跑得"聪明"：

- 继承 session context（不是冷启动）
- 可以对比"上次和这次"
- 可以触发完整工具链（lint/test/git）

适合的场景：**实时监控 + 增量响应**。例如：

```
/loop 5m 监控 staging deploy 状态，如果有新错误就分析根因
/loop 30m 检查 main 分支是否有新合并，如果有就 rebase 当前分支
/loop 2h 扫描日志，如果出现 FATAL 就自动创建 issue
```

这些场景的共性：**间隔周期 + 增量信息 + 即时响应**。不适合的场景是"完成一个长期目标"——那是 `/goal` 的领域。

### 3.3 Devin：The Async Engineer

Devin 把自己定位为"异步工程师"——你不需要管它在做什么，看结果就行。这个定位的工程实现：

- 强隔离（云沙箱，不污染本地）
- 强归因（独立 bot identity）
- 强治理（PR-only，无直接 push）
- 强不透明（用户不可见中间步骤）

适合的场景：**有明确 spec 的隔离任务**。Augment Code（2026）的对比文章总结得很到位：

> Devin 的甜区是"isolated tasks with clear requirements (multi-hour)"。需要跨服务一致性的复杂重构不适合它，因为它没有 shared living spec。

代价是：**调试困难**。中间步骤被刻意隐藏，出问题时只能看最终 PR。如果 PR 错了，你需要在 Slack 里再开一轮对话——而那已经是新任务了。

## 4. 决策矩阵：什么场景选什么

```
┌─────────────────────────────────────────────────────────────────┐
│  你需要 Loop 做什么？                                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [A] 单一长期目标，有明确 deliverable                           │
│      → Codex /goal                                              │
│      例：迁移 N 个文件、修复 specific bug、达成 specific 重构   │
│                                                                 │
│  [B] 持续周期性检查，触发增量响应                               │
│      → Claude Code /loop                                        │
│      例：监控部署、扫描日志、追踪 PR 状态                       │
│                                                                 │
│  [C] 隔离任务，需要异步交付，强治理                             │
│      → Devin                                                    │
│      例：依赖升级、CVE 修复、按 spec 实现新模块                 │
│                                                                 │
│  [D] 复杂 multi-step workflow，需要自定义编排                   │
│      → 自建 Loop（基于 LangGraph / 自研 harness）              │
│      例：Anthropic Multi-Agent Research、复杂业务流程           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 决策检查清单

逐项问自己：

1. **目标是否可在 audit 中验证为"完成"？**
   - 是 → 倾向 `/goal` 或 Devin
   - 否（开放性、持续性）→ 倾向 `/loop` 或自建

2. **是否需要在执行间保持累积上下文？**
   - 是 → 倾向 `/goal`（持久化目标）或 `/loop`（session 内）
   - 否（每次独立） → 倾向 cron + headless agent

3. **是否需要独立身份和强归因？**
   - 是（多人协作、合规） → 倾向 Devin
   - 否（个人开发） → `/goal` 或 `/loop` 都可

4. **是否需要 PR-only 治理？**
   - 默认必须 → 倾向 Devin
   - 用户可自配 → `/goal` 或 `/loop` 都可

5. **预算单位偏好？**
   - Token 维度精细控制 → `/goal`
   - 时间维度简单管理 → `/loop`
   - 算力 + 时间打包 → Devin

## 5. 混合使用模式

实际生产中，三家工具不是互斥的——可以分层使用：

### 模式 A：`/loop` 触发 `/goal`

```
/loop 1d   ← 每天检查"是否有新 CVE 影响项目"
  ↓ 触发条件满足
/goal 修复 CVE-2025-XXXX，确保所有 affected 模块通过测试
  ↓ Codex 持续跑直到 audit 通过
git PR
```

`/loop` 做"调度感知"，`/goal` 做"执行收敛"。

### 模式 B：`/goal` 编排，Devin 执行

```
本地 /goal: 完成 Q3 技术债务清理（meta goal）
  ↓ 本地 Agent 拆解为子任务
对每个子任务 → 通过 API 派发给 Devin（独立沙箱）
  ↓ Devin 执行 + PR
本地 /goal 跟踪所有 PR 状态，全部 merged 才标 achieved
```

本地 Agent 做"导演"，Devin 做"演员"。

### 模式 C：`/loop` 监控 Devin 任务

```
/loop 30m 检查 Devin dashboard，对每个失败任务发送总结到 Slack
```

Claude Code `/loop` 在这里相当于"轻量级运维助手"。

## 6. 实战代码：构建你自己的 Loop Engine

如果三家工具都不满足需求，可以基于 4.2 的四原语自建 Loop Engine。下面是一个最小可用实现，融合了三家的精华：

```python
"""
最小 Loop Engine：Codex 的持久化 goal + /loop 的调度 + Devin 的隔离
"""
from dataclasses import dataclass, field, asdict
from datetime import datetime, timedelta
from enum import Enum
from typing import Optional, Callable, List
import json
import uuid

class GoalStatus(Enum):
    PURSUING = "pursuing"
    PAUSED = "paused"
    ACHIEVED = "achieved"
    UNMET = "unmet"
    BUDGET_LIMITED = "budget_limited"


@dataclass
class Goal:
    """借鉴 Codex /goal 的持久化目标对象"""
    id: str
    objective: str
    status: GoalStatus
    created_at: datetime
    cumulative_tokens: int = 0
    elapsed_seconds: float = 0.0
    audit_log: List[dict] = field(default_factory=list)
    
    def to_dict(self):
        d = asdict(self)
        d["status"] = self.status.value
        d["created_at"] = self.created_at.isoformat()
        return d


@dataclass
class LoopSchedule:
    """借鉴 Claude Code /loop 的调度配置"""
    interval_seconds: int  # 0 = 连续跑（Ralph loop 风格）
    max_iterations: int = 1000
    auto_expire_after_days: int = 7


class LoopEngine:
    """融合三家精华的最小 Loop Engine"""
    
    def __init__(self, goal: Goal, schedule: LoopSchedule, 
                 sandbox_path: str, 
                 max_tokens: int, max_cost: float):
        self.goal = goal
        self.schedule = schedule
        self.sandbox_path = sandbox_path  # Devin 风格隔离
        self.max_tokens = max_tokens
        self.max_cost = max_cost
        self.iteration = 0
        self.started_at = datetime.utcnow()
    
    def run(self, llm_call: Callable, audit_fn: Callable[[Goal], bool]):
        """主循环"""
        while self.iteration < self.schedule.max_iterations:
            # 0. 检查终止条件
            if self._is_expired():
                self._transition(GoalStatus.UNMET, "auto-expired")
                break
            if self._is_budget_exhausted():
                # Codex 风格：软终止，不直接抛
                self._inject_wrapup_prompt(llm_call)
                self._transition(GoalStatus.BUDGET_LIMITED, "budget exhausted")
                break
            if self.goal.status == GoalStatus.PAUSED:
                break
            
            # 1. 注入 continuation prompt（Codex 风格）
            response = self._run_iteration(llm_call)
            self.iteration += 1
            
            # 2. 审计驱动的完成判断（Codex 的 audit-first）
            if response.get("claims_complete"):
                if audit_fn(self.goal):
                    self._transition(GoalStatus.ACHIEVED, "audit passed")
                    break
                else:
                    self.goal.audit_log.append({
                        "iteration": self.iteration,
                        "reason": "claimed complete but audit failed",
                        "timestamp": datetime.utcnow().isoformat(),
                    })
                    # 强制 Agent 继续——不接受未审计的"完成"声明
                    continue
            
            # 3. 周期等待（如果不是连续跑）
            if self.schedule.interval_seconds > 0:
                self._sleep(self.schedule.interval_seconds)
    
    def _run_iteration(self, llm_call: Callable) -> dict:
        """单次迭代"""
        prompt = self._build_continuation_prompt()
        result = llm_call(prompt, sandbox=self.sandbox_path)
        self.goal.cumulative_tokens += result.get("tokens_used", 0)
        return result
    
    def _build_continuation_prompt(self) -> str:
        """Codex 的 continuation.md 思路"""
        return f"""
=== Goal ===
{self.goal.objective}

=== Progress ===
Iteration: {self.iteration}
Tokens used: {self.goal.cumulative_tokens:,} / {self.max_tokens:,}
Status: {self.goal.status.value}

=== Instructions ===
1. Restate the objective as concrete deliverables
2. Build a checklist from those deliverables
3. Continue work from current state
4. Before claiming complete, VERIFY each checklist item directly
5. If genuinely complete, output {{"claims_complete": true, "evidence": [...]}}
"""
    
    def _inject_wrapup_prompt(self, llm_call):
        """Codex budget_limit.md 思路：优雅收尾"""
        prompt = f"""
=== Budget Exhausted ===
Tokens used: {self.goal.cumulative_tokens:,}
Tokens budget: {self.max_tokens:,}

Wrap up gracefully:
1. Summarize current progress
2. List remaining work
3. Commit any WIP changes with a clear "WIP:" prefix
4. Output a handoff note for the next session
"""
        llm_call(prompt, sandbox=self.sandbox_path)
    
    def _is_expired(self) -> bool:
        elapsed = datetime.utcnow() - self.started_at
        return elapsed > timedelta(days=self.schedule.auto_expire_after_days)
    
    def _is_budget_exhausted(self) -> bool:
        return self.goal.cumulative_tokens >= self.max_tokens
    
    def _transition(self, new_status: GoalStatus, reason: str):
        self.goal.audit_log.append({
            "from": self.goal.status.value,
            "to": new_status.value,
            "reason": reason,
            "timestamp": datetime.utcnow().isoformat(),
        })
        self.goal.status = new_status
    
    def _sleep(self, seconds: int):
        # 实际生产用 asyncio.sleep
        import time
        time.sleep(seconds)
    
    def pause(self):
        self._transition(GoalStatus.PAUSED, "user pause")
    
    def resume(self):
        if self.goal.status == GoalStatus.PAUSED:
            self.goal.status = GoalStatus.PURSUING
    
    def persist(self, path: str):
        """跨会话持久化"""
        with open(path, "w") as f:
            json.dump(self.goal.to_dict(), f, indent=2)
```

## 7. 三家的演进方向

到 2026 年初，三家产品都在向"对方的优势区"演进：

- **Codex** 在补 identity 和 write surface（增加 GitHub App 集成、PR-only 模式选项）
- **Claude Code** 在补 state persistence（headless 模式 + cron + 可挂载 .claude/ 目录）
- **Devin** 在补 audit transparency（任务执行 trace 暴露、Interactive Planning 在前期介入）

未来一两年内可能出现 **convergence on a common abstraction**——目标 + 调度 + 沙箱 + 治理 + 预算 + 审计 都成为基础设施，框架差异变成 API 风格差异。但目前阶段，三家的设计哲学差异巨大，选错工具的代价仍然可观。

## 8. 总结：选 Loop 工具是选哲学

回到开头的"依赖更新"场景，正确的工具选择应该是：

- **Codex `/goal`**：不合适——"每天检查"不是单一目标，audit 永远不通过
- **Claude Code `/loop`**：部分合适——能调度，但每次冷启动评估浪费
- **Devin**：最合适——独立任务、强治理、bot 归因，每次依赖升级是一个明确 PR

但注意：如果场景换成"迁移整个 Auth 模块到新版本"，答案就反了——这是 Codex `/goal` 的甜区。

这是本文最重要的一个判断：**Loop 工具不是按"功能多"选，而是按"哲学契合度"选**。

| 你的本质需求 | 对应哲学 | 选谁 |
|---|---|---|
| 跑出一个 deliverable | State Machine | Codex |
| 持续观察 + 增量响应 | Recurring Probe | Claude Code |
| 把工单交出去 | Async Engineer | Devin |
| 都不是 | 自建 | LangGraph + 4.2 四原语 |

下一篇 4.4 将聚焦 Loop 治理三大挑战——Verification Burden、Comprehension Debt、Token Economics——这是 Loop 工具用得多了之后才会浮现的隐性成本。
