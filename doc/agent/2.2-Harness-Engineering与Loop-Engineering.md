# Harness Engineering 完整体系：让 Agent 从"能跑"到"可靠"

> 模型决定了 Agent 的上限，Harness 决定了 Agent 的下限。一个用了 GPT-5 但没有 Harness 的 Agent，不如一个用 Sonnet 但配了完善 Harness 的 Agent 稳定。本文从第一性原理拆解 Harness 的全部组成——为什么需要它、它包含什么、怎么分层建设。

## 0. 开场问题

你部署了一个 coding agent，让它修一个 bug。它做了以下事情：

1. 读了代码，分析正确
2. 生成了一个 patch，逻辑也对
3. 但改的时候顺手删了一个看起来"没用"的 import——实际上那个 import 在另一个文件里被 re-export 使用
4. 提交了。CI 挂了。Agent 看到 CI 挂了，又改了一版——这次把 CI 错误信息当作要 fix 的 bug，越改越离谱
5. 最终提交了一个让代码库状态更差的 PR

问题出在哪？**不是模型不够聪明**——它确实理解了 bug 的根因。问题是：

- 没人告诉它"这个项目中 import 不能随便删"（缺 Guide）
- 它改完没跑类型检查就提交了（缺 Sensor）
- CI 失败后没有"回滚到改之前"的能力（缺 Rollback）
- 没有权限门禁阻止它连续提交有问题的代码（缺 Permission Gate）

这四个缺失，每一个都属于 **Harness**——模型之外的一切。

## 1. 从等式出发：Agent = Model + Harness

Birgitta Böckeler 在 Martin Fowler 站点上给出的这个等式（2025）是整个领域最好的起点：

```
Agent = Model + Harness

其中 Harness = 模型之外的一切工程
         = Guides + Sensors + Tools + Sandbox + Hooks + Observability + ...
```

模型是"大脑"，Harness 是"身体 + 环境 + 制度"。大脑再聪明，如果没有眼睛（Sensor）看路、没有规则（Guide）约束行为、没有安全带（Sandbox）防撞、没有体检（Observability）发现隐患——出事只是时间问题。

## 2. 横向结构：Guides 与 Sensors

### 2.1 Guides（前馈控制）

在 Agent 行动**之前**塑造它的判断。目标是减少犯错概率。

| Guide 类型 | 形态 | 作用时机 |
|---|---|---|
| AGENTS.md / 项目宪法 | Markdown 文件注入 system prompt | 每次调用前 |
| Skills / Playbook | 结构化知识包 | 特定任务触发时 |
| LSP / 类型系统提示 | IDE 协议数据 | 编辑代码时 |
| 架构决策记录（ADR） | 历史决策文档 | Agent 做架构选择时 |
| Tool description | JSON Schema + 使用说明 | 模型选择工具时 |

**关键原则**：Guide 不是越多越好。每条 Guide 都消耗 token、分散注意力。好的 Guide 满足一个条件——你能指出"不加这条 Guide 会出什么具体问题"。

### 2.2 Sensors（反馈控制）

在 Agent 行动**之后**检测是否出错并触发修正。目标是及时发现问题。

| Sensor 类型 | 速度 | 成本 | 可靠性 | 典型用途 |
|---|---|---|---|---|
| Linter / Formatter | 毫秒 | 几乎为零 | 100% 确定性 | 代码风格 |
| 类型检查 | 秒级 | 低 | 100% 确定性 | 逻辑类型错误 |
| 单元测试 | 秒~分钟 | 低 | 高（但覆盖率依赖） | 行为正确性 |
| 集成测试 / E2E | 分钟级 | 中 | 中 | 系统级行为 |
| AI Review | 秒~分钟 | 中（token 费用） | 非确定性 | 语义层面审查 |
| Mutation testing | 分钟~小时 | 高 | 高 | 测试质量验证 |

**Böckeler 的核心警告**：

> 只有 Sensors 没有 Guides，Agent 会重复犯同样的错误；只有 Guides 没有 Sensors，Agent 会执行规则却不知道是否真的有效。

两者必须配对。AGENTS.md 说"不要删 import"（Guide），类型检查/测试会验证"删了之后是否出错"（Sensor）。

### 2.3 Computational vs Inferential

每种 Harness 组件都有两种实现手段：

- **Computational（确定性计算）**：测试、类型检查、静态分析、linter。快、便宜、可靠、应该优先用
- **Inferential（LLM 推理）**：AI code review、LLM-as-judge、语义一致性检查。慢、贵、非确定性，但能覆盖计算手段覆盖不到的语义层

**工程决策规则**：能用 computational 解决的事**绝不**用 inferential。LLM 判断只用在确实需要语义理解的地方——"这段代码是否符合业务意图"比"这段代码是否有语法错误"更适合 inferential。

## 3. 纵向结构：三层 Harness

按"约束什么"来分，Harness 有三个不同深度：

### 3.1 Maintainability Harness（可维护性）

最成熟的一层。工具链齐全：

- Linter（ESLint、Ruff、golangci-lint）
- Formatter（Prettier、Black）
- 依赖审计（Dependabot、Renovate）
- 复杂度检查（SonarQube）
- 代码覆盖率门禁

Agent 场景中的特殊考虑：把这些配成 **git pre-commit hook**，Agent 提交时自动拦截。Claude Code 的 Hooks 机制（PostToolUse 事件绑定）就是这个思路——每次编辑后自动跑 lint/test，Agent 无法跳过。

### 3.2 Architecture Fitness Harness（架构合规）

中等成熟度。工具存在但不普及：

- 模块边界检查（ArchUnit、Nx boundary rules）
- 依赖方向校验（"feature 不允许依赖 core 以外的模块"）
- API 兼容性检查（protobuf breaking change 检测）
- 性能 regression 门禁（benchmark 对比）

这一层的核心概念是 **Fitness Function**——Neal Ford 在《Building Evolutionary Architectures》中提出的：用可执行的检查来保证架构特征不退化。

### 3.3 Behaviour Harness（行为正确性）

最难做的一层。Böckeler 称之为"the elephant in the room"。

问题场景：Agent 提交的代码能编译、能通过 lint、能过类型检查——但**功能不对**。比如把"用户删除"实现成了"软删除"，但业务要求的是"物理删除 + 数据归档"。

这一层没有银弹，但有分层策略：

```
行为正确性验证分层:
├── 单元测试（最基础——但需要已有好的测试）
├── Property-based testing（自动生成边界输入）
├── Contract testing（API 契约验证）
├── Specification-as-test（把需求文档变成可执行验证）
└── LLM-as-judge（语义层面"这符合需求吗"）
```

## 4. Harness 的五层架构：Mechanism vs Policy

boydfd 的分层拆解提供了最实用的架构视角：

```
┌─────────────────────────────────────────────┐
│  Layer 5: Design Instance                    │  Policy
│  (Claude Code, Codex, Cursor, ...)           │  "谁用这些东西"
├─────────────────────────────────────────────┤
│  Layer 4: Pattern                            │  Policy
│  (Plan-Act-Verify, ReAct, Ralph Loop, ...)   │  "怎么组合"
├─────────────────────────────────────────────┤
│  Layer 3: Design Axes (10 个维度)            │  Mechanism
│  (topology, control loop, memory, ...)       │  "有哪些旋钮"
├─────────────────────────────────────────────┤
│  Layer 2: Framework / Runtime                │  Mechanism
│  (tool dispatch, sandbox, MCP, hooks, ...)   │  "提供什么能力"
├─────────────────────────────────────────────┤
│  Layer 1: Cross-cutting                      │
│  (Prompt Eng + Context Eng)                  │  贯穿所有层
└─────────────────────────────────────────────┘
```

下三层是 **Mechanism**（"能做什么"），上两层是 **Policy**（"怎么用"）。

第三层的 **10 个 Design Axes** 是设计时最重要的检查清单：

| 组 | Axis | 问题 |
|---|---|---|
| 结构 | Topology | 单 Agent 还是多 Agent？什么拓扑？ |
| 结构 | Coordination | Agent 间怎么协作？共享状态？消息传递？ |
| 结构 | Session Span | 单轮？多轮？跨天持续？ |
| 行为 | Control Loop | 内部用 ReAct？Plan-Execute？Tree Search？ |
| 行为 | Autonomy | 全自主？HITL？审批门禁？ |
| 行为 | Execution Isolation | 共享环境？容器？microVM？ |
| 资源 | Memory | 无记忆？会话内？跨会话持久化？ |
| 资源 | Tool Type | 只读？读写？不可逆？ |
| 资源 | Verification | 自验证？外部验证？人工验证？ |
| 资源 | Compaction | 无压缩？Masking？LLM 摘要？ |

描述一个 Agent 系统时，沿这 10 个轴具体标注——比贴"multi-agent"这种粗标签有用一百倍。

## 5. Ashby 定律与 Harness 的可行性边界

控制论中 Ashby's Law of Requisite Variety 说：

> 一个调节器要稳定一个系统，其变化度必须至少等于被调节系统的变化度。

翻译成 Harness 语言：**你的 Harness 能覆盖的"情况数"必须至少等于 Agent 可能产生的"异常数"。**

实操含义：

- 如果代码库本身没有规范（什么命名方式都有、什么框架都用），Harness 几乎不可能有效——变化度太高，任何规则都覆盖不完
- **选定技术栈和架构模式本身就是 Harness 的前提条件**——这是一次 "variety-reduction move"，把系统的可能状态空间缩小到 Harness 能管理的范围

反直觉的推论：**Harness 最需要的地方（遗留系统、混乱代码库）恰恰是最难建立 Harness 的地方**。解法是分阶段：先选定边界和规范（reduce variety），再在此基础上加 Sensor 和 Guide。

## 6. Loop Engineering：从被动响应到持续运行

### 6.1 Loop 解决什么问题

当 Harness 让单次 Agent 调用足够可靠后，下一组问题是：

- 我希望它每天凌晨自动跑仓库扫描
- 某个 Issue 被打标签时自动接手
- 一个长任务可以中断、暂停、恢复
- 多个 Agent 并行干活互不干扰
- 跑飞了能快速回滚

这些关心的都是**调度、跨次、并行、预算、回滚**——Loop Engineering 的领地。

### 6.2 Loop 的 5+1 个 Building Blocks

| Block | 作用 | 典型形态 |
|---|---|---|
| **Automations** | 触发 Agent | cron、webhook、event |
| **Worktrees** | 并行隔离 | git worktrees、独立 branch |
| **Skills** | 编码项目知识 | SKILL.md、Playbook |
| **Connectors** | 接触外部系统 | GitHub、Jira、Slack |
| **Sub-agents** | 角色分离 | writer vs verifier |
| **Workspace** | 治理边界 | Identity + Scope + Audit + Rollback |

第六块 Workspace 是 puppyone 的关键贡献——只有 Memory 回答了"能继续"，Workspace 回答了"不出大事"。它包含四个原语：身份（每个 job 有独立 identity）、范围（路径/工具/凭证边界）、审计（不可篡改的操作记录）、回滚（预定义的回滚窗口）。

### 6.3 Loop 的三大风险

1. **Verification Burden**："done is a claim, not a proof"——Agent 说"完成了"不等于真完成了
2. **Comprehension Debt**：Loop 提交代码的速度可能超过你理解它的速度
3. **Token Economics**：

> "If your loop doesn't have a budget policy, it doesn't have a stopping condition—just a failure mode."  — Addy Osmani

### 6.4 Loop 与 Harness 的关系

容易犯的错是把 Loop 画在 Harness "之上"。实际上：

> **Loop 是 Harness 五层结构中 Pattern 层的一员，而非独立层。**

Loop 唯一真正新增的能力是 **self-triggering**。它依赖的其他一切——self-prompting、state persistence——都来自 Harness 已有的 axes。**没有扎实的 Harness，谈 Loop 是空中楼阁。**

## 7. 完整代码示例：一个 Harness 配置的生命周期

```python
"""
演示：一个 Coding Agent 的 Harness 配置
包含：Guide 注入、Sensor 链、Permission Gate、Budget Control
"""
import subprocess
import json
from dataclasses import dataclass
from pathlib import Path
from anthropic import Anthropic

client = Anthropic()

# ===== Guides: 静态知识注入 =====
AGENTS_MD = """
# 项目规约

## 不变量
- 不删除任何现有的 import 语句，除非确认无任何文件 re-export 它
- 所有 API 返回值必须用 TypedDict 类型标注
- 测试文件命名: test_{module}.py
- commit message 格式: type(scope): description

## 架构边界
- services/ 不允许 import controllers/
- models/ 不允许 import services/
- utils/ 是纯函数，不允许有副作用
"""

SKILL_REFACTOR = """
## Refactoring Skill
1. 先跑现有测试确认基线
2. 做修改
3. 跑全量测试验证无回归
4. 如果测试新增或修改，说明原因
"""

# ===== Sensors: 自动化验证链 =====
@dataclass
class SensorResult:
    name: str
    passed: bool
    output: str
    cost_ms: int

def run_sensor_chain(changed_files: list[str]) -> list[SensorResult]:
    """按速度/成本排列的 Sensor 链——快的先跑，慢的后跑"""
    results = []
    
    # Sensor 1: Linter（最快，毫秒级）
    lint_result = subprocess.run(
        ["ruff", "check"] + changed_files,
        capture_output=True, text=True, timeout=10
    )
    results.append(SensorResult(
        name="ruff_lint", 
        passed=lint_result.returncode == 0,
        output=lint_result.stdout + lint_result.stderr,
        cost_ms=50
    ))
    if not results[-1].passed:
        return results  # 快速失败——lint 不过就不跑后面的
    
    # Sensor 2: 类型检查（秒级）
    type_result = subprocess.run(
        ["mypy", "--strict"] + changed_files,
        capture_output=True, text=True, timeout=60
    )
    results.append(SensorResult(
        name="mypy_typecheck",
        passed=type_result.returncode == 0,
        output=type_result.stdout,
        cost_ms=3000
    ))
    if not results[-1].passed:
        return results
    
    # Sensor 3: 单元测试（秒~分钟级）
    test_result = subprocess.run(
        ["pytest", "--timeout=60", "-x"],  # -x: 第一个失败就停
        capture_output=True, text=True, timeout=120
    )
    results.append(SensorResult(
        name="pytest_unit",
        passed=test_result.returncode == 0,
        output=test_result.stdout[-2000:],  # 截断避免 token 爆炸
        cost_ms=15000
    ))
    
    return results


# ===== Permission Gate =====
class PermissionGate:
    """关键操作的权限门禁"""
    
    # 高风险操作需要人工确认
    HIGH_RISK_PATTERNS = [
        "DELETE FROM",
        "DROP TABLE",
        "rm -rf",
        "force push",
        "migration",
    ]
    
    @classmethod
    def check(cls, action: str, args: dict) -> tuple[bool, str]:
        """返回 (allowed, reason)"""
        action_str = json.dumps(args, ensure_ascii=False)
        
        for pattern in cls.HIGH_RISK_PATTERNS:
            if pattern.lower() in action_str.lower():
                return False, f"高风险操作检测: '{pattern}'。需要人工确认。"
        
        return True, "allowed"


# ===== Budget Control =====
@dataclass
class BudgetPolicy:
    max_steps: int = 30
    max_tokens: int = 500_000
    max_file_changes: int = 10
    current_steps: int = 0
    current_tokens: int = 0
    current_file_changes: int = 0
    
    def check(self) -> tuple[bool, str]:
        if self.current_steps >= self.max_steps:
            return False, f"步数预算耗尽 ({self.max_steps})"
        if self.current_tokens >= self.max_tokens:
            return False, f"Token 预算耗尽 ({self.max_tokens})"
        if self.current_file_changes >= self.max_file_changes:
            return False, f"文件修改数超限 ({self.max_file_changes})"
        return True, "within budget"


# ===== Agent with Full Harness =====
def run_harnessed_agent(task: str):
    """带完整 Harness 的 Agent 主循环"""
    
    budget = BudgetPolicy()
    changed_files = []
    
    system_prompt = [{
        "type": "text",
        "text": f"{AGENTS_MD}\n\n{SKILL_REFACTOR}\n\n当前任务: {task}",
        "cache_control": {"type": "ephemeral"}
    }]
    
    messages = [{"role": "user", "content": task}]
    
    tools = [
        {"name": "read_file", "description": "读取文件内容", 
         "input_schema": {"type": "object", "properties": {"path": {"type": "string"}}, "required": ["path"]}},
        {"name": "write_file", "description": "写入文件（会触发 Sensor 链验证）", 
         "input_schema": {"type": "object", "properties": {"path": {"type": "string"}, "content": {"type": "string"}}, "required": ["path", "content"]}},
        {"name": "run_tests", "description": "运行测试套件", 
         "input_schema": {"type": "object", "properties": {}, "required": []}},
    ]
    
    while True:
        # Budget check
        ok, reason = budget.check()
        if not ok:
            print(f"[BUDGET] {reason}")
            break
        
        budget.current_steps += 1
        
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=4096,
            system=system_prompt,
            tools=tools,
            messages=messages,
        )
        
        budget.current_tokens += response.usage.input_tokens + response.usage.output_tokens
        messages.append({"role": "assistant", "content": response.content})
        
        if response.stop_reason == "end_turn":
            break
        
        # 处理工具调用
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                # Permission Gate
                allowed, gate_reason = PermissionGate.check(block.name, block.input)
                if not allowed:
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": f"[BLOCKED] {gate_reason}",
                        "is_error": True,
                    })
                    continue
                
                # 执行工具
                result = execute_with_harness(block.name, block.input, changed_files, budget)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })
        
        messages.append({"role": "user", "content": tool_results})
    
    # 最终 Sensor 链
    if changed_files:
        print("[FINAL SENSOR] 运行完整验证链...")
        results = run_sensor_chain(changed_files)
        for r in results:
            status = "✓" if r.passed else "✗"
            print(f"  {status} {r.name} ({r.cost_ms}ms)")
            if not r.passed:
                print(f"    {r.output[:200]}")


def execute_with_harness(name: str, input_data: dict, 
                          changed_files: list, budget: BudgetPolicy) -> str:
    """带 Harness 的工具执行"""
    if name == "write_file":
        path = input_data["path"]
        budget.current_file_changes += 1
        changed_files.append(path)
        
        # 写文件后立即跑快速 Sensor（lint）
        Path(path).write_text(input_data["content"])
        quick_check = run_sensor_chain([path])  # 只会跑到第一个失败
        
        if all(r.passed for r in quick_check):
            return f"已写入 {path}，快速检查通过"
        else:
            failed = next(r for r in quick_check if not r.passed)
            return f"已写入 {path}，但 {failed.name} 失败:\n{failed.output[:500]}\n请修复。"
    
    elif name == "read_file":
        path = input_data["path"]
        try:
            return Path(path).read_text()[:10000]
        except FileNotFoundError:
            return f"文件不存在: {path}"
    
    elif name == "run_tests":
        results = run_sensor_chain(changed_files)
        return "\n".join(f"{'✓' if r.passed else '✗'} {r.name}: {r.output[:200]}" for r in results)
    
    return "未知工具"
```

## 8. 实践取舍：从哪一层开始

工程上的递进顺序：

1. **先把 Guides 写对**——AGENTS.md + 基本的编码规范，成本最低、见效最快
2. **配 fast Sensor**——pre-commit hook 跑 lint + typecheck，拦截最明显的错误
3. **加 Permission Gate**——不可逆操作必须过门禁
4. **跑通完整工作流**——在跑通之前不要自动化
5. **再考虑 Loop**——Loop 上线必须同时配齐 Workspace 四原语 + 预算策略 + 回滚路径

跳过任何一步直接上更高层，结果通常是把不靠谱的 Agent 用更高的频率放大。

> **"It's not a model problem. It's a configuration problem."** — HumanLayer
