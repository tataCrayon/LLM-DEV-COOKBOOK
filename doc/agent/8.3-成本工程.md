# 8.3 成本工程：Prompt Caching / Batch API / 级联路由 / 预算管理

> Agent 应用在 PoC 阶段觉得"LLM 不就 $0.005/请求嘛"，到生产 100k DAU 才发现一个月烧 $200k。本篇把 Agent 成本拆解成可治理的预算单元，给出 4 个层级的工程降本手段以及一套预算治理体系。

---

## §0 真问题：成本结构的"三大错觉"

新团队对 Agent 成本有 3 个普遍误解：

**错觉 1：成本 ∝ DAU**

实际上 Agent 成本 ∝ **trajectory token 总量** = 用户数 × 平均步数 × 平均上下文长度 × (1 + 工具回环数)。一个 DAU 在 chatbot 时代 = $0.001/天，在 deep research Agent 时代可能 = $5/天，差 5000×。

**错觉 2："Agent 主要烧在 LLM 输出 token"**

实测多 Agent + 长上下文 + 工具回环的应用，**输入 token 通常是输出 token 的 10-100×**。Anthropic 2025 数据：典型 SWE-bench Verified 评测中，输入/输出比可达 50:1。这也是为什么 Prompt Caching（针对输入）成为最重要的降本手段。

**错觉 3："换便宜模型立刻省钱"**

便宜模型常常需要 2-3 倍 token 才能完成同任务（reasoning 不足 → 自我修正回环增加），或者输出质量不达标导致用户重发。FrugalGPT 论文显示"无脑换 Haiku 反而比 Sonnet 更贵"的反直觉案例并不少见。

正确的第一性原理：

```
Cost_per_outcome = (Σ token_in × p_in + Σ token_out × p_out)
                  ── 不是 cost_per_call，是 cost_per_outcome（完成一次业务任务）
                  
                  影响因子：
                  - 模型选型（价差最大 100×）
                  - 缓存命中率（直接 -90%）
                  - Batch async（直接 -50%）
                  - 级联路由（最高 -98%，但工程复杂）
                  - 上下文压缩（输入 token -50%）
                  - 工具调用回环数（Agent 设计本身）
```

## §1 第一性原理：成本工程的四层金字塔

```
                  ┌─────────────────────────┐
                  │   L4: 业务层成本治理      │   预算/配额/计费
                  │   (FinOps 落地)         │   按租户、按 workflow
                  ├─────────────────────────┤
                  │   L3: 模型选择层         │   级联路由 / Router
                  │   (cascading)           │   模型质量分类
                  ├─────────────────────────┤
                  │   L2: 调用模式层         │   Batch API / 异步
                  │   (sync vs async)       │   长尾任务批跑
                  ├─────────────────────────┤
                  │   L1: Token 层          │   Prompt cache / 压缩
                  │   (token economics)     │   Context engineering
                  └─────────────────────────┘
```

每一层有独立的工程手段，从下到上**成本节省的边际递增、复杂度递增**。生产建议按 L1 → L2 → L3 → L4 顺序推进，而不是直接跳到 L3 做高大上的 router。

## §2 L1：Token 层降本——Prompt Caching 是头号武器

### 2.1 Prompt Caching 的成本数学

各家定价（截至 2026）：

| Provider | Cache Write | Cache Read | Read vs Base |
|---|---|---|---|
| Anthropic Claude (5 min TTL) | 1.25× base | 0.1× base | **节省 90%** |
| Anthropic Claude (1 h TTL) | 2.0× base | 0.1× base | 节省 90% |
| OpenAI GPT-4o / GPT-5 | 1.0× base（无 write 加价） | 0.5× base | 节省 50% |
| Google Gemini (implicit) | 1.0× base | 0.25× base | 节省 75% |
| Google Gemini (explicit + storage) | 1.0× base + storage fee | 0.25× base | 同上 |

**关键洞察**：Anthropic 收 25% write premium 但 read 只要 10% base，OpenAI 不收 write premium 但 read 是 50%。**对高复用率场景（Agent 长 system prompt + 大量短交互），Anthropic 更省**；对一次性长 prompt 场景，OpenAI 写一次免溢价更划算。

### 2.2 Agent 场景的实测节省

[Don't Break the Cache (arXiv 2601.06007)](https://arxiv.org/abs/2601.06007) 在 DeepResearch Bench (500+ sessions, 10K-token system prompt) 上实测：

- 成本节省 **41%-80%**（跨 OpenAI/Anthropic/Google）
- TTFT 改善 **13%-31%**
- 收益线性扩展到 500-50K token prompt、3-50 次工具调用

### 2.3 Cache-friendly Prompt 设计原则

**核心规则：把动态内容放到 prompt 末尾，前缀保持稳定**。Agent 系统 prompt 90% 命中率的关键：

```python
# ❌ 反模式：动态内容夹在中间，前缀也变
prompt = f"""你是 {agent_name}，专门处理 {user.tier} 用户。
[10000 字工具说明]
当前时间：{datetime.now()}    # ← 这一行让前缀每秒变一次，缓存击穿
[10000 字 few-shot 示例]
{user_query}
"""

# ✅ 正确模式：稳定前缀 + 末尾注入动态
prompt_prefix = f"""你是 Agent。     # 稳定
[10000 字工具说明]          # 稳定
[10000 字 few-shot 示例]    # 稳定
"""
prompt_suffix = f"""
当前时间：{datetime.now()}
租户：{user.tier}
查询：{user_query}
"""
# 在 prefix 末尾放 cache_breakpoint，suffix 不缓存
```

### 2.4 Cache breakpoint 选择

Anthropic 允许最多 4 个 cache breakpoint，每个对应一段可独立失效的缓存层。生产推荐分层：

```
[BP1] System Identity & Policy         (改动频率 < 月)
[BP2] Tool Definitions (Function calls) (改动频率 < 周)
[BP3] Few-shot Examples                 (改动频率 < 天)
[BP4] Long-term Memory / RAG Context    (改动频率 < 会话)
─────────────────────────────────────
Dynamic: 当前查询、当前工具结果（不缓存）
```

这样改一次 system prompt 不会冲掉 few-shot 缓存。

### 2.5 三大缓存陷阱

[arXiv 2601.06007] 警告："naive full-context caching can paradoxically increase latency"。三个常见陷阱：

1. **动态工具结果塞进缓存块**：每轮工具调用结果变了，缓存击穿；正确做法：工具结果放 dynamic 区；
2. **每次都改前缀的微小细节**（比如 timestamp）：99% 内容一样但前 100 字变了 = 0 命中率；
3. **prompt 长度低于厂商门槛**：Anthropic 1024 token 起，Google Gemini 32K 起；短 prompt 不要折腾缓存。

### 2.6 监控指标

```
cache_read_input_tokens / total_input_tokens  → 命中率，目标 >70%
cost_with_cache / cost_without_cache          → 节省比，目标 <0.3
```

不要看"hit yes/no 二值"，看**输入 token 中有多少是缓存的**。

## §3 L2：调用模式层——Batch API + 异步

### 3.1 Batch API：50% 折扣的隐藏金矿

各家 Batch API：

| Provider | 折扣 | SLA | 接口 |
|---|---|---|---|
| OpenAI Batch API | 50% off | 24h（通常 <1h） | JSONL 上传 |
| Anthropic Message Batches | 50% off | 24h | JSON list |
| Google Vertex Batch | 50% off | 24h | BigQuery / GCS |

**核心约束**：不能用于交互式响应（用户在等）；适合后台批跑。

### 3.2 Agent 中能批跑的场景

被多数团队忽略的批量化机会：

1. **离线 eval**：每次发版要在 5000 题上跑回归 → Batch 跑成本砍半；
2. **数据标注与合成**：用大模型给 100K 条数据打标签、生成 SFT 数据 → 强适合 Batch；
3. **批量摘要 / 翻译 / 嵌入**：用户上传 1000 个文档批处理 → Batch；
4. **预计算缓存**：常见查询提前算好结果写缓存 → Batch 离线生成；
5. **后台 Agent 的非紧急步骤**：比如 deep research 的"对 50 篇论文做摘要"那一步，可以丢 Batch 队列等小时级返回；
6. **A/B test 候选生成**：要在 prompt v1/v2/v3 上跑 1000 个样本对比 → Batch。

### 3.3 工程模式：同步 + Batch 双轨

```python
class LLMRouter:
    async def call(self, request: LLMRequest):
        if request.user_facing or request.latency_sla < 60_000:
            return await self.sync_call(request)        # 用户在等 → 同步
        elif request.deadline_seconds > 3600:
            return await self.batch_submit(request)     # 不急 → Batch 省一半
        else:
            return await self.sync_call(request)
    
    async def batch_submit(self, request):
        # 写入 batch queue，由后台 worker 聚合成 JSONL 提交
        await batch_queue.enqueue(request)
        return PendingBatchResponse(batch_id=...)
```

### 3.4 Batch 反模式

- 把交互式查询走 Batch（用户等 24h，活该被骂）；
- 不做超时回退：Batch 偶尔超 SLA，没有 sync fallback；
- 不批量化合并：一个 Batch job 只放 10 条请求，错失规模效应；
- 不做 Batch 失败重提：单条失败整批不处理（Batch API 返回部分成功，每条独立 JSON）。

## §4 L3：模型选择层——级联路由（Cascade Routing）

### 4.1 FrugalGPT 的核心思想

[FrugalGPT arXiv:2305.05176](https://arxiv.org/abs/2305.05176) 提出三件套：

1. **Prompt Adaptation**：少样本精选、压缩、合并；
2. **LLM Approximation**：精确缓存、语义缓存、小模型微调；
3. **LLM Cascade**：按"便宜 → 贵"顺序调用，置信度足够即返回。

论文声称在某些数据集上 **匹配 GPT-4 表现的同时削减 98% 成本**。生产实际收益 30-70%，取决于任务可分级度。

### 4.2 Cascade 的两个核心组件

**(1) Scoring Function** `g(query, answer) → [0, 1]`

判断当前模型的回答是否"够好"。常见做法：

- **Self-evaluation**：让 LLM 自己判分（cost: 一次 cheap call）；
- **Classifier**：训一个 DistilBERT-class 模型 (1 ms 推理)；
- **Logprob-based**：取 LLM 输出的 confidence (free，但只对 logprob API 可用)；
- **Heuristic**：长度、是否含 "I'm not sure"、是否能解析成 schema (free)。

**(2) Router with threshold**

```python
MODEL_CHAIN = [
    ("gpt-4o-mini",   0.7),   # threshold: 分数 ≥0.7 → 提前返回
    ("claude-sonnet", 0.85),  # 否则升级到 Sonnet，分数 ≥0.85 → 返回
    ("claude-opus",   1.0),   # 最后兜底 Opus
]

async def cascade_call(query: str):
    for model, threshold in MODEL_CHAIN:
        answer = await llm.call(model, query)
        score = await scoring_fn(query, answer)
        if score >= threshold:
            metrics.record_hit(model)
            return answer
    return answer  # last model 兜底
```

### 4.3 阈值怎么调

**严禁拍脑袋**。生产做法：

1. 在 5000-10000 题的验证集上跑全模型，记录 (query, model, answer, score, ground_truth)；
2. 做 grid search：枚举 threshold 组合，目标函数 = **min cost s.t. accuracy ≥ X%**；
3. 把最优阈值固化到配置，每月 retune；
4. 配套 7.4 的 drift 检测：score 分布漂移时触发 retune。

### 4.4 Router 的另一种范式：分类先行

不是"试便宜模型 → 不行升级"，而是"先分类查询难度 → 直接派到对应模型"：

```python
async def router_call(query: str):
    difficulty = await classify_difficulty(query)  # 小分类器 1ms
    if difficulty == "trivial":
        return await llm.call("gpt-4o-mini", query)
    elif difficulty == "moderate":
        return await llm.call("claude-sonnet-4.5", query)
    else:
        return await llm.call("claude-opus-4.5", query)
```

优点：省去 cascade 的"先调一次便宜模型"开销；缺点：分类器训练数据要求高。

**实战推荐**：**简单任务用 classifier router；复杂任务用 cascade**。混合最好。

### 4.5 Semantic Cache

普通缓存（KV by exact prompt）命中率约 8%，**语义缓存（向量匹配）平均 21%，高复用场景 60%**。架构：

```
query → embed → vector_search(top1) → similarity > 0.95?
                                      ├─ yes → return cached_answer
                                      └─ no  → call LLM → cache (query_embed, answer)
```

陷阱：
- 阈值不要拍：用 5k 请求回测，找到 accuracy ≥99% 的阈值；
- 多租户隔离：embed 不分租户 = 数据泄漏；
- 负反馈触发 invalidate：用户点踩 → force-refresh，否则反复返回坏答案。

## §5 L4：预算管理——FinOps 落地

### 5.1 三级预算

```
Org Budget (月预算 $50K)
  ├── Workload A: chatbot         ($20K)
  │     ├── Tenant tier=enterprise  ($15K)
  │     └── Tenant tier=free        ($5K, hard cap)
  ├── Workload B: deep research    ($20K)
  └── Workload C: batch eval       ($10K)
```

每一层都要有：
- **soft limit**：触发告警，但继续服务；
- **hard limit**：触发 → 切便宜模型 / 降级 / 拒服务；
- **time window**：日/月预算双重，防止月头一日烧光。

### 5.2 实时成本计量

每次 LLM 调用必须实时记账（与 7.3 联动）：

```python
@dataclass
class CostEvent:
    tenant: str
    workload: str
    model: str
    input_tokens: int
    output_tokens: int
    cached_tokens: int
    cost_usd: float
    trace_id: str
    timestamp: float

def compute_cost(model: str, in_tok: int, out_tok: int, cached_tok: int) -> float:
    p = PRICING[model]
    cache_cost = cached_tok * p["input"] * 0.1   # Anthropic style
    new_input_cost = (in_tok - cached_tok) * p["input"]
    out_cost = out_tok * p["output"]
    return cache_cost + new_input_cost + out_cost
```

写到 OTel span attribute + 时序库（Prometheus / ClickHouse）。

### 5.3 与 8.2 的成本速度熔断

8.2 §5 已经讲了：当 rolling 5min 成本 > 10× planned rate，立即熔断。**这是预算治理的最后防线**，不要省。

### 5.4 把成本指标接入产品决策

不要只算"总花了多少钱"，要算 **cost-per-outcome**：

| 指标 | 公式 | 用途 |
|---|---|---|
| Cost per task | 总 token cost / 完成任务数 | 跟踪 Agent 设计变化的成本影响 |
| Cost per resolved ticket | 总 cost / 工单解决数 | 与传统人工对比 ROI |
| Cost per code commit | 总 cost / 通过 CI 的代码提交 | SWE Agent 经济性 |
| Marginal cost per +1pp accuracy | Δcost / Δaccuracy | 决定是否上更贵模型 |

7.4 的 A/B test 设计里，OEC 应该是 **质量 / 成本** 的复合指标，不是单纯准确率。

## §6 五种降本策略组合实战

| 场景 | L1 缓存 | L2 Batch | L3 Cascade | L4 预算 | 预期节省 |
|---|---|---|---|---|---|
| 高复用 chatbot（10k DAU） | ✅ semantic+prompt | ❌ | ⚠️ classifier router | ✅ | 50-70% |
| Deep research Agent | ✅ prompt | ⚠️ 中间步骤 | ✅ cascade | ✅ | 40-60% |
| 批量标注 / 合成数据 | ⚠️ | ✅ 必须 | ⚠️ | ✅ | 50-80% |
| SWE Agent | ✅ 重 prompt cache | ❌ | ✅ cheap planner + opus executor | ✅ | 30-50% |
| 多租户 SaaS 网关 | ✅ + 租户隔离 | ⚠️ async only | ✅ | ✅ 必须 | 40-60% |

**真实案例（Anthropic 2025 客户案例平均）**：
- 引入 prompt caching → -50% cost、TTFT -25%；
- 引入 cascade (Haiku → Sonnet → Opus) → 额外 -30%；
- 联合后接近 **-70% 总成本**，质量持平。

## §7 反模式

| 反模式 | 后果 |
|---|---|
| 直接换最便宜模型 | 准确率 ↓ → 用户重发率 ↑ → 总成本可能更高 |
| 无脑 full-context caching | 动态内容击穿，反而 TTFT 上升 |
| Cache key 不分租户 | 数据泄漏，越权读其他人的回答 |
| Batch API 走交互查询 | 用户等 24h |
| Cascade 阈值拍脑袋 | 要么早返回（质量差），要么从不早返回（白做 cascade） |
| Semantic cache 阈值 0.8 | 召回了大量错答案 |
| 不监控 cache 命中率 | 不知道改 prompt 后击穿了缓存 |
| 没有 hard limit | 一个 bug 一夜烧完月预算 |
| 只看总成本不看 cost-per-outcome | 优化方向错误 |
| 优化输出 token 不优化输入 | 输入是输出 10-100×，搞错重点 |

## §8 S0 → S6 路线

| 阶段 | 关键动作 | 累计节省 |
|---|---|---|
| S0 | 没意识，看月账单傻眼 | 0 |
| S1 | 加 prompt caching + cache-friendly 设计 | -40% |
| S2 | 离线评测 / 数据标注走 Batch API | -50% |
| S3 | 加 semantic cache + 多租户隔离 | -55% |
| S4 | 加 cascade routing (难度分类 or 置信度路由) | -65% |
| S5 | 加 FinOps 预算治理 + cost velocity 熔断 | -70% |
| S6 | 把 cost-per-outcome 接入产品决策 + 月度 retune | -75% + 治理闭环 |

**90% 团队的最大单笔收益在 S1**，先把 prompt caching 调好再说其他。

## §9 一页总结

把 8.3 压成一句话：**Agent 成本治理 = 把成本拆成 token/调用/模型/预算四层，每层用独立工具（cache/batch/cascade/budget）；优先做 Prompt Caching（最大单笔收益）、Batch 异步（最简单的 50% off）、Cascade（高复杂度但 -98% 上限），用 cost-per-outcome 而非 cost-per-call 度量。**

承接关系：
- **8.2 可靠性**：成本速度熔断是预算治理的最后一道闸；
- **8.4 延迟**：Prompt cache 同时优化成本与 TTFT，是两章交集；
- **8.5 模型选型**：cascade 路由背后是模型选型矩阵；
- **8.6 数据飞轮**：cascade 中小模型的训练数据来自大模型 trace（自蒸馏）；
- **5.x Context Engineering**：prompt cache 设计与 context 管理深度联动；
- **7.4 持续改进**：cost OEC 是 A/B test 的核心指标。
