## 0. 真问题：Agent 出事的那一刻，你能在 5 秒内停下吗？

凌晨 2:17，监控大盘跳红：某 Agent 在 30 秒内调了 1,400 次 `delete_object`，已删了 870 万条客户记录。值班工程师奔到键盘，发现：

- Agent 主进程在 K8s 里跑得好好的，没有 panic；
- 没有"停止"按钮，平台层只能通过 `kubectl scale --replicas=0` 干掉整个 Agent 服务——**会影响所有租户**；
- 模型层无 abort 接口，只能等本次推理完成；
- Memory / Vector 已经被部分污染，回滚需要 30 分钟以上；
- 审计链断在了"agent 决策出动作"那一步——根因分析延迟到次日下午。

这是 2025 年真实事故的合成版。问题不是"防得不够"，而是 **"出事时刹不住"**。EU AI Act Art.14 要求 high-risk AI 系统必须 "remain capable of being effectively overseen by humans"——可观测+可中断+可回滚是**法定义务**，不是工程奢侈品。

到 2026 年，业内共识：**Kill Switch 不是单个按钮，是一个跨 5 层（模型 / 进程 / 工具 / 数据 / 网关）、3 档（Surgical / Class / Nuclear）的容控架构**。本文给出完整工程化方案。

## 1. 概念分层：Interrupt / Pause / Quarantine / Rollback / Kill 是 5 件不同的事

不要把这 5 个动作混为一谈：

| 动作 | 范围 | 时延要求 | 可逆性 | 触发场景 |
|---|---|---|---|---|
| **Interrupt** | 当前一次 LLM call / tool call | <100ms | 是 | 单次决策已知错误 |
| **Pause** | 单 Agent 实例 / 整条任务 | <1s | 是 | 等待人工核查 / 修复 |
| **Quarantine** | 单 Agent 类 / Memory 子集 / Tool 子集 | <10s | 是 | 怀疑漂移 / 投毒 |
| **Rollback** | 数据 / Memory / Action 序列 | 分钟级 | 是（需要日志） | 已确认错误，需撤销 |
| **Kill** | Agent 类 / 整集群 | <5s（Nuclear） | 否（重启 = 新实例） | 灾难性事故 |

不同动作走不同物理路径：

- Interrupt 通过 LLM 提供商的 abort token / Streaming cancel；
- Pause 通过 Agent Runtime 的状态机（State = `running → paused`）；
- Quarantine 通过 Gateway / Identity 系统的策略闸；
- Rollback 通过 Append-only Log 的逆向重放（与 [6.6](./6.6-Memory与DataSecurity.md) 联动）；
- Kill 通过 Gateway scale-to-zero / JWT 撤销 / 进程信号。

把这 5 件事写进同一份系统手册，是合规审计的硬要求。

## 2. 总体架构：Control Plane / Data Plane / Action Plane

```
        ┌───────────────────────────────────────────┐
        │  Control Plane                              │
        │  - Kill Switch Console（HITL）              │
        │  - Policy / Quota / Rate                     │
        │  - JWT issuer + revoker                      │
        │  - Telemetry → Prometheus / Jaeger / Langfuse│
        └───────────┬─────────────────┬────────────────┘
                    │ control          │ telemetry
                    ▼                  ▲
        ┌──────────────────────────────────────────┐
        │  Data Plane (Agent Gateway)               │
        │  - mTLS / JWT 校验                         │
        │  - Tool allowlist + CEL Authz             │
        │  - Token bucket / RPM 限流                 │
        │  - Circuit Breaker（Action Plane 故障感知） │
        │  - 阻断 / 限速 / 转 quarantine                │
        └───────────┬──────────────┬──────────────┘
                    │ guarded calls │ guarded calls
                    ▼               ▼
        ┌──────────────┐    ┌─────────────────────┐
        │  LLM Provider │    │  Tools / MCP / DB    │
        │  (abort 支持) │    │  (Idempotency / DLQ) │
        └──────────────┘    └─────────────────────┘

        Action Plane = 实际执行层，必须支持 abort + idempotency + DLQ
```

设计原则（与 [Agent Gateway 实践](https://maniak.io/articles/2026-02-21-multi-agent-architecture-agentgateway-kill-switch/) 一致）：

> "Coordinator 决定做什么，Gateway 决定是否允许；Specialist 无状态、可丢弃。"

无状态 + 网关守门 = 可一键停。状态全部托管在 Coordinator + Memory，方便恢复（[5.3 长期记忆](./5.3-长期记忆架构.md)）。

## 3. 信号源：Kill Switch 该被谁触发？

**自动触发（机器）**：

| 信号 | 阈值 | 动作 |
|---|---|---|
| guardrail_block_rate | >5% over 60s | Pause + 告警 |
| tool_error_rate | >10% over 5min | Class quarantine |
| 单 agent token spend | >$50/min | Interrupt + Pause |
| Action 速率（writes/sec） | 业务基线 +5σ | Class quarantine |
| Memory drift score | >0.3 | Quarantine 该 subject |
| Inter-agent 异常协作模式 | 自学习异常检测 | Surgical kill |
| 红队回归 ASR | P0 漏洞 >1% | Block 上线 + alert |
| 合规事件（PII 流出 canary） | 任意命中 | Nuclear（全面停） |

**人工触发**：

- HITL 决策树（[6.4](./6.4-RedTeaming合规审计与人机信任边界.md)）；
- DPO / 安全官 / On-call 的 Web 控制台；
- 监管要求（EU AI Act Art.72 重大事件 → 立即响应）。

**外部触发**：

- 上游模型供应商发布 Safety Advisory（如：检测到训练数据污染） → 自动 quarantine 受影响版本；
- CVE 推送（MCP 工具 CVE）→ 与 [6.5 供应链](./6.5-AgentIdentity与供应链安全.md) 联动 → 撤销该工具 token。

## 4. 三档容控：Surgical / Class / Nuclear

### 4.1 Surgical（外科式，秒级）

撤销**单个 Agent 实例**的能力：

```python
def surgical_kill(agent_did: str, reason: str):
    # 1) JWT 撤销 + StatusList 标记
    jwt_issuer.revoke(agent_did, reason)
    vc_status.revoke(agent_did)
    # 2) 拉黑该 Agent 的入站请求（gateway level）
    gateway.add_deny(subject=agent_did, ttl="1h")
    # 3) 当前在飞 LLM call 与 tool call 全部 abort
    inflight.abort_by_agent(agent_did)
    # 4) 回收正在使用的 sub-task tokens（与 6.3 RAR 联动）
    rar.revoke_all_active(agent_did)
    # 5) 标记 audit event
    audit.emit("surgical_kill", agent_did=agent_did, reason=reason)
```

时延目标：**<5s 全链路收敛**。

### 4.2 Class（类级，10s 级）

按 Agent role / 模型版本 / 工具 / Memory namespace 维度生效：

```python
def class_quarantine(class_selector: dict, reason: str):
    # 例：class_selector = {"role": "customer-support", "version": "3.1.7"}
    policy.set_rate_limit(class_selector, tokens_per_min=0)
    policy.set_route_status(class_selector, status="paused")
    # 触发该 class 在飞任务的 Pause（不撤销，可恢复）
    runtime.pause_all(class_selector)
    audit.emit("class_quarantine", selector=class_selector, reason=reason)
```

恢复时只需 `set_rate_limit` 还原 + `runtime.resume_all`。

### 4.3 Nuclear（核选项，<5s 全停）

```bash
# Gateway 缩到 0
kubectl scale deployment agentgateway --replicas=0
# 同时禁用 LLM 出口（防止有 fallback 路径）
kubectl apply -f netpolicy/deny-egress-llm.yaml
# 撤销所有 active VC（StatusList 批量 revoke）
curl -X POST $VC_REVOKE_API/all-active
# 标记紧急事件，触发 Incident 预案
incident.declare("nuclear_kill", severity="P0")
```

要求：必须**演练**（至少季度一次）。否则真出事时按钮按下去，发现某个绕路还在跑（自然语言别名：纸上谈兵的 Kill Switch 等于没有）。

### 4.4 三档决策树

```
出事级别                          推荐动作
─────────────────────────────────────────────────────
单 Agent 异常 / 可疑              Surgical
某版本 / 某工具普遍异常             Class
跨多 Agent / 数据外泄 / 监管要求    Nuclear
```

不同档位的**回归测试**（恢复演练）成本不同——Nuclear 需要全量重启，对所有租户都会有 RPO/RTO 影响。

## 5. Interrupt：单次决策的中断

### 5.1 LLM Streaming Abort

主流厂商已支持：

- OpenAI / Azure OpenAI: `controller.abort()` on streaming；
- Anthropic Claude: streaming 取消 + `stop_sequence`；
- Google Gemini: SSE close + cancel；
- vLLM / 自部署: `request_abort()` API。

实现规范：

```python
async def llm_call_with_abort(prompt, abort_signal):
    stream = await client.completions.stream(prompt=prompt)
    async for chunk in stream:
        if abort_signal.is_set():
            await stream.close()
            raise InterruptedError("aborted_by_kill_switch")
        yield chunk
```

`abort_signal` 是一个全局可见的 Event，任何监控信号触发都能立刻中止当前推理。**没有 abort 路径的模型不要进生产**。

### 5.2 Tool Call Abort

工具调用的 abort 比 LLM 难，因为：

- 已发出的 HTTP 请求未必能取消；
- 已写入数据库的事务无法 abort，只能 compensate。

设计要求：

- 所有写工具必须支持 **Idempotency Key**（防重 + 可补偿）；
- 工具运行时挂载 **Cancellation Context**（context.WithCancel / asyncio.CancelledError）；
- 每个工具有 **timeout**（即使 abort 失败，超时后必死）；
- 失败请求进 **DLQ**（Dead Letter Queue），人工审视后决定是否 compensate。

## 6. Pause：可恢复的暂停

Pause 用于：等待 HITL 审批、等待外部状态、Hold 高风险任务。需要持久化的 State Machine：

```python
from enum import Enum

class TaskState(str, Enum):
    RUNNING = "running"
    PAUSED  = "paused"
    QUARANTINED = "quarantined"
    COMPLETED = "completed"
    FAILED = "failed"

class TaskRecord(BaseModel):
    task_id: str
    agent_did: str
    state: TaskState
    checkpoint: dict          # 续跑所需的全部上下文
    pause_reason: str | None
    pause_ts: float | None
    resume_authz: str | None  # 恢复时必须的 authz scope
```

恢复路径：

```python
def resume_task(task_id: str, approver: str, authz: str):
    task = task_store.get(task_id)
    if task.state != TaskState.PAUSED:
        raise InvalidState
    if not policy.can_resume(approver, authz, task):
        raise PermissionDenied
    task.state = TaskState.RUNNING
    task_store.update(task)
    runtime.resume(task)
    audit.emit("task_resume", task_id=task_id, approver=approver)
```

要点：

- Pause 必须是**幂等**的（重复 pause 不出错）；
- 恢复必须有**单独 authz**（不能让普通用户解 paused 的 R4 任务）；
- Pause 有最大窗口（默认 7 天），超时自动 quarantine。

## 7. Quarantine：隔离区设计

Quarantine 的对象不只是 Agent，还可以是 Memory / Tool / 模型版本：

| 类型 | 对象 | 恢复路径 |
|---|---|---|
| Agent Quarantine | 该实例 / 类 | 解除 + 漂移指标恢复 |
| Memory Quarantine | 单条 / subject / namespace | [6.6](./6.6-Memory与DataSecurity.md) 双 LLM 审视 |
| Tool Quarantine | 工具 / MCP server | [6.5](./6.5-AgentIdentity与供应链安全.md) 重新 attestation 通过 |
| Model Version Quarantine | 模型 hash | 厂商 advisory 撤销 + 内部回归通过 |
| Action Quarantine | 待执行 action | HITL 复核后释放 |

Quarantine ≠ Kill：内容仍在，但不参与生产路径。

## 8. Rollback：把已发生的"撤回去"

### 8.1 数据 Rollback 的层次

| 层次 | 机制 | 复杂度 |
|---|---|---|
| Memory（[6.6](./6.6-Memory与DataSecurity.md)） | append-only log → revert by ts | 中 |
| 业务 DB | DB-native point-in-time / saga compensate | 高 |
| 外部副作用（邮件已发） | 不可回滚，发"道歉/更正"邮件 | — |
| 文件 / 对象存储 | 版本化 + soft delete + 物理 delete window | 中 |

### 8.2 Saga Compensation 模板

每个工具不只要有 `do(args)`，还必须有 `undo(args, do_result)`：

```python
class ToolSpec(BaseModel):
    name: str
    do: Callable
    undo: Callable | None
    compensable: bool

def execute_with_saga(plan: list[ToolCall]):
    log = []
    try:
        for call in plan:
            res = call.spec.do(call.args)
            log.append((call, res))
        return Outcome(ok=True, log=log)
    except Exception as e:
        # 反向补偿
        for call, res in reversed(log):
            if call.spec.compensable:
                try: call.spec.undo(call.args, res)
                except Exception: dlq.send(call, reason="undo_failed")
        return Outcome(ok=False, error=str(e), log=log)
```

不可补偿的工具（如：发币、删客户）默认要求 R4+ HITL 预审批。

### 8.3 Action Replay vs Reverse

两种风格：

- **Reverse**：每步都有 undo；
- **Replay-from-checkpoint**：把数据库恢复到某 timestamp，然后用新 Agent 版本重跑（适合幂等场景）。

数据敏感场景推荐 Replay；外部副作用场景必须 Reverse。

## 9. Circuit Breaker：模型/系统级"自动保险丝"

### 9.1 系统层 Circuit Breaker

经典 Hystrix / Resilience4j 模式适用于：

- 单工具失败率超阈值 → open（拒绝调用）→ half-open 探测 → close；
- LLM 提供商 5xx 率超阈值 → 自动切备用提供商或 degrade；
- 单 Agent 调用激增 → 触发限流 + alert。

### 9.2 模型表征层 Circuit Breaker（Representation Rerouting, RR）

Gray Swan AI 2024 的 Circuit Breakers 论文（[arXiv:2406.04313](https://arxiv.org/html/2406.04313v2)，2026 仍在更新）证明：

- 在 Layer 10 / 20 加 LoRA 适配器，对"有害表征"应用 short-circuit loss；
- 一旦内部表征命中"有害簇"，模型 **生成转为无意义输出**（拒绝产生有害内容）；
- 不需要 refusal training，不靠规则；
- 训练成本仅 **A100-80GB × 20min × 150 step**；
- 关键数据：
  - Llama-3-8B GCG-T 攻击 ASR：76% → **0%**；
  - 多语言 ASR：19.3% → **3.5%**；
  - Agent forced-function-call ASR：87% → **14%**；
  - 能力保留：MT-Bench / MMLU 几乎不掉。

工程意义：**模型层"自带 Kill Switch"**——比 Guardrail 更深一层，相当于"基因级"安全。建议：

- 自部署模型（vLLM / TGI）端：上 RR 适配器；
- API 模型端：等供应商集成（Anthropic Claude 已透露相关研究方向）；
- 多档防御：模型 RR + 进程 Pause + Gateway Quarantine + 数据 Rollback，**一层失守不等于全失守**。

## 10. 多 Agent / 跨服务的 Kill Switch 一致性

多 Agent 时 Kill Switch 必须**全网一致**——否则某个分支还在跑相当于没停。

### 10.1 Sigterm 风暴 vs Coordinated Shutdown

错误做法：直接给所有进程发 SIGKILL → 写一半的事务挂在不一致状态。

正确做法（与 K8s graceful shutdown 同理）：

```
1. Gateway 接到 kill 信号
2. 立即 stop_accepting_new_traffic
3. 现存 in-flight 请求得到 abort 信号 + 5s grace（用于做 saga compensation）
4. Memory / DB 写入完成 commit / rollback
5. JWT / VC 撤销
6. 进程退出
```

### 10.2 跨服务事务

- Agent 调用 Service A 后调 Service B：必须用 Saga 模式，B 失败回滚 A；
- LongRunning / Workflow 类（如 Temporal / Cadence）：原生支持 cancellation + compensation，强烈推荐。

### 10.3 一致性看板

控制台必须显示：

- 当前 active agents / inflight tasks / pending HITL；
- "Stop All" 按钮的预期影响范围（多少租户、多少任务）；
- Kill 信号的传播状态（每个节点是否确认）。

## 12. 反模式

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| 没有 abort，等本次推理结束 | 出事多挣的钱不够付罚款 | LLM streaming abort + 工具 cancellation |
| Kill Switch = 一个大红按钮 | 单粒度伤害大 | 三档（Surgical/Class/Nuclear） |
| Pause 不持久化 | 重启后丢状态 | Pause = 持久化状态机 |
| 工具无 idempotency / undo | Rollback 不可能 | Spec 强制 do/undo + idempotency_key |
| Quarantine 与 Kill 不分 | 可恢复的也变成毁灭的 | Quarantine 不删数据，仅停路径 |
| 模型层无 Circuit Breaker | 单层 guardrail 可绕过 | 加 RR 适配器（自部署）或选支持的厂商 |
| Kill Switch 没演练 | 真要用时翻车 | 季度演练 + Chaos Drill |
| Telemetry 无 Kill 信号 | 触发不智能 | 8 类自动信号 + alert |
| 多服务无 coordinated shutdown | 残留任务 | Saga + Workflow 引擎 |
| 没有合规归档 | EU/NIST 审计失分 | Runbook + 演练记录 + 事故复盘 |
