# Agent 权限模型：最小权限 / Scoped Credentials / Permission Gate

> 本文是 Agent 系列第 6.3 篇。**6.1 揭示注入是架构缺陷、6.2 把护栏拆成 Input/Output/Tool 三层。本篇专门回答一个问题：当 LLM 决定要"做事"——调工具、写数据、调钱——这个动作是否被授权？**
>
> 经典 web 应用的权限模型是 *用户 → 服务 → 资源*。Agent 时代多了一个 actor：**LLM 自己是新的 principal**。它带着用户的票去访问资源，但它的"决策"来自概率模型。这意味着传统的 OAuth scope、RBAC 都不够——你不能让 Agent 拿到 `email.send` scope 就乱发邮件。
>
> 这一篇会从一个具体的问题切入——**"为什么把客服 Agent 直连生产数据库永远是错的？"**——把 Agent 权限模型的整套工具拆开：最小权限原则在 Agent 时代的新形式、Rich Authorization Requests (RFC 9396) 与 2026 IETF agent 扩展、Permission Gate 的风险分级与 HITL 设计、Scoped Credentials 的多层下发、Capability 模型 vs RBAC vs ABAC 的取舍、以及 Pre-Action Authorization (Vault CTF 中 0% 攻击成功率) 这条新前线。

## 0. 真问题：为什么"按用户身份"授权 Agent 是危险的？

很多团队的第一版 Agent 权限设计都是这样：

```
登录用户 user_alice 调用 Agent →
Agent 拿着 user_alice 的会话 token →
工具调用时直接用这个 token 访问后端 API
```

这个设计在 chatbot 时代勉强可用，在 Agent 时代是危险的。原因有三：

1. **能力放大**：用户在 UI 里手动操作每次都受 UI 约束（按钮、表单、二次确认），而 Agent 拿着同样的 token 可以**在一次 session 中执行数百次操作**——一旦被 IDPI 劫持，爆炸半径远超人类点击。
2. **意图模糊**：用户在 UI 上点"导出报表"是清晰的；用户对 Agent 说"帮我整理数据"，Agent 自己决定要不要导出、导出什么、导出到哪里——**用户授权的是意图，不是具体动作**。
3. **追责模糊**：审计日志只记录"user_alice 调了 export API"，而真实情况是"Agent 在某个 IDPI 影响下代表 alice 调了 API"——这两件事在合规视角下截然不同。

> 一句话总结：**用户的身份令牌不应该原样下发给 Agent。Agent 必须有自己的 actor 身份，且要带着用户授权的最小子集。**

这就把权限设计从"给 Agent 一个 token"升级为四个独立问题：

* **谁是 actor**：Agent 的身份是什么？（→ 6.5 详解）
* **Agent 拿到了什么 scope**：以什么粒度授予？（本篇核心）
* **每个具体动作是否真的被允许**：Permission Gate（本篇核心）
* **什么时候需要人**：HITL 触发条件（本篇核心）

## 1. 第一性原理：Agent 时代的最小权限四原则

最小权限（Principle of Least Privilege, POLP）在 Agent 时代要补强成 4 条：

### 1.1 P1 — Capability Scoping（能力切片）

任何一个 Agent 实例不应同时拥有"读敏感数据 + 写外部状态 + 处理不可信输入"三件事的全部（**Rule of Two**，Meta/Anthropic 2025）。具体落地方式：

* 一个 chat Agent 只读，写操作交给另一个**审批 Agent** 链路
* 数据 Agent 不调用外部 API；调用 API 的 Agent 不读 PII
* 处理网页内容的子 Agent 不能直连数据库

### 1.2 P2 — Just-in-Time Scope Reduction（按需降权）

Agent 不应在 session 开始时就拿到所有 scope。**每个 sub-task 应当临时申请仅够此 task 的更窄 scope**。这是 OAuth 2.1 + Token Exchange (RFC 8693) + RAR (RFC 9396) 的最佳搭档。

### 1.3 P3 — Action-level Authorization（按动作授权）

不是"有 `payment` scope 就能转任意金额"。**每次具体的 `transfer(to=X, amount=Y)` 都要单独鉴权**——金额、对象、时机都进决策。RFC 9396 RAR 是当前最干净的标准方案。

### 1.4 P4 — Lifecycle Binding（绑定生命周期）

token 不绑定 wall-clock，而绑定 **任务状态**。任务结束 / 失败 / 取消 → token 立即失效。这条在 IETF 2026 *RAR Agent Extensions* 草案中被新增为 `lifecycle_binding` 字段。

> *Source*: [draft-chen-oauth-rar-agent-extensions-01, IETF 2026](https://datatracker.ietf.org/doc/draft-chen-oauth-rar-agent-extensions/)。

## 2. RBAC vs ABAC vs Capability vs RAR：四种模型在 Agent 中的取舍

```
                   粒度        动态决策   适合 Agent?
RBAC (角色)        粗（角色名）  ❌         入门级、不够
ABAC (属性)        中（条件）    ✅         主流、必备
Capability (令牌)  细（权能）    ✅✅       Agent-friendly
RAR (RFC 9396)    极细（动作+约束） ✅✅✅  2026 推荐
```

### 2.1 RBAC（Role-Based Access Control）

经典 web。Agent 场景的问题：**角色是静态的**，无法表达"金额 < $100"这种动态约束。**只用作 baseline**——比如 Agent 角色只能调 `read_*` 工具。

### 2.2 ABAC（Attribute-Based Access Control）

加入主体属性（user.dept、agent.tier）+ 资源属性（doc.classification）+ 环境属性（time、ip）+ 动作属性。**生产 Agent 最起码做到 ABAC**，否则连"工作时间内才能转账"都表达不了。

### 2.3 Capability-based

每次 Agent 创建时拿到一组 **不可伪造、可委派、最小化** 的 capability token。每个 capability 描述具体的"我能做什么"，而不是"我是谁"。这与 Agent 的多 sub-agent / 多 sub-task 拓扑天然契合（capability 可以被一个 Agent 切碎下发给子 Agent）。

### 2.4 OAuth 2.0 RAR（RFC 9396）

2023 年 RFC 化的 IETF 标准，2026 年开始成为 Agent 授权的事实选择。把 OAuth 的 `scope` 字段从扁平字符串扩展为**结构化 JSON**：

```json
"authorization_details": [
  {
    "type": "payment_initiation",
    "actions": ["initiate"],
    "locations": ["https://api.bank.example/payments"],
    "instructedAmount": { "currency": "USD", "amount": 50 },
    "creditorAccount": { "iban": "DE..." }
  },
  {
    "type": "account_information",
    "actions": ["read_balances", "read_transactions"],
    "locations": ["https://api.bank.example/accounts"]
  }
]
```

**和传统 scope 的差别**：

* `scope=payments` 是"能不能转"
* RAR 是"能不能转 50 美元 给这个账号 在这个 API"

对 Agent：

1. 表达 transaction-specific consent（不是模糊范围）
2. 通过 `locations` 实现 audience binding（防止 token 被滥用到其他 API）
3. 支持 enrichment（用户在 consent 阶段细化具体账户）
4. 给 RS 提供机器可读的策略材料（不只是 scope 字符串）

> *Source*: [RFC 9396 OAuth 2.0 Rich Authorization Requests](https://www.rfc-editor.org/info/rfc9396/)。

### 2.5 RAR Agent Extensions（IETF 2026 草案）

2026 年 4 月 China Mobile 提交的 IETF 草案 *draft-chen-oauth-rar-agent-extensions-01* 把 RAR 扩展为 Agent-aware：

| 新字段 | 作用 |
|---|---|
| `intent_request.intent` | 人类可读的意图描述（user goal） |
| `intent_request.constraints` | 机器可读的边界（budget、currency） |
| `policy_context.assurance_level` | 要求的策略等级（如 `financial_grade_v1`） |
| `policy_context.compliance_frameworks` | 合规框架（`gdpr`, `pci-dss`） |
| `lifecycle_binding.task_id` | 把 token 绑到任务 ID |
| `lifecycle_binding.termination_states` | 任务结束自动撤销 token |

**两阶段流**：先取 `intent token` → 用 RFC 8693 Token Exchange 换取 sub-task 的窄 token。AS 验证子任务请求是 intent 的"合法且安全的分解"。

新错误码：`policy_requirement_not_met`（策略不达标拒绝）。

这个设计直接回应了 §1 的 4 条原则。

> *Source*: [draft-chen-oauth-rar-agent-extensions-01](https://datatracker.ietf.org/doc/draft-chen-oauth-rar-agent-extensions/)。

## 3. Permission Gate：把"决策"从模型搬到确定性引擎

> "Pre-Action Authorization 把概率防御换成了确定性策略门。在 Vault CTF 上，permissive 策略下社工攻击成功率 74.6%；切到 OAP 受限策略后，**879 次攻击 0% 成功，$5,000 悬赏无人领取**。" — *Deterministic Pre-Action Authorization*, arXiv 2603.20953

### 3.1 Permission Gate 是什么

```
LLM ──(意图调用工具)──▶ ┌─────────────┐ ──ALLOW──▶ Tool 执行
                       │  Permission │ ──DENY───▶ 拒绝并写审计
                       │    Gate     │ ──ESCALATE▶ HITL 审批
                       └─────────────┘
                            │
                ┌───────────┴───────────┐
              Agent Passport         Policy Pack
              (Ed25519 签名身份)      (JSON/YAML 规则)
```

* **同步阻塞 hook**（不是异步事后扫描）
* **Fail-closed**：策略不可达 / passport 失效 → DENY
* **每次决策签名审计**（含输入、输出、规则路径）
* **运行在 framework 层**（LangChain / LangGraph / CrewAI / Letta 都有 adapter）

参考实现 *Open Agent Passport (OAP)*：

* Agent Passport：Ed25519 签名的能力凭证
* Policy Pack：声明式规则，仅支持比较/集合操作（保证可判定）
* 中位延迟 **53ms**（云 API，N=1000）
* Vault CTF 879 次攻击 0% 成功

### 3.2 风险分级 → 决策矩阵

每个工具应预先标注风险等级：

| 等级 | 描述 | 默认策略 |
|---|---|---|
| **R0** 只读、无副作用 | 查文档、查数据 | ALLOW（仅审计） |
| **R1** 写本地状态 | 写自己的 memory、起子任务 | ALLOW + rate limit |
| **R2** 写外部、可回滚 | 创建草稿、保存 issue | ALLOW + idempotent + audit |
| **R3** 写外部、低代价 | 发邮件、发消息 | ALLOW + 频率限制 + canary 检查 |
| **R4** 写外部、高代价 | 转账、生产数据库写 | **ESCALATE → HITL** |
| **R5** 不可逆 / 巨大代价 | 大额转账、大批量删除 | **ESCALATE + 多人 + 锁定时间** |

> 关键：**风险等级是工具静态属性**，不能让 LLM 自己声明。

### 3.3 HITL 触发条件（不只是金额）

工程经验上，下列任一命中都应触发 HITL：

* 工具属于 R4/R5
* 工具参数中含有 **tainted** 字段（来自外部 untrusted 源）
* 同 session 内同工具调用 > 阈值（如转账 > 3 次）
* 累计预算超阈（如 $1,000 / session）
* 偏离意图：动作类型不在 user intent 的合法分解集
* 模型置信度低 / classifier 高分
* 前一步发生过 deny / 异常

**HITL 必须展示原始 action 而非 LLM 摘要**——否则攻击者可以让 Agent 同时执行恶意操作并伪造良性总结骗审批人。

> *Source*: [Strata, Practicing the Human-in-the-Loop 2026](https://www.strata.io/blog/agentic-identity/practicing-the-human-in-the-loop/)。

## 4. Scoped Credentials：多层下发的工程实现

```
┌────────────────────────────────────────────────────────────┐
│ User Identity                                               │
│   - 主用户 OAuth token                                       │
│   - 包含完整 user grants                                     │
└──────────────────────┬──────────────────────────────────────┘
                       │ token exchange (RFC 8693)
                       ▼
┌────────────────────────────────────────────────────────────┐
│ Agent Identity (intent token)                               │
│   - actor=agent_xyz, on_behalf_of=user_alice                │
│   - intent="组织本周 sales 数据", budget=$0                  │
│   - lifecycle_binding=task_42                               │
└──────────────────────┬──────────────────────────────────────┘
                       │ token exchange per sub-task
        ┌──────────────┴──────────────┬───────────────┐
        ▼                              ▼                ▼
┌──────────────┐               ┌──────────────┐   ┌──────────────┐
│ DB read token │               │ Email send   │   │ Slack post   │
│ scope=sales/* │               │ rcpt=team@.. │   │ ch=#sales    │
│ exp=15m        │               │ exp=5m       │   │ exp=5m       │
└──────────────┘               └──────────────┘   └──────────────┘
```

**关键工程要求**：

1. **Sub-task token 的 TTL 极短**（5-15 分钟）
2. **绑定 sub-task ID + audience（locations）**
3. **拒绝传播 user 主 token 给子 Agent**——子 Agent 只能拿 derived
4. **被撤销时使用 token introspection** 实时验证（不能仅靠 `exp`）
5. **审计链**：每个 derived token 包含上一级 token id（`parent_jti`）

## 5. 生产蓝图：把上述零件组装起来

```python
from typing import Literal
from pydantic import BaseModel
from datetime import datetime, timedelta

# ─────────── Tool 元数据 ───────────
class ToolMeta(BaseModel):
    name: str
    risk_tier: Literal["R0", "R1", "R2", "R3", "R4", "R5"]
    idempotent: bool = False
    side_effects: bool = False
    requires_assurance: str = "default"   # 对应 RAR policy_context

# ─────────── 决策结果 ───────────
class GateDecision(BaseModel):
    action: Literal["ALLOW", "DENY", "ESCALATE"]
    reason: str
    policy_path: str
    audit_id: str

# ─────────── Permission Gate ───────────
class PermissionGate:
    def __init__(self, passport, policy_pack, taint, budget):
        self.passport = passport       # Ed25519-signed capability
        self.policy = policy_pack      # 声明式规则
        self.taint = taint             # 来自 6.1
        self.budget = budget           # session 预算计数器

    async def authorize(self, tool_meta: ToolMeta, args: dict, ctx: dict) -> GateDecision:
        # 1. passport 必须有效（fail-closed）
        if not self.passport.verify():
            return self._deny("invalid_passport", "passport.verify")

        # 2. tool 必须在 passport capability 集合中
        if tool_meta.name not in self.passport.capabilities:
            return self._deny("not_in_passport", "passport.cap")

        # 3. R4/R5 → ESCALATE
        if tool_meta.risk_tier in {"R4", "R5"}:
            return self._escalate("high_risk_tier", "tier.high")

        # 4. 任何 tainted 参数进入有副作用工具 → ESCALATE
        if tool_meta.side_effects and self.taint.any_tainted(args):
            return self._escalate("tainted_arg_in_side_effect", "taint.side_effect")

        # 5. 预算检查
        if not self.budget.check(tool_meta.name, args, ctx):
            return self._deny("budget_exceeded", "budget")

        # 6. 跑声明式 policy pack
        decision = self.policy.evaluate(tool_meta, args, ctx)
        return decision  # ALLOW / DENY / ESCALATE

    def _deny(self, reason, path):
        return GateDecision(action="DENY", reason=reason, policy_path=path,
                            audit_id=audit.log_deny(reason, path))

    def _escalate(self, reason, path):
        return GateDecision(action="ESCALATE", reason=reason, policy_path=path,
                            audit_id=audit.log_escalate(reason, path))


# ─────────── Token Exchange ───────────
async def derive_subtask_token(intent_token, action, location, ttl=timedelta(minutes=10)):
    return await oauth.token_exchange(
        subject_token=intent_token,
        requested_token_type="urn:ietf:params:oauth:token-type:access_token",
        actor_token=current_agent_credential(),
        authorization_details=[{
            "type": map_action_to_rar_type(action),
            "actions": [action],
            "locations": [location],
        }],
        # 2026 草案扩展
        lifecycle_binding={"task_id": current_task_id(), "termination_states":
                           ["COMPLETED", "FAILED", "CANCELLED"]},
        expires_in=int(ttl.total_seconds()),
    )


# ─────────── 完整 step ───────────
async def agent_step_with_authz(user_msg: str):
    intent_token = await mint_intent_token(
        user=current_user(), intent=user_msg, constraints={"budget_usd": 100}
    )
    while True:
        resp = await llm.invoke(user_msg, tools=TOOLS)
        if not resp.tool_calls:
            break
        for call in resp.tool_calls:
            meta = TOOL_META[call.name]
            decision = await gate.authorize(meta, call.args, ctx)
            if decision.action == "DENY":
                resp.add_tool_error(call, decision.reason)
                continue
            if decision.action == "ESCALATE":
                ok = await human_approve_with_raw_args(call)  # 必须看 raw
                if not ok:
                    continue
            # 此时换 narrow token
            sub_tok = await derive_subtask_token(intent_token, call.name, meta.location)
            raw = await dispatch(call, token=sub_tok)
            resp.add_tool_result(call, sanitize_tool_result(raw))
    return resp.content
```

要点：

1. Permission Gate 在 LLM 与工具之间**强制 fail-closed**
2. 每次工具调用都换一个 narrow token（RAR + Token Exchange）
3. R4/R5 一律 ESCALATE，不让 LLM 自己评估"是否需要审批"
4. 决策全签名入审计——用于 6.4 红队 / 9.4 决策溯源

## 6. 反模式

| 反模式 | 表现 | 修正 |
|---|---|---|
| **Agent 直接持用户 token** | 用户 OAuth token 原样下发 | Token Exchange，Agent 自己是 actor |
| **粗 scope** | `scope=email` 给所有邮件能力 | RAR + actions/locations |
| **静态 TTL** | token 1 小时不变 | lifecycle_binding 跟任务状态 |
| **无 risk tier** | 所有工具一视同仁 | R0-R5 + 决策矩阵 |
| **HITL 看摘要** | 审批人看 LLM 自己生成的描述 | 必须看 raw args |
| **gate 在 LLM 内部** | 让 LLM 判断"是否高危" | 必须确定性策略引擎 |
| **fail-open** | 策略服务挂了默认 ALLOW | fail-closed |
| **审计链断开** | 子 token 不绑父 token | parent_jti 链 |
| **single-actor 审批** | 单人审批生产级转账 | R5 多人 + cooling-off |
| **不撤销** | 任务完成 token 仍有效 | termination_states 自动撤 |

## 7. S0→S6 路线

| 阶段 | 目标 | 关键动作 |
|---|---|---|
| **S0** | RBAC baseline | Agent 角色固定，仅 R0 工具 |
| **S1** | ABAC 条件化 | 加入 user/dept/time 等条件 |
| **S2** | Token Exchange | Agent 不再持用户主 token |
| **S3** | RAR | scope 升级为 RAR + audience binding |
| **S4** | Permission Gate | 同步 fail-closed 策略门接入 framework |
| **S5** | RAR Agent Extensions | intent_token + lifecycle_binding 接入 |
| **S6** | Pre-Action Authz CTF | 自办 Vault 类红队，目标 ASR 0% |
