## 0. 真问题：为什么 95% 的"评估"其实是自欺

线下 Demo 跑通了，老板很满意，于是 PR 合入，Agent 上线。一周后客服群里炸开锅：客户说"我让它取消订单它给我下了一单"、"问它退款它扯了一堆条款最后没退"、"它说已发邮件其实没发"。回去看日志——LLM 的输出"看起来"全都对：礼貌、结构化、引用了条款、调用了工具。可惜每一步都"看起来对、实际错了"。

下面是 2026 年仍然普遍存在的几种伪评估：

| 伪评估模式 | 现象 | 真正在测什么 |
|---|---|---|
| **抽查式 Demo** | PM 手测 5 个 case，过了就上 | 抽 5 个 happy path，长尾 0 覆盖 |
| **Final-only 自动评估** | 只看最后那段话像不像答案 | 盲了所有过程错误（错误工具/错误参数/绕路） |
| **BLEU/ROUGE 文本相似** | 跟黄金答案算 n-gram 重合 | "措辞像不像"，跟"做对没"无关 |
| **单一 LLM-Judge 打分** | 用 GPT 打 1-5 分，平均当指标 | 受位置偏差/冗长偏差/自偏好严重污染 |
| **Benchmark 数字游戏** | "我们 SWE-bench 70%" | 公开榜可能数据泄露 + 任务分布跟你的业务无关 |
| **延迟/成本不算** | 只比正确率，忽略花了多少钱多少秒 | 正确但 30s/1$ 一次的 Agent 业务上不可用 |

业界 2026 年的关键转折数据：在代码开发任务上，**只看最终输出的 LLM-Judge** 与人类多数票的不一致率约 **31%**；而 **Agent-as-Judge**（看完整轨迹）的不一致率仅 **~0.3%**（[Zhuge et al., 2024, ICML 2025](https://proceedings.mlr.press/v267/zhuge25a.html)）。这意味着 final-only 评估在复杂任务上**几乎没有信号**。

第一性原理拆解评估问题：

```
评估的目的不是产生数字，而是回答两个问题：
  Q1: 这个版本上线，业务指标会涨还是会跌？  → 离线代理指标的预测能力
  Q2: 上次回归是哪个组件/哪步推理坏的？      → 可定位（component/trajectory level）

→ 评估必须分层（end-to-end + trajectory + component），且必须可校准（与人类标注对齐）
→ 评估器自身是"模型"，必须有自己的评估（meta-evaluation）
```

下文按 4 部分展开：评估金字塔（§1-3）、LLM-as-Judge 工程化（§4-5）、Agent-as-Judge 与轨迹评估（§6）、Benchmark 设计与陷阱（§7-8）、生产化路线图（§9）。

## 1. 评估金字塔：业务 / Task / Trajectory / Component 四层

把评估想象成倒置金字塔——上层粗、下层细，每一层回答不同的问题：

```
┌─────────────────────────────────────────────────────────┐
│ L0  业务指标          转化率 / NPS / 退款率 / 工单时长     │ ← 真北
├─────────────────────────────────────────────────────────┤
│ L1  Task Completion   端到端任务成功率（goal-aware judge） │ ← 离线代理
├─────────────────────────────────────────────────────────┤
│ L2  Trajectory        步骤效率 / 计划质量 / 计划遵循度     │ ← 轨迹诊断
├─────────────────────────────────────────────────────────┤
│ L3  Component         Tool 选择正确率 / 参数正确率 / RAG   │ ← 定位故障
└─────────────────────────────────────────────────────────┘
```

**关键工程认知**（[Confident AI, 2026](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide)）：

1. **L0 是真北，L1-L3 是代理指标**。每隔一段时间必须验证 L1/L2/L3 是否还能预测 L0——否则评估失效（goodhart's law：指标一被优化就不再是指标）。
2. **每层选不同评估器**：
   - L0 用 A/B 实验（见 [7.4](./7.4-持续改进闭环.md)）。
   - L1 用 LLM-as-Judge（goal-aware，无需黄金答案）。
   - L2 用 Agent-as-Judge 或专项 Trajectory Judge。
   - L3 用**确定性比对**（tool name / args schema），少量用 LLM-Judge（参数语义）。
3. **指标必须分级**：发版前必须过的硬门（如 tool 选择正确率 ≥95%）vs. 长期监控的软门（如冗余调用比 ≤10%）。

## 2. L1 任务完成度：goal-aware Judge 替代固定参考答案

**问题**：客服 Agent 用户说"帮我把上次那个订单退了吧"——黄金答案是什么？没有。同一个意图可以用 5 个工具组合实现。

**Task Completion Judge 的机制**（[Confident AI, 2026](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide)）：

```python
class TaskCompletionJudge:
    """
    输入: input(用户初始请求), trace(完整执行轨迹), final_output
    步骤:
      1. 从 input + trace 推断用户真实 goal（goal inference）
      2. 检查 final_output 是否实现了 goal
      3. 检查 trace 中是否产生了实现 goal 的"世界状态变化"
         （如真的发了邮件、真的写了 DB、真的扣了款）
    输出: score ∈ [0,1] + reason
    """
    def evaluate(self, case: TraceCase) -> JudgeResult:
        goal = self._infer_goal(case.input, case.trace[:3])  # 看头几步推 goal
        addressed = self._check_goal_addressed(goal, case.final_output)
        side_effects = self._check_side_effects(goal, case.trace)
        return JudgeResult(
            score=0.5 * addressed + 0.5 * side_effects,
            reason=f"goal={goal}; addressed={addressed}; side_effects={side_effects}"
        )
```

**关键设计**：
- **Goal-aware**：避免"答非所问但说得很好"被判通过。
- **副作用校验**：看 trace 里是否真的发生了"应当发生的事"——纯文本评估测不到 `send_email` 失败但模型说"已发"这种 hallucination。τ-bench 正是用这点把 syntax-only 评估甩开（[Sierra AI, 2024](https://sierra.ai/uk/blog/tau-bench-shaping-development-evaluation-agents)）。

**反例**："Agent 说我已成功为您取消订单。" → final-text-judge 通过；但 trace 里只调了 `search_order` 没调 `cancel_order` → goal-aware judge 失败。

## 3. L3 组件级：确定性 + 结构化 比纯 LLM-Judge 更准

工具调用是 Agent 性能的"地基"。地基不稳，上面什么都不可信。组件级评估必须**优先用确定性测**，能确定就别用 LLM。

```python
@dataclass
class ToolCallExpectation:
    name: str                      # 必须调用的 tool
    args_validator: Callable | None  # 参数 schema/语义校验函数
    order_constraint: str = "any"  # "any" | "before:other_tool" | "after:..."
    multiplicity: tuple[int,int] = (1, 1)  # (min, max) 调用次数

def score_tool_correctness(trace: list[ToolCall], expectations: list[ToolCallExpectation]) -> dict:
    """完全确定性，每次结果一致"""
    results = {"select": 0.0, "args": 0.0, "order": 0.0}
    matched = []
    for exp in expectations:
        cands = [c for c in trace if c.name == exp.name]
        if len(cands) < exp.multiplicity[0] or len(cands) > exp.multiplicity[1]:
            continue
        if exp.args_validator and not all(exp.args_validator(c.args) for c in cands):
            continue
        matched.append(exp)
    results["select"] = len(matched) / len(expectations)
    # ... 计算 args/order 维度
    return results
```

**为什么不用 LLM 判 tool 是否调对了？** 因为 schema 比对是封闭问题——`name == "cancel_order" and args["order_id"] == "ORD-123"` 要么真要么假。让 LLM 来判反而引入噪音、成本、不可复现。

**LLM-Judge 在组件级的合理用途**：
- 参数**语义**校验（如 `summary` 参数是否真概括了原文）。
- 工具输出与下游引用的一致性（faithfulness）。

## 4. LLM-as-Judge：从"打分"到"工程化评估器"

LLM-as-Judge 把"评估"从字符串相似度升级成"语义打分"，但若没有工程化处理，自身就是新的不确定来源。**判官必须被当成一个产品来设计**。

### 4.1 三种范式选型

| 范式 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| **Single-output Grading** | 线上监控、回归（无对照） | 便宜，1 次调用 | 绝对分数漂移 |
| **Pairwise Comparison** | A/B prompt/model 比较 | 相对判断稳 | 位置偏差严重 |
| **Reference-based Scoring** | 有黄金答案/上下文 | 最稳 | 需要标注 |

来源：[DeepEval, 2026](https://deepeval.com/blog/llm-as-a-judge)。

### 4.2 三种核心实现技术

**G-Eval**：自然语言定义评估维度，LLM 自动展开为 chain-of-thought 步骤，最后输出加权分。适合主观维度（如"礼貌度"、"专业感"）。**关键**：写明 `evaluation_steps` 比写 `criteria` 更稳定。

**DAG（Directed Acyclic Graph）**：把"是否合格"拆成决策树。每个节点是一个小判断（"输出是 valid JSON 吗？"→否，直接 0 分；是→进下一节点）。适合**有硬约束**的场景。

**QAG（Question-Answer Generation）**：把大判断拆成多个封闭式小问题（"输出是否包含订单号？"、"是否引用了正确条款？"），用 yes/no 答案聚合得分。`FaithfulnessMetric` / `AnswerRelevancyMetric` 的实现思路。

```python
# G-Eval 示例：评估"客服回复的合格度"
politeness_judge = GEval(
    name="Politeness",
    evaluation_steps=[
        "Step 1: Read the user's input and detect emotional state.",
        "Step 2: Check if the agent acknowledges the user's emotion.",
        "Step 3: Check if the agent's tone matches business context (formal for B2B, casual for D2C).",
        "Step 4: Score 0 if rude/dismissive, 0.5 if neutral, 1 if warm and aligned."
    ],
    evaluation_params=[INPUT, ACTUAL_OUTPUT],
    threshold=0.7,
)

# DAG 示例：硬约束 + 软评估混合
refund_correctness = DAG(
    root=Node("is_refund_action", children=[
        Node("if_yes", check="contains_action(trace, 'process_refund')",
             on_fail=Score(0, "未真正调用退款工具")),
        Node("amount_correct", check="trace.refund_amount == expected_amount",
             on_fail=Score(0, "退款金额错误")),
        Node("comm_quality", judge=GEval("Customer Communication"))  # 通过硬门后才进入软评估
    ])
)
```

### 4.3 五类核心偏差与缓解（必背）

研究综述见 [arXiv:2406.07791 - A Systematic Study of Position Bias in LLM-as-a-Judge, 2025 IJCNLP](https://aclanthology.org/2025.ijcnlp-long.18.pdf)。

| 偏差 | 现象 | 量化指标 | 缓解 |
|---|---|---|---|
| **位置偏差（Position）** | A/B 比较时偏好第一个或最后一个 | Swap Consistency（双向打分一致率，理想=1，实测 0.7-0.8） | 双向 swap 取一致；不一致计 tie |
| **冗长偏差（Verbosity）** | 长答案分更高 | 回归 score ~ quality + log(length)，β₂/β₁ 体现长度对分的相对影响（实测可达 0.4） | 校准集回归后剪掉长度贡献；要求"等长归一" |
| **自偏好（Self-preference）** | Judge 偏好与自己同家族的输出 | 多家族 Judge 一致率 | 多 Judge 投票（GPT-4 + Claude + Gemini） |
| **风格偏好（Familiar style）** | 偏好结构化 markdown / 礼貌套话 | 与 human label 偏离度 | 校准 prompt 强调"内容>形式"；用 reference-based |
| **噪声（Noise）** | 同一 case 重复调用得分波动 | 重复调用 std | temperature=0；多次取中位数 |

来源：[Brenndoerfer, 2026](https://mbrenndoerfer.com/writing/position-bias-in-llm-judges)、[Future AGI, 2026](https://futureagi.com/blog/llm-as-a-judge/)。

```python
def pairwise_with_swap(judge: LLMJudge, a: str, b: str) -> str:
    """位置偏差缓解：双向 swap 取一致"""
    v1 = judge.compare(first=a, second=b)
    v2 = judge.compare(first=b, second=a)  # 调换位置
    if v1 == "A" and v2 == "B":  # 都说 a 好
        return "A"
    if v1 == "B" and v2 == "A":
        return "B"
    return "TIE"  # 不一致 → 计为 tie（或送人工）

def length_calibrated_score(raw_score: float, length: int, beta_2: float, ref_length: int = 200) -> float:
    """冗长偏差缓解：基于校准集学的 β₂ 剪掉长度贡献"""
    import math
    return raw_score - beta_2 * (math.log(length) - math.log(ref_length))
```

## 5. Meta-Evaluation：判官自己也要被打分

**判官的可信度 = 与人类标注的一致率**。没有 meta-evaluation 的判官只是"听起来很自信的随机数生成器"。

### 5.1 三个必跟指标

| 指标 | 定义 | 目标值 |
|---|---|---|
| **Agreement with Human** | 在校准集上与人工标注的一致率（accuracy 或 Cohen's κ） | κ ≥ 0.6 才可用 |
| **False Positive Rate** | Judge pass 但人工 fail 的比例 | < 5%（创造虚假信心，最危险） |
| **False Negative Rate** | Judge fail 但人工 pass 的比例 | < 10%（创造噪声，但不会上线坏东西） |

```python
def meta_evaluate(judge: Judge, calibration_set: list[CalibrationCase]) -> MetaResult:
    judge_labels = [judge.evaluate(c).pass_or_fail for c in calibration_set]
    human_labels = [c.human_label for c in calibration_set]
    agreement = sum(j == h for j, h in zip(judge_labels, human_labels)) / len(calibration_set)
    fp = sum(j and not h for j, h in zip(judge_labels, human_labels)) / max(1, sum(judge_labels))
    fn = sum(not j and h for j, h in zip(judge_labels, human_labels)) / max(1, sum(human_labels))
    kappa = cohen_kappa(judge_labels, human_labels)
    return MetaResult(agreement=agreement, fp_rate=fp, fn_rate=fn, kappa=kappa)
```

### 5.2 校准集设计原则

- **来自真实流量**：从生产 trace 抽样，不要拍脑袋造。
- **覆盖失败模式**：故意包含已知错例（红队产物，见 [6.4](./6.4-RedTeaming合规审计与人机信任边界.md)）。
- **2 名以上标注 + 仲裁**：单标注主观性太大，至少二标注 + 一仲裁。
- **滚动维护**：每月新增 ~20% 新 case，逐步淘汰过时 case。
- **大小**：起步 100-200 case 即可有意义；500+ 可建立稳定基线。

## 6. Agent-as-Judge：当任务复杂到 final-only 失效

### 6.1 为什么需要

[Zhuge et al., 2024 (Meta) - Agent-as-a-Judge: Evaluate Agents with Agents](https://proceedings.mlr.press/v267/zhuge25a.html) 实证：在 DevAI（55 个真实开发任务）上，agent-judge 与人类 majority vote 的一致率达 **~99.7%**，而 final-only LLM-judge 仅 **~69%**。差距来自 LLM-judge 看不到中间步骤。

agent-judge 看的是**整段轨迹**——每一步用了什么工具、产出了什么、有没有偏离子目标——它能像一个 senior 工程师 code review 一样指出"第 7 步本来应该跑测试，他直接 commit 了"。

### 6.2 主流方法对照

| 方法 | 机制 | 适用 |
|---|---|---|
| **Agent-as-a-Judge**（[Zhuge 2024](https://proceedings.mlr.press/v267/zhuge25a.html)） | 一个 tool-using judge agent 沿轨迹检查每步 | 软件开发、复杂多步任务 |
| **ChatEval** | 多 persona 辩论投票 | 需要多视角的开放任务 |
| **DEBATE** | Scorer + Critic + Commander 三方对抗 | 防 hallucination |
| **CourtEval** | 法庭隐喻：Judge / Prosecutor / Defender | 高利害 / 决策类任务 |
| **MAJ-EVAL** | 自动生成多 stakeholder + 内部辩论 | 跨利益相关方场景（医疗/金融） |

来源：[A Survey on Agent-as-a-Judge, arXiv:2601.05111, 2026](https://arxiv.org/html/2601.05111v1)。

### 6.3 工程化判官 Agent 模板

```python
class TrajectoryJudge:
    """对每一步给出 verdict + 最终聚合"""
    def __init__(self, llm, requirements: list[Requirement]):
        self.llm = llm
        self.requirements = requirements  # 子目标列表

    def evaluate(self, trace: AgentTrace) -> TrajectoryVerdict:
        step_verdicts = []
        sub_goal_satisfied = {r.id: False for r in self.requirements}
        for i, step in enumerate(trace.steps):
            v = self._judge_step(step, context=trace.steps[:i], requirements=self.requirements)
            step_verdicts.append(v)
            for sat_id in v.satisfied_requirements:
                sub_goal_satisfied[sat_id] = True
        completion = sum(sub_goal_satisfied.values()) / len(self.requirements)
        efficiency = self._score_efficiency(trace)  # 看冗余调用、循环、绕路
        return TrajectoryVerdict(
            completion=completion,
            efficiency=efficiency,
            step_verdicts=step_verdicts,
            unsatisfied=[r for r, sat in sub_goal_satisfied.items() if not sat],
        )
```

**生产注意**：agent-judge 慢、贵、不稳定（每次评估几秒到几分钟）。**不要全量跑**——只在回归套件、A/B 离线评估时跑；线上监控用便宜的 LLM-Judge + 确定性指标即可。

## 7. Benchmark 设计：什么时候用公开榜，什么时候必须自己造

### 7.1 公开榜适合干什么

| Benchmark | 域 | 评估方式 | 当前 SOTA（2026） |
|---|---|---|---|
| [SWE-bench Verified](https://www.swebench.com/) | 真实 GitHub issue 修复 | 跑 patched repo 测试 | ~70%+ |
| [τ-bench / τ²-bench](https://sierra.ai/uk/blog/tau-bench-shaping-development-evaluation-agents) | 客服（零售/航司） | DB 终态 + pass^k 可靠性 | ~50-60% |
| [GAIA](https://huggingface.co/papers/2311.12983) | 通用 assistant | exact match | ~50%+ |
| [WebArena / Visual WebArena](https://webarena.dev) | 真实 web 任务 | URL/state match | ~40-50% |
| [OSWorld](https://os-world.github.io/) | 桌面操作系统 | 终态 match | ~30-40% |
| [MLE-Bench](https://github.com/openai/mle-bench) | Kaggle ML 比赛 | 比赛排名 | medal rate ~30% |
| AgentBench | OS/DB/web/games 多环境 | 任务定义 | 持续刷新 |

公开榜**适合**：
- 选模型时的横向对比基线。
- 验证某个新方法（如新 planning 算法）有效性。
- 学术发表。

公开榜**不适合**：
- 决定要不要上线——你的业务场景跟 benchmark 分布**几乎肯定不一样**。
- 长期监控——容易被刷分（数据泄露、prompt 工程过拟合）。

详见 [7.5 领域专项 Benchmark 深度解读](./7.5-领域专项Benchmark深度解读.md)。

### 7.2 自建 Benchmark 的"三个必须"

**MUST 1：从真实流量采样**。日志里抽 200-500 条 representative case，按"高频 happy path / 长尾 / 失败原因"三层分桶。

**MUST 2：执行级校验，不要只看文本**。借鉴 τ-bench——客户说"取消订单"，benchmark 要能查 DB 看订单状态确实变了。["A benchmark that only checks tool-call syntax or final text would pass agents that look right and do the wrong thing."](https://www.morphllm.com/ai-agent-evaluation)

**MUST 3：每条 case 有"成功的多种合法路径"**。Agent 不应被迫走某条具体路径，只要终态对、副作用对就算成功。

### 7.3 Benchmark 反模式

| 反模式 | 后果 | 修正 |
|---|---|---|
| **抄公开榜，几乎不改** | 模型早就见过，刷不出真实能力 | hold-out 自有数据 + canary 题 |
| **黄金答案是 prompt 直接复制** | 评估等于背书 | 用 goal-based judge，不要看字符串 |
| **没有失败 case 类目分布** | 不知道改进空间在哪里 | 每个 case 标 fail-mode 类目 |
| **跑一次就当真理** | 单次结果方差大 | n=3-5 取中位 + 报告 std |
| **不做难度分层** | 平均分掩盖长尾灾难 | 按 easy/medium/hard 分桶分别报告 |

---

## 8. 评估器自身的成本与延迟治理

LLM-Judge 跑全量 trace 在生产规模会**比 Agent 本身还贵**。常见三档优化：

```
档位 A（精准）：每次 PR 跑完整 500 case + agent-judge → 30 分钟，~$5
档位 B（日常）：CI 每次跑 100 cherry-picked case + LLM-judge → 3 分钟，~$0.5
档位 C（线上监控）：1% 流量打 sampled judge + 100% 确定性 metric → ~$0
```

工程做法：
- **小模型 first-pass + 大模型 second-pass**：用 GPT-4o-mini / Claude Haiku 过初筛，仅可疑/边界 case 升级到 GPT-4 / Claude Sonnet。
- **缓存**：同 trace 同 judge 同 version → cache 复用。
- **批处理**：对独立 case 用 batch API 拿 50% 折扣。
- **指标采样**：高频确定性指标 100% 采，重 LLM-Judge 1-5% 采样 + 异常报警时升采样。

## 9. 生产化路线图：S0 → S6

| 阶段 | 关键动作 | 退出标准 |
|---|---|---|
| **S0 现状盘点** | 列业务 KPI、当前 Agent 使用场景、已知 fail mode | 一份"评估目标 vs 现状"差距文档 |
| **S1 Component 自动化** | 给所有工具加 `tool_correctness` 单测；CI 阻断 | tool select/args 正确率 ≥95% 才能合并 |
| **S2 Task Completion 上线** | 写 goal-aware judge；100 case 校准集；CI 跑 | judge-human κ ≥0.6 |
| **S3 Trajectory + Agent-Judge** | Trajectory metric（efficiency/plan adherence）；离线跑 agent-judge | 回归套件覆盖 top-3 fail mode |
| **S4 Bias 治理** | swap consistency / length-calibration / multi-judge 部署 | judge 在 swap 测试 SC ≥0.85 |
| **S5 在线评估** | 1% 流量 sampled judge；告警接 oncall | 质量 drift 24h 内可发现 |
| **S6 飞轮闭环** | 失败 case 自动入校准集；每月 retrain judge prompt | 每月迭代 1 次校准集；月度 review |

**与红队对偶**（见 [6.4](./6.4-RedTeaming合规审计与人机信任边界.md)）：红队产生的 fail case 必须自动流入评估校准集——这是从攻击者视角持续给评估器加肌肉的唯一方法。

## 10. 反模式速查

| 反模式 | 为什么坏 | 替代 |
|---|---|---|
| 单一 LLM-Judge 一票定终身 | 偏差污染严重 | 多 judge + 与人类 κ 校准 |
| Final-only 评估复杂任务 | trace 错误全部漏检 | 至少加 trajectory 评估 |
| BLEU/ROUGE 跟黄金答案比 | 测的是措辞不是做对 | goal-aware LLM-Judge |
| 不算延迟/成本 | "正确但不可用" | 强制三件套：准确率 + p95 延迟 + 单次成本 |
| 公开榜分数当业务 KPI | 分布错位 + 数据泄露 | 自建 benchmark 反映业务场景 |
| Judge prompt 凭感觉调 | 漂移不可见 | meta-evaluation + 校准集版本化 |
| 一份打分代替分层 | 故障定位无从下手 | L1/L2/L3 各自独立分数 |

---

## 11. 一页总结

> **评估不是上线前的体检，是 Agent 的体重秤——每次迭代必须站一次。**

四层金字塔 L0/L1/L2/L3 把"业务"翻译成"工程信号"。L3 用确定性指标，L2/L1 用 LLM-as-Judge，复杂任务用 Agent-as-Judge 看完整轨迹。Judge 必须有自己的 meta-evaluation——与人类标注 κ ≥0.6 才可上岗，五类偏差（位置/冗长/自偏好/风格/噪声）每个都有量化指标和缓解手段。

Benchmark 选用原则：公开榜做横评不做线上决策；自建 benchmark 必须 (1) 真实流量采样 (2) 执行级校验 (3) 多合法路径。评估系统自身要有成本档位（A/B/C），高频用便宜的确定性 + 采样，低频用昂贵的 agent-judge 跑回归。最终目标是建立**红队 → 评估校准集 → judge 升级 → 回归门禁**的飞轮，让 Agent 每一次发版都能被量化地证明"变好了，没变坏"。
