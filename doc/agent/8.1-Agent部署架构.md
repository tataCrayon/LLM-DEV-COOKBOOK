# 8.1 Agent 部署架构：Stateless + External State / Queue / Serverless vs Long-running

> Module 8 开篇。前 7 个模块解决了"Agent 是什么、怎么写、怎么评"，本模块进入工程化深水区："怎么把它扔到生产环境里，让它稳定跑、不爆账、能扩缩、能恢复、能升级"。8.1 是地基：部署形态决定了 8.2-8.6 所有其他工程问题的求解空间。

---

## §0 真问题：Demo 跑得飞起，上线第一周翻车 5 次

把 Agent 从 notebook 搬到生产环境，新手最容易踩的坑：

1. **写成同步 HTTP 接口**：一次 Agent 任务平均 30 秒、长尾 5 分钟、最长 1 小时，Nginx/ALB 默认 60s 超时，CDN/WAF 链路再砍一半，用户看到 502；
2. **写成 Lambda**：Lambda 最大 15 分钟超时（AWS Lambda 2024 起允许至 15 min，更早只有 5 min），人审等待几小时直接爆掉；
3. **状态全放进程内**：副本一重启，所有跑到一半的 trajectory 全丢；K8s 滚动升级直接把昨晚跑了 10 小时的研究任务清零；
4. **没有解耦**：一波突发流量同时打 LLM Provider + 向量库 + 业务下游，全部限流报错，重试雪崩；
5. **同一份代码处理同步与异步**：会话级聊天（要 TTFT 200ms）和后台 deep research（要跑 30 分钟）跑在同一个 Pod 上，互相挤兑 CPU 与 token quota；
6. **升级即丢任务**：滚动发布把跑到一半的 workflow 杀掉，恢复后从头重跑，副作用（发邮件、扣款）被执行两次。

这些不是 LLM 问题，是 **分布式系统老问题在 Agent 时代被放大** —— Agent 任务**比传统请求长 100×、状态比函数大 10×、副作用代价比 RPC 高 1000×**（一次错误的 LLM 调用 = $0.5；一次错误的工具调用可能 = 退款失败/订单错发）。

第一性原理是 [Functions to Agents 的运行时变迁](https://streamnative.io/blog/from-functions-to-agents-what-changes-in-the-runtime)：

| 维度 | Functions（传统 serverless） | Agents（新范式） |
|---|---|---|
| 生命周期 | 毫秒-秒 | 分钟-小时-天 |
| 状态 | 外置 | 运行时托管的持久记忆 |
| I/O | 单次事件 | 流订阅 + 背压 |
| 通信 | 直接调用 | 事件驱动、松耦合 |
| 伸缩 | 无状态复制 | 分区 + 一致状态分片 |
| 治理 | 调用级监控 | 决策级审计 |

一句话：**Agent 是"会思考的微服务"，不是"会调用 LLM 的函数"**。部署架构必须从这个判断出发。

## §1 第一性原理：部署形态的 3 个根本选择

把所有 Agent 部署架构归约成 3 道选择题：

### 题 1：状态放哪里？（State location）

- **进程内（in-memory）**：最快，但实例死则状态死；只适合**单次请求级**的 ReAct loop；
- **外置存储（external state，DB / Redis / object store / vector store）**：副本可重启可水平扩；这是**生产默认**；
- **运行时托管（managed durable state，如 Restate / Temporal / DBOS / Cloud Run + Postgres）**：开发者按"内存变量"的语义写代码，运行时自动持久化每一步；这是 **2025-2026 的趋势**。

### 题 2：执行是同步还是异步？（Execution model）

- **同步 RPC**：客户端 hold 连接等结果，适合 <30s 的"问答-工具-答"；
- **流式（streaming SSE/WebSocket）**：连接保持但分块返回，适合"边想边说"，TTFT 优化的主战场；
- **异步任务（async with polling / callback / webhook）**：客户端立即拿到 task_id，后续轮询或被回调；适合 30s 以上任务；
- **事件驱动（event bus）**：完全解耦，Agent 是流上的常驻消费者；适合多 Agent / 后台 Agent / 实时反应。

### 题 3：计算放哪里？（Compute substrate）

- **Long-running container**（K8s Deployment / ECS Service / Fargate）：永远在线，状态可以内存优先（仍要外置兜底），适合 high-QPS、低延迟、有 GPU 的场景；
- **Serverless function**（Lambda / Cloud Run / Vercel / Cloudflare Workers）：scale-to-zero、按调用付费，适合突发、低频、低 RPS；
- **Durable serverless**（Restate / Temporal Cloud / DBOS / Inngest / Trigger.dev）：上面两者的"成年版"，serverless 弹性 + workflow 持久性。

这 3 道题 2×3×3 = 18 种组合，但**生产里真正常见的不超过 5 种**，下文逐一拆解。

## §2 五种主流部署模式（按场景反向倒推）

### 模式 A：同步 stateless + external state（聊天助手 / 工具代理）

```
Client ─HTTP/SSE─► API Gateway ─► Stateless App (K8s / Lambda)
                                    │
                                    ├─► LLM Provider
                                    ├─► Redis (conversation state, TTL)
                                    ├─► Vector DB
                                    └─► Tool APIs
```

- **状态**：会话历史压缩后写 Redis，key = session_id，每轮 read-then-update；
- **延迟**：流式 SSE 推回，TTFT 目标 <800ms；
- **扩缩**：HPA 按 RPS + LLM 等待中的并发数（不是 CPU！LLM 调用是 IO-bound，CPU 永远低）；
- **可用性**：任何副本可处理任何请求，单点死亡 = 用户重试一次；
- **适用**：对话产品、单轮多步工具代理、IDE 内 inline assistant。

**关键工程点**：
1. 把"上下文管理"从内存里剥到 Redis，是 **可水平扩容**的前提；
2. SSE / WebSocket 不要直挂到 LB 后的 Pod，加 sticky session 或用 push gateway（AWS API Gateway WebSocket、Cloudflare Durable Object）；
3. LLM Provider 调用是高延迟 IO，单 Pod 并发可拉到 100-500（gunicorn workers × asyncio），不要按 CPU 设并发上限。

### 模式 B：异步任务 + 工作队列（中等长度后台任务，30s-30min）

```
Client ─POST /tasks─► API ──► SQS / Redis Stream / Kafka
                              │
                              ▼
                       Worker Pool (K8s / Fargate)
                              │
                              ├─► State Store (Postgres)
                              ├─► LLM
                              └─► Tools

Client ◄─GET /tasks/{id}─ API ◄─ Postgres
```

- **状态**：任务表（task_id, status, input, output, trajectory, version, created_at），每步 trajectory 追加；
- **并发控制**：Worker concurrency = N，超出排队；细粒度限流交给队列；
- **可恢复**：worker 死亡 → SQS 消息 visibility timeout 到期 → 另一个 worker 接手；但要么任务幂等，要么需要 step-level checkpoint（见模式 D）；
- **适用**：研报生成、批量数据分析、文档处理、定时任务。

**关键工程点**：
1. **可见性超时（visibility timeout）必须 > 任务 p99 时长**，否则任务会被双重消费；超长任务（>15min）用 [SQS extend visibility](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/working-with-messages.html#changing-message-visibility-timeout) heartbeat，每 N 秒延长一次；
2. **DLQ + retry budget**：失败 3 次进死信，避免毒消息打爆 worker；
3. **任务版本号**：worker 启动时记录 git_sha，恢复时按 sha 路由，避免新版本 worker 接管旧版本任务（详见 §4）；
4. 队列选型：Kafka（高吞吐、有 partition key、可重放）/ SQS（运维零负担）/ Redis Stream（轻量、低延迟）/ NATS JetStream（边缘场景）。

### 模式 C：Serverless + durable execution（长任务、人审、跨日工作流）

这是 2025-2026 最大的变化。传统 serverless 处理长任务有三个糟糕选项（见 [Restate 博客](https://www.restate.dev/blog/resilient-serverless-agents)）：

1. 保函数运行 → 贵 + Lambda 15min 上限；
2. 自建 external state + queue → 简单 agent 变成分布式系统；
3. 放弃 serverless 改用 workflow orchestrator → 失去 scale-to-zero。

**Durable Execution** 给出第四条路：每步 LLM 调用、工具调用、人审等待都写入 journal（持久化日志），崩溃后从 journal 重放，已完成的步骤直接读结果**而不重新执行外部副作用**。

```python
# Restate Vercel AI SDK pattern (节选)
execute: async (claim) => {
  # 通知人审，副作用记录到 journal
  await restate.run(() => notifyHumanReviewer(claim, restate.key));
  # durable promise：函数挂起，等回调
  return await restate.promise<boolean>("approval");
}
```

执行流程：

1. 函数运行到 `restate.promise(...)` 时抛出"挂起异常"；
2. Restate 持久化整个状态后让函数实例销毁（Lambda 退出、Vercel 缩容到零）；
3. 几小时后审批 HTTP 回调到达；
4. Restate 重启函数（可能是不同实例），从 journal 重放，跳过已记录步骤，在挂起点恢复 promise；
5. 继续执行后续逻辑。

**关键产品**：
- [Restate](https://www.restate.dev)：开源、push-based、SDK 跨 TS/Python/Java；
- [Temporal](https://temporal.io)：成熟、pull-based worker，企业市场主流；
- [DBOS](https://www.dbos.dev)：Postgres 做 journal，[Cloud Run 上做 Agent](https://www.dbos.dev/blog/making-ai-agents-fault-tolerant-on-google-cloud-run) 几乎零运维；
- [Inngest](https://www.inngest.com) / [Trigger.dev](https://trigger.dev)：开发者友好的托管版；
- AWS Step Functions：成熟但 ASL 语法笨重；新版 SDK 集成已支持类代码写法；
- [AWS Bedrock AgentCore](https://aws.amazon.com/bedrock/agentcore/)（2025 re:Invent）：托管 Agent runtime，内置 memory、identity、observability。

**何时选 durable execution**：
- 任务跨越 15 min Lambda / 60 min Cloud Run 上限；
- 涉及人审 / 异步外部回调（支付、审批、邮件回复）；
- 工具调用昂贵且非幂等（不能容忍重放副作用）；
- 升级期间不希望丢任务（详见 §4）。

### 模式 D：Long-running stateful agent（Devin 类后台 Agent）

```
Browser ─WebSocket─► Gateway ─► Agent Process (Long-running container)
                                   │
                                   ├─► Sandbox (Docker / Firecracker / Modal)
                                   ├─► Memory store (vector + relational)
                                   ├─► Tool gateway
                                   └─► LLM stream
```

- **状态**：进程内（hot）+ Postgres 持久化（cold checkpoint，每 N 秒或每完成一步）；
- **生命周期**：分钟到天，Agent 进程长时间在线，对应一个用户的一个"工作空间"；
- **资源隔离**：每个 Agent 有独立的 sandbox（文件系统、shell、网络），通常 Firecracker microVM 或 Modal sandbox；
- **典型代表**：Devin (Cognition)、Cursor Background Agents、Replit Agent、Manus；

**关键工程点**：
1. **Agent 与 sandbox 解耦**：Agent 进程是策略，sandbox 是执行环境，sandbox 可独立缩放与回收；
2. **Snapshot & restore**：sandbox 应支持 fork（branching 探索）和 snapshot（任务暂停后恢复）；
3. **流式 UX**：thinking trace 实时推送到浏览器；详见 9.4；
4. **后台 + 通知**：用户关掉网页 Agent 继续跑，完成后通过邮件/IM/push 通知；详见 9.5；
5. **运行时是 stateful microservice**：用 K8s StatefulSet + PVC，或托管平台（Modal Sandbox、E2B、Daytona、Coderpad、CodeSandbox）。

### 模式 E：Event-driven multi-agent runtime（多 Agent 协作）

```
                  ┌─── Agent A ─┐
Event Bus ◄──┬───┤             │───► Event Bus
             │   ├─── Agent B ─┤
             │   ├─── Agent C ─┤
             │   └─── Agent D ─┘
             └─── Audit / Trace / Killer
```

- **总线**：Kafka / Pulsar / NATS / Redis Stream；
- **Agent**：消费某些 topic + 发布某些 topic 的有状态进程；
- **协调**：通过事件而非直接 RPC（解耦、可重放、可旁路审计）；
- **分片**：按 conversation_id / user_id partition key，把同一会话所有事件路由到同一 Agent 实例，保证状态一致；
- **典型**：[StreamNative blog](https://streamnative.io/blog/from-functions-to-agents-what-changes-in-the-runtime) 把 Pulsar 直接当 agent runtime，把状态做成 Pulsar table；微软 [Magentic-One](https://github.com/microsoft/autogen) / Salesforce Agentforce 多 Agent 编排。

适用：3.7 Multi-Agent 系统、需要边缘部署 / 多区域低延迟的实时反应类应用。

## §3 决策矩阵：怎么从场景选模式

| 场景 | 任务时长 | 状态量 | 是否需人审 | 推荐模式 |
|---|---|---|---|---|
| 客服对话 / IDE 助手 | <30s | 会话级 | 否 | A 同步 + Redis |
| 文档摘要 / SQL 生成 | 30s-5min | 任务级 | 否 | B 异步 + 队列 |
| 报销审批 / 合同 | 分钟-小时 | 工作流级 | 是 | C Durable execution |
| Deep research / SWE Agent | 分钟-天 | 大、有 sandbox | 否 | D Long-running stateful |
| 物联网 / 实时风控 / 多 Agent | 持续 | 流式 | 否 | E 事件驱动 |

**反模式速查**：

| 反模式 | 后果 |
|---|---|
| 用同步 HTTP 跑 5 分钟任务 | LB/CDN 超时，client 看到 502 |
| 用 Lambda 跑 30 分钟任务 | 15min 强杀，状态全丢 |
| 状态放进程内不外置 | 副本重启即丢数据，无法滚动升级 |
| 把同步对话与后台批跑混在同一 Pod | 资源互相挤兑，长任务把对话挤超时 |
| Queue 没设 DLQ | 毒消息 OOM-kill 所有 worker |
| 工具调用没幂等键 | 重放/重试导致重复扣款、重复发邮件 |
| 升级没固定版本路由 | 跑到一半的任务被新版代码"半路接管"，行为不一致 |
| 没有限流，所有任务并发打 LLM | Provider 429，整体雪崩 |

## §4 三大跨切关注点

### 4.1 幂等性 & 副作用控制（与 8.2 深度联动）

任何在 durable execution / 队列重试中可能被重放的步骤，都必须设计为**幂等**：

```python
async def call_payment_api(order_id: str, amount: float, *, idempotency_key: str):
    # 同一 idempotency_key 在 Provider 侧只生效一次（Stripe / Adyen 都支持）
    return await stripe.PaymentIntents.create(
        amount=amount * 100,
        currency="usd",
        metadata={"order_id": order_id},
        idempotency_key=idempotency_key,  # = f"{task_id}:{step_id}"
    )
```

**幂等键生成原则**：必须由 `task_id` + `step_id` 派生，让重试携带同样的 key；不能用 `uuid4()`，否则每次重试都成新调用。

### 4.2 版本路由：滚动升级的隐藏地雷

跑了一半的任务，**必须在它启动的代码版本上继续**，否则会出现：

- 步骤 1 用 v1 的 prompt 生成了一段中间结果；
- 升级到 v2，步骤 2 用 v2 的解析逻辑去解析 v1 的输出 → schema 不匹配；
- 或更糟：步骤 1 用 v1 的工具签名调了外部 API，步骤 2 用 v2 重放，参数对不上。

[Restate 的解决方案](https://www.restate.dev/blog/resilient-serverless-agents)：把部署 URL 做成不可变的，每次发版分配新 URL，老任务带 deployment_id 路由到老 URL；老版本平时缩容到零，被恢复请求唤醒。Temporal 用 [Worker Versioning](https://docs.temporal.io/workers#worker-versioning) 也是同一思路，DBOS 用 [Cloud Run revision pinning](https://www.dbos.dev/blog/making-ai-agents-fault-tolerant-on-google-cloud-run) 实现。

自己实现要点：
1. 任务表记录 `code_version`（git_sha）；
2. 调度器优先把恢复任务派给同 sha 的 worker；
3. 新版本灰度期间，老 sha 的 worker 池保留 1-2 副本接残余任务；
4. 配套 8.2 的 graceful shutdown：worker 收到 SIGTERM 后**不再消费新消息**，等手头任务完成或 checkpoint 后退出。

### 4.3 多租户 & 配额隔离

生产 Agent 平台必然多租户。每个租户的 LLM 调用、工具调用、sandbox CPU/Mem 都要独立计量与限流：

- **租户级 token budget**：网关在调 LLM 前预估 token cost，扣租户配额；超限快速失败；
- **租户级队列**：高优租户独立 queue 或独立 worker pool，避免吵闹邻居；
- **租户级 sandbox 配额**：Docker / Firecracker CPU/Mem 软硬限；K8s 用 ResourceQuota + LimitRange + NetworkPolicy。

## §5 选型决策树（一页可贴墙）

```
任务最长跑多久？
├── <30s
│    └── 是否需要持续对话？
│         ├── 是 → 模式 A：同步流式 + Redis session
│         └── 否 → 模式 A 退化版：同步 RPC（无状态）
├── 30s - 15min
│    └── 是否有人审 / 异步外部回调？
│         ├── 是 → 模式 C：Durable execution
│         └── 否 → 模式 B：队列 + worker
├── 15min - 1 小时
│    └── 是否需要 sandbox / 持续交互？
│         ├── 是 → 模式 D：Long-running stateful（Cloud Run / Fargate）
│         └── 否 → 模式 C：Durable（首选）or 模式 B + heartbeat extend
├── > 1 小时（含跨日）
│    └── 是否多 Agent 协作？
│         ├── 是 → 模式 E：Event-driven runtime
│         └── 否 → 模式 D / C 二选一，强烈建议 durable execution
└── 持续（无终点）
     └── 模式 E：事件驱动 + 分区状态
```

## §6 平台对比（2025-2026 当下）

| 平台 | 模式 | 长任务 | 持久状态 | Scale-to-zero | 适用 |
|---|---|---|---|---|---|
| AWS Lambda + Step Functions | C | ✅（SFN 1 年） | ✅ | ✅ | AWS 重客户 |
| AWS Bedrock AgentCore | A/D | ✅ | ✅ 内置 | ✅ | AWS 上的标准 Agent |
| Cloud Run + DBOS | C | ✅ | ✅ Postgres | ✅ | GCP 重客户、需 Postgres 兼容 |
| Vercel + Restate | C | ✅ | ✅ | ✅ | TS/Next.js |
| Temporal Cloud | C/D | ✅（无上限） | ✅ | ⚠️ pull worker | 大企业、强工作流语义 |
| Modal | D | ✅ | ✅ | ✅ | Python 生态、GPU agent |
| K8s + 自建 queue | B/D | ✅ | 自己搞 | ❌ | 重运维团队 |
| Cloudflare Workers + Durable Objects | A/D | 受限 | ✅ DO | ✅ | 全球边缘、低延迟 |

选型时把"团队能不能维护它"放在第一位，**不要为了"durable execution"上手就重构成 Temporal**，先用 SQS+Postgres 把模式 B 跑稳，跑到痛了再迁。

## §7 反模式

| 反模式 | 真实代价 |
|---|---|
| Agent 状态全塞 Pod 内存 | K8s 滚动升级一次，所有跑到一半的任务清零；用户投诉率飙升 |
| 同步 HTTP + 长任务 | 任务跑到 31s，CDN 切断，client 拿 502，业务下游已经被调用，订单状态不一致 |
| Lambda 跑 deep research | 15min 强杀，自己实现 checkpoint 还不如直接上 Temporal |
| 把对话和后台批跑放同一服务 | 后台任务把进程 LLM 并发占满，对话 TTFT 飙升到 8s |
| Queue 消息体放整段对话 | 单消息 50KB，Kafka 性能腰斩、SQS 超 256KB 限制；正确做法：消息只放 task_id，正文在 S3/Postgres |
| 不区分租户限流 | 一个大客户的批量任务，把所有人 LLM 配额吃光 |
| 没做 graceful shutdown | 滚动升级时 worker SIGKILL，任务卡在 in_progress，需要人工修数据 |
| 工具调用没 idempotency_key | 重试导致同一笔退款发两次，财务对账崩溃 |

---

## §8 S0 → S6 路线

| 阶段 | 状态 | 部署 | 痛点 |
|---|---|---|---|
| S0 | Demo | 笔记本 / 单机 | 一关机就没了 |
| S1 | 同步上线 | 模式 A，K8s + Redis | 长任务超时 |
| S2 | 加队列 | 模式 B，SQS + workers | 重试副作用问题 |
| S3 | 加幂等 + DLQ + heartbeat | 模式 B 强化 | 人审等待难做 |
| S4 | 引入 durable execution | 模式 C，Temporal/Restate/DBOS | 后台 Agent 体验需求 |
| S5 | 加 long-running stateful | 模式 D + sandbox | 多 Agent / 边缘场景 |
| S6 | 全面事件驱动 | 模式 E + 流平台 | 治理与可观测性升维 |

**S0-S3 是 90% 团队的全部生命周期**，不要过早跳到 S5。

---

## §9 一页总结

把 8.1 压成一句话：**Agent 部署的核心矛盾是"长任务 × 副作用 × 状态" vs "弹性 × 升级 × 多租户"，解法是把状态外置、用 durable execution 处理重试与人审、按任务时长选 5 种模式之一，永远不要把进程内当持久存储。**

承接关系：
- **8.2 可靠性**：把模式 B/C 的幂等、重试、熔断、降级讲透；
- **8.3 成本**：scale-to-zero、prompt cache、级联路由怎么省钱；
- **8.4 延迟**：模式 A 的 TTFT 极致优化；
- **8.5 模型选型**：自托管 vs Managed、开源 vs 闭源；
- **8.6 数据飞轮**：模式 B-E 产生的 trajectory 怎么变成 SFT/DPO 数据；
- **6.3 权限模型**：多租户的工具凭据与权限边界；
- **6.7 Kill Switch**：在所有 5 种模式下都要可一键停车；
- **7.3 可观测性**：每种模式下的 span 树结构。
