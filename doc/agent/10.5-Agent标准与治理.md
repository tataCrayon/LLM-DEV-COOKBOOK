## §0 真问题：2026 年的 Agent 治理，到底在治理什么？

把镜头拉到 2026 年中：

- **MCP** 月下载量 ~97M（Python+TS SDK 综合），Registry 接近 2,000 个服务器，被捐给 Linux Foundation 的 Agentic AI Foundation。
- **A2A** 一年内突破 **150+** 支持机构，22k+ stars，5 种 SDK，AP2 扩展拉来 60+ 金融机构。
- **EU AI Act** 的 GPAI 规则 **2025-08-02 生效**，**2026-08-02** 委员会执法权全面激活，**2027-08-02** 是存量模型最后大限，最高罚款 €15M 或全球年营收 **3%**。
- **W3C Agent Identity Registry Protocol CG** 2026-04-24 启动，目标做 DID + Verifiable Credential 一套 Agent 身份基建。
- **AgentDID** 学术原型在 Sepolia 上跑通：注册 ~$0.88，50v50 并发协议延迟仍 ~13.5s。

这些事件指向同一个判断：**Agent 时代的话语权正在被几个标准+法规框架快速圈定，从写一个 demo 到合规部署一个 Agent，门槛正在结构性抬升**。

> **判句**：未来 18 个月，企业里能跑生产的 Agent，必须能回答四个问题：你接的是哪条标准（MCP/A2A）？身份怎么签？合规归属哪条法规？审计证据怎么留？

## §1 第一性原理：Agent 治理为什么需要专属新标准？

人类社会用了几十年把人/企业的身份、合规、审计建得相对完备：身份证、护照、营业执照、KYC、AML、GDPR。把这套搬给 Agent，会立刻在四点上失灵：

1. **主体多层**：一次操作可能涉及"用户 → 个人 Agent → 第三方 Agent → 工具服务"四级代理，传统 OAuth scope 单维表达不了。
2. **可逆性丧失**：人类犯错可以撤销/调解，Agent 行动以毫秒级、跨域、不可逆方式发生（参考 9.3 与 10.4）。
3. **责任链断裂**：模型/Prompt/工具/数据各方不同，出事到底是谁的责任？
4. **数量与速度**：人类社会每天百万级法律事件，Agent 系统每秒百万级。监管不能逐案审查，必须靠**协议级**机制（签名/审计/可证伪）。

> **判句**：Agent 治理 = 协议（MCP/A2A）+ 身份（DID/VC）+ 法规（EU AI Act 等）+ 可审计基建。四者缺一不可。

## §2 协议层：MCP 与 A2A 的角色固化

### 2.1 MCP 在 2026 的成熟度

MCP 从 Anthropic 2024-11 提案，演化为 Linux Foundation 旗下中立标准（[WorkOS MCP 2026 综述](https://workos.com/blog/everything-your-team-needs-to-know-about-mcp-in-2026)、[MCP Roadmap](https://modelcontextprotocol.io/development/roadmap)）。三次关键修订：

| 时间 | 关键升级 |
|---|---|
| 2025-03 | OAuth 2.1 + DCR (Dynamic Client Registration) |
| 2025-06 | MCP server 正式形为 OAuth Resource Server；强制 RFC 8707 Resource Indicators 防 token 跨服务复用 |
| 2025-11 | 默认从 DCR 换成 **Client ID Metadata Documents (CIMD)**——客户端身份即一个 JSON URL；新增 Tasks/Sampling/Elicitation/Server-side Agent Loops/Extensions 系统 |
| 2026-01 | **MCP Apps**（与 OpenAI 联合）——MCP server 可嵌入沙盒 iframe 提供交互式 HTML UI |

> 重要变化：MCP 已经从"工具调用 RPC"扩展为含**异步 Task、user elicitation、UI 嵌入**的复合协议，与 A2A 的功能边界有部分重叠。

### 2.2 A2A 的明确分工

详见 [10.3](./10.3-A2A协议与Agent经济.md)。这里只强调一句标准化判断：

| 维度 | MCP | A2A |
|---|---|---|
| 治理方 | Linux Foundation / Agentic AI Foundation | Linux Foundation 独立项目 |
| 核心场景 | Agent ↔ 工具/数据 | Agent ↔ Agent（跨组织） |
| 信任 | 同信任域为主 | 跨组织默认 |
| 当前位置 | 接近事实标准（97M 下载/月） | 协议事实标准（150+ 机构、生产案例） |
| 与 OAuth | 已深度整合 | 多种 SecurityScheme 全覆盖 |

> **判句**：MCP 与 A2A 在 2026 中已经形成"双轨垄断"，新协议想插队几乎没可能；理性企业策略是 **早接、早影响 working group、早内化**。

### 2.3 MCP 治理细节（与 Agent 安全直接相关）

- **Session-scoped authorization**：放弃长期 token，朝短期、会话级、可撤销发展
- **Server Cards**（roadmap）：`.well-known` 暴露 server 元数据，做发现不需要连接
- **CIMD**：客户端身份去中心化，避开 DCR 的状态地狱
- **MCP Apps 安全模型**：iframe sandbox + 预声明模板 + JSON-RPC 审计 + 用户同意

这些设计反映了 MCP 团队一个一致的安全哲学：**最小持久权限 + 协议级审计 + 可撤销同意**。这些正是 6.3 / 6.6 中讨论的最小权限原则在协议层的体现。

## §3 身份层：从 OAuth 到 Agent Identity

### 3.1 传统 IAM 的天花板

OAuth 2.0 / SAML / FIDO2 是为**人 + 设备 + 平台**设计的（[AgentDID, arXiv:2604.25189, 2026](https://arxiv.org/html/2604.25189)）：
- 假设**长会话** + **集中权限管理**
- 主体是稳定的用户/服务
- 单点失败、横向扩展瓶颈

把它直接搬给 Agent 立即出现：
- Agent 数量爆炸，集中身份提供方延迟拉爆
- token 一签发就长期有效，丢失后不可见追溯
- 没法区分"同一 Agent 在不同任务中的不同权限"
- 没法验证"Agent 当前是否处于可执行的状态"

### 3.2 Agent Identity 的三层叠加（业界共识方向）

```
┌────────────────────────────────────────────────┐
│ Layer 1 · Organization Identity                │
│  - 谁是这个 Agent 的责任主体（公司/部门）       │
│  - 法人 KYC/营业执照层（与现有合规对接）        │
└────────────────────────────────────────────────┘
┌────────────────────────────────────────────────┐
│ Layer 2 · Agent Identity                        │
│  - 这个 Agent 的 DID + 版本 + 能力 VC           │
│  - 签名 Agent Card / AgentDID 模型              │
│  - 包含静态 (model/tools) + 动态 (state) 双属性 │
└────────────────────────────────────────────────┘
┌────────────────────────────────────────────────┐
│ Layer 3 · Session / Task Authority             │
│  - 当前会话的 scope + 时限 + 委托链              │
│  - AP2 Mandates / OAuth session-scoped token   │
│  - 任务级可审计                                  │
└────────────────────────────────────────────────┘
```

为什么必须三层？因为单一层级表达不了"Acme 公司的 ChatGPT v1.2 实例在 task_xyz 中代用户花最多 ¥800 买高铁票"这种复合授权关系。

### 3.3 标准化进展

| 组织 | 工作组 | 启动时间 | 目标 |
|---|---|---|---|
| W3C | [Agent Identity Registry Protocol CG](https://www.w3.org/community/agent-identity/) | 2026-04-24 | DID method + VC format + 跨组织信任协议 |
| OpenID Foundation | AIIM WG | 2025-Q4 | OIDC 扩展给 Agent 用 |
| IETF | WIMSE WG | 进行中 | Workload Identity 标准 |
| NIST | AI Agent Identity 工作 | 2025-2026 | 政府层面框架 |

**学术原型**：AgentDID（[arXiv:2604.25189](https://arxiv.org/html/2604.25189)）已完整实现 W3C DID + VC，每个 DID 文档双密钥对（admin + operational）+ capabilityInvocation 隔离。**关键创新**：State Verification——Readiness Probe + Context Consistency Hash（SHA-256 canonical session 对比）。

> **判句**：未来 12-18 个月，"Agent 身份能不能跨组织对账"会成为企业 Agent 部署的 gating issue。早期投入接 DID/VC 的团队，未来三五年节省海量集成成本。

## §4 法规层：EU AI Act 与 GPAI Code of Practice

### 4.1 时间线（背下来）

| 日期 | 事件 |
|---|---|
| **2024-08-01** | EU AI Act 正式生效 |
| **2025-02-02** | 禁止性 AI 系统条款生效（社会评分等） |
| **2025-07-10** | GPAI Code of Practice 终版发布 |
| **2025-08-02** | GPAI 规则法律适用；新发布模型须合规（CoP 签署者享首年协作期） |
| **2026-08-02** | 委员会执法权全面激活：审计、调查、罚款 |
| **2027-08-02** | 存量模型（2025-08-02 前上市）的硬截止 |

来源：[Eyreact GPAI 2026 update](https://eyreact.com/eu-ai-act-a-quick-guide-to-the-gpai-code-of-practice-2026-update/)、[European Commission GPAI Code](https://digital-strategy.ec.europa.eu/en/policies/contents-code-gpai)。

### 4.2 GPAI Code of Practice 三大支柱

**1. Transparency（所有 GPAI 提供方）**
- 标准化 Model Documentation Form：架构、训练数据、计算、能耗
- 对下游开发者 14 天窗口提供技术细节
- 文档保留 10 年
- 商业秘密可对外脱敏，但监管在受控环境下可查

**2. Copyright（所有 GPAI）**
- **董事会级别**书面版权合规政策
- 尊重 `robots.txt` 等机器可读 opt-out
- 不爬黑名单盗版站
- 输出端反侵权技术措施 + 权利人投诉渠道

**3. Safety & Security（系统性风险模型，>10²⁵ FLOPs 训练）**
- 完整风险管理框架
- SOTA 评估 + **独立外部 red-teaming**
- 严重事件**直报**欧洲 AI Office + 改进措施

**罚款**：最高 **€15M 或全球年营收 3%**，按高者。

### 4.3 对 Agent 系统的具体影响

虽然 GPAI CoP 主要约束模型提供方，但**部署 Agent 的企业（deployer）**通过下面三条间接被牵动：

1. **依赖模型提供方文档**：你必须能解释你 Agent 用的是哪个模型、哪个版本、有何 evaluation 结果（接 8.2 / 8.3）
2. **Agent 自动操作的版权风险**：Agent 浏览/抓取/生成内容会触发新的版权义务（接 10.4 + 6.5）
3. **Serious incident 上报链**：Agent 致害事件可能逆向追溯到模型提供方（系统性风险），形成跨方责任链

欧盟委员会另有针对 AI Agent 的**初步指引**（separate preliminary guidance），细化"自主浏览/决策/调用 API"的具体义务，预计 2026 H2 - 2027 完整定型。

### 4.4 其他法域速览

| 法域 | 关键文件 | 时间窗 |
|---|---|---|
| 中国 | 《生成式 AI 服务管理暂行办法》《算法推荐管理规定》《深度合成规定》 | 已实施 |
| 美国 | EO 14110 之后各州自定（CA SB-1047 已否决；NY/CO 等推进） | 滚动 |
| 英国 | AI Safety Institute + pro-innovation 框架 | 滚动 |
| OECD/ISO | ISO/IEC 42001 AI Management System | 2023 发布，2026 持续推广 |

> **架构判断**：欧盟是**标准制定者**，中国是**最先严格落地的市场**，美国是**主战场的法规真空**。跨国 Agent 企业必须三地分别合规，不要假设"一处合规处处通行"。

## §5 治理工程化：把法规与标准变成代码

法规与标准最后必须落到代码与流程。下面是一个"Agent 治理 Stack"参考蓝图：

```
┌────────────────────────────────────────────────────┐
│  Layer 5 · Audit & Reporting                       │
│    - 不可篡改审计日志 (append-only)                │
│    - Serious incident 直报渠道                      │
│    - 数据保留 10 年（GPAI 要求）                    │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│  Layer 4 · Compliance Gate                         │
│    - 数据来源版权检查                                │
│    - 模型 evaluation 文档拉取（CoP Model Doc Form） │
│    - 内容输出风险评估（接 6.2）                     │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│  Layer 3 · Identity & Authorization                 │
│    - Agent DID + VC 验证                            │
│    - AP2 Mandates / OAuth session-scoped           │
│    - 三层身份（Org / Agent / Session）              │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│  Layer 2 · Protocol                                 │
│    - MCP 工具调用                                   │
│    - A2A 跨组织协作                                  │
│    - x402 / MPP / AP2 支付                          │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│  Layer 1 · Runtime                                  │
│    - Sandbox (VM/Container/iframe)                  │
│    - Guardrail (Input/Output/Tool 三道)             │
│    - Kill Switch + Rate Limit                       │
└────────────────────────────────────────────────────┘
```

### 5.1 不可篡改审计日志（关键基础设施）

每一次 Agent 行动必须记录：
- `timestamp` (RFC 3339, 单调时钟兜底)
- `agent_did` + `session_id` + `task_id`
- `caller_chain`（principal → agent A → agent B）
- `action_type`（LLM call / tool call / state change）
- `input_hash` + `output_hash`（PII redaction 后）
- `policy_decisions`（哪些 guard 触发，何决策）
- `signature`（agent operational key）

存储采用 append-only（如 AWS QLDB / Tessera / 区块链 anchor），定期 Merkle root 公证。

### 5.2 模型 Documentation 自动拉取

```python
class ModelDocumentationRegistry:
    """对每个使用的 GPAI 模型，拉取并缓存 Model Doc Form."""
    def get(self, model_id: str) -> ModelDoc:
        doc = self.cache.get(model_id)
        if doc is None or doc.is_stale():
            doc = self.provider.fetch_doc_form(model_id)
            self.cache.put(model_id, doc)
        return doc

    def assert_compliance(self, model_id: str):
        doc = self.get(model_id)
        assert doc.transparency_complete, f"{model_id} missing transparency"
        assert doc.copyright_policy_url, f"{model_id} missing copyright policy"
        if doc.systemic_risk:
            assert doc.last_red_team_report.is_recent(months=6)
```

### 5.3 跨域信任协商（A2A + DID）

```python
def negotiate_trust(local_agent, remote_agent_card_url):
    # 1. 拉取并验证 Signed Agent Card
    card = fetch_and_verify_signature(remote_agent_card_url)

    # 2. DID resolve + VC chain validation
    did_doc = resolve_did(card.agent_did)
    vcs = fetch_credentials(did_doc)
    assert verify_chain_of_trust(vcs, trusted_issuers=POLICY.issuers)

    # 3. State verification（readiness probe）
    probe_result = ready_probe(remote_agent, timeout=COMPUTE_DYNAMIC_TIMEOUT())
    assert probe_result.fresh and probe_result.signature_valid

    # 4. 检查授权范围是否覆盖意图
    intent = local_agent.current_intent
    assert intent.is_within_scope(card.declared_scopes)

    return TrustEstablished(card, vcs)
```

## §6 反模式

| 反模式 | 后果 | 替代 |
|---|---|---|
| 自研 Agent 协议，不接 MCP/A2A | 长期被生态孤立、维护成本高 | 直接用上游 SDK，差异化做扩展 |
| 把 OAuth 当 Agent 身份完整方案 | scope 表达不全、token 长期化 | 三层身份 + DID/VC + session-scoped |
| GPAI 合规等到 2026-08-02 再做 | 罚款 €15M / 3% 年营收 | 现在就建 Model Doc + 审计日志 |
| 跨域调用不验证 Signed Agent Card | 被钓鱼/伪造 Agent | 强制签名 + DID resolve + state probe |
| 审计日志可写可删 | 出事拿不出证据 | append-only + 周期 Merkle 公证 |
| 全球合规当作一处合规 | 中国/欧盟/美国各自踩坑 | 三地分别 review + 本地化 deployment |
| 把"上线"等同于"合规完毕" | serious incident 不上报招致重罚 | 上报渠道 + 7×24 监控 + 跨方协同 |
| 不更新 MCP/A2A SDK | 错过 CIMD / session-scope / state verification | 季度级 upgrade plan |

## §7 S0 → S5 落地路线图

### S0：现状（多数企业）
- 用 MCP/A2A 但版本陈旧
- 身份靠 OAuth + 静态 API Key
- 合规只看本地法规
- 审计日志可改

### S1：协议升级（1-2 个月）
- MCP 升到 2025-11 后版本（CIMD / Tasks / Elicitation）
- A2A 升到 v1.0
- 选定 deployment region 对应法规清单

### S2：身份三层（3-6 个月）
- 引入 Signed Agent Card
- 试点 AgentDID 给关键 Agent（金融/医疗/合规相关）
- session-scoped token 全量替换长期 token

### S3：合规自动化（6-9 个月）
- Model Doc Form 自动拉取与校验
- 不可篡改审计日志（append-only + Merkle）
- 内部 GPAI 合规 dashboard

### S4：跨组织生态（9-12 个月）
- 与上下游合作伙伴的 A2A 互联 + 信任协商
- AP2 Mandates 给高金额操作
- serious incident 上报渠道演练

### S5：参与标准（12+ 个月）
- 加入 W3C Agent Identity CG / Linux Foundation MCP/A2A WG
- 把自研扩展贡献回上游
- 在 EU AI Act 配套指引上提交反馈

## §8 一页总结

**判句**：未来 18 个月，Agent 系统的护城河不在模型，而在你接没接对协议、签没签好身份、留没留住审计。

**三件事必须记住**：
1. **MCP（工具）+ A2A（Agent 间）+ AP2/x402/MPP（结算）+ DID/VC（身份）+ EU AI Act（合规）** 是 2026 之后必修的五张牌。
2. **三层身份**（Organization / Agent / Session）是表达"复合代理"的最简结构，少一层都会留漏洞。
3. **EU 时间线**：2026-08-02 委员会执法、2027-08-02 存量模型大限。现在就启动，不要押到截止前。

**做对的事**：
- 早接 MCP 2025-11+ / A2A v1.0，季度级 upgrade
- 投资 DID/VC 实验，跟住 W3C Agent Identity 进展
- 把 Model Doc + 审计日志做成 day-1 基建
- 三地分别合规，不要混淆

**警惕**：
- "我们用 OAuth 就够了" 的幻觉
- 把合规等同于"加一个 disclaimer"
- 自研协议替代生态标准
- 审计日志可改
