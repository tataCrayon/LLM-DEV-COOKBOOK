# Tool Schema 设计原则与工具生态治理

> 一个 Agent 的真实能力边界不取决于模型多聪明，而取决于它手里有什么工具、这些工具被描述得有多好。Tool Schema 就是模型"看到"工具的全部——名字、描述、参数结构。一个坏的 schema 能让 GPT-4o 级别的模型反复调错工具；一个好的 schema 能让小模型精准使用复杂 API。

## 0. 从一个真实的调用失败说起

某 Agent 接入了 20 个内部 API 工具，其中有 `create_ticket` 和 `create_incident`。两者的 description 分别是"创建工单"和"创建事件"——对于模型来说，几乎没有区分度。结果：模型在 35% 的紧急场景中错误地调用了 `create_ticket` 而非 `create_incident`，因为它无法从 description 中理解两者的语义边界。修复方法不是换更大的模型，而是改写 description：

```diff
- "Create a ticket"
+ "Create a non-urgent support ticket for internal IT requests 
+  (e.g., laptop replacement, VPN access). For production incidents 
+  with SLA impact, use create_incident instead."
```

修改后，错误率从 35% 降至 3%。这就是 Tool Schema 设计的杠杆效应。

## 1. 第一性原理：为什么 Schema 设计如此关键

### 1.1 模型如何"看到"工具

当你把工具列表传给模型时，模型收到的是一段结构化文本——`name`、`description`、`parameters`（JSON Schema）。模型基于这段文本做出三个决策：

1. **是否需要调用工具**（vs 直接用参数知识回答）
2. **调用哪个工具**（从候选集中选择）
3. **构造什么参数**（填充 JSON 对象）

每个决策点都是一次 NLU（自然语言理解）操作——模型在用语言能力理解你的"API 文档"。如果文档写得模糊、冗余、或缺乏边界说明，模型的决策就会退化。

### 1.2 经验数据支撑

Anthropic 在 SWE-bench Verified 上的实验表明，仅对工具 description 进行措辞优化（不改任何模型参数或代码），就能带来显著的任务完成率提升。arXiv 2605.00737 的框架化研究进一步揭示：模型对工具的"感知需要度（perceived need）"与"真实需要度（true need）"存在系统性偏差——这个偏差的主要来源不是模型能力，而是 schema 的表达质量。

## 2. 工具命名：第一道信号

### 2.1 命名原则

| 原则 | 正例 | 反例 | 说明 |
|------|------|------|------|
| 动词_名词 格式 | `search_contacts` | `contacts` | 明确操作意图 |
| 命名空间分层 | `jira_issues_search` | `search` | 多工具环境下防冲突 |
| 精确语义 | `get_invoice_by_id` | `get_data` | 减少模型猜测 |
| 区分粒度 | `list_recent_orders` | `query_database` | 窄工具优于宽工具 |

### 2.2 命名空间策略

当 Agent 接入多个服务（Slack、Jira、GitHub、Database），命名空间是第一道区分信号：

```
# 推荐：服务_资源_操作
slack_channels_list
slack_messages_search
jira_issues_create
jira_issues_transition
github_pr_create
github_pr_review

# 不推荐：纯操作名
list_channels
search_messages
create_issue
```

Anthropic 的研究指出，前缀式（`slack_search`）与后缀式（`search_slack`）命名对模型的工具选择准确率有"非平凡的差异"——建议 A/B 测试确定哪种更适合你的工具集。

## 3. Description 撰写：最被低估的 Prompt Engineering

### 3.1 核心原则：写给"新入职同事"

Anthropic 的建议是：把 description 写得像在给一个聪明但没有领域背景的新同事做入职讲解。需要显式化的内容包括：

- 这个工具**做什么**（核心功能）
- **什么时候该用**（正面触发条件）
- **什么时候不该用**（负面边界，指向替代工具）
- **查询/参数格式**的特殊要求
- **返回结果**的结构概要

### 3.2 Bad → Good 对比

```json
// ❌ Bad: 模型无法区分 search_logs 和 get_logs 的语义
{
  "name": "search_logs",
  "description": "Search system logs"
}

// ✅ Good: 明确功能、边界、格式要求
{
  "name": "search_logs",
  "description": "Search application logs by time range and keyword. Returns matching log lines with ±3 lines of context. Use for debugging specific errors or tracing request flows. For real-time log streaming, use tail_logs instead. Query supports Lucene syntax (e.g., 'level:ERROR AND service:payment'). Maximum 1000 results per call; use pagination token for more."
}
```

### 3.3 关键陷阱：Description 中的隐性偏差

Anthropic 团队发现，Claude 的 web search 工具曾在 description 中暗示"当前年份是 2025"，导致模型在所有搜索查询中自动附加"2025"——这不是模型的 bug，而是 description 设计的 bug。工具描述中的任何暗示都会被模型当作行为指导。

## 4. 参数设计（Parameter Schema）

### 4.1 类型系统是你最好的朋友

JSON Schema 提供了丰富的约束能力，充分使用它们：

```json
{
  "name": "create_event",
  "parameters": {
    "type": "object",
    "properties": {
      "title": {
        "type": "string",
        "description": "Event title, 5-100 characters",
        "minLength": 5,
        "maxLength": 100
      },
      "start_time": {
        "type": "string",
        "format": "date-time",
        "description": "ISO 8601 format, e.g., 2025-03-15T14:00:00Z"
      },
      "priority": {
        "type": "string",
        "enum": ["low", "medium", "high", "critical"],
        "description": "Event priority. Use 'critical' only for production-impacting incidents."
      },
      "attendees": {
        "type": "array",
        "items": { "type": "string", "format": "email" },
        "description": "List of attendee email addresses",
        "maxItems": 50
      }
    },
    "required": ["title", "start_time"]
  }
}
```

### 4.2 参数命名规范

| 规范 | 正例 | 反例 | 原因 |
|------|------|------|------|
| 语义完整 | `user_id` | `id` / `uid` | 消除歧义 |
| 避免缩写 | `start_time` | `st` | 模型需要理解含义 |
| 明确单位 | `timeout_seconds` | `timeout` | 防止单位混淆 |
| 自然语言标识 | `customer_name` | `cust_uuid_hex` | 模型擅长处理自然语言，不擅长构造 UUID |

### 4.3 Tool Use Examples：解决 Schema 无法表达的隐性约定

JSON Schema 能约束类型和范围，但无法表达"这些参数在什么场景下配合使用"。Anthropic 的 Tool Use Examples 特性（2025-11）填补了这个空白：

```json
{
  "name": "create_ticket",
  "input_examples": [
    {
      "description": "Simple internal IT request",
      "input": {
        "title": "VPN access for new hire",
        "priority": "low",
        "category": "access-request"
      }
    },
    {
      "description": "Urgent bug report with full context",
      "input": {
        "title": "Payment service timeout in checkout flow",
        "priority": "critical",
        "category": "bug",
        "affected_service": "payment-gateway",
        "sla_hours": 4,
        "escalation_team": "payments-oncall"
      }
    }
  ]
}
```

关键效果：Anthropic 报告，加入 examples 后复杂参数处理准确率从 72% 提升到 90%。

建议：每个工具 1-5 个示例，覆盖最小参数集和完整参数集两种场景，聚焦 schema 歧义点。

## 5. 工具粒度设计：宽工具 vs 窄工具

这是最常见的设计抉择。

### 5.1 宽工具的问题

```json
// ❌ 过宽：模型需要构造 SQL，容易幻觉
{
  "name": "query_database",
  "description": "Execute any SQL query",
  "parameters": { "sql": { "type": "string" } }
}
```

问题：模型不了解表结构、会幻觉列名、可能生成破坏性 SQL。

### 5.2 窄工具的优势

```json
// ✅ 窄且明确：模型只需要提供 invoice_id
{
  "name": "get_invoice_by_id",
  "description": "Retrieve a specific invoice with line items and payment status",
  "parameters": {
    "invoice_id": {
      "type": "string",
      "pattern": "^INV-[0-9]{6}$",
      "description": "Invoice ID in format INV-XXXXXX"
    }
  }
}
```

### 5.3 合并的时机

当用户完成一个意图需要调用 3+ 个 API 时，考虑合并为一个意图级工具：

```
# 原始（3次调用）
search_user(name="jane") → get_user_calendar(id=123) → create_event(calendar_id=456, ...)

# 合并（1次调用）
schedule_meeting(participant_name="jane", title="...", time="...")
```

Anthropic 的建议：将高频链式调用合并为单一工具，保留底层工具给需要细粒度控制的场景。

### 5.4 工具数量上限

MLflow 研究指出：**单个 Agent 超过 8 个工具就是设计问题**的信号。如果工具超过这个数量，应该考虑：
- 拆分为多个 sub-agent，各司其职
- 使用 Tool Search（动态加载）而非全量注入
- 合并语义重叠的工具

## 6. Token 经济学：工具定义的隐性成本

### 6.1 成本模型

每个工具的 schema 定义大约占 200-500 tokens。50 个工具全量加载意味着 ~72K tokens 的上下文预算被工具占据——这还没开始工作。这直接影响：
- 首次响应延迟（更多 prefill）
- 每次请求的美元成本
- 可用于推理的剩余上下文

### 6.2 Anthropic Tool Search 的解决方案

Tool Search Tool 将工具分为两类：
- **Always-loaded**（`defer_loading: false`）：3-5 个高频核心工具
- **Deferred**（`defer_loading: true`）：其余工具按需搜索加载

效果数据：
- 传统方案：50+ 工具 → ~77K tokens 启动开销
- Tool Search：~500 tokens 启动 + 按需 ~3K → 总 ~8.7K tokens
- **85% 的 token 节省**，同时工具选择准确率从 49% 提升到 74%（Opus 4）

### 6.3 响应格式控制

在工具参数中暴露 `response_format` enum（如 `DETAILED` / `CONCISE`），让模型自主选择所需详细度。Concise 模式可节省约三分之二的返回 token。

### 6.4 Programmatic Tool Calling（PTC）

对于需要聚合大量数据的场景，让模型写 Python 代码批量调用工具，只返回聚合结果：

```
# 传统方式（20次推理 + 20次工具调用 + 200KB 上下文）
for each employee:
  get_team → get_expenses → get_budget → compare

# PTC 方式（1次推理 + 1段 Python + 1KB 结果）
async def analyze():
    results = await asyncio.gather(*[check_employee(e) for e in employees])
    return [r for r in results if r.exceeded_budget]
```

Anthropic 数据：token 用量从 43,588 降至 27,297（37% 减少），同时消除了 19+ 次多余的推理 pass。

## 7. 错误处理设计

### 7.1 错误是教学机会

工具的错误返回不是给人看的——是给模型看的。模型需要从错误中理解：
- 到底什么错了
- 应该如何修正
- 有没有替代方案

```json
// ❌ Bad: 模型无法从中学到任何东西
{ "error": "400 Bad Request" }

// ❌ Bad: 原始 traceback 浪费 token 且无指导价值
{ "error": "Traceback (most recent call last):\n  File..." }

// ✅ Good: 结构化、可操作
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "Query failed: column 'user_name' not found in table 'users'. Available columns: id, email, full_name, created_at. Did you mean 'full_name'? Tip: call describe_table('users') first to see the full schema."
  }]
}
```

### 7.2 错误引导策略

| 场景 | 错误引导 |
|------|----------|
| 参数格式错 | 示例正确格式 + 解释约束 |
| 权限不足 | 说明需要的权限 + 指向申请方式 |
| 资源不存在 | 建议搜索类工具 + 可能的替代 ID |
| 结果过多 | 建议加过滤条件 + 当前限制说明 |
| 速率限制 | 建议等待时间 + 批量替代方案 |

## 8. MCP Tool Annotations：声明式行为提示

MCP 2025-03-26 引入了 Tool Annotations 机制，让 Server 向 Client 声明工具的行为特征：

```typescript
interface ToolAnnotations {
  title?: string;           // 显示名称（纯 UX）
  readOnlyHint?: boolean;   // 默认 false: 是否只读
  destructiveHint?: boolean; // 默认 true: 是否有破坏性
  idempotentHint?: boolean; // 默认 false: 是否幂等
  openWorldHint?: boolean;  // 默认 true: 是否访问外部开放环境
}
```

### 8.1 客户端如何使用 Annotations

| Annotation | Client 行为 |
|------------|-------------|
| `readOnlyHint: true` | 可跳过确认对话框（可信 Server 前提下） |
| `destructiveHint: true` | 执行前显示警告 |
| `idempotentHint: true` | 失败时安全重试 |
| `openWorldHint: true` | 标记信任边界跨越，审视返回内容 |

### 8.2 关键限制：Hints ≠ Guarantees

Annotations 是**提示**而非**保证**。规范明确指出："annotations should be treated as untrusted unless they come from a trusted server"。一个恶意 Server 完全可以声明 `readOnlyHint: true` 然后执行写操作。

真正的安全保障应当建立在：
- 网络层隔离（Server 能访问什么）
- 沙箱执行（进程级/容器级限制）
- 权限门控（Host 代码中的硬校验）

### 8.3 Simon Willison 的"致命三角"

安全研究者 Simon Willison 指出，当以下三个条件同时满足时，数据泄露几乎不可避免：

1. 访问私密数据（读邮件、文件、数据库）
2. 暴露于不可信内容（搜索结果、外部 API 返回）
3. 具有外部通信能力（发邮件、写文件、调 webhook）

Tool Annotations 的 `openWorldHint` 正是为检测第三条而设计。未来 MCP 规范可能引入更多维度（`unsafeOutputHint`、`secretHint`、`trustedHint`）来让 Client 自动检测"致命三角"的完整组合。

## 9. 工具生态治理

当工具数量从个位数增长到十位数、百位数时，"设计好一个工具"不再是唯一挑战——你需要治理整个工具生态。

### 9.1 治理维度

| 维度 | 问题 | 解决方案 |
|------|------|----------|
| **发现** | Agent 如何知道有哪些工具可用？ | MCP Registry、Tool Search、工具目录 |
| **版本** | 工具升级如何不破坏现有 Agent？ | 语义化版本 + 能力协商 + 向后兼容 |
| **质量** | 如何确保第三方工具靠谱？ | 官方认证、社区评分、conformance test |
| **安全** | 如何防止恶意/有漏洞的工具？ | 签名验证、沙箱执行、权限分级 |
| **成本** | 工具调用如何归因和计量？ | 调用链追踪、token 归因、预算配额 |
| **生命周期** | 工具如何上线/下线/迁移？ | 注册/发现/废弃通知机制 |

### 9.2 权限分层模型

MLflow 建议的三级权限：

| 层级 | 操作类型 | 审批要求 | 示例 |
|------|----------|----------|------|
| L1 只读 | 查询、搜索、获取 | 自动通过 | `search_logs`, `get_user` |
| L2 写入 | 创建、修改 | 人工确认 | `create_ticket`, `update_config` |
| L3 破坏性 | 删除、转账、部署 | 多重审批 | `delete_production_data`, `deploy_to_prod` |

关键原则：**权限必须在代码中结构化实施，不能仅靠 prompt 约束**。模型可以被注入绕过 prompt 级限制，但无法绕过代码级门控。

### 9.3 工具可观测性

每次工具调用应记录：

```json
{
  "trace_id": "abc-123",
  "tool_name": "jira_issues_create",
  "timestamp": "2025-03-15T14:23:01Z",
  "input": { "title": "...", "priority": "high" },
  "output_tokens": 234,
  "latency_ms": 450,
  "success": true,
  "caller": "agent-session-789",
  "approval_status": "auto_approved",  // L1
  "cost_attribution": "team-engineering"
}
```

关键指标：
- **工具调用成功率**：低于 95% 说明 schema 或后端有问题
- **单步延迟**：超过 2s 需要考虑异步/缓存
- **token 成本每次调用**：监控异常膨胀
- **选择准确率**：模型选对工具的比例

### 9.4 渐进式工具加载策略

```
Layer 0 (Always Loaded): 3-5 核心工具
  ↓ Tool Search 触发
Layer 1 (On-Demand): 20-50 可搜索工具
  ↓ MCP Server 动态注册
Layer 2 (Discoverable): 200+ 注册表工具
```

这个三层架构平衡了响应速度（Layer 0 无搜索延迟）、覆盖面（Layer 1 按需加载）和可扩展性（Layer 2 跨团队/跨组织）。

## 10. 完整代码示例：工具生态治理框架

```python
"""
Tool Governance Framework
演示：权限门控、调用审计、Token 预算、渐进加载
"""
from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Callable
import time
import json


class PermissionLevel(Enum):
    READ = "L1_read"
    WRITE = "L2_write"
    DESTRUCTIVE = "L3_destructive"


class ApprovalStatus(Enum):
    AUTO_APPROVED = "auto_approved"
    PENDING = "pending"
    APPROVED = "approved"
    DENIED = "denied"


@dataclass
class ToolDefinition:
    """工具注册信息"""
    name: str
    description: str
    input_schema: dict
    handler: Callable
    permission_level: PermissionLevel = PermissionLevel.READ
    annotations: dict = field(default_factory=dict)
    version: str = "1.0.0"
    namespace: str = ""
    # Token cost estimation
    avg_input_tokens: int = 50
    avg_output_tokens: int = 200
    # Loading strategy
    defer_loading: bool = False
    # Examples for LLM
    input_examples: list = field(default_factory=list)


@dataclass
class ToolCallRecord:
    """调用审计记录"""
    trace_id: str
    tool_name: str
    timestamp: float
    input_params: dict
    output: Any = None
    latency_ms: float = 0
    success: bool = True
    error: str | None = None
    approval_status: ApprovalStatus = ApprovalStatus.AUTO_APPROVED
    token_cost: int = 0


class ToolRegistry:
    """工具注册与发现"""

    def __init__(self):
        self._tools: dict[str, ToolDefinition] = {}
        self._audit_log: list[ToolCallRecord] = []
        self._token_budget: int = 100_000  # 每次会话 token 预算
        self._token_spent: int = 0

    def register(self, tool: ToolDefinition) -> None:
        full_name = f"{tool.namespace}_{tool.name}" if tool.namespace else tool.name
        self._tools[full_name] = tool

    def search(self, query: str, top_k: int = 5) -> list[ToolDefinition]:
        """基于关键词的工具搜索（生产环境应替换为向量检索）"""
        scored = []
        for name, tool in self._tools.items():
            score = 0
            query_lower = query.lower()
            if query_lower in name.lower():
                score += 3
            if query_lower in tool.description.lower():
                score += 2
            if any(query_lower in kw for kw in tool.annotations.get("keywords", [])):
                score += 1
            if score > 0:
                scored.append((score, tool))
        scored.sort(key=lambda x: x[0], reverse=True)
        return [t for _, t in scored[:top_k]]

    def get_always_loaded(self) -> list[ToolDefinition]:
        """获取始终加载的核心工具（Layer 0）"""
        return [t for t in self._tools.values() if not t.defer_loading]

    def get_schema_for_llm(self, tools: list[ToolDefinition]) -> list[dict]:
        """生成 LLM 可消费的工具 schema 列表"""
        schemas = []
        for tool in tools:
            schema = {
                "name": f"{tool.namespace}_{tool.name}" if tool.namespace else tool.name,
                "description": tool.description,
                "input_schema": tool.input_schema,
            }
            if tool.input_examples:
                schema["input_examples"] = tool.input_examples
            if tool.annotations:
                schema["annotations"] = {
                    k: v for k, v in tool.annotations.items()
                    if k in ("readOnlyHint", "destructiveHint", "idempotentHint", "openWorldHint")
                }
            schemas.append(schema)
        return schemas


class PermissionGate:
    """权限门控：L1 自动通过，L2/L3 需要审批"""

    def __init__(self, auto_approve_levels: set[PermissionLevel] = None):
        self._auto_approve = auto_approve_levels or {PermissionLevel.READ}

    def check(self, tool: ToolDefinition, params: dict) -> ApprovalStatus:
        if tool.permission_level in self._auto_approve:
            return ApprovalStatus.AUTO_APPROVED

        # 生产环境：发送审批请求到 Slack/DingTalk/邮件
        # 此处简化为自动通过 L2，拒绝 L3
        if tool.permission_level == PermissionLevel.WRITE:
            print(f"[PermissionGate] L2 write operation '{tool.name}' — requesting confirmation...")
            return ApprovalStatus.APPROVED  # 简化
        else:
            print(f"[PermissionGate] L3 destructive operation '{tool.name}' — DENIED (requires multi-approval)")
            return ApprovalStatus.DENIED


class TokenBudgetController:
    """Token 预算控制"""

    def __init__(self, budget: int):
        self.budget = budget
        self.spent = 0

    def can_afford(self, tool: ToolDefinition) -> bool:
        estimated_cost = tool.avg_input_tokens + tool.avg_output_tokens
        return (self.spent + estimated_cost) <= self.budget

    def record(self, actual_tokens: int) -> None:
        self.spent += actual_tokens

    @property
    def remaining(self) -> int:
        return max(0, self.budget - self.spent)


class GovernedToolExecutor:
    """整合治理框架的工具执行器"""

    def __init__(self, registry: ToolRegistry, gate: PermissionGate, budget: TokenBudgetController):
        self.registry = registry
        self.gate = gate
        self.budget = budget

    def execute(self, tool_name: str, params: dict, trace_id: str) -> dict:
        """执行工具调用，经过完整治理管线"""
        tool = self.registry._tools.get(tool_name)
        if not tool:
            return {"isError": True, "content": f"Tool '{tool_name}' not found. Use tool_search to discover available tools."}

        # Step 1: 预算检查
        if not self.budget.can_afford(tool):
            return {
                "isError": True,
                "content": f"Token budget exceeded. Remaining: {self.budget.remaining} tokens. "
                           f"Consider using CONCISE response_format or completing the task with current information."
            }

        # Step 2: 权限门控
        approval = self.gate.check(tool, params)
        if approval == ApprovalStatus.DENIED:
            return {
                "isError": True,
                "content": f"Permission denied for '{tool_name}' (requires {tool.permission_level.value}). "
                           f"This operation needs multi-party approval. Contact your admin."
            }

        # Step 3: 执行 + 审计
        start = time.time()
        record = ToolCallRecord(
            trace_id=trace_id,
            tool_name=tool_name,
            timestamp=start,
            input_params=params,
            approval_status=approval,
        )

        try:
            result = tool.handler(**params)
            record.latency_ms = (time.time() - start) * 1000
            record.success = True
            record.output = result
            # 估算实际 token（简化）
            actual_tokens = len(json.dumps(result)) // 4
            record.token_cost = actual_tokens
            self.budget.record(actual_tokens)
            return {"content": result}
        except Exception as e:
            record.latency_ms = (time.time() - start) * 1000
            record.success = False
            record.error = str(e)
            return {
                "isError": True,
                "content": f"Tool execution failed: {str(e)}. "
                           f"Suggestion: verify parameters match the schema and retry."
            }
        finally:
            self.registry._audit_log.append(record)


# ─── 使用示例 ──────────────────────────────────────────────────────

def _handle_search_logs(query: str, time_range: str = "1h", limit: int = 100) -> str:
    return f"Found 23 log entries matching '{query}' in last {time_range}"

def _handle_delete_logs(older_than_days: int) -> str:
    return f"Deleted logs older than {older_than_days} days"

# 注册工具
registry = ToolRegistry()

registry.register(ToolDefinition(
    name="search_logs",
    namespace="observability",
    description=(
        "Search application logs by keyword and time range. Returns matching lines with context. "
        "Supports Lucene query syntax. Use for debugging errors or tracing requests. "
        "For real-time streaming, use tail_logs instead. Max 1000 results per call."
    ),
    input_schema={
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "Lucene query (e.g., 'level:ERROR AND service:payment')"},
            "time_range": {"type": "string", "enum": ["15m", "1h", "6h", "24h", "7d"], "default": "1h"},
            "limit": {"type": "integer", "minimum": 1, "maximum": 1000, "default": 100},
        },
        "required": ["query"],
    },
    handler=_handle_search_logs,
    permission_level=PermissionLevel.READ,
    annotations={"readOnlyHint": True, "openWorldHint": False},
    defer_loading=False,  # Layer 0: always loaded
))

registry.register(ToolDefinition(
    name="delete_logs",
    namespace="observability",
    description=(
        "Permanently delete application logs older than specified days. "
        "DESTRUCTIVE: cannot be undone. Requires L3 multi-party approval. "
        "For archiving instead of deleting, use archive_logs."
    ),
    input_schema={
        "type": "object",
        "properties": {
            "older_than_days": {"type": "integer", "minimum": 30, "description": "Minimum age in days (must be ≥30)"},
        },
        "required": ["older_than_days"],
    },
    handler=_handle_delete_logs,
    permission_level=PermissionLevel.DESTRUCTIVE,
    annotations={"destructiveHint": True, "idempotentHint": True, "openWorldHint": False},
    defer_loading=True,  # Layer 1: on-demand
))

# 执行
gate = PermissionGate(auto_approve_levels={PermissionLevel.READ})
budget = TokenBudgetController(budget=50_000)
executor = GovernedToolExecutor(registry, gate, budget)

# 读操作：自动通过
result = executor.execute("observability_search_logs", {"query": "level:ERROR"}, trace_id="t-001")
print(f"Read result: {result}")

# 破坏性操作：被拒绝
result = executor.execute("observability_delete_logs", {"older_than_days": 90}, trace_id="t-002")
print(f"Delete result: {result}")
```

## 11. 设计检查清单

在发布一个工具之前，过一遍这张清单：

**命名与描述**
- [ ] 名字用 `动词_名词` 或 `namespace_resource_action` 格式？
- [ ] Description 包含：功能 + 使用时机 + 不使用时机 + 格式要求？
- [ ] 与相似工具有明确的边界说明（"for X use Y instead"）？

**参数设计**
- [ ] 所有参数名语义完整、无歧义？
- [ ] 使用了 JSON Schema 的 enum/format/pattern/min/max 约束？
- [ ] 提供了 1-3 个 input_examples 覆盖典型场景？
- [ ] 避免暴露内部 ID（优先自然语言标识）？

**错误处理**
- [ ] 错误返回包含具体修正建议？
- [ ] 设置了 `isError: true` 标记？
- [ ] 指向了替代工具或下一步操作？

**Token 效率**
- [ ] 支持 response_format 或 limit 参数控制输出量？
- [ ] 大结果集有分页/截断机制？
- [ ] 考虑了 defer_loading（>10 工具时）？

**安全与治理**
- [ ] 设置了正确的 permission_level？
- [ ] 添加了 MCP annotations（readOnly/destructive/idempotent/openWorld）？
- [ ] 输入有类型校验（不依赖模型"不会犯错"）？
- [ ] 输出不泄露不必要的内部细节？

**可观测性**
- [ ] 每次调用有 trace_id 贯穿？
- [ ] 记录了 input/output/latency/success？
- [ ] 有异常调用频率的告警？
