# Guardrails 体系：Input / Output / Tool 三层护栏

> 本文是 Agent 系列第 6.2 篇。**6.1 揭示了 prompt injection 是架构性缺陷、给出了 8 层防御纵深栈；6.2 把视角缩近一层——专门研究"护栏（Guardrail）这一类组件该怎么设计。"**
>
> Guardrail 在 2024-2026 年被严重过度承诺：很多团队装一个 Llama Guard / OpenAI Moderation 就以为护栏到位了，实际上这只覆盖了八层栈中的 **L5 一层**。真问题不是"装哪个 guardrail 库"，而是 **"在 Input、Output、Tool 三个环节上分别该塞什么策略、用什么实现、放在什么位置、容忍多少延迟。"**
>
> 这一篇会从一个具体的真问题出发——*"为什么我装了 NeMo Guardrails 还是被 IDPI 打穿？"*——把 Guardrails 抽象成 **6 类失败模式 × 3 个生效环节 × 2 个时间维度（build-time / runtime）的矩阵**，把当前主流库（NeMo Guardrails / Guardrails AI / Llama Guard / Lakera / Azure / OpenAI Moderation）的能力定位讲清楚，并给出一份生产可落地的三层护栏蓝图。

---

## 0. 真问题：装了护栏，为什么还是被打穿？

一个常见的失败模式：

```
团队 A 部署了 Llama Guard 3 在所有用户输入和模型输出上做扫描，
但 Agent 仍然被一个嵌入网页中的 IDPI 打穿，
原因是 Agent 在工具调用阶段读取了外部网页，
而 Llama Guard 只在「用户消息 → 模型」和「模型 → 用户」两个边界做扫描，
工具结果完全在它的视线之外。
```

这背后的认知错误是：**把 Guardrail 当作单点工具，而不是一组分布在数据流多处的策略**。同样典型的错误还有：

* 只做 input guardrail，假设输出一定安全（攻击者用 input 引发 output 中泄露 system prompt → output 没护栏，被泄露）
* 只做 output guardrail，攻击者已经触发了高风险工具调用（订金钱、删数据库），output 才被拦截，已经晚了
* 只用 LLM-as-judge 当 guardrail，每个 turn 加 5-10 秒延迟，p95 失控
* 用同一个 LLM 来做 guardrail——攻击者可以用 prompt injection 同时绕过两道关
* 只装现成 guardrail 库，没有写**自己业务的 policy**——库永远不知道你的退款规则

> 一句话总结：**Guardrail 不是一个组件，是一组沿数据流分布的、定义在 6 类失败模式 × 3 个生效环节上的策略集合。** 真正的工程问题是怎么把这些策略组织起来。

## 1. 第一性原理：Guardrail 的 6 类失败模式

> "What none of them ship is your policy" —— Morph LLM, *LLM Guardrails 2026*

任何 guardrail 库的能力都可以被拆解到下面 6 类失败模式上。前 5 类是 **语义类（semantic）**，需要理解；第 6 类是 **机械类（mechanical）**，正则/schema 就够。

| # | 失败模式 | 例子 | 机械 vs 语义 |
|---|---|---|---|
| F1 | **Jailbreak & Prompt Injection** | DAN、roleplay 越狱、IDPI | 语义（最难） |
| F2 | **PII / Data Leak** | 信用卡号回显、未脱敏简历 | 部分机械（结构化）+ 语义（描述性） |
| F3 | **Toxicity / Harmful Content** | 仇恨/暴力/自伤 | 语义 |
| F4 | **Topic / Policy Enforcement** | 客服 Agent 谈股票投资 | 语义（且高度业务定制） |
| F5 | **Hallucination / Groundedness** | 答案与检索内容不一致 | 语义（需要对照） |
| F6 | **Format / Structural Validation** | JSON schema、字段长度 | 机械 |

> *Source*: [Morph LLM, LLM Guardrails 2026: Failure Taxonomy](https://www.morphllm.com/llm-guardrails)。

**关键结论 1**：F1-F5 必须靠语义模型（小型 classifier 或 LLM），F6 用代码校验。**把所有失败都丢给同一种实现的 guardrail 库，必然漏**。

**关键结论 2**：F4（topic/policy）几乎不可能用现成库覆盖——因为它是**你的业务规则**。所有 guardrail 框架（NeMo、Guardrails AI、Lakera）都要求你自己写 policy。

## 2. 第二性原理：3 个生效环节 + 2 个时间维度

把 guardrail 放在 Agent 数据流的什么位置，决定了它能拦什么、拦不住什么。一个 Agent step 的关键边界有 **三个**：

```
        ┌──────────────────────────────────────────────────────────┐
        │                    Agent Runtime                          │
        │                                                           │
  user  │  ① Input    ┌────────────┐    ② Tool     ┌───────────┐   │
 ──────►│ ─────────► │  LLM Step  │ ─────────────► │  Tools    │   │
        │  Guardrail │            │  Guardrail     │           │   │
        │            └────┬───────┘                └─────┬─────┘   │
        │                 │ ③ Output                     │         │
        │                 │  Guardrail                   ▼         │
        │                 ▼                       ┌────────────┐   │
        │              user                       │ Tool Result│   │
        │                                         │ Guardrail  │   │
        │                                         │ (≈ Input2) │   │
        │                                         └────────────┘   │
        └──────────────────────────────────────────────────────────┘
```

* **① Input Guardrail**：用户消息进入模型之前。截 F1（直接注入）/F3/F4 用户输入侧问题
* **② Tool / Tool-Result Guardrail**：模型决定调用工具时拦工具调用；工具结果回流时拦不可信内容（IDPI）
* **③ Output Guardrail**：模型生成内容回到用户/上游之前。截 F2（PII 回显）/F5（hallucination）/output spoofing

加上 **build-time vs runtime** 两个时间维度：

| 时间 | 形式 | 例子 |
|---|---|---|
| **Build-time** | 红队评测、CI 套件、fine-tune 数据扫描、AgentDojo 回归 | 离线、慢、详尽 |
| **Runtime** | 在每次 step 上同步/异步执行 | 在线、快、必须在 latency budget 内 |

> "Build-time 测的是你预想到的分布；runtime 测的是当前对话的真实分布。"

两层都不能省：build-time 防"你想到的",runtime 防"你没想到的"。

## 3. Input Guardrail：进入模型前的最后一道闸

### 3.1 应做的事

| 任务 | 实现 | 目标失败模式 |
|---|---|---|
| 长度 / 格式 / 编码合法性 | 代码校验 | F6 |
| PII 检测（脱敏 / 拒绝） | regex + NER 小模型 | F2 |
| Jailbreak / Prompt Injection 检测 | classifier (PromptGuard 2 / Lakera / Llama Guard) | F1 |
| Topic 范围校验 | 自训 classifier 或 LLM-as-judge（采样） | F4 |
| Toxicity / 违禁内容 | OpenAI Moderation / Azure CSF / Llama Guard | F3 |

### 3.2 实现选择矩阵

| 库 / 服务 | 类型 | 延迟 | 成本 | 主要覆盖 | 备注 |
|---|---|---|---|---|---|
| **OpenAI Moderation** | 托管 API | ~100-300ms | 免费（API 用户） | F3 | 公开 taxonomy，覆盖窄 |
| **Azure AI Content Safety** | 托管 API | 375-873ms | 按量 | F3, F1 部分 | "Microsoft 生态基础底" |
| **Llama Guard 4 (12B)** | 自托管 / 1B/8B 小版 | 459ms | 自承担算力 | F3, F1 | F1 OpenAI Mod 0.737 / HarmBench 0.961 |
| **PromptGuard 2** | Meta 小型 classifier | ~50ms | 自承担 | F1 | 配 LlamaFirewall ASR 17.6%→1.75% |
| **Lakera Guard** | API | ~200-500ms | 商业 | F1 | 0.697 OpenAI Mod / 0.525 GA Jailbreak |
| **GA Guard / GA Guard Lite** | 商业 | 16-29ms | 商业 | F1, F3, 长上下文 | 长上下文场景显著优于云大厂 |
| **NeMo Guardrails (Nemoguard 8B)** | 自托管 framework | <50ms | 自承担 | F1, F3, F4 | 0.793 OpenAI Mod / 0.875 HarmBench |
| **自训小模型** | 自托管 | 30-90ms | 自承担 | F4 业务专属 | 唯一能覆盖你 policy 的方案 |

> *Source*: [General Analysis, Best AI Guardrails 2026](https://generalanalysis.com/guides/best-ai-guardrails)；Morph LLM。

### 3.3 一个真相：Long-context 场景下大厂 guardrail 性能崩塌

General Analysis 2026 测试显示——Azure AI Content Safety 在 **长上下文 jailbreak benchmark 上 F1 仅 0.046**（短文本是 0.193）。原因是大多数云端 guardrail 默认窗口很小，长文本被截断。

**对策**：长文本场景必须用支持长上下文的 guardrail（如 GA Guard 或自训大窗口分类器），或者在喂给 guardrail 之前做**滑窗扫描**。

### 3.4 设计原则

1. **分层**：mechanical（regex / schema）→ 小 classifier（10-50ms）→ LLM-as-judge（仅高风险路径采样）
2. **快路径 + 慢路径**：所有 turn 走 fast guard；高风险动作（金钱、写库）才触发 heavy guard
3. **同时支持 block / sanitize / quarantine 三种 action**：不是非黑即白
4. **拒绝走自然语言模板**：防止攻击者根据拒绝信息调整 payload

## 4. Output Guardrail：模型与世界之间的最后一道闸

Output 边界要拦的是**模型已经做出的不当内容**。常见任务：

| 任务 | 实现 | 目标失败模式 |
|---|---|---|
| PII / 系统提示词 / Token 回显 | regex + canary token | F2 |
| Markdown / URL / Image src 白名单 | 代码 + URL allowlist | F2（exfil 侧信道） |
| Unicode tag / Bidi override 清洗 | 代码 | F1（隐形指令） |
| 输出 schema 校验 + 重试 | Pydantic / Outlines / Constrained Decoding | F6 |
| Groundedness / Faithfulness | LLM-as-judge / Faithfulness scorer | F5 |
| Toxicity 反向扫描 | OpenAI Moderation / Llama Guard | F3 |

### 4.1 关键点：Canary Token 拦 system prompt 泄露

把一段唯一 random token 注入 system prompt（如 `INTERNAL-CANARY-9F3A2B`），output guardrail 实时扫描该 token——一旦命中，认定 system prompt 已被泄露，立即 block + alert。简单但极其有效。

### 4.2 Hallucination 该不该在 runtime 拦

诚实结论：**runtime 全量拦不住**。Faithfulness 检查需要把 retrieval context + 答案一起跑 LLM-as-judge，p95 至少 1-3 秒。生产做法是 **采样**（10-20%）+ **离线评测**（RAGAS / DeepEval CI）。少数高风险场景（医疗 / 法务）才全量拦。

### 4.3 Output Sanitization Hardening

零成本但 ROI 极高（已在 6.1 提到，这里展开实现）：

```python
import re
import urllib.parse

UNICODE_TAG_RE = re.compile(r"[\U000E0000-\U000E007F]")
BIDI_RE = re.compile(r"[\u202A-\u202E\u2066-\u2069]")

ALLOWED_HOSTS = {"docs.example.com", "cdn.example.com"}

def sanitize_output(text: str) -> str:
    # 1. 去 Unicode tag / bidi override
    text = UNICODE_TAG_RE.sub("", text)
    text = BIDI_RE.sub("", text)
    # 2. 拦截 markdown image：![alt](url) 中 url 必须在白名单
    def _img(m):
        url = m.group(2)
        host = urllib.parse.urlparse(url).hostname or ""
        return m.group(0) if host in ALLOWED_HOSTS else f"[image blocked: {host}]"
    text = re.sub(r"!\[([^\]]*)\]\(([^)]+)\)", _img, text)
    # 3. canary token 检测
    if "INTERNAL-CANARY-" in text:
        raise SystemPromptLeakDetected()
    return text
```

## 5. Tool Guardrail：Agent 时代真正的"主战场"

> 这是 Module 5/6 反复强调的一点：**Agent 之所以比 chatbot 危险得多，根源是它能调工具——执行真实世界的动作**。

Tool Guardrail 要拦三种东西：

### 5.1 Pre-call：参数校验 + 权限校验

| 检查 | 例子 | 实现 |
|---|---|---|
| 参数 schema 合法 | 数字范围、枚举、URL 格式 | Pydantic / JSON Schema |
| 参数没有 taint 污染 | 转账金额来自外部网页内容 | Taint tracker（见 6.1） |
| 当前 Agent 有此工具权限 | 客服 Agent 不能调 `delete_db` | RBAC + scoped credentials（见 6.3） |
| 操作处于预算内 | 单次 session 单工具调用次数上限、累计金额上限 | 工具级 budget counter |
| 操作是否需要 HITL 审批 | 转账 > $100 必须人工确认 | 风险等级 → 审批 gate |

### 5.2 Post-call：工具结果再扫描（关键、被严重低估）

> **大多数 IDPI 在这一层被打穿。** Tool result 是不可信内容入侵 Agent 的最大入口（网页、邮件、issue、用户上传 PDF）。

必做的处理：

1. **Spotlighting wrapping**：所有工具结果在拼回 prompt 前用随机 delimiter 包裹（见 6.1 §4.3）
2. **Classifier 扫描**：把工具结果当作不可信输入，过 PromptGuard 2 / Llama Guard
3. **Taint 标记**：被读取的字段全部打 tainted=True，传播到下游
4. **Schema 抽取**：用 quarantined LLM（Dual-LLM 模式）把不可信原文抽成结构化字段，把字段而非原文回传给主 LLM

### 5.3 Side-effect 控制

| 维度 | 策略 |
|---|---|
| 幂等性 | 写类工具必须支持 idempotency key |
| 回滚 | 高风险操作必须有 reverse 方法（如 `undo_payment(tx_id)`） |
| Dry-run | 上线前所有写类工具必须支持 dry-run mode |
| 速率限制 | 工具级 token bucket / circuit breaker |
| 审计日志 | 所有写类工具调用全量日志 + 不可变追加 |

## 6. 主流 Guardrail 库定位地图

把上面三层贴回库：

```
                             失败模式覆盖
                  F1 inj  F2 PII  F3 tox  F4 pol  F5 hal  F6 fmt
                  ───────────────────────────────────────────────
NeMo Guardrails    ●●●     ●       ●●●     ●●●●    ●        ●●
(Colang DSL)
Guardrails AI       ●       ●●●    ●        ●        ●●     ●●●●
(validator stack)
Llama Guard 3/4     ●●●     ●       ●●●●    ─        ─        ─
(classifier)
PromptGuard 2       ●●●●   ─        ─        ─        ─        ─
Lakera Guard        ●●●     ●       ●●       ─        ─        ─
Azure CSF           ●●     ●       ●●       ─        ─        ─
OpenAI Moderation   ●       ─       ●●●●    ─        ─        ─
GA Guard            ●●●●   ●●     ●●●     可定制   ─        ─
RAGAS / DeepEval    ─        ─       ─        ─        ●●●●   ●
```

**直白结论**：

* **想要单点最强 F1**：PromptGuard 2 或 GA Guard（前者免费、后者商业但延迟最低）
* **想要 framework 写 policy（F4）**：NeMo Guardrails（Colang）或 Guardrails AI（validators）
* **想要结构化输出（F6）**：Guardrails AI / Pydantic AI / Outlines（见 1.6）
* **想要 hallucination 评测（F5）**：build-time 用 RAGAS / DeepEval；runtime 采样用 LLM-as-judge
* **想要"一站式"**：不存在。生产系统必须组合至少 3-4 个

## 7. 生产蓝图：三层护栏的可落地骨架

```python
from typing import Any, Callable
from pydantic import BaseModel
import asyncio

# ─────────── F6 + 业务 schema ───────────
class TransferArgs(BaseModel):
    to_account: str
    amount: float
    memo: str = ""

# ─────────── 三类 guardrail 抽象 ───────────
class InputGuardrail:
    async def check(self, user_msg: str) -> "GuardResult":
        ...

class OutputGuardrail:
    async def check(self, model_out: str, ctx: dict) -> "GuardResult":
        ...

class ToolGuardrail:
    async def pre_call(self, tool: str, args: dict, agent_ctx: dict) -> "GuardResult":
        ...
    async def post_call(self, tool: str, result: Any) -> Any:
        ...


# ─────────── Agent step ───────────
async def agent_step(user_msg: str):
    # ① INPUT
    fast = await fast_input_guard.check(user_msg)        # ~30-50ms classifier
    if fast.action == "block":
        return refuse(fast.reason)
    if fast.action == "sanitize":
        user_msg = fast.cleaned

    # 异步采样跑慢路径，仅审计（不阻塞）
    asyncio.create_task(slow_input_guard.check(user_msg))

    # LLM step（含工具调用 loop）
    while True:
        resp = await llm.invoke(user_msg, tools=TOOLS)
        if not resp.tool_calls:
            break

        for call in resp.tool_calls:
            # ② TOOL pre
            decision = await tool_guard.pre_call(call.name, call.args, ctx)
            if decision.action == "block":
                resp.add_tool_error(call, decision.reason)
                continue
            if decision.action == "approve_required":
                ok = await human_approve(call)           # 必须展示 raw args
                if not ok:
                    continue

            raw = await dispatch(call)

            # ② TOOL post：不可信结果走净化
            cleaned = await tool_guard.post_call(call.name, raw)
            resp.add_tool_result(call, cleaned)

    # ③ OUTPUT
    out = sanitize_output(resp.content)                  # zero-cost L7 cleaning
    out_check = await output_guard.check(out, ctx)
    if out_check.action == "block":
        return refuse_output(out_check.reason)
    return out
```

**关键工程要点**：

1. **fast / slow 双路径**：fast 同步阻塞（<50ms），slow 异步采样（仅审计）
2. **Block / Sanitize / Approve_required / Allow** 四种 action（不是非黑即白）
3. **HITL 必须看 raw call.args**，不能看 LLM 总结
4. **Post-call 净化是 IDPI 的主防线**——别忘了
5. 所有 guard 决策入审计日志（见 6.4）

## 8. 性能与延迟预算

一个生产级 Agent step 的延迟预算（参考值，p95）：

| 阶段 | 预算 | 实现 |
|---|---|---|
| Input fast guard | 30-50ms | PromptGuard 2 / 自训小 classifier |
| LLM 推理（含 tool） | 1-3s | 主模型 |
| Tool 调用（外部 API） | 500ms-2s | 看工具 |
| Tool post-guard（含 spotlighting + classifier） | 50-100ms | 自托管 |
| Output sanitize（mechanical） | <5ms | 代码 |
| Output classifier 扫描 | 30-100ms | 同 fast guard |
| **整 step p95** | **≤5s** | — |

> 经验法则：**runtime guardrail 总延迟预算 ≤ 200ms**（Input 50 + Tool 100 + Output 50）。超过这个量级，p95 就会被 guard 自己吃掉。

## 9. 反模式

| 反模式 | 表现 | 修正 |
|---|---|---|
| **单点 guardrail** | 全靠 Llama Guard 一个 | 至少 Input + Output + Tool 三层 |
| **没有 Tool guardrail** | 工具调用 / 工具结果裸跑 | 必须 pre_call + post_call + spotlighting |
| **同模型自审** | 用主 LLM 自己当 guardrail | 必须独立模型（小型 classifier 或 quarantined LLM） |
| **LLM-as-judge 全量** | 每个 turn 跑大模型 judge | 仅采样 + 高风险路径触发 |
| **拒绝信息泄露策略** | 拒绝时把规则原文吐给攻击者 | 模板化拒绝（"sorry, I can't"） |
| **没有 build-time** | 只装 runtime guard，不做红队/CI | 必须 AgentDojo / 自定义 redteam suite 入 CI |
| **业务 policy 缺失** | 装了通用库，没写自己的 policy | F4 必须自训或 Colang / Validator 编排 |
| **没有 canary token** | system prompt 泄露无法检测 | 注入 canary，output guard 实时扫描 |
| **拦截后无法 sanitize** | 全部 block，UX 灾难 | 同时支持 sanitize / quarantine 状态 |
| **过度依赖云 guard 长文本** | Azure CSF 长上下文 F1 0.046 | 长文本场景换 GA Guard / 自训长窗口 |

| 阶段 | 目标 | 关键动作 |
|---|---|---|
| **S0** | mechanical 兜底 | length / regex / JSON schema |
