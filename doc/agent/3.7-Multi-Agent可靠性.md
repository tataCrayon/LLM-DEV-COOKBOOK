# 3.7 Multi-Agent 可靠性：Blast Radius / 级联失败 / 容错与系统级评估

> 模块三收官篇。3.5 解决"角色清晰"、3.6 解决"分而治之"，本文回答最后一个问题：**当系统里跑着 5 个、20 个、100 个 Agent，其中任何一个会随机说瞎话时，整个系统怎么不崩？**

---

## §0 真问题：从单点失败到整体崩盘

当 Multi-Agent 系统进入生产，最常见的痛点不是单个 Agent 不够聪明，而是：

1. **一个 worker 答错，下游全错。** 没有故障隔离，错误像瀑布一样冲下来。MAST 团队在 7 个主流框架、1,600+ 真实 trace 中观察到，**只有 21.30% 的失败和"验证不足"直接相关；41.77% 来自规约缺陷、36.94% 来自协调失败**（Cemri et al., NeurIPS 2025 Datasets & Benchmarks Spotlight）—— 也就是说，绝大多数生产事故**根本不是模型"不够强"，而是系统级架构问题**。
2. **Blast Radius 不受控。** 一个 sub-agent 误删了文件、误发了邮件、误转了账，没有边界限制爆炸半径。
3. **评估时全是"准确率"，没有"系统鲁棒性"。** 单 Agent benchmark 上分了，组队上线就崩。
4. **Byzantine fault 在 LLM 世界放大十倍。** 经典分布式系统的"crash fault"还可挽救，LLM 加入"semantic fault"（输出合法但语义错误）后，传统 BFT 协议直接失效。MAS-FIRE 实验显示：**GPT-5 在 Table-Critic Blind Trust 模式下 RS（鲁棒性得分）仅 6.32%，而 DeepSeek-V3 RS 反而高达 70.61%**——更强模型在错误的容错模式下反而更脆弱（Wang et al., arXiv:2602.19843, 2026）。
5. **没有 system-level eval。** 团队每次发版都靠"祈祷"，没有可重复的容错评估框架。

本文目标：把 Multi-Agent 可靠性从"靠 LLM 自愈"提升为"按分布式系统范式工程化"。

---

## §1 第一性原理：分布式系统 50 年经验在 Agent 上的复用与失效

分布式系统三大永恒主题：**故障模型 / 一致性边界 / 失败处理协议**。Multi-Agent 系统把它们继承了——但需要打补丁。

### 1.1 故障模型再分类

| 故障类型 | 经典分布式 | Multi-Agent 表现 | 处理范式 |
|---|---|---|---|
| **Crash fault** | 节点宕机 | Agent 超时 / API 抛错 / OOM | 超时 + 重试 + 备份 worker |
| **Omission fault** | 消息丢失 | Tool call 部分失败 / 流式截断 | 幂等 + 可重放 |
| **Byzantine fault** | 节点恶意/任意行为 | 模型幻觉 / 越权 / Prompt Injection | 多副本 + 投票 + 守门 |
| **Semantic fault**（LLM 独有）| —— | 输出语法/格式合法，语义错误 | LLM-as-Judge + 验证器 + 反事实测试 |
| **Specification fault**（MAST 41.77%）| —— | 任务约束错位 / 角色越界 / 指令缺失 | Schema + RoleSpec + 强制 IO 契约 |
| **Coordination fault**（MAST 36.94%）| —— | 消息丢传 / 上下文未对齐 / 死锁 | Blackboard / 事件溯源 / Saga |

**关键洞察**：传统分布式系统假设 "fail-stop"（要么对要么不响应），而 LLM 是 "fail-loud-wrongly"（自信地犯错）。这让多数现成的 BFT 协议（PBFT/Raft）失效——投票投不出真理，因为 N 个副本可能一致地错。

### 1.2 不变量优先于策略

可靠性工程的核心思路从 70 年代 Lampson、80 年代 Lamport、到 2020s SRE 一脉相承：**先定义系统级不变量，再设计能维持不变量的机制**。

Multi-Agent 系统应该明确的不变量（按重要性排序）：

1. **Blast radius bound**：任何 Agent 的副作用最大波及范围有明确上限。
2. **No silent failure**：所有失败要么被处理、要么被显式上报，绝不被吞掉。
3. **Idempotency of side effects**：写操作（DB / API / 文件）幂等可重放。
4. **Compensating transactions exist**：每个不可逆操作有补偿动作（Saga 范式）。
5. **Eventual termination**：Loop 必有终止条件（预算 / 步数 / 时限）。
6. **Auditable trace**：任何决策可回放、可审计、可归因。

写程序时不背这些，等线上事故时刚好对应"事后分析报告"的章节标题。

## §2 数据：MAST + MAS-FIRE 的硬证据

### 2.1 MAST 三轴失败分类（Cemri et al., 2025）

| 失败大类 | 占比 | 代表症状 |
|---|---|---|
| **Specification & System Design**（规约与设计）| 41.77% | 任务定义不清 / 角色边界缺失 / 协议未定义 / 退出条件含糊 |
| **Inter-Agent Misalignment**（协调失败）| 36.94% | 消息丢传 / 上下文窗口断层 / 工作交接信息丢失 / 死锁 |
| **Task Verification**（验证不足）| 21.30% | 输出未验证 / 终止过早 / 错误传播 / 无回溯 |

来源：[Why Do Multi-Agent LLM Systems Fail?](https://arxiv.org/abs/2503.13657) (NeurIPS 2025 D&B Spotlight)，分析了 7 个框架（AutoGen / MetaGPT / ChatDev / HyperAgent / AppWorld / AG2 / Magentic-One）的 1,600+ traces。

**架构启示**：78.71%（41.77 + 36.94）的失败发生在"模型推理之前的系统设计层"。这意味着——**你的"再换个更强的模型"通常没用**，必须先修系统。

### 2.2 MAS-FIRE 注入式可靠性评估（Wang et al., 2026）

MAS-FIRE 把 15 种故障类型（按 7 类组织：Planning / Memory / Reasoning / Action / Configuration / Instruction / Communication）作为"输入扰动"主动注入到 5 个真实 MAS（Self-Discover / DyLAN / Table-Critic / EvoMAC / G-Designer）。

四项指标：

- **RS（Robustness Score）**：注入故障后系统输出仍正确的比例。
- **Of（Offset Frequency）**：错误被检测但修正失败的频率。
- **Lf（Latch Frequency）**：错误被"误锁"为正确答案的频率。
- **Sf（Suppress Frequency）**：错误被静默吞掉的频率。

最反直觉的一条数据：

| 模型 | Table-Critic Blind Trust | Table-Critic Reasoning Verify |
|---|---|---|
| GPT-5 | **RS=6.32%** | RS=68.4% |
| DeepSeek-V3 | RS=70.61% | RS=72.8% |

含义：当 MAS 的容错模式是"无脑信任 critic"时，**越强的模型反而越脆弱**（因为它的 critic 信号更被下游 worker 当真）。容错策略必须和模型能力**共同设计**。

来源：[MAS-FIRE: Benchmarking Fault Injection Robustness on Multi-Agent Systems](https://arxiv.org/abs/2602.19843)。

### 2.3 容错策略四级分类（MAS-FIRE 总结）

| 级别 | 名称 | 例子 | 适用场景 |
|---|---|---|---|
| **L1** | Mechanism Tolerance | DAG 重试 / 超时 fallback / 多副本投票 | 故障显式可观测时 |
| **L2** | Rule Tolerance | Schema 校验 / 黑白名单 / Permission Gate | 故障可静态规则识别时 |
| **L3** | Prompt Tolerance | "请再检查一遍" / 反例式 prompt | 故障可被模型自检时 |
| **L4** | Reasoning Tolerance | LLM-as-Judge / Critic Agent / 反事实推理 | 语义级别 fault |

四级正交，不是替代关系。生产系统应该**全栈部署**。

## §3 失败模式分类与对策（Failure-Driven Design）

把 MAST 的三大轴细分为 14 个高频模式，逐条对应工程对策：

### 3.1 规约层（Specification & Design）

| 模式 | 症状 | 对策 |
|---|---|---|
| 角色越界 | Worker 自作主张调用未授权工具 | RoleSpec.allowed_tools + IAM Permission Gate |
| 输出契约缺失 | 下游不知道如何 parse | Pydantic / JSON Schema + extra='forbid' |
| 退出条件含糊 | Worker 没完没了反思 | max_steps / max_tokens / explicit done_signal |
| 工具语义重叠 | Worker 不知该调哪个 | Tool naming convention + description 互斥校验 |

### 3.2 协调层（Inter-Agent）

| 模式 | 症状 | 对策 |
|---|---|---|
| 上下文断层 | Sub-agent 拿到的输入残缺 | Blackboard + explicit context passing schema |
| 消息丢传 | Async 消息漏处理 | Persistent queue + at-least-once + idempotency key |
| 死锁 | A 等 B，B 等 A | Topological sort + cycle detection + timeout |
| 信息单向流失 | Reduce 阶段丢掉 worker 异议 | Synthesizer "preserve disagreement" 设计 |

### 3.3 验证层（Verification）

| 模式 | 症状 | 对策 |
|---|---|---|
| 早停 | 答错就 return | Verifier-first：必须过校验才算完成 |
| 验证器与生成器同模型 | "自我洗白" | Cross-model verification（不同家族模型） |
| 数值验证缺失 | 输出 "12 dollar" 被当成 12 | 类型守卫 + 单位归一 + 数值范围检查 |
| 黑盒判官 | LLM-as-Judge 偏见 | 多 Judge + Bradley-Terry 校准 + 人工校验抽样 |

### 3.4 反 Byzantine 设计：N-version programming 的现代版

经典 N-version programming 思路：**让 N 个独立实现各自跑一遍，结果不一致就报警**。在 Agent 系统的落地：

- **多模型副本**：同一 sub-task 让 GPT-5 / Claude / DeepSeek 各跑一份，结果不一致触发 critic。
- **多 prompt 风格**：同一模型用不同 prompt（CoT vs 直答 vs few-shot）跑多份，统计一致性。
- **多检索路径**：RAG 多召回器（Hybrid / Vector / BM25）结果对比，分歧时人审。

注意：**多数表决（majority vote）≠ 多数正确**。要配合"分歧时升级"策略，不要静默投票。

## §4 工程不变量与代码骨架

### 4.1 Blast Radius Containment

```python
@dataclass
class BlastRadius:
    """每个 Agent 的副作用边界"""
    writable_paths: list[str]          # 只能写哪些目录
    callable_tools: set[str]            # 只能调哪些工具
    max_external_calls: int             # 最多调几次外部 API
    max_token_budget: int               # 最多花多少 token
    max_dollar_budget: float            # 最多花多少美元
    allowed_outbound_domains: list[str] # 允许 fetch 哪些域名
    side_effect_dry_run: bool = True    # 默认空跑

class AgentSandbox:
    def __init__(self, agent: Agent, blast: BlastRadius):
        self.agent = agent
        self.blast = blast
        self.usage = ResourceUsage()

    def execute_tool(self, tool_name: str, args: dict):
        if tool_name not in self.blast.callable_tools:
            raise PermissionError(f"tool {tool_name} 超出 blast radius")
        if self.usage.external_calls >= self.blast.max_external_calls:
            raise QuotaExceeded("外部调用次数耗尽")
        if self.blast.side_effect_dry_run and tool_is_mutating(tool_name):
            return self._dry_run(tool_name, args)
        result = real_call(tool_name, args)
        self.usage.record(tool_name, result)
        return result
```

### 4.2 Worker Isolation + Cross-Verification

```python
class IsolatedWorker:
    """每个 sub-agent 跑在独立 sandbox 里，结果带 provenance"""
    def __init__(self, role: RoleSpec, sandbox: AgentSandbox):
        self.role = role
        self.sandbox = sandbox

    def run(self, task: SubTask) -> WorkerResult:
        with trace_span(f"worker:{self.role.name}") as span:
            try:
                output = self.sandbox.run(task, timeout=task.timeout)
                return WorkerResult(
                    success=True, output=output,
                    trace_id=span.id, worker_id=self.role.name,
                    verified=False,
                )
            except (PermissionError, QuotaExceeded, TimeoutError) as e:
                return WorkerResult(success=False, error=str(e),
                                    trace_id=span.id, worker_id=self.role.name)

class CrossVerifier:
    """跨模型验证：生成器和验证器必须不同家族"""
    def __init__(self, generator_model: str, verifier_models: list[str]):
        assert all(not same_family(generator_model, v) for v in verifier_models), \
            "verifier 必须跨模型族"
        self.verifiers = verifier_models

    def verify(self, task: SubTask, result: WorkerResult) -> VerifyVerdict:
        votes = []
        for v_model in self.verifiers:
            verdict = call_model(v_model, build_verify_prompt(task, result))
            votes.append(verdict)
        if all(v.passed for v in votes):
            return VerifyVerdict(passed=True, confidence=1.0)
        if any(v.failed for v in votes):
            return VerifyVerdict(passed=False, dissent=votes, escalate_to_human=True)
        return VerifyVerdict(passed=False, confidence=0.5)
```

### 4.3 Compensating Transactions（Saga 范式）

```python
@dataclass
class ReversibleAction:
    do: Callable                  # 正向操作
    undo: Callable                # 补偿动作
    idempotency_key: str
    description: str

class SagaCoordinator:
    """所有副作用按 Saga 模式记录，失败时反向补偿"""
    def __init__(self):
        self.history: list[ReversibleAction] = []

    def execute(self, action: ReversibleAction):
        try:
            result = action.do()
            self.history.append(action)
            return result
        except Exception:
            self._rollback_all()
            raise

    def _rollback_all(self):
        # 反向遍历执行补偿
        for action in reversed(self.history):
            try:
                action.undo()
            except Exception as e:
                emit_alert(f"补偿失败：{action.description}, error={e}")
        self.history.clear()
```

### 4.4 Circuit Breaker（防雪崩）

```python
class AgentCircuitBreaker:
    """Worker 连续失败 N 次后，熔断该 worker 一段时间"""
    def __init__(self, threshold: int = 5, cooldown_s: int = 60):
        self.threshold = threshold
        self.cooldown_s = cooldown_s
        self.failure_count: dict[str, int] = defaultdict(int)
        self.open_until: dict[str, float] = {}

    def can_call(self, worker_id: str) -> bool:
        if worker_id in self.open_until:
            if time.time() < self.open_until[worker_id]:
                return False
            del self.open_until[worker_id]
            self.failure_count[worker_id] = 0
        return True

    def record_failure(self, worker_id: str):
        self.failure_count[worker_id] += 1
        if self.failure_count[worker_id] >= self.threshold:
            self.open_until[worker_id] = time.time() + self.cooldown_s
            emit_alert(f"worker {worker_id} 熔断 {self.cooldown_s}s")
```

### 4.5 Idempotency Key

所有可能被重放的 write 操作必须带幂等键：

```python
def call_tool_idempotent(tool: str, args: dict, idem_key: str):
    """同一 idem_key 在 24h 内只执行一次"""
    cached = idempotency_store.get(idem_key, ttl=86400)
    if cached:
        return cached.result
    result = call_tool(tool, args)
    idempotency_store.put(idem_key, result, ttl=86400)
    return result
```

### 4.6 系统级容错评估（FaultInjector 模式）

```python
class FaultInjector:
    """像 Netflix Chaos Monkey 一样，定期注入故障"""
    SCENARIOS = [
        "drop_random_message",     # 随机丢消息
        "timeout_random_worker",    # 随机超时一个 worker
        "corrupt_subtask_output",   # 把某 sub-output 改成 nonsense
        "swap_role_identity",       # 让 worker 错认自己角色
        "double_invoke_tool",       # 让同一工具被调两次
        "inject_prompt_injection",  # 注入越狱 prompt
    ]
    def run_scenario(self, system: MultiAgentSystem, scenario: str, task: Task):
        with self._inject(scenario, system):
            result = system.run(task)
        return ScenarioReport(
            scenario=scenario,
            final_correct=task.verify(result),
            errors_caught=system.errors,
            blast_radius=system.measured_blast,
        )

class ReliabilityHarness:
    """每发版前跑一遍故障注入，把 RS/Of/Lf/Sf 当成回归测试"""
    def assess(self, system) -> ReliabilityReport:
        scores = []
        for scenario in FaultInjector.SCENARIOS:
            for task in self.eval_set:
                scores.append(FaultInjector().run_scenario(system, scenario, task))
        return aggregate(scores)
```

## §5 反模式：不要这么做

1. **"加一个 critic 就够了"**：单 critic 是单点故障，且容易和 worker 同源同偏见。要多 critic + 跨模型 + 抽样人审。
2. **"用更强的模型就稳了"**：MAS-FIRE 数据反证。容错策略必须随模型能力共同设计。
3. **"用多数投票"**：N 个 worker 可能一致地错。要"分歧时升级"+ 高熵答案不投票。
4. **"超时就重试"**：不带幂等键的重试 = 重复写。要先看 idempotency_key 是否被消费过。
5. **"自检-自纠"**：让出错的模型自己改自己，常常越改越错。要让外部 verifier 拿决定权。
6. **"故障注入太麻烦，上线后再说"**：MAS 注入测试成本很低（vs RLHF），缺它就等于裸奔。
7. **"在每步加一堆校验"**：过度校验会拖垮系统。原则是"高风险点重防、低风险点轻防"。
8. **"Critic 比 worker 弱也行"**：MAS-FIRE 实验显示弱 critic 给强 worker 当判官，比没 critic 更糟。

## §6 演进路线 S0 → S5

| 阶段 | 标志 | 核心能力 |
|---|---|---|
| **S0：玩具** | 无失败处理，挂了就重启 | 无 |
| **S1：被动重试** | try/except 包一层 | 简单 retry |
| **S2：故障隔离** | 每个 worker 沙箱 + blast radius | RoleSpec / Permission Gate / Budget |
| **S3：补偿与熔断** | Saga + Circuit Breaker + Idempotency | 反向恢复 + 防雪崩 |
| **S4：跨模型验证** | Verifier ≠ Generator + 投票分歧升级 | 拒绝同源自检 |
| **S5：Chaos Multi-Agent** | 故障注入回归 + RS/Of/Lf/Sf 上看板 | 每次发版前必跑 |

绝大多数生产 Multi-Agent 系统目前在 S1-S2 之间。S3 是 Production-grade 门槛。S5 是头部团队（Anthropic / OpenAI / Devin）的内部实践。

## §7 一页总结 + 模块三收官

### 3.7 一页总结

```
失败模型：Spec 41.77% / Coord 36.94% / Verify 21.30%（MAST）
关键洞察：78% 失败发生在"系统设计层"，换模型治不了
不变量：blast radius bound / no silent fail / idempotency / compensation / termination / audit
四级容错：Mechanism (DAG/timeout/voting) → Rule (schema/IAM) → Prompt (self-check) → Reasoning (LLM-Judge)
反 Byzantine：N-version + 跨模型 verifier + 分歧升级，拒绝同源自检
工程骨架：Sandbox + Saga + Circuit Breaker + Idempotency Key + FaultInjector
评估：每发版跑 RS/Of/Lf/Sf 四指标，注入 15 种故障当回归测试
```

### 模块三三联画总结（3.5 / 3.6 / 3.7）

模块三回答 Multi-Agent 工程的核心三问：

- **3.5 角色与人格工程** → 解决"谁来干"：把 Persona / Role / Capability Boundary 三层分清，让每个 Agent 知道自己边界、能力、退出条件。
- **3.6 任务分解与结果聚合** → 解决"怎么分、怎么合"：根据任务结构（Linear / Map-Reduce / Tree / Debate）选拓扑，根据信息保留需求（Concat / Reduce / Vote / Consensus / Debate / Synthesize）选聚合。
- **3.7 可靠性** → 解决"出错怎么办"：把分布式系统的 50 年经验复用过来，针对 LLM 的 semantic fault 做反 Byzantine 加固。

三者递进。**没做 3.5（角色不清）+ 3.6（分合不当）就上 3.7（容错），等于给塌方的房子加避震器**——顺序很重要。

模块三的核心架构口令：

> **明确角色 → 干净拓扑 → 兜底容错。三者缺一不可，且不可换序。**
