# 9.2 自主度梯度：Copilot → Auto-pilot → Full Autonomous

> "自主度不是能力的副产品——它是一个独立的设计决策。"  
> — Feng & McDonald, Levels of Autonomy for AI Agents（[arXiv:2506.12469](https://arxiv.org/html/2506.12469v1)）

## §0 真问题：你的 Agent 该有多少自主权？

四个生产场景，同一个 Agent 框架，四种截然不同的正确答案。

**场景 A**：代码补全。开发者写到一半，Agent 推荐接下来 3 行。**正确自主度：L1（建议模式）**——延迟必须 <100ms，误操作代价低（删掉就好），自主执行会打断心流。

**场景 B**：跨文件重构。开发者说"把所有 `class` 组件迁移到 `function` 组件"。**正确自主度：L3（自主执行，人审结果）**——30 个文件逐个确认是噩梦，但直接提 PR 让人 review 恰好。

**场景 C**：财务合同审核。Agent 分析 50 页合同标记风险条款。**正确自主度：L2（协作模式）**——Agent 标注但**绝不能**自主"修改合同"。

**场景 D**：后台日志清理。每天凌晨 3 点删 7 天前的日志。**正确自主度：L4+（全自主，异常时通知）**——人类不需要参与常规清理。

**问题本质**：自主度不是一个系统级开关（"开启自主模式"），而是**按任务、按风险、按信任动态路由**的梯度。本文回答：怎么设计这个梯度？怎么让它随时间演化？

## §1 第一性原理：能力 ≠ 自主度

### 1.1 二轴分离

DeepMind 2023 "Levels of AGI" 论文的核心贡献：**Capability 和 Autonomy 是正交的两个轴**。

```
Autonomy（自主度）
   ▲
   │  ┌─────────────────────┐
L5 │  │ 高能力+高自主        │ ← 理想态（但风险最高）
   │  │ (全自主研究 Agent)   │
   │  ├─────────────────────┤
L3 │  │ 高能力+中自主        │ ← 2026 生产前沿
   │  │ (Devin, Codex CLI)  │
   │  ├─────────────────────┤
L1 │  │ 高能力+低自主        │ ← 保守但安全
   │  │ (ChatGPT Canvas)    │
   │  └─────────────────────┘
   └──────────────────────────→ Capability（能力）
```

**设计含义**：即使你的 Agent 拥有 o3 级别的推理能力，你依然可以（且应该）选择限制它的自主度。自主度是**治理决策**，不是技术决策。

### 1.2 SAE 类比：为什么 5 级分类有效

汽车行业的 SAE J3016 六级自动驾驶标准已经被验证了 10 年。AI Agent 领域在 2025-2026 年收敛到了类似的分级框架（[arXiv:2506.12469](https://arxiv.org/html/2506.12469v1)、CSA 2026、Knight First Amendment Institute 2025）：

| 级别 | SAE 类比 | Agent 中的用户角色 | 控制方式 |
|---|---|---|---|
| **L1** | 驾驶辅助（车道保持） | 操作者（Operator） | 用户逐步触发 |
| **L2** | 部分自动（ACC+LKA） | 协作者（Collaborator） | 用户可随时接管 |
| **L3** | 有条件自动 | 顾问（Consultant） | Agent 主导，人提意见 |
| **L4** | 高度自动 | 审批者（Approver） | 只在红区触发人工 |
| **L5** | 完全自动 | 观察者（Observer） | 仅有紧急停止 |

**关键洞察**（Zylos 2026）：85% 的逐步准确率在 10 步 workflow 中只有 ~20% 的端到端成功率。要在 10 步中达到 80% 的端到端成功，需要**每步 97.9%** 的准确率。这就是为什么 L5 在 2026 仍然没有可信的生产部署。

## §2 五级自主度详解

### 2.1 L1：用户即操作者（Copilot 建议模式）

| 维度 | 描述 |
|---|---|
| 用户角色 | 发起每个动作，Agent 只响应请求 |
| Agent 行为 | 被动建议，不主动行动 |
| 典型交互 | 代码补全、inline suggestion、问答 |
| 必要控制 | 接受/拒绝建议 |
| 产品示例 | GitHub Copilot 补全、Cursor Tab、ChatGPT Canvas |
| 适合场景 | 延迟敏感、试错成本低、用户是领域专家 |

**设计要点**：
- 建议必须 <200ms 出现（否则打断心流）
- 接受一键、拒绝零键（忽略=拒绝）
- 不需要展示推理（太慢）
- 不需要计划预览（单步）

### 2.2 L2：用户即协作者（Pair Programming 模式）

| 维度 | 描述 |
|---|---|
| 用户角色 | 和 Agent 共同规划、分工执行 |
| Agent 行为 | 可独立执行小任务，但用户能随时修改 Agent 工作 |
| 典型交互 | Agent 写一段代码→用户修改→Agent 继续 |
| 必要控制 | 实时编辑 Agent 产出、控制权互传 |
| 产品示例 | Claude Code 默认模式、Cursor Agent mode |
| 适合场景 | 任务需要领域知识+Agent 执行力的结合 |

**设计要点**：
- 共享进度表示（用户和 Agent 都能看到全局）
- 控制权显式转移（"我来做这步" / "你继续"）
- 用户可以**直接修改** Agent 的产出（不只是"请求修改"）
- Agent 在修改后能继续（不重新开始）

### 2.3 L3：用户即顾问（Delegated Execution 模式）

| 维度 | 描述 |
|---|---|
| 用户角色 | 提供方向和偏好，审查结果 |
| Agent 行为 | 自主规划+执行，定期汇报，请求反馈 |
| 典型交互 | Agent 完成一个 PR → 人 review → 合并/修改意见 |
| 必要控制 | 丰富的反馈界面（不只是 approve/reject） |
| 产品示例 | GitHub Copilot Coding Agent、Gemini Deep Research、Replit Agent |
| 适合场景 | 任务定义清晰、可验证、人工逐步审查不经济 |

**设计要点**：
- 结果导向而非过程导向的审查
- Agent 主动请求偏好/专业知识输入（不是等用户发现问题）
- 用户**不需要理解每个中间步骤**，只审查最终产物
- 但中间步骤必须**可访问**（审计需要）

### 2.4 L4：用户即审批者（Background Agent 模式）

| 维度 | 描述 |
|---|---|
| 用户角色 | 预设审批条件，只在例外时介入 |
| Agent 行为 | 完全自主执行，仅在预设条件或异常时暂停 |
| 典型交互 | Agent 在后台工作 → 完成通知 / 异常时升级 |
| 必要控制 | 可定制审批条件 + 操作审计 |
| 产品示例 | Devin 2.0、Claude Code Background + git worktree |
| 适合场景 | 高重复性、规则清晰、可回滚、用户信任已建立 |

**设计要点**：
- 预先定义审批触发条件（不是事后问）
- 操作日志完整+可审计
- 异常升级路径清晰
- 安全隔离（sandbox / worktree / container）
- 凭据存储安全（见 6.6）

### 2.5 L5：用户即观察者（Full Autonomous 模式）

| 维度 | 描述 |
|---|---|
| 用户角色 | 仅监控，不参与 |
| Agent 行为 | 完全自主，包括目标设定和策略选择 |
| 典型交互 | Dashboard 监控 + 紧急停止 |
| 必要控制 | Emergency off switch |
| 产品示例 | 无可信生产部署（Voyager/AI Scientist 是研究系统） |
| 适合场景 | 2026 无适合场景 |

**为什么 L5 在 2026 不可行**（Zylos 研究）：
- 74% 的企业无法解释 Agent 的结论
- 62% 在 2025 经历了 agent-induced incident
- 法律框架不支持无人类问责的 AI 行为（EU AI Act 2026 生效）
- 错误放大效应使得长链自主任务在当前能力水平不可靠

## §3 动态自主度路由：按什么决定用哪一级？

### 3.1 不是系统级开关，是请求级路由

**最大反模式**：在系统设置里放一个"自主模式：低/中/高"开关。

正确设计是**每个请求/每个步骤动态评估自主度**：

```python
def route_autonomy(request, user_profile, step):
    """Per-request autonomy routing"""
    risk = assess_risk(step)
    trust = user_profile.trust_level
    reversibility = step.is_reversible
    
    if risk == "critical" or not reversibility:
        return Level.L2  # 必须人工协作
    
    if risk == "high":
        return Level.L3  # Agent 做，人审结果
    
    if trust >= TRUST_THRESHOLD_HIGH and risk == "low":
        return Level.L4  # 后台自动
    
    return Level.L2  # 默认协作
```

### 3.2 路由的 4 个维度

| 维度 | 低自主度 ← | → 高自主度 |
|---|---|---|
| **后果可逆性** | 不可逆（删库、发邮件） | 完全可逆（生成文件、写 draft） |
| **错误成本** | 高（财务、法律、声誉） | 低（格式、建议） |
| **用户信任** | 新用户 / 首次使用 | 老用户 / 历史无事故 |
| **可验证性** | 无法自动验证 | 有测试/工具可验证 |

**决策矩阵**：

| 可逆 + 低错误成本 + 高信任 + 可验证 → **L4 自动执行** |
|---|
| 不可逆 + 高错误成本 + 任何信任 + 不可验证 → **L2 必须协作** |
| 中间态 → **L3 自主执行 + 人审结果** |

### 3.3 同一任务内的自主度变化

一个复杂任务的不同步骤可能需要不同自主度：

```
任务："帮我重构支付模块并部署到 staging"

步骤 1: 分析现有代码        → L4（只读，无风险）
步骤 2: 生成重构方案        → L3（自主生成，人审方案）
步骤 3: 修改代码            → L3（自主写，人审 diff）
步骤 4: 跑测试              → L4（自动执行，报告结果）
步骤 5: 部署到 staging      → L2（需要确认环境+凭据）
步骤 6: 部署到 production   → L1（人操作，Agent 只提供指令）
```

**设计含义**：自主度不是任务级的，是**步骤级的**。

## §4 Progressive Delegation：信任是怎么积累的

### 4.1 CSA "晋升模型"（2026）

Cloud Security Alliance 的 Agentic Trust Framework 用职级隐喻描述信任积累：

| 阶段 | 隐喻 | Agent 行为 | 晋升条件 |
|---|---|---|---|
| **Intern** | 实习生 | 每步都确认 | 完成 N 次无事故 |
| **Associate** | 初级 | 绿区自动，黄区确认 | 30 天无异常 |
| **Senior** | 资深 | 绿+黄自动，红区确认 | 90 天无事故 + 用户主动提升 |
| **Principal** | 首席 | 仅红区确认 | 用户显式授权 + 治理审批 |

**关键设计**：晋升需要 5 个同时满足的条件（不仅是时间）：
1. 累计操作数达阈值
2. 成功率 > 阈值
3. 用户无投诉
4. 无安全事件
5. 用户主动确认提升

### 4.2 自动降级（Anti-Escalation）

比晋升更重要的是**降级触发**：

| 触发 | 降级幅度 | 恢复条件 |
|---|---|---|
| 单次严重错误（PII 泄露、资损） | 直接回 Intern | 人工重审+重新授权 |
| 连续 3 次用户 reject | 降 1 级 | 5 次连续成功 |
| 异常检测告警 | 降 1 级 + 暂停 | 根因分析完成 |
| 用户主动降级 | 按用户指定 | 用户主动恢复 |
| 底模更换/版本升级 | 回到 Associate | 10 次无事故 |

**底模更换必须重置信任**——新模型行为特征未知。这是很多团队忽略的：升级模型后没重置权限，新模型的不同行为在高自主度下直接造成事故。

### 4.3 Claude Code 的纵向数据

arXiv:2604.14228 提供了真实的信任积累曲线：

- 前 50 次会话：auto-approve 率 ~20%（用户谨慎）
- 50-200 次：逐步上升到 ~30%
- 200-750 次：稳定在 ~40%

**注意**：即使高频用户也只有 40% 的操作选择 auto-approve——这意味着**多数操作用户仍然选择审查**。Progressive Delegation 不是终点是 L5，而是找到用户舒适的均衡点（通常是 L3-L4 之间）。

## §5 架构实现：权限系统怎么建

### 5.1 两种权限哲学

Zylos 2026 的比较：

| 维度 | 规则引擎 (Codex CLI) | ML 分类器 (Claude Code Auto) |
|---|---|---|
| 判断方式 | 白名单/黑名单 + OS 沙箱 | 神经网络分类操作风险 |
| 可审计性 | ✅ 确定性规则 | ❌ 不透明 |
| 鲁棒性 | 规避容易（满足字面规则但语义违规） | 对抗性 prompt 更鲁棒 |
| 可解释性 | ✅ 能解释为什么拒绝 | ❌ 无法解释 |
| 适应性 | ❌ 每个新工具手动加规则 | ✅ 泛化到新操作 |

**推荐**：混合架构——**规则引擎做硬约束（红线不可逾越），ML 分类器做软决策（灰区智能路由）**。

### 5.2 权限定义 schema

```yaml
permissions:
  green_zone:  # L4 自动执行
    - pattern: "read_file:*"
    - pattern: "search:*"
    - pattern: "run_test:*"
    - pattern: "create_file:${workspace}/**"
  
  yellow_zone:  # L3 执行后审查
    - pattern: "edit_file:${workspace}/**"
    - pattern: "git_commit:*"
    - pattern: "api_call:internal:*"
  
  red_zone:  # L2 必须预审批
    - pattern: "delete_file:*"
    - pattern: "api_call:external:*"
    - pattern: "deploy:*"
    - pattern: "send_email:*"
    - pattern: "access:credentials:*"

  forbidden:  # 绝对禁止
    - pattern: "rm_rf:*"
    - pattern: "access:production_db:write"

trust_level: associate  # intern | associate | senior | principal

escalation:
  on_forbidden: deny_and_explain
  on_red_zone: pause_and_ask
  on_uncertainty: route_to_yellow
```

### 5.3 步骤级权限检查流程

```python
class AutonomyGate:
    def __init__(self, permissions, trust_level):
        self.permissions = permissions
        self.trust = trust_level
    
    def check(self, action) -> Decision:
        zone = self.classify_zone(action)
        
        if zone == "forbidden":
            return Decision(
                action="deny",
                reason=f"Action '{action.type}' is not permitted",
                alternatives=self.suggest_alternatives(action)
            )
        
        if zone == "red":
            return Decision(
                action="pause_and_ask",
                preview=self.render_preview(action),
                consequences=self.assess_consequences(action),
                rollback_plan=self.plan_rollback(action)
            )
        
        if zone == "yellow":
            if self.trust >= TrustLevel.SENIOR:
                return Decision(action="execute_and_log")
            return Decision(action="execute_then_review")
        
        # green zone
        return Decision(action="auto_execute")
    
    def classify_zone(self, action):
        """规则引擎 + ML 分类器混合"""
        # 先检查硬规则
        rule_result = self.rule_engine.match(action)
        if rule_result != "unknown":
            return rule_result
        
        # 灰区用 ML 分类器
        risk_score = self.risk_classifier.predict(action)
        if risk_score > 0.8:
            return "red"
        elif risk_score > 0.4:
            return "yellow"
        return "green"
```

## §6 七场景决策指南

| 场景 | 建议自主度 | 理由 | 关键控制 |
|---|---|---|---|
| **代码补全** | L1 | 延迟 <100ms，单步，可忽略 | 接受/忽略 |
| **Bug 修复 (单文件)** | L2-L3 | 需审查 diff，但范围可控 | diff 预览 + approve |
| **跨文件重构** | L3 | 逐文件审查不经济 | PR review |
| **数据分析报告** | L3-L4 | 只读操作，结果可验证 | 结果审查 |
| **发送客户邮件** | L2 | 不可逆+外部影响 | 预览全文+确认 |
| **CI/CD 部署** | L2 (prod) / L4 (staging) | 生产不可逆，staging 可 | 环境分级 |
| **日志清理/备份** | L4 | 高重复、低风险、可逆 | 异常通知 |

## §7 错误放大效应与设计应对

### 7.1 复合准确率问题

Zylos 2026 的核心数据：

```
per_step_accuracy = 0.85
steps = 10
end_to_end_success = 0.85 ** 10 = 0.197  ≈ 20%

# 要达到 80% 端到端成功率
required_per_step = 0.80 ** (1/10) = 0.979  ≈ 97.9%
```

这解释了为什么高自主度需要极高的单步准确率。2026 的 frontier 模型在简单任务上可以 >95%，但在需要领域知识的复杂步骤上只有 60-80%。

### 7.2 Context Drift：65% 失败的根因

Zylos 的数据显示多步任务失败的主要原因：

| 类型 | 占比 | 表现 |
|---|---|---|
| **目标漂移** | 35.9% | Agent 做着做着偏离了原始目标 |
| **上下文污染** | 35.6% | 之前步骤的错误信息积累 |
| **推理漂移** | 17.0% | 逻辑链在长 context 中退化 |
| 其他 | 11.5% | — |

**应对策略**（与自主度设计相关）：
- **Checkpoint + 目标重申**：每 N 步重新对齐原始目标
- **Memory 蒸馏**：压缩执行历史为精炼信念（+21% 稳定性）
- **人工检查点**：在关键节点插入人工审查（即降低该步自主度到 L2）

### 7.3 自主度作为缓解策略

错误放大效应意味着：**不是所有步骤都应该相同自主度**。

策略：
- 链条中的**前几步**用高自主度（通常是信息收集，可逆）
- **中间决策节点**用低自主度（方向性决策，影响后续）
- **后几步执行**根据可逆性决定

```
 L4    L4    L2    L3    L3    L2    L4
 读取→ 搜索→ 决策→ 编写→ 编写→ 审批→ 测试
```

## §8 Multi-Agent 场景的自主度治理

### 8.1 问题：谁给谁授权？

当 Agent A 调用 Agent B 时，B 的自主度从哪来？

arXiv:2506.12469 的框架提出：Agent 的"用户"可以是另一个 Agent。这意味着权限可以**传递**——但传递必须受限：

```
User → grants L3 to Agent A
Agent A → can only grant ≤ L2 to Agent B (不能超过自己被授予的级别 - 1)
Agent B → can only grant ≤ L1 to Agent C
```

**原则**：**权限只能递减传递，永远不能放大**。

### 8.2 Multi-agent 可见性

Microsoft Design 原则："Actions an agent takes, including collaboration with others, are visible and controllable by the user"。

在多 Agent 系统中，用户必须能看到：
- 哪些 Agent 参与了这个任务
- 每个 Agent 的自主度级别是什么
- Agent 之间传递了什么信息
- 每个 Agent 的操作日志

这对应 9.1 透明性原则在 multi-agent 场景的扩展。

## §9 反模式表

| # | 反模式 | 后果 | 解药 |
|---|---|---|---|
| 1 | 系统级"自主度开关"（高/中/低） | 一刀切无法匹配场景 | 请求级动态路由 |
| 2 | 能力提升→自动提升自主度 | 新能力未经验证就高权限 | 能力与自主度解耦 |
| 3 | 升级底模不重置信任 | 新模型行为未知+高权限=事故 | 模型变更→降级到 Associate |
| 4 | L5 YOLO（"反正有回滚"） | 不可逆外部操作无法回滚 | L4 是 2026 天花板 |
| 5 | 晋升只看时间不看表现 | 坏 Agent 混上 Senior | 5 门同时满足 |
| 6 | 只有升级没有降级 | 一次事故后用户失去信任但权限未变 | 自动降级触发器 |
| 7 | 所有步骤同一自主度 | 忽略步骤间风险差异 | 步骤级路由 |
| 8 | 子 Agent 权限放大 | A 给 B 的权限超过 A 自己 | 递减传递原则 |
| 9 | 自主度高但审计不全 | 出事后无法溯源 | 自主度与日志粒度正相关 |
| 10 | 用户无法手动降级 | 不信任时只能停用 | 随时可调的信任旋钮 |

## §10 S0 → S5 演进路线

| 阶段 | 自主度设计 | 信任机制 | 产品示例 |
|---|---|---|---|
| **S0** | 纯问答，无执行 | 无 | ChatGPT 早期 |
| **S1** | L1 建议模式 | 接受/忽略 | Copilot 补全 |
| **S2** | L2 固定协作 | 每步确认 | Cursor Agent basic |
| **S3** | L3 + 绿/黄/红分层 | 静态权限配置 | Claude Code、Codex CLI |
| **S4** | 动态路由 + Progressive Delegation | CSA 晋升模型 + 自动降级 | 内部成熟平台 |
| **S5** | 多 Agent 权限治理 + 自主度证书 | 第三方评估 + autonomy certificate | 未来态 |

**你在哪？** 如果你的 Agent 所有操作用同一套确认规则→S2。如果按风险分区→S3。如果能根据用户历史动态调整→S4。

## §11 一页总结：抄走这页就够

| 问题 | 答案 |
|---|---|
| **自主度分几级？** | L1 操作者 → L2 协作者 → L3 顾问 → L4 审批者 → L5 观察者 |
| **2026 生产天花板？** | L4（有隔离+审计）；L5 无可信部署 |
| **怎么决定用哪级？** | 4 维度路由：可逆性 × 错误成本 × 用户信任 × 可验证性 |
| **系统级还是步骤级？** | 步骤级——同一任务内不同步骤自主度不同 |
| **信任怎么积累？** | CSA 晋升模型：5 门同时通过才升级 |
| **何时降级？** | 严重错误→直接回 Intern；底模变更→降 Associate |
| **子 Agent 权限？** | 只能递减传递，永远不能放大 |
| **错误放大怎么办？** | 关键决策节点降低自主度（L2），信息收集步保持高（L4） |
| **第一性原则** | 能力 ≠ 自主度；自主度是治理决策，不是技术决策 |
