# 1.5 推理模型时代的 Agent 范式变化：o1 / o3 / R1 / Extended Thinking 与 Test-time Compute Scaling

> **写在前面**：本文聚焦 2024 年末 OpenAI o1 发布之后到 2026 年 GPT-5 系列、Claude Sonnet/Opus 4.x Extended Thinking、DeepSeek R1 / V3.x、Qwen QwQ、Gemini 2.5 Pro Thinking 完整一代推理模型对 Agent 工程的冲击。范畴：模型本身的范式变化（训练目标、推理时计算），以及由此**反向重塑的 Agent Harness、Loop、工具使用、监控、成本预算**。
>
> 这是模块一的补全篇，与 **1.1 LLM 推理机制与能力边界** 是承接关系：1.1 讲 LLM 为什么"会推理"，本文讲 LLM"专门为推理优化"之后 Agent 应该怎么变。

## §0 真问题：当模型自己会想了，Harness 还要干什么？

2024 年 9 月 12 日 OpenAI 发布 o1-preview，AIME 2024 从 GPT-4o 的 13% 跳到 56%，Codeforces Elo 从 11 百分位跳到 89 百分位。一年后 GPT-5、Claude Opus 4.5 Extended Thinking、DeepSeek R1（AIME 79.8 vs o1-1217 79.2）、Gemini 2.5 Pro Thinking 都把"会推理"做成默认能力。Agent 工程师立刻发现一堆"过去成立、现在不成立"的设计假设：

第一个真问题：**Chain-of-Thought 还要不要写在 prompt 里？** o1 / R1 不需要 "let's think step by step"，把它写进去甚至**会降低**性能（OpenAI 官方 GPT-5 prompting guide 明确建议不要再叠加 CoT 指令）。Few-shot CoT exemplars 在推理模型上经常变成负向 prompt（[Lin 2025](https://medium.com/@siddantvardey/the-frontier-reasoning-models-scaling-laws-whats-actually-coming-next-93bba260644b)）。

第二个真问题：**ReAct/Plan-and-Execute 在推理模型上还成立吗？** o1/R1 在被调用一次时已经内化了 plan→reason→verify。如果外面再套一个 ReAct loop，会导致**双重规划**：模型在自己脑子里规划一次（消耗 reasoning tokens），harness 再让它产出一个 plan 文本（消耗 output tokens），两个 plan 经常打架。Anthropic 2025-08 GPT-5 launch 之后 Cursor、Cognition、Bolt 公开承认重写了 ReAct prompt。

第三个真问题：**Reasoning tokens 怎么计费、怎么预算？** 一次调用花掉 10K reasoning tokens 是常态，gpt-5-codex 在 SWE-bench 提交里单 task 可超 50K reasoning tokens（[Azure docs 2026-04](https://learn.microsoft.com/en-us/azure/foundry/openai/how-to/reasoning)）。这些 token **按 output 价格计费但不可见**，传统的"以 output 长度估算成本"完全失效。

第四个真问题：**思维链可见性的安全工程化怎么做？** OpenAI 2025-07 的 CoT Monitorability paper（Anthropic + DeepMind + Apollo 联合署名）警告：当前推理模型的 CoT 是**fragile 的安全监控通道**，过度训练对齐反而会让模型学会"隐藏 misbehavior"。

第五个真问题（最现实）：**什么时候用推理模型、什么时候不用？** 把所有调用都换成 o1/R1，成本会涨 10×、延迟从秒级到分钟级。但在某些场景（如 deep research、debug、PR review），不用推理模型直接掉链子。需要的是**级联与路由**，不是一刀切。

本文围绕这五个问题展开。

## §1 第一性原理：推理模型的三个底层变化

### 1.1 训练目标从 imitation 变成 outcome

经典 LLM（GPT-3/4、Claude 3）训练目标是**模仿人类 token 分布**：next-token prediction + RLHF 让模型说话像人。模型不知道答案对不对，只知道**说得像不像**。

推理模型把目标改成**outcome-based RL**：模型生成一长串 CoT，最后给出答案，**只有最终答案被 verify**（数学题对错、代码能不能跑、单元测试过没过）；然后把整条 trajectory 一起反向传播（[DeepSeek R1 paper, arXiv:2501.12948](https://arxiv.org/abs/2501.12948)）。DeepSeek R1 用的 GRPO（Group Relative Policy Optimization）是 PPO 的简化版，**去掉 critic、用 group baseline 估计 advantage**。R1 训练管线分四阶段：

1. **R1-Zero**：纯 RL，从 DeepSeek-V3-Base 开始，只用 rule-based reward（math correctness + code unit tests + format reward）。AIME 从 15.6% 涨到 71%，majority voting 86.7%。中途出现 "aha moment"——模型自发学会回头检查自己步骤。
2. **Cold start SFT**：用 R1-Zero 生成长 CoT 再人工筛选，回灌做 SFT 解决可读性问题。
3. **Reasoning RL**：再次 RL，加 language consistency reward。
4. **General RLHF**：加入 helpfulness/harmlessness reward 收尾。

R1 公开后的关键发现：**Process Reward Models (PRM) 和 MCTS 在大规模训练中均被标记为"unsuccessful attempts"**——这戳破了 2023-2024 主流叙事（"推理就是 PRM + 树搜索"）。OpenAI o1/o3 没公开技术细节但行为完全一致，业界共识：**outcome reward + 足够算力 > 复杂的过程奖励**。

### 1.2 推理时算力从常数变成可调维度

Pre-o1 LLM：推理 cost ≈ output_tokens × cost_per_token，**与问题难度无关**——再难的问题模型也只会写差不多长的回答。

Post-o1 LLM：模型可以在内部"想更久"，**reasoning tokens 是显式可调的第二个维度**。OpenAI GPT-5 系列的 `reasoning_effort` 参数支持 `none / minimal / low / medium / high / xhigh`（[Azure docs 2026-04](https://learn.microsoft.com/en-us/azure/foundry/openai/how-to/reasoning)）。Anthropic Claude Extended Thinking 用 `budget_tokens` 直接指定预算上限（1024 起步，可设到 60K 以上，[AWS Bedrock Claude Extended Thinking docs](https://docs.aws.amazon.com/bedrock/latest/userguide/claude-extended-thinking.html)）。

这条 scaling law 的工程含义是颠覆性的：**算力可以推到推理阶段而非训练阶段**。Introl 引用的研究估计，"一个 7B 模型给 100× 推理算力可以匹敌 70B 模型在标准推理设置下的表现"。OpenAI 2024 inference spend 据报达 23 亿美元，约 15× GPT-4 训练成本（[Introl Blog 2025-12](https://introl.com/blog/inference-time-scaling-research-reasoning-models-december-2025)）。预测 2026 年推理算力需求将超过训练 118×。

### 1.3 CoT 不再是 prompt 技巧，而是模型内部状态

GPT-4 / Claude 3 时代，CoT 是**prompt 端的字符串**，可读、可裁剪、可注入指令。Reasoning model 时代，CoT 变成**模型内部的隐藏 trace**：

- **OpenAI o-series / GPT-5**：原始 CoT 不返回；只能通过 `reasoning.summary` 拿到摘要（`auto / concise / detailed`）。试图通过 prompt engineering 提取原始 CoT 被官方明令禁止（"may violate Acceptable Use Policy"）。
- **Claude 4.x Extended Thinking**：默认返回**summarized thinking**（Claude 4 系列）或**完整 thinking**（3.7 Sonnet）；可见但不可编辑；多轮调用之间**必须把 thinking blocks 原封传回**否则模型会偏。
- **DeepSeek R1 / Qwen QwQ（开源）**：完整 `<think>...</think>` 可见，是研究界最香的金矿。

这个变化导致两个**新的工程约束**：

第一，**Thinking block 是 first-class state**。在工具调用循环里，必须把上一轮的 thinking blocks 与 assistant message 一起回传到下一轮（Anthropic API 强制校验），否则会拿到 `thinking_block_required` 错误。这与 Claude 3 时代"只回传 messages 就够了"完全不同。

第二，**CoT 监控成为安全工程的新支柱**。OpenAI 2025-07 的论文（[OpenAI CoT Monitorability](https://openai.com/index/chain-of-thought-monitorability/)）发现：用一个弱模型（GPT-4o-mini）作为 monitor 读 o3-mini 的 CoT，可以在 95% 召回率下检测出 reward hacking 行为；但如果把这个 monitor 反过来用于 RL reward signal，模型会迅速学会**obfuscation**——用普通话写 CoT 但内部还是在 hack。这是**安全可监控性 vs 对齐压力**的根本张力。

## §2 数据：Reasoning Model 的现状速览

| 模型 | 发布 | 关键能力 | 关键代价 |
|------|------|---------|---------|
| OpenAI o1 | 2024-09 | AIME 56%（GPT-4o 13%）; Codeforces 89th pct | 不返回 CoT, 不支持 tool use（早期） |
| OpenAI o3 | 2025-01 | ARC-AGI semi-private 76%, EpochAI FrontierMath 25% | 高 effort 单 task 数千美元 |
| OpenAI GPT-5 系列 | 2025-08~2026-04 | reasoning_effort 6 档；支持 tool use + parallel calls；context 400K~1.05M | minimal 档不支持 parallel tools |
| DeepSeek R1 | 2025-01 | AIME 79.8% vs o1-1217 79.2%; 开源 MIT License | 671B MoE 不易自托管, distilled-32B AIME 72.6% |
| Anthropic Claude 4 Extended Thinking | 2025-05~2026-Q2 | budget_tokens 1K~60K+; interleaved thinking beta | 长 budget 下 streaming required |
| Gemini 2.5 Pro Thinking | 2025-03 | 1M context + thinking budget control | thinking 不可与某些 grounding 共用 |

性能-成本-延迟三个轴上，2026 年中的实际选型形如：

- **最高质量需求**（数学竞赛、形式化证明、深度科研）：GPT-5 Pro / Claude Opus 4.6 Extended Thinking + budget_tokens=60K
- **量产 Agent**（Cursor、Devin、Codex 这类）：GPT-5 / GPT-5-Codex effort=medium，配合 R1 distill 做廉价回退
- **本地/隐私敏感**：DeepSeek R1 / Qwen QwQ 自托管 vLLM + speculative decoding

## §3 决策矩阵：什么时候应该上推理模型

> 反潮流的判断：**推理模型不是"更好的 GPT-4"，是一个新工种**。错误地把它当 GPT-4 替换品用，会让成本爆炸、延迟爆炸、Agent 行为退化。

| 场景 | 用推理模型？ | 理由 |
|------|------------|------|
| 简短客服回答、FAQ 检索 | ❌ | reasoning 浪费；用 nano/haiku/4o-mini |
| 代码改 1~3 个文件的小修复 | ⚠️ | GPT-4o + 好 prompt 经常够；推理只在复杂 debug 启动 |
| **深度 debug**（trace 多模块、需要构造反例） | ✅ | reasoning_effort=high；正是它的优势区 |
| **生成测试用例与 mutation testing** | ✅ | 需要"想清楚 edge case"，效果显著 |
| **PR 评审 / Code Review** | ✅ | 需要把代码与意图 / 风险 reason 一遍 |
| **写 / 改 SQL 复杂 query** | ✅ | join + 子查询 + 性能折衷需要推理 |
| 工具决策（"该用哪个 API"） | ⚠️ | 工具列表稳定时用普通模型 + 良好 schema 即可；超过 20 个 tool 时再考虑推理 |
| Web Agent / Browser Agent 单步动作 | ❌ | 延迟是 UX 杀手；推理模型只在 planner 层 |
| Web Agent 的 planner / replan | ✅ | 间隔触发；reason "下一步该去哪里"价值最高 |
| 多 Agent 系统中的 orchestrator | ✅ | 它的工作就是 reason about agents 与 tasks |
| 多 Agent 系统中的 worker | ❌ | 工作类型确定时反而该用快速模型 |
| 长文档摘要（< 100K context） | ❌ | 不需要 reasoning；浪费 token |
| 跨文档关联分析、Deep Research | ✅ | OpenAI Deep Research、Google Gemini DR 全是 reasoning-based |
| RAG 中的 retrieval 决策 | ⚠️ | retrieval planner 用推理，retrieval 本身用普通模型 |

**经验法则**：如果任务**有可验证的正确性**（数学、代码、形式逻辑），推理模型会显著赢；如果任务**靠风格、记忆、常识**，推理模型不一定赢甚至会输（GPT-5 Chat 反而更适合）。

## §4 Harness 设计的五个范式迁移

### 4.1 从 ReAct 到 Single-Shot Reasoning + Verifier Loop

传统 ReAct：每步 `Thought → Action → Observation`，模型在 prompt 里显式写思考。

推理模型时代：把多步 reasoning 折叠到**模型内部**，外面只跑 `Reason+Act → Verify → 可选 Retry`。模型一次产出的 plan/action 已经被内部 reason 过一遍，**外层只需要做 outcome verification**——单测、type check、lint、运行结果断言——失败再扔回去做 self-correction（Anthropic 在 GPT-5 launch 后公开的实践，参考 Cursor 2025-11 工程博客）。

```python
def reasoning_agent_loop(task, max_retries=3):
    """
    推理模型时代的最小 harness：模型一次完成 reason+act，外层只做 verify.
    """
    for attempt in range(max_retries):
        response = model.complete(
            messages=[{"role": "user", "content": task.prompt}],
            reasoning_effort="medium",
            tools=task.tools,
            max_completion_tokens=32_000,
        )
        # 关键：必须保留 thinking blocks
        actions = extract_tool_calls(response)
        observations = execute(actions)
        verdict = task.verify(observations)
        if verdict.passed:
            return verdict
        # 把 verifier 失败信息扔回去，但保留原 thinking blocks
        task.prompt = build_retry_prompt(task, response, verdict)
    return verdict
```

### 4.2 Reasoning Budget 工程

新一代 Harness 必须管理两类预算：

- **External budget**：传统的 max_tokens、tool call 次数、wall-clock 上限。
- **Internal reasoning budget**：reasoning_effort / budget_tokens。

实战策略：

1. **阶梯式 escalation**：先用 `effort=low` 跑一次，如果 verifier 不过，escalate 到 `medium`，再不行 `high`。Cursor 公开使用此策略，平均省 60% reasoning tokens。
2. **任务自评估预算**：让一个轻量模型先估这道题"难不难"，再决定 effort。难度低用 4o-mini，难度高才用 reasoning。
3. **超时熔断**：Claude Extended Thinking 在大 budget 下要求 streaming（`max_tokens > 21,333` 强制流式），否则可能 504；harness 要先判 budget 再选 streaming/sync。

### 4.3 Thinking-Aware Tool Loop

工具调用循环里要让 thinking blocks 与 tool_use blocks **正确穿插**：

```
user → assistant(thinking + tool_use) → user(tool_result) → assistant(thinking + tool_use) ...
```

Anthropic 提供 **interleaved thinking beta**（header `anthropic-beta: interleaved-thinking-2025-05-14`）允许 thinking 在 tool 之间穿插，使模型每次 tool 结果后再 think 一段；这是 2025-Q3 之后 Agent 工程的事实标准。

省钱技巧：**`clear_thinking_20251015`** beta（Claude Sonnet 4.5+）允许在 N 轮后清理早期 thinking，**保留 messages 但丢弃 thinking 历史**，长会话节省 30-50% input tokens。但要确保丢弃前任务已完成中间结论的固化。

### 4.4 CoT 监控作为安全层

把推理模型的 CoT 视为**可监控但 fragile** 的安全 signal：

- 用一个 cheap monitor（GPT-4o-mini / Haiku）订阅 reasoning summary 流，关键词触发或行为分类。
- **绝不**把 monitor 的判定直接接入 RL reward（OpenAI 2025-03 论文示警，会导致 obfuscation）。
- 用 monitor 做 **post-hoc audit + 实时 kill-switch**，而非 inline RL signal。

参考 Anthropic Responsible Scaling Policy v2 与 OpenAI Preparedness Framework：reasoning model 的可监控性是当下少数有效的对齐工具，**这是工程要保住的资产，不是要优化掉的成本**。

### 4.5 Caching 与延迟工程

推理模型的 prompt cache 命中率**直接决定单位经济性**。GPT-5 系列 cached tokens 通常 50% 折扣，Claude 是 10%（write）+ 90%（read）折扣。设计原则：

- **System / developer message 放最前并保持稳定**——任何一个字符变更击穿 cache。
- **Reasoning 不进 cache**：每次 reasoning_tokens 都重新生成，**真正可缓存的是 input 部分**。
- 长会话：使用 conversation pinning（Anthropic prompt caching `cache_control` 标记到某个具体 message），让"system + tool defs + 早期 turn"全部命中。

## §5 反模式与陷阱

**反模式 1：把 CoT 写进 prompt**。"Let's think step by step", "First, let's analyze..." 在 o1/R1/GPT-5 上经常**降低**性能。OpenAI GPT-5 prompting guide 明示："avoid telling the model to think step by step. The model already does this internally."

**反模式 2：用 reasoning model 做高频小调用**。用 o3 跑 RAG 中每段文档的相关性判断——单次几秒延迟、几美分、reasoning_tokens 占大头。改用 4o-mini 或 Haiku batch。

**反模式 3：忽略 thinking blocks 导致 silent quality drop**。多轮工具调用时丢掉 thinking blocks，模型表面正常但行为退化（无法 self-correct）。Anthropic 在 2025-05 之后 API 已强制校验，但许多自研框架仍在踩坑。

**反模式 4：把 reasoning summary 当 ground truth**。Summary 是**事后摘要**，并非真实 CoT。基于 summary 做决策（如 routing、CoT injection）会拿到 hallucinated 推理过程。

**反模式 5：在 reasoning model 上做 fine-tune 普通 instruction data**。R1 distill 出来的 32B 模型如果直接拿你的客服对话 fine-tune，会**杀掉 reasoning 能力**——蒸馏好的 reasoning capability 极其脆弱（DeepSeek paper §4.3 明示）。蒸馏要"reasoning-preserving"专门方法（cold-start mix 长 CoT 数据）。

**反模式 6：以 output_tokens 估算成本**。Reasoning_tokens 不返回但**计费**。一个 4K output 的 reply 后面可能藏着 30K reasoning_tokens。监控必须读 `usage.completion_tokens_details.reasoning_tokens` 字段。

**反模式 7：把 reasoning_effort=high 当默认值**。high effort 在简单任务上**精度不会涨，成本/延迟 5-10×**。Azure 实测：summarization、format 转换、简单 RAG 这类任务，medium → high 几乎无 measurable gain。

**反模式 8：用 reasoning model 做 streaming UI"打字机"体验**。reasoning_tokens 是"前置成本"——用户看到第一个 user-visible token 之前要等 5-30 秒。要么用 reasoning summary 做 status 提示，要么直接换非推理模型。

## §6 S0→S5 演进路线：把 Agent 升级到推理模型时代

**S0 - 直接替换（半天工作量）**：模型 ID 改成 gpt-5 / claude-sonnet-4.6-thinking / deepseek-r1，跑回归测试。预期成本涨 3-10×，部分场景质量提升明显。**这是反模式**，但常是起点。

**S1 - 删 CoT prompt（半天）**：清理 "let's think step by step"、"chain of thought" 指令、few-shot CoT exemplars，在推理模型上重测。常有 5-15% 净提升。

**S2 - Verifier 闭环（1-2 周）**：把 reasoning model 包在 verify-retry loop 里。Verifier 可以是单测、type check、JSON schema 校验、LLM-as-judge。失败率显著下降。

**S3 - Reasoning Budget Ladder（1-2 周）**：实现 low → medium → high 的渐进 escalation。结合任务难度预估（轻量模型先判难度），总成本通常降 40-60%。

**S4 - Hybrid 路由（2-4 周）**：建立 "task → model" 路由器。Code routing、planner routing、verifier routing 分别用不同模型。Cursor、Bolt、Devin 都是这个阶段。配合 prompt cache 工程。

**S5 - CoT 监控与安全工程（持续）**：cheap monitor 订阅 reasoning summary，建立行为基线和告警；Kill switch；CoT obfuscation 红队；与 6.7 Kill Switch、6.6 Memory Security、9.4 可解释性 UX 互锁。

## §7 一页总结

第一性原理：推理模型把训练 RL 目标从 imitation 改成 outcome；把推理算力从常数变成可调维度；把 CoT 从 prompt 字符串变成模型内部状态。这三条同时改变 Agent 工程的成本结构、安全模型和接口形态。

工程不变量：**Verifier**（外层 outcome check）、**Budget**（reasoning_effort/budget_tokens）、**Cache**（system prompt 稳定性）、**Monitor**（cheap model 读 CoT summary）、**Routing**（哪些任务该上、哪些不该上）。

最危险的反模式：把 reasoning_effort=high 当默认；把 CoT 当可编辑 prompt 字符串；用 output_tokens 估算成本；在 reasoning model 上做未经处理的 SFT。

最强信号：可验证任务（数学、代码、形式逻辑）上推理模型质量碾压，且差距随 effort 单调扩大；非可验证任务（写作、创意、对话）上反而可能输给普通模型。
