## 0. 真问题：合规要求三家三本经，谁给我把它们映射成一份工程清单？

任何在欧盟、美国、ISO 体系内做生意的 Agent 团队，都必须同时面对三本"AI 经"：

- **EU AI Act**（Regulation (EU) 2024/1689）：法律，违反最高 €35M 或全球营收 7%；
- **NIST AI RMF**（含 GenAI Profile NIST.AI.600-1）：自愿，但被美国联邦机构 / 监管者引用；
- **ISO/IEC 42001:2023**：可认证国际标准，企业拿 cert 当通行证。

业内的真实痛苦：

- 法务、安全、合规、PMO 各拿一份框架，要求工程团队"对齐"；
- 工程师写一份 RoPA 给 GDPR、写一份 TEVV 给 NIST、再写一份 AIMS 给 ISO，**三套文档八成内容重复**；
- 出事时各路审计走查，证据链对不上。

到 2026 年的最佳实践是：**做 Crosswalk（控制映射）→ 一套控制矩阵满足三家** + "做一次、证三遍"。本文就是这份 Crosswalk 的工程版本，把前 7 篇的所有控制点（[6.1-6.7](./6.1-Prompt-Injection攻防.md)）按三家框架整齐对照，直接可作 Definition-of-Done。

## 1. 三家框架快速对比

| 维度 | EU AI Act | NIST AI RMF | ISO/IEC 42001 |
|---|---|---|---|
| 性质 | 法律（强制） | 自愿指南 | 可认证标准（自愿） |
| 适用 | 进入 EU 市场或影响 EU 用户 | 全球可参考 | 全球可认证 |
| 风险分级 | 4 级（不可接受/高/有限/最小） | 自评（无固定层级） | 风险评估覆盖生命周期，无固定层级 |
| 结构 | 风险分级条款 + Annex III/IV | 4 大功能：Govern/Map/Measure/Manage | 管理体系：领导/政策/风险/数据/透明/绩效/改进 |
| GenAI 增量 | GPAI 提供者义务（Art.53/55） | GenAI Profile 600-1（额外子风险类） | Annex A 通用，无 GenAI 专章 |
| 罚则 | 最高 €35M 或全球营收 7% | 无 | 失证 / 商誉损失 |
| 关键里程碑 | 高风险条款 **2027-12-02** 生效（Digital Omnibus 延期） | 1.0 (2023) + GenAI Profile (2024) + CI Profile (2026 草案) | 2023 发布；ISO/IEC 42006:2025 给认证机构标尺 |

> 战略选择（参考 [AgenticRail 2026 crosswalk](https://agenticrail.nz/resources/ai-governance-frameworks-2026/)）：
> - **EU 市场 + 高风险**：EU AI Act 必走；ISO 42001 当 conformity scaffold；NIST 补 US ops；
> - **美国联邦/受监管**：NIST 为基线；ISO 42001 给认证证据；EU AI Act 在涉欧时叠加；
> - **全球化企业**：ISO 42001 当管理体系底座，EU/NIST 按市场叠加。

## 3. NIST AI RMF 1.0 + GenAI Profile

四大功能 × 类别（节选与 Agent 强相关的）：

### 3.1 GOVERN（治理）
- GOVERN-1.1 ~ 1.7：政策、问责、合规角色、文档流程；
- GOVERN-2 / 3：人才与训练；
- GOVERN-4 / 5：组织文化、与利益方沟通；
- GOVERN-6：第三方风险（与 [6.5 供应链](./6.5-AgentIdentity与供应链安全.md) 联动）。

### 3.2 MAP（识别上下文）
- MAP-1：System category；
- MAP-2 / 3：Stakeholders / 影响；
- MAP-4 / 5：风险评估范围；
- MAP-5.1：Negative impacts to be measured。

### 3.3 MEASURE（度量）
- MEASURE-2.3：能力/限制评估；
- MEASURE-2.6：安全（含 RT）；
- MEASURE-2.7：Cybersecurity；
- MEASURE-2.8：Hazardous misuse；
- MEASURE-2.9：Bias / fairness；
- MEASURE-2.10 / 2.11：Privacy / data；
- MEASURE-2.12：Environmental；
- MEASURE-2.13：Drift；
- MEASURE-3：Documenting measurement；
- MEASURE-4：Feedback。

### 3.4 MANAGE（管理）
- MANAGE-1.4：Risk acceptance / boundary；
- MANAGE-2.1 ~ 2.4：Treatments + 第三方 + Override / disengage（**这就是 Kill Switch 条款**）；
- MANAGE-3：Continuous improvement；
- MANAGE-4：Incident response。

### 3.5 GenAI Profile (NIST.AI.600-1) 12 类子风险（必须覆盖）

CBRN, Confabulation, Dangerous/Violent Recommendations, Data Privacy, Environmental, Human-AI Configuration, Information Integrity, Information Security, Intellectual Property, Obscene/Degrading Content, Toxicity/Bias/Homogenization, Value Chain & Component Integration。

> 实操：把 12 类对照到本系列前面的章节——CBRN/Dangerous/Toxicity → [6.2 Output Guardrail](./6.2-Guardrails三层护栏.md)；Confabulation → [6.6 Drift](./6.6-Memory与DataSecurity.md)；Privacy → [6.6 GDPR](./6.6-Memory与DataSecurity.md)；Information Security → [6.1](./6.1-Prompt-Injection攻防.md) + [6.5](./6.5-AgentIdentity与供应链安全.md)；Value Chain → [6.5 Supply Chain](./6.5-AgentIdentity与供应链安全.md)。

## 4. ISO/IEC 42001:2023 结构

ISO 42001 是 management system standard（MSS），结构与 ISO 27001 同系：

- 4. 上下文（理解组织 / 利益相关方 / 范围）
- 5. 领导（高层承诺 / AI 政策 / 角色）
- 6. 规划（风险评估、目标、变更管理）
- 7. 支持（资源、能力、意识、沟通、文档）
- 8. 运行（运行规划与控制 + 事件管理 + 第三方）
- 9. 绩效评估（监控、内审、管理评审）
- 10. 改进（不符合项 / 持续改进）

加 **Annex A 控制集**（38 控制项 / 9 域），关键域：

- A.5 政策；A.6 内部组织；A.7 资源；A.8 影响评估；A.9 生命周期；A.10 数据；A.4 / A.6.1.6 操作日志（Crosswalk 关键）。

ISO/IEC 42006:2025 规定认证机构资质——**到 2026 年开始有真正可信的认证供应**。

## 6. 工程化的统一证据模型

不要给三家各做一套文档。设计一个**Single Source of Truth**：

```yaml
# evidence/control-08-human-oversight.yaml
control_id: 08
title: Human Oversight (HITL)
mappings:
  eu_ai_act: [Article 14]
  nist_rmf: [MANAGE-2.4]
  iso_42001: [Clause 6.1, Annex A.6.2.5]
gdpr: [Article 22]                # 与 GDPR 联动
implementation:
  references:
    - doc/agent/6.4-RedTeaming合规审计与人机信任边界.md#4-人机信任边界
    - doc/agent/6.7-KillSwitch与可控性工程.md#5-Interrupt
  components:
    - PermissionGate            # 6.3
    - HITLBroker                # 6.4 + 6.7
    - AuditEvent (signed)       # 6.4 / 6.7
evidence:
  runbook: runbooks/hitl_decision_tree.md
  metrics:
    dashboard: grafana://agent-hitl
    kpis: [hitl_latency_p95, decoy_recall_rate, override_count]
  drills:
    - 2026-Q1: passed
    - 2026-Q2: passed
  audit_trail: qldb://agent-audit/2026
review_cycle: quarterly
owner: ai-governance@acme.com
last_reviewed: 2026-06-15
```

特性：

- 控制项 ID 与 §5 总表一一对应；
- 一份 yaml 同时映射三家；
- 实现引用本系列章节；
- 可机器化生成 EU AI Act Annex IV 文档、NIST TEVV 报告、ISO 42001 内审输入。

## 7. 12 个月的合规落地路线

### 月 1-2（启动）
- 任命 AI Governance Owner / DPO / Risk Officer；
- 做 AI 资产盘点（每个 Agent 一行）；

- 立即上 [6.4](./6.4-RedTeaming合规审计与人机信任边界.md) §3.1 TEVV 报告骨架。

- §5 总表 #1-12 落地；

- HITL 控制台 + Challenge-Response Checklist；

### 月 5-7（进阶）
- §5 总表 #13-19 落地；
- §6 evidence yaml 自动化生成；
- ISO 42001 内审一次（自评）；

### 月 8-10（认证）
- 选 ISO 42001 认证机构（看 ISO/IEC 42006:2025 资质）；
- 第三方 Conformity Assessment（如需 EU 高风险）；
- Kill Switch 演练（季度，含 Nuclear）。

### 月 11-12（持续）
- Post-Market Monitoring 流水自动化；
- §5 总表全行覆盖；
- 进入下一个 release 周期，把整个流程作为 PR Gate。

## 8. 反模式

| 反模式 | 后果 | 正确做法 |
|---|---|---|
| 三家各做一套文档 | 维护成本爆炸 | Crosswalk + 单一证据源 |
| 把 NIST 当作"代替 EU AI Act 的方法" | NIST 自愿，不能替代法律 | 该走 EU 合规仍要走 |
| 高层签了 Policy 但工程未实现 | 审计 / 罚款 | Policy 落到 evidence yaml |
| Risk 分级随便填一个"低" | EU 高风险义务漏掉 | Annex III 对照 |
| Audit 用普通业务日志 | 不可信 | append-only + 数字签名 |
| HITL 演练只签字不实操 | Kill 时翻车 | 季度演练 + Chaos Drill |
| GPAI 用了上游模型不读 Art.53/55 | 上游违规牵连下游 | 上游条款覆盖 |
| ISO 42001 当 IT 流程审计 | AI 风险特性丢失 | 用 AI-specific 控制（Annex A） |
| FRIA 与 DPIA 各做一份 | 重复 | 合并模板 |
| Post-Market Monitoring 仅 telemetry | 缺 incident triage | telemetry → triage → fix → 闭环 |

## 9. S0→S6 治理成熟度路线

| 阶段 | 表现 |
|---|---|
| S0 | 无 AI 政策；无风险评估 |
| S1 | 有政策但无证据；分级粗略 |
| S2 | NIST RMF 4 函数全覆盖 + 证据 yaml 雏形 |
| S3 | ISO 42001 内审通过 + EU AI Act 高风险义务到位（如适用） |
| S4 | ISO 42001 第三方认证 + EU Conformity Assessment 通过 |
| S5 | Continuous Compliance（PR Gate + Auto-Evidence） |
| S6 | Federated Cross-Vendor Compliance（与上下游 Agent 互证） |

- 6 个月达 S2 是底线；
- 12 个月达 S3 是 EU 合规起跳线；
- 18-24 个月达 S4 是商业差异化资产。
