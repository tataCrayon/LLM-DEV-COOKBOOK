# Context Engineering：从 RAG 到上下文窗口管理

> 如果 Prompt Engineering 是"单次调用内怎么说"，Context Engineering 就是"整个 Agent 生命周期中，上下文窗口里该有什么"。它是 Agent 系统的内存管理——决定了模型每一步能"看见"什么、"忘记"什么、"去哪里找"什么。

## 0. 开场问题

你有一个 coding agent，正在处理一个 100 文件的 monorepo。用户说"把所有 API endpoint 的错误处理统一成 RFC 7807 格式"。Agent 需要：

1. 理解 RFC 7807 是什么（知识检索）
2. 找到所有现有 endpoint 的错误处理代码（代码检索）
3. 记住已经改了哪些文件（状态跟踪）
4. 保持对项目编码规范的遵守（规则持久化）
5. 上下文用到 150K token 时不崩溃（窗口管理）

这五件事没有一件是靠"写好 prompt"能解决的。它们是上下文层的问题——你需要一套系统来决定：此刻的 context window 里该塞什么、不该塞什么、满了怎么办。

Andrej Karpathy 的定义最精炼：

> Context Engineering 是"用恰好正确的信息填充上下文窗口以供下一步使用的精微艺术与科学"。

Cognition（Devin 团队）更直白：这是"构建 AI Agent 的工程师的第一号工作"。

## 1. 为什么"长上下文"不能替代 Context Engineering

2024-2025 年，各厂商疯狂扩窗口：Gemini 2M、Claude 200K、GPT-4o 128K。直觉上，窗口够大就不用管理了——全塞进去就行。

这个直觉是错的，原因在 1.1 篇已经铺过：

**注意力物理学的限制**：Chroma 的实验发现，标称 200K 窗口的模型在 50K token 时就出现显著的检索准确率下降。原因是注意力权重被 softmax 压平——token 越多，每个 token 分到的注意力越少，"lost in the middle"效应越严重。

**经济学的限制**：处理 200K 输入的单次调用，Claude Sonnet 收费约 $0.60。一个 Agent 一次任务可能调用模型 50 次——如果每次都塞满窗口，一个任务 $30。而通过精细的上下文管理，可以把平均输入降到 20-30K，成本降 80%。

**延迟的限制**：Prefill 阶段的时间与输入 token 数线性相关。200K 的 prefill 比 20K 慢一个数量级。Agent 需要快速迭代（试错→修正→再试），延迟直接影响任务完成时间。

结论：**长上下文是保险，不是策略。** 好的 Context Engineering 让你在 80% 的调用中只用 20% 的窗口，留着 200K 应对真正需要全局视野的关键步骤。

## 2. RAG：最经典的 Context Engineering

RAG（Retrieval-Augmented Generation）是 Context Engineering 的原型——用检索代替硬塞。但 2025 年的 RAG 已经远比 2023 年的"向量搜索 + 拼接"复杂。

### 2.1 RAG vs 长上下文：不是非此即彼

Li et al. (2024, arXiv:2407.16833) 的对比研究明确了分界线：

| 场景 | 用 RAG | 用长上下文直接塞 |
|---|---|---|
| 知识库 >500 篇文档 | ✅ 无法全部塞入 | ❌ 物理上不可能 |
| 需要实时更新的知识 | ✅ 可以随时更新索引 | ❌ 需要重新构建上下文 |
| 单篇文档深度分析 | ❌ 检索可能漏掉关联段 | ✅ 全文在窗口内 |
| 跨文档关联推理 | ❌ 各片段缺乏连贯性 | ✅ 模型能看到全貌 |
| 成本敏感 | ✅ 只检索相关片段 | ❌ 全文计费 |

### 2.2 2025 年 RAG 的演进

Windsurf（前 Codeium）的 CTO Varun 在 LangChain 的文章中指出了一个关键问题：**随着代码库增长，纯嵌入检索变得不可靠。** 生产级 RAG 需要多路检索：

```
查询 ─┬─→ 向量嵌入搜索（语义相似性）
      ├─→ 关键词/grep 搜索（精确匹配）
      ├─→ 知识图谱遍历（结构化关系）
      └─→ 文件路径匹配（目录结构感知）
           │
           ▼
      Re-ranking（重排序，融合多路结果）
           │
           ▼
      Top-K 片段 → 塞入上下文
```

这里的每一路都有其不可替代性：嵌入搜索擅长"语义接近但措辞不同"的查询；grep 搜索擅长找精确的函数名/变量名；知识图谱擅长"A 调用了 B，B 依赖 C"这种多跳关系。

### 2.3 RAG 在 Agent 中的特殊需求

传统 RAG 是"一问一答"——用户问一个问题，检索一次，返回答案。Agent RAG 是"多轮持续检索"——Agent 在执行任务的过程中，根据当前步骤的需要动态检索。

这带来两个额外问题：

**检索查询由 Agent 自己生成**——而非人类手写。Agent 的检索查询质量直接影响后续步骤。好的 Agent 会先做"query expansion"（把用户的模糊需求转化为多个精确查询），再做检索。

**检索结果会累积**——Agent 可能在 10 步中检索了 10 次，每次塞 2K token 的结果。到第 11 步时，光检索结果就占了 20K。这时候需要 compaction 或 clearing。

## 3. 上下文四操作：Write / Select / Compress / Isolate

LangChain（2025）提出了一个简洁但完备的 Context Engineering 操作分类：

### 3.1 Write（写出去）

把信息从上下文窗口写到外部存储。两种形态：

**Scratchpad（会话内暂存）**——Agent 在执行过程中把中间状态写到文件或内存对象，而非留在对话历史里。Anthropic 的多 Agent 研究系统在超过 200K token 时，把当前计划写入 memory，后续步骤从 memory 读取而非回溯历史。

**Long-term Memory（跨会话持久化）**——Claude Code 的 `CLAUDE.md`、Cursor 的 rules 文件、ChatGPT 的 memory 功能。这些都是把"跨 session 需要保持的知识"写出去，在新 session 开始时选择性加载回来。

### 3.2 Select（选进来）

决定哪些外部信息拉入当前上下文。这是 RAG 的泛化——不只检索文档，还包括：

- **检索记忆**：从长期记忆中拉取与当前任务相关的历史决策
- **检索工具描述**：当工具集很大时（>20 个），用 RAG 动态选择当前步骤最相关的工具子集，而非全部暴露。研究表明这能提升 3x 的工具选择准确率
- **检索示例**：根据当前输入动态选 few-shot 示例（而非固定示例）

### 3.3 Compress（压缩）

当上下文逼近窗口上限时，保留语义、丢弃冗余。

**JetBrains 的关键发现（NeurIPS 2025 Deep Learning 4 Code Workshop）**：

他们在 SWE-bench Verified（500 instances, 250 turns）上对比了两种压缩策略：

1. **Observation Masking**（SWE-agent 风格）：用占位符替换旧的工具输出，保留推理和动作记录
2. **LLM Summarization**（OpenHands 风格）：用另一个模型把历史压缩成摘要

结果出人意料：**observation masking 在 4/5 的测试设置中胜出**。用 Qwen3-Coder 480B 测试时，masking 比不管理的基线高 2.6% 的解题率，同时便宜 52%。而 LLM summarization 反而导致 Agent 轨迹延长 13-15%——因为摘要"抹平了本该让 Agent 停下来的信号"。

**Anthropic 的锚定摘要模式**：当确实需要 LLM 压缩时，使用四字段结构化摘要——intent（意图）、changes_made（已做更改）、decisions_taken（已做决策）、next_steps（下一步）。这比无结构的"请总结对话"保留了更多关键信号。

### 3.4 Isolate（隔离）

把上下文拆分给不同的执行单元，各自管理各自的窗口。

**Multi-agent 隔离**：每个 sub-agent 有自己的工具集、指令和上下文窗口。Anthropic 报告这可能使用约 15x 于普通 chat 的 token——但每个 agent 在自己的窗口内效率更高。

**沙箱变量隔离**：HuggingFace 的 deep researcher 用 CodeAgent 执行代码，把大型对象（图片、数据表）作为 sandbox 变量保存，不占用 LLM 上下文。

## 4. Compaction 深度剖析：Agent 的核心生存技能

对于长时运行的 Agent，compaction 不是优化——是生存条件。

### 4.1 触发时机

Anthropic 的 Claude Code 在上下文达到窗口 95% 时触发 auto-compact。但研究建议更早启动：

- **~50% 使用率**：已经出现性能退化信号（Chroma 的实验）
- **~64% 使用率**：触发 memory sync 到外部存储（此时上下文仍然连贯，sync 质量高）
- **~80% 使用率**：启动 session handoff（轮换到新 session）

这是一个两阶段监控模式——在"还行"的时候提前保存，在"快满了"的时候果断切换。

### 4.2 Compaction API（Anthropic，2026-01）

Anthropic 在 2026 年 1 月发布了原生 compaction API：

```python
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=4096,
    system="你是编码助手。",
    messages=long_conversation_messages,
    betas=["compact-2026-01-12"],
    context_management={
        "edits": [{
            "type": "compact_20260112",
            "trigger": {"type": "input_tokens", "value": 150_000},
        }]
    }
)
```

关键行为：
- 触发后，整个对话历史（用户消息、助手消息、工具调用、工具结果）被压缩成一个高保真摘要
- 高层事实（意图、关键决策、主要变更）100% 保留
- 具体细节（附录表格、具体统计数字）可能丢失
- 可以提供自定义 `instructions`，但注意：自定义指令**完全替换**默认压缩策略，不是追加

### 4.3 Tool-Result Clearing（手术式清理）

比 compaction 更精细的操作——只清除旧的 `tool_result` 内容，保留工具调用记录和推理过程：

```python
context_management={
    "edits": [{
        "type": "clear_tool_uses_20250919",
        "trigger": {"type": "input_tokens", "value": 100_000},
        "keep": 3,  # 保留最近 3 次工具结果
        "exclude_tools": ["read_file"]  # 某些工具结果不清除
    }]
}
```

在 Anthropic 的测试中，clearing 单次释放约 163K token，把峰值上下文从 335K 压到 173K。

核心逻辑：**工具结果通常是可重新获取的**（再读一次文件、再查一次 API）。清除它们只丢弃了"缓存值"，不丢弃"推理链"。而 compaction 是压缩推理链本身，代价更高。

### 4.4 从动量视角理解 Compaction

Jason Liu（jxnl）提出了一个优雅的类比：如果把 in-context learning 看作梯度下降，那么 compaction 就是 momentum。

```
传统动量: θ_{t+1} = θ_t - α∇L(θ_t) + β(θ_t - θ_{t-1})
对话类比: context_new = compact(context_full) + β(learning_trajectory)
```

好的 compaction 不只保留事实，还保留**方向**——"尝试了 X，失败了，改用 Y 成功了"这种轨迹信息。naive summarization 丢掉的恰恰是这个方向，所以 JetBrains 的实验中 summarization 反而让 Agent 走弯路。

## 5. Session 生命周期管理

长时间运行的 Agent 不可避免要面对 session 边界——要么上下文满了必须切换，要么任务跨天需要恢复。

### 5.1 Cold Start 问题

冷启动对任务型 Agent 是灾难性的：Agent 要么反复尝试恢复已经不存在的状态，要么"过早宣布任务完成"。

**Anthropic 的 Initializer-Coder 模式**：

```
初始化 Agent（只跑一次）：
  → 构建 feature list JSON
  → 生成 init.sh
  → 初始化 git repo
  → 创建 progress file

编码 Agent（每次 session）：
  → 读 progress file + git log
  → 跑 init.sh
  → 跑 E2E tests
  → 选下一个 feature 开始工作
  → 工作完成后 commit + 更新 progress file
```

### 5.2 Handoff 的五层模式

当一个 session 必须切换到新 session 时，handoff 质量决定了新 session 能否无缝接续。好的 handoff 是"讲故事"，不是"倒数据"：

1. **State snapshot**：结构化的当前状态（有 schema 验证）
2. **Narrative context**：3-5 句话描述"我们在干嘛"
3. **Decision log**：已做决策 + 延迟决策 + 原因
4. **Priority queue**：有序的下一步列表
5. **Warnings**：环境 quirks、rate limits、已知的坑

Claude Code 的 `claude-progress.txt` 就是这个模式的实例——每次 session 结束时更新进度文件，下次 session 开始时读取。

### 5.3 分层记忆架构

```
┌──────────────────────────────────────┐
│  Tier 1: In-context working memory    │  ← 活跃对话，易失
├──────────────────────────────────────┤
│  Tier 2: Session-scoped external      │  ← progress files, scratchpad
├──────────────────────────────────────┤
│  Tier 3: Long-term semantic memory    │  ← 向量库/KG，按需查询
├──────────────────────────────────────┤
│  Tier 4: Archive / Cold storage       │  ← 历史 session，极少访问
└──────────────────────────────────────┘
```

**值得持久化到 Tier 2+**：决策及其理由、当前任务/阻塞、环境配置、优先级队列。

**不值得持久化**：原始工具输出、中间计算结果、可重新获取的文件内容。

## 6. 成本工程：让 Context Engineering 产生 ROI

聪明的上下文管理可以"降低 80-90% 的 token 成本，同时提升 26% 的回复质量"（Zylos Research, 2026）。三大杠杆：

### 6.1 Prompt Caching

| 提供商 | 机制 | 折扣 | TTL |
|---|---|---|---|
| Anthropic | `cache_control: {"type": "ephemeral"}` | 90% off（缓存命中） | 5 分钟 |
| OpenAI | 自动缓存相同前缀 | 50% off | ~5-10 分钟 |
| Google Gemini | 显式缓存 API | 75% off | 可配置（1h+） |

设计含义：把 system prompt、tool definitions、AGENTS.md 这类不变内容放在消息最前面，确保每次调用都能命中缓存。

### 6.2 Token 压缩

**LLMLingua-2**：微软的 token 压缩技术，可以在保持 >90% 语义保真度的情况下压缩 5x。适合压缩 RAG 检索结果中的冗余信息，再塞入上下文。

**Selective tier loading**：不是每次调用都加载全部记忆。根据当前步骤的工具调用类型，只加载相关 tier 的记忆子集。

### 6.3 Anti-pattern：全历史重放

最大的成本浪费是每次调用都重放完整对话历史。一个 50 步的 Agent 如果每步都带完整历史，第 50 步的输入是第 1 步的 50 倍。正确做法：sliding window（保留最近 N 步）+ 结构化摘要（压缩更早的步骤）。

## 7. 完整代码示例：带上下文管理的 Agent Loop

```python
"""
演示：一个带完整上下文管理的简单 Agent Loop
覆盖：RAG 检索、working memory、compaction 触发、session handoff
"""
import json
from pathlib import Path
from anthropic import Anthropic

client = Anthropic()

# ===== 配置 =====
MAX_CONTEXT_TOKENS = 180_000
COMPACT_THRESHOLD = 0.7  # 70% 时触发 compaction
HANDOFF_THRESHOLD = 0.9  # 90% 时触发 session handoff
PROGRESS_FILE = Path("./agent-progress.json")

# ===== Working Memory =====
class WorkingMemory:
    """Agent 的工作记忆——存活于 session 内，跨调用保持"""
    
    def __init__(self):
        self.decisions: list[str] = []
        self.files_modified: list[str] = []
        self.current_task: str = ""
        self.blockers: list[str] = []
    
    def to_context_block(self) -> str:
        """压缩为上下文片段，放在 system prompt 末尾"""
        return f"""<working_memory>
当前任务: {self.current_task}
已修改文件: {', '.join(self.files_modified[-10:])}  
关键决策: {'; '.join(self.decisions[-5:])}
阻塞项: {'; '.join(self.blockers) if self.blockers else '无'}
</working_memory>"""
    
    def save_progress(self):
        """Session 结束时持久化到文件"""
        PROGRESS_FILE.write_text(json.dumps({
            "task": self.current_task,
            "decisions": self.decisions,
            "files_modified": self.files_modified,
            "blockers": self.blockers,
        }, ensure_ascii=False, indent=2))
    
    @classmethod
    def load_progress(cls) -> "WorkingMemory":
        """新 Session 开始时从文件恢复"""
        mem = cls()
        if PROGRESS_FILE.exists():
            data = json.loads(PROGRESS_FILE.read_text())
            mem.current_task = data.get("task", "")
            mem.decisions = data.get("decisions", [])
            mem.files_modified = data.get("files_modified", [])
            mem.blockers = data.get("blockers", [])
        return mem


# ===== Context Budget Tracker =====
class ContextBudget:
    """追踪 token 使用，触发 compaction/handoff"""
    
    def __init__(self, max_tokens: int):
        self.max_tokens = max_tokens
        self.current_usage = 0
    
    def update(self, response_usage):
        """从 API response 中更新 usage"""
        self.current_usage = response_usage.input_tokens
    
    @property
    def utilization(self) -> float:
        return self.current_usage / self.max_tokens
    
    @property
    def needs_compaction(self) -> bool:
        return self.utilization >= COMPACT_THRESHOLD
    
    @property
    def needs_handoff(self) -> bool:
        return self.utilization >= HANDOFF_THRESHOLD


# ===== Compaction =====
def compact_messages(messages: list[dict], memory: WorkingMemory) -> list[dict]:
    """
    压缩消息历史。策略：
    1. 保留最近 3 轮完整对话
    2. 更早的用锚定摘要替代
    """
    if len(messages) <= 6:
        return messages
    
    # 用模型生成结构化摘要
    old_messages = messages[:-6]
    recent_messages = messages[-6:]
    
    summary_response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1000,
        system="生成对话历史的结构化摘要。只输出 JSON。",
        messages=[{
            "role": "user",
            "content": f"""将以下对话历史压缩为结构化摘要：

{json.dumps(old_messages, ensure_ascii=False)[:50000]}

输出格式：
{{"intent": "...", "changes_made": [...], "decisions_taken": [...], "next_steps": [...]}}"""
        }]
    )
    
    summary_text = summary_response.content[0].text
    
    # 用摘要替代旧消息
    compacted = [
        {"role": "user", "content": f"[历史摘要] {summary_text}"},
        {"role": "assistant", "content": "理解。我将基于以上历史继续工作。"},
    ] + recent_messages
    
    return compacted


# ===== Main Agent Loop =====
def run_agent(task: str):
    """主 Agent 循环，带完整上下文管理"""
    
    memory = WorkingMemory.load_progress()
    memory.current_task = task
    budget = ContextBudget(MAX_CONTEXT_TOKENS)
    
    # 分层 system prompt（缓存友好排列）
    system_prompt = [
        {
            "type": "text",
            "text": """你是高级软件工程师。你通过工具与代码库交互。

工作规范：
- 每次只改一个文件
- 改完跑测试
- 发现问题立即记录到 working_memory
- 如果连续 3 次失败，上报 blocker""",
            "cache_control": {"type": "ephemeral"}  # 缓存稳定部分
        },
        {
            "type": "text",
            "text": memory.to_context_block()  # 动态部分不缓存
        }
    ]
    
    messages = []
    if memory.decisions:
        # 冷启动恢复：把上次进度作为首条消息
        messages.append({
            "role": "user",
            "content": f"继续任务：{task}\n\n上次进度：已改文件 {memory.files_modified[-3:]}，最近决策：{memory.decisions[-3:]}"
        })
    else:
        messages.append({"role": "user", "content": f"任务：{task}"})
    
    tools = [
        {"name": "read_file", "description": "读取文件", "input_schema": {"type": "object", "properties": {"path": {"type": "string"}}, "required": ["path"]}},
        {"name": "write_file", "description": "写入文件", "input_schema": {"type": "object", "properties": {"path": {"type": "string"}, "content": {"type": "string"}}, "required": ["path", "content"]}},
        {"name": "run_tests", "description": "运行测试", "input_schema": {"type": "object", "properties": {"path": {"type": "string"}}, "required": ["path"]}},
        {"name": "search_code", "description": "搜索代码", "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}},
        {"name": "update_memory", "description": "更新工作记忆", "input_schema": {"type": "object", "properties": {"decision": {"type": "string"}}, "required": ["decision"]}},
    ]
    
    step = 0
    max_steps = 30
    
    while step < max_steps:
        step += 1
        
        # 检查是否需要 compaction
        if budget.needs_handoff:
            print(f"[Step {step}] Context at {budget.utilization:.0%} — triggering handoff")
            memory.save_progress()
            print("Progress saved. Please start a new session to continue.")
            break
        
        if budget.needs_compaction:
            print(f"[Step {step}] Context at {budget.utilization:.0%} — compacting...")
            messages = compact_messages(messages, memory)
        
        # 调用模型
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=4096,
            system=system_prompt,
            tools=tools,
            messages=messages,
        )
        
        # 更新 budget
        budget.update(response.usage)
        
        # 处理响应
        assistant_message = {"role": "assistant", "content": response.content}
        messages.append(assistant_message)
        
        # 检查是否结束
        if response.stop_reason == "end_turn":
            print(f"[Step {step}] Agent 完成任务")
            memory.save_progress()
            break
        
        # 处理工具调用
        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input, memory)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })
            messages.append({"role": "user", "content": tool_results})
    
    return memory


def execute_tool(name: str, input_data: dict, memory: WorkingMemory) -> str:
    """执行工具并更新 working memory（示意实现）"""
    if name == "update_memory":
        memory.decisions.append(input_data["decision"])
        return "已记录"
    elif name == "write_file":
        memory.files_modified.append(input_data["path"])
        # 实际写文件...
        return f"已写入 {input_data['path']}"
    elif name == "read_file":
        # 实际读文件...
        return f"文件内容：..."
    elif name == "run_tests":
        # 实际跑测试...
        return "所有测试通过"
    elif name == "search_code":
        # 实际搜索...
        return "找到 3 个匹配结果..."
    return "未知工具"
```

## 8. 决策清单：Context Engineering 选型指南

面对一个具体的 Agent 项目，按以下顺序决策：

**第一步：评估知识需求**
- 知识量 < 窗口 50% → 直接塞入 system prompt（最简方案）
- 知识量 > 窗口 → 需要 RAG
- 知识需要实时更新 → 需要 RAG + 实时索引

**第二步：评估会话长度**
- 单次调用完成 → 不需要 compaction
- 5-20 步 → observation masking + sliding window
- 20+ 步 / 跨 session → 完整 compaction + progress file + handoff

**第三步：评估并行度**
- 单 Agent 线性执行 → 一个上下文窗口够用
- 多 Agent 并行 → isolate（每个 agent 独立窗口）+ 共享 working memory

**第四步：评估成本预算**
- 高预算 → 长上下文 + 少压缩，优先保真度
- 低预算 → 激进压缩 + prompt caching + selective loading

**第五步：选择 compaction 策略**
- 工具输出为主的历史 → tool-result clearing（手术式，损失小）
- 推理对话为主的历史 → observation masking（比 LLM summarization 便宜且效果好）
- 需要跨 session 保持理解 → 锚定摘要（四字段结构化）
