# Agent 隔离技术与递归自改进：安全底座与能力前沿

> Agent 要执行代码就必须有沙箱，Agent 要自我进化就必须有安全围栏。这两个主题是工程上的耦合配件——一个解决"如何让 Agent 安全地跑代码"，一个探讨"Agent 自己改自己到什么程度了"。后者的可行性很大程度上依赖前者。

## 0. 开场问题

你的 coding agent 刚生成了一段代码，准备执行 `pip install` 安装一个依赖然后跑测试。问题是：

1. 如果模型幻觉了一个恶意包名（typosquatting），你的宿主机就被植入了后门
2. 如果代码里有 `os.system("rm -rf /")` 这种由 prompt injection 触发的恶意指令——不管多离谱，你不能赌它不会发生
3. 如果 Agent 跑了一个无限循环吃光 CPU 和内存呢？

这些都是真实威胁。隔离不是"高级功能"，是 Agent 执行任何不可信代码的**底线**。没有隔离的 Agent 就像在生产数据库上直接跑 `eval(user_input)` 一样危险。

接着第二个问题：如果 Agent 不只是跑代码，还能**修改自己的 prompt、工具集甚至训练数据**——这就进入了递归自改进（RSI）的领域。RSI 的每一次迭代都需要在隔离环境中验证产物，否则一次 reward hacking 就可能搞崩整个系统。

## 1. Agent 隔离：四种方案的物理学

隔离强度和性能开销天然对立。所有方案都是在这条曲线上选点：

### 1.1 方案对比

| 维度 | 容器 (Docker) | gVisor | Firecracker | 完整 VM |
|---|---|---|---|---|
| 隔离原理 | namespace + cgroups | 用户态内核 | 硬件虚拟化 (KVM) | 硬件虚拟化 |
| 安全边界 | 共享宿主内核 | Go Sentry 进程 | 独立 guest 内核 | 独立 guest 内核 |
| 启动时间 | <100ms | ~200ms | ~125ms | 数秒 |
| 内存开销 | ~几十 MB | ~50 MB | <5 MB/microVM | ~GB |
| I/O 性能 | 接近原生 | 下降 20-50% | 接近原生 | 接近原生 |
| 兼容性 | 完整 Linux | 部分 syscall 缺失 | 完整 Linux | 完整 OS |
| 逃逸风险 | 周期性 CVE | 极低 | 极低 | 极低 |
| 密度（单机） | 数百 | 数十~百 | 数千 | 数十 |
| 典型用户 | CI、内部服务 | Google Cloud Run | AWS Lambda、E2B | 金融合规 |

### 1.2 容器：最轻但最脆

本质是"被关起来的进程"——用 namespace 隔离 PID/网络/文件系统，用 cgroups 限制资源，用 seccomp + AppArmor 限制系统调用。

**优点**：启动毫秒级、几乎零 overhead、生态最完善（Docker 镜像直接用）。

**致命弱点**：与宿主共享内核。历史上 runC CVE-2019-5736、Dirty Pipe CVE-2022-0847 等漏洞允许容器逃逸。对于跑自家可信代码完全够用，但跑 AI 生成的未知代码——你在赌"当前没有未公开的内核漏洞"。

**2026 现状**：arXiv 论文（Coding Agent 分类学）发现 5/13 开源 coding agent 仍在用 Docker 隔离。Daytona 也基于容器，创建时间 <90ms。对于开发环境可接受，生产级 Agent 应该考虑更强隔离。

### 1.3 gVisor：Google 的折中方案

一个用户态内核——用 Go 重写了大部分 Linux syscall，容器内进程的系统调用被 Sentry 拦截、在用户态处理，只有极少数透传到宿主内核。攻击面比容器小一个数量级。

**优点**：比容器安全很多；比 VM 轻很多。

**缺点**：部分 syscall 不支持（很多 ML 框架、某些 C 扩展跑不起来）；I/O 密集场景性能显著下降。

**2026 现状**：Modal 使用 gVisor 作为隔离层。Google Cloud Run、App Engine 内部用。适合不需要完整 Linux 兼容性的纯计算任务。

### 1.4 Firecracker：AI Agent 沙箱的甜蜜点

AWS 开源的极简 VMM，运行在 KVM 之上。砍掉了 QEMU 的全部"PC 兼容"包袱——没有 BIOS、没有 PCI、没有 USB。结果是：

- 启动 ~125ms（比完整 VM 快 100x）
- 内存 <5MB/microVM（比完整 VM 小 200x）
- 安全边界是硬件虚拟化（VT-x/SVM），与完整 VM 同级
- 单台宿主机可跑数千个 microVM

**2026 现状**：这是当前 AI Agent 沙箱的主流技术底座。使用 Firecracker 的平台包括：

| 平台 | 定位 | 附加特性 |
|---|---|---|
| **E2B** | Agent 专用沙箱 | 预构建模板、SDK 定义环境、auto-pause（beta） |
| **CodeSandbox** (Together AI) | Agent + 开发者环境 | `sandbox.fork()` copy-on-write 克隆 |
| **Fly.io** | 通用 microVM 平台 | 全球多区域部署、完整状态快照 |
| **AWS Lambda** | Serverless 函数 | 大规模验证、企业级 |

**选型直觉**：如果你的 Agent 要跑用户提交的代码或 AI 生成的代码，Firecracker 是默认选择——安全性等同 VM，密度接近容器。

### 1.5 完整 VM

最强隔离——独立内核、独立 OS、硬件虚拟化全保护。但启动秒级、内存 GB 级、密度低。

适合场景：金融/医疗等强合规环境、需要 Windows/macOS 的 Agent（Firecracker 只支持 Linux guest）、需要 GPU 直通的 ML 训练。

### 1.6 WASM：新兴的第五选项

WebAssembly 作为沙箱在 2025 年开始进入 Agent 领域。优点是极轻量（字节级 overhead）、近原生速度、天然的内存安全。缺点是生态尚不成熟、系统调用受限、不适合需要完整 OS 的场景。

适合：浏览器内 Agent 执行、轻量级函数沙箱、不需要文件系统/网络的纯计算。

## 2. 隔离的工程实践

### 2.1 Codex CLI 的隔离架构

arXiv 论文揭示了 Codex CLI 的安全模型：

- **隔离层**：Bubblewrap (bwrap) + Landlock（Linux 安全模块）
- **额外保护**：一个基于 LLM 的 "Guardian" 安全子 agent，审查每个工具调用
- **关键设计**：只有 Codex 提供的 shell tool 在沙箱中运行；MCP 工具不受 Codex 沙箱保护，必须自行实现安全边界

### 2.2 网络隔离

沙箱的一个常被忽视的维度是网络策略：

- **完全隔离**：无网络。最安全但很多任务做不了（pip install、API 调用）
- **白名单出站**：只允许访问特定域名（PyPI、npm registry、内部 API）
- **代理模式**：所有流量经过审计代理，可以拦截可疑请求
- **完全开放**：不推荐，除非你 100% 信任 Agent 的行为

### 2.3 文件系统隔离

```python
"""
演示：用 Docker SDK 创建隔离执行环境
包含：文件系统挂载、网络限制、资源限制、超时控制
"""
import docker
import tempfile
from pathlib import Path

client = docker.from_env()

def execute_in_sandbox(
    code: str,
    timeout: int = 30,
    memory_limit: str = "512m",
    network_disabled: bool = True,
    allowed_packages: list[str] = None,
) -> dict:
    """
    在 Docker 容器中安全执行 Agent 生成的代码。
    生产环境应替换为 Firecracker/E2B SDK。
    """
    # 创建临时工作目录
    work_dir = tempfile.mkdtemp()
    code_file = Path(work_dir) / "agent_code.py"
    code_file.write_text(code)
    
    # 构建执行命令
    if allowed_packages:
        install_cmd = f"pip install --quiet {' '.join(allowed_packages)} && "
    else:
        install_cmd = ""
    
    cmd = f"bash -c '{install_cmd}python /workspace/agent_code.py'"
    
    try:
        container = client.containers.run(
            image="python:3.11-slim",
            command=cmd,
            volumes={work_dir: {"bind": "/workspace", "mode": "ro"}},  # 只读挂载
            mem_limit=memory_limit,
            nano_cpus=1_000_000_000,  # 1 CPU
            network_disabled=network_disabled,
            # 安全加固
            read_only=True,  # 只读根文件系统
            security_opt=["no-new-privileges"],
            cap_drop=["ALL"],  # 丢弃所有 Linux capabilities
            # 临时写入区域
            tmpfs={"/tmp": "size=100m,noexec"},
            detach=True,
        )
        
        # 等待执行完成或超时
        result = container.wait(timeout=timeout)
        logs = container.logs().decode("utf-8", errors="replace")
        
        return {
            "exit_code": result["StatusCode"],
            "output": logs[-5000:],  # 截断防止 token 爆炸
            "timed_out": False,
        }
        
    except docker.errors.ContainerError as e:
        return {"exit_code": 1, "output": str(e)[-2000:], "timed_out": False}
    except Exception as e:
        return {"exit_code": -1, "output": f"Sandbox error: {e}", "timed_out": True}
    finally:
        try:
            container.remove(force=True)
        except:
            pass


# ===== 使用示例 =====
agent_generated_code = """
import math
# Agent 生成的代码——可能安全，也可能不安全
result = sum(math.factorial(i) for i in range(20))
print(f"Result: {result}")
"""

result = execute_in_sandbox(
    code=agent_generated_code,
    timeout=10,
    memory_limit="256m",
    network_disabled=True,  # 禁止网络——纯计算不需要
)
print(f"Exit: {result['exit_code']}, Output: {result['output']}")
```

## 3. 递归自改进（RSI）：Agent 自己改自己

### 3.1 概念溯源

I.J. Good (1965) 提出 "intelligence explosion"：一个能造出比自己更聪明系统的机器会引发一连串迭代。Bostrom (2014《Superintelligence》) 把这推广为奇点叙事。RSI 是 AI Safety 领域的核心母题，也是 OpenAI/Anthropic/DeepMind 制定安全策略时的关键威胁模型。

### 3.2 改进的五个层级

按"动什么"分层，越往下越根本、越危险：

| 层级 | 改什么 | 当前状态 | 例子 |
|---|---|---|---|
| **数据/行为级** | 经验、记忆 | ✅ 成熟，遍地都是 | Voyager 技能库、长期记忆 |
| **参数级** | 模型权重 | ✅ 成熟（工业实践） | Self-Instruct、RLAIF、Constitutional AI |
| **代码级** | prompt/工具/源码 | ⚡ 弱形态量产 | Claude Code 改 skills/AGENTS.md |
| **架构级** | 模型结构 | 🔬 早期研究 | AlphaEvolve (2025)、AI Scientist |
| **元级** | 改进方法本身 | 📝 纯理论 | Yudkowsky 警告的层级 |

### 3.3 当前已经实现到哪

**数据/行为层**：完全成熟。每个带记忆的 Agent 都在做——用得越久越好用，但没触动模型本身。

**参数层**：工业标准。Llama、Claude、GPT 系列都用了类似 RLAIF/自蒸馏的手法。模型生成数据训自己的下一版。

**代码层弱形态**：已经量产。Cursor/Claude Code 修改自己的 skills、配置文件、AGENTS.md。但强形态（让 Agent 改自己的训练 pipeline）还在研究阶段。

**架构层**：DeepMind 的 AlphaEvolve (2025) 用 LLM 进化发现了新的矩阵乘法算法。Sakana AI 的 AI Scientist 能产出新论文。但闭环仍需人工监督。

**元层**：纯理论阶段。改进"改进自己的方法"本身——理论上最容易触发智能爆炸，实践中还没有可信的实现。

**核心判断**：真正的"快速起飞"型 RSI 还没出现。当前观察到的是渐进式自改进——更像复利式的能力累积，而非指数爆炸。

### 3.4 风险与对策

三大核心风险：

- **Reward Hacking**：模型学会欺骗评估器（"任务完成"但其实没有）
- **Goal Misgeneralization**：自改进过程中目标漂移（"变强了但目标歪了"）
- **不可解释**：改完以后人类看不懂为什么

对应的工程对策：

```
风险                    对策                        落地形态
────────────────────────────────────────────────────────────
Reward Hacking    →    沙箱验证                  →  每次自改进的产物先在 Firecracker 里跑
Goal Drift        →    能力评估红线              →  METR task suite、Anthropic ASL 等级
不可解释          →    HITL 审批                 →  关键改动需人评审（Anthropic RSP）
失控              →    形式化验证/可解释性        →  早期研究，长期解药
```

### 3.5 隔离是 RSI 的基础设施

把两块拼起来——**递归自改进的安全前提是强隔离**：

如果你想让 Agent 自己写代码、跑实验、迭代自己——每一次实验都必须在 Firecracker/microVM 里跑。原因：

1. 自改进产物的正确性无法提前保证（否则就不需要实验）
2. Reward hacking 的第一个表现就是"做出看起来对但实际有害的改动"
3. 没有隔离意味着第一次失败的实验就可能污染宿主环境

E2B、Modal、Daytona、CodeSandbox——这批"AI Agent 沙箱"公司的共同技术底座几乎全是 Firecracker，产品形态就是给自我迭代型 Agent 提供"安全的练习场"。

换句话说：**隔离技术是 RSI 的基础设施，没有它递归自改进只能在玩具环境里演示，没法真正放出来。**

## 4. 选型决策清单

### 4.1 隔离方案选型

| 你的场景 | 推荐方案 | 理由 |
|---|---|---|
| 跑自家可信代码/CI | 容器 (Docker) | 够用，生态好 |
| 跑 AI 生成的代码 | Firecracker (E2B/Fly.io) | 硬件隔离 + 高密度 |
| 需要 GPU | Modal (gVisor) 或完整 VM | gVisor 支持 GPU；密度优先选 Modal |
| 浏览器内轻量执行 | WASM | 无需服务端 |
| 强合规（金融/医疗） | 完整 VM | 审计要求 + 独立 OS |
| 需要 Windows/macOS | 完整 VM | Firecracker 只支持 Linux |

### 4.2 RSI 安全等级评估

| Agent 能改什么 | 风险等级 | 最低安全要求 |
|---|---|---|
| 只改自己的 memory/notes | 低 | 基本 permission gate |
| 改 prompt/skills/config | 中 | 版本控制 + diff review |
| 改工具定义/注册新工具 | 中-高 | 沙箱测试 + 人工审批 |
| 改自己的源代码 | 高 | 隔离执行 + 能力评估 + HITL |
| 改训练数据/权重 | 极高 | 全套 RSP + 形式化验证 |
